Just like a physical storage drive can become fragmented, virtual memory can also suffer from **fragmentation**.

In virtual memory, fragmentation happens when your system's memory becomes broken up into a messy patchwork of small,
scattered chunks. Even if you have plenty of _total_ free memory, it is so disorganized that the computer struggles to use it
efficiently.

There are two distinct types of fragmentation in virtual memory: **Internal** and **External**.

---

## 1. Internal Fragmentation

Internal fragmentation happens **inside** an allocated block of memory.

Virtual memory is managed in fixed-size blocks called **Pages** (almost always 4 KB in size). When a program asks for memory,
the operating system assigns it a page.

- **The Problem:** If a program only needs 1 KB of data to store a quick variable, the operating system still has to give it
  a full 4 KB page.
- **The Result:** The remaining 3 KB inside that page goes completely unused, but it is locked by that program. No other
  software can touch it. This wasted, trapped space inside the page is internal fragmentation.

---

## 2. External Fragmentation

External fragmentation happens **between** the allocated blocks of memory over time as programs open and close.

- **The Problem:** Imagine your virtual memory space is like a long parking lot. As applications launch, they take up
  consecutive spaces. When you close a few random applications, they leave behind empty "holes" of free memory scattered all
  over the lot.
- **The Twist:** Suppose you have five scattered 1 MB holes of free memory (5 MB total free space). If you try to launch a
  new program that requires a single, continuous **3 MB block** of memory, the system might fail to allocate it—even though
  you have 5 MB free—because the free space isn't contiguous (connected together).

---

## How It Affects Your System

When virtual memory becomes heavily fragmented, you will notice a distinct drop in your computer's performance:

- **Increased Page Faults:** The CPU has to work much harder to map fragmented virtual addresses to actual physical RAM. This
  triggers more "page faults," forcing the system to constantly access the slow SSD/hard drive.
- **Memory Allocation Failures:** Apps might crash or give you "Out of Memory" errors, even if your task manager claims you
  still have free RAM/swap space available.
- **System Stuttering:** The Operating System has to constantly spend CPU cycles trying to clean up and reorganize the memory
  layout instead of running your apps.

---

## How the Operating System Fixes It

You don't need to run a "defrag" tool on your RAM or virtual memory; the operating system handles this automatically behind
the scenes using a few clever techniques:

1. **Paging:** By breaking memory down into uniform 4 KB pages rather than random, arbitrary sizes, modern operating systems
   almost entirely eliminate _external_ fragmentation. Even if the physical pages are scattered all over your RAM and SSD,
   the CPU's **Memory Management Unit (MMU)** uses a page table to seamlessly stitch them back together into a neat,
   continuous line for the software.
2. **Compaction:** The OS will occasionally pause for a microsecond to shuffle active memory blocks around, sliding them all
   to one side to merge a bunch of small, scattered holes into one large, usable block of free space.
