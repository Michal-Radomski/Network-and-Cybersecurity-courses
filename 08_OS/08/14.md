In a computer, your running programs need memory (RAM) to store data. However, programs don't access your actual, physical
RAM sticks directly. Instead, the Linux kernel creates an illusion for every program called **Virtual Memory**.

A **Page Table** is the internal data structure used by the kernel to map this illusion (Virtual Memory) to reality (Physical
RAM).

---

## The Core Concept: Pages and Frames

To manage memory efficiently without it becoming a chaotic mess, the kernel chops memory up into fixed-size blocks:

- **Pages:** Chunks of _virtual_ memory allocated to a program (usually **4 KB** in size).
- **Frames:** Chunks of actual, _physical_ RAM hardware.

When a program wants to read data from a specific virtual page, the CPU and the kernel look at the **Page Table** to see
which physical frame that data is actually sitting in.

---

## How the Translation Works (The MMU)

Every time a program executes an instruction to read or write memory, a physical component inside your CPU called the **MMU
(Memory Management Unit)** handles the translation.

1. The program requests data at a **Virtual Address**.
2. The MMU intercepts this request and looks up the active **Page Table** managed by the kernel.
3. The Page Table translates the virtual address into a **Physical Address**.
4. The CPU grabs the data from the actual RAM chip.

```text
 ┌─────────────────┐       ┌─────────────────┐       ┌──────────────────┐
 │ Program requests│ ────> │  MMU looks at   │ ────> │ Data read from   │
 │ Virtual Address │       │   Page Table    │       │ Physical Address │
 └─────────────────┘       └─────────────────┘       └──────────────────┘

```

To make this blazing fast, the CPU caches the most recently used translations in a super-fast hardware buffer called the
**TLB (Translation Lookaside Buffer)**.

---

## Multi-Level Page Tables (The Tree Structure)

If the kernel kept a simple flat list of every single page mapping for a modern 64-bit operating system, the Page Table
itself would take up gigabytes of RAM just to exist!

To fix this, the Linux kernel uses **Multi-Level Page Tables**, which structured like a tree.

On standard 64-bit Linux architectures, this is typically a 4-level or 5-level hierarchy:

1. **PGD** (Page Global Directory) - The root of the tree.
2. **P4D** (Page 4th Directory) - Used in 5-level paging.
3. **PUD** (Page Upper Directory)
4. **PMD** (Page Middle Directory)
5. **PTE** (Page Table Entry) - The final leaf that points directly to the physical RAM frame.

**Why do this?** If a program only uses a tiny bit of memory, the kernel only needs to create the top-level branches of the
tree. The unneeded lower branches are never created, saving a massive amount of system RAM.

---

## Extra Superpowers of Page Tables

Page Tables don't just hold memory addresses; each entry contains special **attribute bits** that give the kernel total
control over security and stability:

- **Present Bit:** Tells the CPU if the data is actually in RAM. If it’s `0`, the data might have been moved to your hard
  drive to save space (Swap space). This triggers a **Page Fault**, telling the kernel to freeze the program for a
  millisecond, pull the data back into RAM, and resume.
- **Read/Write Bit:** Marks memory as read-only or read-write. If a program tries to overwrite code it shouldn't touch, the
  CPU triggers a **Segmentation Fault** and crashes the app safely.
- **NX Bit (No-Execute):** Marks areas of memory (like where your variables are stored) as un-runnable. This stops hackers
  from injecting malicious code into your variables and forcing the CPU to run it.
