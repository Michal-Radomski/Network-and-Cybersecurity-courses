In x86 and x86-64 computer architectures, **CR3** and **CR4** are specific **Control Registers** inside the CPU. They act as
the primary control switches for the **MMU (Memory Management Unit)**, dictating exactly how the kernel translates virtual
memory into physical RAM.

If virtual memory is a map, CR3 tells the CPU _where_ the map is located, and CR4 tells the CPU _how_ to read it.

---

## 1. The CR3 Register: The Page Directory Base Register (PDBR)

The **CR3** register is the heart of memory translation for every individual process running on your system. Its primary job
is to hold the **physical memory address of the very first page table** (the root of the multi-level page table tree, known
as the Page Global Directory or PGD).

### How it works during context switching:

Every time your operating system switches from running one app (like Discord) to another (like VS Code), a **context switch**
occurs.

1. The kernel looks up the memory map for the new program.
2. It overwrites the CR3 register with the physical address of the new program's root page table.
3. Instantly, the CPU's MMU starts using the new map.

This is the exact mechanism that ensures Program A cannot see or modify the memory of Program B—their CR3 registers point to
entirely different places in physical RAM.

> **TLB Flashing Note:** Writing a new address into the CR3 register automatically tells the CPU to invalidate (flush) its
> **TLB (Translation Lookaside Buffer)** cache, because the old virtual-to-physical address mappings are no longer valid for
> the new program.

---

## 2. The CR4 Register: Architectural Feature Control

While CR3 holds data for specific programs, **CR4** is a global configuration register. It contains a collection of
single-bit switches (flags) that turn advanced CPU hardware features on or off. The kernel sets these bits during the
computer's boot sequence.

Several critical bits inside CR4 directly change how the MMU handles virtual memory:

### Key MMU-related Bits in CR4:

- **PAE (Physical Address Extension - Bit 5):** When enabled, this changes the page table structure from a 2-level system to
  a multi-level system, historically allowing 32-bit systems to access more than 4 GB of physical RAM.
- **PGE (Page Global Enable - Bit 7):** Enables "Global Pages." When the kernel switches tasks and updates CR3, it normally
  clears the TLB cache. If a page table entry is marked as "global" and the PGE bit is on, the CPU will _not_ clear that
  specific mapping. This is used for kernel memory, so the OS doesn't waste time reloading its own memory map during context
  switches.
- **PSE (Page Size Extension - Bit 4):** Allows the MMU to use "Huge Pages" (like 2 MB or 4 MB chunks instead of the standard
  4 KB chunks), which drastically speeds up memory performance for database engines and large applications.
- **PCIDE (PCID Enable - Bit 17):** Turns on Process-Context Identifiers. This tags TLB cache lines with a "Process ID
  number," allowing the CPU to keep cached memory maps for multiple programs simultaneously without needing to flush the TLB
  during a context switch.

## Summary: How They Work Together

When a program requests a piece of data, the MMU jumps into action using both registers:

```text
               ┌─────────────────────────────────────┐
               │         CPU reads CR4 Bits          │
               │ (Checks if PAE, PGE, PCID are on)   │
               └──────────────────┬──────────────────┘
                                  │
                                  ▼
               ┌─────────────────────────────────────┐
               │         MMU looks at CR3            │
               │  (Finds the root Page Table address)│
               └──────────────────┬──────────────────┘
                                  │
                                  ▼
 ┌──────────────────┐      ┌──────┴──────┐      ┌───────────────────┐
 │ Virtual Address  │ ───> │     MMU     │ ───> │ Physical RAM Location│
 │ from Application │      │ Translation │      │   Accessed        │
 └──────────────────┘      └─────────────┘      └───────────────────┘

```
