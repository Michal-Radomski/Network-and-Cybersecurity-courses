Because bytecode is meant for a virtual machine and not humans, it is technically just a stream of numeric bytes (1s and 0s,
usually represented as hexadecimals).

However, language runtimes have tools that translate those raw bytes into a readable **assembly-like text format** so
developers can inspect what the computer is actually doing.

Let’s look at a concrete example using **Python**, which compiles code into bytecode before running it.

---

## The Example: Adding Two Numbers

Imagine you write this simple Python function:

```python
def add(a, b):
    return a + b

```

### 1. What the raw bytecode actually looks like:

If you were to open the compiled Python file (`.pyc` file) in a hex editor, the raw bytecode for that specific function looks
like a sequence of numbers like this:

`97 00 7c 00 7c 01 16 00 53 00`

The computer reads these numbers instantly, but they don't mean much to us.

### 2. What the readable bytecode looks like:

Python has a built-in tool called `dis` (short for _disassemble_) that turns those raw numbers into human-readable text. If
we run that tool on our function, this is exactly what it outputs:

```text
  1           0 RESUME                   0

  2           2 LOAD_FAST                0 (a)
              4 LOAD_FAST                1 (b)
              6 BINARY_OP                0 (+)
             10 RETURN_VALUE

```

---

## How to Read This Bytecode

The Python Virtual Machine (PVM) is a **stack-based architecture**. Think of a stack like a physical stack of plates—you can
only add to the top or take from the top.

Here is exactly how the Virtual Machine reads those bytecode instructions line-by-line:

- **`LOAD_FAST 0 (a)`**: Take the value of variable `a` and push it onto the top of the stack.
- **`LOAD_FAST 1 (b)`**: Take the value of variable `b` and push it onto the stack (now both `a` and `b` are sitting on the
  stack).
- **`BINARY_OP 0 (+)`**: Pop the top two values off the stack (`a` and `b`), add them together, and push the result back onto
  the top of the stack.
- **`RETURN_VALUE`**: Grab whatever is currently sitting on top of the stack (the result of the addition) and hand it back to
  whoever called the function.

Every time you run a Python script, thousands of these micro-instructions are being generated and processed in milliseconds
right under the hood!

Because JavaScript was originally built to run inside web browsers, its bytecode system is incredibly sophisticated. Modern
JavaScript engines—like Google Chrome's **V8 engine**—don't just interpret bytecode; they use it as a launching pad to
generate blazing-fast machine code on the fly.

Let’s look at how JavaScript handles the exact same scenario: adding two numbers.

---

## The Example: Adding Two Numbers in JS

Imagine you write this JavaScript function:

```javascript
function add(a, b) {
  return a + b;
}
```

### What V8's Bytecode Looks Like

If you peer into Chrome's V8 engine (using special developer flags), the bytecode generated for this function looks like
this:

```text
Parameter count 3
Register count 0
   0: Ldar a1
   2: Add a0, [0]
   5: Return

```

---

## How to Read V8 Bytecode

Unlike Python, which uses a "stack" (like a stack of plates), Chrome's V8 engine uses an **accumulator register** system.
Think of the accumulator as a temporary notepad where the CPU keeps the result of the very last thing it calculated.

Here is what the V8 engine is doing line-by-line:

- **`Ldar a1`**: (Short for **L**oa**d** **A**ccumulator from **R**egister). This takes the second argument (`b`, stored in
  register `a1`) and writes it down on the temporary notepad (the accumulator).
- **`Add a0, [0]`**: This takes the first argument (`a`, stored in register `a0`), adds it to whatever is currently written
  on the notepad, and updates the notepad with the new total.
- **`Return`**: This grabs the final number written on the notepad and returns it.

_(Note: You might wonder why the parameter count is 3 when we only passed `a` and `b`. JavaScript automatically adds an
invisible third parameter called `this` to every function!)_

---

## The Cool Part: V8's "Just-In-Time" (JIT) Magic

JavaScript bytecode has a superpower that Python bytecode usually doesn't: **Optimization Feedback**.

Look at that `[0]` in the `Add a0, [0]` instruction. That is a feedback vector. Because JavaScript doesn't know what data
types `a` and `b` are until the code actually runs, it uses that slot to watch your code:

1. If you call `add(2, 3)`, the engine says, _"Ah, these are integers,"_ and optimizes the bytecode to handle numbers.
2. If you call that function 10,000 times with integers, a secondary compiler inside V8 called **TurboFan** kicks in. It
   says, _"This function only ever uses integers. Let's completely bypass the bytecode and compile this straight into raw
   x86/ARM Machine Code so it runs at maximum hardware speed."_

This dynamic switching between source code, bytecode, and raw machine code is the main reason modern web applications like
Google Maps or browser-based games can run so smoothly!
