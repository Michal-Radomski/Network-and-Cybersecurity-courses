**`kexec`** (short for **kernel execution**) is a powerful feature in the Linux kernel that allows you to boot into a new
Linux kernel directly from a currently running one—completely bypassing your computer's BIOS or UEFI firmware phase.

In simple terms, it lets you reboot your operating system without actually resetting the hardware.

---

## How a Normal Boot vs. kexec Works

To understand why `kexec` is so special, look at how the restart process changes when you use it:

### The Normal Reboot Process:

1. You trigger a reboot.
2. The operating system shuts down.
3. The motherboard hardware resets.
4. The system loads the **BIOS/UEFI** firmware.
5. The firmware performs hardware checks (POST - Power-On Self-Test) and initializes device controllers.
6. The bootloader (like GRUB) is loaded.
7. The bootloader finally launches the Linux kernel.

- **The Problem:** Steps 3 through 5 can take a long time—sometimes several minutes on large enterprise servers with massive
  amounts of RAM and hardware arrays.

### The kexec Reboot Process:

1. You load a new kernel image directly into your system's memory while the current OS is still running.
2. You trigger `kexec`.
3. The current kernel shuts down your active software services safely.
4. Instead of resetting the hardware, the running kernel hands off total control of the CPU and memory directly to the new
   kernel.
5. The new kernel boots up instantly.

```text
Normal Boot: [OS Shutdown] ──> [Hardware Reset] ──> [BIOS/UEFI] ──> [Bootloader] ──> [New Kernel Boots]
                                      ▲                              ▲
                                      └─────── Bypassed by kexec ────┘
kexec Boot:  [OS Shutdown] ───────────────────────────────────────────────> [New Kernel Boots]

```

---

## What is kexec used for?

Because it skips the hardware initialization phase, `kexec` cuts reboot times down to just a few seconds. It is primarily
used in two major scenarios:

### 1. Rapid Server Upgrades (Zero-Downtime)

In large data centers or cloud environments, every second a server is offline costs money. If a system administrator needs to
apply a critical security patch to the Linux kernel, they can use `kexec` to reboot into the updated kernel in a blink of an
eye, minimizing disruptions to hosted websites or apps.

### 2. Kernel Crash Dumping (`kdump`)

This is the most common automated use of `kexec`. When a Linux server suffers a critical error and crashes (a "kernel
panic"), it can't rely on the crashed kernel to save a log file to the hard drive because the system is corrupted.

Instead, the system uses `kexec` to instantly jump into a secondary, lightweight "capture" kernel that was pre-loaded into a
reserved slice of RAM. This backup kernel boots up safely and copies the memory contents of the crashed system into a crash
dump file (`vmcore`) so developers can analyze exactly what went wrong.

## How to use it (The basic concept)

Using `kexec` manually on Ubuntu involves a user-space tool called `kexec-tools`.

First, you "load" the new kernel and the initial ramdisk (`initrd`) into memory:

```bash
sudo kexec -l /boot/vmlinuz-$(uname -r) --initrd=/boot/initrd.img-$(uname -r) --reuse-cmdline

```

Then, when you are ready to switch, you execute it:

```bash
sudo kexec -e

```

Your screen will flicker for a moment, and suddenly you will be sitting right back at the login screen of a freshly booted
system without your computer ever turning off.
