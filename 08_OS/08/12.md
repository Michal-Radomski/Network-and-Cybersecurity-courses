**OverlayFS** is a modern file system service for Linux that allows you to combine two separate directories into a single,
unified directory.

To the user, it looks like a single normal folder, but behind the scenes, it is actually stacking one directory on top of
another. This is called a **union mount**.

It is the secret sauce behind how technologies like **Docker containers** and **Live Linux USB drives** work so efficiently.

---

## How OverlayFS Works: The 4 Layers

When you set up an OverlayFS, you define four distinct components:

```text
 ┌─────────────────────────────────────────────────────────┐
 │                  Unified Directory                      │  <- What the user sees
 └───────────────────────────┬─────────────────────────────┘
                             │
              ┌──────────────┴──────────────┐
              ▼                             ▼
   ┌────────────────────┐        ┌────────────────────┐
   │    Upper Dir       │        │    Lower Dir       │
   │  (Read-Write)      │        │    (Read-Only)     │
   └──────────┬─────────┘        └────────────────────┘
              │
              ▼
   ┌────────────────────┐
   │    Work Dir        │  <- Internal bookkeeping
   └────────────────────┘

```

1. **Lower Directory (`lowerdir`):** This is the base layer. It is completely **read-only**. It contains your core files
   (like a base operating system or template files) that you never want to accidentally change or delete.
2. **Upper Directory (`upperdir`):** This is the **read-write** layer. Any new files you create, or modifications you make to
   existing files, are saved _only_ in this directory.
3. **Unified Directory (`merged`):** This is the final folder mounted by Linux. When you look inside, you see a combined view
   of both the `lowerdir` and `upperdir` seamlessly blended together.
4. **Work Directory (`workdir`):** An empty directory used internally by Linux for bookkeeping and preparing files before
   they hit the upper directory.

---

## How Files are Handled (Read, Write, Delete)

OverlayFS uses clever rules to decide what happens when you interact with the unified folder:

- **Reading a file:** If a file exists in both layers, OverlayFS will always show you the version from the **Upper Dir**,
  hiding the lower one. If it only exists in the lower layer, it reads it from there.
- **Modifying a file:** If you edit a read-only file from the lower layer, OverlayFS performs a **Copy-Up** operation. It
  instantly duplicates the file into your upper layer and applies your changes there. The original file in the lower layer
  remains untouched.
- **Deleting a file:** Since it can't delete files from the read-only lower layer, it creates a special hidden marker called
  a **whiteout** file in the upper layer. This marker acts as a blind spot, telling the system: _"Pretend this file doesn't
  exist anymore."_

## Why is OverlayFS so popular?

The main benefit of OverlayFS is **efficiency and speed**.

### 1. The Docker Example 🐳

When you run 10 identical Ubuntu containers on a machine using Docker, Docker doesn't copy the entire Ubuntu operating system
10 times. Instead, it loads _one_ copy of Ubuntu into the **Lower Dir** and gives each container its own tiny, empty **Upper
Dir**. All 10 containers share the base operating system, saving massive amounts of disk space.

### 2. Live Boot USBs 💾

When you boot Ubuntu from a USB stick to try it out, the USB hardware itself is treated as a read-only **Lower Dir**. Any
files you create or settings you change while testing are written dynamically to your computer's RAM (acting as the **Upper
Dir**). When you shut down, the RAM clears out, leaving the USB completely pristine for the next boot.
