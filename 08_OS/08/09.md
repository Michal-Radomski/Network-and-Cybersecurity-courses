In JavaScript, you don't have to manually allocate memory (like saying "give me space for a new variable") or manually free
memory when you're done. The JavaScript engine handles this automatically through a process called **Garbage Collection
(GC)**.

The main job of the garbage collector is to monitor memory allocation, figure out when a piece of allocated memory is no
longer needed, and reclaim it so the computer doesn't run out of RAM.

---

## The Core Concept: Reachability

How does JavaScript know if a piece of memory is "no longer needed"? It uses a concept called **reachability**.

A value is considered "reachable" if it can be accessed or used by your program. If a value is unreachable, it is considered
garbage, and the engine deletes it.

There's a base set of values that are _always_ reachable and can never be deleted. These are called **roots**:

- The global variables (e.g., the `window` object in a browser or `global` in Node.js).
- Variables and parameters in the current execution stack (the functions currently being called).

Any other object is reachable if it can be found by starting at a root and following a chain of references.

---

## The Main Algorithm: Mark-and-Sweep

Modern JavaScript engines (like Chrome's V8) use an algorithm called **Mark-and-Sweep** to clean up memory. It runs
periodically in the background and follows two main phases:

### 1. The "Mark" Phase

The garbage collector starts at the **roots** and "paints" or marks them as active. It then looks at all properties inside
those roots and marks the objects they point to. It keeps traveling down the chain, marking every single object it can
physically reach.

### 2. The "Sweep" Phase

Once the collector has visited every reachable path, it scans all the memory in the system. Any object that was **not
marked** during the first phase is deemed unreachable. The collector wipes these unmarked objects from memory, freeing up
that space for future variables.

---

## A Quick Example of Garbage Collection in Action

Look at this simple code snippet:

```javascript
let user = { name: "Alice" }; // Memory is allocated for the object

user = null; // The reference is broken!
```

1. Initially, the global variable `user` points to the object `{ name: "Alice" }`. Because `user` is a root, the object is
   **reachable**.
2. When we set `user = null`, the link is cut.
3. The next time the Mark-and-Sweep algorithm runs, it starts at the roots. Since `user` no longer points to
   `{ name: "Alice" }`, the algorithm cannot reach that object.
4. The object is left unmarked and gets **swept away** (deleted) from memory.

---

## The Real-World Complexity: Generational GC

Because stopping your entire website to scan all memory can cause lag (known as "jank"), modern JS engines split memory into
two main zones:

- **The Young Generation:** This is where new objects are born. Most objects in JavaScript live very short lives (like a
  temporary variable inside a quick `for` loop). The garbage collector cleans this zone incredibly fast and very frequently.
- **The Old Generation:** If an object survives a few rounds of cleanups in the Young Generation, it gets "promoted" to the
  Old Generation. The engine scans this zone much less frequently because it assumes these objects are long-term (like a user
  configuration object that stays active the entire session).
