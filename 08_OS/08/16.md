Since you just learned about how the MMU uses the **CR3 register** and **Page Tables** to translate virtual memory into
physical RAM, you are in the perfect position to understand the **Translation Lookaside Buffer (TLB)**.

In short: The TLB is a super-fast hardware cache built directly into the CPU. Its sole purpose is to make virtual memory
translation happen at lightning speed.

---

## The Problem: Page Tables are Too Slow

As we discussed, when a program wants to read a variable from memory, the CPU's MMU has to look through a multi-level page
table (PGD $\rightarrow$ PUD $\rightarrow$ PMD $\rightarrow$ PTE) to find the physical RAM address.

Because the page tables themselves are stored in your system RAM, the CPU has to make **4 or 5 separate memory lookups** just
to find the address of the _one_ piece of data the program actually asked for!

Reading RAM is incredibly slow compared to how fast the CPU chip runs. If the CPU had to do this 5-step walk every single
time it accessed memory, your computer would crawl at a snail's pace.

---

## The Solution: The TLB Cache

The **TLB** solves this by acting like a "cheat sheet" or an address book right inside the CPU chip.

The very first time a program accesses a page of memory, the MMU does the slow, manual walk through the page tables. But once
it finds the physical address, it takes that translation (Virtual Page $\rightarrow$ Physical Frame) and stores it in the
**TLB**.

The next time the program asks for data on that same page, the MMU skips the page tables entirely. It looks at the TLB, finds
the answer instantly, and grabs the data.

### TLB Hit vs. TLB Miss

- **TLB Hit:** The translation is already cached in the TLB. The CPU gets the physical address instantly (usually in less
  than 1 clock cycle). This happens roughly **99% of the time** in modern computers.
- **TLB Miss:** The translation is _not_ in the TLB. The MMU must do the slow "page table walk" in RAM to find the address,
  and then it saves that result into the TLB for next time.

---

## How the TLB Connects to CR3 (The Context Switch Problem)

Remember how we said the **CR3 register** points to the root page table of the currently running program?

When the operating system switches from running Discord to running VS Code, the kernel changes the value in CR3. Because VS
Code has a completely different memory map, **all the translations currently sitting in the TLB become completely wrong.**

Historically, the moment CR3 was updated, the CPU had to perform a **TLB Flush** (wipe the entire cache clean). The newly
running program would suffer from a barrage of slow "TLB Misses" until the TLB got filled up again.

### The Modern Fix: PCIDs (Controlled by CR4)

To stop this slowdown, modern CPUs use a feature called **PCID (Process-Context Identifiers)**, which is turned on via the
**CR4 register**.

With PCID, the TLB doesn't just store the memory translation; it also tags it with a number identifying _which program_ it
belongs to (e.g., "This translation belongs to Process #3"). When the CPU switches tasks, it doesn't have to wipe the TLB! It
just ignores the tags belonging to other programs, keeping your computer running incredibly smooth.

---

## Quick Analogy

Imagine you are a librarian:

- **Physical RAM** is the massive archive room in the back of the building.
- **The Page Table** is the massive card catalog system you have to flip through to find which shelf a book is on. Walking to
  the catalog and flipping cards for every request takes forever.
- **The TLB** is a small sticky-note pad on your desk. When someone asks for a book, you look at the catalog once, but you
  write the book's exact location on a sticky note. The next 50 people who ask for that same book get their answer in one
  second because you just look at your desk sticky note.
