`gdb` stands for the **GNU Debugger**. It is a powerful command-line tool used in Linux and other Unix-like systems to
inspect what is happening inside a program while it executes—or what it was doing at the moment it crashed.

Think of it as an X-ray machine for your code (most commonly used for C and C++ programs).

---

## Why Use GDB?

When your code has a bug, adding `printf()` statements everywhere can be slow and messy. `gdb` allows you to:

- **Pause execution:** Stop the program at any specific line of code.
- **Inspect variables:** View or change the value of variables on the fly.
- **Trace crashes:** Find out exactly which line of code caused a "Segmentation Fault."
- **Step through code:** Run the program line-by-line to see exactly how data changes.

---

## How to Use It (The Basic Workflow)

### 1. Compile with Debug Symbols

Before using GDB, you must compile your code with the `-g` flag. This tells the compiler to store extra information (like
variable names and line numbers) that GDB needs.

```bash
gcc -g my_program.c -o my_program

```

### 2. Start GDB

Launch GDB by passing your compiled program as an argument:

```bash
gdb ./my_program

```

### 3. Essential GDB Commands

Once inside the GDB prompt (which looks like `(gdb)`), you use short commands to control your program:

| Command                     | Shortcut | What it does                                                                                                |
| --------------------------- | -------- | ----------------------------------------------------------------------------------------------------------- |
| **`run`**                   | `r`      | Starts running your program from the beginning.                                                             |
| **`break [line/function]`** | `b`      | Sets a **breakpoint**. The program will pause here. (e.g., `b 15` or `b main`).                             |
| **`next`**                  | `n`      | Runs the next line of code, but steps _over_ functions.                                                     |
| **`step`**                  | `s`      | Steps _into_ the next line of code (goes inside functions).                                                 |
| **`continue`**              | `c`      | Resumes running the program until the next breakpoint or the end.                                           |
| **`print [variable]`**      | `p`      | Shows the current value of a variable (e.g., `p counter`).                                                  |
| **`backtrace`**             | `bt`     | Shows the "call stack" (the chain of functions that led to the current moment—crucial for finding crashes). |
| **`quit`**                  | `q`      | Exits GDB.                                                                                                  |

---

## A Quick Example

Imagine your program is crashing. You would open it in GDB and type `run`. When it crashes, GDB will stop and show you the
exact line where it failed.

You can then type `bt` to see how it got there, or `p my_variable` to see if a variable holds an unexpected value (like
`NULL` or a giant random number).
