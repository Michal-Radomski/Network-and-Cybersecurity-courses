In computer science, a **stack overflow** occurs when a program running on a computer uses more memory in its execution stack
than has been allocated for it. To understand how this happens, we need to look at how a process organizes its memory.

---

## 1. Process Memory Layout

When a program runs, the operating system allocates a block of memory for it. This memory is typically divided into several
distinct segments:

- **Text (Code) Segment:** Stores the actual compiled machine code instructions.
- **Data Segments:** Stores global and static variables.
- **Heap:** A large pool of memory used for dynamic allocation (e.g., using `malloc` in C or `new` in Java/C++). It typically
  grows _upward_ (toward higher memory addresses).
- **Stack:** A structure used for tracking function calls, local variables, and control flow. It typically grows _downward_
  (toward lower memory addresses).

---

## 2. How the Stack Works

The stack operates on a **Last-In, First-Out (LIFO)** basis. Every time a function is called, a new chunk of memory called a
**stack frame** (or activation record) is pushed onto the top of the stack.

A stack frame typically contains:

- The function's local variables.
- The arguments passed into the function.
- The **return address** (where the CPU should jump back to once the function finishes).

When the function finishes executing, its stack frame is popped off the stack, and the memory is freed up for the next
function call.

---

## 3. The Anatomy of a Stack Overflow

Because the stack and the heap share the same general pool of memory, they grow toward each other. The operating system sets
a strict architecture-defined limit on how large the stack can grow.

A **stack overflow** happens when the stack pointer exceeds this allocated boundary, colliding into another memory segment or
unmapped memory addresses.

> 💥 **What happens next?** The operating system immediately steps in to protect the system's integrity, terminating the
> program with a crash error—most commonly a **Segmentation Fault (SIGSEGV)**.

---

## 4. Primary Causes of Stack Overflow

### A. Infinite or Deep Recursion

This is the most common culprit. If a function calls itself (recursion) without a proper base case to stop it, it will keep
pushing new stack frames indefinitely until the stack runs out of space.

```c
// Example of infinite recursion in C
void causeOverflow() {
    int localVariable = 5;
    causeOverflow(); // No exit condition, will crash the program
}

```

### B. Oversized Local Variables

Allocating massive arrays or structures directly as local variables inside a function can instantly deplete the stack space.

```c
void largeAllocation() {
    // Attempting to allocate ~4MB on the stack.
    // If the stack limit is 1MB, this causes an immediate overflow.
    int hugeArray[1000000];
}

```

---

## Summary of Differences: Stack vs. Heap

| Feature              | The Stack                       | The Heap                                        |
| -------------------- | ------------------------------- | ----------------------------------------------- |
| **Growth Direction** | Downward (usually)              | Upward (usually)                                |
| **Management**       | Automatic (by the CPU/Compiler) | Manual (by the programmer or garbage collector) |
| **Size**             | Small, strict fixed limit       | Large, limited by system RAM                    |
| **Overflow Result**  | **Stack Overflow** (Crash)      | **Out of Memory** error                         |
