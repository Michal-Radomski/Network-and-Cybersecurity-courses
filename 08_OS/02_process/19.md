While JavaScript and TypeScript handle memory management automatically—meaning you rarely have to think about allocating or
freeing bytes manually—these three concepts still absolutely apply.

Because JS/TS run inside an engine (like V8 in Chrome and Node.js), these issues manifest slightly differently than they do
in lower-level languages like C or C++.

---

## 1. Stack Overflow in JS/TS

In JavaScript, the **Call Stack** keeps track of function execution. When a function is called, a new frame is pushed onto
the stack. When the function returns, it is popped off.

A stack overflow happens when the call stack exceeds its maximum size limit set by the browser or Node.js environment.

### How it happens:

This almost always happens due to **infinite recursion** (a function calling itself without a stopping condition).

```typescript
// TypeScript / JavaScript
function standardRecursion(): void {
  standardRecursion(); // High-speed ticket to a stack overflow
}

standardRecursion();
```

### The Result:

The engine will immediately halt execution and throw a specific runtime error:

> ❌ `RangeError: Maximum call stack size exceeded`

---

## 2. Heap Overflow in JS/TS

The **Heap** is where JavaScript stores large, dynamically allocated items like objects, arrays, and closures.

In low-level languages, a heap overflow means writing past a buffer's boundary to corrupt data. In JavaScript, you cannot
directly corrupt memory like that. Instead, a JS "heap overflow" means **the heap memory has completely filled up**, and the
engine has run out of space to allocate new objects.

### How it happens:

Creating massive arrays or objects inside a loop, or pushing immense amounts of data into memory all at once.

```typescript
// TypeScript / JavaScript
const massiveArray: string[] = [];

while (true) {
  // Continuously allocating blocks of strings into the heap
  massiveArray.push("A very long repetitive string that eats up heap memory");
}
```

### The Result:

The V8 engine will crash entirely, often bringing down your Node.js process or crashing the browser tab.

> ❌ `FATAL ERROR: Ineffective mark-compacts near heap limit Allocation failed - JavaScript heap out of memory`

---

## 3. Memory Leaks in JS/TS

JavaScript uses an automatic **Garbage Collector (GC)**. The GC periodically looks for objects that are no longer "reachable"
from the root of the program and frees up their memory.

A memory leak in JS/TS happens when you accidentally keep a reference to an object that you don't need anymore. Because the
reference still exists, the Garbage Collector thinks you still want it and _never_ cleans it up. Over time, your heap memory
grows until you get a heap overflow crash.

### Common Causes in JS/TS:

#### A. Accidental Global Variables

If you assign a value to a variable without declaring it (`const`, `let`, or `var`), JS attaches it to the global `window` or
`global` object. It stays in memory forever.

```javascript
function looseVariable() {
  // Missing 'let' or 'const'. This is now attached to the global object!
  leakedData = new Array(1000000);
}
looseVariable();
```

#### B. Forgotten Event Listeners or Timers

If you set a `setInterval` or add an event listener to a DOM element, but forget to clear it when you're done, the memory
cannot be collected.

```typescript
class DataComponent {
  private data = new Array(1000000);

  constructor() {
    // This timer keeps a reference to 'this' component forever
    setInterval(() => {
      console.log("Doing background work...");
    }, 1000);
  }
}
```

#### C. Closures

A closure happens when an inner function retains access to variables from its outer function. If the inner function is kept
alive, the outer variables are kept alive too.

---

## Summary for JS/TS Developers

| Concept            | What is it in JS/TS?                                                       | The Error You See                                                 |
| ------------------ | -------------------------------------------------------------------------- | ----------------------------------------------------------------- |
| **Stack Overflow** | Too many nested function calls. The Call Stack runs out of slots.          | `RangeError: Maximum call stack size exceeded`                    |
| **Heap Overflow**  | The V8 engine completely runs out of memory for objects/arrays.            | `JavaScript heap out of memory`                                   |
| **Memory Leak**    | Unused objects holding onto references, hiding from the Garbage Collector. | No immediate error; slow performance slowdown leading to a crash. |
