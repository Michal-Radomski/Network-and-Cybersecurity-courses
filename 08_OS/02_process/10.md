When a program is running, the operating system allocates a chunk of memory for the process. One of the most critical parts
of this memory layout is the **Stack**.

The stack is a dynamic memory structure that manages function calls, local variables, and the flow of execution using a
**Last-In, First-Out (LIFO)** approach.

---

## 1. The Anatomy of a Process Memory

To understand the stack, it helps to see where it lives. A typical process memory layout is divided into four main segments:

- **Text (Code):** The compiled machine instructions.
- **Data/BSS:** Global and static variables.
- **Heap:** Dynamically allocated memory (managed by the programmer via `malloc` or `new`).
- **Stack:** Automatically managed memory for function execution.

> 💡 **Growth Direction:** In most modern architectures (like x86), the Heap grows _upward_ (to higher memory addresses),
> while the Stack grows _downward_ (from high memory addresses toward lower ones).

---

## 2. What is a Stack Frame?

Every time a function is called, a new chunk of memory called a **Stack Frame** (or Activation Record) is "pushed" onto the
top of the stack. This frame contains everything that specific function needs to execute.

A typical stack frame includes:

1. **Function Arguments:** Parameters passed into the function by the caller.
2. **Return Address:** The memory address of the instruction the CPU should jump back to once the function finishes.
3. **Saved Frame Pointer:** A reference to the previous stack frame so the CPU can restore the caller's environment.
4. **Local Variables:** Variables declared inside the function.

---

## 3. Step-by-Step Execution Walkthrough

Let's look at a simple execution example:

```c
void funcB() {
    int y = 20;
}

void funcA() {
    int x = 10;
    funcB();
}

int main() {
    funcA();
    return 0;
}

```

### Step 1: `main()` starts

The OS allocates a stack frame for `main()`. The stack pointer points to the top of this frame.

### Step 2: `main()` calls `funcA()`

Before control shifts to `funcA`, the CPU pushes the **return address** (inside `main`) onto the stack. Then, a new stack
frame for `funcA` is created, allocating space for the local variable `x`.

### Step 3: `funcA()` calls `funcB()`

The return address (inside `funcA`) is saved. A new stack frame for `funcB` is pushed onto the top of the stack, allocating
space for `y`.

### Step 4: `funcB()` finishes (The Unwind)

Once `funcB` hits its closing brace, its stack frame is **popped** (destroyed). The CPU reads the saved return address to
jump back into `funcA` exactly where it left off.

### Step 5: `funcA()` finishes

`funcA`'s frame is popped, control returns to `main`, and eventually, the entire stack is cleared when the program exits.

---

## 4. Key CPU Registers Managing the Stack

The CPU relies on two dedicated hardware registers to keep track of this process:

- **Stack Pointer (SP / ESP / RSP):** Points to the very top (current active address) of the stack. It moves every time data
  is pushed or popped.
- **Base/Frame Pointer (BP / EBP / RBP):** Stays fixed at the base of the current stack frame. This gives the CPU a stable
  reference point to find local variables and arguments using offsets (e.g., `BP - 4`).

---

## 5. What Can Go Wrong? (Stack Overflow)

Because the stack has a fixed size allocated by the OS, it can run out of space.

The most common cause is **infinite recursion** (a function calling itself without a base case). Each call adds a new stack
frame until the stack collides with the heap or exceeds its boundaries, causing the notorious **Stack Overflow** error and
crashing the program.
