When a program runs, the operating system allocates a specific chunk of memory for its process. This memory is organized into
a structured layout to manage data efficiently.

Here is how a typical process memory layout looks, with a deep dive into the two most talked-about regions: the **Stack** and
the **Heap**.

---

## The Process Memory Layout Overview

A standard process memory layout is generally divided into four main segments:

1. **Text Segment (Code):** Contains the compiled machine code (the actual instructions) executed by the CPU. This area is
   usually read-only to prevent a program from accidentally modifying itself.
2. **Data Segment:** Stores global and static variables. It is split into initialized data (variables with assigned values)
   and uninitialized data (often called BSS).
3. **Heap:** Used for dynamic memory allocation at runtime.
4. **Stack:** Used for storing local variables and managing function calls.

---

### The Stack: Structured and Fast

The **Stack** is a region of memory that operates on a **LIFO (Last In, First Out)** basis. Every time a function is called,
a new "stack frame" is created at the top of the stack to hold that function's local variables, parameters, and the return
address. When the function finishes, its stack frame is completely wiped out (deallocated) automatically.

- **Management:** Handled automatically by the CPU/compiler. You don't have to allocate or free this memory manually.
- **Size:** Fixed and limited. It is usually much smaller than the heap. If a program uses too much stack memory (like an
  infinite recursive function), it triggers a **Stack Overflow**.
- **Access Speed:** Extremely fast because the memory is allocated in a contiguous block, and the CPU utilizes a dedicated
  register (the stack pointer) to manage it.

> **Analogy:** Think of the stack like a stack of dinner plates. You add new plates to the top, and you must remove the top
> plate before you can access the ones underneath.

---

### The Heap: Flexible and Large

The **Heap** is a large, unstructured pool of memory used for **dynamic memory allocation**. If you don't know how much data
your program will need until it actually runs (like reading a user-uploaded file or creating a variable-sized array), you
allocate that memory on the heap.

- **Management:** Handled manually by the programmer (or by a language's runtime environment). In languages like C/C++, you
  use `malloc()` and `free()`. In languages like Java or Python, a **Garbage Collector** automatically cleans up the heap for
  you.
- **Size:** Flexible and large. It is typically limited only by the physical RAM and virtual memory available on the system.
- **Access Speed:** Slower than the stack. Allocating memory requires searching the heap for a free block, and accessing it
  requires following pointers, which can lead to memory fragmentation.

> **Analogy:** Think of the heap like a massive warehouse. When you need space, the warehouse manager finds an open spot of
> the right size and gives you a receipt (a pointer) to find it later. When you're done, you have to tell the manager to
> clear that spot out.

---

## Stack vs. Heap: A Quick Comparison

| Feature            | The Stack                                | The Heap                                                     |
| ------------------ | ---------------------------------------- | ------------------------------------------------------------ |
| **Allocation**     | Automatic (by the compiler/CPU)          | Manual (by the programmer or garbage collector)              |
| **Data Structure** | LIFO (Last In, First Out)                | No specific structure (flexible pool)                        |
| **Speed**          | Very fast                                | Slower                                                       |
| **Size Limit**     | Small (typically a few megabytes)        | Large (dependent on system RAM)                              |
| **Lifetime**       | Tied to the function's scope             | Tied to the life of the application (until explicitly freed) |
| **Risk**           | Stack Overflow (too many function calls) | Memory Leaks (forgetting to free memory)                     |

### Growth Direction

In a classic architecture, the **Stack** and the **Heap** grow toward each other to maximize memory efficiency. The heap
typically grows "up" (from lower memory addresses to higher ones), while the stack grows "down" (from higher memory addresses
to lower ones). If they ever meet, the system runs out of memory.
