At a very fundamental operating system level: C and JS/TS, **they look exactly the same** because both C and JavaScript have
to run on the same computer hardware. However, **how they use and manage that memory internally is completely different.**

Think of C as a bare-bones apartment where you have to manually place every piece of furniture, while JS/TS is a fully
managed luxury hotel where an automated staff (the JS Engine) organizes everything for you behind the scenes.

Here is a breakdown of how the memory model for JS/TS compares to C.

---

## 1. The Core Structure: Same Names, Different Roles

Both C and JS/TS divide memory into a **Stack** and a **Heap**, but what they put in them is vastly different.

### The Stack

- **In C:** The stack stores function frames, primitive variables (`int`, `char`), and actual data pointers. When a function
  finishes, that memory is instantly freed by moving the stack pointer.
- **In JS/TS:** It is called the **Call Stack**. It is used primarily for tracking function execution flow (LIFO). While
  simple primitives (like small integers) and object reference pointers live here temporarily, the stack is kept very
  lightweight.

### The Heap

- **In C:** The heap is an empty canvas. You use `malloc()` to request a specific number of bytes, and it stays there until
  you manually call `free()`.
- **In JS/TS:** The **Memory Heap** is massive, highly structured, and entirely managed by the engine (like V8 in
  Chrome/Node.js). **Almost everything in JavaScript lives on the heap.** Even things you might think are primitives (like
  long strings or floating-point numbers) are often wrapped as objects and thrown onto the heap.

---

## 2. Key Differences in Memory Layout

If we peek inside the process memory while running both languages, the differences become obvious:

| Feature                 | Process Memory in C                                                                                  | Process Memory in JS/TS (e.g., V8 Engine)                                                                                                                                                                                |
| ----------------------- | ---------------------------------------------------------------------------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ |
| **Data Types**          | Highly typed and packed tightly. An array of 5 integers takes exactly 20 bytes contiguous in memory. | Objects are dynamic. The engine creates hidden structures ("Shapes/Maps") in the heap to keep track of changing object properties.                                                                                       |
| **Text (Code) Segment** | Contains the pre-compiled native machine code (binary) of your app.                                  | Initially contains the code for the **JS Engine itself**. The engine reads your text JS file, converts it to bytecode, and uses a JIT (Just-In-Time) compiler to dynamically generate machine code in memory as it runs. |
| **Pointers**            | Direct hardware memory addresses (e.g., `0x7fff5fbff61a`). You can manipulate them directly.         | Hidden from the developer. JS uses "References" which are abstracted pointers managed safely by the runtime.                                                                                                             |
| **Memory Cleanup**      | **Manual:** The developer must explicitly free every bit of heap memory.                             | **Automatic:** Controlled by a Generational **Garbage Collector** that splits the heap into "New Space" (short-lived data) and "Old Space" (long-lived data).                                                            |

---

## 3. Visualizing a Variable in C vs. JS

To see how the layout differs, let’s see what happens when you create a simple object/structure in both languages.

### In C:

```c
struct User {
    int id;       // 4 bytes
    int age;      // 4 bytes
};
struct User u1 = {101, 25};

```

- **Memory Layout:** If allocated on the stack, it occupies exactly 8 sequential bytes. It is incredibly compact and fast.

### In JS/TS:

```typescript
const user = { id: 101, age: 25 };
```

- **Memory Layout:** 1. The Call Stack holds a variable named `user` containing a hidden reference pointer.

2. That pointer points to a complex structure in the **Heap**.
3. In the heap, V8 doesn't just store `101` and `25`. It stores a **Map** pointer (describing the structure of the object), a
   pointer to its properties, a pointer to its elements, and _then_ the actual values.

---

## Summary

JavaScript/TypeScript memory layout is essentially a **highly advanced application (the engine) running inside a C-like
environment.** Because of this layer of abstraction, JS protects you from severe memory errors like corrupting adjacent data
via a heap overflow, but it comes at the cost of using significantly more raw RAM than a C program would to store the exact
same information.
