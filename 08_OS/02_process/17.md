A **heap overflow** occurs when a program writes more data to a buffer allocated on the heap than the buffer was actually
designed to hold.

To understand how this happens and why it is dangerous, let's look at how the heap operates within a process's memory.

---

## 1. What is the Heap?

The heap is a region of a process's memory used for **dynamic memory allocation**. Unlike the stack, which automatically
allocates and frees memory for function calls in a strict order, the heap is a large, unstructured pool of memory.

Programmers use the heap when:

- They don't know how much memory they will need until the program runs.
- They need variables to persist even after the function that created them finishes.

In languages like C and C++, memory on the heap is managed manually using functions like `malloc()`, `calloc()`, and
`free()`. In languages like Java or Python, the runtime environment manages the heap automatically via a garbage collector.

---

## 2. The Anatomy of a Heap Overflow

When you request memory on the heap, the operating system's memory manager assigns a specific chunk of memory and keeps track
of its size using hidden data called **metadata** (often stored right next to the allocated chunk).

A heap overflow happens when a program accepts input from a user or another source and writes it into a heap buffer without
verifying that the input fits. The extra data "spills over" the boundary of the intended buffer.

### What gets corrupted?

Because the heap is a continuous block of memory, spilling out of one buffer means writing into whatever happens to be next
to it. This can overwrite:

1. **Adjacent Program Data:** Changing the values of other variables, which can cause the program to behave erratically or
   crash.
2. **Heap Metadata:** Overwriting the hidden control structures used by `malloc` and `free`. When the program later tries to
   free that memory, the corrupted metadata can cause an immediate crash or allow arbitrary memory manipulation.
3. **Function Pointers:** If the adjacent memory contains pointers to functions, an attacker could overwrite them to redirect
   the program's execution flow.

---

## 3. A Classic Code Example (C)

Here is a conceptual example of how a heap overflow occurs due to unsafe string copying:

```c
#include <stdio.h>
#include <stdlib.h>
#include <string.h>

int main() {
    // Allocate 10 bytes on the heap
    char *buffer = (char *)malloc(10 * sizeof(char));

    // Unsafe user input that is 20 characters long
    char *userInput = "ThisIsAVeryLongInputString!";

    // strcpy does not check boundaries. It will write all 27 characters
    // into the 10-byte buffer, overflowing into adjacent heap memory.
    strcpy(buffer, userInput);

    printf("Buffer content: %s\n", buffer);

    free(buffer); // This will likely crash the program due to corrupted metadata
    return 0;
}

```

---

## 4. Stack Overflow vs. Heap Overflow

While both are buffer overflow vulnerabilities, they happen in different places and behave differently:

| Feature              | Stack Overflow                                                                     | Heap Overflow                                                                                        |
| -------------------- | ---------------------------------------------------------------------------------- | ---------------------------------------------------------------------------------------------------- |
| **Location**         | The Stack segment.                                                                 | The Heap segment.                                                                                    |
| **Trigger**          | Usually caused by infinite recursion or massive local arrays.                      | Caused by copying too much data into a dynamically allocated buffer.                                 |
| **Mechanism**        | Exceeding the total allocated _size_ boundary of the entire stack.                 | Writing past the boundary of an _individual_ allocated chunk.                                        |
| **Immediate Result** | Instant crash (Segmentation Fault) because the stack hits an unmapped memory wall. | May not crash immediately; often corrupts data silently until the memory is accessed or freed later. |

---

## 5. Mitigation: How to Prevent It

Modern software development relies on several defenses to stop heap overflows:

- **Safe Functions:** Replacing dangerous functions like `strcpy` or `gets` with bounds-checking alternatives like `strncpy`
  or `fgets`.
- **Memory-Safe Languages:** Using languages like Rust, Python, or Java, which automatically manage memory boundaries and
  prevent direct, unchecked pointer manipulation.
- **ASLR (Address Space Layout Randomization):** A security technique used by operating systems to randomize where memory
  segments are located, making it much harder for attackers to predict where to redirect code execution if an overflow
  occurs.
