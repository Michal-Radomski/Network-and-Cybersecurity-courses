The **Text Segment** (also frequently called the **Code Segment**) is a specific region of a process’s memory that contains
the compiled, executable machine code instructions of the program.

When you write code in a language like C, C++, or Rust and compile it, or when a runtime environment prepares a program to
run, those human-readable lines are translated into binary machine instructions (0s and 1s) that the CPU can execute. The
Text Segment is where those instructions live while the program is running.

---

## 1. Key Characteristics of the Text Segment

To protect the integrity of the operating system and the running program, the Text Segment has very strict properties:

- **Read-Only:** The operating system marks this memory region as read-only. A program should never modify its own
  instructions while running. If a program attempts to write data into the Text Segment, the OS will instantly terminate it
  with a **Segmentation Fault**.
- **Fixed Size:** The size of the Text Segment is determined when the program is compiled or loaded. Unlike the Stack or the
  Heap, which grow and shrink dynamically as the program runs, the Text Segment remains exactly the same size from the moment
  the app starts until it closes.
- **Sharable:** If multiple instances of the same program are running simultaneously (for example, if you open three separate
  windows of the same web browser), the operating system only needs to load the Text Segment into physical RAM _once_. All
  three processes can map to the exact same read-only memory, saving a massive amount of system memory.

---

## 2. Where It Sits in Process Memory

In the standard layout of a process's virtual memory, the Text Segment is typically placed at the very bottom (the lowest
memory addresses). This positions it safely away from the Stack and Heap, which are dynamic and constantly changing size.

---

## 3. How the CPU Interacts with It

The CPU relies on a special internal register called the **Instruction Pointer (IP)** or **Program Counter (PC)** to navigate
the Text Segment:

1. The Instruction Pointer holds the exact memory address of the next machine instruction inside the Text Segment.
2. The CPU fetches that instruction, decodes it, and executes it.
3. The Instruction Pointer then increments to point to the next instruction sequentially.
4. When your code encounters a loop, an `if` statement, or a function call, the Instruction Pointer "jumps" to a different
   memory address within the Text Segment rather than moving to the next line.

---

## 4. Text Segment vs. Other Memory Segments

To see how the Text Segment fits into the bigger picture, here is how it compares to the parts of memory that handle data:

| Memory Segment           | What it Stores                                                  | Permissions           | Size Behavior           |
| ------------------------ | --------------------------------------------------------------- | --------------------- | ----------------------- |
| **Text (Code)**          | Compiled machine instructions.                                  | Read-Only, Executable | Fixed                   |
| **Data (Static/Global)** | Global variables initialized by the programmer.                 | Read-Write            | Fixed                   |
| **Stack**                | Local variables, function arguments, return addresses.          | Read-Write            | Dynamic (Grows/Shrinks) |
| **Heap**                 | Dynamically allocated objects and data (e.g., `malloc`, `new`). | Read-Write            | Dynamic (Grows/Shrinks) |

---

## 5. Security Context: Preventing Exploits

Historically, hackers exploited a vulnerability called a **buffer overflow** to inject malicious machine code into the
_Stack_ or _Heap_ and trick the CPU into executing it.

To combat this, modern operating systems enforce a security feature called **W^X (Write Inside or Execute, but not both)**.
Memory regions like the Stack and Heap are marked as _Writeable but Non-Executable_, while the Text Segment is marked as
_Executable but Non-Writeable_. This ensures that malicious code injected into data segments cannot be run by the CPU.
