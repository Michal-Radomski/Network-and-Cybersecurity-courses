At the absolute heart of every computer is the **Central Processing Unit (CPU)**, often called the "brain" of the machine.
The CPU doesn't actually understand complex concepts like "video games" or "web browsers"—instead, it breaks everything down
into billions of tiny, microscopic steps.

This continuous loop of processing instructions is called the **CPU Execution Cycle** (also known as the
**Fetch-Decode-Execute Cycle**).

Here is exactly how it works, step-by-step:

---

## The Key Hardware Players

Before looking at the cycle, it helps to know the specific components inside the CPU that make it happen:

- **Program Counter (PC):** A special register (a tiny, lightning-fast memory slot) that holds the memory address of the
  _next_ instruction to be executed.
- **Instruction Register (IR):** Holds the current instruction that was just fetched from memory.
- **Control Unit (CU):** The manager. It directs the flow of data inside the CPU and decodes instructions.
- **Arithmetic Logic Unit (ALU):** The calculator. It handles all the math (addition, subtraction) and logic (AND, OR,
  comparing numbers).
- **RAM (Random Access Memory):** The main memory outside the CPU where programs and data live while the computer is turned
  on.

---

## The 4 Steps of the Cycle

Every single action your computer takes relies on this four-part loop repeating billions of times per second.

### 1. Fetch

The CPU needs to grab its next instruction from the system's RAM.

- The **Control Unit** looks at the memory address currently stored in the **Program Counter (PC)**.
- It copies the instruction located at that address in RAM and moves it into the CPU's **Instruction Register (IR)**.
- Immediately after fetching, the **Program Counter (PC)** automatically increments (points to the next address), preparing
  for the next round.

### 2. Decode

Computers only understand `0`s and `1`s (binary machine code). The fetched instruction is just a string of binary numbers, so
the CPU has to figure out what it actually means.

- The **Control Unit** reviews the instruction in the IR.
- It breaks down the binary code into an **opcode** (what to do, like "ADD" or "MOVE") and an **operand** (what data to do it
  to, or where that data is located).
- The CU then prepares the internal circuitry of the CPU for the next step.

### 3. Execute

This is where the actual work gets done. Depending on what the instruction was, different parts of the CPU fire up:

- If it’s a math or logic problem (e.g., $5 + 3$ or checking if $X > Y$), the instruction is passed to the **ALU**.
- If it's a data movement instruction, the CPU might grab data from one register and copy it to another.
- If it's a "jump" instruction, the Program Counter might be forcefully changed to a completely different memory address
  (which is how `if/else` statements and loops work in code).

### 4. Store (Writeback)

Once the execution is complete, the CPU has to save the result.

- The output generated by the ALU or control unit is written back to a specific CPU register or directly back into the
  computer's **RAM** so it can be used later.

---

## How Fast Does This Happen?

The speed of this cycle is dictated by the **CPU Clock**. Every time the clock "ticks," the CPU moves forward in the cycle.

When you see a processor advertised as **4.0 GHz**, it means the internal clock ticks **4 billion times per second**. While a
single instruction might take a few clock cycles to completely finish its Fetch-Decode-Execute journey, modern CPUs use a
technique called _pipelining_ to work on different stages of thousands of instructions simultaneously.
