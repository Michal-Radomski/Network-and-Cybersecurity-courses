A **memory leak** occurs when a computer program allocates memory on the heap but fails to release (free) it back to the
operating system when it is no longer needed.

Over time, these un-freed blocks of memory accumulate. If a program leaks memory continuously, it will slowly consume more
and more of the computer's available RAM, leading to performance degradation or eventual system crashes.

---

## 1. How Memory Leaks Happen

To understand a memory leak, it helps to look at the lifecycle of dynamic memory allocation:

1. **Allocate:** The program requests a chunk of memory from the heap (e.g., using `malloc()` in C or `new` in C++).
2. **Use:** The program reads from and writes to this memory.
3. **Release:** The program explicitly tells the operating system it is done with the memory (e.g., using `free()` or
   `delete`).

A memory leak happens when step 3 is forgotten or bypassed, but the program loses the pointer (the address) to that memory.
Once the pointer is gone, the program has no way to access or free that memory ever again. It becomes "orphaned."

---

## 2. A Simple Code Example (C++)

Here is a classic example of how a memory leak occurs inside a loop:

```cpp
#include <iostream>

void leakMemory() {
    // 1. Allocate memory for an integer on the heap
    int* ptr = new int(42);

    // 2. Use the memory
    std::cout << "Value: " << *ptr << std::endl;

    // 3. Missing 'delete ptr;'
    // When this function ends, the local pointer 'ptr' is destroyed.
    // However, the memory it pointed to remains allocated on the heap.
}

int main() {
    while (true) {
        leakMemory(); // Running this continuously will quickly drain the system's RAM
    }
    return 0;
}

```

---

## 3. The Consequences of a Memory Leak

Unlike a stack overflow or a heap overflow, a memory leak usually **does not crash the program immediately**. Instead, it
acts like a slow drain:

- **Performance Slowdown:** As the program consumes more RAM, the operating system may be forced to use "swapping" (writing
  memory data to the much slower hard drive/SSD). This makes the entire computer feel sluggish.
- **Out of Memory (OOM) Crashes:** Eventually, the program (or the entire system) will completely run out of available RAM.
  When this happens, the operating system's OOM killer will step in and abruptly terminate the process.
- **Severe for Long-Running Software:** Memory leaks are particularly dangerous for software meant to run indefinitely
  without restarting, such as operating system kernels, web servers, or background system services.

---

## 4. Comparison: Stack Overflow, Heap Overflow, and Memory Leak

| Feature              | Stack Overflow                                        | Heap Overflow                                                 | Memory Leak                                                      |
| -------------------- | ----------------------------------------------------- | ------------------------------------------------------------- | ---------------------------------------------------------------- |
| **Location**         | The Stack                                             | The Heap                                                      | The Heap                                                         |
| **The Core Problem** | Running out of stack space (too many function calls). | Writing data _past the boundaries_ of an allocated block.     | Forgetting to _free_ an allocated block after use.               |
| **Security Risk**    | Low (mostly just crashes the app).                    | **High** (can be exploited by hackers to run malicious code). | Low to Medium (mostly causes instability and Denial of Service). |
| **Timing of Crash**  | Instantaneous.                                        | Can be instant, or can corrupt data silently and crash later. | Very slow and gradual.                                           |

## 5. How Memory Leaks are Prevented

### A. Garbage Collection

Modern programming languages like Java, Python, C#, and JavaScript use an automatic memory management system called a
**Garbage Collector (GC)**. The GC periodically scans the heap, looks for blocks of memory that no longer have any pointers
tracking them, and automatically frees them. While this prevents most leaks, memory leaks can still happen in these languages
if a program accidentally keeps references to objects it no longer needs (e.g., keeping old data in a global list).

### B. Smart Pointers (C++)

Modern C++ introduces "smart pointers" (like `std::unique_ptr` and `std::shared_ptr`). These are wrappers around raw pointers
that automatically delete the underlying heap memory as soon as the pointer goes out of scope, eliminating the human error of
forgetting to call `delete`.

### C. Profiling Tools

Developers use specialized software called memory profilers (like **Valgrind**, **AddressSanitizer**, or built-in IDE tools)
to track down leaks. These tools monitor allocations while the program runs and report exactly which lines of code allocated
memory that was never freed.
