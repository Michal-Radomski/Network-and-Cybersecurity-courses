In JavaScript, you don't manually allocate memory to the stack or the heap. The JavaScript engine (like V8 in Chrome and
Node.js) handles this automatically via **automatic memory management** and **Garbage Collection**.

However, understanding _how_ it distributes data between the stack and the heap is crucial for writing efficient code.

---

## The Stack vs. The Heap: A Quick Contrast

- **The Stack:** Used for **static memory allocation**. It stores fast, short-lived data: primitive values (numbers, strings,
  booleans, `null`, `undefined`, `symbol`, `bigint`) and **execution contexts** (function calls and their local variables).
  It operates on a Last-In, First-Out (LIFO) basis.
- **The Heap:** Used for **dynamic memory allocation**. It stores large, unstructured, or dynamically sized data: **Objects,
  Arrays, and Functions**.

> **The Key Link:** Variables on the stack that hold objects don't actually contain the object itself. Instead, they hold a
> **reference** (a memory address pointer) that points to the object's location in the heap.

---

## Code Example

Here is a practical breakdown of how the JavaScript engine manages memory during execution.

```javascript
// 1. Primitives are stored directly on the Stack
let age = 16;
let name = "Alex";

// 2. Objects are stored in the Heap.
// 'user' (the variable name) sits on the Stack, but its value is a reference (pointer) to the Heap.
let user = {
  username: "coder123",
  grade: 11,
};

// 3. Modifying a primitive (Stack behavior)
let newAge = age; // Copies the actual value (16) to a new spot on the stack
newAge = 17; // 'age' remains 16, 'newAge' becomes 17

// 4. Modifying an object (Heap behavior)
let admin = user; // Copies the REFERENCE (pointer), not the actual object
admin.grade = 12; // Modifies the object in the Heap

console.log(user.grade); // Outputs: 12 (Because 'user' and 'admin' point to the exact same heap object)

// 5. Function Execution Stack
function calculateSquare(num) {
  let result = num * num; // Allocated on the stack when function runs
  return result;
}

let score = 5;
let squareScore = calculateSquare(score);
// When calculateSquare finishes, its stack frame (including 'num' and 'result') is popped off and cleared instantly.
```

---

## Visualizing the Memory Layout

If we look at the memory state right after `admin.grade = 12` is executed, it looks like this:

| Stack (Fast, Structured)   | Heap (Flexible, Unstructured)                                        |
| -------------------------- | -------------------------------------------------------------------- |
| `age` = `16`               |                                                                      |
| `name` = `"Alex"`          |                                                                      |
| `user` = `Reference #001`  | `Reference #001` $\rightarrow$ `{ username: "coder123", grade: 12 }` |
| `newAge` = `17`            |                                                                      |
| `admin` = `Reference #001` | _(Same object as above)_                                             |
| `score` = `5`              |                                                                      |
| `squareScore` = `25`       |                                                                      |

## Memory Management: Clean up

- **Stack Cleanup:** This is incredibly fast. When a function finishes executing, its "stack frame" is popped off the stack,
  and all local primitives are instantly destroyed.
- **Heap Cleanup (Garbage Collection):** Because heap data can be referenced by multiple parts of a program, it can't be
  dropped instantly. The engine uses a **Garbage Collector (GC)**, primarily running a "Mark-and-Sweep" algorithm. If an
  object in the heap can no longer be reached from the stack (e.g., you set `user = null` and `admin = null`), the GC marks
  it as unreachable and later frees up that memory space.
