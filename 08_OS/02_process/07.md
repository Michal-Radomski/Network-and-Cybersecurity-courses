When you launch a program (like a web browser, a game, or a script), the operating system (OS) creates a **process**. A
process is essentially a program in execution.

The process execution cycle follows a strict journey from the moment you double-click an icon to the moment the program
closes. Here is exactly how it works:

---

## 1. From Disk to RAM (Creation)

A program stored on your hard drive or SSD is just passive code. When you run it, the OS converts it into an active process:

- **Loading:** The OS reads the program's binary code from the disk and loads it into the computer's Random Access Memory
  (RAM).
- **Allocation:** The OS allocates resources, such as memory space for variables and a unique tracking number called a
  **Process ID (PID)**.

---

## 2. The Process Memory Layout

Once in RAM, a process is organized into four distinct segments to keep things organized:

- **Text (Code) Segment:** Contains the actual machine instructions instructions for the CPU to read.
- **Data Segment:** Stores global and static variables.
- **Heap:** Dynamic memory allocated at runtime (grows when the program asks for more memory).
- **Stack:** Stores temporary data like function parameters, local variables, and return addresses.

---

## 3. The CPU Execution Cycle (The Core Process)

The CPU handles the actual "running" of the process using a continuous loop called the **Instruction Cycle**:

1. **Fetch:** The CPU retrieves the next instruction from RAM, guided by a pointer called the **Program Counter (PC)**.
2. **Decode:** The CPU’s control unit figures out what the instruction actually means (e.g., add two numbers, move data, jump
   to a different piece of code).
3. **Execute:** The CPU performs the operation.
4. **Store:** The result is written back to memory or a CPU register, and the Program Counter moves to the next instruction.

---

## 4. Process States (The Lifecycle)

Because your computer runs hundreds of processes at once but only has a limited number of CPU cores, processes must take
turns. A process cycles through various **states**:

- **New:** The process is being created.
- **Ready:** The process is loaded into RAM and waiting for its turn on the CPU.
- **Running:** The CPU is actively executing the process's instructions.
- **Waiting / Blocked:** The process cannot run because it’s waiting for something else (like waiting for you to click a
  button, or waiting for a file to download).
- **Terminated:** The process has finished running or was forced to close, and the OS reclaims its memory.

---

## 5. Context Switching (Multitasking)

To give you the illusion that all your apps are running at the exact same time, the OS uses **Context Switching**.

The OS lets a process run for a tiny fraction of a second (a time slice), then pauses it, saves its exact "state" (where it
was in the code and what variables were loaded), lets another process run, and then restores the first process right where it
left off. This happens thousands of times per second.

---

## 6. Termination

When the process finishes its job (or you close it), it enters the final stage. The OS frees up the RAM, closes any files the
process had open, and removes its PID from the system tracking list.
