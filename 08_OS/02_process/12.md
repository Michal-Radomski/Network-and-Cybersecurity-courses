When a program runs, the operating system allocates a block of memory for the process. This memory is divided into different
segments, and one of the most critical parts is the **heap**.

The heap is a region of process memory used for **dynamic memory allocation**. Unlike the stack, which follows a strict
Last-In, First-Out (LIFO) order and handles memory automatically, the heap is a large, unstructured pool of memory that you
can use flexibly at runtime.

---

## Key Characteristics of the Heap

- **Dynamic Lifetime:** Memory allocated on the heap stays there until you explicitly free it, or until the program
  terminates. It does not automatically disappear when a function finishes executing.
- **Run-time Determination:** You use the heap when you don't know how much memory you'll need ahead of time (e.g., reading a
  file of unknown size or accepting user input).
- **Shared Access:** Any part of your program with a pointer to a piece of heap memory can access or modify it. It is
  globally accessible within the process.
- **Slower Access:** Allocating and deallocating heap memory requires interacting with the OS memory manager, making it
  slower than stack allocation.

---

## How It Works: Allocation and Deallocation

Because the heap is just a giant pool of bytes, the operating system relies on a **memory allocator** to keep track of which
parts are being used and which parts are free.

In languages like **C and C++**, you manage this manually:

- **Allocation:** You request memory using functions like `malloc()` or the `new` keyword. The allocator finds a free block
  of the requested size and returns its address (a pointer).
- **Deallocation:** You must explicitly give the memory back using `free()` or `delete`.

In languages like **Java, Python, or JavaScript**, memory management is automated:

- The runtime environment automatically allocates heap memory when you create objects or arrays.
- A **Garbage Collector (GC)** periodically scans the heap to find memory that is no longer being used and frees it for you.

---

## Common Heap-Related Issues

Because the heap requires careful management, it is a frequent source of programming bugs:

- **Memory Leaks:** This happens when memory is allocated on the heap but never freed after the program is done using it.
  Over time, the program consumes more and more RAM until the system runs out of memory.
- **Dangling Pointers:** Occurs when you free a piece of memory but still try to use the pointer pointing to that old
  address. This can cause crashes or unpredictable behavior.
- **Fragmentation:** As memory is allocated and freed in random sizes, the heap can become "fragmented"—broken up into tiny,
  non-contiguous gaps. You might have plenty of total free memory, but if it's fragmented, the system won't be able to
  allocate a single large, continuous block.

---

## Heap vs. Stack: A Quick Comparison

| Feature        | The Stack                                    | The Heap                                |
| -------------- | -------------------------------------------- | --------------------------------------- |
| **Allocation** | Automatic (by the CPU)                       | Manual (by the programmer or runtime)   |
| **Size**       | Fixed and limited (can cause Stack Overflow) | Flexible and limited only by system RAM |
| **Speed**      | Extremely fast                               | Slower                                  |
| **Structure**  | Highly organized (LIFO)                      | Unstructured pool                       |
