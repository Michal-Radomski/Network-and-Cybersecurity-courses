When developers talk about the **"Python CoW (Copy-on-Write) bug,"** they are usually referring to a notorious architectural
flaw involving how Python's memory management interacts with Unix-based process forking.

Strictly speaking, it is not a "bug" in the sense of broken code; rather, it is a **performance and memory design flaw**
caused by Python's reliance on **Reference Counting** and its **Garbage Collector**.

Here is a breakdown of what the problem is, why it happens, and how Python ultimately fixed it.

---

## The Background: What is CoW _supposed_ to do?

When a master Python process uses `fork()` to spawn multiple worker subprocesses (common in web servers like Gunicorn or
Celery), the operating system tries to save RAM using **Copy-on-Write (CoW)**.

Instead of duplicating the master process's memory, the OS forces the parent and child processes to share the exact same
physical memory pages. As long as the workers only _read_ the data, no extra RAM is consumed. A physical copy of a memory
page is only made when a process attempts to _modify_ (write to) it.

---

## The Flaw: Why Python Breaks CoW

In Python, **even when a child process is only reading data, it triggers massive memory writes.** This completely destroys
the benefits of CoW, forcing the OS to copy almost all the shared memory into the child process anyway.

This happens due to two core components of Python's internal design:

### 1. Reference Counting (The Main Culprit)

Every time a Python object is accessed or read, Python updates its **reference count** (an integer stored inside the object's
memory structure that tracks how many variables point to it).

- **The Trap:** When a worker subprocess reads a shared configuration dictionary or global variable, Python increments its
  reference count.
- **The Consequence:** To the operating system, changing that integer is a **write operation**. The OS immediately panics,
  splits the memory page, and allocates a duplicate copy for the child process.

### 2. Cyclic Garbage Collection (GC)

Python's cyclic garbage collector periodically sweeps through memory to find objects referencing each other in loops.

- **The Trap:** To perform this search, the GC modifies internal flags (`gc_refs`) directly inside the headers of Python
  objects.
- **The Consequence:** Every time the garbage collector runs inside a child process, it touches and modifies nearly every
  object in memory. This triggers a massive wave of CoW page splits, causing the child's memory usage to balloon instantly.

> **The Result:** If you have a 1 GB machine-learning model or large data cache loaded into a parent process, spawning 4
> workers doesn't take 1 GB of total RAM as it should under CoW—it quickly balloons toward 5 GB.

---

## How This Was Solved

For years, Python developers had to use ugly workarounds, like disabling the garbage collector entirely right before forking
(`gc.disable()`). However, recent updates to Python introduced permanent structural fixes:

### Fix 1: GC Tweaks (Python 3.7+)

Python introduced `gc.freeze()`. This allows developers to "freeze" all objects currently tracked by the garbage collector in
the parent process. When the child process runs its own GC, it completely ignores these frozen objects, preventing the GC
from touching their headers and triggering memory copies.

### Fix 2: Immortal Objects (Python 3.12+)

The definitive fix came with **PEP 683 (Immortal Objects)**. Python 3.12 introduced a mechanism where certain objects (like
core runtime constants, integers, None, and strings) can be marked as completely **immortal**.

For these objects, Python completely bypasses reference counting. Their reference counts remain at a fixed, unchanging value.
Because the child processes never modify these numbers, the OS never breaks the Copy-on-Write sharing, resulting in
drastically reduced memory overhead for multi-processed Python apps.
