To manage multitasking, the operating system's kernel maintains centralized data structures to track everything running on
the system. Two of the most important structures for this are the **Kernel Process Table** and the **Kernel Thread Table**.

Here is a breakdown of what they are, what they contain, and how they work together.

---

## 1. The Kernel Process Table

The **Kernel Process Table** is a global data structure that the OS uses to track every active process on the system. Think
of it as the master directory of all running applications.

- **What it contains:** It holds an array or linked list of pointers to **Process Control Blocks (PCBs)**. Each PCB contains
  the global metadata for a process.
- **Key Information Stored:**
- **Process ID (PID):** A unique identifier for the process.
- **Memory Management:** Pointers to page tables, memory limits, and virtual address spaces.
- **Resource Ownership:** A list of open file descriptors, network sockets, and allocated hardware.
- **Security Credentials:** User ID (UID) and Group ID (GID) determining permissions.

---

## 2. The Kernel Thread Table

The **Kernel Thread Table** is a global data structure used by the OS to track every individual thread of execution currently
managed by the kernel.

In modern operating systems (which use kernel-level threading), the scheduler schedules _threads_, not processes. Therefore,
the kernel needs a dedicated table to manage them.

- **What it contains:** It holds pointers to **Thread Control Blocks (TCBs)** for every thread across the entire system.
- **Key Information Stored:**
- **Thread ID (TID):** A unique identifier for the thread.
- **Processor State:** Saved CPU registers, Stack Pointer, and Program Counter (PC).
- **Scheduling Info:** Thread priority, CPU usage stats, and thread state (Ready, Running, Blocked).
- **Link to Parent:** A pointer back to the specific entry in the Process Table that owns this thread.

---

## How They Work Together

The Process Table and Thread Table operate in a **one-to-many relationship**. A single entry in the Process Table (a process)
can correspond to multiple entries in the Thread Table (its threads).

| Feature                  | Kernel Process Table                                                   | Kernel Thread Table                                                             |
| ------------------------ | ---------------------------------------------------------------------- | ------------------------------------------------------------------------------- |
| **Focus**                | Resource allocation and ownership.                                     | Execution state and CPU scheduling.                                             |
| **Shared Data**          | Processes do **not** share memory or files with each other by default. | Threads belonging to the same process **share** the process's memory and files. |
| **What the Kernel Sees** | The container holding the resources.                                   | The actual unit of work to be run on the CPU.                                   |

### An Execution Scenario

When a thread needs to be paused and another resumed (a context switch):

1. The kernel looks at the **Kernel Thread Table** to find a thread marked as "Ready."
2. It saves the current thread's CPU registers into its respective TCB.
3. It loads the new thread's registers from its TCB in the Thread Table.
4. If the new thread belongs to a _different_ process, the kernel also references the **Kernel Process Table** to switch the
   memory mapping (page tables) to the new process's memory space. If it belongs to the _same_ process, it skips this step,
   making the switch much faster.
