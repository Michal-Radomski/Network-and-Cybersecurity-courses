When designing an application, one of the biggest architectural decisions is choosing how to handle concurrent tasks. Two
primary approaches exist: **Multi-threading** and **Multi-processing**.

The fundamental difference lies in how they handle **memory allocation** and **isolation**.

---

## 1. Multi-threaded Architecture

In a multi-threaded application, a single process creates multiple paths of execution (threads). All of these threads live
inside the same process container.

- **Memory Sharing:** Every thread shares the exact same memory space (the heap, global variables, and code). Each thread
  only gets its own private stack and program counter.
- **Communication:** Because they share memory, threads can communicate with each other incredibly fast. They can simply read
  and write to the same variables.
- **Overhead:** Extremely lightweight. Creating a thread or switching between threads takes very little CPU time and memory.
- **The Catch:** **Low fault tolerance.** If one thread encounters a fatal bug or crashes, it can corrupt the shared memory
  and crash the _entire_ process. Race conditions and data corruption are also common risks if two threads try to change the
  same variable at the exact same time.

---

## 2. Multi-process Architecture

In a multi-process application, the operating system spawns entirely separate, isolated processes, each running
independently.

- **Memory Isolation:** Each process gets its own completely dedicated sandbox of virtual memory space. Process A cannot see
  or touch Process B's memory.
- **Communication:** Processes cannot talk to each other directly. They must use slower **IPC (Inter-Process Communication)**
  methods, such as network sockets, pipes, or shared memory files.
- **Overhead:** Heavyweight. Spawning a new process requires allocating a new memory layout, copying descriptors, and
  handling heavier context switches.
- **The Benefit:** **High fault tolerance.** If Process A crashes, Process B keeps running perfectly fine as if nothing
  happened (this is how Google Chrome works—if one browser tab crashes, your whole browser doesn’t go down).

---

## Side-by-Side Comparison

| Feature               | Multi-threading                                             | Multi-processing                                                |
| --------------------- | ----------------------------------------------------------- | --------------------------------------------------------------- |
| **Memory**            | Shared among all threads.                                   | Isolated per process.                                           |
| **Creation Cost**     | Low overhead (Fast).                                        | High overhead (Slower).                                         |
| **Context Switching** | Fast (No need to swap page tables).                         | Slower (Requires flushing/changing page tables).                |
| **Crash Impact**      | One thread crash can kill the whole app.                    | If one process crashes, others survive.                         |
| **Best Used For**     | **I/O-bound workloads** (e.g., chat servers, web scraping). | **CPU-bound workloads** (e.g., video rendering, data analysis). |

---

## How to Choose

### Go with Multi-threading if:

Your application spends most of its time waiting for things to happen (I/O-bound). For example, a web server handling
incoming requests, or an app reading files from disk. Threads allow you to handle thousands of waiting connections
simultaneously with very little RAM usage.

### Go with Multi-processing if:

Your application does heavy math or computation that consumes 100% of a CPU core. Because of programming languages' safety
locks (like Python's GIL), multi-processing is often the only way to bypass restrictions and utilize every single core of a
multi-core processor for true parallel computation.
