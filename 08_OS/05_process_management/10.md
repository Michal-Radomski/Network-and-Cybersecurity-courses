When we talk about computer performance, workloads are generally categorized into two main types based on what resource is
slowing them down the most: **CPU-bound** and **I/O-bound**.

Understanding the difference determines how you write your code, what hardware you buy, and how you optimize your
applications.

---

## 1. CPU-Bound Workloads

A workload is **CPU-bound** if the speed of the task is limited by the performance of the central processing unit (CPU). In
these scenarios, the CPU is running at 100% capacity, spinning its wheels to crunch data as fast as possible.

- **What it looks like:** The program spends its time doing heavy mathematical calculations, data manipulation, or logic
  operations in memory.
- **The Bottleneck:** Adding faster RAM or a faster internet connection won't speed it up. The only way to make it faster is
  to give it a more powerful CPU, more CPU cores, or optimize the algorithm.
- **Examples:**
- Video editing and rendering.
- Compiling a massive software project from source code.
- Training machine learning models or processing complex graphics (often offloaded to GPUs).
- Cryptographic hashing, mining cryptocurrency, or zip-compressing a massive folder.

---

## 2. I/O-Bound Workloads

A workload is **I/O-bound** (Input/Output bound) if the speed of the task is limited by the time spent waiting for data to be
sent or received from external sources. The CPU sits idle most of the time, waiting for other hardware to finish its job.

- **What it looks like:** The program spends its time waiting on storage drives, network requests, database queries, or user
  input.
- **The Bottleneck:** Upgrading to the fastest CPU in the world won't make a difference if the application is stuck waiting
  for a slow hard drive or a congested internet connection.
- **Examples:**
- A web scraper downloading thousands of webpages.
- A database server fetching records from a hard drive.
- A file-copying utility moving 50 GB of data from a computer to a USB flash drive.
- A chat application waiting for users to type and send messages.

---

## Summary Comparison

| Feature                | CPU-Bound                                                                      | I/O-Bound                                                                                       |
| ---------------------- | ------------------------------------------------------------------------------ | ----------------------------------------------------------------------------------------------- |
| **Primary Limitation** | Processor speed and number of cores.                                           | Data transfer speed (Disk, Network, RAM bus).                                                   |
| **CPU Utilization**    | **Very High (close to 100%)** on active cores.                                 | **Low**, the CPU spends a lot of time "sleeping" or waiting.                                    |
| **How to Scale**       | Add faster processors or use **Multi-processing** to spread work across cores. | Use **Asynchronous programming** (like Node.js) or **Multithreading** to handle multiple waits. |
| **Hardware Fix**       | Upgrade the CPU / GPU.                                                         | Upgrade to an SSD, increase network bandwidth, or cache data in RAM.                            |

---

## Why This Matters for Developers

How you handle concurrency in programming depends entirely on this distinction:

- **For I/O-bound tasks:** You want an architecture that doesn't block the program while waiting for data.
  **Asynchronous/event-driven programming** (like Node.js) or standard **multithreading** works beautifully here because one
  thread can handle a request while another thread is waiting for the disk to read.
- **For CPU-bound tasks:** Traditional multithreading won't always help (especially in languages with execution locks like
  Python's old GIL). To speed things up, you must use **parallelism/multi-processing**, which splits the heavy computation
  across separate physical CPU cores.
