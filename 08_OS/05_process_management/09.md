To understand **TLB flushes** and **TLB ASIDs**, we first need to look at what the **TLB (Translation Lookaside Buffer)**
actually does.

When a program runs, it uses virtual memory addresses. The CPU, however, needs physical memory addresses to fetch data from
RAM. Turning a virtual address into a physical address requires looking up a massive guide called a **Page Table**. Because
checking the Page Table in RAM is slow, the CPU uses a super-fast, built-in hardware cache called the **TLB** to store the
most recent address translations.

Here is how the operating system manages this cache using flushes and ASIDs during a context switch.

---

## 1. What is a TLB Flush?

When the operating system switches from running Process A to Process B, a problem arises. Because processes are isolated from
each other, Process A and Process B might use the exact same virtual address (e.g., `0x00400000`), but that address points to
completely different physical locations in RAM.

If the CPU tries to read memory for Process B using the old cache entries left behind by Process A, it will read or corrupt
the wrong data.

To prevent this, the OS can perform a **TLB Flush**.

- **The Action:** The OS completely wipes out (invalidates) all the entries in the TLB cache.
- **The Catch:** Wiping the cache means the next time Process B tries to access memory, it will experience a series of **TLB
  misses**. The CPU is forced to look up translations the slow way (by walking the Page Table in RAM), which creates a
  noticeable performance slowdown right after a context switch.

---

## 2. What is a TLB ASID?

To avoid the performance penalty of flushing the entire TLB every time a context switch happens, modern CPU architectures
(like ARM and x86) introduced **ASID (Address Space Identifier)**.

An ASID is a small, unique hardware tag (usually 8 to 16 bits) that the operating system assigns to each process.

### How ASID Works:

Instead of just storing the virtual address and physical address in the TLB, the CPU also stores the process's ASID tag
inside the cache entry.

- **Process A** runs with `ASID = 1`. Its TLB entries are tagged with a `1`.
- When the OS switches to **Process B** (`ASID = 2`), **the OS does not flush the TLB.**
- When Process B looks up a virtual address, the hardware checks the cache for an entry that matches both the virtual address
  _and_ `ASID = 2`.
- It ignores Process A's entries because the tags don't match.

---

## TLB Flush vs. ASID: Summary Comparison

| Feature                  | TLB Flush Approach                                                    | TLB ASID Approach                                                                           |
| ------------------------ | --------------------------------------------------------------------- | ------------------------------------------------------------------------------------------- |
| **Strategy**             | Erase everything on a context switch.                                 | Keep everything, but tag entries to identify the owner.                                     |
| **Performance**          | **Slower.** Causes "cache coldness" and slowdowns after every switch. | **Faster.** Avoids empty cache penalties; old entries are still there when you switch back. |
| **Hardware Requirement** | Basic CPU capability (older architectures).                           | Requires specialized hardware support in the MMU (Memory Management Unit).                  |

### When is a Flush still needed with ASID?

Even if a CPU supports ASID, the OS must still flush the TLB occasionally. For example, if a process frees up memory or
modifies its own page tables, the old translations are no longer valid, and the OS must explicitly flush those specific
entries from the TLB.
