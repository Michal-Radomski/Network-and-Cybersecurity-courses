When multiple threads or processes run at the same time, they often need to share the same resources (like a file, a
variable, or a printer). If two threads try to modify the same variable at the exact same moment, the data gets corrupted.
This is called a **race condition**.

To prevent this chaos, operating systems use synchronization tools. The two most popular tools are **Mutexes** and
**Semaphores**.

---

## 1. What is a Mutex?

A **Mutex** (short for **Mut**ual **Ex**clusion) is a locking mechanism used to synchronize access to a resource. It is a
binary flag that ensures **only one thread** can access a critical piece of code or data at a time.

Think of a Mutex like a **bathroom key** at a coffee shop:

- If you want to use the bathroom, you must take the key.
- While you are inside, no one else can enter because you have the key.
- When you are finished, you return the key, allowing the next person in line to use it.

### Key Characteristic: Ownership

A Mutex has a strict concept of ownership. **Only the thread that locked the Mutex is allowed to unlock it.** If Thread A
locks a resource, Thread B cannot swoop in and force it open.

---

## 2. What is a Semaphore?

A **Semaphore** is a signaling mechanism that controls access to a resource by using a counter. Instead of allowing just one
thread, a semaphore allows a **specified maximum number of threads** to access a resource simultaneously.

Think of a Semaphore like a **nightclub with a bouncer**:

- The club has a capacity limit of 50 people.
- The bouncer keeps a count. As long as the count is under 50, people can walk straight in.
- Once the club hits 50 people, a line forms outside.
- The moment someone leaves the club, the count drops, and the bouncer lets the next person in line enter.

### Types of Semaphores:

- **Binary Semaphore:** Has a maximum count of 1. It behaves very similarly to a mutex, but without the strict ownership rule
  (any thread can signal it to unlock).
- **Counting Semaphore:** Has a maximum count greater than 1 (e.g., controlling access to a pool of 4 database connections).

---

## Side-by-Side Comparison

| Feature                 | Mutex                                                | Semaphore                                                                   |
| ----------------------- | ---------------------------------------------------- | --------------------------------------------------------------------------- |
| **Mechanism**           | Locking mechanism (Key).                             | Signaling mechanism (Counter).                                              |
| **Simultaneous Access** | Strictly **one** thread at a time.                   | **Multiple** threads (up to a defined limit).                               |
| **Ownership**           | Yes. Only the locking thread can unlock it.          | No. Any thread can signal a change in the count.                            |
| **Best Used For**       | Protecting shared data or variables from corruption. | Managing a limited pool of resources (like a buffer or server connections). |

---

## A Quick Scenario: When to use which?

- **Use a Mutex** if you have a single global variable (like a bank account balance) that multiple threads are trying to
  update. You want to lock the balance, update it, and unlock it so no two threads mess up the math.
- **Use a Semaphore** if you are building a web server that can only handle 10 file downloads at a time to keep from
  crashing. You set a counting semaphore to 10. The 11th user is forced to wait until one of the first 10 finishes
  downloading.
