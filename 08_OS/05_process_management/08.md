**Context switching** is the fundamental mechanism that allows an operating system to be multitasking. It is the process by
which the CPU switches from executing one process or thread to another.

To the human eye, it looks like a computer is running dozens of applications at the exact same time. In reality, a single CPU
core can only do one thing at a time; it just switches between tasks so incredibly fast (often thousands of times per second)
that it creates the illusion of simultaneous execution.

---

## The Core Steps of a Context Switch

When the operating system's scheduler decides to swap the currently running task (Task A) for a waiting task (Task B), it
executes a very specific sequence:

1. **Pause Task A:** The CPU stops executing instructions for Task A.
2. **Save Context:** The current state of Task A—including its program counter (where it left off), CPU registers, and stack
   pointers—is saved into its management structure (either its **Process Control Block (PCB)** or **Thread Control Block
   (TCB)**).
3. **Load Context:** The operating system fetches the saved state of Task B from its respective PCB or TCB.
4. **Resume Task B:** The CPU loads Task B's saved registers and program counter into its actual hardware, and Task B begins
   executing exactly where it last paused.

---

## Thread Switching vs. Process Switching

Context switching happens at two levels, and one is significantly faster than the other.

### 1. Thread Context Switch (Fast)

This happens when the CPU switches between two threads that belong to the **same** process. Because they share the same
virtual memory space, the OS only needs to swap out the CPU registers and stack pointers. The memory layout stays exactly the
same.

### 2. Process Context Switch (Slow)

This happens when the CPU switches to a thread in a **different** process altogether. Because processes are isolated from one
another, the OS must swap the CPU registers _and_ flush the memory page tables (Translation Lookaside Buffer or TLB). This
requires the CPU to reload memory mappings, which introduces a major performance penalty.

---

## What Triggers a Context Switch?

An operating system will initiate a context switch for three main reasons:

- **Multitasking (Time Slicing):** The OS gives every program a tiny window of CPU time (a "time slice"). When that time
  expires, a hardware timer interrupts the CPU, forcing it to switch to the next program.
- **Interrupt Handlers:** If hardware needs attention—like a mouse click, a keyboard press, or a network packet arriving—the
  CPU will pause its current task to handle the hardware event.
- **Blocking Operations:** If a program requests data from a slow storage drive or waits for a webpage to load, it can't
  proceed. The OS moves it to a "Blocked" state and switches to a different program that is ready to work.

---

## The "Context Switch Overhead"

While context switching is necessary for modern computing, it comes with a cost known as **overhead**.

During the microsecond it takes to save one state and load another, the CPU is managing the operating system's administrative
tables rather than running user programs. If an OS switches tasks _too_ frequently, it can lead to a state called
**thrashing**, where the computer spends more time switching between jobs than actually getting any work done.
