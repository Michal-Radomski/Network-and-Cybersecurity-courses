A **Thread Control Block (TCB)** is a data structure used by an operating system's kernel to manage and store information
about a single thread.

Just like a Process Control Block (PCB) manages an entire process, a TCB is created for every individual thread within that
process. Because multiple threads share the same process resources (like memory and open files), the TCB focuses strictly on
the information unique to that specific thread's execution.

---

## Key Components of a TCB

A typical TCB contains several critical pieces of information that the operating system needs to schedule and run the thread:

- **Thread Identifier (Thread ID):** A unique integer assigned by the OS to distinguish this thread from others.
- **Thread State:** Tracks what the thread is currently doing (e.g., Ready, Running, Blocked, or Suspended).
- **Program Counter (PC):** Points to the address of the next instruction the thread is supposed to execute.
- **CPU Registers:** Stores the current values of the processor registers when the thread is paused, allowing it to resume
  exactly where it left off.
- **Stack Pointer:** Points to the thread's independent stack, which holds local variables, function parameters, and return
  addresses.
- **Thread Priority:** Information used by the OS scheduling algorithm to decide which thread gets CPU time next.
- **Parent Process Pointer:** A reference link back to the PCB of the process that owns this thread.

---

## TCB vs. PCB: How They Compare

To understand the TCB better, it helps to see how it splits responsibilities with the Process Control Block (PCB).

| Feature               | Process Control Block (PCB)                          | Thread Control Block (TCB)                             |
| --------------------- | ---------------------------------------------------- | ------------------------------------------------------ |
| **Scope**             | Manages the entire process.                          | Manages a single thread within a process.              |
| **Quantity**          | One per process.                                     | One per thread (a process can have many TCBs).         |
| **Resource Tracking** | Contains global memory, source code, and open files. | Contains only execution states, registers, and stacks. |
| **Overhead**          | High overhead to create, delete, or switch.          | Low overhead; much faster to create and switch.        |

---

## The Role of TCB in Context Switching

When the operating system decides to stop running Thread A and start running Thread B, a **context switch** occurs. The TCB
is central to this process:

1. The CPU pauses Thread A.
2. The current state of the CPU (registers, program counter) is saved into **Thread A's TCB**.
3. The OS loads the saved CPU state from **Thread B's TCB** into the actual CPU hardware.
4. The CPU resumes execution exactly where Thread B previously stopped.

Because threads share the same memory space, switching between TCBs within the same process is significantly faster than
switching between PCBs, as the OS doesn't need to reload memory page tables.
