Because Node.js is inherently single-threaded (for your main application code), you don't typically run into the traditional
multithreaded race conditions where two threads try to modify the same hardware memory address at the same exact time.

However, because Node.js is **asynchronous**, you _frequently_ run into **asynchronous race conditions**. For example, if two
asynchronous functions try to read, modify, and write a file or a database record at the same time, they can overwrite each
other's changes.

To solve this, we can use Mutexes and Semaphores in Node.js. Since these aren't built into the core language, the industry
standard is to use a popular, lightweight library called **`async-mutex`**.

---

First, you would need to install the package in your project:

## 1. How to use a Mutex in Node.js

A **Mutex** ensures that an asynchronous block of code (like a database update or file write) is only executed by **one
request at a time**.

### The Problem Scenario

Imagine an asynchronous function that fetches a bank balance, adds money, and saves it. Without a mutex, if two deposits
happen at the exact same millisecond, they might both read the _old_ balance before either can save, causing one deposit to
disappear.

### The Solution using Mutex:

```javascript
const { Mutex } = require("async-mutex");

const mutex = new Mutex();
let bankBalance = 100; // Shared resource

async function depositMoney(amount, user) {
  console.log(`[${user}] Requesting access to deposit...`);

  // 1. Acquire the lock. If another function has it, this waits.
  const release = await mutex.acquire();

  try {
    console.log(`[${user}] Lock acquired. Current balance: $${bankBalance}`);

    // Simulate a slow asynchronous database read/write
    await new Promise((resolve) => setTimeout(resolve, 1000));

    bankBalance += amount;
    console.log(`[${user}] Deposit successful! New balance: $${bankBalance}`);
  } finally {
    // 2. ALWAYS release the lock in a finally block,
    // otherwise your app will freeze forever if an error occurs!
    release();
    console.log(`[${user}] Lock released.`);
  }
}

// Simulating two identical requests hitting the server at the exact same time
depositMoney(50, "Alice");
depositMoney(30, "Bob");
```

**How it executes:** Alice will grab the lock first, run her code for 1 second, and finish. Bob's code will automatically sit
patiently at `await mutex.acquire()` and won't execute until Alice calls `release()`.

## 2. How to use a Semaphore in Node.js

A **Semaphore** allows you to set a limit on how many asynchronous operations can run concurrently. This is incredibly useful
for **rate-limiting** or protecting external APIs and databases from getting overwhelmed.

### The Scenario

Imagine you want to scrape data from 100 different webpages. If you launch all 100 `fetch` requests at once, the target
website might block your IP address for spamming. A Semaphore allows you to limit it to exactly 3 downloads at a time.

### The Solution using Semaphore:

```javascript
const { Semaphore } = require("async-mutex");

// Initialize a semaphore with a concurrency limit of 3
const semaphore = new Semaphore(3);

async function downloadFile(fileId) {
  // 1. Wait for an available slot and acquire it
  // acquire() returns an array: [value, releaseFunction]
  const [value, release] = await semaphore.acquire();

  try {
    console.log(`[START] Downloading file ${fileId}...`);

    // Simulate downloading a file (takes 2 seconds)
    await new Promise((resolve) => setTimeout(resolve, 2000));

    console.log(`[FINISH] File ${fileId} downloaded.`);
  } finally {
    // 2. Release the slot back to the pool
    release();
  }
}

// Try to launch 6 downloads all at once
for (let i = 1; i <= 6; i++) {
  downloadFile(i);
}
```

**How it executes:** 1. Files 1, 2, and 3 will start downloading immediately. 2. Files 4, 5, and 6 will wait in line. 3. As
soon as File 1 finishes and calls `release()`, File 4 will instantly start.

---

## Summary Checklist

- **Use a Mutex** when you are doing **Read-Modify-Write** operations on files, global variables, or database records where
  only one worker should touch the data at a time.
- **Use a Semaphore** when you want to build a **concurrency throttle** (e.g., limit outbound API calls, concurrent image
  resizing tasks, or open database connections).
- **Crucial Rule:** Always wrap your critical code in a `try...finally` block and call the `release()` function inside
  `finally`. If your code throws an error and you forget to release, your application will deadlock and halt future
  operations.
