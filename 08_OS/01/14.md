While they sound similar, a **program** and a **process** represent two completely different stages of software execution.

The easiest distinction is that a **program is passive**, while a **process is active**.

---

## 1. What is a Program?

A **program** is a passive file stored on your hard drive or SSD. It consists of a static collection of instructions and data
written by a developer and compiled into machine code.

- **State:** Passive/Inert. It just sits on your disk doing nothing until you trigger it.
- **Life Span:** Permanent. It exists on your storage until you manually delete or uninstall it.
- **Memory:** It consumes storage space (megabytes or gigabytes on your disk), but it doesn't use any CPU or RAM.
- **Example:** The `firefox.exe` file sitting in your program files folder, or a compiled binary file in Linux like
  `/bin/ls`.

---

## 2. What is a Process?

A **process** is an active, running instance of a program. When you double-click a program icon, the operating system kernel
copies the program's instructions into RAM, allocates system resources, and begins executing those instructions on the CPU.

- **State:** Active/Dynamic. It is actively executing instructions, managing data, and interacting with the OS.
- **Life Span:** Temporary. It begins when the program is launched and terminates when you close the app or it finishes its
  task.
- **Memory:** It consumes active system RAM and demands CPU clock cycles.
- **Example:** When you open three separate windows of Firefox, you have **one program** stored on your disk, but **three
  separate processes** running simultaneously in your RAM.

---

## Technical Comparison

When a program becomes a process, the operating system wraps it in a special structure within the RAM. A process includes
components that a static program file does not have:

| Component                       | What it is                                                                                                 |
| ------------------------------- | ---------------------------------------------------------------------------------------------------------- |
| **Text Section**                | The actual compiled machine code loaded from the program file.                                             |
| **Data Section**                | Stores global and static variables used by the running code.                                               |
| **The Heap**                    | Dynamically allocated memory that the process requests while running.                                      |
| **The Stack**                   | Temporary data storage for function parameters, return addresses, and local variables.                     |
| **PCB (Process Control Block)** | The OS kernel's internal ID tag for the process, tracking its priority, current CPU state, and open files. |

---

## Summary Metaphor

Think of a **program** as a recipe printed in a cookbook. It is just text on a page sitting quietly on a shelf.

A **process** is the actual act of cooking that recipe in the kitchen. It requires active ingredients (RAM), tools (CPU
registers), a chef (the CPU Control Unit), and it eventually comes to an end when the meal is finished.
