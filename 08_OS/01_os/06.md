An Operating System (OS) architecture defines how the different software components of a system interact with each other and
the underlying hardware. At the absolute center of this architecture sits the **kernel**.

Think of the OS as a high-security building: the kernel is the core management team operating behind closed doors, while the
apps you use are visitors in the lobby.

---

## What is the Kernel?

The **kernel** is the core program of an OS. It loaded into memory during boot-up and remains active until the system shuts
down. Because it interacts directly with the physical hardware, it runs in a highly privileged area of the CPU called
**Kernel Mode** (or Supervisor Mode).

### Core Responsibilities of a Kernel:

- **Process Management:** Deciding which program gets to use the CPU, for how long, and when (scheduling).
- **Memory Management:** Keeping track of physical and virtual memory ($RAM$), ensuring apps don't overwrite each other's
  data.
- **Device Management:** Acting as an intermediary between software and hardware peripherals (keyboards, GPUs, drives) using
  device drivers.
- **File System Management:** Organizing, storing, and retrieving files on storage drives.

---

## The OS Architecture Layers

An OS architecture is generally split into two distinct execution environments to prevent user apps from crashing the entire
system:

```text
+--------------------------------------------------+
|                   User Space                     |
|  [ User Apps ]   [ Web Browser ]   [ UI/Shell ]  |
+--------------------------------------------------+
                        ||  (System Calls)
====================================================
                        ||
+--------------------------------------------------+
|                  Kernel Space                    |
|   [ Process Mgmt ]          [ Memory Mgmt ]      |
|   [ File Systems ]          [ Drivers ]          |
+--------------------------------------------------+
                        ||
+--------------------------------------------------+
|                 Physical Hardware                |
|        [ CPU ]       [ RAM ]       [ Disk ]      |
+--------------------------------------------------+

```

### 1. User Space

This is where your everyday applications run (e.g., Discord, Spotify, video games). Programs here run in **User Mode**, a
restricted mode where they cannot directly touch hardware or access memory assigned to other programs. If a user space app
crashes, it just closes—it doesn't crash your whole computer.

### 2. The Bridge: System Calls

When an app in User Space needs to do something involving hardware (like saving a file or playing audio), it cannot do it
alone. It must issue a **System Call (Syscall)**. This is a formal request that asks the kernel to step in, switch to Kernel
Mode, perform the task safely, and return the result.

### 3. Kernel Space

The privileged zone where the kernel code executes and has unrestricted access to all system hardware and memory.

## Common Types of Kernel Architectures

Different operating systems organize their kernels in different ways. The two most common types are Monolithic and
Microkernels.

| Feature                       | Monolithic Kernel                                                                      | Microkernel                                                                               |
| ----------------------------- | -------------------------------------------------------------------------------------- | ----------------------------------------------------------------------------------------- |
| **Design Philosophy**         | Put all core OS services inside the kernel for maximum speed.                          | Keep the kernel as tiny as possible; move services to user space.                         |
| **What's inside the Kernel?** | Process management, memory management, file systems, device drivers.                   | Only the bare minimum: scheduling, virtual memory, and IPC (Inter-Process Communication). |
| **Pros**                      | **Fast execution speed** because components communicate instantly.                     | **Highly stable and secure**. If a driver crashes, the kernel stays alive.                |
| **Cons**                      | If a single driver bugs out, the entire system can crash (Blue Screen / Kernel Panic). | **Slower performance** due to the overhead of constant communication between spaces.      |
| **Examples**                  | Linux, Android, macOS (Hybrid), Windows (Hybrid).                                      | QNX (used in cars), seL4, Google Fuchsia.                                                 |

> **Note on Hybrid Kernels:** Modern commercial operating systems like Windows and macOS actually use a **Hybrid Kernel**.
> They combine the speed of a monolithic design with the modularity of a microkernel, running certain non-essential drivers
> in user space while keeping performance-critical components in kernel space.
