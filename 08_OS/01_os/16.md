Imagine you are using an app on your phone or computer, like a photo editor, and you want to save a picture to your device.

The app itself isn't allowed to just reach directly into your hard drive and write data wherever it wants—that would be a
massive security risk, as a rogue app could easily delete your files. Instead, the app has to ask the operating system (OS)
to save the file for it.

That request is called a **system call**.

A **system call** is the programmatic way a computer program requests a service from the kernel (the absolute core) of the
operating system. It acts as a secure bridge between user applications and the physical hardware.

---

## The Dual-Mode Bridge

To understand why system calls are necessary, it helps to know that modern CPUs run in two primary modes:

1. **User Mode:** This is where your normal applications (browsers, games, text editors) live. In this mode, programs have
   limited access to memory and absolutely no direct access to hardware.
2. **Kernel Mode:** This is the highly privileged mode where the core of the operating system operates. The kernel has
   unrestricted access to the hardware, memory, and CPU instructions.

When an application in **User Mode** needs to do something involving hardware, it executes a system call. This triggers an
interception (often called a "trap" or "exception") that safely switches the CPU into **Kernel Mode**, performs the requested
task, and then switches back to User Mode to hand the results to the app.

---

## Common Types of System Calls

System calls generally fall into five main categories based on what the program needs to accomplish:

### 1. Process Control

These calls handle the creation, management, and termination of programs.

- **Examples:** Starting a new program (`fork()` or `exec()`), ending a program (`exit()`), or making one program wait for
  another to finish.

### 2. File Management

These are used whenever a program needs to interact with files or directories on a storage drive.

- **Examples:** Creating a file, deleting a file, opening an existing file (`open()`), reading data from it (`read()`), or
  writing data to it (`write()`).

### 3. Device Management

Programs use these to interact with physical hardware peripherals like your mouse, keyboard, monitor, or printer.

- **Examples:** Requesting access to a hardware device, reading inputs from a controller, or sending data to a screen.

### 4. Information Maintenance

These calls transfer information between the operating system and the user program, such as system configurations or time
data.

- **Examples:** Asking for the current time and date, checking the amount of free hard drive space, or getting system version
  info.

### 5. Communication

These handle data transfers between different programs or across a network.

- **Examples:** Creating a network connection to a website, sending data packets over Wi-Fi, or allowing two apps on the same
  computer to talk to each other.

---

## Why Don't Programmers Write System Calls Directly?

If you have ever written a simple programming script, you probably didn't write an explicit system call. Instead, you likely
used a standard library function, like `print()` in Python or `printf()` in C.

Writing raw system calls is incredibly complex and changes depending on the exact CPU architecture and OS. To make life
easier, operating systems provide an **API (Application Programming Interface)**.

- When you write a piece of code to save a file, you use a simple API function provided by your language.
- Behind the scenes, that library function takes care of the messy, low-level details and executes the actual **system call**
  to the OS on your behalf.
