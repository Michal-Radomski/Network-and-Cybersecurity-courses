Installing an app on Windows and Linux feels completely different under the hood. Windows relies on a **decentralized,
installer-based** model, while Linux relies on a **centralized, package-based** model.

Here is a breakdown of how they compare.

---

### 1. Where You Get the Software

- **Windows:** You typically open a web browser, search for the app, go to the developer's website, and download an installer
  file (like an `.exe` or `.msi`). _(Note: The Microsoft Store exists, but downloading from the web is still the norm)._
- **Linux:** You rarely use a browser. Instead, you use a **Package Manager** (like `apt`, `pacman`, or `dnf`) or a software
  center built into the OS. It’s like an App Store, but everything is hosted in official, secure servers called
  **Repositories**.

### 2. How "Dependencies" are Handled

Apps often need other helper files (like libraries or frameworks) to run.

- **Windows:** The installer usually bundles everything it needs into one giant package, or it automatically downloads things
  like .NET Framework or DirectX during setup.
- **Linux:** The package manager handles this automatically. If App A needs Library B to run, Linux will automatically look
  for Library B in the repository, download it, and share it among any other apps that need it. This keeps file sizes much
  smaller.

### 3. File Structure (Where things go)

- **Windows:** Almost everything goes into a single dedicated folder, usually inside `C:\Program Files` or
  `C:\Program Files (x86)`. It also registers the app in the **Windows Registry** (a massive database for system and app
  settings).
- **Linux:** Linux scatters the files across the system based on file type. The executable goes to `/usr/bin`, libraries go
  to `/usr/lib`, and configuration files go to `/etc` or your hidden user folders. There is no system registry; everything is
  handled via simple text files.

### 4. Updating the Apps

- **Windows:** Each app usually manages its own updates. You might see an "Update available" popup when you open the app, or
  it might run a background updater service.
- **Linux:** The system updates **everything at once**. When you run a system update, the package manager checks for updates
  for the OS _and_ every single app you installed through it, updating them all in one go.

---

### Summary Comparison

| Feature              | Windows                                   | Linux                                                   |
| -------------------- | ----------------------------------------- | ------------------------------------------------------- |
| **Primary Method**   | Downloading `.exe` / `.msi` from websites | Using a Package Manager via Terminal or Software Center |
| **Storage Location** | Centered in `Program Files`               | Scattered systematically (`/bin`, `/lib`, `/etc`)       |
| **System Registry**  | Yes (Updates the Windows Registry)        | No (Uses configuration files)                           |
| **Updates**          | Handled individually by each app          | Handled globally by the OS                              |

### The Modern Shift: Flatpaks, Snaps, and AppImages

To make things easier, modern Linux now has formats like **Flatpaks** and **Snaps**. These work more like Windows or macOS
apps—they bundle everything the app needs into one isolated container, allowing you to just click and run without worrying
about system dependencies!

Behind the scenes, Windows and Linux take entirely different approaches to installing an application. Windows uses a
**self-contained installation wizard** that places everything in a dedicated folder, while Linux uses a **package management
system** that breaks the app down and distributes its pieces into system-wide directories.

Here is exactly what happens during the installation process for both operating systems.

---

## The Windows Installation Process

When you double-click an `.exe` or `.msi` installer file on Windows, the following sequence occurs:

### 1. Verification and Privileges

- **UAC Prompt:** The User Account Control (UAC) triggers a pop-up asking for administrator permissions. This grants the
  installer permission to modify system files and the registry.
- **Extraction:** The installer extracts a temporary copy of its contents into your `C:\Users\Username\AppData\Local\Temp`
  directory to prepare for setup.

### 2. Directory Creation and File Copying

- **Target Folder:** The installer creates a dedicated folder for the application, typically inside `C:\Program Files` or
  `C:\Program Files (x86)`.
- **Copying Assets:** It moves all executables (`.exe`), dynamic link libraries (`.dll`), images, and asset files into that
  specific folder.

### 3. Modifying the Windows Registry

- **Configuration:** The installer writes keys into the Windows Registry (specifically under `HKEY_LOCAL_MACHINE` or
  `HKEY_CURRENT_USER`).
- **Integration:** This step tells Windows how to handle file associations (e.g., making sure `.psd` files open with
  Photoshop) and logs the app so it appears in the "Add or Remove Programs" settings menu.

### 4. Creating Shortcuts and Finalizing

- **Shortcuts:** The installer places shortcut links (`.lnk` files) onto your Desktop and into the Start Menu directory
  (`C:\ProgramData\Microsoft\Windows\Start Menu`).
- **Cleanup:** The temporary files used during setup are deleted, and the installer closes.

---

## The Linux Installation Process

When you run an install command (like `sudo apt install app-name` or `sudo pacman -S app-name`), Linux uses a centralized
approach:

### 1. Repository Lookup and Dependency Resolution

- **The Manifest Check:** The package manager contacts the configured online software repositories to find the requested
  package and fetch its metadata.
- **Dependency Tree:** The manager checks if the app requires external libraries (dependencies) to run. If your system
  doesn't have them, it automatically adds those missing libraries to the download queue.

### 2. Downloading and Verifying

- **Payload Retrieval:** The package manager downloads the compressed archive (like a `.deb` or `.rpm` file).
- **GPG Key Verification:** Linux verifies the digital signature of the package against trusted local keys to ensure the
  software hasn't been tampered with or corrupted.

### 3. File Distribution (Unpacking)

Unlike Windows, Linux does not put everything into one folder. It unpacks the archive and scatters files into specific,
standardized directories based on the **Filesystem Hierarchy Standard (FHS)**:

- **Executables** go to `/usr/bin/` or `/usr/local/bin/`
- **Shared Libraries** (`.so` files, the Linux equivalent of `.dll`) go to `/usr/lib/`
- **Global Configurations** go to `/etc/`
- **Icons and Documentation** go to `/usr/share/`

### 4. Running Post-Install Scripts and Registering

- **Configuration Scripts:** The package manager executes built-in scripts to set correct user permissions, update the
  system's environment variables, and rebuild library caches (like running `ldconfig`).
- **Desktop Integration:** A `.desktop` text file is placed in `/usr/share/applications/`, which tells your desktop
  environment to display the app icon in your applications menu.
- **Local Database Update:** The package manager logs the exact version and file paths in its local database, making clean
  uninstallation or global updates incredibly easy later on.
