To understand how an SSD writes, updates, and deletes data, you have to look at its underlying architecture. Unlike a
traditional Hard Disk Drive (HDD) that can overwrite data anywhere on a spinning platter, an Solid-State Drive (SSD) uses
**NAND flash memory**, which has a strict quirk: **it cannot overwrite existing data directly.**

Here is exactly how these operations work under the hood.

---

## 1. The Core Architecture: Pages vs. Blocks

Before looking at the operations, you need to know how an SSD is organized:

- **Pages:** The smallest unit for **reading and writing** data (typically 4KB to 16KB).
- **Blocks:** A collection of pages (usually 128 to 512 pages). This is the smallest unit for **erasing** data.

> ⚠️ **The Golden Rule of SSDs:** Data can be written at the _page_ level, but can only be erased at the _block_ level. You
> cannot erase a single page by itself.

---

## 2. Writing Data (Fresh Write)

When an SSD is brand new or empty, writing data is incredibly fast and straightforward.

- The SSD controller finds empty, pristine pages.
- It writes the data directly to those pages sequentially.
- Because the pages were already clean (erased), there is no extra overhead.

---

## 3. Updating Data (Write Amplification)

Because an SSD cannot just overwrite a page with new information, an "update" is actually a multi-step game of musical chairs
managed by the **Flash Translation Layer (FTL)**.

If you want to update a file in **Page A**:

1. **Write New Data:** The SSD writes the updated information to a completely new, empty page (**Page B**).
2. **Mark Old Data:** The SSD marks the original **Page A** as **"stale" or "invalid."** The data is still there, but the
   drive knows it’s old news.
3. **Remap:** The FTL updates its internal map so that whenever your computer asks for that file, it points to Page B instead
   of Page A.

---

## 4. Deleting Data & Garbage Collection

When you click "Delete" on your computer, the data isn't instantly scrubbed from the drive.

### The Deletion Step (TRIM)

- When you delete a file, the Operating System sends a **TRIM command** to the SSD.
- This command tells the SSD controller, _"Hey, these specific pages contain data we don't need anymore."_
- The SSD marks those pages as **invalid**. No actual erasing happens yet because, remember, it can't erase single pages.

### The Cleanup Step (Garbage Collection)

Eventually, the SSD will start running out of blank, pristine pages. To make room, it triggers a process called **Garbage
Collection**:

1. **Find a Target:** The controller looks for a block full of mixed data (some valid pages, some invalid/deleted pages).
2. **Copy the Good:** It reads only the _valid_ pages from that old block and copies them over to a brand-new, empty block.
3. **Wipe the Block:** Now that the old block contains _only_ invalid data, the SSD erases the entire block in one flash.
4. **Reset:** That block is now completely empty and ready to be written to again.

---

## Summary Comparison

| Operation  | What the Computer Thinks Happens          | What the SSD Actually Does                                                                            |
| ---------- | ----------------------------------------- | ----------------------------------------------------------------------------------------------------- |
| **Write**  | Puts data into an empty slot.             | Fills pristine, empty **pages**.                                                                      |
| **Update** | Overwrites the old file with new changes. | Writes changes to a **new page**; marks the old page as **invalid**.                                  |
| **Delete** | Erases the file immediately.              | Marks pages as **invalid (via TRIM)**. Erases them later at the **block level (Garbage Collection)**. |
