When we talk about **Node.js**, **`epoll`**, and **`io_uring`**, we are looking at the evolution of how Node.js communicates
with the Linux kernel to achieve its legendary high-concurrency, non-blocking I/O.

To understand why this relationship matters, we have to look at how Node.js works under the hood via its underlying I/O
library, **`libuv`**.

---

## 1. The Baseline: Node.js and `epoll`

For over a decade, **`epoll`** has been the backbone of Node.js on Linux. It is a kernel notification mechanism designed for
scalable network I/O.

### How it works:

When Node.js handles thousands of concurrent TCP connections, it doesn't spin up a thread for each one. Instead, it relies on
a **Readiness-based model**:

1. Node.js (via `libuv`) registers thousands of network sockets (file descriptors) with `epoll`.
2. The event loop calls `epoll_wait()`, which blocks until _at least one_ socket is ready to read or write.
3. When data arrives, `epoll` wakes up Node.js, telling it: _"Hey, socket #42 has data ready!"_
4. Node.js then executes a system call (like `read()`) to actually fetch that data into user space, and fires your JavaScript
   callback.

### The Catch with File I/O:

`epoll` is amazing for network sockets, but **it does not support regular files** on disk. Disk I/O cannot be truly
non-blocking in the same way. To fix this, Node.js has to offload file operations (like `fs.readFile`) to a background
**Thread Pool** (libuv's worker threads). This adds context-switching overhead.

---

## 2. The Next Generation: `io_uring`

Introduced to the Linux kernel by Jens Axboe, **`io_uring`** is a radical rethink of asynchronous I/O. Instead of a
**Readiness** model, it uses a **Proactor (Completion) model**.

### How it works:

`io_uring` establishes two ring buffers shared directly between Node.js (user space) and the Linux kernel:

- **Submission Queue (SQ):** Node.js pushes I/O requests into this ring (e.g., _"Read this file"_ or _"Write to this
  socket"_).
- **Completion Queue (CQ):** The kernel pushes completed results into this ring (e.g., _"I finished reading that file, here
  is the data"_).

Because the memory is shared, Node.js can submit and reap dozens of I/O operations with a single `io_uring_enter` system
call—or even **zero system calls** if kernel polling (`IORING_SETUP_SQPOLL`) is enabled.

---

## 3. Key Differences: `epoll` vs. `io_uring`

| Feature                  | `epoll`                                                        | `io_uring`                                                                 |
| ------------------------ | -------------------------------------------------------------- | -------------------------------------------------------------------------- |
| **Model**                | **Readiness**: Tells you _when_ you can perform I/O.           | **Completion**: Does the I/O for you and tells you when it's _done_.       |
| **System Call Overhead** | High. Requires `epoll_wait` + subsequent `read`/`write` calls. | Ultra-low. Uses shared ring buffers; minimizes or eliminates system calls. |
| **File System I/O**      | Unsupported. Forces Node.js to use a slow thread pool.         | Fully supported. True asynchronous disk I/O natively in the kernel.        |
| **Data Copying**         | Requires copying data across the user/kernel space barrier.    | Supports registered buffers to allow **Zero-Copy** I/O.                    |

---

## 4. Where does Node.js stand with `io_uring`?

Because Node.js relies entirely on `libuv` to abstraction-layer the operating system (using `epoll` on Linux, `kqueue` on
macOS, and `IOCP` on Windows), any major architecture shift has to happen inside `libuv`.

- **Experimental Native Integration:** `libuv` has ongoing architectural work to adopt `io_uring`. However, replacing a
  bulletproof ecosystem built around `epoll` is an incredibly delicate task because `io_uring` requires relatively modern
  Linux kernels and handles resource lifecycles differently.
- **The Current Landscape:** In modern Node.js development, `epoll` remains the default engine for standard network loops.
  However, newer runtime competitors like **Bun** and **Deno** have integrated `io_uring` aggressively to claim ultra-low
  latency file and network performance on Linux.

If you want to use `io_uring` in Node.js for specialized, high-performance file streaming or socket servers, developers often
bridge the gap using native C++ addons or NPM packages like `node-io-uring`.
