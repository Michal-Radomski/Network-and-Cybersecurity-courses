The **C10k problem** is a famous challenge in computer networking that refers to the inability of a web server to handle
**10,000 concurrent (simultaneous) connections** at the same time.

The term was coined in 1999 by Dan Kegel, a prominent software engineer, at a time when the internet was growing rapidly, and
existing web servers were crashing under heavy traffic—even though the server hardware had plenty of CPU power and RAM left
to spare.

---

## Why Was It a Problem?

In the early days of the web, servers handled connections using a **one-thread-per-connection** or
**one-process-per-connection** model.

When a user visited a website, the server spawned a dedicated thread to talk to that user. If 10,000 users connected at once,
the server tried to create 10,000 threads.

This caused two massive bottlenecks:

1. **Memory Exhaustion:** Each thread requires its own memory space (stack memory). Multiplying that by 10,000 quickly ate up
   all the server's RAM.
2. **Context Switching Overhead:** The CPU can only do so many things at once. It had to constantly switch between 10,000
   threads to give everyone a turn. The CPU spent more time _switching_ between threads (context switching) than actually
   doing real work, causing the server to grind to a halt.

---

## How Was the C10k Problem Solved?

The solution wasn't just "buy bigger computers." It required a fundamental shift in how operating systems and web servers
manage network traffic.

The industry moved from **blocking, thread-per-connection** architectures to **asynchronous, event-driven** architectures.

### 1. Efficient I/O Multiplexing

Instead of a thread waiting around doing nothing while a user reads a webpage, operating systems introduced new system calls
that allowed a single thread to monitor thousands of connections at once.

- **Linux** introduced `epoll`
- **BSD/macOS** introduced `kqueue`
- **Windows** used `IOCP` (Input/Output Completion Ports)

These tools let the operating system tell the server, _"Hey, out of these 10,000 connections, only these 3 just sent new
data. Deal with them now."_

### 2. Event-Driven Software

Newer web servers and runtimes were built from the ground up to utilize these operating system upgrades. Instead of creating
thousands of threads, they use a tiny pool of threads (often just one per CPU core) running an **Event Loop**.

- **Nginx:** Unlike Apache (which struggled with C10k due to its process-based model), Nginx used an event-driven model and
  easily conquered the C10k barrier, becoming the backbone of the modern high-traffic web.
- **Node.js:** Popularized asynchronous, event-driven JavaScript on the backend, allowing a single thread to handle massive
  amounts of concurrent connections.

---

## What is the Problem Today?

Because the C10k problem was solved in the mid-2000s, engineers have moved the goalposts. Today, high-scale systems talk
about the **C10M problem**—how to handle **10 million** concurrent connections on a single server. Solving that requires
bypassing the standard operating system kernel entirely and using specialized hardware optimization.
