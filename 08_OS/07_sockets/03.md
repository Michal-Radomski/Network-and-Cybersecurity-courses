To understand how a file descriptor works in a TCP connection, it helps to first understand a classic philosophy of Unix-like
operating systems (such as Linux and macOS): **"Everything is a file."** Whether the system is interacting with a hard drive,
a keyboard, a printer, or a network connection over TCP, it treats them all as streams of bytes.

Here is a breakdown of what a file descriptor is and how it manages a TCP connection.

---

## What is a File Descriptor (FD)?

A **File Descriptor (FD)** is simply a non-negative integer (like `3`, `4`, or `5`) that the operating system kernel uses to
track an open resource.

When a program wants to interact with the outside world, it asks the OS to open a channel. The OS creates an internal data
structure to manage that channel and hands the program a number (the FD) as a "receipt." Instead of dealing with complex
network protocols or hardware details directly, the program just tells the OS: _"Hey, send this data to file descriptor 4."_

### The Standard FDs

Every time a process starts, the OS automatically opens three standard file descriptors:

- **0:** Standard Input (`stdin`) - Usually the keyboard.
- **1:** Standard Output (`stdout`) - Usually the terminal screen.
- **2:** Standard Error (`stderr`) - For error messages on the screen.

Any new file or network connection opened after this will start at **3** or higher.

---

## How an FD Works in a TCP Connection

When you establish a TCP connection, the operating system creates a **network socket**. A socket is the software endpoint for
the connection. The OS then assigns a file descriptor to that socket.

Here is the step-by-step lifecycle of how FDs manage a TCP connection:

### 1. Creating the Socket

The application calls a function like `socket()`. The OS allocates network resources and returns an FD (let's say `3`). At
this stage, it's just a blank socket that doesn't know where it's going yet.

### 2. Establishing the Connection

- **For a Client:** The program calls `connect(3, Server_IP, Port)`. The OS uses FD 3 to initiate the TCP three-way handshake
  with the server.
- **For a Server:** The server uses a "listening socket" (e.g., FD `3`) to listen for incoming requests. When a client
  connects, the OS accepts it and creates a _brand new_ socket with a _new_ file descriptor (e.g., FD `4`) specifically for
  that client. This keeps FD `3` free to listen for more incoming connections.

### 3. Reading and Writing Data

Once connected, the program doesn't need to know anything about TCP segments, IP addresses, or network packets. It uses
standard file operations:

- `write(4, buffer, size)` sends data out over the TCP connection.
- `read(4, buffer, size)` grabs incoming data that the OS received from the network.

### 4. Closing the Connection

When the conversation is over, the program calls `close(4)`. The OS finishes sending any leftover data, tears down the TCP
connection using the standard `FIN/ACK` handshake, and frees up the number `4` so it can be reused for future files or
connections.

---

## Behind the Scenes: The OS Tables

The file descriptor integer doesn't hold any network data itself. It is actually an index pointer that points through a chain
of tables managed by the kernel:

1. **Process Descriptor Table:** A private list for your specific program. Integer `4` points to a slot in this table.
2. **System-wide Open File Table:** The OS's master list of everything open across the entire computer. The slot from your
   process table points here. This table tracks things like the status flags (read/write mode).
3. **Network Socket Inode:** For a TCP connection, the entry points to a network socket structure in memory. This structure
   contains the **TCP 4-tuple** that uniquely defines the connection:

- Source IP Address
- Source Port
- Destination IP Address
- Destination Port

> **Note on Limits:** Because file descriptors are tracked in tables, operating systems have limits on how many FDs a single
> process (or the whole system) can have open at once. If a busy web server doesn't close its TCP connections properly, it
> can run out of file descriptors, leading to a "Too many open files" error, which prevents any new clients from connecting.
