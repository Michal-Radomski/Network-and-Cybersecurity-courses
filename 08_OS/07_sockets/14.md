In Ubuntu, there isn't just a single "TCP Listen Backlog" value; it is governed by a combination of a **kernel-level maximum
limit** and the **application-level requested value**.

Here is how the default values break down:

### 1. The Kernel Limit (`somaxconn`)

The absolute maximum backlog limit for a TCP listening socket is defined by the kernel parameter `net.core.somaxconn`. If an
application requests a backlog larger than this value, the kernel will silently truncate it to this maximum.

- **Default Value:** **`4096`** (in modern Ubuntu versions like 22.04 and 24.04).
- _Note:_ In older Ubuntu/Linux releases (prior to kernel 5.4), the default was historically **`128`**.

You can check your system's current live value by running:

```bash
sysctl net.core.somaxconn

```

---

### 2. The IP-Layer Queue (`tcp_max_syn_backlog`)

Another relevant backlog value is `net.ipv4.tcp_max_syn_backlog`. This dictates the maximum number of "half-open" connections
(connections that have sent a `SYN` but haven't yet completed the 3-way handshake).

- **Default Value:** Typically **`256`**, **`512`**, or **`1024`** depending on your system's total RAM, though it can scale
  higher automatically on high-memory systems.

You can check this value via:

```bash
sysctl net.ipv4.tcp_max_syn_backlog

```

---

### 3. Application-Specific Backlogs

When a developer writes a networking application (like Nginx or Apache) and opens a port, they pass a `backlog` argument to
the `listen()` system call.

If the application doesn't explicitly define a number, it often defaults to a standard value or inherits a language-specific
default:

- **Nginx default:** `511`
- **Apache (httpd) default:** `511`
- **Python (`socket.listen()`):** If no value is passed, Python automatically uses a reasonable default (often matching
  `somaxconn`).

> ⚠️ **The Golden Rule:** The actual backlog size used by a running application will always be the **lesser** of the
> application's requested backlog and the kernel's `net.core.somaxconn`.

### How to increase the limits

If you are running a high-traffic server and seeing connection drops, you can increase these limits by adding the following
lines to `/etc/sysctl.conf`:

```text
net.core.somaxconn = 8192
net.ipv4.tcp_max_syn_backlog = 4096

```

Then, apply the changes instantly with `sudo sysctl -p`.
