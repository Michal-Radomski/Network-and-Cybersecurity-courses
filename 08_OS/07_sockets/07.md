When building networked applications, **socket programming** acts as the backbone for communication. Depending on your
application's requirements—such as how many concurrent users you have and how much data you are transferring—different design
patterns are used to handle connections efficiently.

Here is a breakdown of the most common socket programming patterns, ranging from simple baseline models to highly scalable
architecture.

---

## 1. The Iterative Pattern (Blocking / Single-Threaded)

This is the simplest, most fundamental pattern. The server handles exactly **one connection at a time** from start to finish
before moving on to the next.

- **How it works:** The server binds to a port, listens, and calls `accept()`. It blocks (waits) until a client connects,
  processes that client's request entirely, closes the connection, and loops back to `accept()`.
- **Pros:** Incredibly easy to implement and debug; low CPU overhead.
- **Cons:** If Client A is slow or idle, Client B is stuck waiting in the OS queue indefinitely.
- **Best For:** Simple logging tools, local debugging, or environments where clients connect sequentially.

---

## 2. Multi-Threaded / Multi-Process Per Connection

To overcome the blocking nature of the iterative pattern, this pattern spawns a new thread or process for every incoming
client.

- **How it works:** The main thread spins in a loop, listening for connections. The moment `accept()` returns a new client
  socket, the main thread hands that socket off to a **newly created thread** (or child process) and immediately goes back to
  listening.
- **Pros:** True concurrency. Client B doesn't have to wait for Client A to finish.
- **Cons:** **Context switching** and memory overhead. If 10,000 clients connect, creating 10,000 threads will quickly crash
  the system or degrade performance drastically due to memory exhaustion.
- **Best For:** Moderate traffic applications where connections are short-lived or processing time per request is high.

---

## 3. Thread Pool Pattern

Instead of dynamically creating and destroying threads on the fly, this pattern uses a fixed number of pre-allocated worker
threads.

- **How it works:** A "pool" of threads is created at server startup. When a new connection arrives, it is placed into a
  **request queue**. Available worker threads pull connections from the queue, handle them, and return to the pool when done.
- **Pros:** Eliminates the overhead of constantly creating/destroying threads and caps maximum resource usage.
- **Cons:** If all threads are busy and the queue fills up, new clients will experience delays or get rejected.
- **Best For:** Standard web servers and enterprise applications handling predictable, steady traffic.

---

## 4. Non-Blocking / Event-Driven Pattern (Reactor Pattern)

This is the modern standard for high-performance, massive-scale applications (e.g., Node.js, Nginx, Netty). It shifts the
paradigm from "one thread per connection" to "one thread managing thousands of connections via events."

- **How it works:** Sockets are configured to be **non-blocking** (they return immediately instead of waiting for data). A
  central **Demultiplexer** (using OS APIs like `epoll` on Linux, `kqueue` on macOS, or `IOCP` on Windows) monitors all
  sockets. When a socket actually has data ready to read, an event is fired, and a **Reactor** dispatches that event to a
  specific handler.
- **Pros:** Incredible scalability (can handle tens of thousands of concurrent connections—the "C10k problem"—on a single
  thread). Highly efficient resource usage.
- **Cons:** Highly complex architecture; "callback hell" or complex async/await state machines; a single blocking operation
  in a handler can freeze the entire server.
- **Best For:** Real-time chat apps, gaming servers, live streaming, and high-throughput API gateways.

---

## Summary Comparison

| Pattern                    | Complexity | Resource Consumption  | Scalability    | Best Match              |
| -------------------------- | ---------- | --------------------- | -------------- | ----------------------- |
| **Iterative**              | Very Low   | Very Low              | Extremely Low  | Simple/Local tools      |
| **Multi-Threaded**         | Low        | High (per connection) | Low-Medium     | Low-traffic apps        |
| **Thread Pool**            | Medium     | Medium (Capped)       | Medium-High    | Traditional web servers |
| **Event-Driven (Reactor)** | High       | Low                   | Extremely High | Real-time / C10k apps   |
