In the world of networking, **TCP (Transmission Control Protocol)** uses "flags" to manage the state of a connection. Think
of these flags as little light switches inside the packet header that tell the receiving computer exactly what the sender
wants to do—whether that's starting a conversation, acknowledging data, or saying goodbye.

Each flag is a single bit (0 or 1) in the TCP header.

---

## The Primary TCP Flags

### 1. SYN (Synchronize)

The "Let’s talk" flag. It is used during the initial setup of a connection. When you type a URL into your browser, your
computer sends a packet with the **SYN** flag set to **1** to the server to initiate the "Three-Way Handshake."

### 2. ACK (Acknowledgment)

The "I got it" flag. This is used to confirm that a packet was received successfully. Almost every packet sent after the
initial SYN will have the **ACK** flag set. It includes an "Acknowledgment Number" that tells the sender which byte of data
is expected next.

### 3. FIN (Finish)

The "Let’s wrap this up" flag. When a device has no more data to send, it sends a packet with the **FIN** flag. This
initiates a graceful teardown of the connection. Unlike a sudden hang-up, both sides must agree to close their ends of the
"pipe."

### 4. RST (Reset)

The "Hang up immediately" flag. This is the "nuclear option." If a computer receives a packet for a connection it doesn't
recognize, or if there is a serious error, it sends an **RST** to kill the connection instantly without a graceful goodbye.

### 5. PSH (Push)

The "Don't wait" flag. Normally, TCP buffers data (collects a bit of it) before sending it to the application to be more
efficient. The **PSH** flag tells the receiving system to bypass the buffer and send the data to the application immediately.

### 6. URG (Urgent)

The "Priority mail" flag. This tells the receiver that the data contained in the packet should be processed immediately,
ahead of other data in the queue. It is rarely used in modern web traffic but was common in older protocols like Telnet.

---

## How They Work Together: The Three-Way Handshake

To understand these flags in action, look at how a connection starts:

1.  **SYN**: Client says, "I want to sync with you."
2.  **SYN-ACK**: Server says, "I acknowledge your request, and I also want to sync with you."
3.  **ACK**: Client says, "I acknowledge your sync, let's start sending data."

---

## Lesser-Known Flags (Congestion Control)

As networking evolved, a few more flags were added to help manage traffic jams on the internet:

- **ECE (ECN-Echo):** Indicates that the network is congested.
- **CWR (Congestion Window Reduced):** Sent by the host to confirm it received the ECE flag and has slowed down its sending
  rate.

| Flag    | Purpose                 | Analogy                  |

The teardown of a connection is slightly more complex than the setup because it’s a "full-duplex" process. This means that
even if one side is finished sending data, it might still be open to receiving data from the other side.

Because of this, it usually takes four steps (or "four way-handshake") to fully close the doors.

## The Four-Way Handshake (Graceful Termination)

1.  **FIN**: The first device (let's say the Client) sends a packet with the **FIN** flag set. This tells the Server, "I have
    no more data to send to you."
2.  **ACK**: The Server responds with an **ACK**. This means "I hear you. I'm acknowledging your request to close your side."
    At this point, the connection is in a "half-close" state. The Server can still send data to the Client if it needs to
    finish its own tasks.
3.  **FIN**: Once the Server is also done sending data, it sends its own **FIN** packet to the Client, effectively saying,
    "I’m done on my end too."
4.  **ACK**: The Client sends a final **ACK** back to the Server. After a short waiting period (to ensure no late packets are
    floating around), the connection is officially closed.

---

## Why "RST" is Different

While **FIN** is like saying "Goodbye, see you later," **RST** (Reset) is like slamming the phone down.

An **RST** flag is used when:

- A computer receives a packet for a port that isn't open.
- The connection has timed out or become unsalvageable.
- A firewall or security system wants to kill a suspicious connection immediately.

## Port Scanning and Flags

Security professionals (and hackers) use these flags to "poke" at a system to see if it's alive.

- **SYN Scan (Stealth Scan):** The scanner sends a **SYN**. If the target sends a **SYN-ACK**, the port is open. Instead of
  sending the final **ACK** to complete the connection, the scanner sends an **RST** to close it quickly before it's logged.
- **Xmas Scan:** A weird historical trick where a packet is sent with **FIN**, **PSH**, and **URG** all set at once (lighting
  up the packet "like a Christmas tree"). Most modern systems just ignore these or drop them.
