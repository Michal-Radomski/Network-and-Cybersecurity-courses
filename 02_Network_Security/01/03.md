The OSI (Open Systems Interconnection) and TCP/IP (Transmission Control Protocol/Internet Protocol) models are frameworks
used to understand how data moves across a network. While OSI is a theoretical reference tool with seven layers, TCP/IP is
the practical standard with four or five layers that actually runs the modern internet. [1, 2, 3]

## Key Comparisons

| Feature [1, 2, 3, 4, 5, 6, 7, 8, 9] | OSI Model                           | TCP/IP Model                    |
| ----------------------------------- | ----------------------------------- | ------------------------------- |
| Number of Layers                    | 7 Layers                            | 4 or 5 Layers                   |
| Approach                            | Theoretical / Reference             | Practical / Implementation      |
| Development                         | Developed by ISO in 1984            | Developed by DARPA in the 1970s |
| Status                              | Used for teaching & troubleshooting | The actual internet standard    |

---

## The OSI Model (7 Layers)

The OSI model strictly separates network functions into independent layers, making it ideal for systematic troubleshooting.
[1, 10]

- Layer 7: Application – Direct interaction with software (e.g., HTTP for browsers).
- Layer 6: Presentation – Formats, encrypts, and compresses data.
- Layer 5: Session – Manages the start, stop, and restart of connections.
- Layer 4: Transport – Ensures reliable end-to-end data delivery (e.g., TCP).
- Layer 3: Network – Handles routing and logical addressing (IP addresses).
- Layer 2: Data Link – Transfers data between nodes on the same network (MAC addresses).
- Layer 1: Physical – Transmits raw bits over hardware like cables and Wi-Fi. [11, 12, 13, 14, 15, 16]

---

## The TCP/IP Model (4/5 Layers)

The TCP/IP model is more flexible and combines several OSI layers into fewer functional blocks. [6, 17]

- Application Layer – Combines OSI's Application, Presentation, and Session layers.
- Transport Layer – Handles host-to-host communication, similar to OSI Layer 4.
- Internet Layer – Equivalent to OSI's Network Layer; routes packets via IP.
- Network Access (Link) Layer – Combines OSI's Data Link and Physical layers to handle hardware connections. [2, 18, 19, 20,
  21]

## Why Both Exist

Networking professionals use the OSI model to diagnose problems—for example, saying an issue is "at Layer 1" immediately
tells a technician to check the physical cables. However, the software on your computer actually uses the TCP/IP model
because its simpler, protocol-based structure is more efficient for real-world data transmission. [1, 10, 22]

Yes, when you run a React app, your interaction happens at the 7th Layer (Application Layer), but there is a bit more to the
story regarding how it runs across different operating systems.

## 1. Does it happen in the 7th Layer?

Yes. From a networking perspective, the React app lives in the Application Layer.

- The Request: When you type a URL, your browser uses HTTP/HTTPS (Layer 7 protocols) to fetch the app's files (HTML, JS,
  CSS).
- The Execution: Once those files arrive, the browser "runs" the React code. Since the browser is an application providing a
  service directly to you, it is the definition of Layer 7.

## 2. Does it run the same on every OS?

Practically, yes. Technically, it depends on the browser. React doesn't talk to Windows or Linux directly; it talks to the
Browser Engine (like V8 in Chrome/Edge or SpiderMonkey in Firefox). Because the browser acts as a "middleman" or a virtual
environment, the React code is cross-platform. However, small differences can occur because of:

- Browser Engines: A React app might look or behave slightly differently in Safari (on macOS) versus Chrome (on Windows)
  because they interpret CSS and JavaScript slightly differently.
- System Resources: Linux and Windows manage memory and CPU differently. A heavy React app might feel smoother on one OS than
  the other if the browser handles the OS's resources better.
- Fonts and UI Elements: Default scrollbars, fonts, and form buttons (like dropdowns) often use "native" styling, so they
  will look like "Windows style" on a PC and "GTK/Ubuntu style" on Linux.

## The "Sandboxing" Reality

The reason React is so consistent is Sandboxing. Browsers create a safe, isolated box for websites to run in. This box
prevents the React code from knowing too much about your OS (for security), which accidentally ensures that the code behaves
almost identically whether you are on a laptop, a Steam Deck, or an Android phone. Would you like to know how the Transport
Layer (Layer 4) handles the data packets that actually make up your React app?

---

When you load that React app, the Transport Layer (Layer 4) is the "delivery manager." It doesn't care about the code inside;
its only job is to get the data from the server to your browser reliably and in the right order. For a React app, this
happens almost exclusively via TCP (Transmission Control Protocol). Here is the step-by-step breakdown:

## 1. The "Three-Way Handshake" (Connection)

Before a single line of React code is sent, your browser and the server perform a handshake to make sure they are both ready:

- SYN: Your PC says, "I'd like to connect on Port 443 (HTTPS)."
- SYN-ACK: The server says, "I received your request, I'm ready."
- ACK: Your PC says, "Great, I'm starting the download now."

## 2. Segmentation (Breaking it down)

React apps can be large (many megabytes of JavaScript). The Transport Layer breaks that big file into small segments (usually
about 1,500 bytes each).

- Each segment gets a Sequence Number.
- This is like numbering the pages of a book so if they arrive shuffled, the browser can put them back in order.

## 3. Error Recovery (Reliability)

This is where Layer 4 is a lifesaver for web apps. If a packet is lost in the "noise" of the internet:

- The browser realizes a sequence number is missing (e.g., it got segments 1, 2, and 4, but not 3).
- It asks the server to retransmit only segment 3.
- Why this matters for React: If you lost even 1% of your JavaScript code, the app would crash or fail to load. TCP ensures
  100% of the code arrives intact.

## 4. Port Numbers (The Delivery Entrance)

Layer 4 uses Ports to make sure the data goes to the right "door" on your computer.

- Even if you have 10 tabs open, Layer 4 sees that the React data is destined for Port 54321 (a temporary port your browser
  opened), while your Spotify stream is going to Port 4000. This prevents your React code from accidentally ending up in your
  music player.

## Summary: TCP vs. UDP

In networking, there’s a faster alternative called UDP, but it doesn't check for errors.

- React uses TCP: Because if you lose a piece of the code, the app breaks.
- Video Calls (Zoom) use UDP: Because if you lose a tiny "frame" of video, it’s better to just skip it and keep the live
  stream going than to pause and wait for a retransmission.
