The **TLS (Transport Layer Security) handshake** is the process that kicks off a secure communication session between a
client (like your web browser) and a server. It’s essentially a high-tech "meet and greet" where both parties agree on how to
encrypt their data.

Since the release of **TLS 1.3** (the current standard), the process has been streamlined to be faster and more secure than
previous versions. Here is how a modern TLS 1.3 handshake works.

---

### 1. The "Hello" Phase (ClientHello)

The client starts by sending a message to the server. This isn't just a "hi"; it includes a bundle of necessary information:

- **Protocol Version:** Which version of TLS it wants to use.
- **Cipher Suites:** A list of supported encryption algorithms.
- **Client Random:** A string of random bytes used later to generate encryption keys.
- **Key Share:** In TLS 1.3, the client preemptively sends its portion of a cryptographic key to save time.

### 2. The Server Response (ServerHello)

The server looks at the client’s options and picks the most secure ones they both support. It responds with:

- **Selected Cipher Suite:** The encryption method they will use.
- **Server Random:** Its own string of random bytes.
- **Server Key Share:** Its portion of the cryptographic key.
- **Digital Certificate:** Proof of the server's identity (issued by a Certificate Authority).

### 3. Authentication and Key Generation

Now, things happen behind the scenes:

- **Authentication:** The client checks the server's certificate. If the certificate is valid and matches the domain, the
  client trusts the server.
- **Key Calculation:** Using the "Key Shares" exchanged in steps 1 and 2, both parties independently calculate a **Session
  Key**. Thanks to the Diffie-Hellman algorithm, they arrive at the same key without ever actually sending the full key over
  the internet.

### 4. Finishing Up

- **Server Finished:** The server sends an encrypted message stating it is done with its side of the handshake.
- **Client Finished:** The client sends an encrypted message confirming it is ready.

---

## Why This Matters

The handshake achieves three critical goals for your privacy:

| Goal               | Description                                                                            |
| :----------------- | :------------------------------------------------------------------------------------- |
| **Authentication** | You are talking to the real website, not an imposter.                                  |
| **Encryption**     | Data is unreadable to anyone "eavesdropping" on the connection.                        |
| **Integrity**      | Ensures the data hasn't been tampered with while traveling between you and the server. |

### Fun Fact: 0-RTT

In TLS 1.3, if you have visited a site recently, you can use a feature called **0-RTT (Zero Round Trip Time)**. This allows
the client to send encrypted data in the very first message, making the "handshake" nearly instantaneous.
