Performing a vulnerability scan is a core part of maintaining good "cyber hygiene." It’s essentially a health check for your
computer or network to find weaknesses before someone else does.

When performing a scan, it is vital to remember the "Golden Rule" of networking: **Only scan equipment that you own or have
explicit, written permission to test.** Scanning networks without permission can be flagged as malicious activity by ISPs or
security systems.

Here is the standard workflow for a professional-grade vulnerability scan:

---

### 1. Define the Scope

Before you touch a tool, decide what you are testing.

- **External:** Scanning your public-facing IP address or website.
- **Internal:** Scanning the devices inside your home or office Wi-Fi (laptops, IoT devices, printers).
- **Assets:** Make a list of the IP addresses or hostnames you want to check.

### 2. Choose Your Tools

Depending on your goal, you’ll pick a specific type of scanner:

- **Network Scanners (The All-Rounders):** \* **OpenVAS (GVM):** A powerful, open-source engine that checks for thousands of
  known vulnerabilities.
  - **Nessus:** The industry standard. It has a "Nessus Essentials" version that is free for home use (up to 16 IP
    addresses).
- **Web Application Scanners:**
  - **OWASP ZAP:** An open-source tool specifically for finding bugs in websites (like SQL injection or Cross-Site
    Scripting).
- **Lightweight/Manual Scanners:**
  - **Nmap (with NSE):** As we discussed, using `nmap --script vuln` is a great way to do a quick check for specific, famous
    vulnerabilities.

### 3. Configure and Run the Scan

Most scanners follow a similar process:

1.  **Discovery:** The tool pings the targets to see if they are "alive."
2.  **Port Scanning:** It checks which ports are open (like port 80 for web or 445 for file sharing).
3.  **Service Detection:** It "grabs the banner" of the service to see exactly what software is running (e.g., _Apache
    2.4.41_).
4.  **Vulnerability Matching:** The tool compares the software versions it found against a database of known vulnerabilities
    (called **CVEs**—Common Vulnerabilities and Exposures).

### 4. Analyze the Report

A scan will usually categorize findings by severity:

- **Critical/High:** Fix these immediately. These are usually "holes" that allow someone to take control of the system.
- **Medium:** Often "misconfigurations" that leak information.
- **Low/Informational:** Usually just tips on how to harden the system (e.g., "This service doesn't use TLS 1.3").

### 5. Remediation (The Most Important Part)

A scan is useless if you don't act on it. Usually, the fix is one of three things:

- **Patching:** Updating the software to the latest version.
- **Configuration:** Changing a setting (like disabling an old protocol like SSL 3.0).
- **Shielding:** Putting a firewall in front of a device that can't be patched.

---

### Pro-Tip: Authenticated vs. Unauthenticated

- **Unauthenticated (Black Box):** The scanner looks at the device from the outside, like a stranger looking at your house
  from the street.
- **Authenticated (White Box):** You give the scanner a login. It "walks inside" and can see if internal files or registry
  keys are out of date. This provides a much more accurate and deep report.
