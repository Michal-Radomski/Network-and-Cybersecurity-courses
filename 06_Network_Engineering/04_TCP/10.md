**QUIC** (Quick UDP Internet Connections) is a modern, UDP-based network protocol designed to make web traffic faster, more
secure, and more reliable. Originally developed by Google in 2012, it was later standardized by the Internet Engineering Task
Force (IETF) and serves as the underlying foundation for **HTTP/3**.

Think of QUIC as a complete reimagining of the traditional internet handshake, combining the responsibilities of **TCP**
(reliability) and **TLS** (security) into a single, efficient package.

---

## Why QUIC Was Created: The TCP Problem

For decades, the web relied on the combination of TCP (Transmission Control Protocol) and TLS (Transport Layer Security).
While reliable, this duo suffers from two major flaws in the modern mobile era:

1. **Connection Overhead:** To establish a secure connection, TCP and TLS require multiple back-and-forth "handshakes." This
   causes noticeable latency (lag) before data even starts transmitting.
2. **Head-of-Line Blocking:** If you are downloading multiple files over one TCP connection and a single packet of data gets
   lost, _everything_ stops and waits for that one packet to be resent—even if the other files are completely fine.

---

## Key Features of QUIC

QUIC solves these legacy issues through several clever design choices:

### 1. Near-Instant Connections (0-RTT)

Because QUIC handles both the connection setup and the encryption handshake at the same thread, it slashes the time it takes
to connect.

- **1-RTT:** On the first visit to a site, it takes only one round-trip to establish a secure connection.
- **0-RTT (Zero Round-Trip Time):** If you have connected to the server before, QUIC can start sending encrypted data on the
  _very first packet_, making web pages load almost instantly.

### 2. No Head-of-Line Blocking

QUIC introduces **multiplexing without blocking**. It splits data into independent streams. If a packet in "Stream A" is
lost, it is retransmitted in the background, while "Stream B" and "Stream C" continue downloading without interruption.

### 3. Connection Migration (Perfect for Mobile)

Traditional TCP identifies connections using your IP address. If you switch from Wi-Fi to cellular data, your IP changes,
your TCP connection drops, and your downloads or streams freeze.

- QUIC uses a unique **Connection ID** instead of an IP address. If you switch networks, QUIC smoothly migrates the
  connection without dropping your session or forcing you to reconnect.

### 4. Always Encrypted

Unlike TCP, which was built before security was a priority, QUIC has **TLS 1.3 baked directly into its core**. The protocol
headers themselves are largely encrypted, preventing third parties from tampering with or snooping on the connection
metadata.

---

## How It Fits Into the Network Stack

QUIC essentially replaces the traditional transport layer. It runs on top of **UDP**, which allows it to bypass rigid
operating system kernels and be updated easily via applications (like web browsers).

| Traditional Stack | QUIC Stack             |
| ----------------- | ---------------------- |
| HTTP/2            | **HTTP/3**             |
| TLS 1.3           | _Integrated into QUIC_ |
| TCP               | **QUIC**               |
| IP                | IP                     |
| UDP               | **UDP**                |

---

## Current Adoption

If you use the internet today, you are already using QUIC. It powers a massive portion of global web traffic, supported
natively by major browsers (Chrome, Safari, Edge, Firefox) and utilized heavily by tech giants like Google, Meta, and
Cloudflare to stream video, load apps, and serve content at lightning speeds.
