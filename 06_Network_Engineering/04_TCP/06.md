**Congestion control** is a network management strategy used by protocols like TCP to prevent the entire internet
infrastructure—such as routers, switches, and data links—from collapsing under the weight of too much traffic.

If **flow control** is about a sender not overwhelming a single receiver, **congestion control** is about all senders
collectively not overwhelming the shared highway of the internet. When too many packets are pumped into the network at once,
routers get overwhelmed, their memory buffers fill up, and they are forced to start dropping packets. This digital traffic
jam is known as **network congestion**.

To prevent this, TCP uses a "probe-and-adjust" strategy: it dynamically guesses how busy the network is and changes its
transmission speed on the fly.

---

## 1. The Core Metrics: Congestion Window (cwnd)

To manage congestion, TCP introduces a variable called the **Congestion Window ($cwnd$)**.

The $cwnd$ dictates the maximum number of bytes the sender can transmit before waiting for an acknowledgment (ACK) from the
receiver.

- **Small $cwnd$** = Slow sending speed (safe for a busy network).
- **Large $cwnd$** = Fast sending speed (good for an empty network).

The sender is strictly limited by whichever is smaller: the receiver's available buffer space (Flow Control) or the $cwnd$
(Congestion Control).

---

## 2. The Four Phases of TCP Congestion Control

The classic TCP congestion control algorithm (often referred to as TCP Tahoe/Reno) breaks its behavior down into four
distinct phases to safely find the maximum speed of the network.

### Phase 1: Slow Start

When a connection first begins, TCP has no idea how much traffic the network can handle. It starts incredibly conservatively,
but accelerates exponentially.

1. It sets the $cwnd$ to a very small initial value (typically 1 to 10 segments).
2. For **every single ACK** it receives, it doubles the $cwnd$.
3. The speed skyrockets exponentially ($1 \rightarrow 2 \rightarrow 4 \rightarrow 8 \rightarrow 16 \dots$) until it hits a
   safety threshold called the **Slow Start Threshold ($ssthresh$)**.

### Phase 2: Congestion Avoidance

Once the $cwnd$ hits the $ssthresh$, TCP puts on the brakes. Exponential growth is too dangerous now because it might
suddenly cause a massive crash.

- It switches to **linear growth**.
- Instead of doubling, it increases the $cwnd$ by just **one segment** per round-trip time. It slowly "probes" the network to
  see if it can squeeze a little more data through.

### Phase 3: Congestion Detection (The Crash)

Eventually, the sender pushes the network too hard, a router buffer overflows, and a packet is dropped. TCP detects this
packet loss in one of two ways:

- **Scenario A: Triple Duplicate ACKs:** The receiver sends three identical ACKs in a row, signaling that a specific packet
  is missing but other packets are still getting through. The network is congested, but not completely broken.
- **Scenario B: Timeout:** The sender hears absolutely nothing back for a long time. This is a severe sign that the network
  is heavily choked or a router has crashed.

### Phase 4: Fast Retransmit & Fast Recovery

How TCP reacts depends entirely on the type of detection from Phase 3:

- **If a Timeout occurs (Severe Congestion):** TCP panics. It drops the $cwnd$ all the way back to **1 segment**, cuts the
  $ssthresh$ in half, and starts the "Slow Start" phase all over again from scratch.
- **If Triple Duplicate ACKs occur (Mild Congestion):** Modern TCP utilizes **Fast Recovery**. Instead of dropping all the
  way back to 1, it cuts the $cwnd$ exactly in **half**, sets the new $ssthresh$ to that value, and immediately resumes
  linear **Congestion Avoidance**. This prevents the internet from constantly freezing and stuttering.

---

## 3. Modern Evolution: Loss-Based vs. Delay-Based

The traditional approach described above is **loss-based**—it drives as fast as it can until it crashes into a wall (drops a
packet), and then backs up. This can cause a phenomenon called _bufferbloat_, where routers sit with perpetually full
buffers, causing high latency.

Modern internet algorithms have evolved:

- **TCP BBR (Bottleneck Bandwidth and RTT):** Developed by Google, BBR is a **delay-based** algorithm. Instead of waiting for
  packets to drop, it measures the physical round-trip time of the network. If it notices packets are taking longer to
  arrive, it deduces that router buffers are starting to fill up and tapers off its speed _before_ a single packet is lost.
  BBR is widely used today for YouTube traffic and Google services.
