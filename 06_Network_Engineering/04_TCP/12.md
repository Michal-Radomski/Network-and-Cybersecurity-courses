To understand how a server handles network traffic, we have to look under the hood at how the Operating System (OS) kernel
manages **Sockets**, **Connections**, and **Kernel Queues**.

When a client connects to a server via TCP, the data doesn't just instantly appear in your application code. It moves through
a highly orchestrated assembly line managed by the network stack.

---

## 1. Sockets vs. Connections: What’s the Difference?

It is common to confuse these two, but they represent very different things in the OS.

- **Socket:** An OS abstraction (a file descriptor) that acts as an endpoint for communication. Think of it like a physical
  phone jack on a wall. It has an IP address and a port, but it doesn't necessarily mean a call is active.
- **Connection:** A fully established, active virtual circuit between _two_ specific sockets (Client IP:Port + Server
  IP:Port). This is the actual phone call.

A single server socket (e.g., listening on port 80) can handle thousands of concurrent connections. The kernel distinguishes
between them using a **4-tuple**:

$$\text{(Source IP, Source Port, Destination IP, Destination Port)}$$

---

## 2. The Lifeline of a Connection: Kernel Queues

When a client initiates a TCP connection, it triggers the famous **TCP 3-Way Handshake**. The kernel manages this process
using two distinct queues: the **SYN Queue** and the **Accept Queue**.

### Step 1: The SYN Queue (Incomplete Connection Queue)

1. The client sends a `SYN` packet to the server.
2. The server kernel receives it, creates a "half-open" connection record, and stores it in the **SYN Queue**.
3. The server replies with a `SYN-ACK`.

> ⚠️ **Security Note:** If an attacker floods a server with `SYN` packets but never responds to the `SYN-ACK`, the SYN Queue
> fills up. This is a **SYN Flood DDoS attack**. Modern kernels combat this using **SYN Cookies**, where the connection state
> is encoded into the TCP sequence number rather than stored in a queue.

### Step 2: The Accept Queue (Complete Connection Queue)

1. The client responds with the final `ACK`.
2. The kernel moves the connection out of the SYN Queue and into the **Accept Queue**.
3. The connection is now fully established (`ESTABLISHED` state) and ready for the application.

### Step 3: `accept()`

Your application code (written in Go, Node.js, Python, Python, etc.) calls the `accept()` system call. This removes the
connection from the top of the Accept Queue and returns a **new socket file descriptor** dedicated entirely to that specific
client.

---

## 3. What Happens When Queues Fill Up?

If your application is slow or frozen, it won't call `accept()` fast enough. The Accept Queue will fill up.

When the **Accept Queue overflows**, the kernel's behavior depends on the system configuration (specifically the
`net.ipv4.tcp_abort_on_overflow` setting):

| Setting Value     | Kernel Action                                                             | Result for Client                                                                                                                                        |
| ----------------- | ------------------------------------------------------------------------- | -------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **`0` (Default)** | The kernel ignores the final `ACK` from the client.                       | The client thinks the packet was lost and retries (`TCP retransmission`). If the server recovers, the connection succeeds. If not, the client times out. |
| **`1`**           | The kernel immediately replies to the client with a `RST` (Reset) packet. | The client's connection is instantly rejected with a "Connection refused" or "Connection reset by peer" error.                                           |

---

## 4. Key Kernel Parameters for Tuning

If you are running high-throughput systems, the default OS queue sizes are often too small. System administrators tune these
via `sysctl`:

- **`net.ipv4.tcp_max_syn_backlog`**: Determines the maximum size of the SYN Queue.
- **`net.core.somaxconn`**: The absolute maximum size of the Accept Queue. (Applications also pass a `backlog` argument to
  the `listen()` call, but the kernel caps it at `somaxconn`).

---

## Summary of the Flow

1. **`LISTEN` Socket:** Waits for incoming knocks.
2. **`SYN` arrives:** Connection enters **SYN Queue** ($\text{State} = \text{SYN\_RECV}$).
3. **`ACK` arrives:** Connection moves to **Accept Queue** ($\text{State} = \text{ESTABLISHED}$).
4. **`accept()` called:** Application pulls the connection out of the queue, freeing up space, and gets a dedicated **Client
   Socket** to read/write data.
