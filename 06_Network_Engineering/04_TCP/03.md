A **TCP segment** is the unit of data that the Transmission Control Protocol uses to slice up and transmit information across
a network. When an application (like a web browser) wants to send a large chunk of data, TCP breaks that data down into
manageable pieces, slaps a **TCP header** onto each piece, and hands it off to the IP layer.

The entire segment consists of two parts: the **Header** (which carries routing, tracking, and control information) and the
**Data/Payload** (the actual content being sent).

---

## 1. The TCP Segment Layout

Because TCP provides guaranteed reliability and flow control, its header is highly complex. A standard TCP header is usually
**20 bytes** long, but it can expand up to 60 bytes if special "Options" are used.

It is typically visualized in a 32-bit (4-byte) wide grid:

| Bit 0 to 15                                     | Bit 16 to 31                   |
| ----------------------------------------------- | ------------------------------ | -------------- | ------------------------- |
| **Source Port** (16 bits)                       | **Destination Port** (16 bits) |
| **Sequence Number** (32 bits)                   |                                |
| **Acknowledgment Number** (32 bits)             |                                |
| **Data Offset** (4b)                            | **Reserved** (3b)              | **Flags** (9b) | **Window Size** (16 bits) |
| **Checksum** (16 bits)                          | **Urgent Pointer** (16 bits)   |
| **Options** (0 to 40 bytes, variable, optional) |                                |
| **Data / Payload** (Variable length)            | ...                            |

---

## 2. Detailed Breakdown of Header Fields

### The Basics

- **Source Port (16 bits):** Identifies the sending application's port on the local machine.
- **Destination Port (16 bits):** Identifies the receiving application's port on the remote machine (e.g., port 443 for
  HTTPS).

### The Tracking Engine (Crucial for Reliability)

- **Sequence Number (32 bits):** Tracks the piece of data being sent. Instead of counting packet numbers (1, 2, 3), TCP
  counts the _bytes_. If this segment starts with the 501st byte of data, the sequence number will be `501`. This allows the
  receiver to stitch data back together in the exact right order.
- **Acknowledgment Number (32 bits):** Used by the receiver to say, "I got everything up to byte X, and I am now expecting
  byte X+1 next." If the sender doesn't get this number back, it knows data was lost and will resend it.

### Management & Control

- **Data Offset (4 bits):** Also known as the _Header Length_. It tells the receiver where the TCP header ends and where the
  actual data begins.
- **Reserved (3 bits):** Set to zero; kept blank for future protocol upgrades.
- **Flags (9 bits):** These act as "on/off" switches that trigger specific behaviors in the connection. The most famous flags
  are:
- **SYN (Synchronize):** Initiates a connection (used during the 3-way handshake).
- **ACK (Acknowledgment):** Confirms that data or a connection request was received.
- **FIN (Finish):** Safely tears down and closes a connection when data is done transferring.
- **RST (Reset):** Abruptly aborts a connection due to an error.
- **PSH (Push):** Forces TCP to pass data to the application instantly instead of waiting for a full buffer.

### Traffic Control & Safety

- **Window Size (16 bits):** Used for **flow control**. The receiver uses this field to tell the sender, _"My buffer is
  filling up, please don't send me more than X bytes of data right now."_ This prevents a fast computer from overwhelming a
  slow computer.
- **Checksum (16 bits):** A safety check value. The receiver runs a mathematical equation over the entire segment; if the
  result doesn't match the checksum, the packet was corrupted by network noise and is discarded.
- **Urgent Pointer (16 bits):** Only used if the URG flag is turned on. It indicates that certain data inside the payload
  needs to be processed ahead of everything else (rarely used today).

---

## 3. Options (0 to 40 bytes)

If the data offset field indicates the header is larger than 20 bytes, it means TCP **Options** are being used. These are
extra features negotiated between machines, such as:

- **Maximum Segment Size (MSS):** Tells the other side the maximum size of a segment it can physically handle.
- **Selective Acknowledgments (SACK):** Allows a receiver to say, "I got packets 1, 2, and 4, but 3 was lost," so the sender
  _only_ resends packet 3 instead of rewriting everything from 3 onward.

---

## 4. The Data (Payload)

This is the actual payload passed down from the application layer—such as a piece of an HTML webpage, an image file, or an
encrypted chat message.

The size of the data is limited by the **MTU (Maximum Transmission Unit)** of the network routers it passes through. On
standard Ethernet networks, the maximum size of an entire packet is usually **1500 bytes**. Since an IP header takes 20 bytes
and a standard TCP header takes 20 bytes, the Data payload usually maxes out around **1460 bytes** per segment.
