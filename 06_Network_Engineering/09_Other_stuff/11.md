To get a free SSL/TLS certificate from **Let's Encrypt** on Ubuntu (covering versions 22.04, 24.04, and the upcoming 26.04),
the standard tool is **Certbot**.

Certbot automates the entire process: it proves you own the domain, downloads the certificate, and configures your web server
(Nginx or Apache) to use it.

---

### 1. Prerequisites

Before you begin, ensure you have:

- A server running **Ubuntu**.
- A **registered domain name** (e.g., `example.com`).
- A **DNS A Record** pointing your domain to your server's public IP address.
- Port **80** (HTTP) and **443** (HTTPS) open in your firewall (e.g., `ufw`).

---

### 2. Install Certbot

The recommended way to install Certbot in 2026 is via **Snap** to ensure you always have the latest security updates and ACME
features.

```bash
# Update your package list
sudo apt update

# Install Snapd (usually pre-installed on Ubuntu)
sudo apt install snapd -y

# Ensure your Snap core is up to date
sudo snap install core; sudo snap refresh core

# Remove any old Certbot versions installed via apt
sudo apt remove certbot -y

# Install Certbot
sudo snap install --classic certbot

# Create a symlink to ensure the command is in your path
sudo ln -s /snap/bin/certbot /usr/bin/certbot

```

---

### 3. Obtain and Install the Certificate

Certbot has specialized "plugins" for Nginx and Apache. Choose the command that matches your web server.

#### For Nginx

```bash
sudo certbot --nginx -d example.com -d www.example.com

```

#### For Apache

```bash
sudo certbot --apache -d example.com -d www.example.com

```

**During the interactive setup:**

1. **Email:** Enter a valid email address for renewal and security notices.
2. **Terms:** Agree to the Terms of Service.
3. **Redirect:** When asked if you want to redirect HTTP traffic to HTTPS, choose **Option 2 (Redirect)**. This is the modern
   security standard.

---

### 4. How the Validation Works

To issue a certificate, Let's Encrypt must verify that you actually control the domain. Certbot usually handles this via the
**HTTP-01 Challenge**.

1. **Challenge Request:** Certbot communicates with Let's Encrypt.
2. **Token Placement:** Certbot places a temporary token file in your web directory.
3. **Verification:** Let's Encrypt's servers try to download that file via your domain on port 80.
4. **Issuance:** If successful, the certificate is sent to your server.

---

### 5. Verifying Automatic Renewal

Let's Encrypt certificates are only valid for **90 days**. However, Certbot automatically installs a background timer
(`systemd`) that checks twice a day and renews any certificate within 30 days of expiration.

To test if the renewal process is working correctly, run a "dry run":

```bash
sudo certbot renew --dry-run

```

If you see "Congratulations, all simulated renewals succeeded," your server is set to stay secure forever without manual
intervention.

---

### 6. Where are the files?

If you need to manually reference your certificates (e.g., for a custom Node.js app or HAProxy), they are stored here:

- **Full Chain:** `/etc/letsencrypt/live/example.com/fullchain.pem`
- **Private Key:** `/etc/letsencrypt/live/example.com/privkey.pem`

> **Security Note:** Never share your `privkey.pem` file. It is the secret key that allows your server to prove its identity.
