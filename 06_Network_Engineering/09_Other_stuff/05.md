**EventSource** (also known as **Server-Sent Events** or **SSE**) is a web API that allows a server to push real-time updates
to a web page over a single, long-lived HTTP connection.

Unlike WebSockets, which are **bidirectional** (two-way), EventSource is **unidirectional** (one-way). The server talks, and
the client listens. It is much simpler to implement than WebSockets and is perfect for things like live stock tickers, news
feeds, or social media notifications.

---

### Key Characteristics

- **Automatic Reconnection:** If the connection drops, the browser will automatically try to reconnect.
- **HTTP-Based:** It works over standard HTTP, so it doesn’t require special protocols or server configurations like
  WebSockets often do.
- **Text-Only:** It is designed to send text data (usually JSON).
- **Lightweight:** It has very low overhead compared to polling the server every few seconds.

---

### TypeScript Example

To use `EventSource` in TypeScript, you define the listener and handle the incoming data stream. Because the data usually
arrives as a string, you'll want to parse it into a typed object.

#### 1. The Client Side (TypeScript)

```typescript
interface ServerUpdate {
  id: number;
  message: string;
  timestamp: string;
}

const connectToServer = () => {
  // 1. Initialize the connection to the server endpoint
  const eventSource = new EventSource("https://api.example.com/stream");

  // 2. Handle the generic "message" event
  eventSource.onmessage = (event: MessageEvent) => {
    const data: ServerUpdate = JSON.parse(event.data);
    console.log(`New update [${data.id}]: ${data.message}`);
  };

  // 3. Handle specific named events (if the server sends them)
  eventSource.addEventListener("user-count", (event: MessageEvent) => {
    console.log(`Current users online: ${event.data}`);
  });

  // 4. Error handling (e.g., server goes down)
  eventSource.onerror = (error) => {
    console.error("EventSource failed:", error);
    // The browser will automatically attempt to reconnect
  };

  // To close the connection when finished:
  // eventSource.close();
};

connectToServer();
```

#### 2. The Server Side (Conceptual)

For the code above to work, the server must respond with a specific header: `Content-Type: text/event-stream`. The data sent
from the server looks like this:

```text
event: user-count
data: 42

data: {"id": 1, "message": "Hello World", "timestamp": "2026-05-23T10:11:31Z"}

```

### When to use EventSource vs. WebSockets

| Feature          | EventSource (SSE)                | WebSockets            |
| ---------------- | -------------------------------- | --------------------- |
| **Direction**    | Server $\rightarrow$ Client only | Bidirectional         |
| **Protocol**     | Standard HTTP                    | Custom (ws://)        |
| **Reconnection** | Automatic (Built-in)             | Manual implementation |
| **Data Type**    | UTF-8 Text                       | Binary & Text         |
| **Complexity**   | Very Low                         | High                  |

> **Pro-Tip:** If your app only needs to receive updates (like a dashboard or a sports score feed), **EventSource** is almost
> always the better, more stable choice. If you are building a chat app or a multiplayer game where the user is constantly
> sending data back, use **WebSockets**.
