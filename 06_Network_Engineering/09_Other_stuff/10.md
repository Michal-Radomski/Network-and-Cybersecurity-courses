**HAProxy** (High Availability Proxy) is the "Formula 1 car" of the networking world. It is a free, open-source load balancer
and reverse proxy designed for one thing: moving massive amounts of traffic with extreme speed and reliability.

Unlike other tools that try to be a web server, a cache, and a proxy all at once, HAProxy is a **specialist**.

---

### Core Architecture

HAProxy is written in C and uses a single-process, event-driven model. This allows it to handle tens of thousands of
concurrent connections with a tiny memory footprint.

- **Layer 4 & Layer 7:** It can route traffic at the **TCP level** (Layer 4 - fast, simple) or the **HTTP level** (Layer 7 -
  smart, looks at headers and cookies).
- **Health Checks:** It is famous for its "obsessive" health checking. It doesn't just check if a server is up; it can look
  for specific strings in a response to ensure the app is actually functioning.
- **Zero-Downtime:** Modern versions (3.x as of 2026) support hitless reloads, meaning you can update your config without
  dropping a single active connection.

### 1. HAProxy vs. NGINX

This is the classic debate.

- **NGINX** is a "Swiss Army Knife." It’s an excellent web server, a great cache, and a solid proxy. Most small-to-medium
  sites use NGINX because it’s a "one-stop shop."
- **HAProxy** wins on **raw power**. It typically processes ~30-35% more requests per CPU cycle than NGINX. If you are
  GitHub, Reddit, or Stack Overflow, you use HAProxy because every millisecond and every byte of RAM matters at scale.

### 2. HAProxy vs. Envoy

- **Envoy** was built for the world of **Service Meshes** (like Istio). It is designed to be configured by other software via
  APIs rather than humans editing text files.
- **Envoy** is better at "observability" (showing you exactly what’s happening in a microservice web), but it uses
  significantly more CPU and memory than HAProxy to achieve this.

### 3. HAProxy vs. Traefik

- **Traefik** is the "Zero-Config" king. It watches your Docker or Kubernetes cluster and automatically configures itself
  when you launch a new container.
- **HAProxy** requires manual configuration (or a more complex controller). However, Traefik's convenience comes at a
  performance cost; under extreme stress, Traefik is more likely to drop requests than the battle-hardened HAProxy.

---

### Which one should you choose?

- **Choose HAProxy if:** You have a high-traffic site where performance is the #1 priority, or you need advanced TCP-level
  routing and sophisticated health checks.
- **Choose NGINX if:** You need to serve images/HTML files directly from the proxy or need an easy-to-use, all-in-one web
  server.
- **Choose Traefik/Envoy if:** You are living in a modern Kubernetes/Cloud-native world where your servers are constantly
  appearing and disappearing, and you value automation over raw throughput.

Since HAProxy is a **pure proxy** (designed for routing, not for reading files from a hard drive), it needs to point to a
"backend" that actually holds your images, CSS, and JavaScript.

Think of HAProxy as the **receptionist** at a massive hotel. The receptionist knows where everyone is, but they don't carry
your luggage or clean your room—they send you to the person who does.

---

### Common Backends for Static Files

In a typical production environment, HAProxy usually forwards static file requests to one of these three sources:

#### 1. NGINX or Apache (The Most Common)

Since NGINX is world-class at serving static files, it is very common to see HAProxy sitting in front of a cluster of NGINX
servers.

- **Why:** HAProxy handles the SSL/TLS and complex load balancing, while NGINX handles the high-speed file delivery.
- **How:** You use an **ACL** (Access Control List) in HAProxy to detect static file extensions and route them to an "assets"
  pool.

#### 2. Object Storage (S3, Google Cloud Storage, Azure Blobs)

Modern cloud apps often don't store files on servers at all. HAProxy can act as the entry point that redirects or proxies
requests to a bucket.

- **Why:** It simplifies your server management since you don't have to sync local folders across multiple web nodes.

#### 3. Small "Hacks" for Tiny Files (Lua or Errorfiles)

If you only need to serve a **single** tiny file (like `robots.txt` or a `favicon.ico`), you can actually "cheat" inside
HAProxy:

- **Errorfile Hack:** You can define a custom 200 OK response inside an `errorfile` directive. It feels dirty, but it works
  for 1KB files without needing a backend.
- **Lua Scripts:** HAProxy supports Lua, which can be programmed to read a file from disk and return its content, though this
  is not recommended for high-volume traffic.

### How to Configure the Split

In your `haproxy.cfg`, you would separate the "thinking" (API/Logic) from the "delivery" (Static Files) like this:

```haproxy
frontend main_gateway
    bind *:443 ssl crt /etc/ssl/certs/

    # Check if the URL looks like a static asset
    acl is_static path_end .jpg .png .css .js .ico
    acl is_static_path path_beg /static /assets

    # Route based on the ACL
    use_backend static_servers if is_static or is_static_path
    default_backend api_servers

backend static_servers
    balance roundrobin
    # These could be NGINX servers or a CDN endpoint
    server static1 10.0.0.10:80 check
    server static2 10.0.0.11:80 check

backend api_servers
    balance leastconn
    # These are your Node.js, Python, or Go apps
    server app1 10.0.0.20:5000 check

```

---

### The Modern Alternative: HAProxy + CDN

For large-scale websites in 2026, the strategy is often to skip the HAProxy backend for static files entirely:

1. **User** requests `image.jpg`.
2. **DNS** routes them to a **CDN** (Cloudflare, Akamai, etc.).
3. **CDN** serves the file from its cache.
4. **HAProxy** only ever sees the "dynamic" traffic (login, checkout, profile updates) that the CDN can't handle.
