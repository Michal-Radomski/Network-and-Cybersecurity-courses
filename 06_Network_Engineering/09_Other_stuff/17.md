A **Remote Procedure Call (RPC)** is a software communication protocol that allows one program to request a service from
another program located on a different computer in a network, without having to understand the underlying network details.

In simple terms, RPC makes a network call look and feel exactly like a **local function call** inside your own code.

---

## The Core Concept: Local vs. Remote

To understand RPC, compare it to how normal programming works:

- **Local Procedure Call:** If a program on your laptop needs to calculate tax, it calls a function: `calculateTax(amount)`.
  The computer pauses the main program, runs the function in memory, and immediately hands back the result. It is instant and
  reliable.
- **Remote Procedure Call (RPC):** Imagine your laptop doesn't have the processing power or data to calculate the tax. The
  code still writes `calculateTax(amount)`, but behind the scenes, the RPC framework wraps up that request, sends it over the
  internet to a powerful server, waits for the server to calculate it, and brings the answer back.

To the developer writing the code, the function looks completely local.

---

## How RPC Works (Step-by-Step)

Because computers don't naturally share memory over a network, an RPC framework relies on architectural components called
**Stubs** to trick the program into thinking the call is local.

Here is the exact sequence of events when a client calls a remote function:

1. **The Client Call:** The client application calls a local procedure (e.g., `getUser(id)`). This actually invokes a piece
   of auto-generated code called the **Client Stub**.
2. **Marshalling (Serialization):** The Client Stub takes the parameters (like the integer `id`) and packages them into a
   standardized, compressed binary format that can be sent over a network.
3. **Transmission:** The client’s operating system sends this packaged message over the network (via protocols like TCP, UDP,
   or HTTP/2) to the target server.
4. **Delivery & Unmarshalling:** The server's OS receives the packets and hands them to the **Server Stub** (sometimes called
   a skeleton). The Server Stub unpacks the binary data back into variables the server language understands.
5. **Execution:** The Server Stub calls the _actual_ function residing on the server. The server does the work and returns
   the result to the Server Stub.
6. **The Return Journey:** The process reverses. The Server Stub marshals the return data, ships it back across the network,
   the Client Stub unpacks it, and hands it back to the client application.

---

## Why Use RPC?

RPC is the backbone of modern **distributed systems** and **microservice architectures** (used extensively by companies like
Google, Netflix, and Uber).

- **Language Agnostic:** A client written in Python can call an RPC function on a server written in C++ or Go. The stubs
  handle the translation.
- **Simplicity for Developers:** Programmers don't have to write tedious networking code (dealing with sockets, IP addresses,
  packet loss, or raw byte streams) every time they want services to talk to each other.
- **High Performance:** Modern RPC frameworks are optimized for speed, using compact binary formats instead of bulky text
  formats like JSON or XML.

---

## Popular RPC Implementations

While RPC is a conceptual design, several actual frameworks implement it:

- **gRPC (Google RPC):** Created by Google, this is currently the industry standard for microservices. It uses **HTTP/2** for
  transport and **Protocol Buffers (Protobuf)** to compress messages into tiny binary payloads for ultra-fast performance.
- **Apache Thrift:** Developed by Facebook, similar to gRPC, allowing scalable cross-language services development.
- **XML-RPC / JSON-RPC:** Older, simpler versions of RPC that use text-based XML or JSON over standard HTTP rather than
  optimized binary formats.

---

## Challenges of RPC

While RPC makes networking look easy, it introduces hidden complexities that developers must plan for:

- **The Fallibility of Networks:** A local function call virtually never fails. A remote call can fail if a backhoe cuts a
  fiber-optic cable, a router overheats, or the server crashes mid-calculation.
- **Latency:** Local calls take nanoseconds. Remote calls take milliseconds. If a developer forgets a function is remote and
  calls it thousands of times in a loop, it will severely slow down the application.

## --

To use RPC in a React and Node.js ecosystem, the industry-standard framework is **gRPC** (developed by Google).

Because web browsers (React) cannot natively speak the raw HTTP/2 framing that gRPC requires, we use a tool called
**gRPC-Web** alongside a reverse proxy (like Envoy) to bridge the gap between your React frontend and your Node.js backend.

Here is a complete, streamlined guide to setting up a simple RPC system where a React app requests user data from a Node.js
microservice.

---

## The Architecture

1. **Protocol Buffers (`.proto`)**: Define the data structure and available functions.
2. **Node.js Server**: Implements the actual business logic.
3. **Envoy Proxy**: Translates browser HTTP/1.1 requests into gRPC-compatible HTTP/2.
4. **React Client**: Calls the backend using auto-generated JavaScript stubs.

---

## Step 1: Define the Contract (`user.proto`)

Create a file named `user.proto`. This is the shared single source of truth for both your frontend and backend.

```protobuf
syntax = "proto3";

package UserService;

// Define the service and its methods
service User {
  rpc GetUserProfile (UserRequest) returns (UserResponse);
}

// Define the request payload
message UserRequest {
  string user_id = 1;
}

// Define the response payload
message UserResponse {
  string name = 1;
  string email = 2;
}

```

---

## Step 2: The Node.js RPC Server

First, install the official gRPC packages for Node:

```bash
npm install @grpc/grpc-js @grpc/proto-loader

```

Create `server.js`:

```javascript
const grpc = require("@grpc/grpc-js");
const protoLoader = require("@grpc/proto-loader");

// Load the proto file
const packageDefinition = protoLoader.loadSync("user.proto", {
  keepCase: true,
  longs: String,
  enums: String,
  defaults: true,
  oneofs: true,
});
const userProto = grpc.loadPackageDefinition(packageDefinition).UserService;

// Implement the RPC function defined in the proto file
function getUserProfile(call, callback) {
  const userId = call.request.user_id;

  // Mock Database Lookup
  const mockDatabase = {
    123: { name: "Alice Smith", email: "alice@example.com" },
    456: { name: "Bob Jones", email: "bob@example.com" },
  };

  const user = mockDatabase[userId];

  if (user) {
    callback(null, user); // Success: return data
  } else {
    callback({
      code: grpc.status.NOT_FOUND,
      details: "User not found",
    });
  }
}

// Start the gRPC Server
function main() {
  const server = new grpc.Server();
  server.addService(userProto.User.service, { GetUserProfile: getUserProfile });

  // Internal gRPC usually runs on a specific port like 50051
  server.bindAsync("0.0.0.0:50051", grpc.ServerCredentials.createInsecure(), () => {
    console.log("Node.js RPC Server running on port 50051");
  });
}

main();
```

## Step 3: Configure Envoy Proxy (The Browser Bridge)

Browsers can't talk directly to port `50051`. You need an Envoy proxy to sit in the middle, listening on port `8080`,
translating browser requests, and forwarding them to Node.js.

Create an `envoy.yaml` config file:

```yaml
static_resources:
  listeners:
    - name: listener_0
      address:
        socket_address: { address: 0.0.0.0, port_value: 8080 }
      filter_chains:
        - filters:
            - name: envoy.filters.network.http_connection_manager
              typed_config:
                "@type": type.googleapis.com/envoy.extensions.filters.network.http_connection_manager.v3.HttpConnectionManager
                codec_type: auto
                stat_prefix: ingress_http
                route_config:
                  name: local_route
                  virtual_hosts:
                    - name: local_service
                      domains: ["*"]
                      routes:
                        - match: { prefix: "/" }
                          route: { cluster: node_rpc_service, timeout: 0s }
                      cors:
                        allow_origin_string_match:
                          - safe_regex: { google_re2: {}, regex: ".*" }
                        allow_methods: GET, PUT, POST, OPTIONS
                        allow_headers: keep-alive,user-agent,cache-control,content-type,content-transfer-encoding,x-accept-content-transfer-encoding,x-accept-response-streaming,x-user-agent,x-grpc-web,grpc-timeout
                        max_age: "1728000"
                        expose_headers: grpc-status,grpc-message
                http_filters:
                  - name: envoy.filters.http.grpc_web
                    typed_config:
                      "@type": type.googleapis.com/envoy.extensions.filters.http.grpc_web.v3.GrpcWeb
                  - name: envoy.filters.http.cors
                    typed_config:
                      "@type": type.googleapis.com/envoy.extensions.filters.http.cors.v3.Cors
                  - name: envoy.filters.http.router
                    typed_config:
                      "@type": type.googleapis.com/envoy.extensions.filters.http.router.v3.Router
  clusters:
    - name: node_rpc_service
      connect_timeout: 0.25s
      type: logical_dns
      http2_protocol_options: {}
      load_assignment:
        cluster_name: node_rpc_service
        endpoints:
          - lb_endpoints:
              - endpoint:
                  address:
                    socket_address: { address: localhost, port_value: 50051 }
```

_You can spin up this proxy instantly via Docker using:_
`docker run -d -v $(pwd)/envoy.yaml:/etc/envoy/envoy.yaml -p 8080:8080 envoyproxy/envoy:v1.22.0`

## Step 4: Generate Client Stubs for React

To use the `.proto` contract in React, you need to compile it into JavaScript client files. Install the compiler (`protoc`)
on your machine, then run:

```bash
npm install grpc-web google-protobuf

protoc -I=. user.proto \
  --js_out=import_style=commonjs:. \
  --grpc-web_out=import_style=commonjs,mode=grpcwebtext:.

```

This will generate two files: `user_pb.js` (data structures) and `user_grpc_web_pb.js` (the network client methods). Move
these into your React `src/` directory.

---

## Step 5: Consume the RPC in React

Now, you can import your generated code into a React component and invoke the backend service just like a native JavaScript
function.

```jsx
import React, { useState } from "react";
// Import the generated stubs
import { UserClient } from "./user_grpc_web_pb";
import { UserRequest } from "./user_pb";

// Connect to the Envoy Proxy port, NOT the Node server port
const client = new UserClient("http://localhost:8080", null, null);

function App() {
  const [userId, setUserId] = useState("");
  const [profile, setProfile] = useState(null);
  const [error, setError] = useState("");

  const fetchProfile = () => {
    setError("");

    // 1. Create the RPC request object
    const request = new UserRequest();
    request.setUserId(userId);

    // 2. Call the remote procedure directly
    client.getUserProfile(request, {}, (err, response) => {
      if (err) {
        setError(`RPC Error: ${err.message}`);
        setProfile(null);
      } else {
        // 3. Extract data using generated getters
        setProfile({
          name: response.getName(),
          email: response.getEmail(),
        });
      }
    });
  };

  return (
    <div style={{ padding: "20px" }}>
      <h2>gRPC-Web React Demo</h2>
      <input
        type="text"
        placeholder="Enter User ID (123 or 456)"
        value={userId}
        onChange={(e) => setUserId(e.target.value)}
      />
      <button onClick={fetchProfile}>Call Remote Procedure</button>

      {error && <p style={{ color: "red" }}>{error}</p>}

      {profile && (
        <div style={{ marginTop: "20px", border: "1px solid #ccc", padding: "10px" }}>
          <h3>User Profile Data:</h3>
          <p>
            <strong>Name:</strong> {profile.name}
          </p>
          <p>
            <strong>Email:</strong> {profile.email}
          </p>
        </div>
      )}
    </div>
  );
}

export default App;
```

### Alternative Option: tRPC (The Node/React Native Alternative)

If you are building an application where **both** your frontend and backend are written in TypeScript, you might want to look
into **tRPC** instead of gRPC.

tRPC doesn't require schema compilation (`.proto` files) or proxies like Envoy. It shares the TypeScript types directly from
your Node backend to your React frontend, giving you full RPC type safety automatically at compilation time without the
infrastructural overhead of gRPC.
