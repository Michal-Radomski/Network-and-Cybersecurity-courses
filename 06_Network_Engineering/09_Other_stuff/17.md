A **Remote Procedure Call (RPC)** is a software communication protocol that allows one program to request a service from
another program located on a different computer in a network, without having to understand the underlying network details.

In simple terms, RPC makes a network call look and feel exactly like a **local function call** inside your own code.

---

## The Core Concept: Local vs. Remote

To understand RPC, compare it to how normal programming works:

- **Local Procedure Call:** If a program on your laptop needs to calculate tax, it calls a function: `calculateTax(amount)`.
  The computer pauses the main program, runs the function in memory, and immediately hands back the result. It is instant and
  reliable.
- **Remote Procedure Call (RPC):** Imagine your laptop doesn't have the processing power or data to calculate the tax. The
  code still writes `calculateTax(amount)`, but behind the scenes, the RPC framework wraps up that request, sends it over the
  internet to a powerful server, waits for the server to calculate it, and brings the answer back.

To the developer writing the code, the function looks completely local.

---

## How RPC Works (Step-by-Step)

Because computers don't naturally share memory over a network, an RPC framework relies on architectural components called
**Stubs** to trick the program into thinking the call is local.

Here is the exact sequence of events when a client calls a remote function:

1. **The Client Call:** The client application calls a local procedure (e.g., `getUser(id)`). This actually invokes a piece
   of auto-generated code called the **Client Stub**.
2. **Marshalling (Serialization):** The Client Stub takes the parameters (like the integer `id`) and packages them into a
   standardized, compressed binary format that can be sent over a network.
3. **Transmission:** The client’s operating system sends this packaged message over the network (via protocols like TCP, UDP,
   or HTTP/2) to the target server.
4. **Delivery & Unmarshalling:** The server's OS receives the packets and hands them to the **Server Stub** (sometimes called
   a skeleton). The Server Stub unpacks the binary data back into variables the server language understands.
5. **Execution:** The Server Stub calls the _actual_ function residing on the server. The server does the work and returns
   the result to the Server Stub.
6. **The Return Journey:** The process reverses. The Server Stub marshals the return data, ships it back across the network,
   the Client Stub unpacks it, and hands it back to the client application.

## Why Use RPC?

RPC is the backbone of modern **distributed systems** and **microservice architectures** (used extensively by companies like
Google, Netflix, and Uber).

- **Language Agnostic:** A client written in Python can call an RPC function on a server written in C++ or Go. The stubs
  handle the translation.
- **Simplicity for Developers:** Programmers don't have to write tedious networking code (dealing with sockets, IP addresses,
  packet loss, or raw byte streams) every time they want services to talk to each other.
- **High Performance:** Modern RPC frameworks are optimized for speed, using compact binary formats instead of bulky text
  formats like JSON or XML.

---

## Popular RPC Implementations

While RPC is a conceptual design, several actual frameworks implement it:

- **gRPC (Google RPC):** Created by Google, this is currently the industry standard for microservices. It uses **HTTP/2** for
  transport and **Protocol Buffers (Protobuf)** to compress messages into tiny binary payloads for ultra-fast performance.
- **Apache Thrift:** Developed by Facebook, similar to gRPC, allowing scalable cross-language services development.
- **XML-RPC / JSON-RPC:** Older, simpler versions of RPC that use text-based XML or JSON over standard HTTP rather than
  optimized binary formats.

---

## Challenges of RPC

While RPC makes networking look easy, it introduces hidden complexities that developers must plan for:

- **The Fallibility of Networks:** A local function call virtually never fails. A remote call can fail if a backhoe cuts a
  fiber-optic cable, a router overheats, or the server crashes mid-calculation.
- **Latency:** Local calls take nanoseconds. Remote calls take milliseconds. If a developer forgets a function is remote and
  calls it thousands of times in a loop, it will severely slow down the application.
