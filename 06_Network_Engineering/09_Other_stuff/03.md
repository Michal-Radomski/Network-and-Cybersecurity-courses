**SCRAM** (Salted Challenge Response Authentication Mechanism) is a modern security standard used by databases like
**PostgreSQL**, **MongoDB**, and **CockroachDB** to verify a user's identity without ever sending their actual password over
the network.

It is widely considered the successor to the older **MD5** hashing method because it provides much stronger protection
against "man-in-the-middle" and "replay" attacks.

---

### How SCRAM Works (The Handshake)

Unlike simple password matching, SCRAM uses a multi-step exchange. Think of it as a series of cryptographic riddles that both
the client and server must solve to prove they know the password.

1. **The Client's Intent:** The client sends the username to the server.
2. **The Server's Challenge:** The server looks up the user and sends back a **Salt** and an **Iteration Count**.
3. **Proof Generation:** The client hashes their password using that salt and the iterations. It then creates a "Client
   Proof" (a cryptographic signature) and sends it to the server.
4. **Verification:** The server performs the same math. If the signatures match, the server knows the client has the right
   password.
5. **Mutual Authentication:** The server sends its own proof back to the client. This proves to the client that the server is
   legitimate and not a fake "phishing" database.

---

### Key Components

To understand SCRAM, it helps to look at the math variables involved (usually using **SHA-256**):

- **Salt:** A random string added to the password before hashing. This ensures that two users with the same password have
  different hashes, preventing "Rainbow Table" attacks.
- **Iterations:** The number of times the hashing process is repeated (e.g., 4096 times). This makes "brute-forcing" the
  password incredibly slow and expensive for hackers.
- **StoredKey & ServerKey:** Instead of storing the password, the database stores these two derived keys. Even if the
  database is leaked, the attacker cannot easily reverse-engineer the original password.

$$SaltedPassword = PBKDF2(Password, Salt, Iterations)$$

$$ClientKey = HMAC(SaltedPassword, "Client Key")$$

$$StoredKey = SHA256(ClientKey)$$

---

### Why SCRAM is Better Than MD5

| Feature         | MD5 (Old)                        | SCRAM (Modern)                                 |
| --------------- | -------------------------------- | ---------------------------------------------- |
| **Security**    | Vulnerable to collision attacks. | Robust; uses modern SHA-256.                   |
| **Salted?**     | Often no (or weak).              | Always unique per user.                        |
| **Mutual Auth** | Client trusts Server blindly.    | Both prove their identity to each other.       |
| **Performance** | Very fast (good for hackers).    | Intentionally CPU-intensive (bad for hackers). |

---

### Implementation Tip

If you are managing a database like **PostgreSQL**, you often have to explicitly enable this in your configuration:

> `password_encryption = 'scram-sha-256'`

Once enabled, any _new_ passwords set by users will be stored using this more secure method, though older MD5 passwords
usually remain valid until the user changes them.
