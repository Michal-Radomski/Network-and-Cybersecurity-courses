When we say that TCP (Transmission Control Protocol) is a **stream-oriented protocol**, it means that TCP treats the data it
transmits as a continuous, unbroken stream of bytes, rather than as a collection of distinct blocks or individual messages.

To understand what this means in practice, it helps to look at how data is sent, how it is received, and how it compares to
the alternative.

---

## 1. The "Garden Hose" Analogy

Think of a **message-oriented** protocol (like UDP) as a mail carrier delivering distinct letters. If you send three letters,
the recipient receives exactly three distinct envelopes.

TCP, being **stream-oriented**, is more like a garden hose. If you pour separate cups of water (your data) into one end of
the hose, the water mixes together inside the pipe. The person on the other end doesn't get "cups" of water; they just get a
continuous flow of water. They can catch that water in a small glass, a giant bucket, or several random sips—the boundaries
of the original cups are completely gone.

---

## 2. How it Works in Practice

Because TCP is stream-oriented, it decouples (separates) the application's write operations from the network's transmission
operations.

- **At the Sender’s End:** Your application might call a function to send three separate messages: `"Hello"`, `"World"`, and
  `"! "`. TCP doesn't immediately send three separate packets. Instead, it shoves those bytes into a send buffer. Depending
  on network conditions, TCP might bundle them all together into a single network packet: `[HelloWorld!]`.
- **At the Receiver’s End:** The receiving application reads data from its own buffer. It has no native way of knowing
  whether the sender transmitted this data as one large message or three small ones. The receiver might read all 11 bytes at
  once, or it might read 2 bytes now (`"He"`) and the remaining 9 bytes later (`"lloWorld!"`).

---

## 3. The Consequences: Framing is Up to You

Because TCP does not preserve **message boundaries**, the software developer is entirely responsible for slicing the byte
stream back into meaningful pieces. This is called **framing**.

If you don't implement framing, a phenomenon known as **packet sticking** (where multiple messages merge) or **packet
fragmentation** (where a single message is split across multiple reads) will break your application.

Developers usually solve this in the application layer using one of two methods:

- **Fixed-Length Framing:** Every message must be exactly 100 bytes. The receiver just reads 100 bytes at a time.
- **Delimiter-Based Framing:** Ending every message with a specific character, like a newline (`\n`). This is how HTTP/1.1
  and SMTP (email) work.
- **Length-Prefix Framing:** Sending an integer first that states exactly how many bytes of data follow (e.g., _"The next
  message is 14 bytes long"_). This is how protocols like TLS or custom RPC protocols handle it.

---

## Comparison: Stream vs. Message Oriented

| Feature                | TCP (Stream-Oriented)                       | UDP / Homa (Message-Oriented)          |
| ---------------------- | ------------------------------------------- | -------------------------------------- |
| **Data Unit**          | Continuous bytes ($B_1, B_2, B_3...$)       | Distinct packets / messages            |
| **Message Boundaries** | **No.** Disappear into a buffer.            | **Yes.** Preserved exactly as sent.    |
| **Application Duty**   | Must manually parse and "slice" data.       | Can read data packet-by-packet.        |
| **Network Efficiency** | High (TCP packs data tightly into packets). | Variable (can result in wasted space). |
