Running out of TCP source ports—often referred to as **ephemeral port exhaustion**—is a common and frustrating bottleneck in
high-concurrency systems. It occurs when a client machine opens so many outbound connections to a specific destination that
it completely uses up its allocated pool of local ports.

Here is a breakdown of why this happens, how to diagnose it, and the best strategies to fix it.

---

## Why Ephemeral Port Exhaustion Happens

When a client initiates a TCP connection to a server, the operating system maps a unique **4-tuple** to that connection:

$$\text{Source IP} \quad \text{Source Port} \quad \text{Destination IP} \quad \text{Destination Port}$$

The Destination Port (e.g., `80` or `443` for web traffic) and IPs are usually fixed. Therefore, to make every connection
unique, the OS must assign a unique **Source Port** from its ephemeral pool.

By default, the operating system limits this pool (typically around 28,000 to 30,000 ports on modern Linux systems).

### The Killer: The `TIME_WAIT` State

The real culprit is usually not just active connections, but closed ones. When an application closes a TCP connection, the OS
keeps that source port locked in a `TIME_WAIT` state for a period of time (typically **60 to 120 seconds** by default). This
ensures that any delayed network packets belonging to that specific connection are dropped rather than corrupting a new
connection.

If your application spins up 30,000 short-lived requests in less than two minutes, you will completely run out of source
ports, causing new connections to fail immediately with errors like `EADDRINUSE` or `Cannot assign requested address`.

---

## How to Diagnose It

If you suspect you are running out of ports, you can verify it using standard terminal tools.

### 1. Check current TCP states

Run the following command on the client machine to count how many sockets are stuck in `TIME_WAIT`:

```bash
netstat -an | grep TIME_WAIT | wc -l
# Or using the faster 'ss' utility:
ss -s

```

If the number is close to your system's maximum ephemeral port range, you have an exhaustion issue.

### 2. Check your current ephemeral range

To see exactly how many ports your system allows, run:

```bash
cat /proc/sys/net/ipv4/ip_local_port_range
# Typical output: 32768 60999 (This means 28,231 available ports)

```

---

## How to Fix It

Depending on whether you can modify your application code or only have access to the underlying infrastructure, there are
several ways to resolve this.

### Strategy 1: Enable Connection Pooling (Best Practice)

The absolute best way to fix port exhaustion is to prevent it at the application layer. Instead of opening and closing a new
TCP connection for every single request, use **HTTP Keep-Alive** or a persistent connection pool.

- **Why it works:** It allows thousands of requests to reuse a small, fixed set of TCP connections sequentially, bypassing
  the `TIME_WAIT` state entirely.

### Strategy 2: Tweak Linux Kernel Parameters (Infrastructure Fix)

If you cannot change the application code, you can update the OS network stack configurations in `/etc/sysctl.conf`:

- **Expand the port range:** Increase the available pool to the absolute theoretical maximum (around 64,500 ports).

```ini
net.ipv4.ip_local_port_range = 1024 65535

```

- **Enable TCP Tw Reuse:** This allows the Linux kernel to safely recycle a socket in the `TIME_WAIT` state for a new
  outbound connection if it deems it safe from a protocol standpoint.

```ini
net.ipv4.tcp_tw_reuse = 1

```

_(After modifying `/etc/sysctl.conf`, run `sysctl -p` to apply changes)._

### Strategy 3: Add More Local IP Addresses

Remember that uniqueness relies on the 4-tuple. If you have 60,000 ports available per IP address, adding a second network
interface or secondary private IP address (`Source IP`) to your client machine instantly doubles your available source ports
to 120,000. You just have to configure your application or load balancer to bind outbound traffic across the multiple local
IPs.

### Strategy 4: Use a Reverse Proxy / Load Balancer

If your client application is talking to a single backend API or database, place a reverse proxy (like Nginx or HAProxy)
between them. The proxy can handle connection pooling efficiently, shielding your core application from managing raw OS
sockets.
