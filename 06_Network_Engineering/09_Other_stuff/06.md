The Linux Kernel manages TCP connections through a sophisticated networking stack that handles everything from the initial
"handshake" to the complex math of congestion control. It acts as the intermediary between your application code and the
physical network hardware.

---

### 1. The Handshake and Queuing

When a connection starts, the Kernel manages two specific queues to handle the incoming load. This is a critical area for
performance tuning.

- **SYN Queue (Incomplete Connection):** When a `SYN` packet arrives, the Kernel stores the connection details here and sends
  a `SYN-ACK`. The connection is "half-open."
- **Accept Queue (Complete Connection):** Once the `ACK` returns from the client, the connection moves here. It sits in this
  queue until your application calls the `accept()` system call.

---

### 2. The Transmission Control Block (TCB)

For every single active connection, the Kernel maintains a **TCB**. This is a data structure that tracks the entire state of
that specific conversation, including:

- **Local and Remote IP/Ports** (The 4-tuple).
- **Sequence Numbers:** To ensure data arrives in the correct order.
- **Window Sizes:** For flow control (preventing the sender from overwhelming the receiver).
- **Timers:** For retransmissions if a packet is lost.

---

### 3. Buffering: Sending and Receiving

The Kernel doesn't send data the exact microsecond your app requests it. Instead, it uses buffers to optimize throughput.

- **Write Buffer (`sk_write_queue`):** When your app calls `send()`, the data is copied from "User Space" to "Kernel Space"
  into this buffer. The Kernel then decides when to wrap that data in a TCP header and push it to the network card (NIC).
- **Read Buffer (`sk_receive_queue`):** Incoming packets are stripped of their headers and placed here. When your app calls
  `read()`, the Kernel copies the data from this buffer into your application's memory.

---

### 4. The Congestion Control Engine

This is arguably the most "intelligent" part of the Kernel. It constantly monitors how fast packets are being acknowledged.
If it detects dropped packets (packet loss) or increasing delays (latency), it triggers an algorithm to slow down the flow to
prevent a network collapse.

Common algorithms include:

- **Cubic:** The current Linux default; optimized for high-speed networks with high latency.
- **BBR (Bottleneck Bandwidth and RTT):** A newer Google-developed algorithm that focuses on actual throughput rather than
  just packet loss.

---

### 5. Termination and TIME_WAIT

When a connection closes, the Kernel doesn't just delete it instantly. It enters the **TIME_WAIT** state (usually for 60
seconds).

> **Why?** To ensure that any "stray" packets still wandering around the internet don't accidentally get delivered to a _new_
> connection that happens to start on the same port immediately after. The Kernel essentially "quarantines" that port
> combination to keep data integrity high.

---

### Summary of System Calls

Your code interacts with the Kernel's TCP management via these primary system calls:

| System Call     | Kernel Action                                          |
| --------------- | ------------------------------------------------------ |
| `socket()`      | Allocate memory for the TCB and file descriptor.       |
| `bind()`        | Associate the socket with a specific local port.       |
| `listen()`      | Prepare the SYN and Accept queues.                     |
| `connect()`     | Initiate the 3-way handshake (Client side).            |
| `accept()`      | Pull a finished connection off the Accept queue.       |
| `send()/recv()` | Move data between user-space and kernel-space buffers. |
