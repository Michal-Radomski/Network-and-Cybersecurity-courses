A **Host header** is a mandatory field in an **HTTP/1.1 (and newer)** request that tells the server which specific domain
name the client is trying to reach.

Because many different websites can live on a single server with a single IP address, the Host header acts as the "label"
that ensures your request lands on the correct site.

---

### Why It Exists: Virtual Hosting

In the early days of the web, one IP address usually equaled one website. As the internet grew, this became inefficient. Web
servers started using **Virtual Hosting** to host multiple domains (e.g., `site-a.com` and `site-b.com`) on one machine.

When a request arrives at the server's IP address, the server looks at the Host header to decide which folder or application
should handle the request.

---

### How It Works in a Request

When you type `www.example.com` into your browser, the browser performs a DNS lookup to find the IP address. Once connected,
it sends a plain-text request that looks like this:

```http
GET /index.html HTTP/1.1
Host: www.example.com
User-Agent: Mozilla/5.0...
Accept: text/html...

```

- **The Problem:** The server knows you want `/index.html`, but it doesn't know _which_ site's index page you want based on
  the IP alone.
- **The Solution:** The `Host: www.example.com` line tells the server exactly which virtual host to wake up.

---

### Common Use Cases

- **Shared Hosting:** Thousands of small websites often share a few dozen IP addresses. The Host header is the only way the
  server knows which customer's site to display.
- **Reverse Proxies & Load Balancers:** Tools like **Nginx** or **HAProxy** use the Host header to decide which internal
  "backend" server should receive the traffic.
- **CDNs (Content Delivery Networks):** When you request a file from a provider like Cloudflare, they use the Host header to
  identify which client’s cached data to serve you.

---

### Security Risks: Host Header Injection

If a web application isn't configured correctly, it might trust the Host header too much. An attacker can manually change the
Host header in a request to something else.

> **Example:** If a website uses the Host header to generate "Password Reset" links in emails, an attacker could inject their
> own domain. When the victim clicks the link, their secret reset token is sent to the attacker's server instead of the real
> one.

---

### Host Header vs. SNI

People often confuse these two because they both deal with domain names, but they happen at different layers of the
connection:

| Feature     | Host Header                                     | SNI (Server Name Indication)                        |
| ----------- | ----------------------------------------------- | --------------------------------------------------- |
| **Layer**   | **Application Layer** (HTTP)                    | **Transport Layer** (TLS/SSL)                       |
| **Timing**  | Sent _after_ the connection is secure.          | Sent _before_ the connection is secure.             |
| **Purpose** | Tells the web server which **content** to show. | Tells the server which **SSL certificate** to show. |

---

---

The short answer is: **The browser handles the `Host` header automatically.** In 99% of web development scenarios, you should
**not** attempt to set or modify this header manually in your React code. The browser and the underlying TCP/IP stack take
care of it to ensure the request is routed correctly.

---

### 1. Who handles it?

- **The Browser:** When you use `axios.get('https://api.example.com/data')`, the browser extracts `api.example.com` from the
  URL and automatically inserts it into the HTTP `Host` header.
- **Security (The "Why"):** Browsers block developers from changing the `Host` header via JavaScript (Axios/Fetch). If a
  developer could change it, they could perform **Host Header Injection**, potentially tricking a server into sending
  sensitive data (like password reset tokens) to a malicious domain.

---

### 2. How to see the Host Header

Since the `Host` header is part of the raw HTTP request, you can see it using the **Browser DevTools**.

1. Open your React app in Chrome or Firefox.
2. Right-click anywhere and select **Inspect**, then go to the **Network** tab.
3. Refresh the page or trigger your Axios call.
4. Click on the specific request name (e.g., `data` or `users`).
5. Look for the **Headers** sub-tab.
6. Scroll down to **Request Headers**. You will see `Host: yourdomain.com` listed there.

---

### 3. When _should_ a developer handle it?

The only time you, as a developer, need to worry about the `Host` header is when configuring the **Server** (Express.js) or a
**Proxy**, not the React client.

#### A. In Express.js (Behind a Proxy)

If your Express server is sitting behind a load balancer (like Nginx, AWS ALB, or Heroku), the "real" host might get lost.
You need to tell Express to "trust" the proxy so it can read the original host correctly.

```javascript
const express = require("express");
const app = express();

// Tell Express to trust headers like X-Forwarded-Host
app.set("trust proxy", true);

app.get("/", (req, res) => {
  // Now req.hostname will show the actual domain the user typed
  console.log(req.hostname);
  res.send(`You are visiting ${req.hostname}`);
});
```

#### B. In Nginx (As a Reverse Proxy)

If you are using Nginx to point to your Express app, you must explicitly pass the Host header along, or Express will only see
`localhost`.

```nginx
location / {
    proxy_pass http://localhost:5000;
    proxy_set_header Host $host; # This passes the real domain to Express
    proxy_set_header X-Real-IP $remote_addr;
}

```

### Summary Table

| Environment                    | Responsibility | Can you change it?                     |
| ------------------------------ | -------------- | -------------------------------------- |
| **React/Axios**                | **Browser**    | **No** (Blocked for security)          |
| **Node.js (Server-to-Server)** | **Developer**  | **Yes** (via Axios headers)            |
| **Express.js**                 | **Developer**  | **Yes** (Use `app.set('trust proxy')`) |
| **Nginx/Load Balancer**        | **Developer**  | **Yes** (Required for routing)         |

### Troubleshooting Tip

If you are seeing an **"Invalid Host Header"** error in your browser console while developing, it’s not an Axios issue. It’s
a safety feature of the **Webpack Dev Server** or **Vite**. It’s telling you: _"Hey, someone is trying to access this dev
environment from a domain I don't recognize; I'm blocking it for safety."_ To fix that, you update your `vite.config.ts` or
`webpack.config.js` rather than your Axios code.
