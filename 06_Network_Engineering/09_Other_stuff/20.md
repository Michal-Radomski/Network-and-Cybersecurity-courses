**TLS 0-RTT** (Zero Round-Trip Time), also known as **TLS Early Data**, is a feature introduced in **TLS 1.3** designed to
eliminate network latency when reconnecting to a server.

It allows a client to send application data (like an HTTP request) in its very first message to the server, entirely skipping
the traditional cryptographic handshake delay.

---

## The Latency Problem: 1-RTT vs. 0-RTT

To understand why 0-RTT is a big deal, we have to look at how normal connections work.

### Standard TLS 1.3 Handshake (1-RTT)

The first time a browser connects to a secured website, it requires **one full round-trip (1-RTT)** of network communication
to establish encryption keys before any actual data can be sent.

1. **Client $\rightarrow$ Server:** "Hello, here are my cryptographic settings." (Client Hello)
2. **Server $\rightarrow$ Client:** "Hello, here are my settings and certificate." (Server Hello)
3. **Client $\rightarrow$ Server:** _[Handshake complete]_ "Now, please send me the homepage." (HTTP Request)

If you are on a mobile network with a 100ms ping, that mandatory handshake causes a noticeable 100ms delay before the website
even _starts_ loading.

### TLS 0-RTT Handshake (0-RTT)

If a client has connected to the server recently, they can use 0-RTT to resume their session instantly.

1. **Client $\rightarrow$ Server:** "Hello, remember me? Here is my session ticket, **and by the way, here is the HTTP
   request for the homepage.**"
2. **Server $\rightarrow$ Client:** "I remember you. Here is the homepage data you asked for."

The client saves a complete round-trip. To the user, the website feels instantaneous.

---

## How Does 0-RTT Work Behind the Scenes?

0-RTT relies on a concept called **Session Resumption** using **Pre-Shared Keys (PSK)**:

1. **The First Visit:** During the very first (1-RTT) handshake, the server sends the client a encrypted **Session Ticket**
   (New Session Ticket message). This ticket contains a copy of the secret cryptographic keys wrapped in a way that only the
   server can decrypt.
2. **The Disconnect:** The user closes the tab or loses connection briefly.
3. **The Reconnection (0-RTT):** When reconnecting, the client pulls that Session Ticket from its cache. It uses the old keys
   to encrypt its "Early Data" (the HTTP request) and sends both the ticket and the data to the server at the exact same
   time.
4. **The Resolution:** The server decrypts the ticket, grabs the keys, decrypts the early data, and processes the request
   immediately.

---

## The Catch: The Replay Attack Vulnerability

While 0-RTT is incredibly fast, it introduces a massive security risk: **Replay Attacks**.

Because the client sends the encrypted data before the server has a chance to provide a fresh, random number (nonce) to
ensure uniqueness, an eavesdropping hacker can intercept that initial 0-RTT packet and record it. Even though the hacker
cannot _read_ the encrypted data, they can **replay** (send it again) to the server 100 times.

### The Threat Scenario

Imagine you send a 0-RTT request to your bank: `POST /transfer?amount=100&to=friend`. If a hacker intercepts that packet,
they can replay it over and over. The bank server, seeing valid encryption tokens, might execute the transfer multiple times,
draining your account.

---

## Mitigating the Risk: How to Safely Use 0-RTT

Because of the replay risk, browsers and servers enforce strict rules on when 0-RTT can be used:

- **Safe Methods Only (Idempotent Requests):** 0-RTT should **only** be used for requests that do not change state on the
  server. Browsers will only send `GET` requests (like loading an image or a webpage) via 0-RTT. They will _never_ use 0-RTT
  for `POST`, `PUT`, or `DELETE` requests. If a hacker replays a `GET` request for a homepage, the server just sends the
  homepage again, which does no harm.
- **Anti-Replay Windows:** Servers keep a cache of recently received 0-RTT identifiers or use unique timestamps to reject
  duplicate packets that arrive outside a tiny time window.
- **Protocol Fallback:** If a server suspects a replay attack or has cleared its session cache, it will ignore the early data
  and gracefully fall back to a standard 1-RTT handshake.
