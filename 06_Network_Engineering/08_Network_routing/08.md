Networking in Docker is the mechanism that allows isolated containers to communicate with each other, with the host operating
system, and with external networks (like the internet).

When you install Docker, it automatically creates three default networks for you. However, you can also create custom
networks depending on how isolated or connected you want your applications to be.

---

## The Default Network Drivers

Docker uses **network drivers** to manage how containers are wired up. Here are the core types of networks you will interact
with:

### 1. Bridge Network (The Default)

A bridge network is a private, internal network created by Docker on your host machine.

- **How it works:** Containers attached to this network get their own private IP addresses (usually in a range like
  `172.17.0.x`). They can talk to each other and access the outside internet (via Network Address Translation handled by the
  host). However, machines outside the host cannot access these containers unless you explicitly map/publish a port (e.g.,
  `-p 8080:80`).
- **Best for:** Standard application stacks running on a single host machine.

### 2. Host Network

This driver removes the network isolation between the container and the Docker host.

- **How it works:** The container does not get its own IP address allocated. Instead, it binds directly to the host’s network
  interfaces. For example, if a container runs a web server on port `80` using the `host` network, that web server is
  instantly available on the host machine's port `80`.
- **Best for:** Maximizing performance (as it removes the Docker network overhead) and situations where a container needs to
  handle a massive range of ports.

### 3. None Network

This driver completely disables networking for the container.

- **How it works:** The container is given its own isolated network stack, but it has no connection to the internet, the host
  network, or any other container. It only has a local loopback (`127.0.0.1`) interface.
- **Best for:** Running highly secure batch jobs, processing private data, or running computational tasks that require
  absolute network isolation.

### 4. Overlay Network

An overlay network creates a distributed network across **multiple physical Docker hosts**.

- **How it works:** It connects multiple Docker daemons together and allows swarm services to communicate with each other
  securely, even if they are running on completely different servers or cloud virtual machines.
- **Best for:** Multi-host deployments and Docker Swarm/Kubernetes orchestration environments.

## How Docker Networking Works Under the Hood

To understand how Docker achieves this isolation on Linux, it relies heavily on native kernel features that we've touched on
in previous conversations:

1. **Network Namespaces:** Docker creates a distinct, isolated network namespace for every container. This ensures that
   Container A cannot see or interfere with the network interfaces or routing tables of Container B.
2. **Virtual Ethernet Pairs (veth):** To connect the container's isolated world back to the host, Docker creates a virtual
   cable called a `veth` pair. One end of this virtual cable sits inside the container (usually named `eth0`), and the other
   end is plugged into the host’s virtual bridge (usually named `docker0`).
3. **IP Forwarding & iptables:** Docker heavily utilizes Linux `iptables` and requires `net.ipv4.ip_forward=1` to be enabled
   on the host. This allows the host machine to route traffic from the physical internet into the virtual `docker0` bridge,
   mapping incoming host ports directly to the container's internal ports.

## Best Practice: User-Defined Bridge Networks

While Docker provides a default bridge network named `bridge`, production deployments should almost always use **User-Defined
Bridge Networks**.

You can create one using the CLI:

```bash
docker network create my-app-network

```

### Why User-Defined Networks are Superior:

- **Automatic Service Discovery (DNS):** On the default `bridge` network, containers can only talk to each other if they know
  each other's temporary internal IP addresses. On a _user-defined_ network, Docker provides a built-in DNS server.
  Containers can talk to each other using their **container names**. For example, a web app container can connect to a
  database container simply by querying the hostname `db-server`.
- **Better Isolation:** If you have an application stack for a blog and another for a payment system on the same machine,
  putting them on separate user-defined networks ensures they are completely blind to one another, preventing a breach in one
  from affecting the other.

## Common Docker Network Commands

Here is how you manage these networks from the Linux command line:

- **List all networks:** `docker network ls`
- **Inspect a network (see attached containers and IPs):** `docker network inspect <network_name>`
- **Run a container on a specific network:** `docker run -d --name web-app --network my-app-network nginx`
- **Connect an existing running container to a network:** `docker network connect my-app-network existing-container`
