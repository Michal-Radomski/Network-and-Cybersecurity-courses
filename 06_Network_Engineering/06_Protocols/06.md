To understand encryption in HTTPS, you have to look at it as a "team effort" between two different types of encryption. It
uses **Asymmetric Encryption** to introduce the parties and **Symmetric Encryption** to carry out the actual conversation.

Think of it like this: Asymmetric encryption is the secure high-tech vault used to exchange a simple physical key, and
Symmetric encryption is using that physical key to lock and unlock a box quickly.

---

### 1. The Two-Stage Process

#### Stage A: The Handshake (Asymmetric Encryption)

When you first connect to a website, your browser and the server don't share a secret key yet. They use **Asymmetric
Encryption** (also known as Public Key Cryptography) to establish trust.

- **Public Key:** Shared with everyone (contained in the SSL/TLS Certificate). Anyone can use this to encrypt data, but they
  cannot use it to decrypt.
- **Private Key:** Kept secret by the server. It is the only key that can decrypt data encrypted by the Public Key.

> **The Goal:** The browser uses the server's Public Key to encrypt a "pre-master secret" and sends it over. Only the server,
> with its Private Key, can read it. Now, both parties have a shared secret that no one else knows.

#### Stage B: The Data Transfer (Symmetric Encryption)

Once the "shared secret" is established, the heavy lifting begins. Asymmetric encryption is mathematically "expensive"
(slow). To keep your web browsing fast, the connection switches to **Symmetric Encryption**.

- **Session Key:** Both the browser and the server derive the same "session key" from that shared secret.
- **Speed:** This single key is used to both encrypt and decrypt the data. It is incredibly fast and efficient for moving
  large amounts of data (like streaming video or loading images).

---

### 2. Ensuring Integrity (Hashing)

Encryption hides the data, but **Hashing** ensures no one tampered with it in transit. HTTPS uses a Message Authentication
Code (MAC).

Every piece of data sent is bundled with a "hash" (a unique digital fingerprint). If a hacker tries to change even one bit of
the data, the fingerprint won't match when it arrives, and the browser will drop the connection as "insecure."

---

### 3. Summary of the Flow

| Step                | Action                                        | Type of Tech                        |
| ------------------- | --------------------------------------------- | ----------------------------------- |
| **1. Identity**     | Server sends its Certificate/Public Key.      | **Digital Certificates**            |
| **2. Key Exchange** | Browser and server agree on a secret key.     | **Asymmetric (RSA/Diffie-Hellman)** |
| **3. Privacy**      | Actual data is encrypted with the secret key. | **Symmetric (AES)**                 |
| **4. Integrity**    | Data is checked for tampering.                | **Hashing (SHA-256)**               |

---

### Why not just use Asymmetric for everything?

If your computer used Asymmetric encryption for every packet of a YouTube video, your CPU would likely melt (or at least get
very hot), and the video would buffer indefinitely. By using the "Handshake" to trade a Symmetric key, HTTPS gets the
**unbreakable security** of the vault with the **high speed** of the session key.
