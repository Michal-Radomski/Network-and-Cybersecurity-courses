**VRRP (Virtual Router Redundancy Protocol)** is a "buddy system" for routers. Its job is to ensure that if your main router
dies, a backup router automatically takes over so quickly that the users on the network don't even notice.

In a normal setup, if your "Default Gateway" (the router) fails, you lose internet access. VRRP prevents this by creating a
**Virtual Router**.

---

## 1. How VRRP Works

Instead of pointing your computers to the physical IP address of a single router, you point them to a **Virtual IP (VIP)**
address shared by a group of routers.

- **The Master (Active) Router:** This is the router currently doing all the work. It "owns" the Virtual IP and handles all
  the traffic.
- **The Backup Routers:** These routers are standing by. They listen for a "heartbeat" from the Master.
- **The Virtual MAC:** VRRP even creates a fake MAC address (usually starting with `00-00-5E-00-01-XX`). This is important
  because it means the computers on the network don't have to update their ARP tables when a failover happens.

---

## 2. The "Heartbeat" and Failover

The Master router sends out a "VRRP Advertisement" packet every second.

1. **Everything is fine:** The Backup routers hear the advertisement and stay quiet.
2. **The Master Fails:** If the Backup routers don't hear an advertisement for about 3 seconds, they assume the Master is
   dead.
3. **The Election:** The Backup router with the highest **Priority** (a number from 1–255) instantly promotes itself to
   Master and starts answering for the Virtual IP.

---

## 3. Key Benefits

- **High Availability:** Your network stays up 24/7, even during hardware failures or firmware updates.
- **No Client Configuration:** You don't have to go to every computer and change their settings when a router breaks; the
  "Virtual Gateway" stays the same.
- **Load Sharing:** While VRRP is usually for redundancy, you can set up multiple VRRP groups so that Router A handles half
  the traffic and Router B handles the other half, but they still act as backups for each other.

---

## 4. VRRP vs. HSRP

If you’ve heard of **HSRP (Hot Standby Router Protocol)**, it is almost exactly the same thing.

- **HSRP** is a proprietary protocol created by **Cisco**.
- **VRRP** is an **open standard** that works across different brands (like Juniper, HP, and Mikrotik).

The best way to implement VRRP on Linux is using a tool called **Keepalived**. It is the industry-standard software for
handling VRRP and health checking on Linux systems.

To test this properly, you ideally need two Linux machines (or VMs) on the same network. Let’s call them **Node A** and
**Node B**.

---

## 1. The Scenario

- **Virtual IP (VIP):** `192.168.1.100` (This is the IP users will use).
- **Node A (Master):** `192.168.1.10`
- **Node B (Backup):** `192.168.1.11`

---

## 2. Installation

On both nodes, install the Keepalived package:

```bash
sudo apt update && sudo apt install keepalived -y

```

---

## 3. Configuration (The Master Node)

Edit the configuration file on **Node A**: `sudo nano /etc/keepalived/keepalived.conf`

Paste the following (adjusting `eth0` to your actual network interface name):

```conf
vrrp_instance VI_1 {
    state MASTER
    interface eth0           # Change to your interface (e.g., ens33, wlan0)
    virtual_router_id 51     # Must be the same on both nodes
    priority 255             # Higher number wins the election
    advert_int 1

    authentication {
        auth_type PASS
        auth_pass 1234       # Simple password for the "handshake"
    }

    virtual_ipaddress {
        192.168.1.100        # The shared Virtual IP
    }
}

```

---

## 4. Configuration (The Backup Node)

Edit the configuration file on **Node B**: `sudo nano /etc/keepalived/keepalived.conf`

```conf
vrrp_instance VI_1 {
    state BACKUP
    interface eth0
    virtual_router_id 51
    priority 100             # Lower than the Master
    advert_int 1

    authentication {
        auth_type PASS
        auth_pass 1234
    }

    virtual_ipaddress {
        192.168.1.100
    }
}

```

---

## 5. Activation and Testing

Start the service on both nodes:

```bash
sudo systemctl start keepalived

```

### How to verify:

1. **Check IP:** Run `ip addr show eth0` on **Node A**. You should see the `192.168.1.100` address attached to it.
2. **Check Node B:** Run the same command on **Node B**. You should **not** see the Virtual IP (it’s waiting).
3. **The "Failover" Test:** Shut down Node A or stop the service (`sudo systemctl stop keepalived`).
4. **The Result:** Check `ip addr` on Node B. Within a couple of seconds, Node B will "claim" the `192.168.1.100` address!

---

### A Common "Learning" Gotcha:

If you are running this in a Virtual Machine environment (like VMware or VirtualBox), you may need to enable **"Promiscuous
Mode"** on the virtual network adapter settings. VRRP uses a specific multicast address (`224.0.0.18`), and some virtual
switches block it by default.
