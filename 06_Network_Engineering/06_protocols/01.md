Think of the **Domain Name System (DNS)** as the "phonebook of the internet." While humans prefer catchy names like
`google.com`, computers are strictly "numbers people" and communicate via **IP addresses** (like `142.250.190.46`).

DNS is the protocol that translates those human-friendly hostnames into machine-readable IP addresses.

---

## How DNS Works: The Lookup Hierarchy

The DNS process isn't a single "call." It’s a series of handoffs across a distributed hierarchy of servers to find the right
address.

1. **The Recursive Resolver:** This is usually provided by your ISP. It’s the "middleman" that goes out and finds the answer
   for your computer.
2. **Root Nameservers:** The first stop. They don't know the IP, but they know where to find the **TLD (Top-Level Domain)**
   servers (like `.com`, `.org`, or `.edu`).
3. **TLD Nameservers:** These servers manage information for specific extensions. A `.com` TLD server knows which server
   holds the records for `example.com`.
4. **Authoritative Nameservers:** The final destination. This server holds the actual IP address for the specific domain
   you’re looking for.

---

## The Step-by-Step Request

When you type a URL into your browser, the following dance occurs:

1. **Browser Cache:** Your computer first checks if it already knows the IP from a previous visit.
2. **The Query:** If not, the **Recursive Resolver** asks the **Root Server**, "Where is `.com`?"
3. **The Referral:** Root says, "I don't know `google.com`, but here is the IP for the `.com` TLD server."
4. **The Specifics:** The Resolver asks the TLD, "Where is `google.com`?"
5. **The Answer:** The TLD points the Resolver to the **Authoritative Server**. The Authoritative server provides the IP
   (e.g., `142.250.190.46`).
6. **The Connection:** The Resolver gives that IP to your browser, and your browser loads the site.

---

## Common DNS Record Types

The "phonebook" doesn't just store one type of number. It uses different "records" for different purposes:

| Record Type     | Purpose                                                        | Example                                               |
| --------------- | -------------------------------------------------------------- | ----------------------------------------------------- |
| **A Record**    | Points a domain to an **IPv4** address.                        | `example.com` -> `93.184.216.34`                      |
| **AAAA Record** | Points a domain to an **IPv6** address.                        | `example.com` -> `2606:2800:220:1:248:1893:25c8:1946` |
| **CNAME**       | Aliases one name to another (Mapping a sub-domain).            | `www.example.com` -> `example.com`                    |
| **MX Record**   | Directs email to a mail server.                                | `mail.example.com`                                    |
| **TXT Record**  | Holds text information (often used for security/verification). | SPF or DKIM records                                   |

---

## Speed and Efficiency: TTL and Caching

Because checking this hierarchy every time would be slow, DNS uses **Caching**.

- **TTL (Time to Live):** Every DNS record has a TTL value (in seconds). This tells servers how long they can "remember" the
  IP address before they have to ask for a fresh one.
- If a TTL is set to 3600, your computer or ISP will remember that IP for one hour.

---

## Security Considerations

DNS was originally designed for functionality, not security, leading to some vulnerabilities:

- **DNS Spoofing/Poisoning:** An attacker inserts a fake IP address into a resolver's cache, sending users to a malicious
  site.
- **DNSSEC:** A suite of extensions that adds digital signatures to DNS records to ensure the data hasn't been tampered with.
- **DoH (DNS over HTTPS):** Encrypts DNS queries to prevent ISPs or attackers from seeing which websites you are visiting.
