In the world of cybersecurity, **X.509** is the standard format for **Public Key Infrastructure (PKI) certificates**. If TLS
is the "lock" on the door, an X.509 certificate is the "ID card" that proves the server behind that door is who they say they
are.

It is a strictly defined data format used to associate a **Public Key** with an **Identity** (like a domain name,
organization, or person).

---

### 1. What’s inside an X.509 Certificate?

An X.509 certificate isn't just a random string of characters; it is a structured file containing several specific fields:

| Field                  | Description                                                                                   |
| ---------------------- | --------------------------------------------------------------------------------------------- |
| **Version**            | Which X.509 version is used (usually v3, which allows for extensions).                        |
| **Serial Number**      | A unique identifier assigned by the Certificate Authority (CA).                               |
| **Algorithm ID**       | The algorithm used to sign the certificate (e.g., SHA-256 with RSA).                          |
| **Issuer**             | The entity that verified the information and signed the cert (e.g., DigiCert, Let's Encrypt). |
| **Validity Period**    | The "Not Before" and "Not After" dates (the expiration date).                                 |
| **Subject**            | The identity being secured (e.g., `CN=google.com`).                                           |
| **Subject Public Key** | The actual public key and the algorithm it uses.                                              |
| **Signature**          | The digital signature created by the Issuer's private key to prove authenticity.              |

---

### 2. The Chain of Trust

X.509 relies on a hierarchical structure. Your browser doesn't inherently know that `google.com` is legitimate. Instead:

1. **The Leaf Certificate:** `google.com` shows you its certificate.
2. **The Intermediate Certificate:** That cert was signed by an Intermediate CA.
3. **The Root Certificate:** The Intermediate was signed by a Root CA (like IdenTrust or GlobalSign).

Your operating system or browser comes pre-installed with a list of "Trusted Root CAs." As long as the certificate can be
traced back to one of those roots, your browser shows the green padlock.

---

### 3. Common File Extensions

While X.509 is the _standard_, you’ll see it implemented in various file formats on a Linux server:

- **.pem:** (Privacy Enhanced Mail) Base64 encoded ASCII text. Very common.
- **.der:** Binary version of the certificate.
- **.crt / .cer:** Usually PEM format, but technically just stands for "certificate."
- **.pfx / .p12:** A password-protected bundle that usually contains the certificate **and** the private key.

---

### 4. Extensions (The "v3" Magic)

The "v3" version of X.509 introduced **Extensions**, which made the protocol much more flexible. The most important one today
is:

- **Subject Alternative Name (SAN):** This allows one certificate to secure multiple domains (e.g., `google.com`,
  `youtube.com`, and `m.google.com` all on one "ID card").

### 5. Revocation: What if a key is stolen?

If a private key is compromised before the certificate expires, the X.509 system has two ways to "cancel" the ID:

- **CRL (Certificate Revocation List):** A "blacklist" published by the CA.
- **OCSP (Online Certificate Status Protocol):** A way for the browser to ask the CA in real-time: "Is this specific ID still
  valid?"
