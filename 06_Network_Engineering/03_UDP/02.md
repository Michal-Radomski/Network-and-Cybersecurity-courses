The structure of a UDP (User Datagram Protocol) packet—more accurately called a **UDP datagram**—is famously simple. Because
UDP is a connectionless protocol designed for speed, it doesn't need the complex tracking fields found in a TCP header.

A UDP datagram consists of two main parts: the **Header** (which is always exactly 8 bytes) and the **Data** (payload).

---

## The UDP Datagram Layout

The entire header is divided into four fields, each precisely 16 bits (2 bytes) long. It is typically visualized in a 32-bit
wide grid:

| Bit 0 to 15                        | Bit 16 to 31                   |
| ---------------------------------- | ------------------------------ |
| **Source Port** (16 bits)          | **Destination Port** (16 bits) |
| **Length** (16 bits)               | **Checksum** (16 bits)         |
| **Data** (Variable length payload) | ...                            |

---

## Breakdown of the Fields

### 1. Source Port (16 bits / 2 bytes)

- **What it is:** The port number of the sending application.
- **Purpose:** It tells the receiving device where to send a reply if one is needed.
- **Note:** This field is optional. If the sending application doesn't care about a response (for example, a simple broadcast
  status message), this field is set to all zeros.

### 2. Destination Port (16 bits / 2 bytes)

- **What it is:** The port number of the receiving application.
- **Purpose:** It ensures the packet gets delivered to the correct application on the destination machine (e.g., port 53 for
  DNS requests, port 123 for Network Time Protocol).
- **Note:** This field is mandatory.

### 3. Length (16 bits / 2 bytes)

- **What it is:** The total length of the entire UDP datagram (Header + Data).
- **Purpose:** It tells the receiving operating system where the UDP packet ends.
- **Limits:** Because it is a 16-bit field, the theoretical maximum size of a UDP datagram is $65,535$ bytes (8 bytes for the
  header, and up to $65,527$ bytes for data). However, practical network limits (like Ethernet MTU) usually restrict it to
  much less.

### 4. Checksum (16 bits / 2 bytes)

- **What it is:** A mathematical value used for error-checking the integrity of the header and the data.
- **Purpose:** When the receiver gets the packet, it recalculates the checksum. If the calculated value doesn't match the
  value in this field, it means the packet was corrupted during transit, and the receiver quietly discards it.
- **Note:** In IPv4, computing the checksum is optional (though highly recommended) and can be filled with zeros if not used.
  In IPv6, the UDP checksum is mandatory.

---

## 5. The Data (Payload)

This is the actual information being transmitted, such as a piece of a voice call, DNS query details, or game coordinates. It
comes immediately after the 8-byte header and varies in size depending on what the application needs to send.
