**UDP**, which stands for **User Datagram Protocol**, is one of the core protocols of the Internet protocol suite. It
operates at the Transport Layer (Layer 4) of the OSI model and is used to send short messages, called _datagrams_, across a
network.

Unlike its more famous sibling TCP (Transmission Control Protocol), UDP is designed for **speed and efficiency** rather than
absolute reliability.

---

## 1. The Core Concept: "Fire and Forget"

The easiest way to understand UDP is to think of it like regular postal mail.

- You write an address on an envelope and drop it in a mailbox.
- You _assume_ the post office will deliver it, but you don't get a notification when the recipient opens it.
- If you send ten letters in a row, they might arrive out of order, or one might get lost in transit, and you would never
  know unless the recipient told you.

Because of this, UDP is known as a **connectionless** and **unreliable** protocol. "Unreliable" doesn't mean it fails
constantly; it just means the protocol itself does not guarantee that data will reach its destination, arrive in the correct
order, or be checked for missing pieces.

---

## 2. Key Characteristics of UDP

- **No Handshake (Connectionless):** Before sending data, UDP does not establish a formal connection with the recipient. It
  just starts sending packets immediately.
- **Low Overhead:** A UDP packet header is incredibly lightweight—only 8 bytes long. (By comparison, a TCP header is at least
  20 bytes). This means less bandwidth is wasted on administrative data.
- **No Error Correction or Flow Control:** If a packet is dropped due to network congestion, UDP does not ask for a
  retransmission. It simply moves on to the next packet.
- **No Sequencing:** Packets are sent independently. If packet #3 arrives before packet #2 because it took a faster network
  route, UDP delivers them to the application exactly as they arrived.

---

## 3. The UDP Packet Structure

Because it is so simple, a UDP header only contains four fields:

| Field                | Size    | Description                                                                                |
| -------------------- | ------- | ------------------------------------------------------------------------------------------ |
| **Source Port**      | 16 bits | The port of the sending device (optional).                                                 |
| **Destination Port** | 16 bits | The port of the receiving device (required).                                               |
| **Length**           | 16 bits | The total size of the UDP header and data.                                                 |
| **Checksum**         | 16 bits | A basic integrity check to see if the data was corrupted in transit (discarded if broken). |

---

## 4. When is UDP Used?

If UDP loses data, why do we use it? We use it because **speed is more important than perfect accuracy** in certain
scenarios. If a network packet is delayed by even a fraction of a second while trying to retransmit, it becomes useless
anyway.

### Common Use Cases:

- **Live Video Streaming & VoIP:** If you are on a Zoom call and a few packets are dropped, the video might glitch for a
  millisecond, or the audio might crackle. This is much better than the entire call freezing for two seconds while TCP tries
  to fetch the missing pixels.
- **Online Gaming:** In a fast-paced multiplayer game, you need real-time updates on enemy positions. An old packet showing
  where a player was half a second ago is useless; you only care about the most recent data.
- **Domain Name System (DNS):** When you type a website URL, your computer sends a single UDP request to a DNS server asking
  for the IP address. It’s a quick "one-shot" request. If it fails, your browser simply tries again.
- **DHCP:** Used to automatically assign IP addresses to devices joining a network.

---

## 5. TCP vs. UDP: At a Glance

| Feature          | TCP                                               | UDP                                       |
| ---------------- | ------------------------------------------------- | ----------------------------------------- |
| **Connection**   | Connection-oriented (Requires 3-way handshake)    | Connectionless                            |
| **Reliability**  | Guaranteed delivery                               | No guarantee                              |
| **Speed**        | Slower (due to tracking and error checking)       | Faster (minimal overhead)                 |
| **Data Flow**    | Byte stream (data is read as a continuous stream) | Datagrams (packets are sent individually) |
| **Example Uses** | Web browsing (HTTP/HTTPS), Email, File transfers  | Gaming, Video streaming, DNS, VoIP        |
