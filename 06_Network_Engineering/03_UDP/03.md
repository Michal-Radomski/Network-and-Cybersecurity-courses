Here is the breakdown of the pros and cons of using UDP:

---

## The Pros (Advantages)

- **Maximum Speed and Low Latency:** Because UDP doesn't waste time establishing a connection (no 3-way handshake) or waiting
  for acknowledgments, data flows instantly. This makes it ideal for real-time applications.
- **Minimal Overhead:** A UDP header is only 8 bytes long, compared to TCP's minimum of 20 bytes. Less header data means
  lower bandwidth consumption, which is highly efficient for sending small, frequent packets.
- **No Delays from Packet Loss (No Head-of-Line Blocking):** In TCP, if packet #2 is lost, packets #3 and #4 must wait in a
  buffer until packet #2 is retransmitted. UDP doesn't care; it delivers packets to the application the moment they arrive.
- **Supports Broadcast and Multicast:** UDP can easily send a single packet to every device on a local network (broadcast) or
  to a specific group of devices (multicast). TCP can only do one-to-one (unicast) communications.

---

## The Cons (Disadvantages)

- **No Guaranteed Delivery:** UDP is a "best-effort" protocol. If a packet gets dropped due to a weak Wi-Fi signal or a
  congested router, it is gone forever. UDP will never automatically retransmit it.
- **No Packet Ordering:** Packets can take different routes across the internet. UDP does not put packets back in numerical
  order; the receiving application gets them in whatever random order they happen to arrive.
- **No Flow or Congestion Control:** UDP will blindly pump data into a network as fast as the application generates it. It
  cannot detect if the network is clogged, meaning it can easily overwhelm a receiver or worsen network congestion.
- **Security Vulnerabilities (Spoofing):** Because UDP is connectionless and doesn't validate the sender with a handshake, it
  is much easier for attackers to fake (spoof) the source IP address. This makes UDP a frequent target for Distributed Denial
  of Service (DDoS) reflection attacks.
- **Requires Application-Level Logic:** If you want _any_ reliability while using UDP, you have to write the code for it
  yourself within your application (e.g., creating your own custom tracking or retransmission system).

---

## Summary: When to use it?

> **Use UDP if:** You are building something where old data is useless data. If a fraction-of-a-second delay ruins the
> experience—like in a competitive multiplayer video game, a live Discord voice call, or live video streaming—UDP is the
> right choice. **Avoid UDP if:** Every single piece of data must arrive intact. For web browsing (HTTPS), downloading files,
> sending emails, or database communication, you should use **TCP**.
