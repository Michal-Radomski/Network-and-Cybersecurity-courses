**TCP Fast Open (TFO)** is an extension to the Transmission Control Protocol (TCP) designed to speed up the opening of
successive TCP connections between two endpoints.

By allowing data to be exchanged during the initial handshake, TFO eliminates a full round-trip time (RTT) delay, making web
browsing and API calls noticeably faster.

---

## The Problem: The Standard TCP Handshake

In standard TCP, a connection must be fully established using a **3-way handshake** before any application data (like an HTTP
request) can be sent.

1. **SYN:** Client sends a synchronization request.
2. **SYN-ACK:** Server responds and acknowledges.
3. **ACK:** Client acknowledges, and _only now_ can it attach data.

This means it takes **1 full RTT** of waiting around before the server actually learns what the client wants.

---

## The Solution: How TCP Fast Open Works

TFO optimizes this process for subsequent connections to the same server by using a cryptographic **Cookie**. Here is how it
plays out:

### Phase 1: The First Connection (Cookie Request)

1. The client sends a standard `SYN` packet but includes a request for a **TFO Cookie**.
2. The server generates a cookie (by hashing the client's IP address with a secret key) and sends it back in the `SYN-ACK`
   packet.
3. The client caches this cookie for future use and completes the handshake normally.

### Phase 2: Subsequent Connections (The "Fast" Open)

When the client wants to connect to that same server again later:

1. **Data in the SYN:** The client sends a `SYN` packet that contains both the **cached TFO Cookie** and the actual
   **application data** (e.g., an HTTP GET request).
2. **Immediate Processing:** The server validates the cookie. If it's valid, the server accepts the data, processes it, and
   sends back a `SYN-ACK` along with the _response data_ immediately.
3. The connection is fully established, but the client already has its data without waiting for the classic handshake to
   finish.

---

## Why Isn't It Used for Everything?

While TFO sounds perfect, it comes with a few trade-offs and deployment challenges:

- **Security (Replay Attacks):** Because data is sent before the connection is fully validated, a malicious actor could
  intercept the `SYN` packet + data and resend it (replay it) to the server. For this reason, TFO should only be used for
  **idempotent requests** (requests that don't change state, like fetching a webpage) and is risky for non-idempotent actions
  (like authorizing a bank transfer).
- **Middlebox Interference:** Some older routers, firewalls, and middleboxes on the internet drop TCP packets that contain
  unexpected data or options in the `SYN` packet, thinking they are malicious.
- **Resource Consumption:** Servers have to perform cryptographic operations to validate cookies, which can add CPU overhead
  during high-traffic spikes.

---

## Current Adoption

Today, TCP Fast Open is widely supported across major operating systems (Linux, Windows, macOS, iOS, Android) and major web
browsers. It is highly effective for mobile networks where latency (RTT) is naturally higher, providing a much snappier user
experience.
