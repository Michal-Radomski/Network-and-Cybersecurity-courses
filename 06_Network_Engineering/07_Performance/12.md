Securing network access to database servers is one of the most critical steps in protecting sensitive data. Because databases
are the ultimate targets for most cyberattacks, allowing unrestricted or poorly managed network access leaves them highly
vulnerable.

Implementing a robust Network Access Control (NAC) strategy for database servers requires a defense-in-depth approach.

---

## 1. Network Segmentation and Isolation

The golden rule of database security is that **database servers should never be directly accessible from the public
internet.** \* **Private Subnets:** Place database servers in a dedicated private subnet within your Virtual Private Cloud
(VPC) or local network. This subnet should have no direct route to or from the internet.

- **Demilitarized Zone (DMZ):** Public-facing components (like web servers or load balancers) should reside in a separate
  DMZ. They act as the only entry point, proxying necessary requests to the internal network.

---

## 2. The Principle of Least Privilege (Firewalls & Security Groups)

Network firewalls and cloud security groups should be configured to restrict traffic to the absolute minimum required for
operations.

- **IP Whitelisting:** Only allow inbound traffic from the specific IP addresses or subnets of the application servers that
  require data access. Block all other IP addresses by default.
- **Port Restriction:** Close all network ports except the specific port used by the database management system (DBMS).

| Database             | Default Port |
| -------------------- | ------------ |
| MySQL                | 3306         |
| PostgreSQL           | 5432         |
| Microsoft SQL Server | 1433         |
| Oracle Database      | 1521         |
| MongoDB              | 27017        |

---

## 3. Secure Administrative Access

Database administrators (DBAs) and developers often need direct access to the database for maintenance, troubleshooting, or
querying. This access must be tightly controlled.

- **Bastion Hosts / Jump Boxes:** Require administrators to first connect to a hardened, monitored "bastion host" in the DMZ
  via SSH or RDP. From that host, they can then initiate a connection to the database server.
- **Virtual Private Networks (VPNs) / Zero Trust Network Access (ZTNA):** Ensure that any remote administrative access
  requires authentication through a secure corporate VPN or a ZTNA solution that verifies device health and user identity
  before granting network-level access.

---

## 4. Encryption in Transit

Network access control is less effective if the data flowing across the permitted network pathways can be intercepted.

- **Transport Layer Security (TLS/SSL):** Enforce TLS encryption for all connections between application servers, client
  tools, and the database server. This prevents eavesdropping and man-in-the-middle (MitM) attacks.
- **Certificate-Based Authentication:** Use mutual TLS (mTLS) where both the client and the server must present valid
  cryptographic certificates to establish a network connection, ensuring absolute certainty of the client's identity.

---

## 5. Continuous Monitoring and Intrusion Detection

Controlling access is a continuous process that requires real-time visibility.

- **Network Flow Logs:** Enable VPC flow logs or network logging to monitor all traffic attempting to reach the database
  subnet. Track both accepted and denied connection attempts.
- **Intrusion Detection/Prevention Systems (IDS/IPS):** Deploy network-based IDS/IPS to detect anomalous behavior, such as a
  sudden spike in data exfiltration or unauthorized port scanning within the internal network.

> ### 🔒 Best Practice Reminder
>
> Network access control handles _how_ a packet reaches the database server. Once the network connection is permitted, strict
> **Database Authentication** (e.g., IAM roles, multi-factor authentication) and **Database Authorization** (role-based
> access control to specific tables) must take over to ensure comprehensive security.
