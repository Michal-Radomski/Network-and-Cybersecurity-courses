In computer networking, a **listening server** (more commonly referred to as a **listening socket** or a server in a
"listening state") is a software program or backend process configured to wait for incoming connection requests from clients
over a network.

Instead of actively reaching out to other computers, it stays passive, keeping a specific network port open and monitoring it
for any inbound traffic.

---

## How a Listening Server Works

To understand a listening server, it helps to use the analogy of a business telephone line. The server is the office, the
port is the specific phone extension, and the "listening" state is the phone ringing, waiting for a customer to call.

Here is the standard lifecycle of how a server listens and connects:

```
[ Client ]                               [ Listening Server ]
    |                                             |
    |  1. Initiates Connection (SYN)              | (Listening on Port)
    |-------------------------------------------->|
    |                                             | 2. Accepts Connection
    |  3. Establishes Session                     |    & Spawns Worker
    |<--------------------------------------------|
    |                                             |
    |====== 4. Data Exchange (HTTP/FTP/etc.) =====|

```

1. **Binding to a Port:** The server application starts up and claims a specific port number (e.g., port 80 for standard web
   traffic) and an IP address. This is called "binding."
2. **Entering the Listen State:** The server issues a system call (like `listen()` in programming) that tells the operating
   system's network stack to start queueing up any incoming connection requests sent to that port.
3. **The Handshake:** When a client (like your web browser) tries to connect, it sends a request. The listening server
   detects this, accepts the connection, and usually hands the session off to a separate "worker" thread or process.
4. **Back to Listening:** By handing the active conversation off to a worker, the primary listening socket immediately goes
   back to waiting for the _next_ client to call.

---

## Key Concepts

### 1. Ports and Protocols

A listening server relies on specific ports and protocols (usually TCP or UDP) to know what kind of traffic to expect. Common
examples include:

- **Port 80 / 443:** Web servers listening for HTTP/HTTPS requests.
- **Port 22:** SSH servers listening for secure remote management connections.
- **Port 25:** Mail servers listening for SMTP email transfers.

### 2. The Connection Queue (Backlog)

If hundreds of clients try to connect at the exact same millisecond, the listening server can't answer them all instantly. It
keeps a small waiting room called a **backlog queue**. If the queue fills up because the server is too slow to accept
connections, new clients will get a "Connection Refused" error.

### 3. Firewalls and Security

Because a listening server leaves a digital "door" open to the internet, it is a primary target for hackers. Network
administrators use firewalls to restrict who is allowed to talk to a listening server, ensuring that only legitimate traffic
gets through to the listening port.

---

## Checking for Listening Servers

If you are troubleshooting a computer or server, you can see what programs are currently acting as listening servers using
command-line tools.

- **On Windows/Linux/Mac:** Opening a terminal and typing `netstat -an` or `ss -l` (on Linux) will display a list of all
  active network connections and show which ports are currently in the **`LISTENING`** state.
