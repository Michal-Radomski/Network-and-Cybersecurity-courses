When you run `curl -v https://www.example.com`, the `-v` flag stands for **verbose**. Instead of just downloading the
webpage's HTML, `curl` will output a detailed, behind-the-scenes look at the entire network operation.

The output is beautifully structured and color-coded (or symbol-coded) to show the chronological lifecycle of a web request.
Lines starting with `*` indicate `curl`'s internal state messages, lines starting with `>` show the data sent _to_ the server
(the request), and lines starting with `<` show data received _from_ the server (the response).

Here is a step-by-step breakdown of exactly what the output describes, chronologically:

---

## 1. DNS Resolution and TCP Handshake (`*`)

The very first lines show `curl` translating the domain name into an IP address and initiating the connection.

```text
* Trying 93.184.215.14:443...
* Connected to www.example.com (93.184.215.14) port 443

```

- **What it means:** `curl` queried your local DNS cache/resolver, found the IP address for `www.example.com`
  (`93.184.215.14`), and successfully completed the **TCP Three-Way Handshake** on port 443 (the default port for secure
  HTTPS traffic).

---

## 2. The TLS/SSL Handshake (`*`)

Because the URL uses `https://`, `curl` must negotiate an encrypted tunnel before sending any HTTP data.

```text
* ALPN: curl offers h2, http/1.1
* TLSv1.3 (OUT), TLS handshake, Client hello (1):
* TLSv1.3 (IN), TLS handshake, Server hello (2):
* TLSv1.3 (IN), TLS handshake, Certificate (11):
* TLSv1.3 (IN), TLS handshake, Verification (15):
* TLSv1.3 (IN), TLS handshake, Finished (20):
* SSL connection using TLSv1.3 / AEAD-AES256-GCM-SHA384
* Server certificate: www.example.com

```

- **What it means:** \* **ALPN:** `curl` tells the server it is capable of speaking HTTP/2 (`h2`) or HTTP/1.1.
- **Handshake:** You see the single round-trip **TLS 1.3 Handshake** occur (`Client hello` $\rightarrow$ `Server hello`).
- **Certificate & Verification:** The server hands over its X.509 digital certificate, and `curl` verifies it against your
  operating system's built-in pool of trusted Certificate Authorities (CAs).
- **Cipher Suite:** It prints the exact cryptographic algorithms chosen to encrypt the session (e.g., `AES256-GCM`).

---

## 3. The HTTP Request Headers (`>`)

Now that a secure tunnel is open, `curl` constructs and transmits the actual HTTP request. Every line here is sent from your
computer to the server.

```text
> GET / HTTP/1.1
> Host: www.example.com
> User-Agent: curl/8.4.0
> Accept: */*
>

```

- **What it means:**
- `GET / HTTP/1.1`: `curl` asks for the root path (`/`) using the HTTP/1.1 protocol.
- `Host`: Confirms the target domain name (vital for servers hosting multiple sites on one IP).
- `User-Agent`: Identifies the client software making the request (`curl`).
- `Accept: */*`: Tells the server that `curl` will accept any type of response data (HTML, text, JSON, etc.).
- **The Empty Line:** A crucial part of the HTTP protocol indicating that the headers are finished and no request body is
  following.

---

## 4. The HTTP Response Headers (`<`)

The server processes the request and sends back metadata about the response.

```text
< HTTP/1.1 200 OK
< Age: 531343
< Cache-Control: max-age=604800
< Content-Type: text/html; charset=UTF-8
< Date: Thu, 28 May 2026 21:50:00 GMT
< ETag: "3147526947+ident"
< Server: ECS (sab/56B1)
< Content-Length: 1256
<

```

- **What it means:**
- `HTTP/1.1 200 OK`: The server successfully found the page (`200 OK`).
- `Content-Type`: The server warns `curl` that the incoming payload is an HTML document encoded in UTF-8.
- `Content-Length`: The actual payload body will be exactly 1,256 bytes long.
- Caching headers (`Cache-Control`, `ETag`, `Age`) are included to dictate how browsers should store this file.

---

## 5. The Response Body (The HTML Payload)

Following the final response header, the actual content of the webpage is dumped directly into your terminal.

```text
<!doctype html>
<html>
<head>
    <title>Example Domain</title>
...
</body>
</html>

```

- **What it means:** This is the raw HTML document that a standard web browser would parse, style, and render visually as a
  webpage.

---

## 6. Connection Teardown (`*`)

Once all the bytes promised in the `Content-Length` header are received, `curl` clean up the connection.

```text
* Connection #0 to host www.example.com left intact

```

- **What it means:** `curl` has successfully finished downloading the asset. By default, it notes that the underlying TCP
  connection is left "intact" (kept alive) in case you chain multiple requests together, though the utility will now exit and
  close it.
