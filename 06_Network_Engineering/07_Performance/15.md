In a `pg_hba.conf` file, **`ident`** is an authentication method that relies on the client's **operating system username** to
verify their identity.

Instead of asking the user for a password, PostgreSQL asks an external service (or the local OS) who is currently logged into
the client machine. If that OS username matches the requested PostgreSQL database username, access is granted.

---

## How `ident` Works

The behavior of `ident` depends entirely on whether the connection is local or coming over a network:

### 1. Local Connections (Same Server)

When used for local connections (via Unix-domain sockets), `ident` is effectively identical to the **`peer`** authentication
method. PostgreSQL simply checks the OS user ID of the process that initiated the connection.

- _Example:_ If you are logged into Linux as the user `bob`, and you type `psql -d mydb -U bob`, PostgreSQL checks the OS,
  sees you are indeed `bob`, and lets you in without a password.

### 2. Remote Connections (Over a Network)

For TCP/IP connections, `ident` works by communicating with an **ident server** (RFC 1413 daemon) running on the client's
machine.

1. A user on a remote client machine tries to connect to PostgreSQL.
2. PostgreSQL pauses the connection and reaches back to the client machine on **port 113** (the standard ident port).
3. The ident daemon on the client machine replies: _"The user who opened this network connection is named 'alice'."_
4. PostgreSQL checks if 'alice' is allowed to log into the requested database.

---

## The Big Catch: Why `ident` is Rarely Used Today

While `ident` sounds convenient, it has a massive security flaw when used over a network: **It completely trusts the client
machine.**

> ⚠️ **The Security Risk:** If an attacker controls the client machine, or if they spoof an ident server on their own laptop,
> they can configure their ident daemon to lie. When PostgreSQL asks, _"Who is connecting?"_, the attacker's machine can
> simply reply _"root"_ or _"postgres"_, and PostgreSQL will grant them full administrative access.

Because of this, `ident` is considered **insecure for untrusted networks** (like the internet or large corporate subnets). It
is only safe if the client machine is strictly isolated, completely secure, and under your absolute control.

## The `ident` Map File (`pg_ident.conf`)

Sometimes, your Linux username doesn't exactly match your PostgreSQL database username. For example, your Linux user might be
`ann_smith`, but your database user is `production_app`.

To fix this, `ident` can be paired with a map file named **`pg_ident.conf`**.

In `pg_hba.conf`, you would write:

```text
# TYPE  DATABASE        USER            ADDRESS                 METHOD
host    all             production_db   10.0.0.5/32             ident map=mymap

```

Then, inside **`pg_ident.conf`**, you link them together:

```text
# MAPNAME     SYSTEM-USERNAME      PG-USERNAME
mymap         ann_smith            production_db
mymap         boss_man             production_db

```

This tells PostgreSQL: _"If the ident server confirms the OS user is `ann_smith` or `boss_man`, allow them to log in as
`production_db`."_

---

## Summary: What should you use instead?

- For local command-line access on the server itself, use **`peer`**.
- For remote application and user connections, use **`scram-sha-256`** (password-based) or **`cert`** (SSL certificates), as
  they do not rely on trusting the client's operating system.
