When you are designing scalable systems, deciding where to balance your traffic is a critical architectural choice. The terms
**Layer 4 (L4)** and **Layer 7 (L7)** refer to different layers of the **OSI (Open Systems Interconnection) model**, and they
handle traffic with entirely different levels of awareness.

Here is a breakdown of how they work, how they compare, and when to use each.

---

## The Core Difference: Awareness

The fundamental difference lies in **what data the load balancer can see**.

- **Layer 4 (Transport Layer):** Operates on a "blind" routing principle. It only looks at the network protocol,
  source/destination IP addresses, and ports (like TCP or UDP). It has **no idea** what the actual payload contains.
- **Layer 7 (Application Layer):** Operates with full context. It terminates the network connection, decrypts the traffic (if
  using HTTPS), and inspects the actual application data—such as HTTP headers, cookies, URLs, and form data.

---

## Layer 4 Load Balancing

Think of Layer 4 load balancing like a mailroom sorting packages based solely on the zip code written on the box. The sorter
never opens the box.

### How it Works

When a request comes in, the L4 load balancer makes a routing decision based on the packet header. It uses algorithms like
Round Robin or Least Connections to forward the TCP packets directly to a backend server. A connection is established
directly between the client and the backend server, with the load balancer acting as a fast traffic cop.

### Pros

- **High Performance & Low Latency:** Because it doesn't inspect the data payload or handle TLS/SSL decryption, it requires
  minimal CPU power. It can handle massive amounts of throughput incredibly fast.
- **Security by Isolation:** The load balancer doesn't need access to your SSL certificates, reducing the attack surface.
- **Protocol Agnostic:** It works for any traffic using TCP or UDP (e.g., database connections like MySQL, gaming traffic, or
  SSH), not just web traffic.

### Cons

- **No Smart Routing:** It cannot route traffic based on content (e.g., sending `/images` requests to a media server).
- **Inefficient Caching:** It cannot cache content or compress data because it cannot read it.
- **No Cookie-Based Sticky Sessions:** If your application requires a user to stay on the same server based on a session
  cookie, L4 can only approximate this using the user's IP address (which can change or be shared via NAT).

---

## Layer 7 Load Balancing

Think of Layer 7 load balancing like a concierge who opens your letter, reads your request, and directs you to the exact
department best suited to help you.

### How it Works

An L7 load balancer acts as a reverse proxy. It terminates the incoming client connection, decrypts the TLS/SSL layer,
inspects the HTTP/HTTPS request, and then opens a _new_ connection to the chosen backend server.

### Pros

- **Smart/Content-Based Routing:** You can route traffic based on URL paths (e.g., `/api` vs `/static`), host headers
  (`app.example.com` vs `blog.example.com`), language preferences, or device types.
- **Microservices Friendly:** Essential for microservices architectures where different services handle different parts of an
  application.
- **Advanced Features:** It can handle SSL/TLS termination, HTTP/2 multiplexing, data compression, and content caching at the
  edge.
- **Security & WAF Integration:** Because it looks at the data, it can act as a Web Application Firewall (WAF) to block SQL
  injections, cross-site scripting (XSS), or malicious bots before they hit your servers.
- **True Sticky Sessions:** It can read session cookies to ensure a user always hits the exact same backend server for their
  session.

### Cons

- **Resource Intensive:** Decrypting traffic and inspecting application data requires significantly more CPU and memory than
  L4.
- **Higher Latency:** The double-connection overhead (Client ➔ LB ➔ Server) adds a small amount of latency compared to L4.
- **Certificate Management:** You must manage and upload your SSL/TLS certificates directly to the load balancer.

---

## At-a-Glance Comparison

| Feature                | Layer 4 (Transport)                                 | Layer 7 (Application)                             |
| ---------------------- | --------------------------------------------------- | ------------------------------------------------- |
| **Data Inspected**     | IP, Port, TCP/UDP protocol                          | HTTP/HTTPS headers, Cookies, URLs, Payload        |
| **Speed/Throughput**   | Extremely fast, low CPU usage                       | Slower, high CPU usage                            |
| **TLS/SSL Decryption** | No                                                  | Yes (Required to inspect data)                    |
| **Routing Decisions**  | Packet-level (Round robin, IP hash)                 | Content-level (Path, Header, Cookie)              |
| **Common Use Cases**   | Databases, TCP streaming, high-volume raw traffic   | Web apps, Microservices, API gateways             |
| **Examples**           | AWS Network Load Balancer (NLB), HAProxy (TCP mode) | AWS Application Load Balancer (ALB), NGINX, Envoy |

---

## When to Choose Which?

- **Choose Layer 4 if:** You need ultra-high performance, are dealing with millions of requests per second, are balancing
  non-HTTP traffic (like databases or MQTT), or simply need a fast, dumb pipe to distribute raw TCP traffic.
- **Choose Layer 7 if:** You are building a modern web application, need to route traffic to specific microservices based on
  the URL, require sticky sessions, or want to offload SSL decryption and security filtering away from your application
  servers.

> **Pro Tip:** In large enterprise architectures, you don't always choose just one. It is incredibly common to **chain them
> together**. A highly scalable architecture might feature a Layer 4 load balancer at the very front to handle massive
> internet traffic, which then distributes packets to a pool of Layer 7 load balancers that handle the complex routing, SSL
> decryption, and application logic.
