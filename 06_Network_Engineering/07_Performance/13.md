In PostgreSQL, **`pg_hba.conf`** is the primary configuration file used to control host-based authentication (HBA). It acts
as the database’s first line of defense, acting like a network firewall specifically for PostgreSQL.

Before a user can even attempt to log in with a password, the database checks this file to see if the connection attempt is
allowed based on _where_ it's coming from and _who_ is trying to connect.

If a connection attempt doesn't match any rules in this file, access is immediately denied.

---

## How the File is Evaluated

When a client attempts to connect, PostgreSQL reads the `pg_hba.conf` file **from top to bottom**.

- It uses the **first matching rule** it finds.
- Once a match is found, it enforces the specified authentication method.
- If that authentication fails, the connection is rejected immediately; it will _not_ fall back to subsequent lines.

---

## The Record Format

Each active line in `pg_hba.conf` follows a specific structure made of up to seven fields:

```text
# TYPE  DATABASE        USER            ADDRESS                 METHOD  [OPTIONS]

```

Here is a breakdown of what each field means:

### 1. TYPE (Connection Type)

Defines how the client is attempting to connect.

- `local`: Matches connection attempts using Unix-domain sockets (local connections on the server itself).
- `host`: Matches connections over TCP/IP (both unencrypted and encrypted).
- `hostssl`: Matches connections over TCP/IP, but **only** if they use SSL/TLS encryption.
- `hostnossl`: Matches connections over TCP/IP that do **not** use SSL.

### 2. DATABASE

Specifies which database(s) this rule applies to.

- Can be a specific database name (e.g., `production_db`).
- `all`: Matches all databases.
- `replication`: Matches replication connections (used for high availability streaming).
- A comma-separated list (e.g., `db1,db2`).

### 3. USER

Specifies which database user(s) this rule applies to.

- Can be a specific username (e.g., `db_admin`).
- `all`: Matches all users.
- A group name prefixed with a plus sign (e.g., `+marketing_team`).

### 4. ADDRESS

Specifies the client machine's network address. _(This field is omitted for `local` connections)._

- Can be an IPv4 address/subnet (e.g., `192.168.1.50/32` for a single host, or `10.0.0.0/24` for a network).
- Can be an IPv6 address (e.g., `::1/128`).
- `all`: Matches any IP address.
- `samehost` or `samenet`: Matches the server’s own IP or its local network.

### 5. METHOD

Specifies how the user must prove their identity. Common methods include:

- `scram-sha-256`: The recommended, secure password-based authentication (securely hashes passwords).
- `md5`: An older password-based method (deprecated in newer versions).
- `trust`: Unconditionally allows the connection **without a password**. (Use with extreme caution, usually only for `local`
  debugging).
- `reject`: Unconditionally rejects the connection. Useful for explicitly blocking a specific IP.
- `cert`: Uses client SSL certificates for authentication (no password required).
- `gss` / `sspi` / `ldap`: Integrates with external enterprise authentication systems like Active Directory or Kerberos.

---

## Common Examples

Here is what a typical, secure `pg_hba.conf` file might look like:

```text
# Allow any local user on the server machine to connect via Unix sockets using SCRAM
local   all             all                                     scram-sha-256

# Allow the internal application server (IP 10.0.0.5) to connect to the prod database securely
hostssl production_db   app_user        10.0.0.5/32             scram-sha-256

# Allow replica servers within the local subnet to connect for replication
host    replication     replicator      192.168.1.0/24          scram-sha-256

# Reject all other external connections by default (Good practice/Implicit fallback)
host    all             all             0.0.0.0/0               reject

```

---

## Applying Changes

If you modify `pg_hba.conf`, the changes will not take effect automatically. You do not need to restart the entire database
server (which causes downtime), but you must **reload** the configuration.

You can do this via the command line:

```bash
pg_ctl reload

```

Or from within a SQL interface as a superuser:

```sql
SELECT pg_reload_conf();

```
