**Nagle’s Algorithm** is a mechanism designed to improve the efficiency of TCP/IP networks by reducing the number of small
packets sent over the wire.

While it works wonders for bandwidth efficiency, it can wreak havoc on applications that require real-time responsiveness.
Here is a breakdown of how it works, its effects, and when you need to turn it off.

---

## The Problem It Solves: "Small Packet Syndrome"

Every TCP packet requires a minimum header size (usually 20 bytes for TCP and 20 bytes for IP). If an application sends data
one byte at a time—like a user typing in a Telnet session—the resulting packet is 41 bytes long to transport just **1 byte**
of useful data. This results in a massive 4000% overhead, clogging the network with tiny packets.

### How Nagle's Algorithm Works

Nagle’s algorithm fixes this by introducing a simple rule for the sending TCP socket:

1. If the application has enough data to fill a **Maximum Segment Size (MSS)** packet, send it immediately.
2. If the data is smaller than an MSS, hold it in a buffer **until the previous packet's acknowledgment (ACK) is received**
   from the receiver.
3. Once the ACK arrives, or enough data accumulates to fill an MSS, the buffered data is bundled together and sent as a
   single packet.

---

## The Core Effects

### 1. Improved Bandwidth Efficiency

By TCP "coalescing" (batching) small data chunks into fewer, larger packets, Nagle’s algorithm drastically reduces header
overhead. This makes it highly effective for bulk data transfers, file uploads, and web browsing.

### 2. Increased Latency (The Downside)

Because the algorithm intentionally holds back data while waiting for an ACK, it introduces a noticeable delay. If the
network is slow or the receiver takes time to respond, the sender sits on the data, causing a bottleneck.

### 3. The Deadly Combo: Nagle + Delayed ACKs

The worst effect of Nagle's algorithm happens when it interacts with another TCP feature called **Delayed ACKs**.

- **Delayed ACKs** tell the receiver to wait a brief moment (up to 200ms) before sending an ACK, hoping it can piggyback the
  ACK onto an outgoing data packet.
- When a sender using Nagle's algorithm talks to a receiver using Delayed ACKs, a **deadlock** occurs: the sender is waiting
  for an ACK to send more data, and the receiver is waiting for more data to send an ACK. The connection stalls until the
  Delayed ACK timer expires.

---

## When to Keep It vs. When to Disable It

| Scenario                                 | Nagle's Algorithm Status | Why?                                                                                   |
| ---------------------------------------- | ------------------------ | -------------------------------------------------------------------------------------- |
| **File Transfers (FTP, HTTP downloads)** | **Enabled (Default)**    | Maximizes throughput; latency doesn't matter for background downloads.                 |
| **Online Gaming**                        | **Disabled**             | Real-time inputs (like moving or shooting) cannot afford a 100–200ms delay.            |
| **Financial/Trading Apps**               | **Disabled**             | Stock tickers and order execution require sub-millisecond precision.                   |
| **SSH / Telnet**                         | **Disabled (Usually)**   | Modern implementations disable it so keystrokes feel instantly responsive to the user. |

---

## How to Control It

In network programming, you can disable Nagle’s algorithm by setting the `TCP_NODELAY` socket option.

> **`TCP_NODELAY = 1`** > This bypasses Nagle’s buffer, instructing TCP to send data across the network the exact moment the
> application writes it, prioritizing speed over bandwidth conservation.
