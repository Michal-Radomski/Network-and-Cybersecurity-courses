In Linux and other Unix-like operating systems, the philosophy is that **"everything is a file."** Whether you are
interacting with a physical document, a hard drive, a keyboard, or even a network connection, the operating system treats
them all as files.

A **File Descriptor (FD)** is the "handle" the system uses to manage these connections.

---

## 1. The Simple Definition

Think of a file descriptor as a **unique ID number** assigned by the kernel to an open file.

When a process wants to read from or write to a file, it doesn't grab the file directly by its name. Instead, it asks the OS
to open the file. The OS checks permissions, opens the file, and hands the process an integer (like 3, 4, or 5). From that
point on, the process simply says, "Hey OS, write this data to ID #3."

---

## 2. The Standard File Descriptors

Every time a new process (like a terminal command or a script) starts in Linux, it automatically opens three default file
descriptors:

| Integer | Name     | Description     | Default Destination    |
| ------- | -------- | --------------- | ---------------------- |
| **0**   | `stdin`  | Standard Input  | Your Keyboard          |
| **1**   | `stdout` | Standard Output | Your Screen (Terminal) |
| **2**   | `stderr` | Standard Error  | Your Screen (Terminal) |

When you use redirection in the terminal (like `2> error.log`), you are telling the shell to point file descriptor **2** to a
file instead of your screen.

---

## 3. How it Works Internally

The process of mapping an integer to a file involves three layers in the Linux kernel:

1. **The Process Descriptor Table:** Each process has its own private table of integers (0, 1, 2, 3...). Each entry points to
   an entry in the system-wide file table.
2. **The Open File Table:** A system-wide list of all open files. It keeps track of the **file offset** (where you are
   currently reading or writing) and the status flags (read-only, write-only, etc.).
3. **The Inode Table:** This points to the actual physical data on the disk or the specific hardware device.

---

## 4. Key Characteristics

- **Non-negative Integers:** File descriptors are always positive integers.
- **Per-Process Limits:** There is a limit to how many files a single process can have open (often 1024 by default), which
  prevents a single "leaky" program from crashing the whole system.
- **Resource Management:** When a program is finished with a file, it must "close" the file descriptor. If it doesn't, it
  causes a **file descriptor leak**, and eventually, the program won't be able to open anything else.

## 5. Seeing it in Action

You can actually see the file descriptors for any running process on your system by looking in the `/proc` directory. If you
find the Process ID (PID), you can run:

```bash
ls -l /proc/[PID]/fd

```

This will show you a list of numbers and exactly which files or devices they are linked to.
