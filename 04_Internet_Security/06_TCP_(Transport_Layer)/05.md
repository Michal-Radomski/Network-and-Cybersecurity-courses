A **SYN Flooding Attack** is a type of Denial-of-Service (DoS) attack that exploits the fundamental way a TCP connection is
established. It targets the "Handshake" process to overwhelm a server's resources.

---

## 1. The Normal TCP Three-Way Handshake

To understand the attack, you first have to look at how a healthy connection starts:

1.  **SYN:** The client sends a "Synchronize" packet to the server.
2.  **SYN-ACK:** The server acknowledges this by sending a "Synchronize-Acknowledgment" and sets aside a bit of memory (in a
    "backlog queue") to wait for the final step.
3.  **ACK:** The client sends an "Acknowledgment" packet. The connection is now open.

---

## 2. How the SYN Flood Works

In a SYN flood, the attacker sends a massive volume of **SYN** packets but never intends to finish the handshake.

- **Step 1 (The Flood):** The attacker sends thousands of SYN packets to the server. These packets usually have **spoofed
  (fake) IP addresses**.
- **Step 2 (The Trap):** The server, following protocol, responds with a SYN-ACK to each request and opens a "half-open"
  connection in its memory.
- **Step 3 (The Stall):** Because the source IPs are fake or the attacker simply ignores the replies, the final **ACK** never
  arrives.
- **The Result:** The server’s memory queue for "half-open" connections fills up completely. When a legitimate user tries to
  connect, the server has no room left in its table and rejects the connection.

> **Analogy:** Imagine a prankster calling a pizza shop from 100 different phones. The shop picks up, asks "What would you
> like?", and the prankster just stays silent. The shop keeps the line open waiting for an answer. Eventually, all the shop's
> phone lines are busy, and a real customer calling to order a pizza gets a busy signal.

---

## 3. Common Defenses

Modern systems have several ways to fight back against this:

- **SYN Cookies:** This is the most common defense. Instead of saving the connection info in memory immediately, the server
  sends back a SYN-ACK with a specially crafted "Sequence Number" (the cookie). It only allocates memory **after** the client
  sends back a valid ACK.
- **Micro-blocks:** The server allocates a tiny amount of memory instead of a full connection object to save space.
- **Reducing Timeouts:** The server is configured to "give up" on half-open connections much faster, clearing out the queue
  for new users.
- **Firewalls/IPS:** Network security devices can detect the abnormal rate of SYN packets from a single source (or to a
  single destination) and block them automatically.

---

## Summary

| Feature             | Normal Connection              | SYN Flood Attack                           |
| :------------------ | :----------------------------- | :----------------------------------------- |
| **Final ACK**       | Sent by client.                | Never sent (or sent to a fake IP).         |
| **Server State**    | Established (Ready).           | Half-Open (Waiting).                       |
| **Server Resource** | Used briefly during handshake. | Exhausted/Full.                            |
| **Impact**          | Successful communication.      | Service becomes unavailable to real users. |
