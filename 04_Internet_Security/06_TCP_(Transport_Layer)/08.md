**TCP Session Hijacking** is a sophisticated attack where an unauthorized user takes over an active TCP session between two
parties. If a SYN Flood is about stopping a connection and a Reset Attack is about killing one, Session Hijacking is about
**stealing** one.

By the time the hijacking occurs, the victim has already logged in (authenticated). By stealing the session, the attacker
bypasses the need for a username or password.

---

## 1. How the Attack Works

The core of the attack relies on predicting or stealing the **Next Sequence Number** that the server expects from the client.

1.  **Tracking (Sniffing):** The attacker must be able to see the traffic between the client and the server. They use a
    packet sniffer to monitor the current **Sequence (Seq)** and **Acknowledgment (Ack)** numbers.
2.  **Desynchronization:** The attacker sends a "storm" of data or a Reset packet to the legitimate client to "desynchronize"
    them from the server. The goal is to make the client lose track of the session or stop communicating temporarily.
3.  **The Takeover:** The attacker crafts a packet with the client’s IP address, the correct source port, and—most
    importantly—the **exact Sequence Number** the server is expecting next.
4.  **The Hijack:** The server receives the attacker's packet, believes it is the next piece of data from the legitimate
    client, and processes it. The attacker can now send commands (like "delete database" or "change password") as if they
    were the authorized user.

---

## 2. Types of Hijacking

### Blind Hijacking

The attacker **cannot see** the traffic between the client and server. They have to "guess" the next sequence number. While
difficult due to modern 32-bit sequence number randomization, it can still happen if the sequence generation algorithm is
weak or predictable.

### Active Hijacking

The attacker is "in the middle" (Man-in-the-Middle). They see the packets in real-time, modify them, and forward them. This
is the most effective form because the attacker doesn't have to guess; they simply read the Seq/Ack numbers off the wire.

---

## 3. The "ACK Storm" Problem

One side effect of a session hijack is often an **ACK Storm**.

- The server sends data to the attacker.
- The legitimate client receives this data too (since they share an IP in the eyes of the server).
- The client’s computer says, "Wait, this isn't the sequence number I expected!" and sends an ACK back to the server to try
  and resynchronize.
- The server receives the client's ACK, realizes _that_ doesn't match the attacker's sequence, and sends another ACK back.
- This creates a loop of rapid-fire ACK packets that can clog the network.

---

## 4. Prevention and Mitigation

| Defense                  | How it helps                                                                                                                                 |
| :----------------------- | :------------------------------------------------------------------------------------------------------------------------------------------- |
| **Encryption (TLS/SSL)** | This is the #1 defense. Even if an attacker hijacks the TCP layer, they cannot read or write the encrypted data inside without the SSL keys. |
| **IPSec**                | Authenticates and encrypts the packets at the IP layer, making spoofing nearly impossible.                                                   |
| **Randomized ISN**       | Modern Operating Systems use highly random Initial Sequence Numbers (ISN), making "Blind Hijacking" statistically impossible.                |
| **VPNs**                 | Wraps all traffic in an encrypted tunnel, hiding the TCP headers and sequence numbers from sniffers on the local network.                    |

---

### Summary

In a **TCP Session Hijack**, the attacker isn't knocking on the door (SYN Flood) or slamming it shut (Reset Attack); they are
waiting for you to unlock the door with your key, then shoving you out of the way and walking inside as if they were you.
