The **Transmission Control Protocol (TCP)** is one of the main protocols of the Internet Protocol Suite. If the Internet were
a postal service, TCP would be the certified mail—it ensures that every "letter" (data packet) arrives in the right order,
intact, and without anything missing.

---

## 1. Key Characteristics

TCP is defined by a few "golden rules" that make it reliable:

- **Connection-Oriented:** Before data flows, the sender and receiver must perform a "handshake" to establish a connection.
- **Reliability:** It uses acknowledgments (ACKs) to confirm receipt. If a packet is lost, TCP automatically resends it.
- **Ordered Data Transfer:** Packets may take different routes across the internet and arrive out of sequence. TCP
  reassembles them in the correct order before passing them to the application.
- **Flow & Congestion Control:** It prevents the sender from overwhelming the receiver or the network by adjusting the speed
  of data transmission.

---

## 2. The Three-Way Handshake

To start a conversation, TCP uses a three-step process to sync up. Think of it like a polite phone call:

1.  **SYN (Synchronize):** Client says, "I'd like to talk. Here is my starting sequence number."
2.  **SYN-ACK (Synchronize-Acknowledge):** Server says, "I hear you! I'm ready too. Here is my sequence number."
3.  **ACK (Acknowledge):** Client says, "Got it! Let's start sending data."

---

## 3. The TCP Header Structure

Every TCP packet has a "header" containing vital instructions. Since we are dealing with a technical protocol, we can look at
the mathematical logic of how data is tracked using sequence numbers ($Seq$) and acknowledgment numbers ($Ack$):

$$Next\_Expected\_Seq = Received\_Seq + Payload\_Size$$

| Field                     | Purpose                                                                  |
| :------------------------ | :----------------------------------------------------------------------- |
| **Source/Dest Ports**     | Directs data to the right application (e.g., Port 80 for HTTP).          |
| **Sequence Number**       | Keeps track of every byte sent so data can be reordered.                 |
| **Acknowledgment Number** | Tells the sender which byte the receiver expects next.                   |
| **Window Size**           | Indicates how much data the receiver can handle before needing a break.  |
| **Checksum**              | A mathematical check to ensure the data wasn't corrupted during transit. |

---

## 4. When to Use TCP (vs. UDP)

TCP is "heavy" because of all the checking it does. It’s perfect for when accuracy is more important than raw speed.

- **Best for TCP:** Web browsing (HTTP/HTTPS), Email (SMTP), File transfers (FTP), and Secure Shell (SSH). You wouldn't want
  a single line of a bank transfer or an email to go missing.
- **Not for:** Live video streaming or gaming, where a tiny bit of data loss is fine, but a delay (lag) caused by resending
  packets would ruin the experience. Those typically use **UDP (User Datagram Protocol)**.

---

## 5. Terminating the Connection

When the data transfer is finished, TCP uses a four-step "Four-Way Handshake" involving **FIN** (Finish) and **ACK** packets
to gracefully close the connection from both sides, ensuring no data is cut off prematurely.
