If your server is currently under attack, you need to move quickly from **Identification** to **Mitigation**. Every second
counts because your goal is to filter the "bad" traffic while keeping the "good" traffic flowing.

Here is a step-by-step emergency response plan:

---

## 1. Immediate Triage (The "Stop the Bleed" Phase)

Before you can fix it, you need to see what kind of attack it is.

- **Identify the Attack Type:** Check your server logs or monitoring tools (like `htop`, `nload`, or `netstat`).
  - **High CPU/RAM but low bandwidth?** It’s likely an **Application Layer (L7)** attack (e.g., HTTP flood).
  - **Bandwidth pipe is 100% full?** It’s a **Volumetric** attack (e.g., UDP/DNS amplification).
  - **Thousands of "SYN_RECV" connections?** It’s a **SYN Flood**.
- **Enable "Under Attack" Mode:** If you use a CDN like **Cloudflare, Akamai, or AWS Shield**, log in immediately and toggle
  the "I'm Under Attack" button. This forces all visitors to pass a challenge (like a JS check or Captcha) before hitting
  your server.

---

## 2. Server-Level Mitigation

If you don't have a specialized DDoS protection service, you'll have to use your server's built-in tools.

### For SYN Floods (Protocol Attacks)

Enable **SYN Cookies** immediately to prevent your connection table from filling up:

```bash
sudo sysctl -w net.ipv4.tcp_syncookies=1
```

### For IP-Based Floods

If you see a specific range of IPs or a single IP slamming your server in your logs, block them using `iptables` or `ufw`:

```bash
# Block a specific IP
sudo ufw deny from 1.2.3.4
# Block an entire subnet
sudo iptables -A INPUT -s 1.2.3.0/24 -j DROP
```

### Rate Limiting (Nginx/Apache)

If it's an HTTP flood, limit how many requests a single IP can make per second. **In Nginx (`nginx.conf`):**

```nginx
limit_req_zone $binary_remote_addr zone=mylimit:10m rate=5r/s;
```

---

## 3. Network-Level Response

If the attack is large enough (Volumetric), your server's firewall won't help because the "pipe" leading to your server is
already clogged.

- **Contact your ISP/Hosting Provider:** Tell them you are under a DDoS attack. Large providers (like AWS, DigitalOcean, or
  Linode) have "Blackhole Routing" where they can drop traffic at the edge of their network before it reaches your virtual
  machine.
- **Change your IP Address:** If the attack is hitting your direct IP, you might need to provision a new IP and hide it
  behind a proxy/CDN.

---

## 4. Post-Attack Hardening (Prevention)

Once the traffic subsides, you need to ensure this doesn't happen again.

| Defense Layer      | Solution                                                                                               |
| :----------------- | :----------------------------------------------------------------------------------------------------- |
| **DNS**            | Use a provider that masks your "Origin IP" so attackers can't find your server's real address.         |
| **Infrastructure** | Scale your resources. Use Load Balancers to distribute traffic across multiple servers.                |
| **Monitoring**     | Set up alerts for high PPS (Packets Per Second) or sudden spikes in bandwidth.                         |
| **WAF**            | Implement a **Web Application Firewall** to filter out malicious patterns and known botnet signatures. |

---

## Summary Checklist

1.  **Don't Panic:** Check your logs to see if it's Volumetric (Pipe full) or Application (CPU high).
2.  **Toggle CDN protection** (Cloudflare/AWS) if you have it.
3.  **Enable SYN Cookies** and **Rate Limiting** on the server.
4.  **Contact your host** if the bandwidth is totally saturated.
5.  **Analyze logs** after the attack to update your firewall rules.
