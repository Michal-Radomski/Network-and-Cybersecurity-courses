In TCP (Transmission Control Protocol), buffers are temporary storage areas in your computer's memory (RAM) used to manage
the flow of data between an application and the network. Think of them as **waiting rooms** that help bridge the speed gap
between how fast an app can process data and how fast the network can actually carry it.

---

## 1. The Send Buffer

When an application (like a web browser or a file transfer tool) wants to send data, it doesn't just shove it onto the wire
immediately. Instead, it hands the data over to the **Send Buffer**.

- **The Waiting Game:** Data stays here until the TCP stack can package it into segments and send them.
- **Reliability:** TCP is "reliable," meaning it needs to make sure the data actually arrived. The Send Buffer keeps a copy
  of the data even after it's been sent.
- **The Release:** Only after the receiver sends an **ACK** (Acknowledgment) saying "I got it!" is the data cleared from the
  Send Buffer to make room for more.

---

## 2. The Receive Buffer

On the other end of the connection, the **Receive Buffer** collects incoming data packets from the network.

- **Reassembly:** Data packets often arrive out of order. The Receive Buffer holds onto them and reorders them correctly so
  the application sees a continuous, logical stream.
- **The Buffer "Gap":** If the network is faster than the application (e.g., your internet is fast but your CPU is bogged
  down), the Receive Buffer fills up.
- **Flow Control:** This buffer is the basis for **Flow Control**. If the Receive Buffer is almost full, TCP sends a message
  to the sender saying, "Hey, slow down! My window size is shrinking," preventing the sender from overwhelming the receiver.

---

## Key Differences at a Glance

| Feature               | Send Buffer                                          | Receive Buffer                                    |
| :-------------------- | :--------------------------------------------------- | :------------------------------------------------ |
| **Location**          | The Sender's side.                                   | The Receiver's side.                              |
| **Primary Job**       | Retaining data for potential retransmission.         | Reordering and storing data for the app.          |
| **Cleared when...**   | An ACK is received from the peer.                    | The application "reads" the data.                 |
| **If it fills up...** | The application "blocks" (pauses) until space opens. | The sender is told to stop sending (Zero Window). |

### Why do we need them?

Without these buffers, you would have a massive "timing" problem. If the sender transmitted data and it was lost, there would
be no copy to resend. Similarly, if the receiver's application was busy for a split second, the incoming data would simply be
dropped and lost forever. Buffers provide the **elasticity** needed for a smooth, reliable conversation over an unpredictable
network.
