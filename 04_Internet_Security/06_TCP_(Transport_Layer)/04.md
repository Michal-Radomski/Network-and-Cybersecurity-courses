To understand how TCP manages data, you can think of it as a **smart courier service**. It doesn't just deliver packages; it
ensures they are in the right order, makes sure the recipient isn't overwhelmed, and checks if the "roads" (the network) are
blocked.

---

## 1. Sequence Numbers (The "Page Numbers")

TCP treats data as a continuous stream of bytes. Since the internet is "packet-switched," a single file is chopped into many
small segments. These segments can take different paths and arrive out of order.

- **How it works:** Each byte in the stream is assigned a **Sequence Number**. If you send a 1000-byte file and each packet
  holds 100 bytes, the first packet is labeled `Seq=1`, the second `Seq=101`, and so on.
- **The Receiver's Job:** It uses these numbers to "re-assemble" the data. If it receives `Seq=201` before `Seq=101`, it puts
  `201` in the **Receive Buffer** and waits for the missing piece.
- **Acknowledgments (ACKs):** The receiver sends back an ACK number, which tells the sender: _"I have everything up to byte
  X, please send me byte X+1 next."_

---

## 2. Flow Control (The "Bucket Limit")

Flow control is a protective mechanism for the **Receiver**. It prevents a fast sender from "drowning" a slow receiver by
sending data faster than the receiver's application can process it.

- **The Receive Window (RWND):** In every ACK packet, the receiver includes a "Window Size." This is literally the amount of
  free space currently available in its **Receive Buffer**.
- **The Rule:** The sender is strictly forbidden from sending more data than the current Window Size allows.
- **The Result:** If the receiver's app slows down and the buffer fills up, the Window Size drops to **0**. The sender stops
  immediately and waits for a "Window Update" before sending more.

---

## 3. Congestion Control (The "Traffic Jam")

While Flow Control protects the _receiver_, Congestion Control protects the **Network**. If too many computers send too much
data at once, routers get overwhelmed and start dropping packets.

TCP uses four main algorithms to handle this:

1.  **Slow Start:** The sender starts by sending a tiny amount of data. If it gets an ACK, it doubles the amount. It keeps
    doubling until it hits a limit or loses a packet.
2.  **Congestion Avoidance:** Once a certain threshold is hit, the sender stops doubling and starts increasing the amount
    very slowly (linearly) to "feel out" the network's capacity.
3.  **Fast Retransmit:** If the sender receives three identical ACKs for the same packet, it assumes that packet was lost and
    resends it immediately without waiting for a timer to expire.
4.  **Fast Recovery:** After a packet loss, instead of dropping all the way back to a "Slow Start," TCP cuts its speed in
    half and tries to build back up.

---

## Summary: The Difference

| Concept                | Who are we protecting? | Key Mechanism                      |
| :--------------------- | :--------------------- | :--------------------------------- |
| **Sequence Numbers**   | The Data Integrity     | Byte numbering and ACKs.           |
| **Flow Control**       | The Receiver           | **RWND** (Receive Window size).    |
| **Congestion Control** | The Network            | **CWND** (Congestion Window size). |

> **Pro Tip:** The sender actually looks at _both_ the RWND and the CWND and chooses the **smaller** of the two as its limit.
> This ensures it respects both the receiver's memory and the network's bandwidth.
