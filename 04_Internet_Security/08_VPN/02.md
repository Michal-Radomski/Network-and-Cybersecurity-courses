At its core, an **SSH Tunnel** (or SSH Port Forwarding) is a method of transporting arbitrary networking data over an
encrypted SSH connection. Think of it as a secure, private pipe nested inside a public or untrusted network connection.

### How It Works

When you create a tunnel, you tell your SSH client to listen on a specific port. Any traffic sent to that port is "wrapped"
in encryption, sent through the SSH server, and then "unwrapped" on the other side to be delivered to its final destination.

---

### The Three Main Types

| Type                        | Direction                     | Common Use Case                                                                    |
| --------------------------- | ----------------------------- | ---------------------------------------------------------------------------------- |
| **Local Port Forwarding**   | Client $\rightarrow$ Server   | Accessing a database on a remote server that isn't exposed to the internet.        |
| **Remote Port Forwarding**  | Server $\rightarrow$ Client   | Letting a friend access a web server running on your laptop behind a firewall.     |
| **Dynamic Port Forwarding** | Client $\rightarrow$ Multiple | Using your SSH server as a SOCKS proxy to browse the web securely on public Wi-Fi. |

---

### 1. Local Port Forwarding (`-L`)

This is the most common form. It forwards a port from your local machine to the remote server, which then connects to a
destination.

- **Command Structure:** `ssh -L local_port:destination_host:destination_port user@ssh_server`
- **Example:** You want to use a GUI tool on your laptop to manage a MySQL database running on a remote server that only
  allows local connections.
- `ssh -L 3306:localhost:3306 user@remote-server`
- Now, pointing your tool to `localhost:3306` actually connects you to the remote database.

### 2. Remote Port Forwarding (`-R`)

This "inverts" the connection. It allows the remote server to access a port on your local machine.

- **Command Structure:** `ssh -R remote_port:destination_host:destination_port user@ssh_server`
- **Example:** You are developing a website on your local machine (`localhost:8080`) and want to show a client. If you have
  an SSH server in the cloud, you can forward a port on that cloud server back to your laptop.

### 3. Dynamic Port Forwarding (`-D`)

Instead of forwarding a single port, this turns your SSH client into a **SOCKS proxy server**.

- **Command Structure:** `ssh -D local_port user@ssh_server`
- **Example:** If you’re on sketchy hotel Wi-Fi, you can run `ssh -D 8080 user@home-server`. If you configure your browser to
  use `localhost:8080` as a SOCKS proxy, all your web traffic is tunneled through your home connection, hiding your activity
  from the hotel.

---

### Why Use It?

- **Security:** It adds a layer of strong encryption to protocols that are normally unencrypted (like VNC or old HTTP sites).
- **Bypassing Firewalls:** If a corporate firewall blocks a certain site but allows SSH, you can tunnel your traffic out.
- **Remote Access:** It provides a way to reach services on private networks without needing a full-blown VPN.

> **Note:** While powerful, SSH tunneling is often monitored by network administrators because it can be used to bypass
> security policies. Always ensure you have permission to "tunnel" out of a restricted network!
