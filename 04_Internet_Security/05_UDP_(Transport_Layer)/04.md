The **User Datagram Protocol (UDP)** is one of the core members of the Internet Protocol Suite. Often referred to as a
"connectionless" protocol, it sits at the Transport Layer and provides a way for applications to send messages (datagrams)
with the absolute minimum of overhead.

Unlike its counterpart, TCP, UDP doesn't care about handshakes, acknowledgments, or whether the data actually arrives. It's
essentially the "fire and forget" method of networking.

---

## The UDP Header

The UDP header is famous for its simplicity. While a TCP header is at least 20 bytes, a UDP header is consistently only **8
bytes**.

### 1. Source Port (16 bits)

Identifies the sender's port. It is optional; if the client doesn't expect a reply, this can be set to zero.

### 2. Destination Port (16 bits)

Identifies the receiving process or service on the destination machine (e.g., DNS uses port 53).

### 3. Length (16 bits)

Specifies the length in bytes of the entire UDP segment (header + data). The minimum length is 8 bytes (just the header).

### 4. Checksum (16 bits)

Used for error-checking the header and data. In IPv4, this is optional (though highly recommended), while in IPv6, it is
mandatory. If the checksum fails, the receiver simply drops the packet without asking for a retransmission.

---

## Key Characteristics of UDP

UDP is designed for speed and efficiency rather than reliability. Here is how it functions in practice:

- **Connectionless:** There is no "setup" phase. Data is sent immediately without a three-way handshake.
- **Unreliable Delivery:** It does not guarantee that packets will arrive, nor does it ensure they arrive in the correct
  order.
- **No Flow or Congestion Control:** UDP will pump data into the network as fast as the application allows, even if the
  network is congested.
- **Low Latency:** Because there is no overhead for retransmission or sequencing, it is significantly faster than TCP for
  real-time applications.

---

## When is UDP Used?

If UDP is "unreliable," why use it? It turns out that for some tasks, speed is more important than perfect accuracy.

| Use Case             | Why UDP?                                                                                          |
| :------------------- | :------------------------------------------------------------------------------------------------ |
| **Streaming & VoIP** | A split-second glitch in audio is better than the entire call freezing to wait for a lost packet. |
| **Gaming**           | Real-time position updates must be instant; old data is useless.                                  |
| **DNS**              | Requests are small and need to be lightning-fast. If it fails, the browser just tries again.      |
| **DHCP**             | Used for assigning IP addresses when a full connection hasn't been established yet.               |

## Comparison: UDP vs. TCP

To visualize the difference, think of **TCP** as a registered letter that requires a signature, and **UDP** as a postcard
thrown into a mailbox.

| Feature         | UDP                 | TCP                      |
| :-------------- | :------------------ | :----------------------- |
| **Reliability** | Unreliable          | Reliable                 |
| **Ordering**    | No guaranteed order | Ordered sequences        |
| **Speed**       | Very Fast           | Slower (due to overhead) |
| **Header Size** | 8 Bytes             | 20-60 Bytes              |
| **Data Flow**   | Datagram-based      | Stream-based             |
