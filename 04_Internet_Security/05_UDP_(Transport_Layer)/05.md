A **UDP (User Datagram Protocol) flood** is a type of Denial-of-Service (DoS) or Distributed Denial-of-Service (DDoS) attack
that overwhelms a target server’s ability to process and respond to traffic.

Unlike TCP, UDP is a **connectionless** protocol. This means it doesn't require a "handshake" to verify the connection,
making it incredibly easy for attackers to send a massive volume of packets without waiting for a response.

---

## How the Attack Works

The goal is to exhaust the target’s resources—specifically its bandwidth and its CPU power. Here is the step-by-step process:

1.  **Packet Storm:** An attacker sends a large number of UDP packets to random ports on the target host.
2.  **Verification:** For every packet received, the target server checks if an application is listening on that specific
    port.
3.  **The "ICMP Destination Unreachable" Response:** When the server realizes no application is listening, it consumes
    resources to generate and send back an **ICMP (Internet Control Message Protocol)** packet notifying the sender that the
    destination was unreachable.
4.  **Resource Exhaustion:** As the volume of incoming UDP packets increases, the server becomes so busy checking ports and
    sending ICMP responses that it can no longer handle legitimate traffic, eventually crashing or becoming unresponsive.

---

## Common Types of UDP Attacks

### 1. UDP Reflection/Amplification

This is a more sophisticated version where the attacker "reflects" traffic off a third-party server (like a DNS or NTP
server).

- **The Trick:** The attacker spoofs the source IP address of the packets to be the **victim's IP**.
- **The Result:** The third-party server sends its large response directly to the victim. Because these responses are often
  much larger than the initial request, the attack is "amplified," allowing a small botnet to generate massive traffic.

### 2. Fraggle Attack

A legacy version of a UDP attack where an attacker sends a large number of UDP echo packets to an IP broadcast address, using
a spoofed source IP (the victim’s). Every device on that broadcast network then sends a response to the victim.

---

## Why UDP Attacks Are Dangerous

- **Easy to Spoof:** Since there is no handshake, it is trivial for attackers to fake the "Sender" IP address.
- **Hard to Filter:** Legitimate services like DNS, VoIP, and online gaming rely on UDP, so blocking all UDP traffic isn't an
  option for most businesses.
- **Bandwidth Saturation:** These attacks often focus on "filling the pipe," making it impossible for any traffic (UDP or
  otherwise) to reach the server.

---

## Mitigation Strategies

Defending against UDP floods requires a multi-layered approach:

| Strategy                     | Description                                                                                                                 |
| :--------------------------- | :-------------------------------------------------------------------------------------------------------------------------- |
| **Rate Limiting**            | Limiting the number of ICMP responses the server can send per second to conserve CPU.                                       |
| **Firewalls/ACLs**           | Configuring network firewalls to drop suspicious UDP packets or block non-essential ports.                                  |
| **Anycast Redundancy**       | Using a network of servers (Anycast) to spread the attack traffic across many locations so no single server is overwhelmed. |
| **DDoS Protection Services** | Using third-party "scrubbing centers" that filter out malicious traffic before it ever reaches your network.                |
