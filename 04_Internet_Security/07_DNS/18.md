A **DNS Rebinding Attack** is a clever and sophisticated security exploit that bypasses the browser's **Same-Origin Policy
(SOP)**.

While most DNS attacks focus on stealing traffic, DNS rebinding is designed to turn your web browser into a "proxy" that an
attacker can use to communicate with private devices inside your home or office network (like a router, a smart thermostat,
or an internal server).

---

## The Concept: Bypassing the Same-Origin Policy

The **Same-Origin Policy** is a fundamental web security rule: a script on `evil-attacker.com` is not allowed to read data
from `your-bank.com` or your local router at `192.168.1.1`.

DNS Rebinding tricks the browser into thinking that the attacker’s server and your private device are actually the **same
origin**.

---

## How the Attack Works

### 1. The Bait

You visit a malicious website (or an innocent site with a malicious ad). The browser asks for the IP address of
`attacker.com`.

### 2. The Short-Lived Response

The attacker’s DNS server responds with its own real IP address (e.g., `54.1.2.3`), but it sets the **TTL (Time to Live)** to
a very short duration, such as **1 second**. This tells the browser not to cache the address for long.

### 3. The Malicious Script

The website loads a script that waits for a few seconds. After the TTL expires, the script makes a new request back to
`attacker.com`.

### 4. The Switch (Rebinding)

Because the previous DNS record expired, the browser asks the DNS server for the IP again. This time, the attacker’s server
responds with a **private IP address** (e.g., `192.168.1.1`, your router's login page).

### 5. The Payload

The browser thinks, _"I'm still talking to attacker.com, so it's safe."_ It allows the script to send commands to
`192.168.1.1`. The script can now steal your router settings, access files on your NAS, or control smart home devices,
sending that private data back to the attacker.

---

## Why Is It So Dangerous?

- **Invisible to Firewalls:** Since the request originates from **inside** your network (your browser), the firewall sees it
  as legitimate outgoing traffic.
- **Bypasses Authentication:** Many IoT devices or internal tools don't require passwords if the request comes from the
  "local" network, assuming the environment is safe.
- **No Interaction Needed:** The user just has to keep the malicious tab open for a few seconds.

---

## Real-World Examples

- **Stealing Crypto:** Attackers have used rebinding to access local Ethereum wallets that expose an RPC interface on
  localhost.
- **Smart Home Control:** Researchers have demonstrated using rebinding to track users via Google Home or Roku devices by
  querying their internal APIs.

---

## How to Protect Yourself

| Target                 | Prevention Method                                                                                                                                                        |
| ---------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------ |
| **For Developers**     | Use **Host Header Validation**. Servers should check if the `Host` header in the HTTP request matches their expected IP or domain. If it says `attacker.com`, reject it. |
| **For Network Admins** | Enable **DNS Rebinding Protection** (standard on most modern routers and firmware like OpenWRT/DD-WRT). This blocks DNS responses that contain private IP addresses.     |
| **For Users**          | Keep your browser updated and ensure your local IoT devices have strong, unique passwords—never rely on "local network" trust.                                           |

---

> **Fun Fact:** This attack essentially exploits the fact that "Trust" in networking is often based on IP addresses, while
> "Trust" in browsers is based on Domain Names. The Rebinding attack is the wedge driven between those two definitions.
