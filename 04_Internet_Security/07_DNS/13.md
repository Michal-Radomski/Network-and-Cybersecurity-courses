A **DNS Local Cache Poisoning attack** (also known as DNS Spoofing) occurs when an attacker manages to insert a fraudulent
DNS entry into a local cache. Once the cache is "poisoned," any user on that system or network who tries to visit a
legitimate site (like `bank.com`) is silently redirected to a malicious IP address controlled by the attacker.

Since the browser believes the IP address came from a trusted source, the user has no way of knowing they are being diverted.

---

## How the Attack Works

To successfully poison a cache, an attacker must "win a race" against the legitimate DNS server.

### 1. The Setup

The attacker waits for a user to request a domain that isn't currently in their local cache. Alternatively, the attacker can
trigger the request themselves if they are on the same network.

### 2. The Request

The user's computer sends a query to the **Recursive Resolver** (or the local `systemd-resolved` cache) asking: _"What is the
IP for example.com?"_

### 3. The Race (The Attack)

The Resolver goes out to the internet to find the answer. While it is waiting, the attacker sends a **fake DNS response** to
the Resolver. For this to work, the attacker must correctly guess two things:

- **The Transaction ID:** A 16-bit number in the DNS header.
- **The Port Number:** The UDP port the Resolver is using to listen for the answer.

### 4. The Poisoning

If the fake response arrives **before** the real response from the authoritative server, the Resolver accepts the fake data
as truth. It then:

1. **Caches the fake IP** (e.g., pointing `google.com` to `6.6.6.6`).
2. **Sends the fake IP** back to the user’s browser.
3. **Discards the real response** when it finally arrives, because the "case is already closed."

---

## What it looks like in the Cache

If you were to dump your Ubuntu cache (using the `killall -USR1` method we discussed) during an attack, you would see a
discrepancy:

**Normal Entry:** `example.com  IN  A  93.184.216.34  (TTL 3600)`

**Poisoned Entry:** `example.com  IN  A  192.168.1.50  (TTL 86400)` _(Note: Attackers often set a very high TTL so the fake
entry stays in your cache for days.)_

---

## Types of Poisoning

### Local/LAN Poisoning (ARP Spoofing)

On a local Wi-Fi network, an attacker can use **ARP Spoofing** to tell your computer that _they_ are the router. Once all
your traffic flows through their laptop, they can intercept your DNS queries and instantly provide fake answers.

### Remote Poisoning (Kaminsky Attack)

Historically, attackers used a method called the **Kaminsky Attack** to poison large ISP resolvers. They would query
thousands of non-existent subdomains (like `1.google.com`, `2.google.com`) to force the resolver to make constant requests,
giving the attacker thousands of chances to "guess" the Transaction ID and inject a fake record for the main domain.

---

## How to Prevent It

Modern DNS has several layers of defense against poisoning:

- **Source Port Randomization:** Resolvers now use random ports for every query, making it much harder for an attacker to
  guess where to send the fake response.
- **DNSSEC (DNS Security Extensions):** This is the ultimate fix. It adds a digital signature to DNS records. If an attacker
  injects a fake IP, the signature won't match, and the Resolver will reject the packet.
- **DoH/DoT (DNS over HTTPS/TLS):** By encrypting the DNS query, an attacker on the local network cannot even see that you
  are making a request, let alone forge a response.
