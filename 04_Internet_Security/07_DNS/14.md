To see if your Ubuntu system is actively using **DNSSEC** to protect you from cache poisoning, you can use the `resolvectl`
tool.

When DNSSEC is active, `systemd-resolved` will cryptographically verify the signatures of the DNS records it receives. If an
attacker tries to "poison" the cache with a fake IP, the signature will be invalid, and the system will simply refuse to
resolve the domain.

---

## 1. Check if DNSSEC is Enabled

Run this command to see the global and per-interface DNS settings:

```bash
resolvectl status

```

Look for the **DNSSEC** line under your active network interface (like `enp3s0` or `wlp2s0`).

- **`+DNSSEC`**: It is supported and active.
- **`allow-downgrade`**: It will use DNSSEC if the server supports it, but will fall back to regular DNS if it doesn't (this
  is the common default).
- **`no`**: DNSSEC is disabled.

---

## 2. Test the Validation in Real-Time

You can test if your system correctly identifies a "poisoned" or "bad" record by querying a domain specifically designed to
fail DNSSEC validation.

Run this command:

```bash
resolvectl query fail01.dnssec-failed.org

```

**What to look for:**

- **If it is working:** The command should return an error like `Data is not authenticated` or `SERVFAIL`. This means your
  system caught the "fake" data and blocked it.
- **If it returns an IP:** It means your system is ignoring DNSSEC signatures and would be vulnerable to a poisoning attack.

---

## 3. How to Enable DNSSEC on Ubuntu

If you found that DNSSEC is "no" or not working, you can enable it manually.

1. **Open the configuration file:**

```bash
sudo nano /etc/systemd/resolved.conf

```

2. **Find the line `#DNSSEC=no**`(or`allow-downgrade`) and change it to:

```text
DNSSEC=yes

```

3. **Save and Restart the service:**

```bash
sudo systemctl restart systemd-resolved

```

---

## Summary: The Layers of Defense

Cache poisoning is much harder today than it was ten years ago. Your Ubuntu system uses a "Defense in Depth" strategy:

| Defense Layer                 | Purpose                                                                                          |
| ----------------------------- | ------------------------------------------------------------------------------------------------ |
| **Source Port Randomization** | Makes it statistically nearly impossible for an attacker to "guess" where to send a fake packet. |
| **Transaction ID (16-bit)**   | A unique "handshake" number that must match between query and response.                          |
| **DNSSEC**                    | Uses public-key cryptography to ensure the data hasn't been tampered with.                       |
| **DoH (DNS over HTTPS)**      | Encrypts the entire conversation so local attackers can't even see the query to spoof it.        |
