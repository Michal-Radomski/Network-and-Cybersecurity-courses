A DNS packet is a binary message usually sent over **UDP** (User Datagram Protocol) on **Port 53**. It has a very rigid
structure to ensure that different types of computers can read it instantly.

Every DNS packet—whether it is a **Question** from your browser or an **Answer** from a server—follows the exact same format.

---

## The DNS Packet Structure

A DNS packet is divided into five distinct sections:

| Section        | Description                                                       |
| -------------- | ----------------------------------------------------------------- |
| **Header**     | Information about the query (ID, flags, counts).                  |
| **Question**   | What you are looking for (e.g., "What is the IP of google.com?"). |
| **Answer**     | The actual resource records (the IPs) responding to the question. |
| **Authority**  | Points toward the authoritative nameservers.                      |
| **Additional** | Extra helpful info (like the IP addresses of the nameservers).    |

---

## 1. The Header (12 Bytes)

The header is the "envelope" of the packet. It tells the receiver how to process the rest of the data.

- **ID (Transaction ID):** A 16-bit identifier assigned by the client to match replies with queries.
- **Flags:** This includes:
- **QR:** Is this a Query (0) or a Response (1)?
- **AA:** Authoritative Answer flag.
- **RD:** Recursion Desired (tells the server to go find the answer if it doesn't have it).

- **Counts:** Four 16-bit fields that tell the receiver how many entries are in the following sections (e.g., "I have 1
  question and 2 answers").

## 2. The Question Section

This contains the specific request.

- **Name:** The domain name (e.g., `google.com`). In binary, this is encoded as labels (e.g., `6google3com0`).
- **Type:** The record type being requested (e.g., **A** for IPv4, **AAAA** for IPv6, **MX** for mail).
- **Class:** Almost always **IN** for "Internet."

## 3. The Resource Record Sections (Answer, Authority, Additional)

These three sections use the same format to provide data back to the user.

- **Name:** The domain name the record belongs to.
- **Type & Class:** Matches the question.
- **TTL (Time To Live):** How many seconds this answer can be cached before it must be deleted.
- **Data Length:** How big the actual data is.
- **RDATA:** The actual "payload"—the IP address itself.

---

## How to see a real DNS packet

You don't have to imagine what this looks like. You can see the "human-readable" version of a packet using the `dig` command
on Ubuntu.

Run this:

```bash
dig google.com

```

### Deciphering the `dig` output:

- **`id: 12345`**: That's your Transaction ID from the Header.
- **`flags: qr rd ra`**: These are your Header Flags.
- **`QUESTION SECTION`**: This is your Question.
- **`ANSWER SECTION`**: These are the Resource Records (IPs).

### Binary view (The "Real" Packet)

If you were to look at this in a packet sniffer like **Wireshark**, it looks like a string of Hexadecimal numbers.

> `7b 42 01 00 00 01 00 00 00 00 00 00 06 67 6f 6f 67 6c 65 03 63 6f 6d 00 00 01 00 01`

In this string, the `06` tells the computer that a 6-letter word ("google") is coming next!
