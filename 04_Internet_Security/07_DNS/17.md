While the name is the same, **Time to Live (TTL)** serves two completely different purposes depending on whether you are
talking about **DNS** (Data Longevity) or **Ping/IP Packets** (Distance/Safety).

---

## 1. TTL in DNS (The "Expiration Date")

In DNS, TTL is a timer that tells a resolver how long to **cache** a record before asking for a fresh one.

When a nameserver gives your computer an IP address, it essentially says: _"Here is the IP for google.com, and you can trust
this information for the next 3600 seconds."_

- **How it works:** Every second the record sits in your cache, the TTL count decreases. When it hits zero, the record is
  "stale," and the next time you visit that site, your computer must perform a new DNS query.
- **Why it exists:** It balances speed vs. accuracy.
- **High TTL (e.g., 86400s / 1 day):** Reduces traffic and speeds up loading (since it stays in cache), but if the website
  changes its IP address, users will be directed to the wrong place until the day is up.
- **Low TTL (e.g., 60s / 1 minute):** Allows for near-instant updates (great for security or switching servers), but puts a
  high load on DNS servers.

---

## 2. TTL in Ping / IP Packets (The "Hop Limit")

In networking (like when you use `ping` or `traceroute`), TTL is **not** a measurement of time. Instead, it is a counter of
**how many routers (hops)** a packet is allowed to pass through before it is destroyed.

- **How it works:** 1. Your computer sends a packet with a set TTL (often 64, 128, or 255).

2.  Every time the packet hits a router, that router subtracts **1** from the TTL.
3.  If the TTL hits **0**, the router drops the packet and sends an "ICMP Time Exceeded" message back to you.

- **Why it exists:** To prevent **Infinite Loops**. If there is a bug in the internet's routing tables, a packet could circle
  around forever, clogging up cables. TTL ensures that every packet eventually "dies" if it can't find its destination.

---

## Comparison Table

| Feature                 | DNS TTL                                  | Ping/IP TTL                                        |
| ----------------------- | ---------------------------------------- | -------------------------------------------------- |
| **Unit of Measurement** | **Seconds** (Time)                       | **Hops** (Number of Routers)                       |
| **Primary Purpose**     | Caching & Traffic Reduction              | Preventing Infinite Loops                          |
| **What happens at 0?**  | The record is deleted from cache.        | The packet is destroyed.                           |
| **Analoggy**            | The "Best Before" date on a milk carton. | A "Battery Life" that drains every mile you drive. |

## Seeing it in Action on Ubuntu

### Check DNS TTL:

Run `dig google.com`. Look at the number in the second column of the **ANSWER SECTION**.

```bash
google.com.    254    IN    A    142.250.190.46

```

_Here, the TTL is 254 seconds. If you run the command again, you'll see that number counting down._

### Check Ping TTL:

Run `ping 8.8.8.8`.

```bash
64 bytes from 8.8.8.8: icmp_seq=1 ttl=117 time=12.4 ms

```

_Google sent this packet with a TTL (likely 128). It passed through 11 routers to get to you ($128 - 11 = 117$)._

### A Fun Trick:

If you want to see exactly which routers are "eating" your TTL, run `traceroute`. It works by sending packets with a TTL of
1, then 2, then 3, deliberately forcing each router along the path to "kill" the packet and identify itself!
