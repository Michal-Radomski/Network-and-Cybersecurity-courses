A **Reverse DNS Lookup (rDNS)** is the process of resolving an IP address back to its associated domain name. While a
standard (forward) DNS lookup asks, "What is the IP for `google.com`?", a reverse lookup asks, "What domain is associated
with the IP `8.8.8.8`?"

This is accomplished using a specific type of DNS record called a **PTR (Pointer) Record**.

---

## How Reverse DNS Works

Reverse lookups utilize a specialized domain hierarchy. For IPv4, this is the `in-addr.arpa` domain.

1. **The IP Reversal:** To look up the IP `192.168.1.50`, the system reverses the octets to `50.1.168.192`.
2. **The Query:** It appends the special suffix to create `50.1.168.192.in-addr.arpa`.
3. **The Result:** The DNS server returns the PTR record, such as `mail.example.com`.

### Why use rDNS?

- **Email Spam Filtering:** Mail servers check if the sender's IP matches the domain they claim to be. If the rDNS fails or
  doesn't match, the email is often marked as spam.
- **Logging & Analytics:** Servers log IP addresses, but rDNS makes those logs human-readable (e.g., seeing
  `crawl-66-249-66-1.googlebot.com` instead of just an IP).
- **Network Troubleshooting:** Tools like `traceroute` use rDNS to show the names of the routers your data passes through.

---

## Reverse DNS Attacks

Attackers can exploit the trust systems place in rDNS results to perform reconnaissance or bypass security checks.

### 1. rDNS Spoofing (Bypassing Access Control)

Some legacy systems or simple "Allow Lists" verify users based on their domain name via rDNS.

- **The Attack:** An attacker controls their own IP space and sets the PTR record to something deceptive (e.g.,
  `trusted.internal.company.com`).
- **The Result:** If a server only checks the rDNS string to grant access, it might mistakenly trust the attacker.
- **Defense:** Always perform **Forward-Confirmed reverse DNS (FCrDNS)**. This means after you get the domain from the rDNS,
  you do a _forward_ lookup on that domain to see if it points back to the original IP.

### 2. Information Reconnaissance

rDNS is a goldmine for attackers during the "footprinting" phase of a hack.

- **The Attack:** Attackers perform "IP sweeping"—bulk reverse lookups on a range of a company's IP addresses.
- **The Result:** The PTR records often reveal the function of the servers (e.g., `dev-server.corp.com`,
  `finance-db-01.corp.com`, or `vpn-gateway.corp.com`). This helps the attacker pick the most vulnerable or valuable targets.

### 3. Log Poisoning / Terminal Injection

Because rDNS results are often written directly into system logs or displayed in admin dashboards:

- **The Attack:** An attacker sets a PTR record containing malicious scripts or terminal escape sequences (e.g.,
  `<script>alert('XSS')</script>`).
- **The Result:** When an administrator views the logs in a web-based tool or a terminal, the script might execute, leading
  to **Cross-Site Scripting (XSS)** or a messed-up terminal display.

## Summary Table

| Feature          | Forward DNS          | Reverse DNS (rDNS)  |
| ---------------- | -------------------- | ------------------- |
| **Input**        | Domain Name          | IP Address          |
| **Record Type**  | A or AAAA Record     | PTR Record          |
| **Zone**         | `example.com`        | `in-addr.arpa`      |
| **Primary Risk** | Phishing/Redirection | Recon/Access Bypass |

> **Pro-Tip:** If you are setting up a mail server, a valid PTR record is not optional—without it, your "delivery rate" will
> likely be near zero as other servers will treat you as a suspicious "unnamed" entity.
