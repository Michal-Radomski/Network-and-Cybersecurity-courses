The **Kaminsky Attack**, discovered by security researcher Dan Kaminsky in 2008, is one of the most famous and sophisticated
versions of **DNS Cache Poisoning**.

Before this attack was discovered, poisoning a cache was considered extremely difficult because an attacker had to "win a
race" against a legitimate DNS server. If the attacker lost the race, the real answer would be cached, and the attacker would
have to wait for the **TTL** (Time To Live) to expire (which could be hours or days) before trying again.

Kaminsky found a way to bypass the TTL entirely, allowing an attacker to try thousands of times per second until they
succeeded.

---

## How the Attack Works

### 1. The Strategy: Subdomain "Grinding"

Instead of trying to poison `google.com` directly, the attacker asks the target DNS resolver for a non-existent subdomain,
like `random123.google.com`.

- **The Resolver:** "I don't have that in my cache. Let me go ask the Authoritative Server for `google.com`."
- **The Race:** The resolver sends a query to the internet. The attacker now sends a flood of fake responses.

### 2. The "Infinite" Retries

If the attacker fails to guess the **Transaction ID** for `random123.google.com`, it doesn't matter! They immediately send a
request for `random456.google.com`, then `random789.google.com`.

Because each subdomain is unique, the resolver's cache doesn't have an entry for it. This forces the resolver to start a new
"race" every single time, giving the attacker thousands of chances to guess the ID in a matter of seconds.

### 3. The "Bait and Switch" (The Payload)

The genius of the Kaminsky attack isn't just about the subdomains. In the "Additional Section" of the fake DNS packet, the
attacker includes a record that says:

> _"By the way, the nameserver for the entire `google.com` domain has moved. It is now at the IP address [Attacker's IP]."_

If the resolver accepts the fake response for the junk subdomain, it **also** accepts the fake nameserver information.

### 4. The Result

The resolver's cache is now poisoned for the **entire domain**. Any future request for `google.com`, `mail.google.com`, or
`drive.google.com` will now go directly to the attacker's server.

---

## Why was it so dangerous?

Before this attack, DNS was considered "secure enough" because the odds of guessing a 16-bit Transaction ID
($2^{16} = 65,536$ possibilities) before a fast server responded were very low. Kaminsky proved that with enough subdomains,
you could brute-force those 65,536 possibilities in under a minute.

## How it was fixed: Source Port Randomization

The immediate fix (which you now see on your Ubuntu system) was to randomize the **Source Port**.

- **Before:** DNS queries always came from Port 53. An attacker only had to guess the 16-bit Transaction ID.
- **After:** DNS queries come from a random port (e.g., Port 49213). Now, an attacker has to guess the **Transaction ID AND
  the Port Number**.

This increased the entropy from $16$ bits to roughly $32$ bits ($2^{32} \approx 4 \text{ billion}$ possibilities), making the
"race" almost impossible to win by pure luck.

---

### The Final Shield: DNSSEC

As we discussed, even with randomized ports, a very determined attacker could technically still win. **DNSSEC** (which you
just enabled) is the only "perfect" fix because even if the attacker wins the race and gets their packet in first, the
**cryptographic signature** will be wrong, and your Ubuntu machine will drop the packet.
