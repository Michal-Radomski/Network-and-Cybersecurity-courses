The **Address Resolution Protocol (ARP)** is the "glue" that connects the Network Layer (Layer 3) to the Data Link Layer
(Layer 2).

While humans and software use **IP addresses** to identify devices, hardware only understands **MAC addresses**. ARP is the
mechanism used to find the MAC address of a device when only its IP address is known.

---

## 1. Why do we need ARP?

Imagine you want to send a letter to "Room 202" in a hotel. You know the room number (the IP), but you don't know the name of
the person staying there (the MAC). To deliver the message directly to their hand, you need to link that room number to a
specific person.

In a local network:

- **IP Addresses** are logical and can change.
- **MAC Addresses** are physical and unique to the hardware.
- **The Problem:** An Ethernet frame cannot be sent without a destination MAC address.

---

## 2. How the ARP Process Works

When a device (the "Source") wants to talk to another device (the "Destination") on the same network, it follows these steps:

### Step A: Check the ARP Cache

Before shouting for help, the device checks its own **ARP Table** (a local temporary memory) to see if it already knows the
MAC address for that IP.

### Step B: The ARP Request (The Shout)

If the address isn't in the cache, the source sends an **ARP Request**. This is a **broadcast** frame, meaning it is sent to
every single device on the local network.

> "Who has IP address 192.168.1.5? Tell 192.168.1.10."

### Step C: The ARP Reply (The Answer)

The device that actually owns that IP address sends back an **ARP Reply**. This is a **unicast** message (sent directly to
the requester).

> "I have 192.168.1.5! My MAC address is 00:AA:11:BB:22:CC."

### Step D: Update and Send

The source device saves this mapping in its ARP table for future use and finally encapsulates its data into an Ethernet frame
to start the actual communication.

---

## 3. Types of ARP

There are a few specialized versions of this protocol:

- **Proxy ARP:** When a router answers an ARP request on behalf of a device that is on another network.
- **Gratuitous ARP:** A device sends an ARP request for its _own_ IP address. This helps detect IP conflicts (if someone else
  answers, the IP is already taken) or updates other devices' caches after a hardware change.
- **Reverse ARP (RARP):** The opposite of ARP—used by a device to find its own IP address if it only knows its MAC address
  (mostly obsolete now, replaced by DHCP).

---

## 4. Security Note: ARP Spoofing

ARP was designed for efficiency, not security. It is "stateless," meaning a device will believe an ARP reply even if it never
asked for one.

In an **ARP Spoofing** (or Poisoning) attack, a malicious actor sends fake ARP replies to a switch. This tricks the network
into sending data meant for the Router to the Attacker instead, enabling "Man-in-the-Middle" attacks.

### Quick Summary Table

| Feature          | Description                                       |
| :--------------- | :------------------------------------------------ |
| **Layer**        | Layer 2 (Data Link), though it works with Layer 3 |
| **Function**     | Maps IP Address $\rightarrow$ MAC Address         |
| **Request Type** | Broadcast (to everyone)                           |
| **Reply Type**   | Unicast (to the requester)                        |
| **Storage**      | Stored in the ARP Cache/Table                     |

---

To view and manage your **ARP Table** (also known as the ARP Cache) on Ubuntu, you can use a few different commands depending
on which package you have installed.

### 1. The Modern Way (`ip` command)

The `iproute2` suite is the standard in modern Linux distributions. It provides a clean, easy-to-read output.

**Command:**

```bash
ip neighbor show
```

_Or the shorthand:_

```bash
ip neigh
```

**What the output looks like:**

- **REACHABLE:** The mapping is valid and has been confirmed recently.
- **STALE:** The mapping is still in the table, but the system needs to verify it the next time it sends data.
- **DELAY/PROBE:** The system is currently trying to verify the MAC address.

---

### 2. The Classic Way (`arp` command)

If you have `net-tools` installed, you can use the traditional `arp` command. If it's not installed, you can get it via
`sudo apt install net-tools`.

**Command:**

```bash
arp -a
```

**What the output looks like:** This displays the cache in a BSD-style format, listing the hostname (if known), IP address,
and MAC address (HWaddress) associated with each interface.

---

### 3. How to Clear the ARP Cache

Sometimes you need to force the system to "re-learn" the network (for example, if you suspect ARP Spoofing or a device
changed its NIC).

**To delete a single entry:**

```bash
sudo ip neighbor del [IP_ADDRESS] dev [INTERFACE]
```

_Example: `sudo ip neighbor del 192.168.1.5 dev eth0`_

**To flush the entire table:**

```bash
sudo ip neigh flush all
```

---

### Summary of Flags in the ARP Table

When looking at your table, you might see these headers:

| Field                 | Meaning                                                     |
| :-------------------- | :---------------------------------------------------------- |
| **Address/Host**      | The IP address of the neighbor device.                      |
| **HWaddress/MAC**     | The 48-bit physical address of that device.                 |
| **Interface (iface)** | Which network card (Ethernet, Wi-Fi) is seeing that device. |
