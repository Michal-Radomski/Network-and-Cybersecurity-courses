**ARP Cache Poisoning** (also known as ARP Spoofing) is a cyberattack where a malicious actor sends falsified ARP messages
onto a local area network. The goal is to link the attacker’s MAC address with the IP address of a legitimate device—usually
the default gateway (router)—causing traffic intended for that IP to be sent to the attacker instead.

---

### How the Attack Works

The attack exploits a fundamental design flaw in the ARP protocol: **it is stateless and trusts all replies.** Devices will
update their ARP cache even if they never sent a request for that information.

1.  **The Setup:** The attacker, a victim PC, and a Router are all on the same LAN.
2.  **The Poisoning:** The attacker sends "Gratuitous ARP" replies to both the victim and the router.
    - **To the Victim:** "I am the Router (IP), here is my MAC address (Attacker's MAC)."
    - **To the Router:** "I am the Victim (IP), here is my MAC address (Attacker's MAC)."
3.  **The Hijack:** Both the victim and the router update their ARP tables with the false information. Now, when the victim
    tries to go to the internet, the data goes to the attacker first.

---

### The Goals of the Attacker

Once the ARP cache is poisoned, the attacker can execute several types of attacks:

- **Man-in-the-Middle (MitM):** The attacker intercepts the data, reads or modifies it, and then forwards it to the real
  destination. The victim usually has no idea their traffic is being monitored.
- **Denial of Service (DoS):** The attacker can link multiple IP addresses to a single non-existent MAC address, effectively
  "dropping" all traffic intended for those IPs so they lose internet access.
- **Session Hijacking:** By capturing traffic, the attacker can steal unencrypted session cookies or login credentials to
  take over web accounts.

---

### Detection and Prevention

Because ARP happens at such a low level (Layer 2), standard firewalls often don't catch it. However, there are several ways
to defend against it:

#### 1. Static ARP Entries

You can manually map an IP to a MAC address so the computer ignores any broadcast updates. This is effective but a nightmare
to manage in large networks.

- _Ubuntu command:_ `sudo arp -s [IP] [MAC]`

#### 2. Dynamic ARP Inspection (DAI)

High-end network switches have a feature called **DAI**. The switch maintains a list of "trusted" MAC-to-IP bindings. If it
sees an ARP packet that doesn't match the list, it drops the packet before it can reach the rest of the network.

#### 3. ARP Monitoring Tools

Software like **Arpwatch** or **XArp** can monitor the network for "flapping" MAC addresses (where an IP address is suddenly
associated with a new MAC) and alert administrators immediately.

#### 4. Use Encryption (HTTPS/VPN)

While ARP spoofing lets an attacker _see_ the traffic, they can't read it if it is encrypted. Using a VPN or ensuring sites
use HTTPS protects the data even if the local connection is compromised.

### Summary Table: Normal vs. Poisoned

| Feature                  | Normal Traffic                                | Poisoned Traffic                                                     |
| :----------------------- | :-------------------------------------------- | :------------------------------------------------------------------- |
| **Victim's Gateway MAC** | Router's actual MAC                           | Attacker's MAC                                                       |
| **Data Path**            | Victim $\rightarrow$ Router $\rightarrow$ Web | Victim $\rightarrow$ Attacker $\rightarrow$ Router $\rightarrow$ Web |
| **Traffic Visibility**   | Private to the router                         | Visible to the attacker                                              |
| **Detection Difficulty** | N/A                                           | High (without specialized tools)                                     |
