At its core, **Netfilter** is a framework inside the Linux kernel that allows various network-related operations to be
implemented in the form of customized handlers. It is the "engine room" that makes features like firewalls, NAT (Network
Address Translation), and packet mangling possible.

If you’ve ever used `iptables` or `nftables`, you’ve used Netfilter—those are just the user-facing tools used to talk to the
Netfilter framework.

---

## How It Works: The Five Hooks

Netfilter operates by placing "hooks" at specific points along the networking stack. As a packet travels through the kernel,
it triggers these hooks, allowing Netfilter modules to inspect, drop, or modify the packet.

1. **PREROUTING:** Triggered as soon as a packet enters the network interface.
2. **INPUT:** Triggered if the packet is destined for the local system itself.
3. **FORWARD:** Triggered if the packet is destined for another host (the system acts as a router).
4. **OUTPUT:** Triggered when a packet is created locally and is about to go out.
5. **POSTROUTING:** Triggered just before a packet leaves the network interface.

---

## Key Components

Netfilter isn't just one piece of code; it’s an ecosystem of different subsystems:

### 1. Packet Filtering

This is the classic "firewall" function. It decides whether a packet should be allowed to pass (**ACCEPT**) or be blocked
(**DROP** or **REJECT**) based on rules like IP addresses or port numbers.

### 2. Connection Tracking (conntrack)

This allows Netfilter to be "stateful." Instead of looking at every packet in isolation, it remembers the state of
connections. It knows if a packet is part of an existing conversation or a brand-new (and potentially suspicious) request.

### 3. Network Address Translation (NAT)

Netfilter handles the rewriting of source or destination IP addresses. This is what allows your home router to let multiple
devices share a single public IP address.

### 4. Packet Mangling

This is used for specialized alterations to the packet header, such as changing the **TTL** (Time to Live) or marking packets
for specific routing behaviors (Quality of Service).

---

## Evolution: From `iptables` to `nftables`

For decades, `iptables` was the standard way to interact with Netfilter. However, it became complex and inefficient for
modern high-speed networking.

- **iptables:** Uses a rigid structure with different binaries for different protocols (e.g., `ip6tables`, `arptables`).
- **nftables:** The modern successor. It provides a more efficient virtual machine-like execution inside the kernel, a
  unified syntax for all protocols, and better performance by reducing code duplication.

---

## Why It Matters

Without Netfilter, Linux wouldn't be the powerhouse it is in the world of networking. It turns a standard Linux box into a
sophisticated enterprise-grade router and security gateway, providing the foundation for tools like **ufw**, **firewalld**,
and even the networking logic inside **Docker** and **Kubernetes**.
