To see the "memory" of your firewall in action on Ubuntu, you can use the `conntrack` tool. This interacts directly with the
Netfilter connection tracking subsystem.

### 1. Install the Utility

The tool isn't always installed by default. You can grab it with:

```bash
sudo apt update && sudo apt install conntrack

```

### 2. View Active Connections

To see every connection your kernel is currently tracking, run:

```bash
sudo conntrack -L

```

This will output a list of flows. Each line looks something like this:
`tcp 6 431999 ESTABLISHED src=192.168.1.5 dst=93.184.216.34 sport=54321 dport=443 src=93.184.216.34 dst=192.168.1.5 sport=443 dport=54321 [ASSURED] mark=0 use=1`

**How to read this output:**

- **`tcp`**: The protocol.
- **`431999`**: The timeout (how many seconds until this entry is deleted if no more packets are seen).
- **`ESTABLISHED`**: The current state of the connection.
- **The first `src/dst**`: The original direction of the packet.
- **The second `src/dst**`: What the firewall expects the reply to look like.

### 3. Live Monitoring

If you want to watch connections being created and destroyed in real-time (like a "live feed" of your firewall's brain), use:

```bash
sudo conntrack -E

```

---

## Managing the Table Limits

As mentioned, the connection tracking table has a maximum size. If you ever suspect your firewall is "full," you can check
the limits and current count:

- **Check current count:** `sysctl net.netfilter.nf_conntrack_count`
- **Check maximum limit:** `sysctl net.netfilter.nf_conntrack_max`

### Adjusting the Limit

If you are running a high-traffic server and seeing "table full" errors in `dmesg`, you can temporarily increase the limit:

```bash
sudo sysctl -w net.netfilter.nf_conntrack_max=131072

```

---

## Summary: Why this is powerful

By looking at this table, you can see exactly:

1. **Who** is connected to your machine.
2. **How long** they've been connected.
3. **Whether** the firewall considers the conversation "healthy" (`ESTABLISHED`) or suspicious.
