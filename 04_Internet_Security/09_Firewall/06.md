In `iptables`, **Tables** are the highest level of organization. Think of them as "departments" in a factory: each department
has a specific job (filtering, address changing, or modifying data), and packets are sent to the relevant department
depending on what needs to be done.

There are four primary tables you will encounter, plus one specialized table for security.

---

## 1. Filter Table (The Security Guard)

This is the **default** and most commonly used table. If you don't specify a table name in your command, `iptables` assumes
you mean `filter`.

- **Purpose:** To decide whether a packet should be allowed to reach its destination or be dropped.
- **Chains:** `INPUT`, `FORWARD`, and `OUTPUT`.

## 2. NAT Table (The Receptionist)

NAT stands for **Network Address Translation**. This table is used when a packet creates a new connection and needs its
source or destination address modified.

- **Purpose:** Routing packets to different networks or "hiding" private IP addresses behind a single public IP.
- **Chains:** \* `PREROUTING`: For changing the destination IP (DNAT).
- `POSTROUTING`: For changing the source IP (SNAT/Masquerading).
- `OUTPUT`: For NAT-ing locally generated packets.

## 3. Mangle Table (The Specialist)

The `mangle` table is used for specialized alteration of the packet's IP headers.

- **Purpose:** Changing things like **TTL** (Time to Live), **TOS** (Type of Service), or placing "marks" on packets so other
  tools (like traffic control) know how to handle them.
- **Chains:** All five hooks (`PREROUTING`, `INPUT`, `FORWARD`, `OUTPUT`, `POSTROUTING`).

## 4. Raw Table (The Fast Track)

The `raw` table is used mainly for configuring exemptions from **connection tracking** (`conntrack`).

- **Purpose:** It allows you to mark packets that should not be handled by the stateful tracking system. This is useful for
  high-traffic services (like heavily loaded DNS) to reduce CPU overhead.
- **Chains:** `PREROUTING` and `OUTPUT`.

## 5. Security Table (The Compliance Officer)

This is used for **Mandatory Access Control** (MAC) networking rules, typically in conjunction with **SELinux**.

- **Purpose:** To mark packets with security contexts that SELinux can interpret.
- **Chains:** `INPUT`, `FORWARD`, and `OUTPUT`.

---

## Comparison of Table Responsibilities

| Table        | Primary Role        | Common Use Case                                    |
| ------------ | ------------------- | -------------------------------------------------- |
| **Filter**   | Packet Filtering    | Blocking a specific IP or port.                    |
| **NAT**      | Address Translation | Port forwarding or sharing an internet connection. |
| **Mangle**   | Header Modification | Setting Quality of Service (QoS) bits.             |
| **Raw**      | Tracking Exemption  | Disabling connection tracking for speed.           |
| **Security** | SELinux Integration | Marking packets for mandatory access controls.     |

---

## The Order of Operations

Packets don't just hit these tables randomly; there is a strict **priority**. For example, at the `PREROUTING` hook, the
`raw` table is checked first, then `mangle`, then `nat`. This ensures that if you want to "un-track" a packet in the `raw`
table, it happens before the `nat` table tries to look for an existing connection.
