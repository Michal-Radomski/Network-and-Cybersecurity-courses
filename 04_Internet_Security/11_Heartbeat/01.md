In distributed systems and networking, a **Heartbeat Protocol** is a monitoring mechanism used to verify that a device (a
"node") is still functioning and reachable. Much like a biological pulse, it consists of periodic signals sent from one node
to another to indicate "I am still alive."

---

## How it Works

The protocol generally involves two roles: the **Sender** (the node being monitored) and the **Receiver** (the monitoring
service or peer node).

1. **The Signal:** At a predefined interval (e.g., every 500ms), the sender transmits a small packet of data to the receiver.
2. **The Timeout:** The receiver maintains a timer. If it does not receive a heartbeat within the expected window (plus a
   small grace period), it assumes the sender has failed.
3. **The Action:** Once a node is marked as "dead," the system triggers a recovery process, such as reassigning its tasks to
   a healthy node or removing it from a load balancer pool.

---

## Key Components

### 1. Heartbeat Interval

This is the frequency of the pulses.

- **Short intervals** detect failures faster but consume more network bandwidth and CPU.
- **Long intervals** reduce overhead but mean the system stays "broken" longer before anyone notices.

### 2. Timeout Threshold

To account for network jitter or temporary congestion, receivers usually wait for $n$ missed heartbeats before declaring a
failure. If the interval is $t$, the timeout $T$ is often calculated as:

$$T = n \times t + \delta$$

where $\delta$ represents network latency.

---

## Push vs. Pull Models

| Model              | Mechanism                                                            | Best For...                                                                           |
| ------------------ | -------------------------------------------------------------------- | ------------------------------------------------------------------------------------- |
| **Push (Common)**  | The node actively sends "I'm alive" messages to the monitor.         | Large-scale clusters; reduces load on the monitor.                                    |
| **Pull (Polling)** | The monitor sends a "Are you alive?" request, and the node responds. | Scenarios where the monitor needs to verify specific health metrics during the check. |

---

## Why is it used?

- **High Availability:** In a server cluster, if the primary database stops sending heartbeats, the secondary database can
  automatically take over (failover).
- **Load Balancing:** A load balancer uses heartbeats to ensure it isn't sending traffic to a crashed web server.
- **Cluster Membership:** In technologies like **Kubernetes** or **Apache Cassandra**, heartbeats help the system maintain an
  accurate list of which machines are currently part of the network.

### A Quick Note on the "Heartbleed" Bug

Don't confuse the **Heartbeat Protocol** with the **Heartbleed** vulnerability. Heartbleed was a specific security flaw in
OpenSSL's implementation of the TLS Heartbeat Extension, where an attacker could trick a server into leaking private memory
data. The protocol itself is a standard architectural pattern, while Heartbleed was a coding error in one specific version of
it.
