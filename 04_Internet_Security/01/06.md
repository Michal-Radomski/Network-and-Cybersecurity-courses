To secure your Node.js backend, you should configure it to listen only on the specific **Private IP** assigned to your server
(or internal network interface) rather than the public-facing one. This ensures that even if your firewall fails, the
application is physically unable to hear requests coming from the outside world.

---

## 1. The "Listen" Command

By default, many Node.js tutorials tell you to leave the host blank or use `0.0.0.0`.

- **`0.0.0.0`**: This tells Node to listen on **every** available network interface (Public IP, Private IP, and Localhost).
  This is the least secure setting.
- **`127.0.0.1`**: This listens only to requests coming from the same machine (Localhost).
- **Private IP (e.g., `10.0.0.5`)**: This listens only to requests coming from within your internal network (like your Load
  Balancer or other microservices).

### How to code it (Express.js)

Instead of just passing a port to `app.listen()`, pass the specific Private IP as the second argument:

```javascript
const express = require("express");
const app = express();

const PORT = 3000;
// Replace with your server's actual internal private IP
const PRIVATE_IP = "10.0.0.5";

app.listen(PORT, PRIVATE_IP, () => {
  console.log(`Server strictly listening on private network at http://${PRIVATE_IP}:${PORT}`);
});
```

---

## 2. Using Environment Variables

In a professional setup, you should never hardcode an IP address. Use a `.env` file so the app remains portable between your
local machine and the production server.

**`.env` file:**

```bash
HOST=10.0.0.5
PORT=3000
```

**`server.js` file:**

```javascript
const host = process.env.HOST || "127.0.0.1"; // Default to localhost for safety
const port = process.env.PORT || 3000;

app.listen(port, host, () => {
  console.log(`Running on ${host}:${port}`);
});
```

---

## 3. The 3-Tier Security Architecture

In 2026, the standard way to deploy a Node/React app securely involves three distinct layers using Private IPs:

1.  **Public Subnet:** Contains only a **Load Balancer**. It has a Public IP to talk to the world and a Private IP to talk to
    your servers.
2.  **Private Subnet (App Layer):** Your **Node.js servers** live here. They have **only** Private IPs. They can only hear
    the Load Balancer.
3.  **Data Subnet (Database Layer):** Your **Database** (MongoDB/Postgres) lives here. It has a Private IP and a strict Net
    Mask (like `/32`) that only allows connections from the App Layer's IPs.

---

## 4. Why this matters for SQLi/NoSQLi

Even if an attacker finds a way to exploit a NoSQL injection in your code, if your database is on a private subnet with no
public IP, they cannot "exfiltrate" the data to their own server as easily. They are trapped inside your private network
walls.

**Summary of your Secure Setup:**

- **React App:** Served via a CDN or Load Balancer (Public).
- **Node.js API:** Listening on a **Private IP** (Internal).
- **Database:** Listening on a **Private IP** with a `/32` mask (Locked to the API).
