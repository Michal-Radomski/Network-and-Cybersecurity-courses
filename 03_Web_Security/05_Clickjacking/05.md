In 2026, using iframes in a React application is still **common**, but the way they are used has shifted. They are no longer
the go-to for building layout sections; instead, they are specialized tools used for **security isolation** and **third-party
integration**.

While modern React development favors API-driven integrations or SDKs, iframes remain the standard for embedding content you
don't control, such as YouTube videos, Google Maps, or secure payment gateways (like Stripe).

---

## 1. Is it Secure?

Iframes are inherently a "double-edged sword." They provide a security boundary, but they also introduce specific
vulnerabilities if not configured correctly.

### The Risks

- **Clickjacking:** An attacker could embed your site in an invisible iframe on their own site to trick users into clicking
  buttons (e.g., "Delete Account").
- **Cross-Site Scripting (XSS):** If the content inside the iframe is malicious, it could attempt to steal data from the
  parent site or perform actions on the user's behalf.
- **Information Leakage:** Without strict policies, an iframe might have access to the user's camera, microphone, or
  location.

### The Modern Security Checklist

To make an iframe secure in a React app, you must use these defensive layers:

- **The `sandbox` Attribute:** This is your most important tool. It restricts what the iframe can do.
  ```jsx
  <iframe
    src="https://example.com"
    sandbox="allow-scripts allow-forms" // Restricts everything else (like popups or top-level navigation)
    title="Third-party Content"
  />
  ```
- **Content Security Policy (CSP):** Use the `frame-ancestors` directive in your server headers to control which sites are
  allowed to embed _your_ app.
- **`X-Frame-Options`:** A legacy but still effective header (`DENY` or `SAMEORIGIN`) to prevent your site from being iframed
  elsewhere.
- **Origin Validation:** If you use `postMessage` to communicate between React and the iframe, **always** check
  `event.origin` to ensure the message is from a trusted source.

---

## 2. Is it Common?

Yes, but for specific use cases. Developers generally avoid iframes for internal app features because they are hard to style
and break the "Single Page App" (SPA) feel.

### Common Use Cases (2026)

| Use Case               | Why use an iframe?                                                            |
| :--------------------- | :---------------------------------------------------------------------------- |
| **Payment Gateways**   | Keeps sensitive credit card data off your servers (PCI compliance).           |
| **Third-Party Media**  | Embedding YouTube, Spotify, or Google Maps without writing complex API logic. |
| **Isolated Sandboxes** | Tools like CodeSandbox use iframes to run user-generated code safely.         |
| **Legacy Systems**     | Porting an old app into a new React shell without a full rewrite.             |

### When to Avoid iframes

- **SEO-heavy content:** Search engines struggle to index content inside iframes.
- **Global Styling:** You cannot easily apply your React app's CSS to the content inside an iframe.
- **Performance:** Each iframe is essentially a new browser instance, which consumes more memory and can slow down your app.

---

## Summary

Using an iframe in React is **secure if you use the `sandbox` attribute and proper CSP headers**. It is **common for
third-party embeds**, but it is considered an anti-pattern for internal navigation or core UI components in modern React
architecture.

> [!TIP] If you need to communicate between your React app and the iframe, consider using a library like
> `iframe-resizer-react` to handle the "janky" parts like height adjustments and messaging.
