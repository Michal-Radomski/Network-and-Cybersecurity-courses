**Clickjacking** (also known as a "UI Redress Attack") is a deceptive technique where an attacker tricks a user into clicking
on something different from what the user perceives. It effectively "hijacks" the user's mouse clicks to perform unintended
actions on a different website.

---

## How the Attack Works

The core of clickjacking relies on the ability to embed one website inside another using an `<iframe>` and manipulating its
visibility.

1.  **The Decoy Page:** The attacker creates a harmless-looking webpage (e.g., a "Play this game" or "Click here for a free
    prize" site).
2.  **The Hidden Target:** The attacker embeds the target website (like your bank’s "Delete Account" page or a "Follow"
    button on social media) in an invisible `<iframe>` right on top of the decoy.
3.  **The Overlay:** Using CSS, the attacker sets the target iframe's `opacity` to `0` (making it invisible) but keeps it
    positioned directly over the decoy button.
4.  **The Trick:** When the user clicks the visible "Play Game" button, they are actually clicking the invisible "Delete
    Account" button on the target site.

---

## Real-World Examples

- **Social Engineering:** Tricking a user into "Liking" a page or following an account without their knowledge.
- **Privacy Settings:** Tricking a user into enabling their webcam or microphone by clicking what they think is a "Close"
  button on an ad.
- **Financial Theft:** Forcing a user to confirm a pre-filled funds transfer while they believe they are clicking "Skip" on
  an intro video.

## How to Prevent Clickjacking

Since you are running a **React/Express** app, you have two primary lines of defense to prevent your site from being "framed"
by malicious actors.

### 1. The `X-Frame-Options` Header

This is the older, widely supported method. It tells the browser whether your site is allowed to be put in a frame.

- **DENY:** No one can frame your site.
- **SAMEORIGIN:** Only your own site can frame itself (useful if you use your own modals).

**In Express (using Helmet):**

```javascript
const helmet = require("helmet");
app.use(helmet.frameguard({ action: "deny" }));
```

### 2. Content Security Policy (CSP) `frame-ancestors`

This is the modern and more flexible approach. It supersedes `X-Frame-Options` in modern browsers.

- `frame-ancestors 'none';` (Equivalent to DENY)
- `frame-ancestors 'self';` (Equivalent to SAMEORIGIN)

**In Express:**

```javascript
app.use(
  helmet.contentSecurityPolicy({
    directives: {
      "frame-ancestors": ["'none'"],
    },
  })
);
```

### 3. Frame-Busting Scripts (Legacy)

Before headers were standard, developers used JavaScript to check if they were in a frame:

```javascript
if (top !== self) {
  top.location = self.location;
}
```

_Note: This is easily bypassed by attackers and should only be used as a fallback._

## Why this matters for your React app

If your React app handles sensitive user data (like a dashboard or settings page), an attacker could frame your login or
settings page. Because the user is already logged into your site in their browser, the "invisible" click carries their
session cookies and performs the action as if they intended it.

By using **Helmet.js** in your Express backend, you can automatically add these headers to every response, making your app
"un-framable" by default.

---

---

Since you are running an **Express.js** backend to serve your **React** frontend, using the **Helmet** middleware is the
industry standard for securing your HTTP headers.

It’s essentially a "security-in-a-box" tool that sets various headers to protect against Clickjacking, XSS, and other common
attacks.

### 1. Install Helmet

In your Express project folder, run:

```bash
npm install helmet
```

---

### 2. Implementation in `server.js`

Add this to your main Express file. For maximum security, you should configure the **Content Security Policy (CSP)** to allow
your React app to function while blocking malicious external scripts.

```javascript
const express = require("express");
const helmet = require("helmet");

const app = express();

// Use Helmet with a secure configuration
app.use(
  helmet({
    // 1. Prevent Clickjacking
    frameguard: {
      action: "deny",
    },
    // 2. Modern CSP (Protects against XSS and Data Injection)
    contentSecurityPolicy: {
      directives: {
        "default-src": ["'self'"], // Only allow resources from your own domain
        "script-src": ["'self'"], // Only allow scripts from your own domain (no inline scripts)
        "style-src": ["'self'", "'unsafe-inline'"], // React often needs 'unsafe-inline' for styles
        "img-src": ["'self'", "data:"],
        "connect-src": ["'self'"], // Restricts where your React app can fetch data from
        "upgrade-insecure-requests": [], // Automatically upgrades HTTP to HTTPS
      },
    },
    // 3. Prevent Sniffing (Forces browser to stick to the declared MIME type)
    noSniff: true,
    // 4. Hide 'X-Powered-By: Express' (Stops attackers from knowing your tech stack)
    hidePoweredBy: true,
  })
);

// Your API routes and React static file serving below...
```

---

### 3. What these headers actually do

| Header                        | Purpose                                                                                                  |
| :---------------------------- | :------------------------------------------------------------------------------------------------------- |
| **Content-Security-Policy**   | The "Wall." It defines exactly which scripts, images, and styles are allowed to load.                    |
| **X-Frame-Options**           | Stops your site from being loaded in an `<iframe>` (Stops **Clickjacking**).                             |
| **X-Content-Type-Options**    | Prevents the browser from trying to guess if a file is a script when it's labeled as text.               |
| **Strict-Transport-Security** | (HSTS) Forces the browser to only communicate with your server over **HTTPS**.                           |
| **Referrer-Policy**           | Controls how much information is sent in the `Referer` header when a user clicks a link to another site. |

---

### 4. Important Tip for React Developers

If you use the `unsafe-inline` directive for styles, it's generally okay for React (especially if you use styled-components
or inline CSS), but **never** use `'unsafe-inline'` for `script-src` in production.

If you do, you effectively disable the primary protection against **XSS**, because an attacker could then inject a `<script>`
tag directly into your page.

### 5. Verify Your Setup

Once your app is running with PM2, you can test your security grade by entering your URL at
**[securityheaders.com](https://securityheaders.com)**. It will give you a letter grade (aim for an A!).
