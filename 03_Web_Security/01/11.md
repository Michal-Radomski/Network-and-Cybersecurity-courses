Cookies can be set on both, but they are used for different purposes depending on where you trigger them.

## 1. Set on the Node.js Server (Most Common)

The server sets the cookie by sending a Set-Cookie header in its HTTP response. This is the standard way to handle security
(like logins).

- How it works: When you log in, Node.js sends a response to the browser saying, "Here is the data you asked for, and also,
  please save this cookie."
- Best for: Authentication tokens (JWTs) and Session IDs.
- Security: You can use the HttpOnly flag. This makes the cookie invisible to JavaScript, meaning a hacker cannot steal it
  using a malicious script (XSS attack).

## 2. Set on the React.js Client

The client sets the cookie directly using JavaScript (e.g., document.cookie = "theme=dark").

- How it works: React code running in the browser writes directly to the browser's storage without asking the server.
- Best for: User Interface (UI) preferences, like "Dark Mode" vs. "Light Mode," or saving a "Dismissed Welcome Modal" flag.
- Security: These cookies are less secure because any script running on the page can read or change them.

---

## Comparison Table

| Feature     | Set by Node.js (Server)      | Set by React.js (Client)   |
| ----------- | ---------------------------- | -------------------------- |
| Mechanism   | Set-Cookie HTTP Header       | document.cookie in JS      |
| Primary Use | Login/Auth & Security        | UI Settings & Tracking     |
| Security    | High (can use HttpOnly)      | Low (accessible by all JS) |
| Persistence | Controlled by server headers | Controlled by JS code      |

## Which one should you choose?

- If the data is sensitive (like a login session): Set it on the Node.js server with HttpOnly: true.
- If the data is purely for the UI (like a language preference): It is fine to set it in React.
