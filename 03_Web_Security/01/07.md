Common Gateway Interface (CGI) is a standard protocol that allows a web server to communicate with external programs to
generate dynamic web pages. [1, 2, 3] Think of it as the "bridge" between a web server (like Apache) and a script or program
on the server's hard drive. [4, 5]

## How it Works

In the early days of the web, servers could only send "static" files (like a simple .html file). CGI was created so the
server could actually do something: [6, 7, 8]

1.  Request: You fill out a form on a website and click "Submit."
2.  Hand-off: The web server receives the data and passes it to a CGI script (written in languages like Perl, C++, or
    Python).
3.  Processing: The script runs, perhaps talks to a database, and generates new HTML on the fly.
4.  Response: The CGI script sends that HTML back to the server, which then sends it to your browser. [9, 10, 11, 12, 13]

## Key Characteristics

- Language Independent: CGI doesn't care what language the script is written in, as long as the server can execute it.
- Separate Processes: Every time a request comes in, the server starts a brand new process for that script. Once the task is
  done, the process closes.
- The "CGI-BIN": You may have seen URLs containing /cgi-bin/. This is the traditional folder on a server where these
  executable scripts are stored for security reasons. [14, 15, 16, 17, 18]

## CGI vs. Modern Web Development

While CGI was the foundation of the dynamic web, it is rarely used for high-traffic sites today because of performance: [19,
20, 21, 22]

- The Problem: Starting a new process for every single visitor is very "expensive" for a computer's CPU and memory. If 1,000
  people click a button at once, the server has to start 1,000 separate programs.
- The Modern Solution: Technologies like FastCGI, Node.js, or PHP-FPM keep the program running in the background constantly,
  so they can handle thousands of requests without the "startup" delay of traditional CGI. [23, 24]
