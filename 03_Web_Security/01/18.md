WebSocket security is unique because it starts as an HTTP request but quickly "upgrades" to a permanent, persistent
connection. This transition creates specific security requirements. Here is the breakdown of how WebSocket security works:

## 1. The Protocol: Use wss://

Just as you use https instead of http, you should always use wss:// (WebSocket Secure).

- What it does: It runs the WebSocket connection over a TLS/SSL encrypted tunnel.
- Why it matters: It prevents "Man-in-the-Middle" attacks where someone on your Wi-Fi could read the messages being sent back
  and forth. It also helps bypass restrictive firewalls/proxies that might block non-encrypted traffic [1, 3].

## 2. No Same-Origin Policy (SOP)

This is the most critical security difference: WebSockets are not restricted by the Same-Origin Policy.

- The Risk: Any website can attempt to open a WebSocket connection to your server.
- The Solution: You must manually check the Origin header on the server during the handshake. If the request isn't coming
  from your trusted frontend domain, the server should reject the connection [1, 2].

## 3. Authentication & Authorization

Unlike standard web pages, WebSockets don't have a "login page." You must verify the user during the initial handshake:

- Ticket-based Auth: Before opening the socket, the client makes a standard HTTP request to get a temporary "token." It then
  passes this token as a query parameter when connecting to the WebSocket [2].
- Cookie Sharing: If your WebSocket server is on the same domain as your web app, it can read the user's session cookies
  during the handshake [1].

## 4. Protecting Against "Cross-Site WebSocket Hijacking" (CSWSH)

Because WebSockets automatically send cookies during the handshake, a malicious site could trick a logged-in user into
opening a connection to your server.

- Prevention: Use CSRF tokens or strictly validate the Origin header as mentioned above [1, 3].

## 5. Rate Limiting and Payload Validation

Because a WebSocket connection stays open, a single user could potentially "flood" your server with thousands of messages a
second, causing a Denial of Service (DoS).

- Validation: Always validate the data format (usually JSON) coming through the socket.
- Throttling: Implement limits on how many messages a single connection can send per minute [2].

## Summary Checklist

| Security Layer | Action                                                        |
| -------------- | ------------------------------------------------------------- |
| Encryption     | Always use wss://.                                            |
| Origin Check   | Reject any connection that doesn't match your domain.         |
| Handshake Auth | Verify a token or cookie before allowing the "Upgrade."       |
| Data Integrity | Treat all incoming socket messages as untrusted "user input." |

---

To validate the Origin header in a Node.js WebSocket server, you can use the verifyClient option provided by the ws library.
This function runs during the initial handshake, allowing you to reject unauthorized connection attempts before they are
upgraded to a full WebSocket connection. [1, 2, 3, 4]

## Node.js Server Origin Validation Example

The following code demonstrates how to restrict connections to a specific list of trusted domains:

const { WebSocketServer } = require('ws'); // 1. Define your trusted frontend domainsconst ALLOWED_ORIGINS =
['https://your-app.com', 'http://localhost:3000']; const wss = new WebSocketServer({ port: 8080, // 2. The verifyClient hook
intercepts the handshake verifyClient: (info, callback) => { const origin = info.origin;

    console.log(`Connection attempt from origin: ${origin}`);

    // 3. Check if the incoming origin is in our allowed list
    if (ALLOWED_ORIGINS.includes(origin)) {
      callback(true); // Connection accepted
    } else {
      console.warn(`Rejected unauthorized origin: ${origin}`);
      // Send a 403 Forbidden status code back to the browser
      callback(false, 403, 'Forbidden');
    }

} });

wss.on('connection', (ws) => { console.log('Client connected securely!'); ws.send('Welcome to the secure WebSocket server.');
});

## Why this is effective

- Browser-Enforced: Browsers automatically attach the Origin header to every WebSocket request, and it cannot be easily
  modified or spoofed by client-side JavaScript.
- Early Rejection: By using verifyClient, the server rejects the connection at the HTTP level (handshake) rather than letting
  the socket open and then closing it, which saves server resources.
- Security Context: It prevents Cross-Site WebSocket Hijacking (CSWSH) by ensuring that only scripts running on your official
  domains can open a tunnel to your backend. [2, 3, 5, 6, 7]

## Important Limitation

While this protects against malicious websites, it does not stop non-browser tools (like a custom script or Postman) from
connecting, as they can manually set any Origin header they want. For full security, you should combine origin checking with
token-based authentication (like JWT). [5, 8, 9, 10]
