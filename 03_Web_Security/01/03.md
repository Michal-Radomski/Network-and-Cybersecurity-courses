Bytecode is a compact, intermediate form of code that sits exactly between the high-level JavaScript you write and the binary
machine code the computer's CPU actually understands. [1, 2] In the context of a browser, think of it as a "half-baked"
version of your script. It is no longer human-readable text, but it isn't yet ready for the hardware to run. [3, 4]

## Why do browsers use it?

Browsers generate bytecode primarily for speed and portability: [5, 6, 7, 8]

- Faster Execution: Interpreting raw text is slow. Bytecode is much faster for the engine to read and execute than plain
  text.
- Platform Independence: The same bytecode can be sent to any device (Windows, Mac, or Phone) because it isn't tied to a
  specific processor yet.
- Efficiency: It’s easier for the engine’s
  [JIT (Just-In-Time) compiler](https://web.stanford.edu/class/cs98si/slides/overview) to turn bytecode into highly optimized
  machine code later. [1, 2, 3, 5, 9]

## Bytecode vs. Machine Code

| Feature [2, 3, 10, 11, 12, 13, 14] | Bytecode                                    | Machine Code                      |
| ---------------------------------- | ------------------------------------------- | --------------------------------- |
| Readability                        | Numeric codes (hexadecimal/binary).         | Pure binary (0s and 1s).          |
| Execution                          | Needs a virtual machine or interpreter.     | Runs directly on the CPU.         |
| Portability                        | Runs on any platform with the right engine. | Specific to one type of hardware. |

## Real-World Example

In Chrome's [V8 engine](https://v8.dev/), the interpreter (called Ignition) takes your script and produces bytecode like
LdaSmi [42] (Load Small Integer 42). While this looks like assembly language to us, the browser sees it as a specific set of
numbers it can process instantly. [13, 15, 16, 17, 18]
