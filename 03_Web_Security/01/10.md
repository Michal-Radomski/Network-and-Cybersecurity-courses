To understand the difference, think of Cookies as an ID card you carry in your pocket, and a Session as a file folder stored
in the server's filing cabinet. [1]

## 1. Cookies (Client-Side)

A cookie is a small text file stored on your computer by your browser. [2, 3, 4]

- Storage: Your browser (Chrome, Safari, etc.).
- How it works: When a website wants to remember you, it sends a "Set-Cookie" header. Your browser saves it and sends it back
  to the server every time you visit a new page on that site.
- Lifecycle: Can last for years (persistent) or disappear when you close the tab (session cookie).
- Best for: Storing user preferences (dark mode), tracking for ads, or remembering a "Remember Me" checkbox. [5, 6, 7, 8, 9]

## 2. Sessions (Server-Side)

A session is a way to store data about a user on the web server. [10]

- Storage: The server's memory or a database (like Redis).
- How it works: Since the server can't "see" who you are between clicks, it creates a unique Session ID. It stores your data
  (like your shopping cart) on the server and gives you a cookie containing only that ID.
- Lifecycle: Usually expires quickly (e.g., 20 minutes of inactivity) or when the browser is closed.
- Best for: Sensitive info, like "Is this user logged in?" or "What is in their shopping cart?" [11, 12, 13, 14, 15]

---

## How They Work Together

Most modern websites use both to keep you logged in: [16]

1.  You log in with your password.
2.  The server validates you and creates a Session in its database.
3.  The server sends back a Cookie to your browser that says: session_id=abc123.
4.  On your next click, your browser sends that session_id back.
5.  The server looks up abc123 in its cabinet and says, "Oh, that's John! Let him in." [17, 18, 19, 20, 21]

## Key Differences

| Feature [22, 23, 24, 25, 26] | Cookie                       | Session                           |
| ---------------------------- | ---------------------------- | --------------------------------- |
| Location                     | Browser / User's Device      | Web Server                        |
| Security                     | Lower (User can see/edit it) | Higher (User cannot see the data) |
| Size Limit                   | Small (typically 4KB)        | Large (limited by server memory)  |
| Data Type                    | Only Strings                 | Objects, Arrays, complex data     |

Security Note: Because cookies are stored on the user's machine, you should never store a plain password or a user's bank
balance in a cookie. You store those in a Session and just use the cookie as the "key" to unlock it. [27, 28, 29, 30, 31]
