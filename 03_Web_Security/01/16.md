The Same-Origin Policy (SOP) is a fundamental security mechanism in web browsers that prevents a script on one website from
accessing data on another website. [1, 2] It acts as a "security wall" to ensure that a malicious site can't steal your
sensitive data (like login sessions or personal info) from a legitimate site you have open in another tab. [3, 4, 5, 6, 7]

## 1. What defines an "Origin"?

Two URLs have the same origin only if these three things are identical: [8, 9]

1.  Protocol (e.g., http vs https)
2.  Host/Domain (e.g., example.com vs ://example.com)
3.  Port (e.g., :80 vs :8080)

| URL to Compare [10, 11, 12, 13, 14] | Base URL: https://example.com | Result                                          | Reason |
| ----------------------------------- | ----------------------------- | ----------------------------------------------- | ------ |
| https://example.com                 | Same Origin                   | Same protocol and host.                         |        |
| http://example.com                  | Different Origin              | Different protocol (HTTP vs HTTPS).             |        |
| https://example.com                 | Different Origin              | Different host (subdomain counts as different). |        |
| https://example.com:443             | Same Origin                   | 443 is the default port for HTTPS.              |        |

## 2. Why is it necessary?

Without SOP, if you were logged into your bank at bank.com and visited a malicious site evil.com in another tab, a script on
evil.com could: [15]

- Make a request to ://bank.com.
- Read your private financial data.
- Perform actions (like transferring money) using your saved login cookies. [16, 17]

SOP blocks evil.com from reading the response coming back from bank.com. [18]

## 3. What is NOT blocked?

SOP is mainly about preventing scripts from reading data. Browsers generally allow: [19, 20]

- Embedding: You can display images (<img>), play videos (<video>), or load scripts (<script>) from other origins.
- Linking: You can click a link that takes you to a different site.
- Writing: You can often send data (like a form submission) to another origin, though you usually won't be allowed to read
  the response. [21, 22, 23, 24, 25]

## 4. How to bypass it safely?

Sometimes you want your frontend (on mysite.com) to talk to your API (on ://mysite.com). To allow this, we use CORS
(Cross-Origin Resource Sharing). The server sends a header like Access-Control-Allow-Origin: https://mysite.com, which tells
the browser, "I trust this specific site; let it read my data." [26, 27, 28, 29, 30]
