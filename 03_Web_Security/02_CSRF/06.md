Defending against Cross-Site Request Forgery (CSRF) requires a "defense-in-depth" approach. While modern browser defaults
have made CSRF harder to exploit, relying on a single layer is risky.

Here are the primary countermeasures used to secure web applications today.

---

## 1. Anti-CSRF Tokens (Synchronizer Token Pattern)

This is the most common and robust defense. The server generates a unique, cryptographically strong, and unpredictable token
for the user's session.

- **How it works:** The server embeds this token into every state-changing form or as a header in AJAX requests.
- **Validation:** When the user submits the form, the server compares the token in the request to the one stored in the
  user's session.
- **Why it works:** Because of the **Same-Origin Policy (SOP)**, an attacker’s site cannot "read" the token from your site,
  meaning they cannot include it in their forged request.

---

## 2. SameSite Cookie Attribute

This is a browser-level defense that instructs the browser whether to send cookies with cross-site requests.

| Attribute Value       | Behavior                                                                                                                                                                                        |
| :-------------------- | :---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **`SameSite=Strict`** | Cookies are only sent if the request originates from the same site. It offers the best protection but can break user experience (e.g., clicking a link from an email won't keep you logged in). |
| **`SameSite=Lax`**    | (Current Browser Default) Cookies are not sent on "sub-requests" (like images or frames) but **are** sent when a user navigates to the site via a top-level link.                               |
| **`SameSite=None`**   | Cookies are sent in all contexts. This requires the `Secure` flag and offers no CSRF protection.                                                                                                |

---

## 3. Double Submit Cookies

This is a "stateless" defense often used in APIs or single-page applications (SPAs) where the server doesn't want to track
tokens in a database.

1.  The server sends a random value in a cookie (e.g., `csrf_cookie=xyz`).
2.  The client-side code reads this cookie and copies the value into a hidden form field or a custom HTTP header.
3.  The server verifies that the value in the cookie matches the value in the request parameter/header.
4.  **Security Note:** Since an attacker cannot read cookies from your domain, they cannot mirror the value into the request.

---

## 4. Custom Request Headers

CSRF attacks typically rely on HTML tags (like `<form>` or `<img>`) which can only trigger "simple" requests. Browsers do not
allow cross-origin requests to have custom headers (like `X-Requested-With` or `X-CSRF-Token`) without first sending a **CORS
preflight (OPTIONS)** request.

If your server requires a custom header for all POST/PUT/DELETE actions, the browser will block the attacker's attempt before
it even reaches your business logic.

---

## 5. User Interaction (Challenge-Response)

For high-value transactions (like changing a password or transferring a large sum of money), you should force the user to
interact with the application again. This breaks the "automatic" nature of CSRF.

- **Re-authentication:** Asking for the password again.
- **CAPTCHA:** Requiring the user to solve a puzzle.
- **One-Time Password (OTP):** Sending a code to their phone/email.

---

## Summary of Best Practices

- **Never use GET for state changes:** Ensure GET requests are "read-only."
- **Use Framework Defaults:** Most modern frameworks (Django, Rails, Spring, Angular) have CSRF protection built-in and
  enabled by default. Don't disable them.
- **Set `SameSite=Lax` or `Strict`:** Always configure your session cookies with these attributes.
- **Verify Origins:** Check the `Origin` and `Referer` headers on the server side to ensure the request is coming from your
  own domain.
