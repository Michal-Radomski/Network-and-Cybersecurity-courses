The **Referer header** (note the famous misspelling in the HTTP spec) is an HTTP request header that identifies the address
of the web page that linked to the resource being requested.

Essentially, it tells the server: **"Hey, I’m coming from this specific URL."**

---

## How It Works

When you click a link on `Website-A.com` that leads to `Website-B.com`, your browser automatically includes the URL of
`Website-A` in the request sent to `Website-B`.

### Example

1.  You are reading an article on `https://news-site.com/tech-news`.
2.  You click a link to a product on `https://shop.com/laptop`.
3.  The browser sends this to `shop.com`:
    ```http
    GET /laptop HTTP/1.1
    Host: shop.com
    Referer: https://news-site.com/tech-news
    ```

---

## Use Cases for the Referer Header

### 1. Analytics and Tracking

Website owners use this to see where their traffic is coming from (e.g., Google search, a social media post, or a specific
blog).

### 2. Basic CSRF Protection

As we discussed earlier, a server can check the Referer header to ensure the request originated from its own domain. If
`bank.com` sees a request to transfer money, but the Referer is `evil-hacker-site.com`, it can block the request.

> **Note:** This is considered a weak defense because the header can sometimes be omitted by browsers or stripped by proxy
> servers.

### 3. Preventing "Hotlinking"

Image hosting sites use this to prevent other websites from stealing their bandwidth. If the Referer isn't their own site,
they can block the image from loading on the external page.

---

## Privacy and Security Concerns

Because the Referer header can contain sensitive information (like search queries or user IDs embedded in a URL), modern web
standards allow developers to control how much information is shared.

### The Referrer-Policy

You can control this behavior using the `Referrer-Policy` HTTP header. Common settings include:

- **`no-referrer`**: The header is never sent.
- **`same-origin`**: Only sent when navigating within the same website.
- **`strict-origin-when-cross-origin`**: (Modern default) Sends the full URL when on the same site, but only sends the domain
  (e.g., `https://news-site.com/`) when going to a different site.

---

## The "Famous Misspelling"

If you noticed the missing "r" in the header name (**Referer** vs. **Referrer**), you're not alone! It was misspelled in the
original 1996 HTTP proposal. By the time the error was noticed, it was already so widely implemented that it became a
permanent part of the internet's infrastructure to avoid breaking existing websites.

---

To protect your users' privacy and prevent leaking sensitive data (like tokens or internal URL structures) to other sites,
you should set a **Referrer-Policy**.

In an Express.js app, the best way to handle this—along with other security headers—is by using the **Helmet** middleware.

---

## 1. Setting the Policy in Express

First, install Helmet:

```bash
npm install helmet
```

Then, configure the `referrerPolicy` middleware. For most modern React + Express applications,
`strict-origin-when-cross-origin` is the recommended balance between utility and privacy.

```typescript
import express from "express";
import helmet from "helmet";

const app = express();

// Use Helmet to set security headers
app.use(
  helmet({
    referrerPolicy: {
      policy: "strict-origin-when-cross-origin",
    },
  })
);

// This tells the browser:
// 1. If moving within my own site: Send the full URL.
// 2. If moving to another site (HTTPS): Only send the domain (e.g., https://mysite.com/).
// 3. If moving from HTTPS to HTTP: Send nothing at all.
```

---

## 2. Why "Referrer" checking is only a secondary defense

While you can use the `Referer` header to help block CSRF, it shouldn't be your only defense. Here is why:

### The Weaknesses

- **Privacy Tools:** Some browser extensions or VPNs strip the Referer header entirely for privacy. If your server _requires_
  the header to be present, you might block legitimate users.
- **Policy Control:** An attacker can use `<meta name="referrer" content="no-referrer">` on their own site to ensure the
  browser doesn't send the header when attacking your site, making it look like a direct "bookmark" visit.
- **Spoofing:** While browsers prevent JavaScript from setting the Referer header, it can sometimes be manipulated in
  non-browser environments or through very complex browser bugs.

---

## 3. Difference between Origin and Referer

When building your Express security logic, you might see both headers. Here is the difference:

| Header      | Content                                                  | Best Use Case                                                                                                                       |
| :---------- | :------------------------------------------------------- | :---------------------------------------------------------------------------------------------------------------------------------- |
| **Origin**  | Just the domain (e.g., `https://example.com`)            | **CSRF Defense.** It’s more reliable because it doesn't contain the full path and is sent with almost all POST/PUT/DELETE requests. |
| **Referer** | The full URL (e.g., `https://example.com/user/settings`) | **Analytics & Path Tracking.** Useful for seeing exactly which page triggered a request, but less privacy-friendly.                 |

## Summary Checklist for your Express App

- [x] **CSRF Tokens:** Use `double-csrf` for state-changing actions.
- [x] **SameSite Cookies:** Set your session cookies to `Lax` or `Strict`.
- [x] **Security Headers:** Use `helmet` to set the `Referrer-Policy` and `X-Content-Type-Options`.
- [x] **Origin Check:** For high-security endpoints, verify that `req.headers.origin` matches your frontend domain.
