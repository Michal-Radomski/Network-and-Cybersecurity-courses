It is important to clear up a common misconception: **React does not have built-in CSRF protection.**

Because React is a client-side library, it cannot protect itself. CSRF protection must be a **cooperative effort** between
your React frontend and your Backend API. React's job is simply to attach the security credentials that the server requires.

The most common way to handle this in React is using an **Axios Interceptor** or a custom **Fetch wrapper**.

---

### The Strategy: Double Submit Cookie / Header Pattern

In this flow, the server sends a CSRF token in a cookie. Your React app reads that cookie and sends the value back in a
custom header (e.g., `X-XSRF-TOKEN`) for every "state-changing" request (POST, PUT, DELETE).

---

### 1. Automated Implementation with Axios

Axios is popular because it has CSRF support "baked in." You just need to tell it which cookie to look for and which header
to use.

```typescript
import axios from "axios";

// Create an instance with CSRF defaults
const apiClient = axios.create({
  baseURL: "https://api.yourdomain.com",
  withCredentials: true, // Crucial: tells the browser to send cookies

  // These are the default settings, but it's good to be explicit:
  xsrfCookieName: "XSRF-TOKEN",
  xsrfHeaderName: "X-XSRF-TOKEN",
});

// Now, every POST request automatically includes the token from the cookie
export const updateProfile = (data: any) => {
  return apiClient.post("/user/update", data);
};
```

---

### 2. Implementation with Fetch (using Interceptors)

If you prefer the native `fetch` API, you'll need to manually grab the token from the document cookies. Since
`document.cookie` is a messy string, we usually use a small helper function.

```typescript
// Helper to get a cookie by name
const getCookie = (name: string): string | null => {
  const value = `; ${document.cookie}`;
  const parts = value.split(`; ${name}=`);
  if (parts.length === 2) return parts.pop()?.split(";").shift() || null;
  return null;
};

// Custom fetch wrapper
const secureFetch = async (url: string, options: RequestInit = {}) => {
  const csrfToken = getCookie("XSRF-TOKEN");

  const headers = {
    ...options.headers,
    "Content-Type": "application/json",
    "X-XSRF-TOKEN": csrfToken || "", // Attach the token
  };

  return fetch(url, { ...options, headers });
};

// Usage in a component
const handleSubmit = () => {
  secureFetch("/api/delete-account", { method: "POST" });
};
```

---

### 3. Critical Security Requirements

Even with the code above, your protection will fail if the server isn't configured correctly. Ensure your backend team does
the following:

1.  **Expose the Cookie:** The CSRF cookie must **NOT** be `HttpOnly`. If it is `HttpOnly`, your React code cannot read it to
    put it in the header. (Note: This is specifically for the _Token_ cookie, not your _Session_ cookie).
2.  **Verify the Header:** The server must check that the `X-XSRF-TOKEN` header matches the value in the cookie.
3.  **SameSite Attribute:** The session cookie should still be set to `SameSite=Lax` or `Strict` as a baseline defense.

### Summary Checklist for React

- [ ] Avoid using GET requests for any action that modifies data.
- [ ] Use `withCredentials: true` in Axios or `credentials: 'include'` in Fetch.
- [ ] Use an interceptor to inject the CSRF token into headers.
- [ ] Ensure your API validates the token on every POST/PUT/DELETE.
