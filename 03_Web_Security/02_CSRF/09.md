To complete the loop, the **Express.js** backend must do two things: provide the CSRF token to the React frontend (usually
via a cookie) and then verify that the token comes back in the headers of any state-changing request.

Here is the implementation using the modern `csrf-csrf` library, which follows the **Double Submit Cookie** pattern.

### 1. Installation

First, install the necessary middleware:

```bash
npm install csrf-csrf cookie-parser
```

---

### 2. The Express Configuration

This setup configures the server to look for a header named `x-xsrf-token` and a cookie named `XSRF-TOKEN`.

```typescript
import express from "express";
import cookieParser from "cookie-parser";
import { doubleCsrf } from "csrf-csrf";

const app = express();

// 1. Standard body parsing and cookie parsing
app.use(express.json());
app.use(cookieParser("your-secret-key")); // Use a strong secret for cookie signing

// 2. Configure CSRF Protection
const { generateToken, doubleCsrfProtection } = doubleCsrf({
  getSecret: () => "your-secret-key",
  cookieName: "XSRF-TOKEN", // The name React/Axios looks for
  cookieOptions: {
    httpOnly: false, // CRITICAL: React must be able to read this cookie
    sameSite: "lax",
    path: "/",
    secure: process.env.NODE_ENV === "production",
  },
  getTokenFromRequest: (req) => req.headers["x-xsrf-token"], // Where React sends it
});

// 3. Endpoint to give the token to React
// React should call this when the app loads or user logs in
app.get("/api/csrf-token", (req, res) => {
  const token = generateToken(req, res);
  res.json({ token });
});

// 4. Protect your sensitive routes
// This middleware will automatically look for the header and compare it to the cookie
app.post("/api/user/update", doubleCsrfProtection, (req, res) => {
  res.json({ status: "Profile updated safely!" });
});

app.listen(3000);
```

---

### 3. Why `httpOnly: false`?

Usually, we say all cookies should be `httpOnly: true` to prevent XSS attacks. However, in the **Double Submit Cookie**
pattern:

- The **Session Cookie** (which identifies who you are) must be `httpOnly: true`.
- The **CSRF Token Cookie** must be `httpOnly: false` because the React code needs to read its value to copy it into the
  `x-xsrf-token` header.

---

### 4. Handling Error Responses

When a CSRF check fails (e.g., an attacker tries to forge a request without the token), the middleware will throw an error.
You should catch this to prevent your server from crashing and to send a clean message.

```typescript
// Error handling middleware
app.use((err, req, res, next) => {
  if (err.code === "EBADCSRFTOKEN") {
    // Handle CSRF errors specifically
    res.status(403).json({ error: "Invalid CSRF token. Potential attack detected." });
  } else {
    next(err);
  }
});
```

---

### Summary of the React + Express Handshake

1.  **Initial Load:** React calls `GET /api/csrf-token`.
2.  **Server Response:** Express sets the `XSRF-TOKEN` cookie and returns the token in the JSON body.
3.  **Client Storage:** React/Axios now has the cookie stored in the browser.
4.  **The Action:** User clicks "Update Profile".
5.  **The Request:** Axios reads the `XSRF-TOKEN` cookie and attaches it to the `x-xsrf-token` header in the `POST` request.
6.  **The Verification:** Express sees the POST request, compares the header value to the cookie value, and only allows the
    code to run if they match.
