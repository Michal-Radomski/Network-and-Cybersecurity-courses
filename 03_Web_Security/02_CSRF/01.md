CSRF (Cross-Site Request Forgery) is a cyberattack that tricks a logged-in user into performing an action they didn't intend
to do on a different website. If XSS is about stealing data, CSRF is about impersonating a user to execute a command.

## How a CSRF Attack Works

Imagine you are logged into your bank at bank.com. Because you are logged in, your browser has a Session Cookie for that
site.

1.  The Trap: You visit a malicious site, evil-site.com, in a different tab.
2.  The Hidden Request: On evil-site.com, there is a hidden piece of code (like an invisible form or an image tag).
3.  The Forgery: The code automatically triggers a request to your bank: POST https://bank.com
4.  The Browser’s Help: Your browser thinks, "Oh, I have a cookie for bank.com! I'll attach it to this request so the server
    knows who is asking."
5.  The Execution: The bank’s server sees your valid cookie and thinks you personally clicked "Transfer." It processes the
    theft.

## Why does this happen?

## It happens because browsers automatically include all cookies associated with a domain whenever a request is made to that domain, regardless of which website triggered the request.

## How to Prevent CSRF

As a developer, you can stop this using two main methods:

## 1. Anti-CSRF Tokens (The Most Common)

When the server sends you a form, it includes a hidden, random "token" (a long string of characters).

- When you submit the form, the server checks if the token matches.
- A hacker on evil-site.com can trigger a request, but they cannot read your bank's page to find out what that secret token
  is. Without the token, the server rejects the request.

## 2. SameSite Cookie Attribute

This is a newer, simpler fix. When setting a cookie in Node.js, you can add the SameSite attribute:

- SameSite=Strict: The cookie is never sent if the request comes from a different website.
- SameSite=Lax: (Modern Browser Default) The cookie is only sent if you are actually navigating to the site (e.g., clicking a
  link), but not for hidden requests like images or background forms.

## Summary: CSRF vs. XSS

- XSS: A hacker runs a script inside your site to steal data (like document.cookie).
- CSRF: A hacker uses an outside site to trick your browser into "pressing a button" on your site using your existing
  session.
