When a SQL injection vulnerability exists, an attacker isn't limited to just viewing data. If the database user account has
the necessary permissions, they can perform **Data Manipulation Language (DML)** or **Data Definition Language (DDL)**
operations to permanently alter the database.

---

## 1. Modifying Existing Records (`UPDATE`)

This is commonly used to change administrative settings, reset passwords, or manipulate financial balances.

**The Vulnerable Scenario:** A profile update page uses a query like:
`UPDATE users SET bio = 'user_input' WHERE user_id = 123;`

**The Attack:** An attacker sets their bio to: `Hacked' , password = 'password123' WHERE user_id = 1 --`

**The Resulting SQL:**
`UPDATE users SET bio = 'Hacked' , password = 'password123' WHERE user_id = 1 --' WHERE user_id = 123;`

The `--` (comment) cuts off the original restriction, allowing the attacker to change the admin's password (typically
`user_id = 1`) instead of their own bio.

---

## 2. Injecting New Records (`INSERT`)

Attackers use this to create "backdoor" accounts or inject malicious scripts into the database that will later be executed in
the browsers of other users (Stored Cross-Site Scripting).

**Example:** On a registration form, an attacker might provide a crafted username that closes the current `INSERT` statement
and starts a new one: `' ); INSERT INTO users (username, password, role) VALUES ('attacker', 'pwned', 'admin'); --`

---

## 3. Deleting Data (`DELETE` and `DROP`)

This is the most destructive form of injection, often used for "ransomware" style attacks or pure vandalism.

- **Deleting Rows:** Using a payload like `' OR 1=1; --` in a delete query can wipe an entire table's contents.
- **Dropping Tables:** If the application supports stacked queries (multiple SQL statements separated by a semicolon), an
  attacker can follow a legitimate `SELECT` with a destructive command: `' ; DROP TABLE orders; --`

---

## 4. The "Stacked Query" Technique

Modifying a database often requires executing an entirely new command. This is possible through **Stacked Queries**, where
the attacker terminates the original query with a semicolon (`;`) and appends their own.

**Original Query:** `SELECT * FROM products WHERE id = 10;` **Injected Input:**
`10; UPDATE users SET role = 'admin' WHERE id = 5;`

> **Note:** Not all database drivers or APIs support stacked queries (e.g., PHP's `mysql_query()` does not, while
> `mysqli_multi_query()` does).

---

## How to Prevent Database Modification

Since these attacks are high-impact, the following defenses are critical:

### Technical Defenses

- **Parameterized Queries:** Always use placeholders (e.g., `?` or `:name`) so that the database engine never interprets user
  input as a command.
- **Escaping Input:** Use database-specific escaping functions as a secondary layer of defense.

### Architectural Defenses

- **Principle of Least Privilege:** This is the most effective way to stop database modification. If your web application
  only needs to read data, the database user should only have `SELECT` permissions. It should **never** have `DROP`,
  `TRUNCATE`, or `GRANT` permissions.
- **Web Application Firewalls (WAF):** A WAF can detect and block common SQLi patterns (like `DROP TABLE` or `OR 1=1`) before
  they reach your server.
