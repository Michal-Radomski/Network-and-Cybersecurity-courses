An **SQL Injection (SQLi)** is a type of vulnerability where an attacker "injects" malicious SQL code into a query through an
input field. This happens because the application fails to properly sanitize user input, allowing the attacker to manipulate
the database's logic.

If successful, an attacker can bypass logins, view private user data, modify or delete records, and in some cases, gain full
administrative control over the database server.

---

### How the Attack Works

Imagine a standard login form. The backend code likely looks something like this:

```sql
-- The intended query
SELECT * FROM users WHERE username = 'admin' AND password = 'password123';
```

If the application is vulnerable, it simply takes the text from the input box and pastes it into the string. An attacker
could enter the following into the **username** field:

`admin' --`

The resulting SQL query becomes:

```sql
SELECT * FROM users WHERE username = 'admin' --' AND password = '...';
```

In SQL, `--` is a comment. This tells the database to **ignore everything after the username**. The database checks if the
user 'admin' exists, sees the comment, and logs the attacker in without ever checking the password.

---

### Common SQLi Techniques

| Technique               | Description                                                                                                                                                                |
| :---------------------- | :------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **In-Band (Classic)**   | The attacker uses the same communication channel to launch the attack and gather results (e.g., the data appears right on the webpage).                                    |
| **Inferential (Blind)** | The page doesn't show data, but the attacker observes changes in the page's response or timing (e.g., "If the first letter of the admin password is 'A', wait 5 seconds"). |
| **Union-Based**         | Uses the `UNION` operator to combine the results of the original query with a completely different table (like the table containing all passwords).                        |
| **Out-of-Band**         | The attacker makes the database server send the data to an external server they control (e.g., via DNS or HTTP requests).                                                  |

### How to Prevent SQL Injection

The golden rule of web security is: **Never trust user input.**

#### 1. Use Prepared Statements (Parameterized Queries)

This is the most effective defense. Instead of building a string, you use "placeholders" (`?`). The database treats the input
strictly as **data**, never as **executable code**.

**Vulnerable (Node.js/Express example):**

```javascript
const query = "SELECT * FROM users WHERE id = " + req.body.id; // DANGEROUS
```

**Secure (Using Prepared Statements):**

```javascript
const query = "SELECT * FROM users WHERE id = ?";
db.execute(query, [req.body.id]); // SAFE
```

#### 2. Use an ORM (Object-Relational Mapper)

Tools like **Sequelize**, **Prisma**, or **Mongoose** (for NoSQL) automatically use prepared statements under the hood,
making it much harder to accidentally write a vulnerable query.

#### 3. Input Validation

Check that the input is what you expect. If you are asking for a `user_id`, ensure the input is actually a number before
sending it to the database.

#### 4. Principle of Least Privilege

Do not run your web application using a database "root" or "admin" account. Create a specific user that can only access the
tables it needs and has no permission to drop tables or shut down the server.

> **Historical Note:** One of the most famous "hacks" in pop culture is the "Little Bobby Tables" comic, which humorously
> illustrates how a school's database was wiped because they didn't sanitize their student enrollment inputs.
