At its core, a **SQL Injection (SQLi)** against a `SELECT` statement occurs when an application takes user input and inserts
it directly into a database query without proper sanitization. This allows an attacker to "break out" of the intended data
field and append their own SQL commands.

---

## How the Vulnerability Happens

Imagine a website with a search feature that looks up products by name. The backend code might look like this:

`query = "SELECT name, description FROM products WHERE name = '" + user_input + "';"`

### 1. The Intended Use

If a user searches for `Laptop`, the database executes: `SELECT name, description FROM products WHERE name = 'Laptop';`

### 2. The Malicious Input

An attacker doesn't enter a product name. Instead, they enter: `' OR '1'='1`

The resulting query becomes: `SELECT name, description FROM products WHERE name = '' OR '1'='1';`

Because `'1'='1'` is always true, the database ignores the name filter and returns **every single row** in the products
table.

---

## Common Objectives of SELECT Injection

When attacking a `SELECT` statement, hackers usually aim for one of three things:

### A. Bypassing Authentication

Often used on login forms. By injecting `' OR 1=1--`, an attacker can trick the database into validating a login without a
correct password, as the `WHERE` clause evaluates to true for the first user in the table (usually the admin).

### B. Data Extraction via UNION

The `UNION` operator is the most powerful tool for a `SELECT` injection. It allows an attacker to combine the results of the
original query with a completely different table.

**Example Input:** `' UNION SELECT username, password FROM users--`

This forces the application to display sensitive user credentials in the area where it normally shows product descriptions.

### C. Database Fingerprinting

Attackers use `SELECT` statements to "map" the database. By using functions like `@@version` (SQL Server) or `user()`
(MySQL), they can determine the database type and version to plan more sophisticated attacks.

---

## The "Invisible" Attack: Blind SQLi

Sometimes, the web page doesn't display the results of the `SELECT` query directly. In these cases, attackers use **Blind
SQLi**:

- **Boolean-Based:** The attacker asks the database true/false questions (e.g., "Does the admin password start with 'A'?")
  and observes if the page loads normally or shows an error.
- **Time-Based:** The attacker commands the database to pause for 10 seconds if a condition is true. If the page takes 10
  seconds to load, the attacker knows their guess was correct.

---

## How to Prevent It

The rule of thumb is **never trust user input.**

1.  **Prepared Statements (Parameterized Queries):** This is the gold standard. It treats user input as literal data, never
    as executable code.
2.  **Input Validation:** Use allow-lists to ensure input matches expected formats (e.g., an ID must be a number).
3.  **Principle of Least Privilege:** Ensure the database account used by the web app only has access to the tables it
    absolutely needs.
