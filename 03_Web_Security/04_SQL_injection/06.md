A **Prepared Statement** (also known as a **Parameterized Query**) is the most effective defense against SQL Injection. It
works by separating the SQL code from the data, ensuring the database treats user input strictly as a value and never as a
command.

Think of it like a **template**: you send the structure of the query to the database first, and then you "fill in the blanks"
later.

---

## How It Works: The Two-Step Process

In a standard (vulnerable) query, the code and data are mashed together into one string. In a Prepared Statement, they are
handled in two distinct phases:

### Phase 1: Preparation (The Template)

The application sends the SQL query structure to the database. Instead of the actual data, it uses **placeholders** (usually
`?` or `:name`).

- **Query:** `SELECT * FROM users WHERE username = ? AND status = ?;` The database parses, compiles, and optimizes this
  "template" before it ever sees your data.

### Phase 2: Execution (The Data)

The application binds the user's input to the placeholders and sends it to the database.

- **Data:** `['alice', 'active']`

Because the database has already compiled the SQL structure in Phase 1, it knows exactly what the command is. Even if the
data contains malicious code like `' OR 1=1--`, the database treats it as a literal, harmless string.

---

## Why It Stops SQL Injection

Imagine an attacker tries to log in with this username: `' OR '1'='1`

### Without Prepared Statements (Vulnerable):

The string is concatenated: `SELECT * FROM users WHERE username = '' OR '1'='1';` The database sees the `'` and interprets
the rest as a command. **The logic of the query changes.**

### With Prepared Statements (Secure):

The database expects a value for the first `?`. It looks for a user whose literal name is **exactly** the string
`' OR '1'='1`. It doesn't care about the quotes or the `OR` logic; it just treats the whole thing as a single piece of text.
Since no user is named that, the attack fails.

## Code Example: Node.js (mysql2)

If you are using the `mysql2` library in your Express backend, here is how you implement it:

```javascript
// DON'T DO THIS (Vulnerable)
const query = "SELECT * FROM users WHERE id = " + req.query.id;

// DO THIS (Secure Prepared Statement)
const userId = req.query.id;
db.execute(
  "SELECT * FROM users WHERE id = ?",
  [userId], // The data is passed in a separate array
  (err, results) => {
    console.log(results);
  }
);
```

## Key Benefits

| Feature         | Benefit                                                                                                           |
| :-------------- | :---------------------------------------------------------------------------------------------------------------- |
| **Security**    | Completely prevents SQL Injection by isolating data from logic.                                                   |
| **Performance** | If you run the same query many times (e.g., in a loop), the database only has to compile it once.                 |
| **Cleanliness** | You don't have to manually escape quotes (`'`) or worry about data types; the database driver handles it for you. |

---

## Limitations

- **Structural Elements:** You cannot use prepared statements for SQL identifiers like table names or column names (e.g.,
  `SELECT * FROM ?`). For those, you must use an "allow-list" in your code.
- **Complex Logic:** Very complex dynamic queries can sometimes be harder to write as templates, but modern ORMs (like
  Sequelize or Prisma) handle this beautifully for you.
