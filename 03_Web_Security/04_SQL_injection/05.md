Since you're interested in the **defensive side**, let's dive into how developers actually stop these attacks in the real
world. Security isn't just about one "magic fix"—it's about building layers of defense so that if one fails, others are there
to catch the threat.

---

## 1. The Gold Standard: Parameterized Queries

This is the single most effective defense against SQL Injection. Instead of building a string, you send the query and the
user data to the database **separately**.

- **How it works:** The database receives a template (the "parameterized" query) and treats the user input strictly as a
  value, never as part of the executable code.
- **Analogy:** It’s like a form at a bank. You can write whatever you want in the "Name" box, but the bank will never mistake
  your name for a command to "Give this person all the money."

```sql
-- The '?' acts as a placeholder that cannot be escaped
SELECT * FROM users WHERE username = ?;
```

---

## 2. Context-Aware Output Encoding

This is the primary defense against **XSS**. Since XSS happens when data is rendered in a browser, you must "neutralize" the
data before displaying it.

- **The Technique:** You convert characters that have special meaning in HTML/JavaScript into safe equivalents.
- **Example:** The character `<` becomes `&lt;` and `>` becomes `&gt;`.
- **Result:** If an attacker tries to inject `<script>`, the browser sees `&lt;script&gt;` and displays it as plain text
  instead of executing it.

---

## 3. Input Validation (The "Allow-list" Approach)

Never try to "clean" bad input (a "deny-list"). Attackers are too creative for that. Instead, define exactly what **good**
input looks like.

- **Strict Types:** If you expect a User ID, ensure the input is an integer.
- **Regex Validation:** If you expect a zip code, only allow 5 digits.
- **Dropdowns/Enums:** If a user is choosing a "Sort By" option, check their input against a list of allowed columns (e.g.,
  `price`, `date`, `rating`).

---

## 4. The Principle of Least Privilege (PoLP)

This is an architectural safeguard. You should assume that an injection _might_ happen and limit the damage the attacker can
do.

- **Database Level:** The web application's database user should not be an `admin` or `owner`. If the app only needs to read
  blog posts, give it `SELECT` permissions on the `posts` table—and nothing else.
- **OS Level:** The web server should run as a low-privileged user (like `www-data`) so that if a **Command Injection**
  occurs, the attacker can't access root files or install software.

---

## 5. Modern Security Headers

You can tell the browser to be more skeptical of the data it receives by using HTTP headers:

- **Content Security Policy (CSP):** Tells the browser exactly which scripts are allowed to run and where they can come from.
  This is a massive roadblock for XSS.
- **X-Frame-Options:** Prevents your site from being loaded in an iframe, stopping "Clickjacking" attacks.

---

## Which layer should you start with?

If you are working on an existing project, the priority is usually:

1.  **Parameterized Queries** (to stop the most destructive data theft).
2.  **Output Encoding** (to protect your users).
3.  **Least Privilege** (to minimize the "blast radius" of a successful hit).
