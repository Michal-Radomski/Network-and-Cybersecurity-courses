Running a React (Frontend) and Express (Backend) app with **PM2** is standard for production, but "securely" requires moving
beyond the default commands.

Here is your security-hardened roadmap for 2026.

---

## 1. Environment & Infrastructure

Before touching PM2, ensure your application environment is locked down.

- **NODE_ENV=production:** This is critical. Express uses this to disable verbose error messages (which leak your folder
  structure and code hints to attackers) and to enable performance caching.
- **The "Non-Root" Rule:** **Never** run PM2 as the root user. If an attacker exploits your Express app and it's running as
  root, they have full control over your server.
  - Create a dedicated user: `sudo useradd -m pm2user`
  - Switch to that user to start PM2.
- **Reverse Proxy (Nginx/Caddy):** Do not expose Express directly to the web. Use Nginx or Caddy to handle **SSL/TLS
  termination** and basic rate limiting. Your app should only listen on `localhost` (127.0.0.1).

---

## 2. Using an Ecosystem File

Don't use `pm2 start app.js`. Use an `ecosystem.config.js` file. It keeps your security configurations version-controlled and
consistent.

```javascript
// ecosystem.config.js
module.exports = {
  apps: [
    {
      name: "express-api",
      script: "./server/index.js",
      instances: "max", // Cluster mode (better availability)
      exec_mode: "cluster",
      env_production: {
        NODE_ENV: "production",
        PORT: 5000,
        // Load secrets from a .env file or vault, NOT hardcoded here
      },
      // Security: Limit how much memory the app can use to prevent DoS via memory leak
      max_memory_restart: "1G",
      // Do not restart if the app crashes too many times (prevents log-spamming DoS)
      max_restarts: 10,
      min_uptime: "5s",
    },
  ],
};
```

---

## 3. Handling Secrets Securely

**Never** commit your `.env` file or hardcode credentials in your code.

- **Runtime Injection:** Use the `--env` flag with PM2: `pm2 start ecosystem.config.js --env production`.
- **Encrypted Secrets:** In 2026, it is best practice to use a secret manager (like AWS Secrets Manager or HashiCorp Vault)
  or at least an encrypted `.env` file that is decrypted at runtime.

---

## 4. Secure Startup & Persistence

If your server reboots, you want your app to come back online automatically—but securely.

1.  **Generate Startup Script:** Run `pm2 startup`.
2.  **Follow Instructions:** It will provide a command starting with `sudo`. Run it.
3.  **Save State:** After your app is running, run `pm2 save`. This locks in the current process list.
4.  **Freeze Config:** Use `pm2 dump` to ensure the configuration is written to disk.

---

## 5. Express-Specific Hardening

PM2 only manages the process; the code inside must still defend itself.

- **Helmet.js:** Use this middleware to set secure HTTP headers (HSTS, CSP, etc.).
- **Rate Limiting:** Use `express-rate-limit` to prevent brute-force attacks on your API endpoints.
- **CORS:** Explicitly define which domains can access your Express API. **Never use `origin: '*'` in production.**

---

## Summary Checklist

| Security Layer    | Action                                                                    |
| :---------------- | :------------------------------------------------------------------------ |
| **Identity**      | Run PM2 as a limited `pm2user`, not `root`.                               |
| **Network**       | Bind Express to `127.0.0.1` and use Nginx for HTTPS.                      |
| **Monitoring**    | Use `pm2 monit` or `pm2-logrotate` to prevent logs from filling the disk. |
| **Configuration** | Use `NODE_ENV=production` in your `ecosystem.config.js`.                  |
| **Persistence**   | Use `pm2 startup` to survive reboots without manual intervention.         |

## Pro-Tip for PM2 Users

Since you are running your app with PM2, ensure your process is running under a **Low-Privilege User**. If a command
injection occurs while your app is running as `root`, the attacker has "God mode" over your entire server. If it's running as
a restricted user like `www-data`, their access is limited to just the app folder.

---

---

Running your PM2 processes under a dedicated, low-privilege user is one of the most effective ways to mitigate the "blast
radius" of a **Command Injection** attack. If an attacker manages to execute a shell command, they will only have the
permissions of that restricted user rather than full system control.

---

## 1. Create a Restricted System User

First, create a user that has no login shell and no sudo privileges. This user should only exist to run your application.

```bash
# Create a system user named 'nodeapp'
sudo useradd -m -s /bin/bash nodeapp
```

---

## 2. Set Directory Permissions

You need to ensure that the `nodeapp` user owns your application files so it can read and execute them, but it shouldn't have
access to sensitive system files.

```bash
# Change ownership of your project folder to the new user
sudo chown -R nodeapp:nodeapp /var/www/my-react-app
```

---

## 3. Set Up PM2 for the New User

To run PM2 as `nodeapp`, you should switch to that user account.

```bash
# Switch to the nodeapp user
sudo su - nodeapp

# Start your application using your ecosystem file
pm2 start /var/www/my-react-app/ecosystem.config.js --env production
```

---

## 4. Configure Secure Persistence (The Startup Script)

When the server reboots, you want the system to restart PM2 specifically **as the nodeapp user**.

1.  **Generate the command:** While still logged in as `nodeapp`, run: `pm2 startup`
2.  **Execute the output:** PM2 will output a command that looks like `sudo env PATH=...`. Copy that command.
3.  **Exit and Run:** Type `exit` to return to your admin user, then paste and run that command.
4.  **Save the state:** Switch back to the user (`sudo su - nodeapp`) and run `pm2 save`.

---

## 5. Security Hardening Tips for PM2

### Resource Limits

Attackers sometimes try to crash a server by consuming all its memory (a "Denial of Service" attack). You can prevent this in
your `ecosystem.config.js`:

```javascript
module.exports = {
  apps: [
    {
      name: "my-app",
      script: "./server.js",
      // Restarts the app if it exceeds 500MB to prevent system-wide slowdowns
      max_memory_restart: "500M",
      // Keeps logs from growing indefinitely and filling the disk
      max_size: "10M",
      retain: 5,
    },
  ],
};
```

### Log Management

By default, PM2 logs everything to disk. If an attacker triggers thousands of errors, they can fill your hard drive.

- **Install the logrotate module:** `pm2 install pm2-logrotate`
- This automatically compresses and deletes old logs.

### Disable the "Tui" in Production

While `pm2 monit` is great for debugging, avoid leaving monitoring tools or dashboards exposed to the public internet. Use a
reverse proxy (like Nginx) to ensure only the necessary API ports are reachable.

---

## Summary of the "Safe Run" Flow

1.  **App Level:** Use `helmet.js` and `express-rate-limit`.
2.  **Process Level:** Run PM2 under the `nodeapp` user with `max_memory_restart`.
3.  **System Level:** Use `pm2-logrotate` and an Nginx reverse proxy for SSL.
