Running a React (Frontend) and Express (Backend) app with **PM2** is standard for production, but "securely" requires moving
beyond the default commands.

Here is your security-hardened roadmap for 2026.

---

## 1. Environment & Infrastructure

Before touching PM2, ensure your application environment is locked down.

- **NODE_ENV=production:** This is critical. Express uses this to disable verbose error messages (which leak your folder
  structure and code hints to attackers) and to enable performance caching.
- **The "Non-Root" Rule:** **Never** run PM2 as the root user. If an attacker exploits your Express app and it's running as
  root, they have full control over your server.
  - Create a dedicated user: `sudo useradd -m pm2user`
  - Switch to that user to start PM2.
- **Reverse Proxy (Nginx/Caddy):** Do not expose Express directly to the web. Use Nginx or Caddy to handle **SSL/TLS
  termination** and basic rate limiting. Your app should only listen on `localhost` (127.0.0.1).

---

## 2. Using an Ecosystem File

Don't use `pm2 start app.js`. Use an `ecosystem.config.js` file. It keeps your security configurations version-controlled and
consistent.

```javascript
// ecosystem.config.js
module.exports = {
  apps: [
    {
      name: "express-api",
      script: "./server/index.js",
      instances: "max", // Cluster mode (better availability)
      exec_mode: "cluster",
      env_production: {
        NODE_ENV: "production",
        PORT: 5000,
        // Load secrets from a .env file or vault, NOT hardcoded here
      },
      // Security: Limit how much memory the app can use to prevent DoS via memory leak
      max_memory_restart: "1G",
      // Do not restart if the app crashes too many times (prevents log-spamming DoS)
      max_restarts: 10,
      min_uptime: "5s",
    },
  ],
};
```

---

## 3. Handling Secrets Securely

**Never** commit your `.env` file or hardcode credentials in your code.

- **Runtime Injection:** Use the `--env` flag with PM2: `pm2 start ecosystem.config.js --env production`.
- **Encrypted Secrets:** In 2026, it is best practice to use a secret manager (like AWS Secrets Manager or HashiCorp Vault)
  or at least an encrypted `.env` file that is decrypted at runtime.

---

## 4. Secure Startup & Persistence

If your server reboots, you want your app to come back online automatically—but securely.

1.  **Generate Startup Script:** Run `pm2 startup`.
2.  **Follow Instructions:** It will provide a command starting with `sudo`. Run it.
3.  **Save State:** After your app is running, run `pm2 save`. This locks in the current process list.
4.  **Freeze Config:** Use `pm2 dump` to ensure the configuration is written to disk.

---

## 5. Express-Specific Hardening

PM2 only manages the process; the code inside must still defend itself.

- **Helmet.js:** Use this middleware to set secure HTTP headers (HSTS, CSP, etc.).
- **Rate Limiting:** Use `express-rate-limit` to prevent brute-force attacks on your API endpoints.
- **CORS:** Explicitly define which domains can access your Express API. **Never use `origin: '*'` in production.**

---

## Summary Checklist

| Security Layer    | Action                                                                    |
| :---------------- | :------------------------------------------------------------------------ |
| **Identity**      | Run PM2 as a limited `pm2user`, not `root`.                               |
| **Network**       | Bind Express to `127.0.0.1` and use Nginx for HTTPS.                      |
| **Monitoring**    | Use `pm2 monit` or `pm2-logrotate` to prevent logs from filling the disk. |
| **Configuration** | Use `NODE_ENV=production` in your `ecosystem.config.js`.                  |
| **Persistence**   | Use `pm2 startup` to survive reboots without manual intervention.         |
