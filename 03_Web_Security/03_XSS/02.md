**Content Security Policy (CSP)** is an added layer of security that helps detect and mitigate certain types of attacks,
including Cross-Site Scripting (XSS) and data injection attacks.

Think of it as a **"Trusted Source List"** for the browser. You, as the developer, tell the browser exactly which domains,
scripts, and resources are allowed to execute on your page. If an attacker manages to inject a malicious script from an
unknown source, the browser will simply refuse to run it.

---

## How It Works

CSP is implemented by sending a `Content-Security-Policy` HTTP header from your server to the client's browser.

### The Basic Syntax

A policy consists of **directives**, each followed by one or more **sources**.

```http
Content-Security-Policy: default-src 'self'; script-src 'self' https://apis.google.com; object-src 'none';
```

- **`default-src 'self'`**: By default, only allow resources (images, scripts, fonts) from your own domain.
- **`script-src 'self' https://apis.google.com`**: Only allow scripts from your domain and Google's API domain.
- **`object-src 'none'`**: Disallow plugins like Flash.

---

## Key Directives

| Directive             | What it controls                                                            |
| :-------------------- | :-------------------------------------------------------------------------- |
| **`script-src`**      | Where JavaScript can be loaded from.                                        |
| **`style-src`**       | Where CSS stylesheets can be loaded from.                                   |
| **`img-src`**         | Where images can be loaded from.                                            |
| **`connect-src`**     | Which URLs you can connect to via `fetch`, `XMLHttpRequest`, or WebSockets. |
| **`frame-ancestors`** | Which sites can embed your page in an `<iframe>` (Prevents Clickjacking).   |

---

## Why CSP is a Powerhouse for XSS Prevention

In a standard environment, the browser executes any code it finds in the HTML. With a strict CSP, you can disable the two
biggest XSS vectors:

1.  **Inline Scripts:** By default, CSP blocks `<script>alert('hacked')</script>` written directly in the HTML.
2.  **`eval()`:** CSP blocks the use of the `eval()` function, which converts strings into executable code.

> **Pro Tip:** If you _must_ use an inline script, you can use a **Nonce** (number used once). You generate a random string
> on the server, include it in the CSP header, and add it to the script tag: `<script nonce="random123">`. The browser will
> only run inline scripts that have a matching nonce.

## Implementation in Express.js

The easiest way to set up a CSP in Node.js is using **Helmet**.

```javascript
import helmet from "helmet";

app.use(
  helmet.contentSecurityPolicy({
    directives: {
      "default-src": ["'self'"],
      "script-src": ["'self'", "https://trusted-scripts.com"],
      "img-src": ["'self'", "https://images.com"],
      "style-src": ["'self'", "'unsafe-inline'"], // 'unsafe-inline' allows internal <style> tags
      "upgrade-insecure-requests": [], // Automatically upgrades HTTP to HTTPS
    },
  })
);
```

## Report-Only Mode

Switching on a strict CSP can easily "break" your website if you forget to whitelist a legitimate font or analytics script.
To avoid this, you can use: `Content-Security-Policy-Report-Only`

In this mode, the browser **will not block** anything, but it **will send a JSON report** to a URL you specify every time a
policy violation occurs. This allows you to test your policy in production before enforcing it.

### Summary

- **CSP is a gatekeeper:** It defines what is "authorized" to run.
- **It stops XSS:** Even if an attacker injects a script, the browser won't execute it if the source isn't whitelisted.
- **Defense in Depth:** It shouldn't be your _only_ defense, but it’s one of the strongest "safety nets" you can have.
