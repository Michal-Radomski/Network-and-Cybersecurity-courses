Cross-Site Scripting (XSS) remains one of the most common web vulnerabilities. Because XSS can occur in different ways
(Reflected, Stored, or DOM-based), a single defense is rarely enough. Security experts recommend a **Defense-in-Depth**
strategy.

### 1. Output Encoding (The Primary Defense)

The most effective way to prevent XSS is to ensure that user-supplied data is never interpreted as active content (like
`<script>` tags) by the browser.

- **Context-Aware Encoding:** You must encode data based on where it will be placed in the HTML.
  - **HTML Body:** Convert `<` to `&lt;` and `>` to `&gt;`.
  - **HTML Attributes:** If putting data in `value="..."`, you must also encode quotes.
  - **JavaScript Context:** If placing data inside a `<script>` block, use Unicode escapes (e.g., `\u003C`).
- **Use Modern Frameworks:** Libraries like **React, Angular, and Vue** automatically encode data by default, which
  significantly reduces the risk of accidental XSS.

### 2. Content Security Policy (CSP)

A CSP is an HTTP response header that tells the browser which sources of content (scripts, styles, images) are trusted.

- **Restrict Script Sources:** You can tell the browser to only execute scripts from your own domain: `script-src 'self'`.
- **Disable Inline Scripts:** A strong CSP blocks `<script>alert(1)</script>` by default unless you provide a unique "nonce"
  (a random one-time token) for that specific script.
- **Prevent `eval()`:** CSP can disable dangerous functions like `eval()` that execute strings as code.

### 3. Input Validation and Sanitization

While encoding handles how data is _displayed_, validation handles how it is _received_.

- **Input Validation:** Use "allow-lists" rather than "block-lists." For example, if a field asks for a phone number, only
  accept digits and dashes. Reject anything else immediately.
- **HTML Sanitization:** If your application _must_ allow users to provide HTML (e.g., a blog editor), use a battle-tested
  library like **DOMPurify**. It "cleans" the HTML by stripping out dangerous tags like `<script>` and attributes like
  `onerror`.

### 4. Use Secure Cookie Flags

If an XSS attack succeeds, the attacker’s primary goal is often to steal the user’s session cookie. You can stop this using:

- **HttpOnly:** This flag prevents JavaScript from accessing the cookie via `document.cookie`. Even if an attacker executes a
  script, they cannot steal the session token.
- **SameSite:** Setting this to `Strict` or `Lax` helps prevent the cookie from being sent in cross-site requests, mitigating
  related attacks like CSRF.

---

### Summary Table: Countermeasures by Context

| Defense Layer       | Purpose                        | Key Technique                                     |
| :------------------ | :----------------------------- | :------------------------------------------------ |
| **Server-Side**     | Stop bad data from entering.   | Input Validation (Allow-lists).                   |
| **Rendering**       | Prevent data from being code.  | Context-aware Output Encoding.                    |
| **Browser Policy**  | Limit damage if XSS occurs.    | Content Security Policy (CSP).                    |
| **Data Protection** | Prevent session theft.         | `HttpOnly` and `Secure` cookie flags.             |
| **DOM-Specific**    | Prevent client-side injection. | Use **Trusted Types** API and avoid `.innerHTML`. |

> **Pro-Tip:** Never try to write your own XSS filter using Regular Expressions. Attackers have thousands of ways to bypass
> simple string replacements (like using different encodings or malformed tags). Always use standard libraries and
> browser-level security features.
