In the context of web security, a **nonce** stands for "number used once." It is a unique, cryptographically strong, random
string generated by the server for every single page request.

### How Nonces Prevent XSS

When you implement a **Content Security Policy (CSP)**, you typically block all inline scripts to prevent attackers from
injecting malicious code. However, sometimes your app _needs_ a specific inline script to run. By attaching a `nonce` to your
script tag, you tell the browser: "I trust this specific script because it has the secret password for this request."

Even if an attacker manages to inject a `<script>` tag, they won't know the unique nonce for that specific request, so the
browser will refuse to execute their code.

---

### Implementation: Express.js (The Server)

The server's job is to generate the nonce, include it in the CSP header, and pass it to the frontend. We use the `crypto`
module for generation and `helmet` to manage the CSP headers.

```javascript
const express = require("express");
const crypto = require("crypto");
const helmet = require("helmet");

const app = express();

app.use((req, res, next) => {
  // 1. Generate a random base64 nonce
  res.locals.nonce = crypto.randomBytes(16).toString("base64");
  next();
});

app.use(
  helmet.contentSecurityPolicy({
    directives: {
      scriptSrc: ["'self'", (req, res) => `'nonce-${res.locals.nonce}'`],
      // Other directives...
    },
  })
);

app.get("*", (req, res) => {
  // 2. Send the nonce to your HTML template or React app
  res.render("index", { nonce: res.locals.nonce });
});
```

---

### Implementation: React (The Frontend)

In a typical Single Page Application (SPA), your React code lives in a bundled JS file (which is allowed by `'self'`).
However, you often need nonces for **inline styles** or **third-party analytics** injected into your `index.html`.

#### 1. Adding to `index.html`

If you are server-side rendering (SSR) or using a template engine, you inject the nonce directly into the script tags:

```html
<script nonce="<%= nonce %>">
  window.__INITIAL_DATA__ = { user: "Gemini" };
</script>
```

#### 2. Accessing in React Components

If you are dynamically injecting styles or scripts within a React component, you need to ensure that the `nonce` generated by
the server is accessible (usually via a global variable or a meta tag).

```javascript
import React from "react";

const MyComponent = ({ nonce }) => {
  return (
    <div>
      <h1>Hello World</h1>
      {/* React doesn't support the nonce attribute on style/script tags directly 
          via JSX in some older versions, so check your version requirements. */}
      <style property="stylesheet" nonce={nonce}>
        {`body { background: #f0f0f0; }`}
      </style>
    </div>
  );
};
```

---

### Critical Considerations

- **Never Reuse Nonces:** A nonce must be regenerated on **every** request. If a nonce is static, an attacker can find it and
  bypass your CSP.
- **Avoid 'unsafe-inline':** The whole point of using a nonce is to remove `'unsafe-inline'` from your `script-src`
  directive.
- **Strict CSP:** For modern apps, look into the **Strict CSP** approach, which uses nonces along with `strict-dynamic` to
  simplify management of third-party scripts.

> **Security Note:** If you are using a purely static site (e.g., hosted on S3 or Netlify), you cannot use nonces because
> there is no server to generate a unique string per request. In those cases, use **Subresource Integrity (SRI)** hashes
> instead.
