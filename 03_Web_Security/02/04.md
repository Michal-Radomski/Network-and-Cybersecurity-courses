Cross-Site Request Forgery (CSRF) is a vulnerability where an attacker tricks a victim's browser into performing an unwanted
action on a different website where the victim is currently authenticated.

When this attack targets **GET services**, it is particularly dangerous because GET requests are designed to be "idempotent"
(safe and non-state-changing) by standard web conventions, yet developers often use them to trigger actions for convenience.

---

## How the Attack Works

The core of the attack relies on two facts:

1.  **Automatic Cookie Handling:** Browsers automatically include ambient credentials (like session cookies) with every
    request made to a specific domain.
2.  **Predictability:** The attacker knows exactly what the URL and parameters look like for a specific action.

### The Mechanism

1.  **The Setup:** A user logs into their bank (`bank.com`) and receives a session cookie.
2.  **The Trap:** While the session is active, the user visits a malicious site (or a site with a malicious ad) controlled by
    an attacker.
3.  **The Trigger:** The malicious site contains a hidden element (like an image tag) that points to the bank's GET-based
    service.
4.  **The Execution:** The browser attempts to "load" the image, sending the request to the bank along with the user's valid
    session cookie. The bank sees a legitimate session and processes the request.

---

## Why GET Services are Vulnerable

In a properly designed system, GET requests should only retrieve data. However, if a developer uses a GET request to change
state (e.g., deleting a profile, transferring funds, or changing a password), it becomes trivial to exploit.

### Example Scenario

Imagine a service that allows a user to delete their account via a GET request:
`https://social-app.com/settings/delete?confirm=true`

An attacker can embed this into an innocent-looking HTML tag on a completely different website:

```html
<img src="https://social-app.com/settings/delete?confirm=true" width="0" height="0" />
```

Because `<img>`, `<script>`, and `<iframe>` tags do not follow **Same-Origin Policy (SOP)** restrictions for making requests,
the browser will fire that GET request the moment the page loads. The user never sees it happen.

---

## Comparison: GET vs. POST in CSRF

While POST requests can also be used in CSRF attacks (usually via hidden forms and JavaScript's `autosubmit`), GET requests
are significantly easier to exploit:

| Feature               | GET-based CSRF                         | POST-based CSRF                           |
| :-------------------- | :------------------------------------- | :---------------------------------------- |
| **Ease of Execution** | Extremely High (requires a single tag) | High (requires a hidden form + JS)        |
| **Interaction**       | Zero interaction required              | Zero interaction required                 |
| **Bypass Mechanisms** | Harder to block with basic firewalls   | Often filtered by stricter security rules |
| **Visibility**        | Can be hidden in images/links          | Usually requires a script-driven form     |

---

## Prevention Strategies

### 1. Follow REST Principles

The most fundamental fix is to ensure that **GET requests never change state.** Any action that creates, updates, or deletes
data should strictly use POST, PUT, or DELETE methods.

### 2. Anti-CSRF Tokens

This is the gold standard. The server generates a unique, cryptographically strong, and unpredictable token for the user's
session.

- The token must be included in any state-changing request.
- Since an attacker cannot "read" the token from the bank's site (due to Same-Origin Policy), they cannot include it in their
  forged request.

### 3. SameSite Cookie Attribute

Modern browsers support the `SameSite` attribute on cookies:

- **`SameSite=Strict`**: The cookie is only sent if the request originates from the same site where the cookie was set.
- **`SameSite=Lax`**: (Modern default) Cookies are withheld on cross-site sub-requests (like images) but sent when a user
  navigates to the URL (like clicking a link).

### 4. Custom Headers

Requiring a custom HTTP header (like `X-Requested-With`) for sensitive actions can stop CSRF because browsers cannot add
custom headers to cross-origin requests without a preflight check (CORS).
