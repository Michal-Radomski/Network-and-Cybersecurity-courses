While GET-based CSRF is often easier to pull off with a simple `<img>` tag, **POST-based CSRF** is arguably more common in
modern web applications where state-changing actions (like changing a password or transferring funds) are correctly
restricted to POST methods.

Since an attacker cannot trigger a POST request using a simple image or link, they must rely on **hidden forms** and
**JavaScript**.

---

## How the POST Attack Works

The core logic remains the same: the attacker exploits the browser's habit of automatically attaching session cookies to
requests made to the target domain.

### 1. The Setup

The attacker identifies a vulnerable POST endpoint, such as: `POST https://bank.com/transfer`
`Payload: amount=1000&to=attacker_account`

### 2. The Malicious Page

The attacker hosts a webpage (or injects code into a compromised site) containing a hidden HTML form. This form is configured
to send data to the target bank.

```html
<form id="attack-form" action="https://bank.com/transfer" method="POST">
  <input type="hidden" name="amount" value="1000" />
  <input type="hidden" name="to" value="attacker_account" />
</form>

<script>
  // Automatically submit the form as soon as the victim loads the page
  document.getElementById("attack-form").submit();
</script>
```

### 3. Execution

- The victim, logged into `bank.com`, visits the attacker's site.
- The JavaScript executes immediately and invisible to the user.
- The browser sends the POST request to the bank, including the victim's authentication cookies.
- The bank processes the transfer, thinking the user initiated it.

---

## Why POST isn't a "Security Feature"

Many developers mistakenly believe that moving an action from GET to POST "fixes" CSRF. It does not. It only changes the
**delivery mechanism**.

While browsers block cross-origin **reading** of data (Same-Origin Policy), they traditionally allowed cross-origin
**sending** of data via forms. This is known as a "simple request" in the context of CORS (Cross-Origin Resource Sharing).

---

## Defenses Against POST CSRF

### 1. Anti-CSRF Tokens (Synchronizer Token Pattern)

The server embeds a unique, secret token in the hidden fields of legitimate forms.

- **Legit request:** `<input type="hidden" name="csrf_token" value="abc123xyz">`
- **Attack request:** The attacker doesn't know the token, so their form submission fails server-side validation.

### 2. SameSite Cookie Attribute

This is the most effective modern defense. By setting cookies to `SameSite=Lax` or `SameSite=Strict`, the browser will refuse
to send the cookie during a POST request that originates from a different site.

### 3. Double Submit Cookies

If the server is stateless (like some APIs), it can send a random value in a cookie and require the client to send that same
value in a hidden form field or header. The attacker can't read the cookie to copy the value into the form, so the "double
submit" check fails.

### 4. Verifying Origin/Referer Headers

The server can check the `Origin` or `Referer` headers to ensure the request came from a trusted domain. However, these
headers can sometimes be stripped by privacy tools or spoofed in very specific edge cases, so they are usually used as a
secondary defense.

## Summary Table

| Feature                 | GET CSRF                                    | POST CSRF                          |
| :---------------------- | :------------------------------------------ | :--------------------------------- |
| **Primary Tool**        | HTML Tags (`<img>`, `<script>`)             | Hidden Forms + Auto-submit JS      |
| **Payload Visibility**  | Visible in URL logs/history                 | Hidden in the request body         |
| **Complexity**          | Extremely Low                               | Low to Moderate                    |
| **SameSite Protection** | `Lax` allows some GETs; `Strict` blocks all | Blocked by both `Lax` and `Strict` |
