In the JavaScript ecosystem, the implementation of CSRF countermeasures depends on whether you are working on the
**Server-Side** (Node.js/Express) or the **Client-Side** (React, Vue, or Vanilla JS).

The most common approach is the **Synchronizer Token Pattern**.

---

## 1. Server-Side: Express.js (Node.js)

In modern Node.js development, the `csurf` package was historically the standard, but it is now deprecated. Most developers
now use the `csrf-csrf` or `double-csrf` libraries, or manage tokens via the **SameSite** cookie attribute.

### Example using `csrf-csrf`

This library generates a token, stores it in an encrypted cookie, and validates it against a header in incoming requests.

```typescript
import express from "express";
import cookieParser from "cookie-parser";
import { doubleCsrf } from "csrf-csrf";

const app = express();

// 1. Setup middleware
app.use(express.json());
app.use(cookieParser("your-secret-key"));

const { generateToken, doubleCsrfProtection } = doubleCsrf({
  getSecret: () => "your-secret-key",
  cookieName: "x-csrf-token",
  cookieOptions: {
    sameSite: "lax", // Protects against most cross-site POSTs
    secure: true,
  },
});

// 2. Endpoint to give the client a token
app.get("/api/csrf-token", (req, res) => {
  const token = generateToken(req, res);
  res.json({ token });
});

// 3. Apply protection to state-changing routes
app.post("/api/transfer-money", doubleCsrfProtection, (req, res) => {
  res.json({ message: "Money transferred safely!" });
});
```

## 2. Client-Side: Fetch / Axios

Once your server provides a token, the frontend must intercept it and include it in the headers of every sensitive request.

### Using Fetch (Vanilla JS/TS)

```typescript
async function performAction() {
  // 1. Fetch the token from the server
  const response = await fetch("/api/csrf-token");
  const { token } = await response.json();

  // 2. Include the token in the headers of your POST request
  const result = await fetch("/api/transfer-money", {
    method: "POST",
    headers: {
      "Content-Type": "application/json",
      "x-csrf-token": token, // The server looks for this specific header
    },
    body: JSON.stringify({ amount: 100, to: "friend_account" }),
  });

  return result.json();
}
```

## 3. The "No-Library" Modern Approach (SameSite)

If you don't want to manage tokens manually, you can lean heavily on modern browser security. By setting your session cookies
to `SameSite=Strict`, you effectively kill CSRF for most applications.

```typescript
// Example of setting a secure session cookie in Express
res.cookie("sessionId", "12345", {
  httpOnly: true, // Prevents XSS from reading the cookie
  secure: true, // Only sent over HTTPS
  sameSite: "strict", // Browser won't send this cookie if the request is cross-site
});
```

### Which one should you use?

- **Use `SameSite=Lax` or `Strict`** as your first line of defense. It's built into the browser and requires no extra code.
- **Use Anti-CSRF Tokens** if you need to support older browsers or if your API is used in complex cross-origin scenarios
  where `SameSite` might be too restrictive or insufficient.
