To understand **CSRF**, you first have to understand the mechanics of a **Cross-Site Request**. While the names sound
similar, one is a standard web function and the other is a specific security exploit.

---

## 1. What is a Cross-Site Request?

A **Cross-Site Request** is simply any request made by a browser from one "origin" (domain) to another.

The modern web is built on these. Every time a website loads a font from Google, a video from YouTube, or an image from a
different server, your browser is making a cross-site request.

### The Mechanism: Cookies

The "danger" in these requests comes from how browsers handle **Cookies**.

- Browsers are designed to be helpful. If you have a login cookie for `Facebook.com`, your browser "remembers" you.
- If you are on `NewsSite.com` and it contains a Facebook "Like" button, your browser sends a request to Facebook to load
  that button.
- **Critically:** The browser automatically attaches your Facebook login cookie to that request so Facebook knows who you
  are.

---

## 2. What is CSRF (Cross-Site Request Forgery)?

**CSRF** is an attack that tricks your browser into using your active login session to perform an action you didn't intend to
do. It is often called "session riding."

### The Anatomy of an Attack

For a CSRF attack to work, three things must be true:

1.  **A Relevant Action:** The target site has an action the attacker wants to trigger (e.g., "Transfer Money" or "Change
    Password").
2.  **Cookie-Based Session Management:** The site identifies the user solely through cookies.
3.  **No Unpredictable Parameters:** The attacker can guess exactly what the URL or form data needs to look like to trigger
    the action.

### A Typical Example:

Imagine you are logged into your bank at `bank.com`. In another tab, you visit a malicious site, `evil-site.net`.

1.  **The Trap:** `evil-site.net` contains a hidden piece of code, such as:
    `<img src="https://bank.com/transfer?amount=1000&to=attacker_account" style="display:none;">`
2.  **The Request:** Your browser tries to load that "image." Since it's a request to `bank.com`, the browser automatically
    attaches your bank login cookies.
3.  **The Execution:** The bank's server receives the request, sees your valid cookie, and thinks _you_ clicked a button to
    transfer money. It processes the transaction.

---

## 3. The Difference at a Glance

| Feature              | Cross-Site Request                  | CSRF (The Attack)                               |
| :------------------- | :---------------------------------- | :---------------------------------------------- |
| **Nature**           | A standard web technology.          | A security vulnerability/exploit.               |
| **Intent**           | Usually benign (loading resources). | Malicious (unauthorized actions).               |
| **User Interaction** | Part of normal browsing.            | Happens in the background without user consent. |
| **Data Flow**        | Usually "Read" (fetching an image). | Usually "Write" (changing data/state).          |

---

## 4. How is CSRF Prevented?

Because this was a major issue in the early 2000s, modern web development has several standard "shields":

- **Anti-CSRF Tokens:** The server gives your browser a "secret handshake" (a long, random string) every time you load a
  form. When you submit the form, you must send that string back. An attacker on a different site won't know your secret
  string, so their forged request will be rejected.
- **SameSite Cookie Attribute:** This is a browser setting that tells the browser: _"Only send this cookie if the user is
  actually on my website."_ If the request comes from a cross-site source, the cookie is stripped away, rendering the request
  unauthenticated.
- **Custom Headers:** Requiring a specific header (like `X-Requested-With`) that can only be sent via JavaScript. Because of
  the **Same-Origin Policy**, a malicious site usually cannot force your browser to add custom headers to a cross-site
  request.
