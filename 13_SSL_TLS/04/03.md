A **cipher suite** is a standardized cryptographic recipe book. It is a single, structured string of text that tells a
computer exactly which security algorithms to use when establishing a secure network connection (like HTTPS).

Whenever you connect to a secure website, your browser and the website's server must agree on a cipher suite to ensure your
data stays private and untampered with.

---

## The Anatomy of a Cipher Suite

To understand how they work, it helps to break one down. Because TLS 1.3 is the modern, secure standard, its cipher suites
are streamlined and look like this:

### `TLS_AES_256_GCM_SHA384`

If we dissect this string, it reveals three essential components:

1. **`TLS`**: This simply identifies the protocol being used (Transport Layer Security).
2. **`AES_256_GCM`**: This is the **Symmetric Encryption Algorithm**. It defines how the actual data (like your passwords or
   credit card numbers) will be scrambled and unscrambled after the connection is established.

- _AES_256_ means it uses the Advanced Encryption Standard with a 256-bit key.
- _GCM_ (Galois/Counter Mode) ensures that the data isn't altered in transit (providing both confidentiality and integrity).

3. **`SHA384`**: This is the **Hashing Algorithm** (specifically SHA-3). It acts as a digital fingerprinting tool used for
   key generation and to verify that the data hasn't been maliciously modified.

---

## How TLS 1.3 Changed the Game

In older versions of TLS (like TLS 1.2), cipher suites were much longer and more complicated because they also had to define
the _Key Exchange_ and _Certificate Signing_ algorithms. A TLS 1.2 suite looked like this:
`TLS_ECDHE_RSA_WITH_AES_128_GCM_SHA256`

TLS 1.3 completely overhauled this system for two major reasons:

- **Massive Cleanup:** TLS 1.3 banned older, vulnerable algorithms (like RC4, DES, and MD5). Instead of hundreds of
  potentially weak combinations, TLS 1.3 relies on just **5 ultra-secure cipher suites**.
- **Separation of Concerns:** In TLS 1.3, the negotiation for _how_ keys are exchanged (e.g., using ECDHE) happens
  independently of the cipher suite. This allowed the cipher suite strings to become much shorter and faster to process.

---

## The 5 Standard TLS 1.3 Cipher Suites

Today, almost all modern secure internet traffic relies on these five combinations:

| Cipher Suite                   | Security Level     | Common Use Case                                                             |
| ------------------------------ | ------------------ | --------------------------------------------------------------------------- |
| `TLS_AES_256_GCM_SHA384`       | **Military-Grade** | Standard high-security web traffic.                                         |
| `TLS_CHACHA20_POLY1305_SHA256` | **High & Fast**    | Optimized for mobile devices or hardware without built-in AES acceleration. |
| `TLS_AES_128_GCM_SHA256`       | **Strong**         | Standard web traffic balancing speed and security.                          |
| `TLS_AES_128_CCM_SHA256`       | **Strong**         | Often used in constrained environments (like IoT devices).                  |
| `TLS_AES_128_CCM_8_SHA256`     | **Lightweight**    | Designed specifically for low-power IoT networks.                           |

> 🤝 **The Handshake Agreement:** During the initial connection (the handshake), your browser sends a list of its supported
> cipher suites to the server. The server looks at the list, picks the most secure one it also supports, and says, _"Let's
> use this recipe."_ If they can't agree on a suite, the connection fails.
