While RSA is heavily used in HTTPS for digital signatures and authentication, it is **not** used to encrypt the actual data
(like passwords, credit card numbers, or webpage content) sent back and forth.

Instead, HTTPS uses a hybrid approach: **RSA (or ECC)** is used for identity verification and exchanging keys, while **AES
(Symmetric Encryption)** is used to encrypt the bulk of the data.

Here are the main reasons why RSA is not used for data encryption:

---

### 1. RSA is Too Slow and Resource-Intensive

RSA is based on complex modular exponentiation of massive numbers (usually 2048 or 4096 bits long). This requires a lot of
CPU power.

- **Symmetric encryption algorithms** like AES use fast, bit-level shuffling and substitutions that can be executed directly
  by specialized instructions on modern computer processors (AES-NI).
- Attempting to encrypt a 100MB file or a video stream with RSA would cripple the performance of both the web server and your
  smartphone or laptop.

### 2. File Size Limitations

Mathematically, an RSA key can only encrypt a piece of data that is _smaller_ than the key size itself.

- A **2048-bit RSA key** can only encrypt a maximum of **245 bytes** of data at a time.
- If you wanted to encrypt a simple 5MB image, your browser would have to chop the image into tens of thousands of tiny
  blocks, encrypt each block individually with RSA, and stitch them back together. This is highly inefficient and creates
  security vulnerabilities.

### 3. Lack of Forward Secrecy

Historically, RSA was used in HTTPS to encrypt the "Pre-Master Secret" (the key used to generate the session keys). However,
this created a massive security flaw: **if an attacker recorded all your encrypted internet traffic today, and managed to
steal the server's RSA private key a year from now, they could decrypt _all_ the past traffic they recorded.**

Modern HTTPS (TLS 1.3) completely banned this method. It now mandates **Diffie-Hellman (DH or ECDHE)** for key exchange,
which provides **Perfect Forward Secrecy (PFS)**. This ensures that even if a server's private key is stolen in the future,
past sessions remain completely secure.

---

## How It Works Instead: The Hybrid System

To get the best of both worlds, HTTPS uses a hybrid encryption system:

1. **Asymmetric Encryption (The Handshake):** Your browser and the server use certificates (authenticated via RSA or ECC) and
   Diffie-Hellman to securely agree on a temporary, shared "session key" without anyone else intercepting it.
2. **Symmetric Encryption (The Data Transfer):** Once that shared key is safely established, both sides switch entirely to
   **AES encryption** to scramble and unscramble the actual data being sent. The key is thrown away as soon as you close the
   tab.
