While you can technically define almost any valid elliptic curve for cryptography, the internet relies on a small,
standardized pool of curves to ensure security, high performance, and universal compatibility.

In TLS 1.3, these curves are referred to as **Named Groups**.

---

## The Standard Curves Used in TLS

The vast majority of modern TLS traffic relies on three main elliptic curves:

### 1. X25519 (The Modern Favorite)

Introduced to TLS relatively recently, **X25519** has rapidly become the preferred choice for modern web browsers and
servers.

- **Design:** It uses a specific curve shape called a _Montgomery curve_.
- **Why it’s popular:** It was designed from the ground up by cryptographer Daniel J. Bernstein to be incredibly fast and
  highly resistant to specific types of hacking attacks (like side-channel attacks). It is also simpler to implement securely
  in software than older curves.

### 2. P-256 (secp256r1) (The Institutional Standard)

Created by the National Institute of Standards and Technology (NIST), **P-256** is the most widely supported curve in the
world.

- **Design:** It is a _Weierstrass curve_.
- **Why it’s popular:** Because it is backed by NIST, it is a requirement for many government, banking, and legacy corporate
  systems. Almost 100% of TLS-compliant devices support it.

### 3. P-384 (secp384r1)

Also created by NIST, **P-384** is the heavier, high-security sibling of P-256.

- **Why it’s used:** It offers a higher level of security (equivalent to a 192-bit symmetric key, compared to P-256's 128-bit
  security). It is used when protecting ultra-sensitive data, though it requires slightly more processing power.

## A Real-World Example: How It Looks in Action

When your browser connects to a website using TLS 1.3, it offers these curves to the server during the handshake. If you were
to look at a network packet capture (using a tool like Wireshark) during a TLS 1.3 handshake, you would see the curve
negotiation explicitly listed.

### The Client Hello Packet

Inside the `supported_groups` extension of the **Client Hello** message, your browser sends an ordered list of curves it
knows how to use:

```text
Extension: supported_groups (len=10)
    Supported Groups List (5 groups)
        Supported Group: x25519 (0x001d)
        Supported Group: secp256r1 (0x0017)
        Supported Group: secp384r1 (0x0018)

```

### The Server Hello Packet

The server reads this list, picks its preferred curve (usually `x25519` if both support it), and responds in the **Server
Hello** by sending its public key share calculated _specifically_ on that curve:

```text
Extension: key_share (len=38)
    Key Share Entry: Group: x25519
        Key Exchange length: 32
        Key Exchange: 9a3f2b8c... [32 bytes of public key data on the X25519 curve]

```

By agreeing on `x25519`, both your browser and the server instantly know the mathematical formulas and fixed starting
coordinates they need to use to calculate your secure session keys.
