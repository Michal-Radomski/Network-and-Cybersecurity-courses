**X25519** is a specific, highly optimized mathematical curve used for **Elliptic Curve Diffie-Hellman (ECDH)** key
exchanges.

Designed by renowned cryptographer Daniel J. Bernstein (djb) and released in 2005, it was created to fix the performance and
security flaws found in older, government-designed curves. Today, it is the default choice for securing connections in TLS
1.3, SSH, Signal, WhatsApp, and Tor.

---

## What Makes X25519 Different?

To understand why X25519 took over the internet, you have to look at the curve it replaced: NIST's **P-256**. While P-256 is
secure, it is mathematically complex and difficult to implement perfectly in software.

X25519 was built from scratch with three main goals: **Speed, Simplicity, and Bulletproof Security.**

### 1. The Montgomery Curve Shape

Most traditional elliptic curves use the **Weierstrass** equation ($y^2 = x^3 + ax + b$). X25519 instead uses a **Montgomery
curve** equation:

$$y^2 = x^3 + 486662x^2 + x$$

The big trick with a Montgomery curve is that computers can perform the "point bouncing" math using **only the
X-coordinate**, completely ignoring the Y-coordinate. Cutting the math criteria in half makes calculations blazing fast.

### 2. Side-Channel Attack Immunity

When a computer crunches cryptographic numbers, it leaks tiny amounts of physical information—like subtle variations in how
much power the CPU draws or exactly how many milliseconds a calculation takes. Hackers can use precise timers to measure
these leaks and steal keys (known as a **Side-Channel Attack**).

Older curves required different mathematical operations depending on whether a bit in the private key was a `0` or a `1`.
X25519 uses a sequence called the **Montgomery Ladder**. Every single operation takes the exact same amount of time and
power, regardless of what the private key looks like. To an attacker measuring the CPU, the math looks completely uniform and
uncrackable.

### 3. No "Twist" Vulnerabilities (Twist Security)

Every elliptic curve has a mathematical shadow called its "quadratic twist." If a hacker sends a fake, malicious public key
that isn't actually on the real curve, an older system might accidentally slip onto the "twist" curve, leaking the private
key.

X25519 was intentionally designed to be **twist-secure**. Even if an attacker passes a completely invalid or malicious point
to the algorithm, the math safely handles it without exposing any private data.

## Behind the Name: Curve25519 vs. X25519

You will often see these two terms used interchangeably, but there is a subtle technical distinction:

- **Curve25519:** This is the name of the _underlying elliptic curve_ itself. The number **25519** comes from the math: the
  curve operates over a prime number that is $2^{255} - 19$.
- **X25519:** This is the name of the _specific key exchange protocol_ (Diffie-Hellman) that uses that curve.

## Summary of Benefits

| Feature            | Older Curves (like P-256)                                 | X25519                                            |
| ------------------ | --------------------------------------------------------- | ------------------------------------------------- |
| **Speed**          | Fast, but complex                                         | Blazing fast (optimized for modern CPUs)          |
| **Math Focus**     | Requires both X and Y coordinates                         | Uses X-coordinate only                            |
| **Timing Attacks** | Vulnerable if poorly written                              | Inherently immune by design                       |
| **Trust**          | Designed by the NSA (historically viewed with skepticism) | Designed openly by public academic cryptographers |
