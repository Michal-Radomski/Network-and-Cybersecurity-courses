Establishing a TLS 1.3 session is designed to be incredibly fast and secure. Unlike older versions (like TLS 1.2), which
required two full round-trips to set up a connection, **TLS 1.3 achieves a complete handshake in just one round-trip (1
RTT)**.

Here is exactly how a TLS 1.3 session is established, step-by-step.

---

## The 1-RTT Handshake Process

### 1. Client Hello (The Opening Move)

The client starts the conversation by sending a **Client Hello** message to the server. This message contains:

- **Supported Cipher Suites:** A list of cryptographic algorithms the client supports. (TLS 1.3 only supports highly secure,
  modern AEAD ciphers).
- **Key Share:** The client guesses which key exchange algorithm the server will prefer, generates its own public key share
  using that algorithm (usually a Diffie-Hellman variant), and sends it right away.

### 2. Server Response (The Heavy Lifting)

The server receives the Client Hello, processes the key share, and responds with its own single flight of messages:

- **Server Hello:** The server selects the cipher suite it wants to use.
- **Key Share:** The server sends its own cryptographic public key share. At this exact point, **both sides can calculate the
  shared secret** (master key) used to encrypt all future communications.
- **Encrypted Extensions:** This contains protocol modifiers that don't need to be public. Because the key share happened,
  this and everything following it is already encrypted!
- **Certificate:** The server sends its digital certificate to prove its identity.
- **Certificate Verify:** A digital signature created by the server using its private key, proving it actually owns the
  certificate it just sent.
- **Finished:** A cryptographic check ensuring the handshake hasn't been tampered with.

### 3. Client Finished (The Wrap-Up)

The client verifies the server’s certificate and signature. If everything checks out, the client sends its own **Finished**
message to confirm that its cryptographic handshake is complete and secure.

From this point forward, application data (like the website's HTML, images, etc.) flows freely over a fully encrypted
channel.

---

## Key Differences: TLS 1.2 vs. TLS 1.3

| Feature             | TLS 1.2                           | TLS 1.3                                      |
| ------------------- | --------------------------------- | -------------------------------------------- |
| **Handshake Speed** | 2 RTT (Two round-trips)           | 1 RTT (One round-trip)                       |
| **0-RTT Mode**      | No                                | Yes (For returning visitors)                 |
| **Privacy**         | Handshake is mostly unencrypted   | Handshake is encrypted almost immediately    |
| **Legacy Ciphers**  | Allowed (RSA key exchange, SHA-1) | Banned (Only secure, modern ciphers allowed) |

> 💡 **What is 0-RTT?** TLS 1.3 also supports a "Zero Round-Trip Time" mode. If a client has connected to the server
> recently, it can use cached session information to send encrypted application data (like an HTTP request) in its very first
> message, making the connection instant.
