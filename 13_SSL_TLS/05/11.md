Choosing between `restart` and `reload` comes down to whether you need to bring the web server completely down to apply
changes, or if you just want it to pick up new configuration files on the fly.

Here is the breakdown of the differences and when to use each.

---

### The Short Answer

- Use **`systemctl reload apache2`** when you only changed configuration files (like adding a new virtual host or changing a
  directory setting). It has **zero downtime**.
- Use **`systemctl restart apache2`** when you made major changes, like installing a new Apache module, changing ports, or
  updating the SSL certificate configuration. It involves a **brief moment of downtime**.

---

### Detailed Differences

| Feature              | `systemctl reload apache2`                                                              | `systemctl restart apache2`                                                                 |
| -------------------- | --------------------------------------------------------------------------------------- | ------------------------------------------------------------------------------------------- |
| **What it does**     | Instructs Apache to read the new configuration files without stopping the main process. | Completely stops the Apache service and starts it back up from scratch.                     |
| **Downtime**         | **None.** Active user connections are not dropped.                                      | **Brief downtime.** Current connections are cut off, and new ones must wait until it boots. |
| **Process ID (PID)** | The main process ID stays the same; only worker processes are recycled.                 | The main process is killed, and a brand new PID is assigned.                                |
| **When to use**      | Tweaking `.conf` files, enabling sites (`a2ensite`), changing rewrite rules.            | Enabling Apache modules (`a2enmod`), changing network ports, major updates.                 |

---

### When to use `reload` (The Gentle Approach)

`reload` sends a `HUP` (Hangup) signal to Apache. The main server process stays alive, tells the current "worker" processes
to finish serving their existing requests, and spawns new worker processes that read the updated configuration files.

**Best used for:**

- Enabling or disabling a website configuration (`a2ensite yoursite.conf` / `a2dissite`).
- Modifying block configurations (e.g., changing a `Directory` permission or adding a redirect).
- Fixing a typo in a configuration file (after checking it with `apache2ctl configtest`).

---

### When to use `restart` (The Hard Reset)

`restart` completely kills the Apache process and all its workers immediately. It then initializes the entire application
again.

**Best used for:**

- Installing or enabling a new Apache module (e.g., `a2enmod rewrite` or `a2enmod ssl`).
- Changing the ports Apache listens on (e.g., editing `ports.conf` from port 80 to 8080).
- Upgrading the Apache software itself.
- Clearing out stuck, frozen, or malfunctioning worker processes if the server is acting up.

> 💡 **Pro-Tip:** Before running either command, always run **`sudo apache2ctl configtest`**. This checks your configuration
> files for syntax errors. If there is a mistake, a `reload` might fail to load the new config, but a `restart` will
> completely crash your site and prevent Apache from starting back up until it's fixed!

---

---

For uploading a new version of your web page, you should use:

**`systemctl reload apache2`**

### Why `reload` is enough:

- **Static Files (HTML, CSS, JS, Images):** If you just replaced your website's files (like your `.html` files or images),
  Apache reads these straight from the disk every time a user requests them. You actually don't even _need_ to reload Apache
  for these changes to show up.
- **Config Updates:** If your new website version required you to change something in your Apache configuration files (like
  adding a new redirect or updating a virtual host setup), a `reload` is perfect because it applies those changes instantly
  with **zero downtime** for your visitors.

### When you might need `restart` instead:

The only time you would need to `restart` after an update is if your new page relies on a backend service or a brand new
Apache module you just installed to support the new code (for example, enabling a specific caching module or changing PHP
versions).
