When it comes to encrypting data, **AES** (Advanced Encryption Standard) is the go-to algorithm. However, AES by itself only
encrypts a single block of data (16 bytes). To encrypt larger files or streams of data, we use **modes of operation**.

Two of the most popular modes are **CBC** (Cipher Block Chaining) and **GCM** (Galois/Counter Mode). Here is a breakdown of
how they work, how they differ, and why one is generally preferred today.

---

## 1. CBC (Cipher Block Chaining)

CBC is a classic, traditional mode of encryption. It introduces the concept of "chaining," meaning each block of plaintext
depends on the block that came before it.

### How it works:

- **Initialization Vector (IV):** Since the first block doesn't have a previous block to chain to, it uses a random starting
  block called an IV. This ensures that encrypting the same text twice results in different ciphertext.
- **The Chain:** Before a plaintext block is encrypted, it is mixed (using an XOR operation) with the _previous_ block's
  ciphertext.
- **The Catch:** Because block 2 depends on block 1, block 3 depends on block 2, and so on, **CBC cannot be parallelized**
  during encryption. You have to do it one block at a time.

> **Security Note:** CBC only provides **confidentiality** (hiding the data). It does not provide **integrity** (proving the
> data hasn't been tampered with). To secure CBC properly, developers have to manually combine it with a MAC (Message
> Authentication Code), usually via a method called _Encrypt-then-MAC_.

---

## 2. GCM (Galois/Counter Mode)

GCM is a modern, highly efficient mode of operation. It is an **AEAD** (Authenticated Encryption with Associated Data) mode,
meaning it handles both encryption and tamper-proofing at the same time.

### How it works:

- **Counter Mode:** Instead of chaining blocks together, GCM uses a counter (e.g., 1, 2, 3...). It encrypts the counter
  value, and then XORs that with the plaintext.
- **Parallel Processing:** Because the counter values are predictable, **GCM can encrypt and decrypt blocks completely in
  parallel**. This makes it incredibly fast on modern CPUs.
- **Built-in Authentication:** GCM multiplies the ciphertext by a special mathematical field (Galois field) to generate an
  authentication tag (often called a "Mac tag"). If an attacker changes even a single bit of the ciphertext, the tag won't
  match, and the system rejects the data.

---

## Key Differences at a Glance

| Feature                 | CBC Mode                                              | GCM Mode                                       |
| ----------------------- | ----------------------------------------------------- | ---------------------------------------------- |
| **Primary Function**    | Confidentiality only                                  | Confidentiality + Authentication (AEAD)        |
| **Speed / Performance** | Slower (Sequential)                                   | Faster (Parallelized)                          |
| **Tamper Protection**   | No (Requires a separate MAC)                          | Yes (Built-in authentication tag)              |
| **Vulnerabilities**     | Prone to padding oracle attacks if implemented poorly | Vulnerable if the IV/Nonce is ever reused      |
| **Modern Status**       | Legacy (Still used, but being phased out)             | Industry Standard (Used in TLS 1.3, SSH, etc.) |

---

## Which one should you use?

In almost all modern applications, **GCM is the winner**.

Because GCM provides built-in authentication, it protects you from malicious tampering out of the box. CBC is much easier to
implement incorrectly, which historically has led to famous cryptographic exploits (like Padding Oracle attacks). Unless you
are working with legacy systems that specifically require CBC, GCM is the safer and faster choice.
