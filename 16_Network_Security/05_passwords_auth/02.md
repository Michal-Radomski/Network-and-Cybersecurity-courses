Just like **scrypt**, both **PBKDF2** and **bcrypt** are specialized cryptographic algorithms designed to secure passwords.
Instead of hashing a password instantly (which makes it easy for hackers to guess millions of combinations per second), these
algorithms are intentionally designed to be slow.

Here is a breakdown of how PBKDF2 and bcrypt work, how they differ, and why they are used.

---

## 1. PBKDF2 (Password-Based Key Derivation Function 2)

**PBKDF2** is a widely accepted, standardized key derivation function. It takes a user's password, adds a "salt" (a random
string of data to prevent precomputed attacks), and runs it through a standard hashing algorithm (like SHA-256) **thousands
of times**.

- **How it Works (Iterations):** PBKDF2 relies on a parameter called the **iteration count**. For example, if the iteration
  count is set to 600,000, PBKDF2 will hash the password, take the result, hash it again, and repeat this cycle 600,000 times
  before producing the final key.
- **The Benefit:** A legitimate user logging in only has to wait a fraction of a second for their computer to run those
  600,000 rounds once. However, an attacker trying to guess millions of passwords has to run all 600,000 rounds for _every
  single guess_, severely slowing down their attack.
- **The Downside:** PBKDF2 is entirely **CPU-bound** and uses almost no RAM. This means hackers can use highly parallel
  hardware—like GPUs or custom ASIC chips—to crack PBKDF2 hashes incredibly quickly because they don't have to worry about
  memory bottlenecks.

---

## 2. Bcrypt

Released in 1999, **bcrypt** is a password hashing function based on the Blowfish symmetric block cipher. It has been a gold
standard for web developers securing user databases for over two decades.

- **How it Works (Work Factor):** Similar to PBKDF2, bcrypt uses a "work factor" (or cost parameter) to determine how slow
  the hashing process should be. The crucial difference is that bcrypt's work factor is **exponential** ($2^{\text{cost}}$).
  If you increase the cost parameter by 1, it takes twice as long to calculate the hash. This makes it very easy for
  administrators to increase security as computer hardware gets faster over time.
- **Built-in Salt:** Unlike some older algorithms where developers had to manually handle and store salt values, bcrypt
  automatically generates and embeds a secure, random salt directly into the final hash string.
- **GPU Resistance:** Because bcrypt relies on a lookup table mechanism from the Blowfish cipher, it requires a small amount
  of memory to run. While it is not nearly as "memory-hard" as **scrypt** or **Argon2**, this small memory requirement still
  makes it significantly harder and more expensive to attack using GPUs compared to PBKDF2.

---

## Quick Comparison: PBKDF2 vs. Bcrypt

| Feature                   | PBKDF2                                                              | Bcrypt                                                                  |
| ------------------------- | ------------------------------------------------------------------- | ----------------------------------------------------------------------- |
| **Primary Resource Used** | CPU only                                                            | CPU + small amount of memory                                            |
| **Scaling Mechanism**     | Linear iteration count (e.g., 600,000 rounds)                       | Exponential work factor ($2^{\text{cost}}$)                             |
| **GPU/ASIC Resistance**   | Low (highly vulnerable to specialized hardware)                     | Moderate (harder to optimize on GPUs)                                   |
| **Industry Standards**    | Highly standardized (FIPS-approved, required by PCI-DSS)            | De facto standard for web applications (Django, Ruby on Rails, Node.js) |
| **Best Use Case**         | Legacy systems, cryptographic key generation (like file encryption) | Web application user password hashing                                   |
