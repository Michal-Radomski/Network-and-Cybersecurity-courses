Requiring a physical device (such as a security key, TPM, or smart card) to unlock an encrypted disk, the options differ
between Windows and Linux.

## Windows

### Option 1: TPM (built into many PCs)

Windows' built-in full-disk encryption, BitLocker, is designed to use the computer's TPM (Trusted Platform Module).

Typical setup:

1. Enable TPM in the computer's firmware if necessary.
2. Turn on BitLocker for the system drive.
3. Store the recovery key somewhere safe.
4. Configure BitLocker to require:
   - TPM only (automatic unlock after integrity checks),
   - TPM + PIN, or
   - TPM + USB startup key (on supported editions and configurations).

This provides hardware-backed protection because the encryption key is sealed to the TPM.

### Option 2: USB startup key

BitLocker can also require a USB flash drive containing a startup key before Windows boots (availability depends on edition
and policy settings). The USB itself is not cryptographically secure like a security key, but it acts as a required physical
factor.

### Option 3: FIDO2 security keys

Windows supports FIDO2 security keys for Windows sign-in and authentication to services, but BitLocker does **not** natively
use FIDO2 security keys to unlock the system drive before boot.

## Linux

Linux offers more flexibility.

### Option 1: TPM2 (recommended)

Modern distributions support TPM2 for unlocking encrypted disks.

Typical stack:

- LUKS2
- `systemd-cryptenroll`
- TPM2

Example:

```bash
sudo systemd-cryptenroll --tpm2-device=auto /dev/nvme0n1p3
```

This enrolls the TPM as one of the unlock methods while keeping your existing recovery passphrase.

Benefits:

- Hardware-protected key
- Automatic boot on the original machine
- Recovery passphrase still works

---

### Option 2: FIDO2 security key (excellent choice)

LUKS2 can use a FIDO2 security key such as a:

- YubiKey
- SoloKey
- Nitrokey

With `systemd-cryptenroll`:

```bash
sudo systemd-cryptenroll --fido2-device=auto /dev/nvme0n1p3
```

At boot:

- insert the key,
- touch it if required,
- the disk unlocks.

This is one of the strongest and most convenient hardware-authentication methods available for Linux.

---

### Option 3: Smart cards or PKCS#11 tokens

LUKS can also use:

- PIV smart cards
- PKCS#11-compatible tokens
- Hardware security modules (HSMs)

Example:

```bash
systemd-cryptenroll --pkcs11-token-uri=auto
```

This is common in enterprise environments.

## Comparison

| Hardware           | Windows                     | Linux                        |
| ------------------ | --------------------------- | ---------------------------- |
| TPM                | ✅ Excellent                | ✅ Excellent                 |
| USB startup key    | ✅ Supported                | Possible with custom setup   |
| FIDO2 security key | ❌ Not for BitLocker unlock | ✅ Native support with LUKS2 |
| Smart card         | Limited for disk unlock     | ✅ Supported                 |
| YubiKey            | Not for BitLocker pre-boot  | ✅ Very good support         |

## Encrypting a disk

Encryption is typically a one-time setup.

- **Windows:** Enable BitLocker, which encrypts the drive in the background. Decryption (if you choose to remove encryption
  later) can also be initiated from BitLocker settings.
- **Linux:** Create a LUKS-encrypted volume (usually during installation or by reformatting the target disk), then enroll
  hardware authentication methods such as TPM2 or a FIDO2 key. Existing encrypted disks can have additional unlock methods
  added without re-encrypting the data.

## Best practice

For most users:

- **Windows:** BitLocker with **TPM + PIN** provides strong protection against both drive theft and unauthorized boot.
- **Linux:** **LUKS2 + TPM2** for seamless unlocking on your own machine, or **LUKS2 + FIDO2 security key** if you want
  unlocking to require possession of a removable hardware token. Keep a strong recovery passphrase enrolled as a backup in
  case the hardware device is unavailable.
