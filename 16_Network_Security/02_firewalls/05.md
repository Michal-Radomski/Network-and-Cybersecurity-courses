Both `iptables` and `nftables` are command-line utilities used to configure the Linux kernel's built-in firewall. While they
serve the same core purpose‚Äîfiltering network packets‚Äî`nftables` is the modern successor designed to replace the aging
`iptables` framework.

Here is a breakdown of how they work, how they compare, and why the transition happened.

---

## What is iptables?

For decades, `iptables` was the de facto standard for Linux firewalls. It interacts with the kernel's **Netfilter** framework
to inspect, modify, or drop network packets based on a set of rules.

`iptables` organizes its rules using a hierarchy of **Tables**, **Chains**, and **Targets**:

- **Tables:** Categorize rules by their broad purpose (e.g., `filter` for basic blocking, `nat` for Network Address
  Translation, `mangle` for altering packet headers).
- **Chains:** Attached to specific points in a packet's journey through the network stack (e.g., `INPUT` for incoming
  traffic, `FORWARD` for traffic passing through, `OUTPUT` for outgoing traffic).
- **Targets:** The action taken when a packet matches a rule (e.g., `ACCEPT`, `DROP`, `REJECT`).

### The Main Limitation of iptables

Every protocol protocol layer requires its own separate tool. For example, you use `iptables` for IPv4, `ip6tables` for IPv6,
`arptables` for ARP packets, and `ebtables` for ethernet bridging. This creates massive code duplication and makes managing
complex dual-stack (IPv4/IPv6) firewalls highly inefficient.

---

## What is nftables?

Introduced in 2014 and now the default in most modern Linux distributions (like Debian, Ubuntu, and Red Hat), `nftables` is a
complete rewrite of the Linux firewall infrastructure.

Instead of multiple separate tools, `nftables` replaces `iptables`, `ip6tables`, `arptables`, and `ebtables` with a single
unified tool: `nft`.

It achieves this by introducing a tiny, high-performance **virtual machine** inside the Linux kernel. The `nft` command
compiles your firewall rules into bytecode, which the kernel execution engine then runs at extreme speeds.

---

## Key Differences: iptables vs. nftables

| Feature                   | iptables                                                                                 | nftables                                                                          |
| ------------------------- | ---------------------------------------------------------------------------------------- | --------------------------------------------------------------------------------- |
| **Command-Line Tools**    | Multiple (`iptables`, `ip6tables`, `ebtables`, etc.)                                     | A single unified tool (`nft`)                                                     |
| **Built-in Architecture** | Fixed tables and chains are always loaded, even if empty.                                | Completely modular. You create only the tables and chains you need.               |
| **Performance**           | Slower rule evaluation; rules are checked linearly one-by-one.                           | Highly optimized via a kernel bytecode VM and native lookup sets.                 |
| **Rule Modification**     | Replacing a single rule requires replacing the entire table (can cause race conditions). | Supports atomic rule updates, allowing you to add/delete rules safely on the fly. |
| **Syntax**                | Verbose, rigid, and command-line option heavy (`-m tcp --dport 80`).                     | Cleaner, more readable, structured syntax similar to a programming language.      |

### Syntax Comparison

To give you an idea of how the usability improved, here is how you allow HTTP (port 80) and HTTPS (port 443) traffic in both
systems.

**In `iptables`:** You have to write two separate rules because it cannot handle multiple ports natively in a single simple
argument without specific extensions.

```bash
iptables -A INPUT -p tcp --dport 80 -j ACCEPT
iptables -A INPUT -p tcp --dport 443 -j ACCEPT

```

**In `nftables`:** You can group protocols, ports, and addresses naturally using sets `{}` in a single rule.

```text
nft add rule inet my_table my_chain tcp dport { 80, 443 } accept

```

_(Note: `inet` allows this single rule to apply to both IPv4 and IPv6 traffic simultaneously)._

---

## The Verdict

While `iptables` is still widely known and deeply embedded in legacy systems and scripts, **`nftables` is the present and
future of Linux firewalls**. Most modern operating systems include compatibility layers (`iptables-nft`) that translate old
`iptables` commands into `nftables` expressions behind the scenes, ensuring older applications don't break while still
leveraging the speed of the new engine.
