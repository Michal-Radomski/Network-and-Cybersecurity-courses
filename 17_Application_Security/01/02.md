The **Software Development Lifecycle (SDLC)** is a structured process used by software development teams to design, develop,
test, and deploy high-quality software. Think of it as a blueprint or a step-by-step roadmap for building an application from
scratch.

The main goal of the SDLC is to minimize risks, stay within a budget, and deliver software that meets or exceeds customer
expectations.

---

## The 6 Phases of SDLC

Every software project goes through these core phases, regardless of the specific methodology used:

### 1. Planning and Requirement Analysis

This is the most important phase. The team gathers requirements from stakeholders (clients, users, and sales teams) to figure
out exactly what the software needs to accomplish.

- **Key Questions:** What problem are we solving? Who will use this? What is the budget and timeline?

### 2. Design and Architecture

Based on the requirements, software architects create the technical blueprint for the system. This includes designing the
user interface (UI), database structures, system interfaces, and choosing the right technology stack.

- **Key Questions:** How will the software components interact? What will the user experience look like?

### 3. Development / Coding

This is where the actual building happens. Developers write the code according to the design specifications defined in the
previous phase. This is typically the longest phase of the cycle.

- **Key Questions:** Are we following secure coding standards? Is the code efficient and well-documented?

### 4. Testing

Once the code is built, QA (Quality Assurance) engineers test the software to find bugs, glitches, and vulnerabilities. They
ensure that the software actually works the way it was designed to.

- **Key Questions:** Does the application break under heavy load? Are there any security vulnerabilities? Does it meet the
  user requirements?

### 5. Deployment

After passing the testing phase, the application is released to production. This means it is made available to the public or
the internal team that requested it. It could mean launching a website, publishing a mobile app to an app store, or
installing software on company servers.

- **Key Questions:** How do we deploy the application smoothly without disrupting existing users?

### 6. Maintenance and Operations

Once the software is live, the work isn't done. Users will inevitably find bugs that weren't caught in testing, or they will
ask for new features. The development team monitors the app's performance, fixes issues, and releases updates.

- **Key Questions:** How is the software performing in the real world? What updates do we need to plan for the next version?

## Popular SDLC Models

How a team moves through these phases depends on the **SDLC methodology** they choose. Here are the three most common
approaches:

### Waterfall Model

The oldest and most traditional method. It is a linear, sequential approach where you finish one phase completely before
moving to the next.

- **Pros:** Easy to manage, clear documentation, great for projects with fixed requirements that won't change.
- **Cons:** Very rigid. If you discover a flaw in phase 4 (Testing), it is incredibly difficult and expensive to go back to
  phase 2 (Design) to fix it.

### Agile Model

The most popular modern approach. Instead of treating the project as one giant lifecycle, Agile breaks the software down into
small, manageable pieces called "increments." Teams work in short cycles (usually 2 to 4 weeks) called **sprints**, going
through all the SDLC phases for just a few features at a time.

- **Pros:** Highly flexible, allows for continuous feedback from the client, and accommodates changing requirements easily.
- **Cons:** Projects can easily experience "scope creep" (growing bigger than originally intended) if not managed properly.

### DevOps / DevSecOps

DevOps is a modern evolution of the SDLC that merges software development (**Dev**) with IT operations (**Ops**). It relies
heavily on automation to continuously integrate and deploy code updates. When security is integrated directly into this
automated loop from the very beginning, it is called **DevSecOps**.

- **Pros:** Incredibly fast deployment times, highly reliable, fixes security bugs before they reach production.
- **Cons:** Requires specialized tools and a major cultural shift within a company.
