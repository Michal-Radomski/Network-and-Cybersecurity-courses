The **OWASP Top 10** is a globally recognized awareness document published by the Open Worldwide Application Security Project
(OWASP). It represents a broad consensus among security experts regarding the most critical security risks to web
applications.

Rather than just a compliance checklist, developers and security teams use it as a foundational guide to prioritize
vulnerability management and build secure-by-design applications.

---

## The OWASP Top 10 (2025 Edition)

The **2025 update** reflects the shifting landscape of modern applications—moving from simple code-level bugs to
architectural, supply chain, and deployment risks.

| Rank    | Risk Category                             | Core Threat                                                                                |
| ------- | ----------------------------------------- | ------------------------------------------------------------------------------------------ |
| **A01** | **Broken Access Control**                 | Users accessing data or functions outside of their intended permissions.                   |
| **A02** | **Security Misconfiguration**             | Insecure default settings, open cloud storage, or missing security headers.                |
| **A03** | **Software Supply Chain Failures**        | Compromised third-party packages, libraries, or developer tools.                           |
| **A04** | **Cryptographic Failures**                | Weak or missing encryption exposing sensitive data (e.g., passwords, SSNs).                |
| **A05** | **Injection**                             | Malicious input (like SQL or OS commands) executed directly by an interpreter.             |
| **A06** | **Insecure Design**                       | Flaws in application architecture and business logic that coding fixes alone cannot solve. |
| **A07** | **Authentication Failures**               | Weak identity verification, session management, or missing MFA.                            |
| **A08** | **Software or Data Integrity Failures**   | Accepting code or data (like serialised objects) without verifying its authenticity.       |
| **A09** | **Security Logging & Alerting Failures**  | Lack of visibility during active attacks, making breaches impossible to track.             |
| **A10** | **Mishandling of Exceptional Conditions** | Verbose error handling that leaks data, or failing "open" instead of "closed."             |

### Deep Dive into Key Categories

### A01:2025 – Broken Access Control

Access control ensures users only interact with resources they are authorized to see. When this breaks, attackers can view,
modify, or delete data belonging to other users, or escalate their privileges to act as administrators.

- **Example:** Modifying a URL parameter from `example.com/app/userInfo?id=123` to `id=124` to view another user's private
  dashboard.

### A02:2025 – Security Misconfiguration

Even perfectly written code can be vulnerable if the server, cloud instance, or database is configured poorly. This includes
leaving default "admin/admin" credentials active, failing to restrict directory listing, or running unnecessary system
services.

### A03:2025 – Software Supply Chain Failures

Modern software relies heavily on open-source packages. If an attacker compromises a popular library or package manager (like
npm or PyPI) and injects malicious code, any application pulling that dependency automatically becomes infected.

### A04:2025 – Cryptographic Failures

Previously known as _Sensitive Data Exposure_, this focuses on the root cause: failing to protect data at rest or in transit.

- **Example:** Sending credit card data over unencrypted HTTP protocol, or storing user passwords using outdated hashing
  algorithms like MD5.

### A05:2025 – Injection

This happens when raw user input is concatenated directly into system queries instead of being sanitized. The interpreter
mistakes the user's input for executable code.

- **Example:** SQL Injection (SQLi), where a malicious SQL statement is typed into a login box to bypass authentication or
  dump database records.

### A06: Insecure Design

Insecure design is different from insecure implementation. You can write perfect, bug-free code, but if the underlying
architecture or business logic is flawed, the application remains vulnerable. Security must be designed into the system from
the start rather than treated as an afterthought.

- **Example:** A retail website allows users to reset their password by answering a single security question like _"What is
  your favorite color?"_ Even if the code executing this reset is flawlessly written, the design itself is inherently
  insecure.

### A07: Authentication Failures

This category deals with confirming a user's identity, assertions of their identity, and session management. When
applications fail to properly verify who a user is, attackers can hijack user sessions or compromise accounts.

- **Example:** Permitting incredibly weak passwords (like `password123`), not enforcing Multi-Factor Authentication (MFA), or
  failing to invalidate session IDs when a user logs out.

### A08: Software or Data Integrity Failures

This risk focuses on code and data that is accepted and executed without verifying its integrity. It often involves software
updates, data distribution pipelines, or "deserialization" (converting data from a file or network back into program
objects).

- **Example:** An application accepts serialized user state data from a cookie and reconstructs it on the server without
  verifying a digital signature. An attacker can modify the serialized data to elevate their privilege level to "admin"
  before the server processes it.

### A09: Security Logging & Alerting Failures

Without adequate logging and active monitoring, security breaches cannot be detected, responded to, or analyzed. Many major
breaches go unnoticed for months because systems fail to flag suspicious, repetitive activities.

- **Example:** An attacker attempts to brute-force admin credentials. Because the application does not log failed login
  attempts or trigger alerts for multiple rapid failures, the attack succeeds without anyone noticing.

### A10: Mishandling of Exceptional Conditions

This category involves how an application behaves when things go wrong. If an application crashes or encounters an error, it
must fail securely (failing "closed" rather than "open") and avoid leaking diagnostic information to the end-user.

- **Example:** A database connection times out, and instead of showing a generic "Something went wrong" message, the
  application displays a raw stack trace on the screen containing database server IPs, table structures, and internal code
  paths.
