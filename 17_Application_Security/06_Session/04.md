Large platform architectures like Udemy often utilize a hybrid approach to authentication and session management. While their
exact internal architecture isn't fully public, major engineering organizations typically handle this by using different
tools for different parts of the system.

Here is how a hybrid approach functions in a large ecosystem like Udemy:

### 1. The Core Application: Stateful Sessions

For the primary web platform (where users log in, browse courses, and manage their dashboard), large platforms generally rely
on **stateful sessions** backed by an efficient server-side data store like Redis.

- **Why:** Security and immediate control. If a user changes their password, logs out from all devices, or an account is
  flagged for fraud, the platform needs the ability to instantly revoke that user's access. Traditional database-backed
  sessions allow the server to delete the session instantly.

### 2. Internal Microservices: Stateless JWTs

Udemy relies on a microservices architecture. When a user requests a page, that single request might require data from the
"Video Streaming service," the "Payments service," and the "Progress Tracking service."

- **Where JWTs fit:** Instead of every single microservice asking the central database _"Is this session valid?"_ thousands
  of times per second (which creates a massive performance bottleneck), a central API gateway verifies the user's session
  cookie once.
- The gateway then generates a short-lived **JWT** containing the user's ID and permissions, and passes it along to the
  internal microservices. The microservices can verify the JWT instantly using cryptography without querying the database.

### 3. Public APIs and Mobile Apps

- **Mobile Apps:** Mobile applications often use a form of OAuth2 or token-based authentication (like JWTs or long-lived
  access tokens) rather than standard cookies because mobile operating systems store and transmit headers more reliably than
  browser cookies.
- **Third-Party Integrations:** For business clients (Udemy Business) or integrations with Single Sign-On (SSO) systems like
  Okta or Azure AD, standard identity tokens (JWTs) are utilized to bridge the connection securely between systems.

---

### Summary of the Workflow

1. **You log in on the website:** The server validates your credentials and drops a secure, stateful **session cookie** in
   your browser.
2. **You watch a video:** The web traffic hits the platform's API gateway. The gateway validates your session cookie.
3. **Behind the scenes:** The gateway translates your session into an internal **JWT** and fires it off to various backend
   microservices so they can securely deliver the video content and update your progress seamlessly.
