Imagine you want to use a new online photo printing service, and it asks to access your photos stored on Google Photos.

In the old days of the internet, you would have to give that printing website your actual Google username and password. This
was incredibly dangerous—that website could now read your emails, change your password, and lock you out.

**OAuth (Open Authorization)** was invented to solve this exact problem.

---

## What is OAuth?

OAuth is an open-standard **delegation protocol**. It allows a website or app to access resources (like your profile data,
photos, or contacts) hosted by another service **without you ever giving away your password**.

Instead of a password, OAuth uses **Access Tokens**. Think of an access token like a keycard at a hotel: the hotel doesn't
give you the master key to the building; they give you a electronic card that _only_ opens your specific room door, and it
automatically expires after a few days.

---

## How OAuth Works: The 4 Main Roles

To understand OAuth, it helps to look at the four roles involved in every single OAuth transaction:

1. **The Resource Owner (You):** The user who owns the data and wants to grant access to it.
2. **The Client (The App):** The third-party website or app that wants access to your data (e.g., the photo printing
   website).
3. **The Authorization Server (The Gatekeeper):** The engine that displays the login screen, asks you for permission, and
   issues the secure token (e.g., Google's login server).
4. **The Resource Server (The Vault):** The API or server that actually holds your data (e.g., the Google Photos database).

---

## The Standard OAuth Flow (Simplified)

When you click that shiny **"Connect to Google Photos"** button, a highly orchestrated dance happens behind the scenes:

```
+--------+                               +-----------------------+
|        |--(A)- Click "Connect App"---->|                       |
|        |                               |  Authorization Server |
|        |<-(B)- "Do you grant access?"--|     (e.g., Google)    |
|  User  |                               |                       |
|        |--(C)--- "Yes, I approve"----->|                       |
|        |                               +-----------------------+
|        |                                           |
|  (The  |                                          (D) Issues
| Client |                                       Authorization
|  App)  |<---------------------------------------- Code
|        |                                           |
|        |--(E)- Sends Auth Code + Secret ----------->
|        |                                           |
|        |<-(F)- Receives Access Token --------------+
+--------+

```

1. **The Request:** The Client App redirects your browser to Google's Authorization Server.
2. **The Consent:** Google asks you: _"This app wants to view your photos. Do you allow this?"_
3. **The Handshake:** Once you click "Allow," Google sends a temporary **Authorization Code** back to the Client App.
4. **The Exchange:** The Client App takes that code and exchanges it securely with Google behind the scenes for a
   long-lasting **Access Token**.
5. **The Access:** The Client App presents that Access Token to the Google Photos database, which verifies it and hands over
   your photos.

## OAuth 2.0 vs. OpenID Connect (OIDC)

A very common point of confusion is mixing up **OAuth** and **OpenID Connect**.

- **OAuth 2.0 is strictly for Authorization:** It answers the question: _"Does this app have permission to access my files?"_
  It does **not** tell the app who you are.
- **OpenID Connect (OIDC) is for Authentication:** Built directly on top of OAuth 2.0, OIDC adds a new feature called an **ID
  Token** (usually a JWT). This answers the question: _"Who is the user currently logged in?"_ Whenever you click a button
  that says **"Sign in with Google"** or **"Log in with Apple"** to create an account on a new app, you are actually using
  **OpenID Connect**.

---

## Why OAuth is a Lifesaver for Security

- **No Password Sharing:** You never trust third-party apps with your primary master password.
- **Granular Scopes:** You can give an app permission to _read_ your calendar events without giving it permission to _delete_
  or edit them.
- **Easy Revocation:** If you decide you don't trust a third-party app anymore, you can go into your Google or Apple account
  settings and instantly revoke their access token, cutting them off completely without needing to change your password.
