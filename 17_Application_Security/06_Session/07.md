**JOSE** stands for **JSON Object Signing and Encryption**.

It is not a single tool, but rather a framework of open standards (RFCs) created by the IETF. Its purpose is to provide a
standardized, secure way to transfer claims or data between two parties as JSON objects.

If you have ever used a JWT (JSON Web Token), you have used the JOSE framework under the hood.

---

## The 4 Main Pillars of JOSE

The JOSE framework is divided into four primary specifications, each handling a specific job in the lifecycle of a secure
token:

### 1. JWS (JSON Web Signature) — _Integrity_

JWS focuses on **signing** data. It takes a JSON payload and signs it using a cryptographic key (symmetric or asymmetric).
This ensures that the data cannot be tampered with while in transit. Anyone can read the data, but if they change even a
single character, the signature becomes invalid.

- - Analogy:\* A signed check. Anyone can read who it’s for and the amount, but the signature proves it came from the right
    person and hasn't been altered.

### 2. JWE (JSON Web Encryption) — _Confidentiality_

JWE focuses on **encrypting** data. It takes a JSON payload and scrambles it entirely using cryptographic algorithms. The
data becomes completely unreadable to anyone who intercepts it, and it can only be read by someone holding the correct
decryption key.

- _Analogy:_ A letter locked inside a digital safe. Only the person with the key can see what is written inside.

### 3. JWK (JSON Web Key) — _Key Management_

JWK defines a standardized JSON format for representing cryptographic keys. Instead of dealing with clunky, raw `.pem` or
`.der` key files, JWK allows systems to share public keys as clean, readable JSON structures.

- _Example:_ An authorization server might expose a public URL (a JWKS or "JSON Web Key Set") containing its public keys so
  that API gateways can automatically download them and verify incoming tokens.

### 4. JWA (JSON Web Algorithms) — _The Cryptographic Suite_

JWA is the dictionary of all the cryptographic algorithms permitted within the JOSE ecosystem. It standardizes the short
names used in the headers of tokens, such as `HS256` (HMAC using SHA-256), `RS256` (RSA using SHA-256), or `A256GCM` (AES GCM
using a 256-bit key).

---

## How Does JWT Fit Into This?

A **JWT (JSON Web Token)** is simply the overarching data structure that puts JOSE to work.

A JWT is like an abstract blueprint for a token. When you actually implement a JWT in the real world, it _must_ become either
a **JWS** (a signed token, which is what 95% of developers use) or a **JWE** (an encrypted token).

## Why Do We Use JOSE?

Before JOSE, web applications struggled to securely pass data back and forth without constantly querying a central database.
JOSE solved this by making tokens:

- **Web-friendly:** It relies entirely on JSON and Base64URL encoding, making it lightweight and easy to pass via HTTP
  headers, URLs, or cookies.
- **Standardized:** Because it follows strict RFC guidelines, a JWS generated by a backend written in Go or Python can be
  natively parsed and verified by a backend written in Node.js without any compatibility issues.
