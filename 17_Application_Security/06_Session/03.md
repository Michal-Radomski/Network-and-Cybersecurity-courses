When working with modern web security and APIs, you will often hear about tokens. While **JWT** is a broad term that most
developers use daily, it actually has a more specific sibling called **JWE**.

Both are part of the **JOSE (JSON Object Signing and Encryption)** framework, but they serve entirely different purposes: one
guarantees **honesty (integrity)**, while the other guarantees **secrecy (confidentiality)**.

---

## 1. JWT (JSON Web Token)

A **JWT** is an open standard (RFC 7519) that defines a compact and self-contained way for securely transmitting information
between parties as a JSON object.

When people say "JWT," they are almost always referring to a **JWS (JSON Web Signature)**.

### How it Works

A standard JWT is **signed, not encrypted**. This means the data inside the token is completely visible to anyone who
intercepts it, but it cannot be altered without the server noticing. Think of it like a postcard written in pen: anyone can
read it, but if someone tries to cross out a word and rewrite it, the signature becomes invalid.

### Structure

A JWT is separated by dots (`.`) into three base64url-encoded parts:

1. **Header:** Contains metadata about the token (e.g., the signing algorithm like `HS256`).
2. **Payload:** Contains the actual data (claims), such as `userId`, `roles`, and expiration time.
3. **Signature:** Created by hashing the encoded header, encoded payload, and a secret key. This ensures the token hasn’t
   been tampered with.

- **Best Used For:** Authorization tokens (e.g., Access Tokens, ID Tokens in OAuth/OIDC) where the client needs to read the
  data, and the server just needs to verify it hasn't been modified.

---

## 2. JWE (JSON Web Encryption)

A **JWE** (RFC 7516) takes things a step further by **encrypting** the data.

### How it Works

Unlike a standard JWT, a JWE completely hides the payload. If someone intercepts a JWE, it will look like a completely random
string of garbage characters. Only someone holding the correct decryption key can read the contents. Think of it like a
letter locked inside a digital safe.

### Structure

Because encryption requires more cryptographic steps, a JWE is split into **five** parts separated by dots instead of three:

1. **Protected Header:** Describes the encryption algorithms used.
2. **Encrypted Key:** The content encryption key (CEK), which is itself encrypted using the recipient's public key.
3. **Initialization Vector (IV):** Random data used to ensure the same plaintext doesn't encrypt to the same ciphertext
   twice.
4. **Ciphertext:** The actual payload, now completely encrypted and unreadable.
5. **Authentication Tag:** Used to ensure the integrity of the ciphertext (preventing tampering).

- **Best Used For:** Transmitting highly sensitive or regulated data (like PII, health records, or financial data) where you
  cannot risk the client or any middleman reading the payload.

---

## Direct Comparison

| Feature          | JWT (Standard / JWS)                           | JWE (Encrypted)                                 |
| ---------------- | ---------------------------------------------- | ----------------------------------------------- |
| **Primary Goal** | **Integrity** (Proof that data hasn't changed) | **Confidentiality** (Proof that data is hidden) |
| **Visibility**   | Publicly readable (Base64 decoded)             | Completely hidden (Encrypted ciphertext)        |
| **Format**       | 3 parts (`header.payload.signature`)           | 5 parts (`header.key.iv.ciphertext.tag`)        |
| **Performance**  | Fast (Hashing/Signing is lightweight)          | Slower (Encryption/Decryption takes more CPU)   |
| **Use Case**     | Everyday API authentication & session tokens   | Passing medical, financial, or strict PII data  |

---

## Can a JWT be a JWE?

**Yes.** A JWT is the overarching abstract concept. A JWT can be implemented either as a signed token (JWS) or an encrypted
token (JWE).

In fact, you can even **nest** them: you can sign a token (JWS) so everyone knows it's authentic, and then encrypt that
signed token inside a JWE so nobody can read it until it reaches its final destination.
