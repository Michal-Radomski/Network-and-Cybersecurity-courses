**Application Security (AppSec)** and **Web Application Security (WebSec)** are two closely related fields in cybersecurity,
but they differ in scope.

Think of it this way: Application Security is the overarching umbrella, while Web Application Security is a specialized,
highly critical section under that umbrella.

---

## 1. Application Security (AppSec)

Application Security is the general practice of making software applications more secure, usually by finding, fixing, and
preventing security vulnerabilities throughout the entire software development lifecycle (SDLC).

- **Scope:** It covers _all_ types of software, including desktop applications (like Microsoft Word), mobile apps (like
  Instagram), embedded software (like the code running your smart fridge), and yes, web applications.
- **When it happens:** It starts before a single line of code is even written (during the design phase) and continues long
  after the app is launched.

### Key Practices in AppSec

- **Secure Coding:** Training developers to write code that isn't easily exploited.
- **Threat Modeling:** Analyzing the app's design early on to spot potential security flaws before they are built.
- **Security Testing:** \* **SAST (Static Application Security Testing):** Scanning the raw source code for bugs before the
  app runs.
- **DAST (Dynamic Application Security Testing):** Testing the running application from the outside to see if it can be
  hacked.

- **Dependency Scanning:** Checking third-party libraries and open-source code packages to ensure they don't introduce
  vulnerabilities.

---

## 2. Web Application Security (WebSec)

Web Application Security is a specific subset of AppSec. It focuses exclusively on the security of websites, web apps (like
Google Docs or Netflix), and the web APIs that connect them.

Because web applications are constantly exposed to the public internet, they are prime targets for hackers.

### Unique Challenges of WebSec

- **Public Exposure:** Unlike a desktop app that sits on a user's local computer, a web app is live on the internet $24/7$,
  meaning anyone from anywhere can attempt to attack it.
- **Complex Ecosystem:** Web apps rely on web browsers, internet protocols (HTTP/HTTPS), web servers, and databases, all of
  which introduce different attack vectors.

### Common Web App Threats (The OWASP Top 10)

The Open Worldwide Application Security Project (OWASP) maintains a list of the most critical web app vulnerabilities. Some
of the most famous include:

- **Injection (e.g., SQL Injection):** When an attacker tricks a web form into sending malicious commands to the backend
  database to steal data.
- **Broken Authentication:** When poorly implemented login systems allow attackers to steal user sessions or brute-force
  passwords.
- **Cross-Site Scripting (XSS):** When an attacker injects malicious scripts into a trusted website, which then execute in an
  innocent user's browser.

### Key Defense Tools in WebSec

- **WAF (Web Application Firewall):** A filter that sits between the web app and the internet, analyzing traffic to block
  known malicious attacks.
- **HTTPS/TLS:** Encrypting the data sent between the user's browser and the web server so it cannot be intercepted.
- **Input Validation:** Ensuring that any data a user types into a website (like a username or search query) is thoroughly
  cleaned and checked before the server processes it.

---

## Summary of Differences

| Feature                 | Application Security (AppSec)                                     | Web Application Security (WebSec)                                         |
| ----------------------- | ----------------------------------------------------------------- | ------------------------------------------------------------------------- |
| **Scope**               | Broad (All software: mobile, desktop, cloud, web).                | Narrow (Websites, web apps, and APIs only).                               |
| **Primary Environment** | Local machines, servers, mobile devices, etc.                     | The internet / web browsers.                                              |
| **Core Goal**           | Securing the software itself from the ground up.                  | Securing the application from internet-faced attacks and exploits.        |
| **Key Example**         | Ensuring a mobile banking app encrypts data stored on your phone. | Ensuring the bank's website doesn't allow SQL injection in the login bar. |
