**HTTP Strict Transport Security (HSTS)** is a powerful web security policy mechanism designed to protect websites against
man-in-the-middle attacks, such as protocol downgrade attacks and cookie hijacking.

It directly addresses the **Cryptographic Failures** category of the OWASP Top 10 by forcing web browsers to interact with a
website _only_ using secure HTTPS connections, completely ignoring any attempts to connect via unencrypted HTTP.

---

## The Problem It Solves: The First-Contact Vulnerability

Even if a website supports HTTPS, users often type a simple address like `example.com` into their browser, or click a legacy
link that uses `http://`.

Normally, the process works like this:

1. The browser attempts an unencrypted connection to `http://example.com`.
2. The server receives the request and sends back a redirect (like a `301 Moved Permanently`) telling the browser to go to
   `https://example.com`.
3. The browser establishes a secure HTTPS connection.

That brief window during **Step 1 and 2** is highly vulnerable. If an attacker is intercepting the user's network traffic
(like on a public Wi-Fi network), they can intercept that initial HTTP request and serve a fake version of the website before
the redirect to HTTPS ever happens. This is known as a **SSL/TLS Strip** or protocol downgrade attack.

---

## How HSTS Works

HSTS fixes this vulnerability by allowing the server to tell the browser: _"Never talk to me over HTTP again. If you try,
automatically turn it into HTTPS before sending the request."_

When a user visits a site over HTTPS, the server responds with a specific HTTP header called `Strict-Transport-Security`.

### The HSTS Header Syntax

```http
Strict-Transport-Security: max-age=63072000; includeSubDomains; preload

```

- **`max-age=63072000`**: Tells the browser to remember this rule for a specific amount of time in seconds (63,072,000
  seconds is exactly 2 years). Every time the user visits the site, this timer resets.
- **`includeSubDomains`**: Extends this rule to all subdomains (e.g., `api.example.com` and `blog.example.com`).
- **`preload`**: Authorizes the website to be included in the **HSTS Preload List**.

---

## The HSTS Preload List

There is still a tiny vulnerability: the _very first time_ a browser ever visits a website, it doesn't know about the HSTS
header yet.

To close this loop, major browser vendors (Google, Mozilla, Apple, Microsoft) maintain a hardcoded **HSTS Preload List**. If
a website owner submits their domain to this list, browsers ship with the knowledge that the site requires HTTPS. The browser
will _never_ attempt an unencrypted HTTP connection to that site, even on the very first visit.

---

## Key Security Benefits

- **Prevents Downgrade Attacks:** Attackers cannot force a user's browser onto an unencrypted HTTP connection.
- **Protects Session Cookies:** It stops sensitive session tokens from accidentally leaking over an unencrypted network
  request.
- **Blocks Certificate Override Exploits:** If a website uses HSTS and an attacker tries to present a fake, self-signed SSL
  certificate, the browser will display a critical error and **completely block the user** from clicking "Proceed anyway."
  This prevents users from accidentally bypassing vital security warnings.
