**Software Supply Chain Failures** became a standalone category in the OWASP Top 10 to address a growing reality: you don't
just have to secure the code _you_ write, you also have to secure the code, tools, and infrastructure you _import_.

Modern software is rarely built from scratch. Developers rely heavily on open-source libraries, third-party plug-ins, public
repositories, and automated delivery pipelines (CI/CD). If an attacker compromises any link in that chain, they can slip
malicious code directly into your final product without ever touching your network.

---

## How Software Supply Chain Failures Happen

Attackers target the weakest links in the software lifecycle, usually falling into these common categories:

### 1. Malicious Dependency Ingestion (Typosquatting)

Attackers publish malicious packages to public registries (like npm, PyPI, or Maven) using names that are nearly identical to
popular, legitimate libraries.

- **Example:** A developer accidentally types `npm install bootstrapp` instead of `bootstrap`. The typo library installs
  perfectly fine but secretly steals API keys in the background.

### 2. Dependency Confusion

Many companies mix internal private packages with public packages. Attackers figure out the names of a company's internal
private components and upload a malicious version with the exact same name—but a higher version number—to a public
repository. The building system automatically grabs the "newer" public (malicious) version.

### 3. Compromised Third-Party Tools & Plugins

If a developer's favorite IDE plugin, browser extension, or a common utility tool is hijacked by hackers, it can be used to
inject malicious code during the development process.

### 4. CI/CD Pipeline Attacks

If an attacker gains access to your automated build servers (like GitHub Actions, Jenkins, or GitLab CI), they can modify the
build scripts to inject malware right before the software is compiled and shipped to customers.

---

## Real-World Impact

A single supply chain failure can have a massive ripple effect because one compromised library can infect thousands of
downstream applications.

- **Mass Malware Distribution:** You unknowingly ship malware directly to your users, completely ruining brand trust.
- **Data and Credential Theft:** Malicious dependencies often look for environment variables containing passwords, cloud
  keys, or database credentials and exfiltrate them.
- **Widespread Infrastructure Compromise:** Ransomware or backdoors can be planted deep inside production systems.

---

## How to Prevent It

Securing the supply chain requires validating everything that enters your ecosystem:

> 🛡️ **The Golden Rule:** Never trust a third-party resource blindly. Verify its integrity, source, and contents before
> letting it enter your build environment.

- **Use a Software Bill of Materials (SBOM):** Maintain a continuously updated inventory of every single component, version,
  and dependency used in your software. Think of it as an ingredient list for your app.
- **Lock Down Dependencies:** Use lockfiles (like `package-lock.json` or `poetry.lock`) to ensure your build system always
  pulls the exact, verified version of a library rather than grabbing the latest unvetted update.
- **Vulnerability Scanning:** Implement automated tools (like GitHub Dependabot or OWASP Dependency-Check) into your pipeline
  to scan for known vulnerabilities in your third-party packages.
- **Verify Signatures:** Use cryptographic signatures to verify that the code you are downloading actually comes from the
  trusted author and hasn't been tampered with mid-transit.
- **Host Private Registries:** Route internal dependencies through a secure, private repository manager to prevent dependency
  confusion attacks.
