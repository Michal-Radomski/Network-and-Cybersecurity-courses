**Cryptographic Failures** (previously known as _Sensitive Data Exposure_) focuses on the root cause rather than the symptom:
the failure to properly protect sensitive data at rest and in transit using cryptography.

It sits high on the OWASP Top 10 because when cryptography fails, it often leads to catastrophic data breaches involving
passwords, credit card numbers, health records, and personal identifiers.

---

## Common Causes of Cryptographic Failures

Cryptographic failures rarely happen because someone cracked a complex math equation; they usually happen because of
implementation mistakes.

### 1. Transmitting Data in Cleartext

Sending data over unencrypted protocols like HTTP, FTP, or SMTP allows attackers to intercept it using simple packet-sniffing
tools.

- **Example:** A login form that submits passwords over `http://` instead of `https://` allows anyone on the same public
  Wi-Fi network to read the password in plain text.

### 2. Using Weak or Outdated Algorithms

Using cryptographic algorithms that have been broken by modern computing power is just as bad as using no encryption at all.

- **Example:** Using **MD5** or **SHA-1** for hashing passwords, or using **DES** or **RC4** for encryption. Attackers can
  easily bypass or reverse these algorithms.

### 3. Poor Key Management

Encryption is only as secure as the keys used to lock and unlock the data.

- **Example:** Hardcoding encryption keys or API secrets directly into the application's source code (where anyone with
  access to the GitHub repository can see them), or reusing the same key indefinitely without rotation.

### 4. Lack of Proper Salting

When storing passwords, applications should use a "salt"—a unique, random string added to the password before it is hashed.
Without unique salts, attackers can use pre-computed tables of hashes (called Rainbow Tables) to instantly crack millions of
stolen password hashes.

---

## Real-World Impact

When cryptography fails, the defensive walls crumble completely:

- **Mass Identity Theft:** Millions of user records can be exposed in plain text.
- **Regulatory Fines:** Massive penalties under laws like GDPR or PCI-DSS for failing to protect user privacy.
- **Loss of Trust:** Complete destruction of a company's reputation once customers realize their private data was poorly
  guarded.

---

## How to Prevent It

Proper cryptography requires following industry standards and avoiding "homegrown" encryption methods.

> 🔑 **The Golden Rule:** Never invent your own cryptography. Use widely accepted, standard libraries and keep your data
> encrypted both while it's moving and while it's stored.

- **Encrypt All Data in Transit:** Enforce HTTPS across your entire website using strict policies like HSTS (HTTP Strict
  Transport Security) to ensure no data is sent in the clear.
- **Encrypt Sensitive Data at Rest:** Ensure databases, backups, and cloud storage volumes holding sensitive data are
  encrypted using strong algorithms like AES-256.
- **Use Strong Password Hashing Functions:** Never use standard hashing algorithms for passwords. Use adaptive, slow hashing
  algorithms specifically designed for password storage, such as **Argon2id**, **bcrypt**, or **PBKDF2**.
- **Protect Encryption Keys:** Store keys securely in dedicated key management systems (like AWS KMS or HashiCorp Vault).
  Never commit keys to version control software.
