**Broken Access Control** is a critical security vulnerability that occurs when a web application fails to properly enforce
restrictions on what authenticated users are allowed to do.

In a secure system, **Access Control** ensures that users can only access the data and functions they have explicit
permission to use. When it is "broken," attackers can bypass these checks to access unauthorized data, modify or delete
content, or even take over administrative functions.

Because it is incredibly common and highly damaging, it frequently occupies the **#1 spot on the OWASP Top 10** list of web
application security risks.

---

## How Broken Access Control Happens

Access control issues usually fall into a few primary categories:

### 1. Vertical Privilege Escalation (Privilege Creep)

This happens when a regular user is able to access functions or data reserved for higher-privileged users, such as
administrators.

- **Example:** A standard user alters the URL from `example.com/user/dashboard` to `example.com/admin/dashboard` and
  successfully gains access to the admin panel because the server forgot to verify their role.

### 2. Horizontal Privilege Escalation

This occurs when a user accesses resources belonging to another user with the same privilege level.

- **Example:** A user logs in and sees their account URL: `example.com/view_invoice?id=1001`. If they change the ID to `1002`
  and can see another customer's private invoice, this is a broken access control flaw known as **Insecure Direct Object
  References (IDOR)**.

### 3. Metadata Manipulation

Attackers tamper with the data sent to the server (like cookies, JSON Web Tokens (JWT), or hidden form fields) to trick the
application into thinking they have different permissions.

- **Example:** Changing a cookie value or API request body from `"isAdmin": false` to `"isAdmin": true`.

### 4. CORS Misconfiguration

Cross-Origin Resource Sharing (CORS) configurations might be set too permissively (e.g., using wildcards like `*`), allowing
unauthorized API access from untrusted origins.

---

## Real-World Impact

When access control is broken, the consequences can be severe:

- **Data Breaches:** Unauthorized viewing of sensitive user data, medical records, or financial information.
- **Data Modification/Destruction:** Attackers altering or deleting critical database records.
- **Complete System Compromise:** If an attacker escalates their privileges to an administrator, they can lock out legitimate
  users and control the entire application.

---

## How to Prevent It

Securing access control requires a "deny by default" mindset. Developers should implement the following best practices:

> 💡 **The Golden Rule:** Access control must always be enforced on the **trusted server side**, never solely on the client
> side (the user's browser).

- **Deny by Default:** Lock down every page and API endpoint. Only grant access to specific roles explicitly.
- **Implement Principle of Least Privilege (PoLP):** Ensure users and processes only have the exact minimum permissions
  necessary to complete their tasks.
- **Centralize Access Control:** Instead of writing authorization checks on every single page (which leads to human error),
  use a centralized framework or middleware to handle permissions uniformly.
- **Disable Directory Browsing:** Ensure that web servers do not list file directories, which could expose backup files or
  sensitive code configurations to the public.
- **Log and Audit:** Log all access control failures (e.g., repeated unauthorized attempts to access admin pages) so security
  teams can detect and respond to potential attacks early.
