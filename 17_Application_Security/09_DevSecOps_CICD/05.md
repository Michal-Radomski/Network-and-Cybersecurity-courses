In production, CI/CD pipelines, and active development, **it is significantly more secure to specify concrete versions (or
pin dependencies using lockfiles) rather than using `npm i package@latest`.**

While `npm i package@latest` is fine for quick local prototyping or intentionally upgrading a tool, relying on `@latest` in
your workflows introduces serious security, stability, and reproducibility risks.

---

## 1. Why `package@latest` is a Security Risk

Using `@latest` (or loose semver ranges like `*` or `^` without a lockfile) exposes your application to immediate supply
chain threats:

- **Compromised Maintainer Accounts / Malicious Releases:** If a developer account or registry token is compromised, an
  attacker can publish a malicious version (e.g., `v2.0.1`) containing secret-stealing scripts or backdoors. Anyone
  installing `@latest` during that window automatically executes the compromised code.
- **Bypassing Testing:** Installing `@latest` dynamically at build or deployment time means your code is running on a version
  that was never tested in your development or staging environments.
- **Typosquatting & Brandjacking Attacks:** A slight typo when running `npm i package@latest` can immediately execute
  malicious code from a similarly named package.

---

## 2. Comparison: Concrete Versions vs `@latest`

| Aspect                   | Concrete Version (`package@1.2.3`)                                     | Latest Flag (`package@latest`)                                             |
| ------------------------ | ---------------------------------------------------------------------- | -------------------------------------------------------------------------- |
| **Deterministic Builds** | **Guaranteed.** Everyone builds against the exact same codebase.       | **Non-deterministic.** Builds vary depending on _when_ `npm install` runs. |
| **Supply Chain Defense** | **High.** Shields you against sudden zero-day supply chain injections. | **Low.** Directly fetches whatever is newest in the registry right now.    |
| **Breaking Changes**     | **Safe.** Protects against unexpected API changes.                     | **Risky.** Unannounced major/minor breaks can bring down builds.           |
| **Auditability**         | **High.** Changes are recorded in git commits / PRs.                   | **Low.** Upgrades happen silently without explicit code review.            |

---

## 3. Recommended Best Practices

### Use Lockfiles and `npm ci`

When deploying or running builds in a CI/CD pipeline, never use `npm install`. Use `npm ci` instead:

```bash
# Reads directly from package-lock.json to install the exact dependency tree
npm ci

```

`npm ci` ignores loose semver ranges, verifies integrity hashes (SHA-512), and fails if `package.json` and
`package-lock.json` do not match.

### Automate Version Upgrades safely

Instead of fetching `@latest` at runtime, update dependencies intentionally using automated tools in your repository:

- **Dependabot / Renovate:** Automatically creates Pull Requests when a new version of a package is released.
- **CI Integration:** Your test suite runs on the PR to verify that the upgrade doesn't break anything or introduce flagged
  vulnerabilities before human review and merge.

### Pinning Exact Versions

To prevent `npm install` from auto-updating packages in your `package.json`, save exact versions without the caret (`^`) or
tilde (`~`):

```bash
npm install --save-exact package-name@1.2.3

```
