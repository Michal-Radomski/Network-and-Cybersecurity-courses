At their core, **DevOps**, **CI/CD**, and **DevSecOps** are modern software development practices designed to help teams
build, test, and release software faster, more reliably, and more securely.

Here is a breakdown of how they work and fit together.

---

## 1. DevOps (Development + Operations)

Traditionally, **Software Developers** (who build code) and **IT Operations** (who deploy and manage servers) worked in
isolated "silos." Developers wanted to release new features quickly, while Operations wanted to keep systems stable by
minimizing changes.

**DevOps** is a cultural and technical movement that unites these two teams into a continuous, collaborative lifecycle:

- **Goal:** Shorten the development cycle while maintaining high software quality.
- **Core Idea:** "You build it, you run it." Teams share ownership of planning, coding, building, testing, deploying,
  operating, and monitoring software.

---

## 2. CI/CD (Continuous Integration & Continuous Delivery/Deployment)

If DevOps is the philosophy, **CI/CD** is the technical engine that makes it run. It relies on automated pipelines to move
code from a developer's computer to live production servers.

### **Continuous Integration (CI)**

- **What it is:** Developers frequently merge their code changes into a central repository (like GitHub or GitLab).
- **How it works:** Every time code is pushed, automated tests run to check for bugs and ensure new code doesn't break
  existing features.

### **Continuous Delivery (CD)**

- **What it is:** Code changes that pass automated tests are automatically built and prepared for release to production.
- **Key Detail:** Deployment to production usually requires a manual approval button click from a human.

### **Continuous Deployment (CD)**

- **What it is:** Every change that passes the automated testing suite is automatically deployed directly to live production
  without manual intervention.

---

## 3. DevSecOps (Development + Security + Operations)

In traditional DevOps, security testing was often saved for the very end of the release process. If a vulnerability was
discovered right before deployment, the release had to be delayed, or worse, security flaws went unnoticed in production.

**DevSecOps** integrates security into **every stage** of the DevOps pipeline—a practice known as **"Shifting Left."**

- **Goal:** Make security a shared responsibility from day one, rather than an afterthought.
- **Key Practices:**
- **Static Code Analysis (SAST):** Scanning raw code for security vulnerabilities as it's written.
- **Dependency Scanning:** Checking third-party libraries/packages for known security flaws.
- **Automated Compliance:** Ensuring configuration files meet security standards automatically during the build process.

---

## Quick Comparison

| Term          | Focus                | Key Benefit                                                      |
| ------------- | -------------------- | ---------------------------------------------------------------- |
| **DevOps**    | Culture & Process    | Breaks down silos between developers and operations.             |
| **CI/CD**     | Automation           | Automatically tests, builds, and deploys code quickly.           |
| **DevSecOps** | Security Integration | Catches security vulnerabilities early in the development cycle. |
