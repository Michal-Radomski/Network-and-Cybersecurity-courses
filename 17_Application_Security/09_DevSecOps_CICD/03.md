Securing a CI/CD pipeline involves two distinct aspects: **Securing the Code in the Pipeline** (preventing vulnerabilities
from entering production) and **Securing the Pipeline Itself** (protecting the build servers, access tokens, and
infrastructure from being compromised).

---

## 1. Security & Protection Across Pipeline Stages

A secure CI/CD pipeline embeds security checkpoints at every phase of delivery without slowing down automation.

```
┌─────────────────┐     ┌─────────────────┐     ┌─────────────────┐     ┌─────────────────┐
│  Code / Commit  │ ──> │  Build & Scan   │ ──> │ Deployment Gate │ ──> │ Runtime & Ops   │
└─────────────────┘     └─────────────────┘     └─────────────────┘     └─────────────────┘
  • Signed Commits        • Dependency Check      • Policy Enforcement     • Immutable Deploy
  • Pre-commit Hooks      • Container Scan        • Provenance Verification• Secret Redaction
  • Secret Scanning       • Artifact Signing      • Approval Gates         • Runtime Audit Log

```

### Stage 1: Commit & Version Control (Source Protection)

- **Secret Detection:** Scans code automatically before or upon pushing (using tools like GitGuardian or Trufflehog) to block
  API keys or passwords from reaching public or internal repositories.
- **Commit Signing:** Uses GPG/SSH keys to verify developer identity and ensure commit authenticity.
- **Branch Protection Rules:** Requires mandatory pull requests, code reviews, and passing status checks before code can be
  merged into `main` or release branches.

### Stage 2: Build & Test Stage (Integrity & Analysis)

- **Dependency & License Checks (SCA):** Verifies third-party packages for known vulnerabilities (CVEs) and malicious
  packages (e.g., typosquatting attacks).
- **Static Analysis (SAST):** Scans application source code for security flaws (e.g., SQL injection, XSS) during
  compilation/build.
- **Container Image Scanning:** Checks container base images (e.g., Dockerfiles) for OS-level vulnerabilities before pushed
  to a registry.
- **Software Bill of Materials (SBOM):** Generates an inventory of all component versions, libraries, and modules included in
  the build.

### Stage 3: Deployment & Delivery (Access & Gates)

- **Cryptographic Artifact Signing:** Digitally signs build artifacts (using tools like Sigstore/Cosign) so production
  environments only run trusted, un-tampered binaries.
- **Policy Enforcement (OPA / Gatekeeper):** Automatically blocks deployments that violate security policies (e.g., running
  containers as root or exposing insecure ports).
- **Environment Separation:** Isolates staging and production pipelines with least-privilege credentials.

## 2. Protecting the Pipeline Infrastructure

Attacks like SolarWinds or Codecov targeted the pipeline itself rather than the code base. Protecting the build environment
requires specific security controls:

| Threat / Risk                  | Protection Mechanism                                                                                                                                                    |
| ------------------------------ | ----------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **Leaked Pipeline Tokens**     | **Ephemeral Credentials:** Use Short-lived OIDC (OpenID Connect) tokens rather than static API keys stored in pipeline variables.                                       |
| **Malicious Pipeline Scripts** | **Immutability & Least Privilege:** Restrict who can edit `.github/workflows` or `Jenkinsfile` configs; run runner agents in isolated, disposable sandboxes/containers. |
| **Dependency Hijacking**       | **Private Proxy / Caching:** Use internal registries or lockfiles with cryptographic hash verification (e.g., `npm-shrinkwrap.json`, `go.sum`).                         |
| **Unauthorized Deployment**    | **Multi-Factor Approval Gates:** Require manual sign-offs or hardware security keys (FIDO2) for production releases.                                                    |

---

## 3. Essential Security Frameworks

To structure pipeline protection, teams commonly align with established industry frameworks:

- **SLSA (Supply-chain Levels for Software Artifacts):** A security framework developed by Google and OpenSSF to prevent
  tampering and guarantee artifact integrity.
- **NIST SP 800-204D:** Strategies for enhancing software supply chain security in CI/CD environments.
