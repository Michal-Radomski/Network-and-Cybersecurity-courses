`npm-shrinkwrap.json` is a configuration file generated by the Node Package Manager (**npm**) that locks down the exact
version tree of all dependencies, sub-dependencies, and transient packages for a Node.js project.

It works almost identically to a standard `package-lock.json` file, but with one critical operational difference:
**`npm-shrinkwrap.json` is designed to be published to the npm registry along with your package.**

---

## 1. Key Differences: `package-lock.json` vs `npm-shrinkwrap.json`

| Feature                        | `package-lock.json`                                            | `npm-shrinkwrap.json`                                                                            |
| ------------------------------ | -------------------------------------------------------------- | ------------------------------------------------------------------------------------------------ |
| **Primary Goal**               | Lock dependency versions during local development & CI builds. | Guarantee identical dependency versions for downstream consumers installing a published library. |
| **Published to npm Registry?** | **No** (npm ignores it during `npm publish`).                  | **Yes** (npm includes it during `npm publish`).                                                  |
| **Target Use Case**            | Applications, microservices, and internal projects.            | Published CLI tools, libraries, or reusable npm packages.                                        |
| **Precedence**                 | Lower priority than `npm-shrinkwrap.json`.                     | **Highest priority** (overrides `package-lock.json` if both exist).                              |

---

## 2. Why Use `npm-shrinkwrap.json`?

When you publish a package (e.g., a library or CLI tool) with regular `package.json` semver ranges like `^1.2.0`, users who
install your package (`npm install your-package`) will automatically fetch the latest minor or patch versions of your
sub-dependencies.

If a third-party dependency publishes a breaking patch update or gets compromised, your end users could experience bugs or
security vulnerabilities.

By including `npm-shrinkwrap.json`:

- **Deterministic Builds:** You enforce that every end user installs the exact same dependency tree you tested against
  locally.
- **Security & Integrity:** Locks explicit package hashes (`integrity` SHA-512 checks) to prevent supply chain attacks like
  dependency hijacking.

---

## 3. How to Create and Use It

You can convert an existing `package-lock.json` into a shrinkwrap file using the npm CLI:

```bash
# Converts your package-lock.json into npm-shrinkwrap.json
npm shrinkwrap

```

### File Structure

The internal JSON structure is identical to `package-lock.json`:

```json
{
  "name": "example-cli-tool",
  "version": "1.0.0",
  "lockfileVersion": 3,
  "requires": true,
  "packages": {
    "node_modules/express": {
      "version": "4.18.2",
      "resolved": "https://registry.npmjs.org/express/-/express-4.18.2.tgz",
      "integrity": "sha512-5/PsL6iGPwN/XomXQSsRAe0KP2CBO3qZaC5Q/pzG50vji84X6AH28DQ3AJu662GTHQAON8VU6/jgL5tNwS/g1w=="
    }
  }
}
```

---

## 4. Best Practices & Warnings

> **Caution for Library Authors:** While shrinkwrapping CLI tools (like `create-react-app` or `eslint`) is widely
> recommended, shrinkwrapping standard libraries can lead to dependency duplication or version conflicts for applications
> consuming multiple libraries. Use it intentionally based on your project's distribution model.
