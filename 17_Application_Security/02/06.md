When navigating the world of internet engineering and cybersecurity, you will constantly run into two crucial acronyms:
**RFC** and **CVE**.

While **RFC** is about building the standards that make the internet work, **CVE** is about identifying and tracking the
security flaws that could break it.

---

## 1. RFC (Request for Comments)

An **RFC (Request for Comments)** is a formal document drafted by the **Internet Engineering Task Force (IETF)** and other
bodies. Despite the casual-sounding name "Request for Comments," RFCs are actually the **official, binding standards and
protocols** of the internet.

### What RFCs Do

Whenever a new technology or protocol is created to help computers talk to each other, it is written down as an RFC. This
ensures that whether you are using an Apple device, a Windows PC, or a Linux server, they all speak the exact same digital
language.

### Famous Examples of RFCs

Every foundational internet technology you use daily is defined by one or more RFCs:

- **RFC 791:** Internet Protocol (IPv4)
- **RFC 2616 (and updates):** Hypertext Transfer Protocol (HTTP/1.1)
- **RFC 5321:** Simple Mail Transfer Protocol (SMTP) — how emails are sent.
- **RFC 4122:** Universally Unique Identifiers (UUIDs).

### The Lifecycle of an RFC

RFCs are never modified once they are published. If a protocol needs to be updated or a bug is found in the standard, a
brand-new RFC is published that officially **"obsoletes"** or **"updates"** the older one.

---

## 2. CVE (Common Vulnerabilities and Exposures)

A **CVE (Common Vulnerabilities and Exposures)** is a unique, publicly identified security flaw in software or firmware. The
CVE program is overseen by the **MITRE Corporation** and funded by the US Department of Homeland Security.

### Why CVEs Matter

Before CVEs existed, different security companies used different names for the same software bug, causing massive confusion.
The CVE system provides a **common dictionary** so that when a security researcher, a software developer, and a system
administrator talk about a vulnerability, they know exactly which bug they are referring to.

### The Format of a CVE

Every registered vulnerability is assigned a unique tracking number in this format:

$$\text{CVE-YYYY-NNNNN}$$

- **CVE:** The prefix.
- **YYYY:** The year the vulnerability was officially discovered or reported.
- **NNNNN:** A unique sequential identifier.

> **Famous Example: Log4Shell (CVE-2021-44228)** This was a critical vulnerability discovered in late 2021 in a widely used
> Java logging library (Log4j). Because it had a standardized CVE number, organizations worldwide could instantly search
> their codebases to see if they were running the vulnerable version.

---

## RFC vs. CVE: Key Differences

| Feature            | RFC (Request for Comments)                            | CVE (Common Vulnerabilities and Exposures)                       |
| ------------------ | ----------------------------------------------------- | ---------------------------------------------------------------- |
| **Purpose**        | To define standard internet protocols and systems.    | To catalog and track known security vulnerabilities in software. |
| **Focus**          | **How things _should_ work** (Design & Construction). | **How things _actually_ broke** (Security Flaws).                |
| **Governing Body** | Internet Engineering Task Force (IETF).               | MITRE Corporation.                                               |
| **Tone**           | Constructive, standardizing, and architectural.       | Defensive, diagnostic, and risk-oriented.                        |
