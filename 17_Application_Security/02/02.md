**API Security** refers to the practices, tools, and strategies used to protect Application Programming Interfaces (APIs)
from abuse, unauthorized access, and malicious attacks.

Because modern applications—from mobile apps to cloud services—rely on APIs to transfer data and connect different systems,
APIs have become one of the primary targets for cybercriminals. If web applications are the "front doors" of a business, APIs
are the back alleys and service entrances; if left unlocked, they grant direct access to sensitive databases.

## The Core Pillars of API Security

To secure an API, you must implement defense-in-depth across five key areas:

### 1. Authentication (Who are you?)

Verifying the identity of the client (user, server, or application) making the request.

- **Best Practices:** Use robust token-based authentication like **OAuth 2.0** and **OpenID Connect** rather than simple,
  easily-stolen API keys. Never hardcode keys in client-side code.

### 2. Authorization (What are you allowed to do?)

Ensuring that an authenticated user has the explicit right to perform the requested action on the requested resource.

- **Best Practices:** Implement strict access control checks on every single API endpoint. Just because a user is logged in
  doesn't mean they should be allowed to view or edit another user's account data.

### 3. Encryption (Keep it private)

Protecting data while it is traveling over the network.

- **Best Practices:** Force **Transport Layer Security (TLS)** (HTTPS) for all API communication to prevent
  "man-in-the-middle" eavesdropping. Sensitive payload data can also be encrypted at the application layer.

### 4. Rate Limiting and Throttling (Don't get overwhelmed)

Controlling how many requests a user or IP address can make to your API within a specific timeframe.

- **Best Practices:** Set strict limits to prevent **Denial of Service (DoS)** attacks and stop brute-force bots from
  guessing passwords or scraping data.

### 5. Input Validation and Sanitization (Don't trust the client)

Never assuming the data coming into your API is safe.

- **Best Practices:** Validate incoming data against a strict schema (e.g., ensuring a "zip code" field only contains
  numbers). Sanitize inputs to prevent Injection attacks.

## The OWASP API Security Top 10 (Quick Snapshot)

Much like the standard OWASP Top 10, there is a dedicated list specifically for APIs. The most notorious API threats include:

| Threat                                         | What It Means                                                                                                                        | How to Stop It                                                                        |
| ---------------------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------ | ------------------------------------------------------------------------------------- |
| **BOLA (Broken Object Level Authorization)**   | An attacker manipulates an ID in an API request to access someone else's data (e.g., changing `/api/v1/user/99` to `/99` to `/100`). | Always validate that the logged-in user actually owns the requested resource ID.      |
| **BFLA (Broken Function Level Authorization)** | An ordinary user accesses administrative endpoints by guessing the URL (e.g., sending a request to `/api/v1/admin/deleteUser`).      | Implement role-based access control (RBAC) checks on every administrative endpoint.   |
| **Unrestricted Resource Consumption**          | The API doesn't limit things like file upload sizes, memory usage, or request frequency.                                             | Set rate limits, payload size limits, and execution timeouts.                         |
| **Unsafe Consumption of APIs**                 | Trusting data received from third-party APIs without validating it first.                                                            | Treat external APIs exactly like untrusted user input—validate and filter everything. |

---

> **The Role of API Gateways** Many organizations deploy an **API Gateway** as a single entry point for all API requests. The
> gateway acts as a reverse proxy, handling traffic routing, enforcing rate limits, performing authentication, and logging
> metrics before the requests even reach the backend services.
