To build secure web applications, developers must defend against both **client-side** attacks (aimed at the user's browser)
and **server-side** attacks (aimed at the backend infrastructure).

Two critical concepts that address these distinct environments are **Content Security Policy (CSP)** and **Server-Side
Request Forgery (SSRF)**.

---

## 1. Content Security Policy (CSP)

**Content Security Policy (CSP)** is a client-side security standard implemented as an HTTP response header. It allows
website administrators to restrict the resources (such as JavaScript, CSS, Images, and Iframes) that the browser is allowed
to load and execute for a given page.

### Why CSP Matters

Its primary purpose is to detect and mitigate **Cross-Site Scripting (XSS)** and data injection attacks. By default, browsers
trust all code delivered from a website. If an attacker successfully injects a malicious script, the browser runs it. CSP
stops this by telling the browser: _"Only run scripts that come from these specific, trusted domains."_

### How CSP Works (An Example)

When a server sends a webpage to a user, it includes the `Content-Security-Policy` header:

```http
Content-Security-Policy: default-src 'self'; script-src 'self' https://apis.google.com; img-src 'self' https://images.com;

```

This policy tells the browser:

1. **`default-src 'self'`**: Only load resources (images, fonts, etc.) from the website’s own domain by default.
2. **`script-src 'self' https://apis.google.com`**: Only execute JavaScript files originating from the site's own domain
   **or** Google's API domain. Any other script (including inline scripts injected by an attacker) will be blocked.
3. **`img-src 'self' https://images.com`**: Only load images from the local domain or `images.com`.

## 2. Server-Side Request Forgery (SSRF)

**Server-Side Request Forgery (SSRF)** is a critical server-side vulnerability where an attacker abuses a web application's
functionality to force the hosting server to make unauthorized HTTP requests to an arbitrary destination.

### How SSRF Works

In a standard web request, the client (user) requests a resource. In an SSRF attack, the attacker manipulates the application
into making the request _on behalf of_ the server.

Because the backend server sits inside the organization's private network, it often has access to internal resources that are
completely hidden from the public internet.

#### A Typical Scenario:

1. A web application has a feature where it fetches a profile picture from a URL provided by the user:
   `https://example.com/preview?url=https://other-site.com/avatar.png`
2. An attacker changes the `url` parameter to point to an internal-only IP address or metadata endpoint:
   `https://example.com/preview?url=http://169.254.169.254/latest/meta-data/` _(This is the local AWS cloud metadata
   service)_
3. The vulnerable backend server fetches the internal URL and returns the sensitive cloud credentials or configuration files
   directly to the attacker.

### Key Mitigation Strategies for SSRF

- **Strict Input Validation / Allowlists:** Only allow the server to fetch resources from a predefined list of trusted
  external domains.
- **Disable Unused URL Schemes:** If the application only needs to fetch HTTP/HTTPS images, block it from using schemes like
  `file://`, `gopher://`, or `dict://` which attackers use to read local server files.
- **Network Segmentation:** Ensure the backend web server cannot communicate with local system loopbacks (`localhost`,
  `127.0.0.1`) or private internal database subnets unless absolutely necessary.

---

## Summary: Client-Side vs. Server-Side

| Concept                                | Target Environment                       | Main Security Goal                                                                           | How It Is Managed                                                     |
| -------------------------------------- | ---------------------------------------- | -------------------------------------------------------------------------------------------- | --------------------------------------------------------------------- |
| **Content Security Policy (CSP)**      | **Client-Side** (User's Web Browser)     | Prevents XSS, unauthorized data exfiltration, and malicious script execution.                | Configured via HTTP response headers (`Content-Security-Policy`).     |
| **Server-Side Request Forgery (SSRF)** | **Server-Side** (Backend Infrastructure) | Prevents attackers from using the server as a proxy to attack internal or external networks. | Managed via input validation, network segmentation, and API gateways. |
