While the standard OWASP Top 10 focuses broadly on web application vulnerabilities, APIs behave differently. They expose
underlying application logic and database structures directly to clients, creating a unique set of security challenges.

To address these distinct threats, the **OWASP API Security Top 10** was created, highlighting the most critical risks
specific to Application Programming Interfaces (APIs).

---

## The OWASP API Security Top 10

| Rank      | Risk Category                                          | What It Means                                                                                                                                    |
| --------- | ------------------------------------------------------ | ------------------------------------------------------------------------------------------------------------------------------------------------ |
| **API1**  | **Broken Object Level Authorization (BOLA)**           | An attacker manipulates object IDs in an API request to access data they shouldn’t be allowed to see.                                            |
| **API2**  | **Broken Authentication**                              | Poorly configured login endpoints, token handling, or API keys allow attackers to impersonate legitimate users.                                  |
| **API3**  | **Broken Object Property Level Authorization (BOPLA)** | The API permits users to read or modify specific object fields/properties they should not have access to.                                        |
| **API4**  | **Unrestricted Resource Consumption**                  | The API has no limits on requests, file uploads, or execution time, leaving it vulnerable to DoS attacks or high cloud hosting bills.            |
| **API5**  | **Broken Function Level Authorization (BFLA)**         | Regular users are able to execute administrative or restricted functions due to missing endpoint-level checks.                                   |
| **API6**  | **Unrestricted Access to Sensitive Business Flows**    | Attackers automate interactions (like buying up concert tickets or spamming comments) because there are no safeguards against high-volume abuse. |
| **API7**  | **Server-Side Request Forgery (SSRF)**                 | The API is coerced into sending unauthorized HTTP requests from the hosting server to an internal or external destination.                       |
| **API8**  | **Security Misconfiguration**                          | Insecure default settings, unencrypted connections (missing TLS), or verbose error messages containing sensitive stack traces.                   |
| **API9**  | **Improper Inventory Management**                      | Failing to track deprecated API versions (e.g., leaving a vulnerable `v1` endpoint active alongside a secure `v2` endpoint) or "shadow" APIs.    |
| **API10** | **Unsafe Consumption of APIs**                         | Blindly trusting data received from third-party APIs or integrations without validating, filtering, or sanitizing it first.                      |

## Deep Dive into the Top Vulnerabilities

### API1: Broken Object Level Authorization (BOLA)

BOLA (previously known as IDOR) is the most common and dangerous API vulnerability. Because APIs frequently use parameters
like database IDs directly in URLs, attackers will try to guess or sequentialize these IDs to view other users' data.

- **The Attack:** An attacker intercepts a request to `api.com/v1/profile?id=5501`. They change the ID to `5502`. If the API
  serves up user 5502's profile details without checking if the attacker actually owns that account, it is a BOLA
  vulnerability.
- **The Fix:** Implement strict object-level authorization checks. The code must explicitly verify: _"Does the user making
  this request have permission to access the object with ID 5502?"_

### API3: Broken Object Property Level Authorization (BOPLA)

This category combines two older risks: **Excessive Data Exposure** and **Mass Assignment**. It focuses on unauthorized
interactions at the field-level of an object.

- **The Attack (Read side):** A mobile app calls `GET /api/v1/users/me`. The backend API returns the user's entire record,
  including their password hash and home address, assuming the mobile frontend will simply "hide" those fields. An attacker
  can easily intercept the raw JSON response to read the hidden data.
- **The Attack (Write side):** A user submits a profile update. They intercept the request and inject `"role": "admin"` into
  the payload. Because the API automatically binds the input parameters to the database object (mass assignment), the user is
  upgraded to an administrator.
- **The Fix:** Only serialize and return fields that the client explicitly needs. Utilize strict "allowlists" for input
  properties to prevent users from binding values to fields they shouldn't edit.

### API6: Unrestricted Access to Sensitive Business Flows

This is not necessarily an "implementation bug," but rather a design oversight. APIs that expose a specific, highly
exploitable flow (such as purchasing a hard-to-find gaming console, creating accounts, or booking reservations) can be easily
abused by automated bots.

- **The Attack:** An attacker writes a script to buy up a limited stock of sneakers the microsecond they go live on an online
  storefront, effectively locking out human buyers.
- **The Fix:** Implement advanced behavioral analysis, CAPTCHAs, or velocity checks to distinguish between human-initiated
  actions and automated bot scrapers.

### API9: Improper Inventory Management

Modern software deployment happens rapidly. It is easy for old, testing, or staging APIs to be left online and forgotten.

- **The Attack:** An enterprise updates their main API to `v2` with robust MFA. However, developers forget to decommission
  the legacy `/v1` endpoint. An attacker discovers the `/v1` endpoint, which does not require MFA, and uses it to access the
  production database.
- **The Fix:** Maintain an active, updated API inventory. Ensure outdated versions are deprecated and decommissioned
  promptly, and protect testing environments with the same security rigor as production.
