## What is Software Composition Analysis (SCA)?

**Software Composition Analysis (SCA)** is an automated security practice used to identify and manage the open-source
components embedded within a software application.

Modern software is rarely built entirely from scratch; developers rely heavily on open-source libraries, packages, and
frameworks to speed up development. While this is highly efficient, it introduces significant risks if those external
packages contain security flaws or restrictive legal licenses. SCA tools scan your code to build a comprehensive inventory of
these dependencies and flag potential risks.

---

## The Three Core Functions of SCA

### 1. Identifying Vulnerabilities

SCA tools scan your project's manifest files (like `package.json` for Node.js, `pom.xml` for Java, or `requirements.txt` for
Python) to catalog all open-source dependencies. It then cross-references this list against databases of known security
flaws—such as the **National Vulnerability Database (NVD)**—to alert you if a library you are using has a documented exploit.

### 2. License Compliance

Open-source software isn't always free of restrictions. Some components come with highly restrictive "copyleft" licenses
(like the GNU General Public License or GPL) that legally obligate a company to make their proprietary source code public.
SCA tools automatically detect the licenses attached to your dependencies to prevent accidental legal exposure.

### 3. Inventory and SBOM Generation

SCA tools are the primary mechanism for generating a **Software Bill of Materials (SBOM)**. By mapping out not just your
direct dependencies, but also _transitive dependencies_ (the libraries that your libraries rely on), SCA provides complete
visibility into your software's true digital footprint.

---

## Where SCA Fits in the Security Pipeline

SCA operates as part of the **Shift Left** philosophy, meaning security testing is integrated early in the development
lifecycle rather than right before a product launches.

```
[ Developer Writes Code ]
           │
           ▼
[ SCA Tool Scans Dependencies ] ──► (Alerts developer to outdated/vulnerable libraries)
           │
           ▼
[ Code Committed to Repository ]
           │
           ▼
[ CI/CD Pipeline Build ] ────────► (Blocks build if severe vulnerabilities are found)

```

---

## SCA vs. SAST: What’s the Difference?

Organizations often use SCA alongside **Static Application Security Testing (SAST)**, but they look at entirely different
things:

- **SAST (Inside-Out):** Scans the **proprietary code** written by your internal developers. It looks for flaws like poor
  encryption, SQL injection vulnerabilities, or bad coding practices written in-house.
- **SCA (Outside-In):** Scans the **third-party code** imported into the application. It doesn't analyze how your developers
  wrote code; it analyzes the safety and legality of the building blocks they borrowed from the open-source community.
