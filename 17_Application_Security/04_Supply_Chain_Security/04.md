## What is a Software Bill of Materials (SBOM)?

A **Software Bill of Materials (SBOM)** is a formal, structured record that lists all the components, libraries, and modules
used to build a piece of software.

Think of it exactly like an **ingredient list on a food label**. If you have a peanut allergy, you check the label before
eating. Similarly, if a major cyber vulnerability is discovered in a specific software library, an organization checks their
SBOMs to instantly see if any of their applications contain that dangerous "ingredient."

---

## Why SBOMs Matter

Historically, companies bought and deployed software as a "black box"—they knew what the software did, but had no idea what
third-party or open-source code was hidden inside it. SBOMs change this by providing total transparency.

- **Rapid Vulnerability Response:** When a new vulnerability (like the infamous Log4j flaw) is announced, security teams
  don't have to waste weeks manually scanning code. They can search their SBOM inventory database in seconds to identify
  every affected application.
- **Supply Chain Integrity:** Software vendors provide SBOMs to their customers to prove that their software is
  well-maintained and free of known, unpatched exploits.
- **Regulatory Compliance:** Governments and regulatory bodies around the world increasingly mandate that software vendors
  selling to critical infrastructure or government agencies must provide an SBOM.

---

## What Information is Inside an SBOM?

According to standard security frameworks, a comprehensive SBOM must include specific foundational data fields for every
component:

- **Supplier Name:** Who created the component.
- **Component Name:** The specific name of the library or package.
- **Version:** The exact version number currently in use.
- **Dependency Relationship:** How the component relates to the rest of the software (e.g., is it a direct component, or was
  it pulled in automatically by another library?).
- **Unique Identifiers:** Standardized tags like a **PURL (Package URL)** or **CPE (Common Platform Enumeration)** so
  security tools can perfectly match the component against global vulnerability databases.

---

## Standard SBOM Formats

Because software systems need to read and process SBOMs automatically, they cannot just be written in a standard text
document. They are generated in machine-readable formats (typically JSON or XML) using one of two dominant global industry
standards:

1. **CycloneDX:** A lightweight, security-focused standard developed by the OWASP foundation. It is highly optimized for
   rapid supply chain risk analysis and identifying vulnerabilities.
2. **SPDX (Software Package Data Exchange):** An open standard managed by the Linux Foundation. It is particularly robust at
   tracking software licenses and intellectual property compliance, alongside security tracking.
