## What is AWS IAM?

**AWS Identity and Access Management (IAM)** is a core security service in Amazon Web Services that controls
**Authentication** (who you are) and **Authorization** (what you can do).

It acts as the primary gatekeeper for your cloud environment. Whenever a person or an application tries to interact with an
AWS resource—like uploading a file to an S3 bucket or spinning up a virtual server—IAM checks to see if they are
authenticated and if they have explicit permission to perform that action.

---

## The 4 Core Elements of IAM

To understand IAM, you need to understand its four building blocks: **Users, Groups, Roles, and Policies.**

### 1. IAM Users

An IAM User represents a specific individual or application that interacts with AWS. A user has permanent credentials, which
can be a username and password (for signing into the web console) or Access Keys (for using the command line or APIs).

> _Best Practice:_ Avoid using the original **Root User** account for daily work. Create an individual IAM user (or use
> Federated single sign-on) for each person on your team.

### 2. IAM User Groups

An IAM Group is a collection of users. Instead of assigning security permissions to 20 different developers one by one, you
can create a "Developers" group, attach the required permissions to the group, and simply add the users to it. This prevents
administrative mistakes.

### 3. IAM Roles

An IAM Role is an identity that doesn't belong to a specific person. Instead, it is **temporarily assumed** by whoever needs
it. Roles do not use long-term passwords or access keys; they use short-lived, auto-rotating security credentials.

- **Service Roles:** You can give an EC2 virtual server an IAM Role so it can securely pull data from a database without you
  having to hardcode passwords inside your software.
- **Cross-Account Access:** You can allow a user from a completely different AWS account to temporarily assume a role in your
  account to perform a task.

### 4. IAM Policies

Policies are the actual rulebooks. They are written in **JSON** (JavaScript Object Notation) and define exactly what actions
are allowed or denied. You attach policies directly to Users, Groups, or Roles.

Here is a simplified example of an IAM Policy that allows a user to look at files in a specific S3 bucket, but nothing else:

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Effect": "Allow",
      "Action": "s3:ListBucket",
      "Resource": "arn:aws:s3:::my-company-data-bucket"
    }
  ]
}
```

---

## The Rule of Evaluation Logic

When an identity makes a request in AWS, IAM evaluates permissions using a strict, predictable logic:

1. **Implicit Deny (Default):** By default, all requests are blocked. If no policy explicitly grants a user permission to do
   something, they can't do it.
2. **Explicit Allow:** If a policy states that a user is allowed to perform an action, the gate opens.
3. **Explicit Deny (The Ultimate Trump Card):** If _any_ policy attached to the user explicitly states "Deny," the request is
   immediately blocked, **overriding** any "Allows" that might exist.

---

## Standard IAM Best Practices

- **Principle of Least Privilege:** Never give anyone full administrator access unless absolutely necessary. Give users only
  the exact permissions they need to complete their current job.
- **Enforce MFA:** Multi-Factor Authentication should be mandatory for the Root Account and all human users accessing the AWS
  console.
- **Ditch Long-Term Access Keys:** Use IAM Identity Center or IAM Roles for human logins and cloud workloads to avoid leaking
  long-term programmatic keys in code repositories.
