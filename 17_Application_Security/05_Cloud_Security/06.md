## What is a Container?

In software development, a **container** is a lightweight, standalone package that contains everything needed to run a piece
of software. This includes the application's code, its runtime environment, system tools, and libraries.

To understand containers, think of physical shipping containers:

- Before physical shipping containers, loading cargo onto a ship was difficult because boxes, barrels, and bags all had
  different shapes.
- A standard shipping container standardizes the shape. As long as the cargo fits inside the container, any crane can lift
  it, and any ship can carry it.

Similarly, a software container standardizes how code is packaged. It isolates the application from the underlying computer
it is running on. This means a developer can build an app on their own laptop, package it into a container, and it will run
exactly the same way on a corporate cloud server or a different operating system without any modifications.

---

## What is Container Security?

**Container Security** is the practice of protecting the entire container ecosystem—from the code inside the container to the
infrastructure that runs it—from threats, unauthorized access, and vulnerabilities.

Because containers share the host operating system's kernel and spin up or down in seconds, securing them requires different
tools than traditional virtual servers.

---

## The Core Pillars of Container Security

### 1. Image Security (The Supply Chain)

Containers are built using a blueprint called a "container image." If the base image you download from the internet contains
malware or vulnerable code, every container you run from it will be insecure.

- **Vulnerability Scanning:** Security tools automatically scan container images in a repository to check for outdated
  libraries or known security flaws before they are allowed to run.
- **Minimal Base Images:** Security teams use the smallest possible base images (like "Distroless" or Alpine Linux) that
  remove unnecessary tools (like web browsers or package managers) so hackers have fewer targets to exploit.

### 2. Runtime Security (The Active Environment)

Once a container is actively running, it must be monitored for malicious behavior.

- **Behavioral Monitoring:** If a container that is supposed to only serve web traffic suddenly tries to modify system files
  or download unauthorized scripts from the internet, runtime security tools will immediately flag or block it.
- **Immutability:** Containers are designed to be temporary and unchangeable. Instead of updating a live container,
  developers destroy it and launch a new version. Runtime security ensures no one is logging into a running container to
  manually alter its settings.

### 3. Isolation and Access Control (The Host Infrastructure)

Because multiple containers run on a single host machine, a major risk is **container breakout**—where a hacker compromises
one container and uses it to hack into the main host computer or neighboring containers.

- **Run as Non-Root:** Containers should never be run with administrative privileges ("root access"). If a non-root container
  is hacked, the attacker is trapped inside a heavily restricted sandbox.
- **Network Segmentation:** Virtual firewalls are used to isolate containers so they can only communicate with the specific
  applications they need to talk to.
