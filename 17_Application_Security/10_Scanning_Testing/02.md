**Dynamic Application Security Testing (DAST)**—often referred to as "black-box testing"—is a security testing methodology
that analyzes a running application from the outside-in.

Unlike SAST, which inspects the underlying source code, DAST simulates real-world cyberattacks against a fully deployed
application or service to uncover operational security vulnerabilities.

---

## 1. How DAST Works

DAST tools act much like automated penetration testers, interacting with an application through its front-end interfaces, web
browsers, or API endpoints.

```
┌─────────────────┐       HTTP Requests (Attacks)       ┌─────────────────┐
│                 │ ──────────────────────────────────> │                 │
│   DAST Scanner  │                                     │ Running App /   │
│ (Black-Box Tool)│ <────────────────────────────────── │ Staging Env     │
└─────────────────┘       HTTP Responses (Behavior)     └─────────────────┘

```

1. **Crawling:** The tool navigates through the running application to map out all accessible pages, form inputs, URLs, and
   API endpoints.
2. **Fuzzing & Attack Simulation:** The scanner sends malicious payloads (e.g., manipulated inputs, unexpected headers) to
   these endpoints.
3. **Response Analysis:** The tool analyzes the application's responses, status codes, and error logs to identify
   vulnerabilities or unexpected behaviors.

---

## 2. Key Vulnerabilities Detected by DAST

- **Runtime & Operational Flaws:** Authentication bugs, broken access control, session hijacking risks.
- **Server & Cloud Misconfigurations:** Insecure HTTP headers, exposed administrative portals, CORS misconfigurations.
- **Reflected & DOM-based Flaws:** Reflected Cross-Site Scripting (XSS), insecure redirects, and input handling issues
  exposed during execution.
- **API Weaknesses:** Insecure Direct Object References (IDOR), unauthenticated API endpoints, or weak rate limiting.

---

## 3. Comparing SAST vs. DAST

| Feature                 | SAST (Static)                                          | DAST (Dynamic)                                                 |
| ----------------------- | ------------------------------------------------------ | -------------------------------------------------------------- |
| **Approach**            | White-Box (Inside-out)                                 | Black-Box (Outside-in)                                         |
| **Execution**           | Analyzes unexecuted code/binaries                      | Requires a running application                                 |
| **SDLC Stage**          | Early (Coding / Build)                                 | Later (Test / Staging / QA)                                    |
| **Vulnerability Types** | Code flaws (e.g., hardcoded secrets, unsafe functions) | Runtime & configuration issues (e.g., auth bugs, server leaks) |
| **Language Dependence** | Requires specific language parsers                     | Language-independent (tests HTTP/network behavior)             |
| **Remediation Details** | Points directly to line number in code                 | Reports endpoint URL and HTTP payload/response                 |

---

## 4. Best Practices for Implementing DAST

- **Run in Staging Environments:** Execute DAST scans against isolated test or staging environments to prevent accidental
  data corruption or denial-of-service (DoS) conditions in production.
- **Use Authenticated Scans:** Provide the DAST tool with test credentials or API keys so it can crawl deeper into protected
  areas of the application behind login screens.
- **Combine with SAST (or IAST):** Use DAST alongside static analysis to achieve comprehensive security coverage—catching
  both code-level flaws and runtime infrastructure issues.
