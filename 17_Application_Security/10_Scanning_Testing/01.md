**Static Application Security Testing (SAST)**—often called "white-box testing"—is a security methodology that analyzes an
application's source code, byte code, or binaries for security vulnerabilities without executing the program.

It acts as an automated "code reviewer" that inspects code early in the software development lifecycle (SDLC) to catch flaws
before code is compiled or deployed.

---

## 1. How SAST Works

SAST tools build a model of the application's structure, data flow, and control flow to trace how data moves through the
codebase.

```
┌──────────────────┐     ┌──────────────────┐     ┌──────────────────┐
│   Source Code    │ ──> │   SAST Engine    │ ──> │ Vulnerability    │
│  (Unexecuted)    │     │  (Rule Matching &│     │ Report & IDE     │
└──────────────────┘     │   Data Flow)     │     │ Feedback         │
                         └──────────────────┘     └──────────────────┘

```

1. **Parsing:** The analyzer converts source code into an **Abstract Syntax Tree (AST)** or intermediate representation.
2. **Data Flow & Control Flow Analysis:** The engine tracks untrusted user inputs ("sources") to see if they reach sensitive
   functions ("sinks") without proper sanitization or validation (known as _Taint Analysis_).
3. **Rule Matching:** Code patterns are compared against predefined rule sets or security standards (e.g., OWASP Top 10,
   CWE).

---

## 2. Key Vulnerabilities Detected by SAST

- **Injection Flaws:** SQL injection, Command injection, Cross-Site Scripting (XSS).
- **Hardcoded Credentials:** API keys, passwords, or tokens accidentally committed in source code.
- **Insecure Cryptography:** Use of weak or deprecated algorithms (e.g., MD5, SHA-1, ECB mode).
- **Buffer Overflows:** Memory safety misconfigurations (primarily in languages like C/C++).
- **Insecure Configuration:** Missing security headers, exposed debug flags, or permissive CORS settings.

---

## 3. Advantages vs. Limitations

| Advantages                                                                                                  | Limitations                                                                                                            |
| ----------------------------------------------------------------------------------------------------------- | ---------------------------------------------------------------------------------------------------------------------- |
| **Shifts Left:** Finds flaws early when they are cheapest and easiest to fix.                               | **High False Positive Rate:** Frequently flags safe code patterns, requiring developer triage.                         |
| **100% Code Coverage:** Analyzes all accessible source code branches, including unexecuted paths.           | **No Runtime Visibility:** Cannot detect operational flaws, server misconfigurations, or authentication failures.      |
| **Exact Location Tracking:** Points directly to the file name and line number containing the vulnerability. | **Language Dependent:** Tools must be explicitly designed to support the specific programming language/framework used. |

---

## 4. Best Practices for Implementing SAST

- **Integrate into IDEs & Pull Requests:** Provide real-time feedback to developers as they write code (e.g., via IDE
  extensions or pre-commit checks) rather than delaying until build time.
- **Tune Rulesets Regularly:** Disable irrelevant rules and whitelist false positives to reduce alert fatigue and keep
  developer trust high.
- **Break Builds Wisely:** Configure CI/CD gates to only block deployments for High or Critical severity findings to avoid
  disrupting developer velocity.
