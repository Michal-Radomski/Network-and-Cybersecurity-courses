**CodeQL** and **Zed Attack Proxy (ZAP)** are two of the most popular open-source security tools used in modern DevSecOps
pipelines. While both focus on finding vulnerabilities, they approach application security from opposite directions:
**CodeQL** is a static analysis (SAST) engine, while **ZAP** is a dynamic scanning (DAST) tool.

---

## 1. CodeQL (Static Security Analysis)

Developed by GitHub (Microsoft), **CodeQL** treat code like data. It compiles source code into a relational database,
allowing developers and security engineers to query the codebase for security vulnerabilities, logic bugs, and data-flow
flaws using a SQL-like query language (QL).

### How CodeQL Works

1. **Database Creation:** CodeQL extracts your codebase into a structured database containing the Abstract Syntax Tree (AST),
   data-flow graph, and control-flow graph.
2. **Executing Queries:** You run pre-built or custom queries against the database to trace how untrusted data moves from a
   source (e.g., HTTP request parameter) to a sink (e.g., database execution call).
3. **Automated Feedback:** In GitHub, CodeQL runs automatically via **GitHub Advanced Security / Code Scanning** and leaves
   inline annotations directly on pull requests.

### Key Capabilities & Strengths

- **Deep Variant Analysis:** Once a zero-day or new vulnerability type is discovered, you can write a query to search across
  thousands of repositories for similar pattern variants.
- **Precise Taint Tracking:** Highly effective at tracing complex data flows across multiple files and functions.
- **Integrations:** Deeply integrated into GitHub Actions, CI/CD pipelines, and IDEs (VS Code extension).

---

## 2. OWASP Zed Attack Proxy / ZAP (Dynamic Security Analysis)

Maintained under the Open Web Application Security Project (OWASP) and the Software Security Project (SSP), **ZAP** is an
open-source dynamic web application security scanner and intercepting proxy.

### How ZAP Works

ZAP sits as a "man-in-the-middle" proxy between the tester's browser (or an automated script) and the target web application.

1. **Passive Scanning:** Inspects all HTTP/HTTPS requests and responses flowing through the proxy to flag missing security
   headers, insecure cookies, or information leaks without modifying traffic.
2. **Active Scanning / Attack Simulation:** Automatically crawls (spiders) the application and sends malicious payloads (SQL
   injection strings, XSS vectors, path traversal attempts) to endpoints to observe response behaviors.
3. **Manual Penetration Testing:** Provides an interactive GUI for security professionals to intercept, modify, and replay
   individual requests on the fly.

### Key Capabilities & Strengths

- **Language-Agnostic:** Tests the application from the outside via HTTP, regardless of what programming language or
  framework powers the back end.
- **Automation Ready:** Runs in headless/containerized modes (Docker) inside CI/CD pipelines to run automated baseline scans
  against staging environments.
- **Extensible Ecosystem:** Offers an extensive marketplace of add-ons, scripts, and API integrations for customization.

---

## 3. Comparison: CodeQL vs. OWASP ZAP

| Feature              | CodeQL                                                               | OWASP ZAP                                                       |
| -------------------- | -------------------------------------------------------------------- | --------------------------------------------------------------- |
| **Category**         | **SAST** (Static Application Security Testing)                       | **DAST** (Dynamic Application Security Testing)                 |
| **Testing Approach** | White-Box (Analyzes source code database)                            | Black-Box / Gray-Box (Tests running HTTP endpoints)             |
| **Execution Needs**  | Requires source code (and build environment for compiled languages)  | Requires a running instance of the application                  |
| **SDLC Stage**       | Early (Coding, Pull Requests, Build phase)                           | Later (Staging, QA, Pre-production testing)                     |
| **Best For**         | Finding code-level flaws, injection paths, and structural bugs early | Catching runtime bugs, auth flaws, and server misconfigurations |
| **Automation**       | GitHub Actions, CI pipelines                                         | Docker containers, ZAP Automation Framework, Jenkins            |

---

## Summary

In a mature DevSecOps workflow, **CodeQL and ZAP are complementary, not competing.** CodeQL catches security bugs in pull
requests _before_ code is merged, while ZAP tests the compiled, running application in staging to ensure that configurations,
authentication routines, and deployment environments are secure.
