## What is Threat Modeling?

**Threat modeling** is a structured process used in application security to identify potential security threats,
vulnerabilities, and optimization choices early in the development lifecycle. Instead of waiting until an application is
fully built to test its security, threat modeling allows security teams and developers to find design flaws before a single
line of code is written.

Think of it as looking at a blueprint for a house and realizing, _"Hey, we put a massive window right next to the front door
lock—someone could easily smash it and reach inside."_ It’s about thinking like an attacker to build a better defense.

---

## The Four Core Questions

Every effective threat modeling framework breaks the process down into four fundamental questions:

1. **What are we building?** (Understanding the architecture, data flows, and boundaries).
2. **What can go wrong?** (Brainstorming potential attacks and vulnerabilities).
3. **What are we going to do about it?** (Determining how to mitigate or fix the identified threats).
4. **Did we do a good job?** (Retrospective review to ensure the mitigations work and the model is accurate).

---

## The Threat Modeling Process

A typical threat modeling session follows a logical workflow to map out and dissect an application.

### 1. Model the Application (Deconstruct)

First, teams map out how data moves through the application. This is usually done using a **Data Flow Diagram (DFD)**. The
diagram highlights:

- **Trust Boundaries:** Points where data transitions from an untrusted zone (like the public internet) to a trusted zone
  (like an internal database).
- **Data Stores:** Databases, file systems, or caches.
- **Processes:** Code, microservices, or cloud functions that manipulate data.
- **External Entities:** Users, third-party APIs, or administrators.

### 2. Identify Threats

Once the architecture is clear, the team systematically identifies what could go wrong. To do this without guessing,
industries rely on frameworks. The most popular framework is **STRIDE**, created by Microsoft:

| Threat Category            | Description                                                                 | Security Property Violated | Example                                                                                 |
| -------------------------- | --------------------------------------------------------------------------- | -------------------------- | --------------------------------------------------------------------------------------- |
| **S**poofing               | Pretending to be someone or something else.                                 | Authentication             | An attacker logging in as an administrator using a stolen session token.                |
| **T**ampering              | Modifying data or code without authorization.                               | Integrity                  | Changing a product price from $100 to $1 in a web request.                              |
| **R**epudiation            | Claiming you didn't perform an action (and the system can't prove you did). | Non-repudiation            | A user deleting a critical database table, but the system logs don't record who did it. |
| **I**nformation Disclosure | Exposing sensitive data to unauthorized people.                             | Confidentiality            | An error message displaying plain-text user passwords on the screen.                    |
| **D**enial of Service      | Crashing or slowing down the system so legitimate users can't access it.    | Availability               | Flooding a login page with millions of fake requests to crash the server.               |
| **E**levation of Privilege | Gaining higher access levels than permitted.                                | Authorization              | A regular customer changing a URL parameter to gain access to the admin dashboard.      |

### 3. Mitigate Threats

For every threat identified, the team must decide on a course of action:

- **Mitigate:** Fix the issue by implementing a security control (e.g., adding encryption or multi-factor authentication).
- **Eliminate:** Remove the feature entirely if the risk is too high and the feature isn't necessary.
- **Transfer:** Push the risk elsewhere (e.g., outsourcing payment processing to Stripe instead of handling credit cards
  internally).
- **Accept:** Acknowledge the risk if the cost of fixing it outweighs the potential damage (usually reserved for low-risk
  items).

### 4. Validate

Finally, the team reviews the completed model. Did the developers implement the agreed-upon fixes? Did the security controls
actually work? This step ensures the threat model stays updated as the application evolves over time.

## Why Threat Modeling Matters

- **It's Cheaper:** Fixing a flaw in the design phase might take an hour of discussion. Fixing that same flaw after the app
  is deployed could require weeks of rewriting code, emergency patches, and potential legal fines.
- **It Fosters a Security Culture:** It brings developers, product managers, and security engineers to the same table,
  helping everyone understand _how_ and _why_ security decisions are made.
- **It Moves Beyond "Checklists":** Penetration testing and automated scanners are great at finding standard code bugs, but
  they often miss systemic, fundamental logic flaws in how an app is designed. Threat modeling catches those design flaws
  early.
