**SAML** and **SSO** are two foundational concepts in modern identity management that work together to simplify and secure
how users log into multiple applications.

Here is a clear breakdown of what each term means and how they connect.

---

## What is SSO (Single Sign-On)?

**Single Sign-On (SSO)** is a _concept and workflow_. It allows a user to log in once with a single set of credentials (like
a username and password) and automatically gain access to multiple independent software systems or applications without
having to log in again for each one.

### Benefits of SSO:

- **Better User Experience:** Users only need to remember one set of credentials and spend less time typing passwords.
- **Stronger Security:** IT teams can enforce centralized security policies (like mandatory Multi-Factor Authentication/MFA
  and strict password requirements) in one place.
- **Easier Offboarding:** When an employee leaves, revoking access at the central identity source instantly cuts off access
  to all connected applications.

---

## What is SAML (Security Assertion Markup Language)?

**SAML** (pronounced _SAM-ul_) is an open-standard _protocol_—specifically an XML-based framework—used to pass authentication
and authorization data between two parties.

While SSO is the _goal_ (logging in once), SAML is one of the primary _technologies_ used under the hood to make web-based
SSO actually work.

---

## How SAML Works: The Three Key Roles

SAML SSO relies on three main components communicating with each other:

1. **The User / Principal:** The human trying to access an application.
2. **The Identity Provider (IdP):** The central system that stores credentials and verifies who the user is (e.g., Okta,
   Microsoft Entra ID/Azure AD, Ping Identity).
3. **The Service Provider (SP):** The application or service the user wants to access (e.g., Salesforce, Slack, Zoom, Google
   Workspace).

```
   [ User ] ──────── 1. Tries to access App ───────> [ Service Provider ]
      │                                                     │
      │ <─────── 2. Redirects to IdP for login ──────────────┘
      │
      ├─── 3. Enters Credentials & MFA ───> [ Identity Provider ]
      │                                             │
      │ <─── 4. Receives Signed SAML Assertion ─────┘
      │
      └─── 5. Forwards SAML Assertion ────> [ Service Provider ]
                                                    │
        <─── 6. Validates XML & Grants Access ──────┘

```

### The Step-by-Step SAML Flow:

1. **Access Request:** The user tries to access a Service Provider (e.g., opening Slack).
2. **Redirect to IdP:** Slack checks for an active session. If none exists, it redirects the user's browser to the enterprise
   Identity Provider (e.g., Okta).
3. **Authentication:** The user logs in at the IdP prompt (using a password, MFA, passkey, or certificate).
4. **Assertion Generation:** Once verified, the IdP generates a **SAML Assertion**—a digitally signed XML document containing
   proof of identity, timestamp, and optional user attributes (like email address, job role, or group memberships).
5. **Assertion Post:** The IdP sends this SAML Assertion back to the user's browser, which automatically posts it to the
   Service Provider.
6. **Access Granted:** The Service Provider verifies the cryptographic signature on the SAML Assertion using a pre-shared
   public certificate from the IdP. If valid, the user is logged in automatically.

---

## Summary Comparison

| Feature         | Single Sign-On (SSO)                                    | SAML (Security Assertion Markup Language)                         |
| --------------- | ------------------------------------------------------- | ----------------------------------------------------------------- |
| **What is it?** | An **architectural concept** / authentication approach. | A specific **protocol / standard** (XML-based).                   |
| **Purpose**     | To let users log into multiple apps with one identity.  | To safely transport identity data between an IdP and an SP.       |
| **Analogy**     | Using a single passkey to open all doors in a building. | The standardized, official ID badge required to open those doors. |
