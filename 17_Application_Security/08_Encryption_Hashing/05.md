In AppSec, **Salt** and **Pepper** are two cryptographic techniques used to make password hashes vastly more secure against
dictionary attacks, rainbow tables, and data breaches.

Think of password hashing like a recipe: a plain hash is the basic dish, while salt and pepper are additional seasonings that
make it unique and much harder for hackers to recreate.

---

## 1. What is a Salt?

A **salt** is a **unique, cryptographically random string** added to a user's password _before_ it is run through a hashing
function (like Argon2id or bcrypt).

$$\text{Stored Hash} = \text{Hash}(\text{Password} + \text{Unique Salt})$$

### How it works:

- **Unique Per Account:** Every single user gets a completely different salt when they create or change their password.
- **Publicly Stored:** The salt is stored in the database right next to the hashed password in plain text.

### What problem it solves:

1. **Destroys Rainbow Tables:** Attackers often use massive, pre-computed lookup tables ("rainbow tables") to instantly look
   up common password hashes. Salting makes these pre-computed tables useless because every hash calculation must include the
   random salt.
2. **Hides Identical Passwords:** If User A and User B both choose `Password123!`, their hashes will look completely
   different in the database because their salts are unique.

---

## 2. What is a Pepper?

A **pepper** is a **system-wide secret string** combined with the password and salt before hashing.

$$\text{Stored Hash} = \text{Hash}(\text{Password} + \text{Unique Salt} + \text{Secret Pepper})$$

### How it works:

- **System-Wide Secret:** Unlike a salt, the pepper is usually identical across the application (or tied to a specific secret
  key).
- **Stored Separately:** The pepper is **never** stored in the database. Instead, it is kept in a separate, highly secure
  location—like an environment variable, an Application Key Vault, or a Hardware Security Module (HSM).

### What problem it solves:

- **Defends Against Database Leaks:** If a hacker gets full access to your database via SQL Injection, they get both the
  password hashes _and_ the salts. However, because the pepper lives on a completely different server or vault, the attacker
  still cannot brute-force the password hashes offline without that secret key.

---

## Summary Comparison

| Feature                   | Salt                                                      | Pepper                                                       |
| ------------------------- | --------------------------------------------------------- | ------------------------------------------------------------ |
| **Uniqueness**            | Unique for **every user account**.                        | One secret for the **entire application**.                   |
| **Storage**               | Stored in the **database** alongside the hash.            | Stored **externally** (e.g., Environment Variable, KMS/HSM). |
| **Secrecy**               | Public / Non-secret.                                      | Strictly secret.                                             |
| **Main Threat Mitigated** | Pre-computed rainbow tables & identical password mapping. | Database leaks / SQL Injection compromises.                  |
