In Application Security (AppSec), **Password Management** covers how applications handle, store, and process user credentials
safely to prevent unauthorized access and data breaches.

Because human-entered passwords remain a primary target for attackers, AppSec focuses on securing the entire password
lifecycle—from input to database storage.

---

## Core Pillars of AppSec Password Management

### 1. Secure In-Transit (The Wire)

- **TLS Everything:** Credentials must always be encrypted in transit using strong, modern TLS configurations to prevent
  interception (e.g., credential sniffing on untrusted networks).
- **Direct Post Requests:** Passwords should never be passed in URL query parameters, logs, or referral headers.

### 2. Secure Storage (At Rest)

Storing plaintext passwords—or even using simple, fast hashes like MD5 or SHA-256—is a major vulnerability because attackers
can reverse them using precomputed rainbow tables or high-speed GPU cracking.

- **Salted, Adaptive Hashing:** Modern AppSec mandates slow, memory-hard hashing algorithms specifically designed to
  frustrate brute-force attempts:
- **Argon2id** (Industry standard recommendation)
- **bcrypt**
- **pbkdf2** or **scrypt**

- **Unique Salts:** Every password must be assigned a unique, cryptographically random salt before hashing to ensure
  identical passwords result in completely different stored hashes.

### 3. Password Policies & Usability

Modern standards (such as **NIST SP 800-63B**) have shifted the focus from arbitrary complexity rules to practical, effective
defenses:

- **Allow Long Passwords:** Support long passphrases (e.g., up to 64–128 characters) including spaces and all printable
  characters.
- **Breached Password Screening:** Validate new passwords against lists of previously leaked/compromised passwords (e.g., via
  APIs like Have I Been Pwned).
- **Avoid Unhelpful Constraints:** Stop forcing arbitrary periodic password resets or requiring complex character
  combinations that lead users to create predictable patterns (e.g., `Spring2026!`).

### 4. Authentication Defense Mechanisms

- **Rate Limiting & Throttling:** Implement strict rate-limiting on login and password reset endpoints to defend against
  automated credential stuffing and brute-force attacks.
- **Secure Reset Flows:** Password reset tokens must be cryptographically secure, random, short-lived, single-use, and sent
  out-of-band (e.g., email or SMS).
- **Account Enumeration Prevention:** Ensure error messages do not reveal whether an email address exists in the system
  (e.g., standardizing responses to _"If an account exists with that email, a reset link has been sent"_).

---

## The Modern Shift: Moving Beyond Passwords

Because passwords carry inherent human-factor risks, AppSec increasingly emphasizes reducing reliance on passwords
altogether:

- **Multi-Factor Authentication (MFA):** Enforcing step-up authentication using time-based one-time passwords (TOTP), push
  notifications, or hardware security keys (FIDO2/WebAuthn).
- **Passkeys & WebAuthn:** Leveraging public-key cryptography on user devices to allow passwordless logins (e.g., using Touch
  ID, Face ID, or hardware security keys).
- **Federated Identity / SSO:** Delegating authentication to enterprise identity providers (IdPs) via protocols like OpenID
  Connect (OIDC) or SAML.
