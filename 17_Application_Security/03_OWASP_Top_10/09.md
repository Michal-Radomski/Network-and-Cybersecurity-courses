**Security Logging and Alerting Failures** (formerly known as _Insufficient Logging & Monitoring_) focuses on the visibility
gap in an application.

This vulnerability doesn’t mean an attacker broke in because of bad code; it means that **once an attacker is inside, the
system is completely blind to what they are doing.** Without proper logging, auditing, and alerting, attacks can go unnoticed
for weeks, months, or even years.

---

## Common Causes of Logging & Alerting Failures

These failures happen when an application doesn't record important security events, stores logs insecurely, or fails to warn
the security team when something suspicious occurs.

### 1. Important Events Are Not Logged

The system fails to record critical actions that could indicate an attack.

- **Example:** The application doesn't log failed login attempts, password changes, high-value financial transactions, or
  administrative actions. If a hacker attempts to brute-force an admin account 10,000 times, no one ever knows.

### 2. Logs Are Only Stored Locally

Logs are kept in plain text files directly on the local application server rather than being sent to a secure, centralized
server.

- **The Failure:** If an attacker compromises the server, the very first thing they will do is delete or alter the local log
  files to erase all evidence of their presence.

### 3. Cleartext Sensitive Data in Logs

Developers sometimes log too much information to help them debug errors, accidentally writing sensitive data into the log
files.

- **Example:** Writing plain-text passwords, session tokens, or credit card numbers into system logs. If an attacker gains
  read access to the logs, they get a goldmine of credentials.

### 4. Lack of Real-Time Alerting

The application generates logs, but they just sit in a storage repository. No one is actively monitoring them, and there are
no automated alerts for obvious malicious behavior.

- **Example:** A database suddenly exports 5 million user records at 3:00 AM. The event is logged, but no alarm is raised,
  allowing the attacker to comfortably exfiltrate the data and log out.

---

## Real-World Impact

- **Extended Breach Dwell Time:** The industry average time to detect a data breach is often over 200 days. This prolonged
  "dwell time" gives attackers months to quietly steal data, plant backdoors, and move laterally across networks.
- **Inability to Perform Forensics:** When a breach is finally discovered, a lack of clear logs means security teams cannot
  figure out _how_ the attacker got in, _what_ data they stole, or _which_ accounts are still compromised.
- **Compliance and Legal Penalties:** Regulations like GDPR, HIPAA, and PCI-DSS mandate strict auditing and logging
  standards. Failing to maintain these logs can result in massive compliance fines.

---

## How to Prevent It

Building a strong logging and alerting system requires turning raw data into actionable security intelligence.

> 🚨 **The Golden Rule:** If it's a security-sensitive action, log it immediately. If it looks like an active attack, alert
> the security team automatically.

- **Log Contextual Data:** Ensure all logs include user identifiers, timestamps, IP addresses, and whether the action
  succeeded or failed. (Be careful never to log passwords or secret keys).
- **Centralize and Protect Logs:** Stream logs in real-time to a dedicated, centralized log management platform or SIEM
  (Security Information and Event Management) system. Ensure these logs are append-only so they cannot be deleted or modified
  by users or attackers.
- **Establish Thresholds and Alerts:** Set up automated alerts for high-risk behavior, such as a single IP address failing to
  log in 50 times in one minute, or an unexpected administrative login from an unusual country.
- **Create an Incident Response Plan:** Ensure that when an alert triggers, your team has a clear, rehearsed playbook
  detailing exactly how to contain the threat and investigate the system.
