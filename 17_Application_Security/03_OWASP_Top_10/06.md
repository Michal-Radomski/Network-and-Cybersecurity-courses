**Insecure Design** focuses on flaws inherent to the architecture and design of an application rather than mistakes made
during the coding phase.

It was added to the OWASP Top 10 to highlight a crucial distinction: **you can write perfect, bug-free code and still have an
insecure application if the underlying design is fundamentally flawed.** If a building is designed without locks on the front
doors, it doesn't matter how well the walls are built—it's still insecure.

---

## The Difference Between Insecure Design and Insecure Coding

It helps to think of design and implementation as two different phases of building software:

- **Insecure Coding (Implementation Flaw):** The design required a secure login wall, but the developer forgot to use
  parameterized queries, creating a SQL Injection vulnerability. This is a bug that can be patched in the code.
- **Insecure Design (Architectural Flaw):** The application was designed to allow users to reset their passwords by answering
  a single, easily guessable security question (like "What color was your first car?"). The code works exactly as intended
  without bugs, but the business logic itself is fundamentally insecure.

---

## Common Examples of Insecure Design

Design flaws often stem from a lack of threat modeling or failing to anticipate how an attacker might abuse a feature.

### 1. Lack of Rate Limiting on Costly Operations

An application allows users to generate a complex, PDF-formatted financial report. Because the design doesn’t limit how many
times a user can click "Generate" per minute, an attacker can script thousands of concurrent requests, overloading the
server's CPU and causing a Denial of Service (DoS) for everyone else.

### 2. E-Commerce Price Manipulation

A retail app splits its checkout process across multiple steps. The system calculates the total price in Step 1, but relies
on the client's browser to pass that total price string over to the final payment processing gateway in Step 3. An attacker
intercepts the network request at Step 3 and changes the price from `$100.00` to `$1.00`.

### 3. Trusting Client-Side Validation

A voting web application uses frontend JavaScript to ensure a user can only click the "Vote" button once. However, the
backend server doesn't check if the user's account has already voted. An attacker completely bypasses the browser UI and
sends thousands of automated API requests directly to the backend server.

---

## Real-World Impact

- **Business Logic Abuse:** Attackers exploit the intended workflow of the application to cheat systems, bypass payment
  walls, or rig voting systems.
- **System Resource Exhaustion:** Flawed logic can allow malicious actors to drain server processing power, storage, or
  external API credits.
- **Irreparable Flaws:** Design flaws are notoriously difficult and expensive to fix after software has been fully built and
  deployed because fixing them often requires rewriting core architectural foundations.

---

## How to Prevent It

Fixing insecure design requires moving security to the very beginning of the software development lifecycle, a concept known
as **"shifting left."**

> 📐 **The Golden Rule:** You cannot code your way out of a bad design. Security must be a primary requirement during the
> initial planning phases, not an afterthought.

- **Establish a Secure Development Lifecycle (SDLC):** Integrate security professionals into the initial brainstorming and
  architecture phases of a project.
- **Use Threat Modeling:** Before writing a single line of code, map out the application's architecture and actively ask,
  _"How could an attacker abuse this specific workflow?"_
- **Leverage Secure Design Patterns:** Use battle-tested architectural frameworks and patterns that inherently prevent common
  flaws (such as centralized authentication and robust server-side validation).
- **Segregate Tiers:** Keep frontend components completely isolated from critical business logic and database layers,
  ensuring the server always has the final say on data integrity and authorization.
