**Injection** is a classic and highly dangerous security vulnerability on the OWASP Top 10 list. It happens when an
application takes untrusted, user-supplied data and sends it directly to an interpreter (like a database engine, a system
shell, or an XML parser) as part of a command or query.

Because the application fails to separate the _data_ from the _executable code_, the interpreter gets tricked into executing
unintended commands or accessing data without proper authorization.

---

## Common Types of Injection

Injection flaws can happen anywhere an application talks to an external interpreter. The most frequent types include:

### 1. SQL Injection (SQLi)

This occurs when user input is concatenated directly into a database query.

- **The Scenario:** A login form asks for a username. Instead of a normal name, an attacker types: `' OR '1'='1`.
- **The Result:** The database interprets `'1'='1` as a true statement, bypasses the password check entirely, and logs the
  attacker in as the first user in the database (usually the administrator).

### 2. Command Injection

This happens when user input is passed directly to the underlying operating system's command line shell.

- **The Scenario:** A web utility lets a user type an IP address to ping a server. An attacker types: `8.8.8.8; rm -rf /`.
- **The Result:** The server executes the ping command, immediately followed by the semicolon `;` separator, and then runs
  the second command, which attempts to delete files on the server.

### 3. Cross-Site Scripting (XSS) / HTML Injection

While XSS has its own historical categories, it is fundamentally a form of injection where malicious JavaScript is injected
into a web page and executed by a victim's browser.

---

## Real-World Impact

Injection attacks are a favorite among hackers because they strike at the core data layer of an application:

- **Total Data Exposure:** Attackers can read every piece of data inside a database, including passwords, credit cards, and
  personal records.
- **Data Manipulation:** Databases can be wiped, corrupted, or altered (e.g., changing account balances).
- **Remote Code Execution (RCE):** In the worst cases, command injection allows an attacker to take complete control of the
  host server, using it to launch further attacks inside the corporate network.

---

## How to Prevent Injection

Preventing injection requires a strict separation between data and commands.

> 🛠️ **The Golden Rule:** Treat all user input as fundamentally hostile. Never concatenate user input directly into
> executable strings or commands.

- **Use Parameterized Queries (Prepared Statements):** For databases, always use prepared statements. This forces the
  database engine to treat user input strictly as a literal value, never as executable code, completely neutralizing SQLi.
- **Input Validation (Allow-listing):** Implement strict validation on the server side. If a field expects a number, reject
  anything that contains letters or special characters.
- **Escape Special Characters:** If you must pass input to an interpreter, use contextual escaping functions designed to
  strip out or neutralize characters that hold special meaning to that interpreter (like quotes, semicolons, or backslashes).
- **Avoid Shell Commands:** Whenever possible, use built-in programming language APIs to interact with the system instead of
  calling raw OS shell commands.
