A **Message Authentication Code (MAC)** is a cryptographic security tool used to verify two things about a message:
**integrity** (that the message hasn’t been changed) and **authenticity** (that it actually came from the sender it claims to
come from).

While standard hashing ensures that data hasn't been accidentally corrupted, a MAC protects data from being _intentionally
altered by an attacker_.

---

## How a MAC Works

A MAC relies on a **symmetric shared secret key** known only to the sender (Alice) and the receiver (Bob).

1. **Generation:** Alice takes her message and the secret key, runs them through a MAC algorithm, and generates a short code
   (the MAC). She sends both the message and the MAC to Bob.
2. **Verification:** Bob receives the message and the MAC. He takes the received message and the exact same secret key, and
   runs them through the MAC algorithm on his end.
3. **The Check:** If Bob’s calculated MAC matches the MAC Alice sent, Bob knows two things: the message was not modified in
   transit, and it must have been sent by Alice (since only she has the key).

```text
Alice: [Message + Secret Key] ───> MAC Algorithm ───> Generated MAC
                                                               │
(Sent over internet) ───> [Message] ───────────────────────────┼───> Do they match?
                          [MAC] ───────────────────────────────┘
                                                               │
Bob:   [Message + Secret Key] ───> MAC Algorithm ───> Calculated MAC

```

### Why regular hashing isn't enough

If Alice only sent a standard hash (like SHA-256) of her message, an attacker (Eve) could intercept it, alter the message,
calculate a _new_ hash for the altered message, and send it to Bob. Bob would have no way of knowing it was tampered with.
Because a MAC requires a secret key, Eve cannot generate a valid MAC for her fake message.

---

## What is HMAC?

**HMAC** stands for **Hash-based Message Authentication Code**. It is a specific, highly secure type of MAC that involves a
cryptographic hash function (like SHA-256 or SHA-3) combined with a secret cryptographic key.

Instead of reinventing the wheel, HMAC builds a security wall around existing, trusted hashing algorithms.

### Why not just use `Hash(Key + Message)`?

Early developers tried to create MACs by simply gluing the key and the message together and hashing them (e.g.,
`Hash(Key + Message)`).

However, this approach is vulnerable to a clever cryptographic attack called a **length extension attack**. If an attacker
knows the length of the key, they can append extra malicious data to the end of the message and calculate a perfectly valid
hash for it, completely bypassing security without ever knowing what the secret key actually was.

### How HMAC Fixes This (The Two-Pass System)

HMAC prevents this flaw by hashing the data twice using a clever "inner" and "outer" padding system.

Here is the simplified formula for how HMAC works:

$$HMAC(K, m) = H\Big((K \oplus opad) \parallel H\big((K \oplus ipad) \parallel m\big)\Big)$$

Where:

- **$H$** is the chosen hash function (like SHA-256).
- **$m$** is the message.
- **$K$** is the secret key.
- **$ipad$ and $opad$** (Inner and Outer Pad) are fixed constants that alter the key.
- **$\oplus$** represents the XOR bitwise operation.
- **$\parallel$** represents concatenation (gluing strings together).

**In plain English:**

1. **The Inner Pass:** The key is mixed with the inner pad ($ipad$), glued to the front of the message, and hashed.
2. **The Outer Pass:** The key is mixed with a different outer pad ($opad$). The result of the _inner pass_ is glued to the
   back of it.
3. **The Final Hash:** This entire combination is hashed one more time.

By nesting the hashes inside one another, HMAC completely destroys an attacker’s ability to use length extension attacks,
making it incredibly secure.

---

## Where are MACs and HMACs Used?

- **API Authentication:** When your phone communicates with a bank's server, it often uses HMAC to sign the request. The
  server verifies the HMAC to ensure the request wasn't altered mid-flight and came from your authorized app.
- **Network Protocols:** HMAC is a foundational component of secure internet protocols like **TLS/SSL** (HTTPS), **IPsec**
  (used in VPNs), and **SSH**.
- **JSON Web Tokens (JWT):** Web applications frequently use HMAC (specifically HS256) to securely sign session tokens given
  to users upon logging in.
