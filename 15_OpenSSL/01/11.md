The **Digital Signature Algorithm (DSA)** is a Federal Information Processing Standard (FIPS) for digital signatures.
Conceived in 1991 by the National Institute of Standards and Technology (NIST), it is used to provide **authentication**,
**data integrity**, and **non-repudiation** for electronic documents.

Unlike the Diffie-Hellman algorithm (which is used to _share secrets_), DSA is used to _prove identity_ and verify that a
message hasn't been altered during transit.

---

## The Core Concept: How it Works

Think of a digital signature like a wax seal on a letter. It proves who sent the letter (authentication) and ensures the
letter hasn’t been opened or altered (integrity).

DSA achieves this using **asymmetric cryptography** (a public/private key pair):

- **Private Key:** Known only to the signer. Used to _create_ the digital signature.
- **Public Key:** Known to everyone. Used by anyone to _verify_ the digital signature.

---

## The Three Steps of DSA

### 1. Key Generation

Before signing anything, the user must generate their keys based on mathematical parameters (primes $p$ and $q$, and a
generator $g$):

- The user chooses a **private key** ($x$), which is a random integer.
- The user calculates their **public key** ($y$) using the formula:

$$y = g^x \pmod p$$

### 2. Signing the Message

When Alice wants to sign a digital document:

1. **Hashing:** She runs the message through a cryptographic hash function (like SHA-2 or SHA-3) to create a fixed-size
   "digest" of the message.
2. **Random Element:** She chooses a random, temporary per-message secret value ($k$).
3. **Signature Creation:** Using the hash digest, her private key ($x$), and the temporary value ($k$), she performs modular
   arithmetic to output two numbers: **$r$** and **$s$**.

These two numbers $(r, s)$ together form the **digital signature**, which is attached to the message.

### 3. Verifying the Signature

When Bob receives the message and the signature $(r, s)$, he wants to make sure Alice actually sent it and that it hasn't
been modified. He uses Alice's **public key** to verify it:

1. Bob computes the hash of the received message.
2. He calculates a value $w$, which is the modular inverse of $s$.
3. He uses $w$, the message hash, and Alice's public key ($y$) to calculate two values ($u_1$ and $u_2$).
4. Finally, he calculates a verification value $v = (g^{u_1} \cdot y^{u_2} \pmod p) \pmod q$.

> **The Verdict:** If $v = r$, the signature is **valid**. If they don't match, the signature is invalid, meaning either the
> message was tampered with, or the wrong private key was used to sign it.

---

## Why DSA is Secure

Like Diffie-Hellman, DSA's security relies on the **Discrete Logarithm Problem**. It is easy to compute the public key $y$
from the private key $x$, but mathematically computationally infeasible to reverse-engineer $x$ from $y$.

Furthermore, because the signature generation involves a hashing step, even a tiny change to the message (like changing a
single letter or a decimal point) will result in a completely different hash, causing the verification step to fail
completely.

---

## Modern Evolution: ECDSA

While standard DSA is still used, modern security systems widely prefer **ECDSA** (Elliptic Curve Digital Signature
Algorithm). ECDSA uses the same conceptual framework as DSA but relies on elliptic curve cryptography instead of modular
exponentiation. This allows ECDSA to offer the exact same level of cryptographic security but with much smaller key sizes,
leading to faster computations and less bandwidth usage.
