**Secure Boot** is a security standard developed by the PC industry to ensure that a computer boots up using _only_ software
that is trusted by the original equipment manufacturer (OEM). It prevents malware, rootkits, and unauthorized operating
systems from hijacking your computer during the startup process before the main operating system even loads.

---

## How Secure Boot Works (The Core Concept)

Secure Boot relies entirely on asymmetric cryptography (public and private keys) embedded right into your computer's
motherboard firmware (**UEFI**, which replaced the old BIOS).

1. **The Factory Keys:** When a motherboard is manufactured, the OEM injects public keys into the UEFI firmware. The most
   critical one is the **Microsoft Production CA key**, which is included on almost every consumer PC.
2. **The Handshake:** When you turn on your computer, the UEFI firmware checks the digital signature of the first piece of
   software it loads (the bootloader).
3. **The Validation:** \* If the bootloader is signed with a private key that matches a trusted public key in the UEFI, the
   PC boots normally.

- If the bootloader is unsigned, tampered with, or signed by an unknown key, the UEFI blocks it, and the computer refuses to
  boot, showing a security warning.

---

## Secure Boot on Windows

Windows and Secure Boot are tightly integrated. Because Microsoft controls the dominant public key pre-installed on
motherboards, the process is seamless for Windows users.

- **The Boot Chain:** When you power on a Windows PC, the UEFI verifies the Windows Boot Manager (`bootmgfw.efi`). Once
  verified, Windows takes over and uses its own keys to verify the next steps: the Windows kernel, system drivers, and
  antimalware software.
- **Windows 11 Requirement:** Microsoft made Secure Boot a strict hardware requirement for Windows 11. If Secure Boot is
  turned off or unsupported by the motherboard, Windows 11 will officially refuse to install or update.

---

## Secure Boot on Linux

Historically, Secure Boot caused major headaches for Linux because Linux developers do not own the keys hardcoded into your
motherboard. To make Linux work without forcing users to turn off Secure Boot, the community developed two main solutions:

### 1. The "Shim" Bootloader (The Easy Way)

Most major, user-friendly Linux distributions (like Ubuntu, Fedora, Debian, and Red Hat) use a brilliant workaround called a
**Shim**.

- **Microsoft's Blessing:** The Linux distribution creates a tiny, simple bootloader called a "Shim." They send this Shim to
  Microsoft, and Microsoft signs it using their widely trusted third-party certificate.
- **The Handshake:** When your PC boots, the UEFI sees Microsoft's signature on the Shim and allows it to run.
- **The Hand-off:** Once the Shim is running, it takes over the boot process. The Shim contains the _Linux distribution's_
  own public key, which it uses to verify and launch the actual Linux kernel.

### 2. Machine Owner Keys (MOK)

If you are using a highly customizable distribution like Arch Linux, or if you compile your own custom Linux kernel modules
(like proprietary NVIDIA graphics drivers), Microsoft hasn't signed your files.

- Linux uses a utility called **MOK (Machine Owner Key)**.
- When you install a custom driver, the system generates a unique key pair.
- Upon reboot, a blue screen interface pops up before the OS loads, asking you to manually authorize this new key into your
  motherboard's firmware.
- Once you approve it, your custom Linux kernel or drivers are trusted locally by your specific machine.

---

## Summary: Windows vs. Linux Secure Boot

| Feature             | Windows                                                      | Linux                                                                                      |
| ------------------- | ------------------------------------------------------------ | ------------------------------------------------------------------------------------------ |
| **Trust Mechanism** | Uses Microsoft keys natively hardcoded into the motherboard. | Uses a Microsoft-signed "Shim" or custom Machine Owner Keys (MOK).                         |
| **Ease of Use**     | Completely invisible and automatic.                          | Automatic on major distros (Ubuntu/Fedora); requires manual MOK setup on advanced distros. |
| **Strictness**      | Mandated by the OS (Windows 11 won't run without it).        | Optional, though highly recommended for security.                                          |
