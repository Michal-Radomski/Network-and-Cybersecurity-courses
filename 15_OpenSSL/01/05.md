In public-key cryptography (asymmetric cryptography), keys are used in specific ways depending on whether you want to achieve
**secrecy (encryption)** or **authenticity (signing)**.

Here is exactly how the process works for both scenarios.

---

## 1. Using a Public Key to Encrypt (Confidentiality)

You use someone’s **public key** when you want to send them a secret message that _only they_ can read.

### The Process:

1. **The Sender Gets the Public Key:** Alice wants to send a secure message to Bob. She obtains Bob’s public key (which is
   freely available to anyone).
2. **Encryption:** Alice writes her message and uses Bob’s public key to run it through an encryption algorithm. This
   scrambles the text into unreadable ciphertext.
3. **Transmission:** Alice sends the ciphertext over the internet. Even if a hacker intercepts it, they cannot read it.
4. **Decryption:** Bob receives the ciphertext. He uses his matching **private key** (which only he possesses) to decrypt the
   message back into readable text.

> **Why this works:** The public key and private key are mathematically linked. A message encrypted with a specific public
> key can _only_ be decrypted by its corresponding private key.

---

## 2. Using a Private Key to Sign (Authenticity & Integrity)

You use your **private key** to digitally sign a document when you want to prove to the world that the message definitely
came from you, and that it hasn't been altered.

### The Process:

1. **Hashing the Message:** Alice wants to sign a message. Her computer first runs the message through a hashing function to
   create a unique, fixed-length digital fingerprint of the message (called a "hash").
2. **Signing (Encryption):** Alice encrypts this hash using her **private key**. This encrypted hash is her **digital
   signature**.
3. **Sending:** Alice sends the original message along with the digital signature to Bob.
4. **Verification:** \* Bob takes the received message and runs it through the same hashing function to get the current hash.

- Bob then uses Alice’s **public key** to decrypt the digital signature, revealing the original hash Alice generated.
- Bob compares the two hashes. If they match perfectly, Bob knows two things: Alice definitely sent it (authenticity), and
  the message wasn't changed in transit (integrity).

---

## Summary Comparison

| Goal                                      | Key Used to Lock (Encrypt/Sign) | Key Used to Unlock (Decrypt/Verify) | Who Holds the Keys?                                       |
| ----------------------------------------- | ------------------------------- | ----------------------------------- | --------------------------------------------------------- |
| **Encryption** (Keep it secret)           | Recipient's **Public Key**      | Recipient's **Private Key**         | Anyone can lock it; only the recipient can unlock it.     |
| **Digital Signature** (Prove who sent it) | Sender's **Private Key**        | Sender's **Public Key**             | Only the sender can lock it; anyone can unlock/verify it. |

---

---

Combining both techniques is exactly how secure, modern communication works. When you sign a message _and_ encrypt it, you
ensure **both confidentiality** (only the recipient can read it) and **authenticity** (the recipient knows for sure it came
from you).

Here is how this scenario plays out step-by-step between **Alice (the Sender)** and **Bob (the Recipient)**.

---

## The Scenario: Alice sends a secure, signed message to Bob

To pull this off, Alice needs two things before she starts:

1. Her own **Private Key**.
2. Bob’s **Public Key**.

### Step 1: Alice Signs the Message (Authenticity)

First, Alice creates the proof that the message is genuinely from her.

- Alice writes her message.
- Her system creates a unique digital fingerprint (hash) of that message.
- She encrypts that hash using her own **Private Key**. This creates her **Digital Signature**.
- She attaches this signature to the original message.

### Step 2: Alice Encrypts Everything (Confidentiality)

Now, Alice needs to lock the message so only Bob can open it.

- Alice takes the combined package (the message + her digital signature).
- She encrypts the entire package using **Bob’s Public Key**.
- The message is now a scrambled mess of ciphertext. Even if someone intercepts it, they cannot read the message or see the
  signature. She sends it to Bob.

---

## The Receiving End: How Bob opens it

When Bob receives the package, he reverses the process by doing the exact opposite steps.

### Step 3: Bob Decrypts the Package

- Bob uses his own **Private Key** to decrypt the package.
- Because only Bob has his private key, he is the only person on Earth who can unlock this layer. He now has the readable
  message and Alice's digital signature.

### Step 4: Bob Verifies the Signature

- Bob takes Alice’s **Public Key** to decrypt the digital signature, which reveals the original hash Alice made.
- Bob runs the decrypted message through the same hashing function to generate a new hash.
- He compares the two hashes. If they match, Bob knows with absolute certainty that Alice wrote the message and that it
  wasn't altered by anyone else.

---

## Summary of the Key Flow

To keep it simple, think of it as a two-envelope system:

$$\text{Message} \xrightarrow{\text{Signed with Alice's Private Key}} \text{Signed Message} \xrightarrow{\text{Encrypted with Bob's Public Key}} \text{Secure Package}$$

- **Alice uses:** Her Private Key (to sign) $\rightarrow$ Bob's Public Key (to encrypt).
- **Bob uses:** His Private Key (to decrypt) $\rightarrow$ Alice's Public Key (to verify).
