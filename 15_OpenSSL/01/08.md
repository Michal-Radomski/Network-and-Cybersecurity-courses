To understand the boot process of a modern computer (especially one running Linux), it helps to look at **Shim** and **GRUB**
as a relay team. Both are bootloaders, but they have completely different jobs at different stages of the startup process.

Here is a breakdown of what Shim is, followed by how it differs from GRUB.

---

## What is Shim?

**Shim** is a very small, simple, and lightweight bootloader designed for a single purpose: to bridge the gap between UEFI
Secure Boot (which trusts Microsoft keys) and Linux operating systems (which use their own keys).

Because almost all computer motherboards come from the factory pre-loaded only with Microsoft’s digital signatures, a
standard Linux bootloader would be blocked by Secure Boot. To fix this, the Linux community created Shim. They sent it to
Microsoft, who reviewed it and signed it with their trusted digital signature.

When your computer boots up:

1. The motherboard’s firmware (UEFI) checks **Shim**.
2. It sees Microsoft's trusted signature and allows Shim to run.
3. Once running, Shim takes over and uses a built-in list of **Linux signatures** to verify and launch the next, more complex
   bootloader (usually GRUB).

---

## What is GRUB?

**GRUB** (Grand Unified Bootloader) is a full-featured, heavy-duty bootloader. Its job is to actually talk to you (the user)
and load the operating system into the computer's memory.

GRUB is what gives you the visual menu when you turn on your computer, allowing you to choose between different operating
systems (like dual-booting Windows and Linux) or choose an older version of the Linux kernel if something goes wrong.

---

## The Core Differences: Shim vs. GRUB

While both are essential for booting a modern Linux machine with Secure Boot enabled, they handle completely different tasks.

| Feature                | Shim                                                                               | GRUB                                                                              |
| ---------------------- | ---------------------------------------------------------------------------------- | --------------------------------------------------------------------------------- |
| **Primary Purpose**    | To bypass the Secure Boot restriction by using a Microsoft-approved signature.     | To provide a user menu, initialize hardware environment, and load the OS kernel.  |
| **Complexity**         | Extremely simple; it has no user interface and just acts as a security gatekeeper. | Highly complex; supports menus, themes, custom commands, and file system drivers. |
| **Who Signs It?**      | **Microsoft** (Third-Party Marketplace CA).                                        | **The Linux Distribution** (e.g., Ubuntu, Fedora) or the user themselves.         |
| **Order of Execution** | Runs **first** (directly after the UEFI firmware).                                 | Runs **second** (launched only after Shim verifies it).                           |
| **User Interaction**   | Invisible to the user (unless managing custom MOK keys via a basic blue screen).   | Highly interactive; displays the boot menu where you select your OS.              |

### The Visual Workflow:

```text
[ UEFI Firmware ]
       │ (Checks Microsoft signature)
       ▼
   [ SHIM ]
       │ (Checks Linux distribution signature)
       ▼
   [ GRUB ]
       │ (Displays boot menu & loads kernel)
       ▼
 [ Linux OS ]

```
