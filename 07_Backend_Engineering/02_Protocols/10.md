## What are Multiplexing and Demultiplexing?

In networking, **multiplexing** and **demultiplexing** are the processes that allow a computer to run multiple network
applications simultaneously (like web browsing, streaming music, and joining a Zoom call) over a single physical internet
connection.

Without these processes, your computer would only be able to do one network task at a time.

- **Multiplexing (at the Sender):** The process of gathering data chunks from different applications, packaging each chunk
  with header information (like ports) to create packets, and funneling them all into a single stream of data sent into the
  network.
- **Demultiplexing (at the Receiver):** The reverse process. The receiving computer accepts the single incoming stream of
  data, looks at the header information, separates the packets, and delivers them to the correct target applications.

---

## Where Is It Done?

Multiplexing and demultiplexing happen primarily at the **Transport Layer (Layer 4)** of the OSI model, directly inside the
host computer's operating system network stack.

### The Mechanism: Ports and Sockets

To understand _where_ and _how_ this happens, you have to look at **Sockets** and **Ports**.

A **Socket** is an internal doorway between the application layer and the transport layer. When an application wants to
network, the OS assigns it a unique **Port Number** (e.g., HTTP defaults to port 80/443, Minecraft defaults to 25565).

- **On the Sending Host:** The transport layer takes data from multiple sockets, slaps a header on them containing the
  destination port, and multiplexes them onto the network.
- **On the Receiving Host:** The transport layer receives the packets from the network layer, inspects the **Destination
  Port** inside the header, and directs the data to the specific socket tied to that port.

---

## Which Protocols Are Used?

Because this is a transport layer job, the two primary protocols responsible for multiplexing and demultiplexing are **UDP**
and **TCP**. However, they handle the process slightly differently based on how they identify a unique connection.

### 1. Connectionless Demultiplexing (UDP)

A UDP socket is identified by a **2-tuple**:

1. **Destination IP Address**
2. **Destination Port Number**

If two different computers send UDP packets to your server's Port 5000, the OS will direct _both_ sets of packets to the
exact same UDP socket. The application itself has to look inside the data to figure out who sent what.

### 2. Connection-Oriented Demultiplexing (TCP)

Because TCP tracks unique, stable connections, a TCP socket is identified by a **4-tuple**:

1. **Source IP Address**
2. **Source Port Number**
3. **Destination IP Address**
4. **Destination Port Number**

If two different clients connect to your web server on Port 443, the OS transport layer performs demultiplexing by looking at
all four pieces of information. It will seamlessly route them to **two entirely different TCP sockets**, keeping the
conversations strictly isolated.

## Other Types of Multiplexing in Networking

While the application-to-port mapping is the most common software-based multiplexing, multiplexing also happens lower down
the stack at the **Physical/Data Link layers** to share physical cables or wireless spectrum:

| Type                          | Layer    | How it Works                                                 | Common Use Case                |
| ----------------------------- | -------- | ------------------------------------------------------------ | ------------------------------ |
| **FDM** (Frequency Division)  | Physical | Divides a channel into different frequency bands.            | Cable TV, Radio, Wi-Fi         |
| **TDM** (Time Division)       | Physical | Users take turns transmitting data in designated time slots. | Cellular networks (GSM)        |
| **WDM** (Wavelength Division) | Physical | Multiplexes different colors of laser light.                 | Fiber-optic internet backbones |
