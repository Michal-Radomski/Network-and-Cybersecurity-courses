**HTTP/2** is a major revision of the HTTP network protocol used by the World Wide Web. Released in 2015 (based on Google's
experimental SPDY protocol), its primary goal was to address the performance limitations of HTTP/1.1 without changing the
core semantics of HTTP—meaning methods like `GET` and `POST`, status codes, and headers remained exactly the same.

Here is a breakdown of how HTTP/2 works, its core features, and how it improves web performance.

---

## The Problem with HTTP/1.1

To understand HTTP/2, it helps to understand what made HTTP/1.1 slow:

- **Head-of-Line (HOL) Blocking:** In HTTP/1.1, a browser can only request one resource per TCP connection at a time. If a
  large image or a slow script is being downloaded, all other requests behind it are blocked until it finishes.
- **Connection Overload:** To bypass HOL blocking, modern browsers open up to 6 parallel TCP connections per domain. However,
  opening new connections requires a costly TCP handshake every time, which wastes server resources and increases latency.
- **Redundant Headers:** HTTP is stateless, meaning every single request must carry headers (like cookies, user-agent, etc.).
  In HTTP/1.1, these headers are sent as plain text every single time, wasting bandwidth on repetitive data.

---

## Key Features of HTTP/2

HTTP/2 solves these bottlenecks by introducing a completely new way of packaging and transporting data between the client and
the server.

### 1. Binary Framing Layer

Unlike HTTP/1.1, which processes requests as plain text, HTTP/2 breaks communication down into a **binary framing layer**.

- Data is converted into smaller, discrete chunks called **frames** (e.g., HEADERS frames, DATA frames).
- This makes parsing data significantly faster and less error-prone for computers compared to reading text.

### 2. Multiplexing

This is the most impactful feature of HTTP/2. Instead of opening multiple TCP connections, HTTP/2 opens **one single TCP
connection** per domain.

- Within this single connection, multiple requests and responses can happen simultaneously.
- The binary frames of different files (CSS, JS, images) are interleaved and sent at the same time. The browser reassembles
  them on the other end.
- This completely eliminates Head-of-Line blocking at the application layer.

### 3. Header Compression (HPACK)

Webpages often require dozens of requests, and sending identical text headers with each request is highly inefficient. HTTP/2
utilizes **HPACK compression**, which:

- Compresses the headers using Huffman coding.
- Forces the client and server to maintain a shared "header table." If a header doesn't change between requests, the client
  only sends an index reference to that table rather than the full text strings, saving massive amounts of bandwidth.

### 4. Stream Prioritization

Because all resources are flying through a single connection simultaneously, the browser needs a way to tell the server what
to send first. HTTP/2 allows the browser to assign a "weight" or priority to streams. For example, a critical CSS file
required to render the layout of the page will be prioritized over a lower-priority image file at the bottom of the page.

### 5. Server Push

Historically, a server had to wait for the browser to request an asset. With Server Push, if a browser requests `index.html`,
the server can anticipate that the browser will also need `styles.css` and `script.js` and actively "push" those files into
the browser's cache before the browser even asks for them.

---

## Summary of Differences

| Feature               | HTTP/1.1                                          | HTTP/2                              |
| --------------------- | ------------------------------------------------- | ----------------------------------- |
| **Format**            | Text-based                                        | Binary                              |
| **Connections**       | Multiple parallel TCP connections                 | One single TCP connection           |
| **Concurrency**       | One request per connection at a time (Sequential) | Fully Multiplexed (Simultaneous)    |
| **Headers**           | Plain text, often repetitive                      | Compressed via HPACK                |
| **Resource Delivery** | Client must pull every file                       | Server can proactively "Push" files |

> **Note:** While HTTP/2 greatly optimized web performance, it had one flaw: if a packet was lost at the network level, the
> entire single TCP connection would freeze to recover it (TCP-level Head-of-Line blocking). This eventually led to the
> development of **HTTP/3**, which replaces TCP altogether with a UDP-based protocol called QUIC.
