**STUN** stands for **Session Traversal Utilities for NAT**. It is a standardized network protocol used to help devices
behind a NAT (Network Address Translation) gateway discover their public IP address, the type of NAT they are behind, and the
specific internet-side port associated with a local port.

In simple terms, STUN helps a device answer the question: _"How does the rest of the public internet see me?"_

---

## Why is STUN Needed?

In a typical home or office network, devices are assigned private IP addresses (like `192.168.1.5`). When these devices
connect to the internet, the router uses NAT to mask those private addresses behind a single public IP address.

This works perfectly for standard web browsing, where the device initiates the connection. However, it creates a massive
problem for **Peer-to-Peer (P2P) communications**, such as:

- **VoIP calls** (Voice over IP)
- **Video conferencing** (like Zoom or WebRTC-based applications)
- **Online gaming** (connecting directly to another player)

For a direct connection to happen, Person A needs to know Person B's public IP and port, and vice versa. Because the devices
only know their internal private IPs, they use a **STUN server** to find out their public identities.

---

## How STUN Works (Step-by-Step)

The STUN process is lightweight and happens in a matter of milliseconds:

```
[Internal Device] ----(1. Discovery Request)----> [STUN Server]
  (192.168.1.5)                                   (Public IP)

[Internal Device] <---(2. Discovery Response)--- [STUN Server]
                    "Your Public IP is 203.0.113.50
                     and your Port is 50001"

```

1. **The Request:** A device inside a private network sends a STUN "Discovery Request" to a public STUN server on the
   internet.
2. **The Inspection:** As the request passes through the local router, the router performs NAT, swapping the device's private
   IP/port for a public IP/port.
3. **The Response:** The STUN server receives the packet, looks at the source header to see where it came from, and notes the
   public IP and port. It then packs this information inside a "Discovery Response" message and sends it back to the device.
4. **The Result:** The internal device opens the packet and now knows its own public-facing IP and port. It can now share
   this information with a peer to establish a direct connection.

---

## The Limitations of STUN

STUN is incredibly efficient because it allows data to flow directly between peers without an expensive middleman server
bandwidth-wise. However, **STUN does not work with all types of NAT.**

Specifically, STUN fails when a network uses **Symmetric NAT** (commonly found in strict corporate networks).

- With Symmetric NAT, the router assigns a _different_ public port for every unique destination the device talks to.
- Therefore, the public port the router opens to talk to the STUN server will not be the same port used to talk to the peer.
  The information STUN provides becomes instantly useless.

### The Backup Plan: TURN and ICE

Because STUN fails in Symmetric NAT environments, it is rarely used alone. It is usually part of a larger framework called
**ICE (Interactive Connectivity Establishment)**, which acts as a fallback strategy:

- **STUN:** Try to make a direct connection first (Fastest, cheapest).
- **TURN (Traversal Using Relays around NAT):** If STUN fails because of a strict NAT, the traffic is entirely relayed
  through a third-party server. (Guaranteed to work, but expensive and introduces latency).
