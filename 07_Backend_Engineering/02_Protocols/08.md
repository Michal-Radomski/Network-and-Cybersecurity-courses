## What is UDP?

**UDP (User Datagram Protocol)** is one of the core communication protocols of the Internet protocol suite. Operating at the
transport layer, it provides a simple, low-overhead way to send data messages—called **datagrams**—across a network.

Unlike its more famous sibling TCP (Transmission Control Protocol), UDP is **connectionless** and **unreliable** (in
networking terms, "unreliable" just means it doesn't guarantee delivery, not that it's broken).

---

## How UDP Works: "Fire and Forget"

To understand UDP, it helps to compare it to mailing a standard postcard. You write the address, drop it in the mailbox, and
hope it gets there. You don't get a confirmation text when it arrives, and if it gets lost in transit, the post office
doesn't automatically mail a duplicate.

Here is how UDP handles data:

- **No Handshake:** TCP requires a "three-way handshake" to establish a connection before any data flows. UDP skips this
  entirely. It just starts blasting packets to the destination IP and port.
- **No Guarantees:** UDP does not care if the packets actually reach the destination.
- **No Ordering:** Packets can take different routes across the internet. With UDP, if Packet B was sent after Packet A,
  Packet B might still arrive first. UDP will not re-order them.
- **No Flow Control:** UDP will send data as fast as the application generates it, even if the receiving network is too
  congested to handle it.

---

## The UDP Packet Structure (Header)

Because UDP doesn't track connections or packet numbers, its header is incredibly lightweight. A UDP header is only **8
bytes** long, compared to a standard TCP header which is at least 20 bytes.

| Field                | Size    | Purpose                                                                      |
| -------------------- | ------- | ---------------------------------------------------------------------------- |
| **Source Port**      | 2 Bytes | The port of the sending application (optional).                              |
| **Destination Port** | 2 Bytes | The port of the receiving application.                                       |
| **Length**           | 2 Bytes | The total length of the UDP header and data.                                 |
| **Checksum**         | 2 Bytes | Used for basic error-checking (to see if the data was corrupted in transit). |

If the checksum detects that a packet was corrupted during transit, UDP simply drops it. It does not ask for a
retransmission.

---

## Why Use UDP? (The Advantages)

If UDP loses data and doesn't guarantee order, why do we use it? The answer is **speed and efficiency**.

1. **Ultra-Low Latency:** Without a connection handshake or acknowledgment packets traveling back and forth, UDP is much
   faster than TCP.
2. **Low Overhead:** Small headers mean less bandwidth is wasted on protocol data, leaving more room for the actual payload.
3. **Broadcast and Multicast:** UDP can send a single packet to multiple destinations simultaneously, which TCP cannot do.

---

## Common Use Cases

UDP is preferred in scenarios where **speed is critical, and losing an occasional packet doesn't ruin the experience.**

- **Online Gaming:** If you are playing a fast-paced shooter, you need real-time position updates. If packet #42 is dropped,
  your computer doesn't want to wait for a retransmission; it just wants packet #43 (the absolute newest position data).
- **Live Video Streaming & VoIP:** In a Discord call or a live sports stream, a dropped packet results in a tiny glitch in
  audio or a pixelated frame for a millisecond. Waiting for TCP to retransmit would cause the entire stream to freeze and
  buffer.
- **DNS (Domain Name System):** When you type a website URL, your computer sends a single UDP request to a DNS server and
  gets a single UDP response back. It’s incredibly fast.
- **VPNs (like OpenVPN or WireGuard):** VPN tunneling often uses UDP because encapsulating network traffic inside a heavy TCP
  connection can cause severe slowdowns.
