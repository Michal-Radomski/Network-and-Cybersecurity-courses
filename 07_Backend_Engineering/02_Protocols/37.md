**HTTPS** (Hypertext Transfer Protocol Secure) is the secure version of HTTP. It is the protocol used to encrypt the data
sent between your web browser and a website.

When you use standard HTTP, data is sent in **cleartext**. If you type a password into a regular HTTP site, anyone sitting on
the same network (like a hacker on public Wi-Fi) can read it. HTTPS solves this by wrapping the data in an encrypted layer
using **TLS (Transport Layer Security)**.

The entire HTTPS communication process can be broken down into two main phases: **The TLS Handshake** (securing the
connection) and **Data Transfer** (encrypting the actual traffic).

---

## Phase 1: The TLS Handshake (Setting Up Security)

Before any web pages, images, or passwords are sent, the browser and server must agree on how they will encrypt the data.
This is called the TLS Handshake, and it happens in a fraction of a second.

### 1. The "Client Hello"

Your browser kicks off the communication by sending a message to the website's server. This message includes:

- The TLS versions the browser supports (e.g., TLS 1.3).
- A list of **Cipher Suites** (combinations of cryptographic algorithms) the browser knows how to use.
- A string of random bytes (Client Random).

### 2. The "Server Hello"

The server replies back with its choices:

- The selected TLS version and Cipher Suite it wants to use from the browser's list.
- Its own string of random bytes (Server Random).
- **The SSL/TLS Certificate:** This is the most crucial part. The certificate contains the server’s **Public Key** and is
  signed by a trusted Certificate Authority (like Let's Encrypt or DigiCert), proving the website is legally who it claims to
  be.

### 3. Authentication & Verification

Your browser takes the server's certificate and verifies it against a built-in list of trusted Certificate Authorities.

- If the certificate is expired, fake, or issued to a different domain, your browser stops the connection and throws a "Your
  connection is not private" warning.
- If it checks out, the browser trusts the server's **Public Key**.

### 4. Key Exchange (Generating the Secret Key)

The browser and server must now create a shared secret key to encrypt the rest of their communication. Depending on the TLS
version (usually TLS 1.2 or 1.3), they use an algorithm like Diffie-Hellman to securely exchange cryptographic pieces.

By combining the **Client Random**, **Server Random**, and a shared secret pre-master key, both sides mathematically
calculate the exact same **Session Key** (Symmetric Key) simultaneously.

### 5. Handshake Finished

Both sides send an encrypted message stating, _"Everything I send from this moment on will be encrypted with our new Session
Key."_ The handshake is complete.

---

## Phase 2: Encrypted Data Transfer

Now that the handshake is over, the heavy lifting of asymmetric encryption is done. Asymmetric encryption (using
public/private keys) is highly secure but computationally slow.

For the actual browsing session, HTTPS switches to **Symmetric Encryption** using the shared **Session Key** they just
created.

```
[Browser] ----(Data encrypted with Session Key)----> [Server]
    |                                                    |
(Encrypts data)                                      (Decrypts data)

```

- **When you send data:** Your browser takes your request (like clicking a link or submitting a form), encrypts it using the
  Session Key, and sends it to the server.
- **When the server responds:** The server receives the ciphertext, decrypts it using its matching copy of the Session Key,
  processes your request, encrypts the resulting HTML/images with that same key, and sends it back.

Once you close the browser tab or terminate the session, that specific Session Key is destroyed. If you visit the website
again, a brand new handshake occurs.

---

## The Three Core Guarantees of HTTPS

HTTPS exists to provide three fundamental security protections:

1. **Encryption (Privacy):** Keeps the communication secret. An eavesdropper can only see _which_ domain you are visiting
   (e.g., `google.com`), but they cannot see the specific pages you look at, your search queries, or your passwords.
2. **Data Integrity (Anti-Tampering):** Ensures data cannot be modified or corrupted during transit. If an ISP or an attacker
   tries to inject ads or malware into the website's data mid-flight, the browser will detect that the encryption hashes
   don't match and reject the data.
3. **Authentication (Trust):** Proves that you are talking to the actual website and not an imposter. It prevents
   "Man-in-the-Middle" attacks where a hacker routes your traffic through their own server to steal your data.
