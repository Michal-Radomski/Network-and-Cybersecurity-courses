**HTTP/3** is the latest generation of the Hypertext Transfer Protocol (HTTP), finalized by the IETF in 2022. While HTTP/2
was a massive upgrade over HTTP/1.1, it had a fundamental flaw: it relied on TCP.

HTTP/3 completely drops TCP in favor of a new transport protocol called **QUIC** (Quick UDP Internet Connections). By moving
to QUIC, HTTP/3 solves the final remaining bottlenecks of web performance, offering faster connection times and seamless
handling of packet loss.

---

## The Problem HTTP/3 Solves: TCP Head-of-Line Blocking

In HTTP/2, all requests and responses are multiplexed into chunks (frames) and sent over a **single TCP connection**. This
works beautifully until a network hiccup occurs.

Because TCP guarantees that data must be delivered in the exact order it was sent, if a single packet containing a piece of
an image is lost in transit, **TCP pauses everything**. The browser cannot process the CSS, JavaScript, or text frames that
arrived after it until that single lost packet is retransmitted and received. This is called **TCP Head-of-Line (HOL)
Blocking**.

HTTP/3 eliminates this issue by swapping TCP for QUIC.

---

## What is QUIC?

Originally developed by Google and later standardized by the IETF, **QUIC** is a brand-new transport layer protocol that runs
on top of **UDP** (User Datagram Protocol).

UDP is famously fast because it doesn't check if packets arrive or force them into a specific order. QUIC takes the raw speed
of UDP and builds a layer of reliability, congestion control, and security directly on top of it.

### Core Features of QUIC and HTTP/3

### 1. True Multiplexing (No HOL Blocking)

Like HTTP/2, QUIC streams multiple files over a single connection. However, QUIC treats every stream as **entirely
independent**. If a packet belonging to "Stream A" (an image) is lost, the network drops it and requests a retry _only_ for
Stream A. Meanwhile, "Stream B" (a CSS file) and "Stream C" (a script) continue to download completely uninterrupted.

### 2. Zero-RTT Connection Handshakes (0-RTT)

Setting up a secure HTTP/2 connection requires multiple network round-trips: first a TCP handshake to establish a connection,
and then a TLS handshake to encrypt it.

QUIC combines the connection handshake and the cryptographic handshake into a **single step**.

- **1-RTT:** On your very first visit to a website, the connection is established in just one round-trip.
- **0-RTT:** If you have visited the site before, the client can use remembered security keys to send encrypted data in the
  _very first packet_ it fires off, resulting in near-instantaneous page loads.

### 3. Connection Migration

Traditional connections are tied to an IP address. If you are downloading a video on your smartphone using home Wi-Fi and
step outside onto a cellular (LTE/5G) network, your IP address changes. Under TCP, the old connection breaks, and the app has
to negotiate a brand-new connection from scratch, causing your video to freeze.

QUIC solves this using a unique **Connection ID** that is independent of your IP address. When your device switches networks,
it simply tells the server, "Hey, I'm the same Connection ID, just at this new IP." The data stream continues seamlessly
without a single dropped frame.

### 4. Mandatory Built-in Security

In HTTP/2, encryption (TLS) is technically optional in the protocol spec, even though web browsers forced its adoption. In
HTTP/3, **encryption is baked natively into QUIC**. The transport protocol integrates TLS 1.3 directly, meaning a QUIC
connection cannot exist without being fully encrypted. Even the packet headers are hidden from network snoopers.

## HTTP/1.1 vs. HTTP/2 vs. HTTP/3

| Feature                   | HTTP/1.1                  | HTTP/2                          | HTTP/3                                     |
| ------------------------- | ------------------------- | ------------------------------- | ------------------------------------------ |
| **Transport Protocol**    | TCP                       | TCP                             | **QUIC (over UDP)**                        |
| **Streams**               | 1 request per connection  | Multiplexed (Shared TCP stream) | **Multiplexed (Independent QUIC streams)** |
| **Head-of-Line Blocking** | At the Application layer  | At the Transport layer (TCP)    | **Completely Eliminated**                  |
| **Handshake Latency**     | High (Separate TCP + TLS) | Medium (Separate TCP + TLS)     | **Low (Combined 1-RTT / 0-RTT)**           |
| **IP Address Change**     | Breaks connection         | Breaks connection               | **Seamless Connection Migration**          |
| **Encryption**            | Optional (TLS)            | Optional (but standard)         | **Mandatory (TLS 1.3 built-in)**           |

---

## Current Adoption

You are likely using HTTP/3 every day without realizing it. Because tech giants like Google, Meta, and Cloudflare heavily
back it, a massive portion of global internet traffic (including YouTube, Instagram, and major e-commerce sites) runs on
HTTP/3. Modern browsers (Chrome, Firefox, Edge, Safari) all support it natively. If a network router blocks UDP traffic
(which some corporate firewalls do), browsers will automatically and gracefully fall back to HTTP/2.
