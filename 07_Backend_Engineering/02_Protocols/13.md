## What is HTTP/1.1?

**HTTP/1.1 (Hypertext Transfer Protocol version 1.1)** was released in 1997 as an upgrade to the original HTTP/1.0. It became
the bedrock of the modern web and remained the dominant protocol for nearly two decades.

While HTTP/1.0 required the browser to open a completely new TCP connection for every single asset (every image, stylesheet,
or script) on a webpage, HTTP/1.1 introduced massive optimizations to make web browsing faster and more efficient.

### Key Features of HTTP/1.1

- **Persistent Connections (Keep-Alive):** By default, HTTP/1.1 reuses a single TCP connection to send and receive multiple
  requests and responses. This eliminates the heavy overhead of performing a TCP three-way handshake for every single file.
- **Pipelining:** This allowed a client to send multiple requests down the same connection without waiting for the previous
  response to come back. (However, in practice, pipelining was prone to bugs and rarely enabled by default in browsers).
- **Chunked Transfer Encoding:** This allowed servers to start streaming content to the browser before knowing the exact
  final size of the file, which is crucial for dynamically generated webpages.
- **Cache Control:** It introduced sophisticated caching headers (like `Cache-Control` and `ETag`) to help browsers
  understand when they could load a file locally instead of fetching it from the server again.
- **The Host Header:** HTTP/1.1 made the `Host` header mandatory. This allowed a single physical server with a single IP
  address to host thousands of different websites (virtual hosting).

### The Major Flaw: Head-of-Line Blocking

Despite its upgrades, HTTP/1.1 suffered from a major structural bottleneck known as **Head-of-Line (HoL) Blocking**.

Because requests on a single TCP connection had to be processed in strict, sequential order, if the very first request (e.g.,
a heavy script) got delayed or stuck, **every other request behind it was blocked** until that first file cleared. To bypass
this, modern browsers resort to opening up to 6 parallel TCP connections per domain, which hogs system and network resources.

## Other Versions of HTTP

The web eventually outgrew HTTP/1.1, leading to the development of newer, faster versions.

### 1. HTTP/0.9 and HTTP/1.0 (The Early Years)

- **HTTP/0.9 (1991):** The original, extremely raw version of the protocol. It only supported a single method (`GET`) and
  could only return raw HTML text—no headers, no status codes, and no images.
- **HTTP/1.0 (1996):** Formally defined the protocol. It added headers, status codes (like `200 OK` and `404 Not Found`), and
  support for multi-media files (images, videos) via MIME types. However, its "one TCP connection per file" architecture made
  it incredibly sluggish as websites grew larger.

### 2. HTTP/2 (2015)

HTTP/2 was a massive overhaul designed explicitly to fix the Head-of-Line blocking issue of HTTP/1.1.

- **Multiplexing:** Instead of text, HTTP/2 converts data into binary frames and splits them into multiple independent
  streams over a **single TCP connection**. The browser can request hundreds of images, scripts, and CSS files _at the exact
  same time_, and they will all download concurrently down one pipe.
- **Header Compression (HPACK):** HTTP headers are often repetitive and bloated. HTTP/2 compresses them heavily, saving
  significant bandwidth.
- **Server Push:** Allows a server to speculatively send resources to the browser before the browser even asks for them
  (e.g., sending the CSS file immediately alongside the HTML).

### 3. HTTP/3 (2022)

While HTTP/2 fixed Head-of-Line blocking at the application layer, it still suffered from it at the _transport_ layer.
Because HTTP/2 uses TCP, if a single packet is lost on the network, TCP freezes **all** streams while it waits for the
missing packet to be retransmitted.

HTTP/3 completely drops TCP in favor of **QUIC (Quick UDP Internet Connections)**, a protocol built on top of UDP.

- **No Transport HoL Blocking:** In HTTP/3, if a packet belonging to an image is lost, only that specific image pauses. Your
  stylesheets, text, and other images keep downloading completely uninterrupted.
- **Zero Round-Trip Handshakes (0-RTT):** By combining the connection handshake and the TLS security handshake into a single
  step, HTTP/3 can connect and start streaming data almost instantly.
- **Connection Migration:** If you are downloading a video on your phone using Wi-Fi and suddenly walk out the front door
  onto a cellular network, HTTP/3 seamlessly migrates the connection without dropping or restarting the download.

## Summary Comparison

| Version      | Transport Protocol | Text or Binary | Multiplexing                 | Core Limitation                                              |
| ------------ | ------------------ | -------------- | ---------------------------- | ------------------------------------------------------------ |
| **HTTP/1.1** | TCP                | Text-based     | No                           | Application-level Head-of-Line blocking.                     |
| **HTTP/2**   | TCP                | Binary         | Yes (inside TCP)             | Transport-level Head-of-Line blocking if packets drop.       |
| **HTTP/3**   | UDP (via QUIC)     | Binary         | Yes (completely independent) | Requires network hardware to widely support UDP web traffic. |
