**TURN** stands for **Traversal Using Relays around NAT**. It is a network protocol used to bypass strict firewalls or
symmetric NATs by **relaying** network traffic through an intermediary server.

If STUN is the protocol that asks _"How do you see me?"_, TURN is the protocol that says _"I give up trying to connect
directly; please pass this note for me."_

---

## Why is TURN Needed?

While protocols like STUN are great for establishing direct Peer-to-Peer (P2P) connections, they fail completely when one or
both sides of the connection are behind a **Symmetric NAT** or a strict corporate firewall.

With Symmetric NAT, the router assigns a unique public port for every single destination IP and port. This means the public
mapping a device gets when talking to a STUN server cannot be reused to talk to a peer. Because the strict security of the
NAT blocks unsolicited incoming traffic from unknown peers, a direct connection becomes mathematically and architecturally
impossible.

TURN is the ultimate fallback mechanism for technologies like WebRTC (used in Google Meet, Discord, Zoom) and VoIP when P2P
fails.

---

## How TURN Works (Step-by-Step)

Instead of forcing a direct connection between two peers, TURN introduces a mutually accessible server in the public cloud to
act as a multimedia relay.

1. **The Allocation:** The internal device (Peer A) connects to a public TURN server and requests an "allocation."
2. **The Relay Assignment:** The TURN server grants the allocation and provides Peer A with a unique, public-facing **Relay
   Transport Address** (an IP and port owned by the TURN server).
3. **The Permission:** Peer A tells the TURN server that it permits traffic from Peer B's IP address.
4. **The Communication:** Peer A sends its media packets to the TURN server. The TURN server looks at the packet, matches it
   to the allocation, and forwards (relays) it to Peer B. When Peer B replies, it sends data back to the TURN server's relay
   address, which forwards it back down to Peer A.

## TURN vs. STUN: The Key Differences

In real-world applications, TURN servers almost always incorporate STUN functionality as well. However, their operational
roles are very different:

| Feature             | STUN                              | TURN                                         |
| ------------------- | --------------------------------- | -------------------------------------------- |
| **Primary Role**    | Discovery (Finds public IP/Port). | Relay (Forwards all traffic).                |
| **Connection Type** | Direct Peer-to-Peer (P2P).        | Mediated (Server-to-Peer).                   |
| **Success Rate**    | ~80% (Fails on Symmetric NAT).    | ~100% (Bypasses almost all NATs/Firewalls).  |
| **Server Cost**     | Very Cheap (Low bandwidth/CPU).   | Very Expensive (High bandwidth/CPU).         |
| **Latency**         | Low (Direct routing).             | Higher (Packets must travel to a middleman). |

---

## The Catch: Why Avoid TURN If Possible?

While TURN guarantees that a call or connection will succeed, network engineers treat it as a last resort because of **cost
and performance**:

- **Bandwidth Overhead:** In a STUN/P2P setup, video and audio data flow directly between users. The application provider
  pays nothing for that data transit. With TURN, _gigabytes or terabytes_ of heavy video data must stream through the
  application provider's cloud servers, leading to massive bandwidth bills.
- **Latency & Quality:** Because data has to take a detour through a TURN server rather than taking the shortest path across
  the internet, it adds latency. This can result in noticeable audio lag or video stuttering in real-time applications.

### The ICE Framework

Because of these trade-offs, applications use **ICE (Interactive Connectivity Establishment)**. ICE will always try STUN
first to get a cheap, fast, direct P2P connection. If it determines after a few milliseconds that the NAT is too strict, it
seamlessly falls back to TURN to ensure the connection doesn't drop.
