**HTTPS over QUIC** (formally standardized as **HTTP/3**) represents the most radical overhaul to web communication
infrastructure since the inception of the internet.

Historically, HTTPS relied on a strict, layered stack: **HTTP** (the data) sat on top of **TLS** (the encryption), which sat
on top of **TCP** (the transport protocol), which ran over **IP**.

HTTPS over QUIC collapses these boundaries. It eliminates TCP entirely, moving the web to **UDP** (User Datagram Protocol)
and fusing the transport layer (QUIC) and the encryption layer (TLS 1.3) into a single, tightly coupled protocol.

---

## The Protocol Stack Evolution

To understand why this change is so monumental, look at how the protocol stack has evolved:

| Traditional HTTPS (HTTP/1.1 & HTTP/2) | HTTPS over QUIC (HTTP/3)                                |
| ------------------------------------- | ------------------------------------------------------- |
| **Application:** HTTP / HTTP/2        | **Application:** HTTP/3                                 |
| **Security:** TLS 1.2 or TLS 1.3      | **Transport & Security:** QUIC _(incorporates TLS 1.3)_ |
| **Transport:** TCP                    | **Base Transport:** UDP                                 |
| **Network:** IP                       | **Network:** IP                                         |

---

## Core Features and Advancements

By eliminating TCP and integrating encryption directly into the transport layer, HTTPS over QUIC solves the fundamental
architectural flaws of the modern web.

### 1. Zero Head-of-Line Blocking (HOLB)

In HTTP/2 over TCP, you can download multiple files (images, JS, CSS) at the same time over a single connection using
"multiplexing." However, TCP views all of this data as one giant, continuous stream of bytes. If a single packet containing a
piece of an image is dropped over a poor connection, **TCP freezes the entire connection** for all files until that dropped
packet is retransmitted. This is called Head-of-Line Blocking.

**How QUIC fixes it:** QUIC natively understands stream multiplexing at the transport layer. Every file being downloaded is
treated as a completely independent stream. If packets for Stream A (an image) are lost, Stream B (a stylesheet) and Stream C
(a script) keep downloading completely unaffected.

### 2. Blazing Fast Connection Setup (0-RTT & 1-RTT)

Under traditional HTTPS over TCP, a browser has to do a multi-step dance just to start talking:

1. TCP Handshake (1 Round Trip Time / RTT)
2. TLS Handshake (1 to 2 RTTs) This means data only starts moving after 2 to 3 round trips between the browser and server.

**How QUIC fixes it:** QUIC merges the connection configuration and the TLS 1.3 handshake into a single step.

- **1-RTT:** For a brand new connection, a secure, encrypted tunnel is established in just **one round trip**.
- **0-RTT (Zero Round Trip Time):** If you have visited the site recently, the browser can send encrypted application data
  (like an HTTP GET request) inside the _very first packet_ it fires to the server, resulting in near-instantaneous page
  loads.

### 3. Connection Mobility (IP Hopping)

TCP connections are bound to a strict 4-tuple: _Source IP, Source Port, Destination IP, Destination Port_. If any of these
change, the TCP connection breaks. If you are downloading a video on your smartphone via Wi-Fi and step outside onto cellular
data, your IP address changes. The TCP connection dies, the video buffers, and your device has to execute a brand-new TCP/TLS
handshake.

**How QUIC fixes it:** QUIC introduces a concept called a **Connection ID (CID)**. The connection is tracked by a unique
random number rather than your IP address. If you switch from Wi-Fi to LTE, your phone sends a QUIC packet with the same
Connection ID from your new IP address. The server instantly recognizes it, and your download continues seamlessly without a
single millisecond of interruption.

### 4. Total Handshake Encryption

While traditional TLS encrypts the website data, it still leaves a decent amount of metadata exposed in plain text during the
initial connection setup (like the server name indication or packet numbers).

QUIC integrates TLS 1.3 so deeply that **almost the entire packet header is encrypted**. External onlookers, eavesdroppers,
or nosy network operators can only see that UDP traffic is flowing to a specific destination; they cannot see packet numbers,
stream IDs, or connection control signals.

## The Challenges of Deploying QUIC

Despite its massive performance benefits, HTTPS over QUIC has a few distinct hurdles:

- **UDP Blocking:** Because UDP was historically used for minor traffic (like DNS or video streaming) or DDOS attacks, many
  corporate firewalls and enterprise networks strictly block or throttle UDP traffic on port 443 out of caution.
- **The Fallback Loop:** Browsers are designed to handle this seamlessly. If a browser tries to connect via HTTPS over QUIC
  and the UDP packets are blocked or timeout, it automatically and silently drops back down to HTTP/2 over standard TCP.
- **CPU Overhead:** Encrypting and processing every individual UDP packet at high speeds takes more server CPU power than
  handling highly-optimized native TCP packets.
