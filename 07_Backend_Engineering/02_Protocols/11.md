## What is TLS?

**TLS (Transport Layer Security)** is a cryptographic protocol designed to provide secure communication over a computer
network. It is the invisible backbone of internet security, responsible for privacy, data integrity, and authentication when
data travels between a client (like your web browser) and a server.

If you see a padlock icon in your browser's address bar and a URL that starts with `https://` instead of `http://`, your
session is being secured by TLS.

> **Note on TLS vs. SSL:** SSL (Secure Sockets Layer) was the original security protocol developed by Netscape in the 1990s.
> TLS is simply the modern, upgraded version of SSL. While the tech industry moved away from SSL decades ago (SSL 3.0 was
> deprecated in 2015), people still frequently use the term "SSL" or "SSL/TLS" out of habit. Today, everyone uses TLS.

---

## The Three Pillars of TLS Security

TLS guarantees three fundamental protections for network data:

1. **Encryption:** Hides the data from eavesdroppers. If a hacker intercepts the packets while you are typing your credit
   card number into a website, they will only see a scrambled mess of random characters.
2. **Authentication:** Proves that the server you are talking to is actually who it claims to be. This prevents
   "Man-in-the-Middle" attacks where a malicious router pretends to be your bank.
3. **Integrity:** Ensures that the data cannot be altered or tampered with during transit without detection. If a bad actor
   tries to change a packet's contents, the receiving system will detect the corruption and drop it.

---

## How TLS Works: The TLS Handshake

Before an application can send encrypted data (like an HTTPS webpage load), TLS must establish a secure connection. It does
this via a process called the **TLS Handshake**, which typically runs right after the TCP three-way handshake.

While older versions (TLS 1.2) took multiple back-and-forth round trips, the modern standard, **TLS 1.3**, completes the
handshake in just **one round trip (1-RTT)**:

### Step 1: Client Hello

The client (browser) kicks off the handshake by sending a message to the server. This message includes:

- The TLS version it supports (e.g., TLS 1.3).
- A list of **Cipher Suites** (the cryptographic algorithms it knows how to use).
- A random string of bytes generated by the client.

### Step 2: Server Hello & Certificate

The server responds with its own choices and credentials:

- It selects the TLS version and cipher suite they will use from the client's list.
- It sends its **Digital Certificate** (which contains the server's public key and is verified by a trusted third-party
  Certificate Authority like Let's Encrypt or DigiCert).
- It sends its own random string of bytes.

### Step 3: Key Generation

Instead of passing a secret password over the internet, both the client and server use a mathematical process called a
**Diffie-Hellman key exchange**. By combining their own private data with the public data they just shared, both sides
independently calculate the exact same secret **Symmetric Session Key**.

### Step 4: Handshake Finished

Both sides send an encrypted "Finished" message. From this exact millisecond forward, all data passing between them is
encrypted using that shared session key.

---

## Symmetric vs. Asymmetric Encryption in TLS

TLS brilliantly combines two different types of cryptography to maximize both security and speed:

- **Asymmetric Encryption (Public/Private Keys):** This is used _only during the handshake_. It requires a lot of
  computational power, but it allows two entities who have never met to securely agree on a secret key without anyone else
  being able to intercept it.
- **Symmetric Encryption (Shared Key):** This is used _for the actual data transfer_. Once the handshake is over, both sides
  use the single shared session key to encrypt and decrypt data. Symmetric encryption is incredibly fast and efficient,
  allowing your computer to stream high-definition video securely in real-time.

---

## Where Does TLS Sit?

In the networking stack, TLS is an intermediary layer. It sits comfortably **between the Transport Layer (TCP) and the
Application Layer (HTTP, SMTP, FTP).**

Because it sits right below the application layer, almost any standard internet protocol can be wrapped in TLS for instant
security:

- **HTTP** becomes **HTTPS** (Secure Web Browsing)

Transport Layer Security (TLS) does not fit neatly into a single OSI layer. It acts as an intermediate layer between the
Transport layer (Layer 4) and the Application layer (Layer 7). [1, 2, 3] When mapping it directly to the strict 7-layer OSI
model, it is most often categorized under the Presentation layer (Layer 6) or the Session layer (Layer 5): [4, 5, 6, 7]

- Presentation Layer (Layer 6): It manages data encryption, decryption, compression, and formatting.
- Session Layer (Layer 5): It establishes, maintains, and terminates the secure connection (the TLS Handshake) between the
  client and server. [4, 5, 6, 7, 8]

How it works in practice: TLS runs on top of a reliable transport protocol like TCP. Even though "Transport" is in its name,
it serves application protocols (like HTTP for HTTPS) by providing secure, encrypted sockets rather than replacing the
transport layer itself. [1, 2, 9] You can read more about how this standardizes networking on
[Wikipedia's TLS Page](https://en.wikipedia.org/wiki/Transport_Layer_Security) or see how it interacts with the stack in the
[TechTarget TLS Overview](https://www.techtarget.com/searchsecurity/definition/Transport-Layer-Security-TLS). [3, 7]
