While **WebSockets** and **WebRTC** are both pillars of real-time web communication, they are built for fundamentally
different tasks.

**WebSockets** is a protocol designed for continuous, two-way communication between a **client and a server**. **WebRTC**
(Web Real-Time Communication), on the other hand, is a framework designed for direct, ultra-low latency **peer-to-peer (P2P)
streaming** of video, audio, and data.

Here is a breakdown of what WebSockets are, followed by a direct comparison with WebRTC.

---

## What are WebSockets?

A WebSocket is a persistent, bi-directional communication channel established between a client (like a browser) and a server.

### How WebSockets Work

Traditional HTTP relies on a request-response model—the client asks, and the server answers. If the server has new data, it
has to wait for the client to ask again. WebSockets solve this via an **HTTP Upgrade**:

1. The client sends a standard HTTP request asking to upgrade the connection.
2. If the server agrees, the connection switches from HTTP to the WebSocket protocol (`ws://` or `wss://`).
3. The underlying **TCP connection** stays open indefinitely. Now, either the client or the server can push data at any
   moment without the overhead of HTTP headers.

---

## WebSockets vs. WebRTC: The Comparison

They are not rivals; in fact, they are highly complementary. The easiest way to understand their differences is by looking at
their architecture, protocol layers, and primary use cases.

| Feature               | WebSockets                                                    | WebRTC                                                                                              |
| --------------------- | ------------------------------------------------------------- | --------------------------------------------------------------------------------------------------- |
| **Architecture**      | **Client-Server** (Star topology)                             | **Peer-to-Peer** (Mesh topology)                                                                    |
| **Transport Layer**   | **TCP** (Reliable, ordered, higher overhead)                  | Primarily **UDP** (Fast, unordered, lower latency via DTLS/SRTP)                                    |
| **Primary Data Type** | Text, JSON, and small binary data strings                     | Real-time Audio, Video, and raw byte streams                                                        |
| **Latency**           | Low latency (~100ms)                                          | Ultra-low latency (<50ms, optimized for live human interaction)                                     |
| **Data Integrity**    | **Guaranteed.** Packets are delivered in order. No data loss. | **Flexible.** Prioritizes speed over perfection (can drop video frames to preserve real-time flow). |
| **Setup Complexity**  | Low. Handled via standard web servers.                        | High. Requires infrastructure for NAT traversal (STUN/TURN) and signaling.                          |
| **Security**          | Optional encryption via TLS (`wss://`).                       | **Mandatory encryption** built into the protocol (DTLS/SRTP).                                       |

## Deep Dive into Key Differences

### 1. Connection Path (Where does the data go?)

- **WebSockets:** Every piece of data must pass through your central server. If User A wants to send a message to User B, it
  goes: `User A ➔ Server ➔ User B`.
- **WebRTC:** Once the connection is established, data flows directly between the users: `User A ➔ User B`. The server is
  entirely bypassed for data transmission, which drastically lowers server bandwidth costs for heavy files or video.

### 2. Protocol Performance (TCP vs. UDP)

- **WebSockets run on TCP.** TCP ensures that if a packet is lost in transit, everything pauses until that packet is
  retransmitted. This is perfect for text chat—you don’t want letters arriving out of order.
- **WebRTC primarily uses UDP.** If you are in a video call and a single frame drops, WebRTC doesn't pause the call to fetch
  it; it simply skips to the next frame. This keeps the stream flowing in true real-time.

### 3. The "Signaling" Catch

WebRTC cannot establish a connection on its own. Two browsers don't inherently know how to find each other over the internet
due to firewalls and routers (NAT). They need an intermediary to exchange connection details (IP addresses, device
capabilities, etc.). This process is called **Signaling**, and WebRTC does not provide a built-in method for it.

---

## How They Work Together (The Hybrid Approach)

Because WebRTC requires a middleman to set up its peer-to-peer connection, **WebSockets are frequently used as the signaling
channel for WebRTC.**

> **Example:** In an application like Discord or Zoom:
>
> 1. You open the app. A **WebSocket** connects you to the server to handle text chat, your friend list, and presence status
>    ("Online", "Idle").
> 2. You click "Call". The **WebSocket** transfers "signaling data" between you and your friend to coordinate the call setup.
> 3. Once coordinated, a **WebRTC** connection is established directly between your browser and your friend's browser,
>    handling the actual video and voice streams.
