## What is X.509?

**X.509** is an international standard defines the format for **public key certificates**. Managed by the International
Telecommunication Union (ITU), it is the global blueprint used to create the digital certificates that secure internet
traffic, encrypt emails, and verify device identities.

When you look at a website's security certificate (the "lock" icon in your browser), you are looking at an X.509 certificate.
It acts like a digital passport or driver's license for the internet: it binds a public cryptographic key to a specific
identity (like a domain name or organization) and is signed by a trusted authority to prove its validity.

---

## The Public Key Infrastructure (PKI) Ecosystem

An X.509 certificate doesn't work in a vacuum; it is the core component of a **Public Key Infrastructure (PKI)**. The PKI
model relies on trusted third parties to prevent spoofing.

- **The Subject:** The entity (website, server, person) requesting the certificate.
- **The Certificate Authority (CA):** A highly secure, trusted organization (like Let's Encrypt, DigiCert, or GlobalSign)
  that validates the identity of the Subject and issues the certificate.
- **The Validator (Client):** Your web browser or operating system. Browsers come pre-installed with a list of trusted Root
  CAs. When you visit a site, the browser checks the site's X.509 certificate against its trusted list to ensure it was
  signed by a legitimate CA.

---

## Anatomy of an X.509 Certificate

An X.509 certificate is written in a data format called **ASN.1** (Abstract Syntax Notation One) and contains specific
structural fields. The modern standard is **X.509 Version 3**, which includes the following standard fields:

| Field                       | Purpose                                                                                                                     | Example / Content                                           |
| --------------------------- | --------------------------------------------------------------------------------------------------------------------------- | ----------------------------------------------------------- |
| **Version**                 | Identifies which X.509 version applies.                                                                                     | Version 3 ($v3$)                                            |
| **Serial Number**           | A unique identifier assigned by the CA to distinguish it from other certificates.                                           | `0A:3B:4C...`                                               |
| **Signature Algorithm**     | The cryptographic algorithm the CA used to sign the certificate.                                                            | `sha256WithRSAEncryption`                                   |
| **Issuer**                  | The identity of the Certificate Authority that generated and signed it.                                                     | `CN=DigiCert TLS RSA SHA256 CA G1`                          |
| **Validity Period**         | The strict "Not Before" and "Not After" dates indicating when the certificate is active.                                    | Expired certificates are automatically blocked by browsers. |
| **Subject**                 | The identity of the entity being secured.                                                                                   | `CN=*.google.com` (Common Name)                             |
| **Subject Public Key Info** | The actual public cryptographic key belonging to the subject, along with its algorithm.                                     | RSA (2048-bit) or ECDSA                                     |
| **Extensions (v3 only)**    | Additional properties, such as **SAN (Subject Alternative Name)**, which allows one certificate to secure multiple domains. | `DNS:example.com`, `DNS:www.example.com`                    |
| **CA Signature**            | The actual digital signature generated by the CA's private key to seal the certificate against tampering.                   | Binary cryptographic hash                                   |

## Common X.509 File Extensions and Encodings

While X.509 dictates the internal structure of the data, the certificate can be saved and transmitted in a few different file
formats:

### 1. DER (.der, .cer)

- **Format:** Binary
- **Description:** The raw, binary ASN.1 encoded data. It is typically used by Java platforms and Windows systems but is not
  easily readable in a standard text editor.

### 2. PEM (.pem, .crt, .key)

- **Format:** ASCII Text (Base64)
- **Description:** The most common format used by Apache, Nginx, and Linux systems. It takes the raw binary DER data,
  translates it into readable Base64 text, and wraps it in distinct markers:

```text
-----BEGIN CERTIFICATE-----
MIIFMDCCBBigAwIBAgIQD7...[Base64 Encoded Data]...
-----END CERTIFICATE-----

```

### 3. PKCS#12 (.pfx, .p12)

- **Format:** Binary (Password Protected)
- **Description:** A wrapper format used to bundle a public X.509 certificate alongside its matching **private key** and
  intermediate chain into a single, password-encrypted file. This is highly utilized for importing/exporting certificates
  between servers.

## Where is X.509 Used?

X.509 is highly versatile and underpins security across various technological domains:

- **HTTPS/TLS:** Securing web traffic via web browsers.
- **Code Signing:** Used by developers to digitally sign software applications so operating systems (like Windows
  SmartScreen) know the file hasn't been modified by malware.
- **Secure Email (S/MIME):** Encrypting and digitally signing email communications to prove authorship and prevent
  eavesdropping.
- **SSH Client Certificates:** Authenticating users to enterprise servers without relying on standard passwords or raw SSH
  keys.
- **Enterprise Wi-Fi (802.1X):** Authenticating devices safely onto secure corporate networks.
