**gRPC** (which stands for **gRPC Remote Procedure Call**) is an open-source, high-performance framework developed by Google
in 2015. It is designed to connect services in a microservices architecture with ultra-low latency and high efficiency.

Instead of using traditional REST APIs where services talk to each other by hitting URLs (like `GET /api/users/1`), gRPC
allows a client application to **directly call a method on a server application on a totally different machine as if it were
a local function.**

---

## The Core Concept: Remote Procedure Calls (RPC)

In a traditional REST setup, you have to manually serialize data to JSON, send an HTTP request, receive JSON back, and
deserialize it.

With gRPC, the boundary between the network and your code disappears. If your server has a function called `getUser(id)`,
your frontend or another microservice can literally just call `server.getUser(id)` in its own codebase. The gRPC framework
handles all the messy network routing, serialization, and transmission behind the scenes.

---

## The Two Pillars of gRPC

gRPC achieves its extreme speed and efficiency by combining two distinct technologies: **Protocol Buffers** and **HTTP/2**.

### 1. Protocol Buffers (Protobuf)

Instead of using JSON or XML to format data, gRPC uses **Protocol Buffers**, which is Google’s open-source mechanism for
serializing structured data.

- **The `.proto` File:** You start by writing a strongly-typed contract defining your data structures and services in a file.
- **Code Generation:** You run this file through the Protobuf compiler, and it automatically generates native code classes
  for your chosen languages (Java, Python, Go, TypeScript, etc.).
- **Binary Serialization:** Unlike JSON, which is plain text and human-readable, Protobuf compresses data into a highly
  optimized **binary format**. This makes payload sizes drastically smaller and significantly faster to parse.

### 2. HTTP/2 Transport

gRPC uses **HTTP/2** as its underlying transport protocol. This gives gRPC several advanced capabilities that traditional
REST (usually running on HTTP/1.1) lacks:

- **Multiplexing:** Multiple requests and responses can be sent simultaneously over a single TCP connection without blocking
  each other.
- **Header Compression (HPACK):** Reduces network overhead by compressing repetitive metadata.
- **Bi-directional Streaming:** Client and server can send a continuous stream of messages to each other at the same time
  over a single connection.

---

## gRPC Streaming Modes

While REST is strictly a request-response model, gRPC supports four distinct types of communication:

1. **Unary (Simple RPC):** The client sends a single request and gets a single response back. (Behaves just like a
   traditional API call).
2. **Server Streaming:** The client sends one request, and the server returns a stream of multiple responses (e.g., streaming
   a live stock ticker feed).
3. **Client Streaming:** The client sends a stream of multiple messages, and the server waits until it receives all of them
   before returning a single response (e.g., uploading a large file in chunks).
4. **Bidirectional Streaming:** Both client and server send a stream of messages simultaneously in whatever order they want
   (e.g., a real-time chat or multi-player gaming connection).

## gRPC vs. REST

| Feature              | gRPC                                               | REST                                          |
| -------------------- | -------------------------------------------------- | --------------------------------------------- |
| **Data Format**      | Protocol Buffers (Binary, small)                   | JSON / XML (Text, larger)                     |
| **Contract**         | Strict (`.proto` file is mandatory)                | Loose (OpenAPI/Swagger is optional)           |
| **Protocol**         | HTTP/2 exclusively                                 | HTTP/1.1 or HTTP/2                            |
| **Streaming**        | Supports Client, Server, and Bi-directional        | Generally limited to Server-Sent Events (SSE) |
| **Browser Support**  | Poor (Requires a proxy like `grpc-web`)            | Native and excellent                          |
| **Primary Use Case** | Internal microservices, IoT, High-performance apps | Public APIs, web apps, frontend-to-backend    |

## Why Use gRPC? (And When Not To)

### Advantages:

- **Incredible Performance:** Binary payloads and HTTP/2 mean gRPC can be up to 7x to 10x faster than REST/JSON APIs.
- **Language Agnostic:** You can write your server in Go and your client in Java; the `.proto` file translates between them
  seamlessly.
- **Compile-Time Type Safety:** Because code is generated directly from your API contract, you can catch errors during
  compilation rather than at runtime.

### Disadvantages:

- **Not Browser Friendly:** Web browsers cannot natively manipulate HTTP/2 frames in the way gRPC requires. To use gRPC in a
  web browser, you must use a proxy middleware, which adds complexity.
- **Human Unreadable:** Because data is sent as binary, you cannot easily inspect payloads using standard network tools or
  tools like Postman without special configuration.
- **Steep Learning Curve:** Setting up code-generation pipelines and learning Protocol Buffers is more complex than throwing
  together a quick REST endpoint.
