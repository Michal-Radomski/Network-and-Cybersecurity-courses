In WebRTC, **Signaling** is the process of exchanging metadata between two devices to coordinate communication.

Ironically, even though WebRTC is a **peer-to-peer (P2P)** technology, devices _cannot connect to each other directly out of
nowhere_. They don't know each other's public IP addresses, they don't know what features the other's camera supports, and
they are usually blocked by firewalls.

To solve this, they use a **Signaling Server**—a mutually reachable third-party server (usually using WebSockets, HTTP, or
MQTT)—to act as a temporary mailbox to swap connection details. Once that exchange is done, the signaling server steps aside,
and the peers talk directly.

---

## What Information is Exchanged During Signaling?

To start a chat, the two browsers must swap three things:

1. **Session Control Messages:** Commands to open, configure, or close the call (e.g., "User A is calling you").
2. **Media Capabilities (SDP):** Text profiles describing what codecs, resolutions, and media types (audio/video) each device
   supports.
3. **Network Information (ICE Candidates):** The IP addresses and ports where each device can be reached.

---

## Step-by-Step: The Process of Beginning a WebRTC Chat

Imagine **Alice** wants to start a video chat with **Bob**. Here is exactly how the browser's JavaScript engine, the
signaling server, and network servers interact to make it happen.

### Phase 1: Local Media Preparation

1. **Accessing Media:** Alice’s browser requests permission to use her camera and microphone via
   `navigator.mediaDevices.getUserMedia()`.
2. **Creating the Connection:** Alice's browser creates a JavaScript object called an `RTCPeerConnection`.
3. **Attaching Tracks:** Alice attaches her audio and video tracks to this `RTCPeerConnection` object so it knows what data
   it will eventually send to Bob.

### Phase 2: The SDP Offer/Answer Handshake

4. **Generating the Offer:** Alice’s browser calls `peerConnection.createOffer()`. This generates an **SDP Offer** (a block
   of text stating: _"I want to do a video call using these specific video and audio codecs"_).
5. **Setting Local Description:** Alice stores this offer inside her own browser by calling
   `peerConnection.setLocalDescription(offer)`.
6. **Sending the Offer:** Alice sends this SDP text blob through the **Signaling Server** over to Bob.
7. **Bob Receives the Offer:** Bob’s browser receives Alice's SDP offer from the signaling server. He stores it as his target
   endpoint by calling `peerConnection.setRemoteDescription(offer)`.
8. **Bob Prepares Media:** Bob prepares his own camera and microphone streams and attaches them to his own
   `RTCPeerConnection`.
9. **Generating the Answer:** Bob’s browser calls `peerConnection.createAnswer()`. This generates an **SDP Answer** (stating:
   _"I accept the call, and I also support those codecs"_).
10. **Setting Descriptions:** Bob saves this as his `setLocalDescription(answer)` and sends it back through the Signaling
    Server to Alice.
11. **Alice Receives the Answer:** Alice receives Bob's answer and saves it via
    `peerConnection.setRemoteDescription(answer)`.

> **Status Check:** At this point, both browsers fully agree on the media types, codecs, and settings they will use. However,
> they still don't know how to physically route packets to each other.

### Phase 3: ICE Candidate Gathering & Exchange

This phase actually happens in parallel with Phase 2 via a process called **Trickle ICE**.

12. **Triggering ICE:** The moment `setLocalDescription` is called on either browser, the internal ICE framework kicks into
    gear.
13. **Gathering Candidates:** Each browser hits the local network interface and talks to a public **STUN/TURN server** to
    discover its public IP address and port configurations. Each discovery is packaged into a tiny snippet of data called an
    **ICE Candidate**.
14. **Exchanging Candidates:** Every time Alice’s browser discovers a new candidate, a JavaScript event (`onicecandidate`)
    fires. Alice immediately sends that candidate through the signaling server to Bob. Bob does the exact same thing, sending
    his candidates to Alice.
15. **Adding Candidates:** As Alice and Bob receive each other's candidates via the signaling server, they feed them into
    their local peer connection objects using `peerConnection.addIceCandidate(candidate)`.

### Phase 4: Direct Connection and Media Flow

16. **Connectivity Checks:** Both browsers now have a list of ways to reach the other. They start sending rapid, invisible
    UDP packets to each other's candidate addresses to test which path is open and fastest.
17. **Handshake Secure:** Once the best working network path is agreed upon, a secure connection handshake (DTLS/SRTP) takes
    place directly between Alice and Bob to encrypt the stream.
18. **The Chat Begins:** The P2P connection switches to "Connected." Alice and Bob's browsers receive an `ontrack` event
    containing the remote video/audio stream, which is instantly hooked up to an HTML `<video>` element on the screen.

The signaling server's job is officially done, and the real-time chat is live!
