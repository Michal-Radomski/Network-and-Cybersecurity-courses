HTTP headers are key-value pairs sent between a client (browser) and a server in every request and response. They carry
essential metadata about the request, the response, the system configuration, and the data payload itself.

To make them easier to understand, they can be categorized based on how they are used.

---

## 1. Request Headers (Sent by the Client)

These headers give the server context about who the client is, what they want, and what format they expect back.

- **`Host`**
- **Description:** Specifies the domain name of the server and the port number (if applicable).
- **Example:** `Host: www.example.com`
- **Why it matters:** This is mandatory in HTTP/1.1. It allows a single physical server with one IP address to host multiple
  different websites (virtual hosting).

- **`User-Agent`**
- **Description:** Contains a string that identifies the browser, operating system, and web engine being used by the client.
- **Example:** `User-Agent: Mozilla/5.0 (Windows NT 10.0; Win64; x64) ...`
- **Why it matters:** Servers use this to tailor content to specific devices (like serving a mobile-friendly site) or to
  block automated scraping bots.

- **`Accept` & `Accept-Encoding**`
- **Description:** `Accept` tells the server what content types (like HTML, JSON, or images) the browser can understand.
  `Accept-Encoding` tells the server which compression algorithms (like `gzip` or `br` for Brotli) the browser supports.
- **Example:** `Accept: text/html`, `Accept-Encoding: gzip, deflate, br`

- **`Authorization`**
- **Description:** Used to pass credentials (like API keys or tokens) to authenticate the client with the server.
- **Example:** `Authorization: Bearer y38a9f...`

- **`Referer`**
- **Description:** (Note the historical misspelling). It contains the absolute or partial address of the previous webpage
  from which a link to the currently requested page was followed.
- **Example:** `Referer: https://www.google.com/`

---

## 2. Response Headers (Sent by the Server)

These headers provide the client with information about the server and how to handle the incoming response.

- **`Server`**
- **Description:** Identifies the software used by the hosting server to handle the request.
- **Example:** `Server: nginx/1.25.4`
- **Why it matters:** While useful for diagnostics, security professionals often obscure or remove this header so attackers
  don't know exactly what software vulnerabilities to target.

- **`Location`**
- **Description:** Used during a redirection (like a `301 Moved Permanently` or `302 Found` status code) to tell the browser
  which URL it should automatically jump to next.
- **Example:** `Location: https://example.com/new-page`

- **`Set-Cookie`**
- **Description:** Used to send a piece of data (a cookie) from the server to the browser, which the browser will store and
  send back on future requests.
- **Example:** `Set-Cookie: session_id=abc123xyz; Secure; HttpOnly`
- **Why it matters:** This is how websites remember that you are logged in or keep track of items in a shopping cart.

---

## 3. Representation Headers (Describe the Payload)

These headers describe the actual body of the message (the HTML file, image, or video) being sent. They can appear in both
requests and responses.

- **`Content-Type`**
- **Description:** Tells the recipient what the media type (MIME type) of the data actually is, so it knows how to render it.
- **Example:** `Content-Type: text/html; charset=utf-8` or `Content-Type: application/json`
- **Why it matters:** Without this, a browser wouldn't know whether to display a file as text, run it as javascript, or
  render it as a JPEG image.

- **`Content-Length`**
- **Description:** Indicates the size of the message body sent to the recipient, measured in bytes.
- **Example:** `Content-Length: 3495`

- **`Content-Encoding`**
- **Description:** Lists what compression has been applied to the payload, matching what the client requested in
  `Accept-Encoding`.
- **Example:** `Content-Encoding: br`

---

## 4. Cache Control Headers (Performance Optimization)

Caching headers dictate how browsers and intermediate servers (like CDNs) should save and reuse files locally to speed up the
web.

- **`Cache-Control`**
- **Description:** The primary header for managing caching policy. It can contain multiple directives like `max-age` (how
  long a file is valid in seconds), `no-cache` (force revalidation), or `private` (only the user's browser can cache it).
- **Example:** `Cache-Control: public, max-age=31536000` (Cache for 1 year)

- **`ETag` (Entity Tag)**
- **Description:** A unique identifier string (often a hash) assigned by a server to a specific version of a resource.
- **Example:** `ETag: "33a64df551425fcc55e4d42a148795d9"`
- **Why it matters:** If a browser needs a file but notices it already has a saved copy with an ETag, it sends that tag back
  to the server. If the file hasn't changed, the server replies with a fast `304 Not Modified` status code, skipping the file
  download entirely.

---

## 5. Security Headers (Protecting the User)

Modern response headers used to instruct the browser to restrict certain behaviors and prevent cyber attacks.

- **`Strict-Transport-Security` (HSTS)**
- **Description:** Forces the browser to communicate with the website exclusively over secure HTTPS connections, even if the
  user types `http://`.
- **Example:** `Strict-Transport-Security: max-age=63072000; includeSubDomains`

- **`Content-Security-Policy` (CSP)**
- **Description:** Restricts where the browser is allowed to load scripts, styles, and images from, heavily mitigating
  Cross-Site Scripting (XSS) attacks.
- **Example:** `Content-Security-Policy: default-src 'self' https://trustedscripts.com`
