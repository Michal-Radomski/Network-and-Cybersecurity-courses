The WebSocket handshake is the bridge used to transition an active HTTP connection into a persistent, bidirectional WebSocket connection. It utilizes the standard HTTP/1.1 Upgrade mechanism to safely negotiate the protocol switch over a single TCP socket. [1, 2, 3, 4] 
## The Handshake Workflow

Client                                      Server

  |                                           |
  | ---- Standard TCP / TLS Handshake ------> |
  |                                           |
  | ---- HTTP GET (Upgrade: websocket) -----> |
  | <--- 101 Switching Protocols ------------ |
  |                                           |
  | <=== Full-Duplex WebSocket Protocol ====> |

## 1. The Client Request
The client (typically a browser) initiates the process by sending a specially structured HTTP GET request to the server. [4, 5] 

GET /chat HTTP/1.1
Host: server.example.com
Upgrade: websocket
Connection: Upgrade
Sec-WebSocket-Key: dGhlIHNhbXBsZSBub25jZQ==
Origin: http://example.com
Sec-WebSocket-Protocol: chat, superchat
Sec-WebSocket-Version: 13


* Upgrade: websocket: Tells the server the client wants to change the application protocol.
* Connection: Upgrade: Confirms that this is a network protocol transition.
* Sec-WebSocket-Key: A randomly generated, Base64-encoded 16-byte value used to prevent proxy caching errors and validate that the server explicitly supports WebSockets.
* Sec-WebSocket-Version: Specifies the protocol version (usually 13 as defined by [IETF RFC 6455](https://datatracker.ietf.org/doc/html/rfc6455)).
* Sec-WebSocket-Protocol: An optional list of preferred application-level sub-protocols (e.g., STOMP, MQTT). [1, 6, 7, 8, 9, 10] 

## 2. The Server Response
If the server supports the protocol and accepts the parameters, it replies with an HTTP 101 Switching Protocols status code. [6, 11] 

HTTP/1.1 101 Switching Protocols
Upgrade: websocket
Connection: Upgrade
Sec-WebSocket-Accept: s3pPLMBiTxaQ9kYGzzhZRbK+xOo=
Sec-WebSocket-Protocol: chat


* HTTP/1.1 101 Switching Protocols: Confirms the server is officially changing protocols.
* Sec-WebSocket-Accept: Prove validation of the client's key. The server takes the client's Sec-WebSocket-Key, concatenates it with a globally unique magic string (258EAFA5-E914-47DA-95CA-C5AB0DC85B11), computes the SHA-1 hash, and returns the Base64-encoded result.
* Sec-WebSocket-Protocol: Acknowledges the single, mutually agreed-upon sub-protocol. [8, 9, 11, 12, 13] 

## Connection State
Once the client verifies the Sec-WebSocket-Accept value, the handshake is officially complete. The HTTP layer is completely detached. The under-the-hood TCP socket stays open, immediately switching to the WebSocket binary framing protocol for low-overhead, full-duplex communication. [3, 6, 14, 15, 16]
