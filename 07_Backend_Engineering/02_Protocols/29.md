**WebRTC** (Web Real-Time Communication) is a free, open-source project and network framework that enables web browsers and
mobile applications to exchange live audio, video, and generic data directly with one another.

Its defining feature is **peer-to-peer (P2P) communication**. Unlike traditional web traffic that travels from a browser to a
central server and back, WebRTC allows data to cut out the middleman and stream directly from Browser A to Browser B. This
results in ultra-low latency—frequently under 200 milliseconds—making it the backbone of modern video conferencing tools,
cloud gaming, and live streaming platforms.

---

## The Three Core Components (APIs)

WebRTC is built into modern browsers and exposed to developers through three main JavaScript APIs:

### 1. `MediaStream` (getUserMedia)

This component handles the capture of audio and video from the user's device. It requests permission to access the device's
camera and microphone, handles echo cancellation, and packages the media into synchronized tracks that are ready to be
transmitted.

### 2. `RTCPeerConnection`

This is the heart of WebRTC. It handles the entire lifecycle of the peer-to-peer connection. It takes care of:

- Maintaining the connection stability.
- Managing bandwidth variations.
- Encrypting the media streams.
- Decoding incoming audio/video data in real-time.

### 3. `RTCDataChannel`

Beyond video and audio, WebRTC allows peers to send raw data directly to each other. This is highly customizable—developers
can choose between reliable delivery (like TCP, where every packet is guaranteed) or unreliable delivery (like UDP, where
speed is prioritized and dropped packets are ignored). This is widely used for multiplayer browser gaming, file sharing, and
real-time text chat.

## How WebRTC Establishes a Connection

Setting up a direct connection between two devices over the internet is incredibly difficult due to security firewalls and
routers (NATs) that mask a device's true public IP address. WebRTC relies on an external infrastructure to bridge this gap.

### 1. Signaling (The Introduction)

Before two peers can connect, they need to exchange information about their media capabilities (e.g., "I support 1080p
video") and connection protocols. This exchange is called **Signaling**. WebRTC does not define a signaling protocol;
developers typically use WebSockets, HTTP, or MQTT to pass these setup messages back and forth via a central server.

### 2. ICE and STUN Servers (Finding the Public IP)

Most devices sit behind a router using Network Address Translation (NAT), meaning they only know their local IP address (like
`192.168.1.5`). To find their public, internet-facing IP, WebRTC uses a **STUN (Session Traversal Utilities for NAT)**
server. The peer pings the STUN server, which simply replies, _"Here is the public IP address and port I see you coming
from."_ The peer can then share this address with its target peer during signaling.

### 3. TURN Servers (The Ultimate Fallback)

Sometimes, corporate firewalls or symmetric NATs are so restrictive that a direct P2P connection is physically impossible. In
these scenarios, WebRTC falls back to a **TURN (Traversal Using Relays around NAT)** server. A TURN server acts as a relay
middleware—both peers connect to it, and it forwards the media packets between them. While this technically breaks the
"peer-to-peer" rule and costs server bandwidth, it ensures the connection succeeds nearly 100% of the time.

## Why Use WebRTC?

### Advantages:

- **Ultra-Low Latency:** Because it runs primarily on UDP and cuts out server hops, communication is fast enough for
  real-time human interaction.
- **No Plugins Required:** It works natively out-of-the-box in Chrome, Safari, Firefox, and Edge without installing
  third-party software.
- **Mandatory Security:** WebRTC strictly enforces encryption. Media and data are encrypted using SRTP (Secure Real-time
  Transport Protocol) and DTLS (Datagram Transport Layer Security), meaning data cannot be intercepted mid-transit.
- **Adaptive Bitrate:** If a user's internet connection suddenly degrades, WebRTC automatically lowers the video resolution
  or frame rate to prevent the call from dropping entirely.

### Challenges:

- **Complex Infrastructure:** Managing STUN, TURN, and signaling servers requires significant backend orchestration.
- **Mesh Scalability Problems:** In a pure P2P setup (a "Mesh" network), if 5 people join a call, your browser has to upload
  its video stream 4 separate times (once to each person). This quickly overwhelms standard home upload speeds, meaning large
  group calls usually require advanced architecture media servers (like SFUs or MCUs) to aggregate the streams.
