The way threads work in Node.js is unique and often misunderstood. People frequently say "Node.js is single-threaded," which
is technically true for the JavaScript code you write, but fundamentally false for how Node.js operates under the hood.

Node.js actually uses a **hybrid thread architecture** composed of a single main thread and a background pool of worker
threads.

---

## 1. The Main Thread (The Event Loop)

When you start a Node.js application, it initializes exactly **one main thread**. This thread is responsible for executing
all of your JavaScript code, handling incoming HTTP requests, and orchestrating the application.

Inside this main thread runs the **Event Loop**. Instead of spawning a new thread for every incoming connection (like
traditional servers do), Node.js puts events (like a user clicking a button or a request hitting a server) into an event
queue and processes them sequentially on this single thread.

### The "Don't Block the Event Loop" Golden Rule

Because all your JavaScript runs on this single main thread, if you write a heavy synchronous calculation (like sorting a
massive array or calculating Fibonacci numbers), **the entire application freezes** for every other user until that
calculation is finished.

---

## 2. Behind the Scenes: Libuv and the Worker Pool

While your JavaScript is single-threaded, Node.js uses a C++ library called **libuv** to handle asynchronous operating system
tasks.

Libuv provides Node.js with a background **Worker Pool** (also known as the Thread Pool). By default, this pool has **4
threads** (though you can scale it up to 128 using the `UV_THREADPOOL_SIZE` environment variable).

When you call an asynchronous, non-blocking function in Node.js, the main thread offloads the heavy lifting to these
background libuv threads.

### What goes to the Thread Pool?

- **File System Operations (`fs`):** Reading or writing massive files to disk.
- **Cryptography (`crypto`):** Hashing passwords (like `bcrypt` or `pbkdf2`), generating random bytes, or handling SSL/TLS.
- **Compression (`zlib`):** Zipping or unzipping files.
- **DNS Lookups (`dns.lookup`):** Resolving domain names to IP addresses.

### What does NOT use the Thread Pool?

Network I/O (like fetching data from an external API or handling database queries) does _not_ use the libuv thread pool.
Instead, libuv interacts directly with the operating system kernel (using epoll on Linux, Kqueue on macOS, or IOCP on
Windows). The kernel handles the network wait, leaving Node's threads completely free.

---

## 3. Worker Threads (True CPU Multi-Threading)

Historically, if you had a heavy CPU-bound task (like video processing or machine learning data crunching), Node.js was a
poor choice. However, Node.js introduced the `worker_threads` module, which allows you to create **true parallel threads**
running JavaScript code.

Each Worker Thread operates in complete isolation with its own V8 JavaScript engine instance and its own Event Loop.

```javascript
const { Worker, isMainThread, parentPort } = require("worker_threads");

if (isMainThread) {
  // This is the Main Thread. Spin up a worker thread.
  const worker = new Worker(__filename);

  // Listen for the result from the worker thread
  worker.on("message", (result) => {
    console.log(`The heavy CPU math result is: ${result}`);
  });
} else {
  // This code runs inside the isolated Worker Thread!
  let heavyCalculation = 0;
  for (let i = 0; i < 1e9; i++) {
    heavyCalculation += i;
  }

  // Send the data back to the main thread
  parentPort.postMessage(heavyCalculation);
}
```

### How Workers Talk to the Main Thread

Because workers run in isolation, they don't share memory by default (avoiding standard multi-threaded bugs like race
conditions). They communicate by passing messages back and forth using an HTML5-style messaging API. If you explicitly _need_
shared memory for extreme performance, Node.js allows threads to share data structures using `SharedArrayBuffer`.
