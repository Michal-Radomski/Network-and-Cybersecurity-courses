**Memcached** is an open-source, high-performance, distributed memory-object caching system. It is designed to speed up
dynamic web applications by alleviating database load, storing data in RAM as simple **key-value pairs**.

Whenever an application needs data, it checks Memcached first. If the data is there (a cache hit), it's returned instantly.
If not (a cache miss), the app queries the slower relational database and saves the result back into Memcached for next time.

While both **Memcached** and **Redis** are popular, in-memory, key-value data stores used primarily for caching, they have
evolved into very different tools.

---

## The Core Differences

The simplest way to look at them is: **Memcached is a specialized, ultra-fast caching layer, while Redis is a Swiss Army
knife that can act as a cache, a database, and a message broker.**

### 1. Data Structures

- **Memcached:** Only supports simple **strings** as both keys and values. If you want to store an array or an object, your
  application must serialize it into a string (like a JSON string) before saving it.
- **Redis:** Supports a wide variety of advanced data structures, including Strings, Lists, Sets, Sorted Sets, Hashes,
  Bitmaps, and HyperLogLogs. This allows you to manipulate data directly inside memory (e.g., appending an item to a list
  without downloading the whole list first).

### 2. Architecture and Threading

- **Memcached:** It is **multi-threaded**. It utilizes multiple CPU cores efficiently right out of the box, making it scale
  vertically with ease when handling massive throughput of simple key-value reads/writes.
- **Redis:** Historically relies on a **single-threaded** event loop architecture for executing commands (though modern
  versions use background threads for specific tasks like deleting data). Because it is mostly single-threaded, a single
  long-running command can block other operations.

### 3. Persistence (Data Survival)

- **Memcached:** It is **strictly volatile**. It lives entirely in RAM. If the server reboots, crashes, or updates, 100% of
  your cached data is instantly wiped out.
- **Redis:** Offers **optional persistence**. It can periodically snapshot data to a disk (RDB) or log every write operation
  to a file (AOF). If a Redis server restarts, it can reload its state from disk, meaning it can technically function as a
  primary database.

### 4. Memory Management & Eviction

- **Memcached:** Uses an internal memory management system called **Slab Allocation**. It carves up memory into pre-allocated
  chunks of fixed sizes to eliminate memory fragmentation. When it runs out of space, it strictly uses a Least Recently Used
  (LRU) algorithm to evict old data.
- **Redis:** Relies on standard memory allocation, which can sometimes lead to fragmentation. However, it gives you highly
  granular control over eviction policies (e.g., Volatile-LRU, AllKeys-LFU, Random, or completely rejecting new writes when
  full).

## Summary Comparison Table

| Feature                  | Memcached             | Redis                                             |
| ------------------------ | --------------------- | ------------------------------------------------- |
| **Primary Use Case**     | Pure, simple caching. | Caching, complex data storage, message brokering. |
| **Data Types**           | Strings only.         | Strings, Lists, Sets, Hashes, Sorted Sets, etc.   |
| **Threading Model**      | Multi-threaded.       | Mostly Single-threaded.                           |
| **Persistence**          | No (RAM only).        | Yes (RDB snapshots / AOF logs).                   |
| **Pub/Sub Capabilities** | No.                   | Yes (Built-in messaging queues).                  |
| **Clustering/Scaling**   | Client-side sharding. | Native clustering and replication.                |

## When should you choose which?

### Choose Memcached if:

- You have a highly straightforward use case: you just need a standard cache layer to store HTML fragments, raw database
  queries, or serialized session data.
- You are dealing with massive scale and require heavy multi-threaded read/write performance.
- You want something dead-simple that requires very low operational overhead.

### Choose Redis if:

- You need complex capabilities, like sorting data, tracking user leaderboards, or managing chat rooms (via Pub/Sub).
- You need your data to survive server restarts (persistence).
- You need built-in geospatial queries or complex caching strategies.
