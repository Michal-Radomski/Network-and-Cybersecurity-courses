In Node.js, managing concurrency can be a bit confusing because the platform handles asynchronous tasks in a few different
ways.

Here is the breakdown of the **Main Thread**, **Thread Pool (Libuv)**, and **Worker Threads**, and how they differ in
purpose, control, and behavior.

---

## 1. The Main Thread

This is the heart of any Node.js application. Node.js is famously "single-threaded," and this is the specific thread that
phrase refers to.

- **What it does:** It executes your main JavaScript code, manages the **Event Loop**, handles incoming requests, and
  orchestrates callbacks.
- **Behavior:** It runs a single task at a time. If you write a heavy synchronous loop (like calculating Fibonacci numbers or
  parsing a massive JSON file), you will **block** this thread, causing the entire application to freeze for all users.
- **Memory:** It owns the primary memory heap of your application.

---

## 2. The Thread Pool (Libuv)

Because JavaScript is single-threaded, Node.js needs a way to handle heavy, blocking system-level tasks without freezing your
app. It does this using a C++ library called **Libuv**, which maintains a background thread pool.

- **What it does:** It handles specific, heavy asynchronous I/O and OS tasks that cannot be done efficiently by the OS kernel
  alone.
- **What runs here?** \* **File System (`fs`) operations** (e.g., `fs.readFile`)
- **Cryptography (`crypto`)** (e.g., `crypto.pbkdf2` password hashing)
- **Compression (`zlib`)**
- **DNS lookups** (`dns.lookup`)

- **Behavior:** You do not manage these threads manually. When you call an asynchronous function like `fs.readFile()`,
  Node.js offloads it to the Libuv thread pool. Once the thread finishes the task, it pushes the result back to the Main
  Thread's Event Loop.
- **Configuration:** By default, Libuv spins up **4 threads**, but you can increase this up to 1024 by changing the
  environment variable: `process.env.UV_THREADPOOL_SIZE = 8;`.

---

## 3. Worker Threads (`worker_threads` module)

Introduced to solve Node.js’s biggest weakness: **CPU-intensive JavaScript tasks.** Worker Threads allow you to run actual
parallel JavaScript code on separate threads.

- **What it does:** It allows you to explicitly create new threads to execute heavy JavaScript calculations (e.g., video
  processing, AI/ML data crunching, massive array manipulation) without blocking the Main Thread.
- **Behavior:** Unlike Libuv threads (which you don't control), you have full lifecycle control over Worker Threads. You
  spawn them, write the JS code they execute, and terminate them when done.
- **Memory:** Each Worker Thread runs its own isolated V8 instance and Event Loop. However, they can efficiently share memory
  using `SharedArrayBuffer` or pass messages via a built-in messaging channel (`postMessage`).

---

## Summary Comparison Table

| Feature                  | The Main Thread                                            | Thread Pool (Libuv)                                                 | Worker Threads                                                          |
| ------------------------ | ---------------------------------------------------------- | ------------------------------------------------------------------- | ----------------------------------------------------------------------- |
| **Who manages it?**      | Node.js runtime automatically.                             | Libuv (C++ background layer) automatically.                         | **You** (the developer) via the `worker_threads` module.                |
| **What code runs here?** | Your standard JavaScript application code.                 | Internal C++ code for specific OS/IO operations.                    | Isolated, heavy JavaScript code assigned by you.                        |
| **Primary Purpose**      | Handling network requests, Event Loop, light JS execution. | Preventing file I/O, crypto, and DNS from blocking the main thread. | Running CPU-bound JavaScript tasks in parallel.                         |
| **Default Number**       | Exactly **1**.                                             | Default is **4** (configurable).                                    | **0** (Only exists if you explicitly create them).                      |
| **Memory Sharing**       | N/A (Main heap).                                           | Managed internally by C++.                                          | Isolated memory, but can use `SharedArrayBuffer` for zero-copy sharing. |

> **A Quick Rule of Thumb:**
>
> - If you are handling **network I/O** (HTTP requests), the **Main Thread** handles it natively and efficiently.
> - If you are reading a **file from disk** or hashing a password, Node.js automatically uses the **Thread Pool**.
> - If you need to **calculate 1 million digits of Pi** in JavaScript, you should spin up a **Worker Thread**.
