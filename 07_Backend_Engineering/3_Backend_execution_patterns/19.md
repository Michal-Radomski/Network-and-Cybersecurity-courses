In backend development, **idempotency** is a critical design principle which ensures that making the same API request
multiple times will yield the exact same result as making it once, without causing unintended side effects.

Think of it like an elevator button: pressing it ten times doesn't make the elevator arrive ten times faster or take you to
ten different floors. It registers the request once and safely ignores the subsequent clicks.

---

## Why Idempotency Matters (The Real-World Problem)

In a perfect world, a client sends a request, the backend processes it, and the client gets a success message. But networks
are unreliable.

Imagine a user purchasing a $100 jacket on an e-commerce site:

1. The client sends a `POST /payments` request.
2. The backend successfully charges the user's credit card.
3. **The network drops** right before the backend can send the "Success" response back to the user.
4. The user sees a loading spinner freeze, assumes it failed, and clicks the "Submit Payment" button again.

Without idempotency, the user is charged **$200** for a single jacket. With idempotency, the backend recognizes it’s a
duplicate request, skips the charge step, and simply returns the successful receipt from the first execution.

---

## HTTP Methods and Idempotency

The HTTP specification explicitly defines which standard methods should be naturally idempotent:

| HTTP Method  | Idempotent? | Description                                                                                                                                                      |
| ------------ | ----------- | ---------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **`GET`**    | **Yes**     | Fetching data does not alter the state of the server. Reading a user profile 10 times changes nothing.                                                           |
| **`PUT`**    | **Yes**     | Replaces a resource entirely. Updating an email to `test@example.com` 5 times leaves the email as `test@example.com`.                                            |
| **`DELETE`** | **Yes**     | Deleting an item removes it. The first time it's deleted (`200 OK`). The next 4 times it's gone (`404 Not Found`), but the server's state didn't change further. |
| **`POST`**   | **No**      | Typically used to _create_ resources. Sending a "Create Comment" POST request 5 times will natively create 5 duplicate comments.                                 |

## How to Implement Idempotency (The Idempotency Key Pattern)

Since `POST` requests are not natively idempotent, backends implement a design pattern utilizing an **Idempotency Key**
(usually a unique UUID generated by the frontend).

```
Client                                     Backend & Cache (Redis)
  |                                                |
  | ─── POST /charge (Idempotency-Key: uuid-1) ──> |
  |                                                | 1. Checks Redis for 'uuid-1'
  |                                                |    (Not found -> Creates lock)
  |                                                | 2. Processes credit card charge
  |                                                | 3. Stores response in Redis
  | <── 返回 200 OK (Success) ────────────────────── |
  |                                                |
  | ─── [Network Retry] Same Request & Key ──────> |
  |                                                | 1. Checks Redis for 'uuid-1'
  |                                                |    (Found! Already processed)
  | <── 返回 200 OK (Cached Response) ────────────── | 2. Skips charging again!

```

### The Step-by-Step Mechanism:

1. **Frontend Generation:** Before making a critical request (like submitting a payment), the client generates a unique
   string, such as a UUID (e.g., `Idempotency-Key: 9b1deb4d-3b7d-4bad-9bdd-2b0d7b3dcb6d`), and attaches it as an HTTP header.
2. **Backend Check:** The backend intercepts the request and checks a fast in-memory store (like Redis) to see if this
   specific key has been processed recently.
3. **Handling Scenarios:**

- **Scenario A (Brand New Request):** The backend doesn't find the key. It marks the key as `IN_PROGRESS` in Redis, executes
  the heavy business logic (charging the card, saving to DB), saves the final response payload into Redis with a Time-To-Live
  (TTL, e.g., 24 hours), and returns the response to the client.
- **Scenario B (Duplicate Request):** The backend finds the key in Redis and sees it is already `SUCCESSFUL`. It completely
  skips your business logic, pulls the cached response out of Redis, and hands it right back to the client.
- **Scenario C (Concurrent Request):** If a duplicate request arrives while the first one is still processing
  (`IN_PROGRESS`), the backend rejects it with a `409 Conflict` or asks the client to retry in a moment, preventing race
  conditions.

## Key Considerations for Developers

- **Scope your keys carefully:** It's best practice to isolate idempotency keys by the user ID (e.g., Redis key:
  `idempotency:user_123:key_uuid`). This prevents a malicious or broken client from guessing or reusing someone else's UUID
  to break their system.
- **Database Constraints:** As a fallback defense layer, utilize unique SQL database constraints. For example, a unique
  composite index on `user_id` and `order_id` will physically block your database from accidentally generating twin rows even
  if your caching layer blips.
