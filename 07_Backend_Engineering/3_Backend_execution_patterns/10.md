When you boot up a Node.js application, it comes with a suite of default behaviors and resource limits configured out of the
box. These defaults are designed to keep the runtime lightweight, but they can be heavily customized to fit your
application's needs.

Here is a breakdown of the most critical default settings in Node.js and exactly how you can change them.

---

## 1. Memory Limits (V8 Heap Size)

Node.js runs on the Google V8 engine, which historically capped memory usage to prevent runaway scripts from crashing the
entire host machine.

- **The Default:** \* In modern 64-bit systems, the default maximum heap size is dynamically configured based on available
  system memory, often capping out around **4 GB** (or roughly 25-50% of physical RAM on smaller systems).
- **How to change it:** If you are processing massive datasets in memory or running into
  `FATAL ERROR: Ineffective mark-compacts near heap limit Allocation failed - JavaScript heap out of memory`, you can
  explicitly set the limit using the `--max-old-space-size` flag (measured in megabytes).

```bash
# Increase the limit to 8 GB
node --max-old-space-size=8192 app.js

```

---

## 2. The Libuv Thread Pool Size

As mentioned previously, Node.js offloads heavy asynchronous OS/system tasks (like file I/O and crypto) to a background C++
thread pool.

- **The Default:** **4 threads**.
- **How to change it:** If your application handles massive amounts of file streaming, image editing, or bcrypt password
  hashing simultaneously, those 4 threads will bottleneck. You can increase this limit up to **1024** by setting an
  environment variable _before_ the Node.js runtime initializes.

```bash
# On Linux/macOS
export UV_THREADPOOL_SIZE=8
node app.js

# On Windows (cmd)
set UV_THREADPOOL_SIZE=8 && node app.js

```

---

## 3. Network & HTTP Defaults

Node.js includes built-in configurations for how it handles network requests and connections.

- **The Defaults:**
- `keepAlive`: **`false`** by default for the global HTTP agent. Node closes connections after a request finishes, rather
  than reusing them.
- `maxSockets`: **`Infinity`** by default. Node will spawn as many concurrent sockets to a single origin as the OS allows.

- **How to change it:** For high-performance microservices, you should explicitly configure a custom HTTP Agent to reuse TCP
  connections, which drastically reduces latency.

```javascript
const http = require("http");

const customAgent = new http.Agent({
  keepAlive: true,
  maxSockets: 100, // Caps simultaneous connections to a single host
  keepAliveMsecs: 1000, // Keeps idle sockets open for 1 second
});

// Pass the agent into your fetch or request options
```

---

## 4. Garbage Collection Behavior

V8 manages memory automatically, but its default "stop-the-world" garbage collection pauses can sometimes introduce latency
spikes.

- **The Default:** Node decides when to run GC cycles automatically based on memory pressure.
- **How to change it:** You can expose the garbage collector to your JavaScript code for debugging purposes, or tweak how
  aggressively it triggers.

```bash
# Exposes a global gc() function you can call manually in JS
node --expose-gc app.js

# Changes the interval of the garbage collector scavenging
node --min-semi-space-size=16 --max-semi-space-size=64 app.js

```

---

## 5. Module Resolution System (CommonJS vs. ESM)

Node.js historically relied entirely on CommonJS (`require`), but fully supports ECMAScript Modules (`import/export`).

- **The Default:** CommonJS (`.js` files are treated as CommonJS unless specified otherwise).
- **How to change it:** You can change the default behavior for your entire project by adding a property to your
  `package.json`:

```json
{
  "type": "module"
}
```

_(Alternatively, using the `.mjs` extension forces ESM, while `.cjs` forces CommonJS)._
