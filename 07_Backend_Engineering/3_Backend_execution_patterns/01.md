In computing, **Processes** and **Threads** are the foundational units of execution that a CPU manages to run software.

The easiest way to understand the difference is to think of a **Process** as a factory, and a **Thread** as a worker inside
that factory.

---

## 1. What is a Process?

A **Process** is an independent, executing program that has been loaded into the computer's memory. When you double-click an
application icon (like Microsoft Word or Google Chrome), the operating system creates a new process for it.

Every process is completely isolated from other processes. It is given its own private sandbox of resources by the operating
system, including:

- **Memory Space:** Its own dedicated virtual address space (where its code and data live).
- **File Descriptors:** Open network connections, files, and hardware handles.
- **Security Context:** Specific user permissions.

Because they are isolated, if one process crashes (e.g., a buggy video editing software), it won't instantly crash your web
browser or your operating system.

---

## 2. What is a Thread?

A **Thread** (often called a _lightweight process_) is the smallest unit of execution _within_ a process. A process cannot
execute code on its own; it must have at least one thread (called the **Main Thread**).

A single process can spawn multiple threads to perform different tasks concurrently. Crucially, **all threads inside a
process share that process’s memory and resources.**

For example, inside a Microsoft Word process:

- **Thread 1** is listening to your keyboard inputs and rendering text on the screen.
- **Thread 2** is running in the background checking your spelling and grammar.
- **Thread 3** is automatically saving your document to the hard drive every few minutes.

---

## Process vs. Thread: Key Differences

| Feature                | Process                                                                          | Thread                                                                                     |
| ---------------------- | -------------------------------------------------------------------------------- | ------------------------------------------------------------------------------------------ |
| **Definition**         | An executing instance of a program.                                              | A path of execution within a process.                                                      |
| **Memory**             | Has its own independent memory space.                                            | Shares memory space with the parent process and sibling threads.                           |
| **Creation Cost**      | **Heavyweight:** Takes significant time and CPU to create and terminate.         | **Lightweight:** Fast to create and destroy.                                               |
| **Communication**      | Difficult. Requires **IPC** (Inter-Process Communication) like sockets or pipes. | Easy. Threads can directly read/write to the shared memory variables.                      |
| **Isolation / Safety** | Highly Isolated. If one process crashes, others keep running safely.             | Low Isolation. If one thread encounters a fatal error or crashes, the entire process dies. |
| **Context Switching**  | Slow. The CPU must swap out the entire memory map and register states.           | Fast. The CPU only needs to swap registers and stack pointers.                             |

---

## Memory Architecture Breakdown

To look at it from a technical perspective, memory allocation is where they diverge completely:

```
+-------------------------------------------------------------+
|                      PROCESS MEMORY                         |
|                                                             |
|  [ Text / Code ]     [ Global Data ]      [ Heap Memory ]   |
|                                                             |
|   +--------------------+       +--------------------+       |
|   |      THREAD 1      |       |      THREAD 2      |       |
|   |  Registers / Stack |       |  Registers / Stack |       |
|   +--------------------+       +--------------------+       |
+-------------------------------------------------------------+

```

- **Shared Resources:** The **Code**, **Global Data**, and **Heap** (where dynamically allocated variables live) are shared
  by _all_ threads in that process.
- **Private Resources:** Each thread only gets its own **Stack** (to keep track of local variables and function calls) and
  **Registers** (to track what instruction the CPU is currently executing).

---

## The Core Trade-Off: Concurrency vs. Safety

Developers have to choose between multi-processing and multi-threading depending on what they are building:

- **Multi-threading** is highly efficient because threads can talk to each other instantly through shared memory. However, it
  introduces synchronization bugs like **Race Conditions** and **Deadlocks**, where two workers try to modify the exact same
  piece of data at the same millisecond, causing corruption.
- **Multi-processing** is incredibly stable. Google Chrome famously uses a multi-process architecture where every single
  browser tab is its own independent process. If a heavy website crashes Tab A, Tab B and the rest of the browser remain
  completely unaffected. The downside is that Chrome consumes a massive amount of RAM because each tab has to duplicate core
  browser memory structures.
