To see how software utilizes a computer's CPU and memory, we can look at how programs are structured. Building on the
concepts of processes (the isolated application "factories") and threads (the "workers"), operating systems execute code
using three main architectures: **Single-Threaded Processes**, **Multi-Processes**, and **Multi-Threaded Processes**.

---

## 1. Single-Threaded Process

A **Single-Threaded Process** is the simplest execution model. It consists of exactly one process containing exactly **one
thread** of execution (the main thread).

- **How it works:** Tasks are executed strictly one after the other in a linear sequence (synchronously). If Task A takes 5
  seconds, Task B cannot start until those 5 seconds are up.
- **The Analogy:** A small coffee shop with one worker. This worker takes the order, grinds the beans, steams the milk, and
  rings up the customer. If a customer orders a complex drink, the entire line must wait.
- **Pros:** \* Incredibly simple to write and debug.
- No risk of concurrency bugs (like two threads fighting over the same data).

- **Cons:** \* Cannot utilize modern multi-core processors; it runs on just a single CPU core.
- **Blocking:** If a task takes a long time (like loading a massive file from disk), the entire application freezes and
  becomes unresponsive to the user.

- **Famous Example:** **Node.js** and browser-side **JavaScript** are natively single-threaded (though they use asynchronous
  event loops to dodge the freezing issue).

---

## 2. Multi-Threaded Process

A **Multi-Threaded Process** consists of a single process that spawns **multiple threads** that execute tasks concurrently.
Crucially, all of these workers share the exact same memory space (heap, data, and code) allocated to that process.

- **How it works:** The operating system schedules different threads across different cores of the CPU. Because they share
  memory, they can read and write to the same variables instantly to cooperate on a job.
- **The Analogy:** A single coffee shop kitchen, but with **four workers** sharing the same espresso machine, milk jugs, and
  cash register. One takes orders, one steams milk, one pulls espresso shots, and one bags pastries.
- **Pros:**
- Highly efficient usage of CPU and RAM.
- Fast communication between tasks because they share memory.
- Highly responsive UI (one thread handles user clicks while background threads do the heavy lifting).

- **Cons:**
- **Concurrency Issues:** If two threads try to modify the exact same variable at the exact same millisecond, the data gets
  corrupted (**Race Conditions**).
- **Shared Fate:** If one thread encounters a fatal error and crashes, it corrupts the shared memory space, and the _entire
  process_ (the whole app) crashes.

- **Famous Example:** **Java** applications, **video games** (where physics, audio, and rendering all run on separate
  threads), and traditional web servers like **Apache**.

---

## 3. Multi-Processes

A **Multi-Process** architecture involves splitting an application's workload across **multiple, entirely independent
processes**. Each process has its own completely isolated memory sandbox and its own thread(s).

- **How it works:** Because their memory spaces are completely locked away from one another, they cannot talk to each other
  directly. If they need to share data, they must use specialized operating system channels called **IPC (Inter-Process
  Communication)**, such as network sockets, pipes, or shared files.
- **The Analogy:** A coffee company that opens **three separate storefronts** down the street from one another. If Storefront
  A accidentally floods or catches fire, Storefront B and Storefront C keep serving coffee completely uninterrupted. However,
  if Storefront A runs out of milk, they can't just reach over to Storefront B's fridge; they have to coordinate a formal
  delivery truck (IPC) to move the milk over.
- **Pros:**
- **Fault Isolation:** Absolute stability. If one process crashes, it has zero impact on the survival of the other processes.
- **Security:** One process cannot accidentally or maliciously read or modify the private memory of another process.
- Easily scales across multiple physical CPU sockets.

- **Cons:**
- **Resource Heavy:** Duplicating memory spaces for multiple processes consumes a massive amount of RAM.
- High overhead to create, destroy, or switch between processes.

- **Famous Example:** **Google Chrome** (every browser tab and extension is an isolated process) and the **PostgreSQL**
  database.

---

## Summary Comparison

| Metric               | Single-Threaded             | Multi-Threaded                     | Multi-Process                        |
| -------------------- | --------------------------- | ---------------------------------- | ------------------------------------ |
| **Memory Isolation** | N/A (Only one thing exists) | **Low:** Shared among all threads  | **High:** Totally isolated sandboxes |
| **RAM Usage**        | Lowest                      | Low / Efficient                    | Highest (Heavy duplication)          |
| **Crash Impact**     | Whole app dies              | One bad thread kills the whole app | One bad process dies; others survive |
| **Complexity**       | Very Low                    | High (Requires locks/mutexes)      | Medium (Requires IPC protocols)      |
| **CPU Core Usage**   | Maxes out at 1 Core         | Spreads across all available Cores | Spreads across all available Cores   |
