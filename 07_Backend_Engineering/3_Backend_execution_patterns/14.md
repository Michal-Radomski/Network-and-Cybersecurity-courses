Once the backend uses the `accept()` system call to grab a connection from the queue, it receives a dedicated **File
Descriptor (FD)** for that specific client. Now, the actual conversation begins.

Reading and sending data over a socket isn't like reading a static file from a hard drive. Because networks are
unpredictable, data arrives in unpredictable chunks.

Here is exactly how a backend handles reading from and writing to a socket at the low level.

---

## 1. How the Backend Reads Socket Data

When a client sends data (like an HTTP POST request or a chat message), the data travels across the internet as a series of
TCP packets.

### Step A: The Kernel Buffer

Your application doesn’t read directly from the network wire. When packets arrive at the server’s network interface card
(NIC), the OS Kernel processes them, strips away the TCP headers, reorders any packets that arrived out of order, and places
the raw bytes into an OS-managed **Kernel Receive Buffer** dedicated to that specific File Descriptor.

### Step B: The `read()` or `recv()` System Call

Your backend application issues a system call (like `read(fd, buffer, length)`) to fetch those bytes.

- **The App Buffer:** Your application allocates a small chunk of its own memory (a user-space buffer) to hold the incoming
  data.
- **The Handoff:** The OS copies the bytes from the _Kernel Receive Buffer_ into your _Application Buffer_.

### The Trick: Dealing with Streams

TCP is a **stream protocol**, not a message protocol. It has no idea what an "HTTP Request" or a "JSON object" is; it just
sees a continuous stream of incoming bytes.

If a client sends a 5MB image, your application's `read()` call might only grab 64KB at a time. Your backend code must run a
loop, continually reading chunks, and stitching them together in memory until it detects that the message is complete (e.g.,
finding the `\r\n\r\n` delimiter that marks the end of HTTP headers, or matching Content-Length).

---

## 2. How the Backend Sends Socket Data

Sending data follows the exact reverse path, but it introduces a unique challenge: your backend can actually output data
faster than the client's network can accept it.

### Step A: The `write()` or `send()` System Call

When your code wants to send a response (like an HTML file or a JSON payload), it executes a system call like
`write(fd, data)`.

### Step B: The Kernel Send Buffer

Just like receiving, the OS copies your data out of the application's memory and drops it into a **Kernel Send Buffer**. The
OS Kernel then takes over, breaks that data down into TCP packets, and streams it out across the physical network card.

As soon as the data is copied into the Kernel Send Buffer, your application code is told _"Success!"_—even if those packets
haven't actually left the machine yet.

---

## 3. The Big Bottleneck: Network Backpressure

What happens if your backend is trying to send a massive 100MB video file to a mobile phone user on a weak 3G connection?

1. Your backend frantically pumps data into the **Kernel Send Buffer** via `write()` calls.
2. The phone can only download data at 1MB per second.
3. The **Kernel Send Buffer** fills up completely.

Once that buffer is full, the OS kernel will block further `write()` calls from your application. If your backend is
single-threaded (like Node.js), blocking that thread means your entire server freezes.

### How Modern Backends Handle This (Non-Blocking I/O)

To prevent freezing, modern backends configure sockets to be **Non-Blocking**.

If the Kernel Send Buffer is full and your app tries to write data, the OS doesn't block. Instead, it immediately returns a
specific error code: `EWOULDBLOCK` or `EAGAIN`. This tells your application: _"I'm full right now. Hold onto that data in
your own memory, and I'll let you know when I have space."_

In Node.js, this is handled automatically via **Streams**. When you call `response.write(chunk)` and it returns `false`,
Node.js pauses reading from the source. Once the OS clears out the network buffer, it fires a `'drain'` event, telling
Node.js it's safe to start sending chunks again. This balancing act is called managing **backpressure**.
