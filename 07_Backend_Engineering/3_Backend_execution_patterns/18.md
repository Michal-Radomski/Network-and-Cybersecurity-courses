At its most fundamental level, a **socket** is an endpoint for communication between two machines over a network.

If you think of the internet or a network as a massive postal system, an IP address is the street address of a building, and
a **socket is the specific mailbox or apartment number** within that building where a specific conversation happens.

---

## 1. The Anatomy of a Socket

A network socket is defined by a unique combination of two things:

1. **An IP Address:** Identifies _which machine_ on the network to talk to (e.g., `192.168.1.5`).
2. **A Port Number:** Identifies _which specific application_ on that machine should receive the data (e.g., port `80` for
   web traffic, port `22` for SSH).

When your web browser connects to a server, a **Socket Pair** is created to form a complete, bidirectional communication
channel:

- **Client Socket:** `192.168.1.50 : 54321` (Your computer's IP + a temporary port)
- **Server Socket:** `93.184.216.34 : 443` (The website's IP + standard HTTPS port)

Because this pairing is completely unique, the operating system knows exactly which incoming data packets belong to your
browser tab versus a background app.

---

## 2. How Sockets Work (The Software Layer)

To an operating system kernel, a socket is treated as a form of **File Descriptor (FD)**. This means once a socket connection
is established, an application reads from it and writes to it just like it would read or write to a text file on a hard
drive.

- **Writing to a socket:** Your application pushes data into the socket. The OS kernel takes that data, wraps it in network
  protocols (like TCP/IP), breaks it into packets, and sends it over the physical network wire.
- **Reading from a socket:** When network packets arrive at a machine, the OS strips away the network headers, drops the raw
  data bytes into the socket's internal buffer, and alerts the application that data is ready to be read.

---

## 3. The Two Main Types of Sockets

Depending on how data needs to be transferred, sockets generally use one of two primary transport protocols:

### A. Stream Sockets (TCP)

- **How they work:** They establish a dedicated, reliable, connection-oriented channel via a "three-way handshake" before any
  data is sent.
- **Characteristics:** Data is guaranteed to arrive in the exact order it was sent, and if a packet is lost in transit, the
  socket automatically requests a retransmission.
- **Use Cases:** Web browsing (HTTP/HTTPS), email (SMTP), file transfers (FTP), and database connections.

### B. Datagram Sockets (UDP)

- **How they work:** They are entirely connectionless. An application fires packets (datagrams) at an IP and port and
  completely forgets about them.
- **Characteristics:** There is no guarantee the data will arrive, nor that it will arrive in order. However, because it
  skips the overhead of handshakes and error-checking, it is incredibly fast.
- **Use Cases:** Live video streaming, online video games, and VoIP (voice calls) where speed is more important than perfect
  accuracy.

## Summary Metaphor

> Imagine a socket like a **landline telephone connection**:
>
> 1. The phone number is the **IP Address**.
> 2. The extension number to reach a specific desk is the **Port**.
> 3. The physical telephone handset you speak into and listen out of is the **Socket**.
