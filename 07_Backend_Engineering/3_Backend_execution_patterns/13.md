At its core, a **File Descriptor (FD)** is just a simple, non-negative integer (like `0`, `1`, `3`, `12`) that an Operating
System kernel uses to identify and track an open resource.

To the OS, **"everything is a file."** This means a file descriptor isn't just used for files on your hard drive; it is the
handle used to read from and write to network sockets, pipes, database connections, and even your keyboard and screen.

Think of it like a **claim ticket** at a coat check: you give the attendant your jacket (the resource), they give you a
number (the file descriptor), and whenever you want to do something to that jacket, you just show them the number.

---

## How It Works Under the Hood

When a process (like a Node.js app or a Python script) wants to open a file or create a network socket, it makes a system
call to the OS Kernel.

```
[ Your Application ]
       │
       │  1. "Open 'data.txt'" (System Call)
       ▼
[ OS Kernel Layer ] ── 2. Allocates file in memory
       │              3. Creates pointer in File Descriptor Table
       ▼
[ Your Application ]
       │
       ◀─ 4. Returns Integer (e.g., File Descriptor 3)

```

1. **The Kernel** opens the resource and allocates memory for it.
2. **The Kernel** maintains a private **File Descriptor Table** for your specific process. It drops a pointer to that
   resource into the table.
3. **The Index** of that pointer in the table becomes your File Descriptor.
4. **The Kernel** returns just that integer back to your application.

From that point on, your app doesn't say _"Write 'Hello' to `/users/documents/data.txt`"_. It tells the OS: **"Write 'Hello'
to FD 3."**

---

## The Default File Descriptors (0, 1, 2)

Every single time a new process starts up on a Unix-like system (Linux, macOS), the OS automatically opens three file
descriptors for it by default:

| FD Integer | Standard Name              | What it represents                         | Default Destination             |
| ---------- | -------------------------- | ------------------------------------------ | ------------------------------- |
| **`0`**    | `stdin` (Standard Input)   | Where the app reads input data from.       | Your keyboard / terminal input. |
| **`1`**    | `stdout` (Standard Output) | Where the app writes its normal data/logs. | Your terminal screen.           |
| **`2`**    | `stderr` (Standard Error)  | Where the app writes error messages.       | Your terminal screen.           |

When you run `console.log()` in Node.js or `print()` in Python, under the hood, the runtime is simply executing a system
write command directly to **FD 1**.

## Why File Descriptors Matter in Backend Development

If you are building high-performance backends, you will eventually encounter file descriptors, usually when things go wrong.

### 1. The "Too many open files" Error

Operating systems put a strict limit on how many file descriptors a single process can have open at once to prevent memory
exhaustion (often a default of 1024 or 4096 on standard Linux setups).

If your web server gets 5,000 simultaneous users, and each user requires a network socket (which consumes 1 FD), your
application will crash with an **`EMFILE: too many open files`** error.

### 2. Resource Leaks

If you write code that opens a file or a database connection inside a loop but forget to close it (e.g., forgetting to call
`file.close()` or `socket.end()`), that integer stays active in the process's File Descriptor Table. This is called a **File
Descriptor Leak**. Eventually, the table fills up, and your application won't even be able to accept new incoming network
requests.

### 3. I/O Multiplexing (How Node.js handles thousands of connections)

Systems like Node.js use OS mechanisms like `epoll` (Linux) or `kqueue` (macOS). Instead of constantly checking every single
network connection to see if data has arrived, Node.js hands a list of thousands of **File Descriptor integers** to the OS
kernel and says: _"Wake me up only when one of these specific FDs receives data."_ This is what makes modern asynchronous I/O
so incredibly fast.
