To understand WebSocket proxying at Layer 4 (L4) and Layer 7 (L7), it helps to look at where these layers sit in the OSI
model. A WebSocket connection starts as a standard HTTP request (Layer 7) and then "upgrades" to a long-lived, bidirectional
TCP connection (Layer 4).

Because of this hybrid nature, proxying WebSockets can be handled differently depending on whether the proxy operates at the
transport layer or the application layer.

---

## Layer 4 (Transport Layer) WebSocket Proxying

Layer 4 proxying operates at the **TCP/UDP level**. The proxy does not read, understand, or modify the actual HTTP headers or
the WebSocket frames passing through it. It simply routes raw TCP packets based on IP addresses and port numbers.

### How it Works

1. The client opens a TCP connection to the L4 proxy.
2. The proxy opens a corresponding TCP connection to the backend server.
3. The proxy stitches these two TCP streams together.
4. The WebSocket handshake (HTTP Upgrade) and all subsequent data frames pass through the proxy completely untouched.

### Pros & Cons

- **🚀 High Performance:** Because the proxy doesn't spend CPU cycles parsing application data (like HTTP headers or
  WebSocket frames), it is incredibly fast and consumes very little memory.
- **🔒 Seamless Encryption (Passthrough):** If using TLS (WSS), the proxy can pass the encrypted traffic straight to the
  backend without needing to manage SSL/TLS certificates.
- **❌ No Application-Aware Routing:** You cannot route traffic based on HTTP paths (e.g., routing `/chat` to one server and
  `/live-feed` to another) or cookies.
- **❌ Limited Security Insights:** The proxy cannot inspect the traffic for malicious WebSocket payloads or handle
  authentication at the gateway.

---

## Layer 7 (Application Layer) WebSocket Proxying

Layer 7 proxying operates at the **Application level** (HTTP/HTTPS). The proxy fully understands the HTTP protocol,
recognizes the `Upgrade: websocket` header, and actively participates in the WebSocket protocol.

### How it Works

1. The client terminates its TCP and TLS connection _at the proxy_.
2. The proxy inspects the incoming HTTP request. It looks for the `Connection: Upgrade` and `Upgrade: websocket` headers.
3. If allowed, the proxy evaluates its routing rules (based on URL paths, cookies, or headers) and opens a _new_ connection
   to the appropriate backend server, forwarding the upgrade request.
4. Once the handshake is complete, the proxy actively translates and forwards WebSocket data frames back and forth.

### Pros & Cons

- **🧠 Intelligent Routing:** You can route connections based on the URL path, hostnames, cookies, or user authentication
  headers.
- **🛡️ Advanced Security:** The proxy can perform SSL/TLS termination, inspect WebSocket frames for malicious content,
  enforce rate limiting, and handle authentication before the traffic ever hits your backend.
- **🔄 Protocol Buffering & Offloading:** The proxy can handle the heavy lifting of managing thousands of idle client
  connections, shielding the backend servers.
- **❌ Higher Resource Consumption:** Parsing HTTP headers and managing WebSocket frames requires significantly more CPU and
  memory than simple packet forwarding.

---

## Key Differences Summary

| Feature                 | Layer 4 Proxying                                                   | Layer 7 Proxying                                                  |
| ----------------------- | ------------------------------------------------------------------ | ----------------------------------------------------------------- |
| **Primary Focus**       | Packet routing (IP & Port)                                         | Content routing (URLs, Headers, Cookies)                          |
| **Protocol Awareness**  | Only TCP                                                           | HTTP and WebSocket protocols                                      |
| **SSL/TLS Termination** | Optional (Can pass encrypted traffic directly)                     | Required if you want to inspect or route traffic                  |
| **Performance**         | Extremely high, low latency                                        | Moderate (higher CPU/Memory usage)                                |
| **Use Case Example**    | High-throughput, simple load balancing (e.g., HAProxy in TCP mode) | API Gateways, complex microservices (e.g., NGINX, Envoy, Traefik) |

---

## Which one should you choose?

- **Choose Layer 4** if you need maximum throughput, have a single monolithic backend or a simple cluster that handles all
  WebSocket traffic uniformly, and want to offload SSL termination directly to your application servers.
- **Choose Layer 7** if you are running a microservices architecture (where different WebSocket endpoints live on different
  services), need a centralized place to handle security/authentication, or want to inject or inspect HTTP headers during the
  initial handshake.
