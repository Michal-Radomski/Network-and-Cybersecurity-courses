Using `EXPLAIN SELECT COUNT(*)`—or more accurately, utilizing the `EXPLAIN` command to extract row estimates—is a fantastic
way to bypass the performance destruction of a real count.

However, you don't actually run `EXPLAIN SELECT COUNT(*)`. Instead, you use `EXPLAIN` on your **base query** to read the
database's internal statistics.

Here is exactly how it works, how to implement it in your Node.js backend, and the major catch you need to look out for.

---

## How it Works Under the Hood

When you prepend `EXPLAIN` to a query, the database engine doesn't actually execute the query or look at the live rows.
Instead, it consults its **Query Planner**. The planner looks at internal statistics tables (which the database updates
periodically in the background) to guess how many rows will be returned.

If you run this in PostgreSQL:

```sql
EXPLAIN SELECT * FROM orders WHERE status = 'completed';

```

The database will instantly return a text plan that looks something like this:

```text
Index Scan using orders_status_idx on orders  (cost=0.42..8.44 rows=42512 width=68)

```

Notice the **`rows=42512`**. That is the database's cached estimate. It took 0.1 milliseconds to get that number, whereas a
real `COUNT(*)` might have taken 5 seconds of heavy disk reading.

---

## How to Implement This in Node.js

Since `EXPLAIN` returns a text execution plan or a JSON structure (depending on your database), your Node.js backend has to
parse that output to extract the number.

### 1. PostgreSQL Implementation (using `pg`)

PostgreSQL allows you to request the explain plan format as JSON, making it incredibly easy to parse in JavaScript.

```javascript
const { Pool } = require("pg");
const pool = new Pool();

async function getEstimatedCount(userId) {
  // We use EXPLAIN (FORMAT JSON) on the target query
  const query = `
    EXPLAIN (FORMAT JSON) 
    SELECT * FROM orders 
    WHERE user_id = $1 AND status = 'completed'
  `;

  try {
    const res = await pool.query(query, [userId]);

    // Postgres returns a nested JSON structure
    // The estimate is located inside the root plan object under 'Plan Rows'
    const estimatedRows = res.rows[0]["QUERY PLAN"][0]["Plan"]["Plan Rows"];

    return estimatedRows; // e.g., 42512
  } catch (err) {
    console.error("Failed to fetch estimate", err);
    return 0;
  }
}
```

### 2. MySQL Implementation (using `mysql2`)

MySQL provides an `EXPLAIN` syntax that outputs a clean table structure.

```javascript
const mysql = require("mysql2/promise");
const connection = await mysql.createConnection(/* config */);

async function getMySQLQueryEstimate(userId) {
  const query = `
    EXPLAIN SELECT id FROM orders 
    WHERE user_id = ? AND status = 'completed'
  `;

  try {
    const [rows] = await connection.execute(query, [userId]);

    // MySQL returns a row for each step. We sum the 'rows' column.
    const estimate = rows.reduce((acc, stage) => acc + (stage.rows || 0), 0);

    return estimate;
  } catch (err) {
    console.error(err);
    return 0;
  }
}
```

## The Trade-off: Accuracy vs. Speed

While this method is blazingly fast, it comes with one massive caveat: **It is an estimate, not a promise.**

### When it is highly accurate:

- On tables that don't change millions of times a minute.
- If your database auto-vacuum or `ANALYZE` routines run frequently to keep internal statistics fresh.

### When it lies to you:

- **Right after heavy writes:** If your app just bulk-inserted 500,000 rows a few seconds ago, `EXPLAIN` will not know they
  exist yet until the database updates its statistics.
- **Complex `WHERE` clauses:** If your query has deeply nested `OR`, `AND`, or string-matching (`LIKE '%text%'`) conditions,
  the query planner's guessing accuracy drops significantly.

## Summary Checklist for the Frontend

If you switch your backend API to use `EXPLAIN` estimates for pagination, make sure your frontend UI reflects this
compromise:

- Change UI text from `"1 to 10 of 42,512 results"` to **`"About 42,500 results"`**.
- If the estimate says `0`, but you want to be safe, do a fallback check or simply show a "No items found" state dynamically
  when the actual data array comes back empty.
