Securing a full-stack application with a **Node.js** backend and a **React.js** frontend requires implementing security
layers at both ends.

Here is a practical checklist based on the **OWASP Top 10** and the **OWASP Node.js/API Security Cheat Sheets**, mapped
directly to your tech stack.

---

## 1. Injection Prevention (OWASP A03:2021)

Node.js applications often interact with SQL (PostgreSQL, MySQL) or NoSQL (MongoDB) databases. Injection occurs when
untrusted user input is executed as code or a query.

- **Never use String Concatenation for Queries:** Avoid constructing queries dynamically using template literals or string
  addition.
- **Use Parameterized Queries / ORMs:** Use tools like Prisma, Sequelize (SQL), or Mongoose (NoSQL). They automatically
  parameterize or sanitize inputs.
- **Sanitize NoSQL Keys:** Attackers can inject operators (like `{"$gt": ""}`) into Express body parsers to bypass password
  checks. Use `express-mongo-sanitize` to strip `$` and `.` characters from incoming requests.

```javascript
// BAD: Vulnerable to SQL injection
const user = await db.query(`SELECT * FROM users WHERE email = '${req.body.email}'`);

// GOOD: Parameterized query (or use an ORM)
const user = await db.query("SELECT * FROM users WHERE email = $1", [req.body.email]);
```

## 2. Secure Data Transmission & HTTP Headers (OWASP A05:2021)

Misconfigured servers often leak critical environment information or fail to leverage browser security features.

- **Use Helmet middleware:** Helmet automatically configures secure HTTP response headers (e.g., hiding
  `X-Powered-By: Express`, setting `X-Content-Type-Options`, and configuring standard protection against clickjacking).
- **Implement Cross-Origin Resource Sharing (CORS) correctly:** Do not use `origin: '*'` in production. Restrict the `origin`
  explicitly to your React frontend's domain.

```javascript
const express = require("express");
const helmet = require("helmet");
const cors = require("cors");
const app = express();

app.use(helmet()); // Sets protective headers

app.use(
  cors({
    origin: process.env.FRONTEND_URL, // e.g., https://my-app.com
    credentials: true, // Allow cookies/authorization headers
  })
);
```

## 3. Broken Authentication & Session Management (OWASP A07:2021)

Because React handles the UI and Node.js manages the session, how you transfer and store authentication data determines your
vulnerability to Cross-Site Scripting (XSS) and Cross-Site Request Forgery (CSRF).

- **Never Store JWTs in React's `localStorage`:** If your React application suffers an XSS vulnerability, malicious scripts
  can read `localStorage` and steal the token.
- **Use `httpOnly` and `Secure` Cookies:** Issue JWTs or session IDs from Node.js in cookies configured with:
- `httpOnly: true` (Prevents React/JavaScript from accessing the cookie, blocking XSS theft).
- `secure: true` (Forces the cookie to only be sent over HTTPS).
- `sameSite: 'strict'` or `'lax'` (Provides built-in defense against CSRF attacks).

- **Hash Passwords Properly:** Use `bcrypt` or `argon2` with a robust salt factor. Never use fast, standard hashing
  algorithms like MD5 or SHA-256 for password storage.

## 4. Cross-Site Scripting (XSS) Defenses

While React natively escapes text values to prevent XSS, you can still introduce security holes if you bypass its safeguards.

- **Avoid `dangerouslySetInnerHTML`:** Avoid using this attribute in React unless the content has been rigorously sanitized
  on the server side or using a client-side library like `DOMPurify`.
- **Sanitize User-Supplied HTML:** If users are allowed to upload rich text (HTML), sanitize it on your Node.js backend using
  `sanitize-html` before saving it to the database.

## 5. Denial of Service (DoS) & Rate Limiting

Node.js operates on a single-threaded event loop. If an attacker floods your API with requests or sends massive JSON
payloads, they can easily crash or freeze the server.

- **Implement Rate Limiting:** Restrict how many requests an IP address can make within a given window using
  `express-rate-limit`.
- **Limit Request Body Sizes:** Prevent attackers from sending huge payloads to crash your JSON parser.

```javascript
const rateLimit = require("express-rate-limit");

// Limit each IP to 100 requests per 15 minutes
const limiter = rateLimit({
  windowMs: 15 * 60 * 1000,
  max: 100,
  message: "Too many requests from this IP, please try again later.",
});
app.use("/api/", limiter);

// Restrict incoming JSON payloads to 10kb
app.use(express.json({ limit: "10kb" }));
```

## 6. Software and Data Integrity Failures (OWASP A06 & A08:2021)

The Node.js ecosystem relies heavily on npm packages. Vulnerabilities in deeply nested dependencies can put your backend at
risk.

- **Audit Dependencies Regularly:** Run `npm audit` locally and integrate tools like Snyk or GitHub's Dependabot into your
  CI/CD pipeline to catch vulnerable packages early.
- **Enforce Lockfiles:** Always use `npm ci` rather than `npm install` in your deployment or Docker build scripts. This
  ensures the environment mirrors your exact `package-lock.json` file, preventing the accidental installation of compromised
  or altered downstream packages.
- **Disable Lifecycle Scripts:** Prevent malicious npm packages from running arbitrary code on your system during
  installation by running:

```bash
npm config set ignore-scripts true

```

---

## 7. Security Logging and Monitoring (OWASP A09:2021)

Without proper logging, you cannot detect if your application is currently under attack or audit what occurred after a
breach.

- **Do Not Use `console.log` for Production Logs:** Use a structured logging framework like `pino` or `winston`. They handle
  high throughput gracefully and output standardized JSON formats that can be parsed by monitoring tools (Datadog, ELK
  stack).
- **Never Log Sensitive Data:** Write middleware carefully to ensure passwords, full credit card numbers, and JWTs are
  stripped out before writing to logs.
