A **JSON Web Token (JWT)** is an open standard (RFC 7519) used to securely transmit information between parties as a JSON
object. Because it is digitally signed, the information can be verified and trusted. JWTs are most commonly used for
**authentication** and **stateless information exchange**.

---

## How a JWT Works (High-Level)

A JWT is typically composed of three parts separated by dots (`.`): **Header**, **Payload**, and **Signature**.

- **Header:** Specifies the token type (JWT) and the signing algorithm (e.g., HMAC SHA256 or RSA).
- **Payload:** Contains the "claims," which is the actual data being passed (like user ID, username, and token expiration
  time).
- **Signature:** Created by hashing the encoded header, encoded payload, and a secret key. This ensures the token wasn't
  tampered with along the way.

---

## The Pros of Using JWT

### 1. Stateless and Scalable

This is the biggest selling point. Traditional session authentication requires the server to store session data in memory or
a database (like Redis). With JWT, all the user data is stored directly _inside_ the token on the client side. The server
doesn't need to look up a database to know who you are; it just validates the signature. This makes horizontal scaling
(adding more servers) incredibly easy.

### 2. Great for Mobile and Microservices

Cookies don't always play nice with mobile apps or across different domains. JWTs are usually sent via the HTTP
`Authorization` header (`Authorization: Bearer <token>`), making them highly compatible with iOS, Android, and decoupled
microservice architectures.

### 3. Built-in Expiration

JWTs include a registered claim called `exp` (expiration time). The token automatically becomes invalid after a certain
period, which forces the client to re-authenticate or use a refresh token without requiring backend intervention to track the
clock.

### 4. Reduced Database Load

Because the server can trust the payload data just by verifying the signature, you reduce the number of database queries
needed per request to fetch basic user details.

---

## The Cons of Using JWT

### 1. Difficult to Invalidate Prematurely

Because JWTs are stateless, the server has no natural way to "delete" a token before it expires. If a user logs out, or if an
admin wants to ban a user immediately, a JWT remains valid until its `exp` time hits.

> **The Workaround:** To fix this, developers often have to build a "blacklist" or "allowlist" in a database like Redis.
> However, doing this reintroduces statefulness, stripping away one of JWT's main benefits.

### 2. Security Vulnerabilities (If Misconfigured)

If someone steals a user's JWT (via Cross-Site Scripting or XSS), they have full access to that user's account until the
token expires. Furthermore, early implementations suffered from vulnerabilities where attackers could change the header
algorithm to `"none"` to bypass signature verification entirely (though modern libraries block this by default).

### 3. Data Overhead (Size)

JWTs are Base64URL-encoded, which makes them larger than a standard session ID cookie. If you start stuffing a lot of user
roles, permissions, and profile data into the payload, the token size grows. Because this token is sent with _every single
HTTP request_, it can increase bandwidth usage and slow down performance.

### 4. Data is Visible to Anyone

A common misconception is that JWTs encrypt data. **They do not.** They are only _signed_ to prevent tampering. Anyone can
take a JWT, drop it into a decoder (like jwt.io), and read the payload. You should never store sensitive information like
passwords, credit card numbers, or SSNs inside a JWT.

---

## Summary: When should you use JWT?

| Use JWT When...                                                 | Avoid JWT When...                                                  |
| --------------------------------------------------------------- | ------------------------------------------------------------------ |
| You are building a **microservices** architecture.              | You are building a traditional, monolithic web app.                |
| You need to support **mobile apps** alongside web apps.         | You need the ability to **instantly revoke** sessions at any time. |
| Your app requires **high scalability** across multiple servers. | You need to store **highly sensitive data** in the session.        |
