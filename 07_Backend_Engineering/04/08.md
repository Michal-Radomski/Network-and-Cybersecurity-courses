The most secure and industry-standard way to handle a refresh token in a React/Node.js application is to store it in an
**`httpOnly`, `Secure`, and `SameSite=Strict` (or `Lax`) cookie**.

You should **never** store a refresh token in `localStorage` or `sessionStorage` because those are accessible via JavaScript,
making your application highly vulnerable to Cross-Site Scripting (XSS) attacks.

Here is the architectural blueprint and code implementation for the best approach.

---

## The Strategy: Split-Token Architecture

To maximize security, you handle the two tokens differently:

1. **Access Token:** Short-lived (e.g., 15 minutes). Sent to React as a JSON response and stored **in-memory** (just a
   standard JavaScript variable or state).
2. **Refresh Token:** Long-lived (e.g., 7 days). Sent from Node as an **`httpOnly` cookie**. The React frontend cannot read
   this token via JavaScript, but the browser will automatically attach it to API requests directed at your auth server.

---

## 1. Backend Setup (Node.js / Express)

When the user logs in, you generate both tokens. You send the access token in the response body, and the refresh token in a
cookie.

```javascript
// authController.js
const jwt = require("jsonwebtoken");

export const login = async (req, res) => {
  // ... Validate user credentials ...
  const userId = user._id;

  const accessToken = jwt.sign({ userId }, process.env.ACCESS_TOKEN_SECRET, { expiresIn: "15m" });
  const refreshToken = jwt.sign({ userId }, process.env.REFRESH_TOKEN_SECRET, { expiresIn: "7d" });

  // Set the refresh token in an HTTP-only cookie
  res.cookie("refreshToken", refreshToken, {
    httpOnly: true, // Prevents JavaScript access (XSS protection)
    secure: true, // Requires HTTPS (use false only in local development)
    sameSite: "strict", // Prevents CSRF attacks
    maxAge: 7 * 24 * 60 * 60 * 1000, // 7 days in milliseconds
  });

  // Send the access token in the body
  return res.json({ accessToken });
};
```

### The Refresh Endpoint

You need a specific endpoint to issue a new access token when the old one expires. It reads the cookie automatically.

```javascript
export const refresh = async (req, res) => {
  const cookies = req.cookies;
  if (!cookies?.refreshToken) return res.status(401).json({ message: "Unauthorized" });

  const refreshToken = cookies.refreshToken;

  jwt.verify(refreshToken, process.env.REFRESH_TOKEN_SECRET, (err, decoded) => {
    if (err) return res.status(403).json({ message: "Forbidden" });

    // Issue a brand new, short-lived access token
    const accessToken = jwt.sign({ userId: decoded.userId }, process.env.ACCESS_TOKEN_SECRET, { expiresIn: "15m" });

    res.json({ accessToken });
  });
};
```

## 2. Frontend Setup (React & Axios)

Because the access token is kept in React state, it will disappear if the user refreshes the page. To fix this, you use Axios
**interceptors**.

Think of an interceptor as a checkpoint: whenever a 401 (Unauthorized) error occurs because an access token expired, the
interceptor pauses the request, silently calls the `/refresh` endpoint to get a new access token, updates the state, and
retries the original request seamlessly.

### Axios Instance Configuration

Ensure `withCredentials: true` is set. This tells Axios to automatically include the `httpOnly` refresh token cookie in its
requests.

```javascript
import axios from "axios";

const api = axios.create({
  baseURL: "https://api.yourdomain.com",
  withCredentials: true, // Crucial for sending/receiving cookies
});

// A placeholder variable for the in-memory access token
let inMemoryToken = null;

export const setAccessToken = (token) => {
  inMemoryToken = token;
};

// Request interceptor: Attach the access token to headers if it exists
api.interceptors.request.use(
  (config) => {
    if (inMemoryToken) {
      config.headers["Authorization"] = `Bearer ${inMemoryToken}`;
    }
    return config;
  },
  (error) => Promise.reject(error)
);

// Response interceptor: Catch 401 errors and refresh the token
api.interceptors.response.use(
  (response) => response,
  async (error) => {
    const originalRequest = error.config;

    // If the error is 401 and we haven't retried this request yet
    if (error.response?.status === 401 && !originalRequest._retry) {
      originalRequest._retry = true;

      try {
        // Call the refresh endpoint (the browser sends the httpOnly cookie automatically)
        const response = await axios.get("https://api.yourdomain.com/refresh", {
          withCredentials: true,
        });

        const { accessToken } = response.data;
        setAccessToken(accessToken); // Update memory

        // Update the original failed request header and retry it
        originalRequest.headers["Authorization"] = `Bearer ${accessToken}`;
        return api(originalRequest);
      } catch (refreshError) {
        // Refresh token itself expired or is invalid -> Force logout
        setAccessToken(null);
        window.location.href = "/login";
        return Promise.reject(refreshError);
      }
    }
    return Promise.reject(error);
  }
);

export default api;
```

---

## 3. How to Handle Page Refreshes

When a user reloads the browser tab, your React `inMemoryToken` turns back to `null`.

To handle this elegantly, call your `/refresh` endpoint immediately inside a root-level `useEffect` (or your Auth Context
provider) when the application mounts for the first time. If the user has a valid refresh token cookie, they will be silently
logged back in instantly without seeing a login screen.

---

## Summary Checklist for Production

- [ ] Is your Refresh Token cookie configured with `httpOnly: true`?
- [ ] Is `secure: true` enabled in production (requires HTTPS)?
- [ ] Is `sameSite: 'strict'` or `'lax'` enabled to prevent Cross-Site Request Forgery?
- [ ] Is your Access Token kept strictly in-memory (React State) and **never** written to `localStorage`?
- [ ] Did you set `withCredentials: true` on your frontend Axios/Fetch configurations?
