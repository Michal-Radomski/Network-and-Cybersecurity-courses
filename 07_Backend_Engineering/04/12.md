Integrating Kerberos into a modern React/Node.js web application typically involves a protocol called **SPNEGO** (Simple and
Protected GSSAPI Negotiation Mechanism). This allows the browser to securely pass the user's Kerberos ticket to your Node.js
backend via HTTP headers.

Because Kerberos is designed for internal networks, this setup is almost exclusively used in enterprise intranet environments
where users are logged into a Windows Domain (Active Directory).

Here is the architectural breakdown and implementation guide to connect the pieces.

---

## The Architecture Overview

1. **The Client (React):** The browser automatically handles the Kerberos handshake. JavaScript/React does not manage keys or
   passwords; it simply tells the browser to include credentials.
2. **The Backend (Node.js):** Validates the incoming ticket against your Key Distribution Center (KDC / Active Directory)
   using machine credentials.
3. **The Session:** Because Kerberos tickets are heavy and meant for network layers, the standard practice is to use Kerberos
   _only once_ at initial login to verify identity, and then issue a standard, short-lived **JWT or Session Cookie** to React
   for subsequent API calls.

---

## Step 1: Pre-requisites & Environment Setup

Before writing code, your Node.js server needs to be recognized by your Kerberos domain.

1. **Create a Service Account:** In Active Directory, create a user account for your Node.js backend.
2. **Generate a Keytab File:** A keytab file contains the SPN (Service Principal Name) and cryptographic keys, allowing your
   Node app to authenticate with the KDC without a plaintext password. Run this on your domain controller:

```bash
ktpass /out nodeapp.keytab /princ HTTP/api.yourdomain.com@YOURDOMAIN.COM /mapuser node_service_account /pass YourPassword /crypto AES256-SHA1 /ptype KRB5_NT_PRINCIPAL

```

3. **Move the Keytab:** Place the `nodeapp.keytab` file securely on the server hosting your Node.js application.

---

## Step 2: Node.js Backend Implementation

You will need a library capable of handling GSSAPI/SPNEGO token verification. A reliable approach in Node.js uses the
`kerberos` and `express-gssapi` (or similar wrapper) packages.

### 1. Install Dependencies

Your Node environment will need the build tools (`node-gyp`) installed because the `kerberos` library compiles native C++
bindings.

```bash
npm install express kerberos jsonwebtoken cookieParser

```

### 3. Setup Express Middleware

Configure your backend to intercept requests, check for the `Authorization: Negotiate <ticket>` header, and validate it using
the keytab file.

```javascript
const express = require("express");
const kerberos = require("kerberos");
const jwt = require("jsonwebtoken");
const app = express();

// Configure Kerberos environment variables pointing to your keytab
process.env.KRB5_KTNAME = "/path/to/secure/nodeapp.keytab";

app.get("/api/auth/kerberos", async (req, res) => {
  const authHeader = req.headers.authorization;

  // Step 1: If no ticket is sent, challenge the browser to send one
  if (!authHeader || !authHeader.startsWith("Negotiate ")) {
    res.setHeader("WWW-Authenticate", "Negotiate");
    return res.status(401).send("Authentication Required");
  }

  // Step 2: Extract the base64 Kerberos ticket from the header
  const ticket = authHeader.substring(10);

  try {
    // Step 3: Initialize Kerberos server context
    const serverContext = await kerberos.initializeServer("HTTP@api.yourdomain.com");

    // Step 4: Step the token to validate it against the KDC via the keytab
    await serverContext.step(ticket);

    // If successful, extract the authenticated username (e.g., john.doe@YOURDOMAIN.COM)
    const username = serverContext.username;

    // Clean up context memory
    await serverContext.clean();

    // Step 5: Transition to Web Authentication (Issue a short-lived JWT)
    const accessToken = jwt.sign({ user: username }, process.env.JWT_SECRET, { expiresIn: "1h" });

    // Send the token back to React
    return res.json({ token: accessToken, user: username });
  } catch (error) {
    console.error("Kerberos validation failed:", error);
    return res.status(403).send("Invalid Kerberos Ticket");
  }
});

app.listen(3000, () => console.log("Server running on port 3000"));
```

## Step 3: React Frontend Implementation

React itself cannot access Kerberos tickets directly due to browser security models. Instead, you must instruct the browser
to negotiate the authentication implicitly via `fetch` or `axios`.

### Adjusting API Requests

When making the initial login request to your Node API, you must explicitly pass credentials flags (`credentials: 'include'`
for Fetch, or `withCredentials: true` for Axios). This signals the browser to look for a local Kerberos ticket and attach the
`Negotiate` header.

```javascript
import React, { useEffect, useState } from "react";
import axios from "axios";

function App() {
  const [user, setUser] = useState(null);
  const [loading, setLoading] = useState(true);

  useEffect(() => {
    // Attempt Windows/Kerberos Silent Login
    axios
      .get("https://api.yourdomain.com/api/auth/kerberos", {
        withCredentials: true, // Forces the browser to attach OS-level Kerberos/NTLM tokens
      })
      .then((response) => {
        const { token, user } = response.data;
        // Store the returned web token (JWT) in state for future API requests
        localStorage.setItem("webToken", token);
        setUser(user);
        setLoading(false);
      })
      .catch((err) => {
        console.error("Kerberos handshake failed or was rejected.", err);
        setLoading(false);
      });
  }, []);

  if (loading) return <div>Logging you in via Windows Single Sign-On...</div>;

  return <div>{user ? <h1>Welcome, {user}!</h1> : <h1>Access Denied</h1>}</div>;
}

export default App;
```

## Critical Troubleshooting Notes for This Setup

- **Intranet Zone Requirements:** Browsers (Chrome, Edge, Safari) will **refuse** to automatically send Kerberos tickets
  unless the target API domain belongs to the local intranet zone. For testing or enterprise-wide deployments, you usually
  need to push a Group Policy (GPO) configuring the `AuthServerAllowlist` policy to include your domain (`*.yourdomain.com`).
- **HTTPS Only:** Browsers typically block SPNEGO/Negotiate tokens over unencrypted HTTP channels. Ensure your Node.js
  application is listening over TLS/HTTPS even in development environments.
- **DNS Names Matter:** Kerberos maps services to exact domain names. You cannot test Kerberos using `localhost` or an IP
  address (`127.0.0.1`). You must map a local domain name in your development machine's `hosts` file that aligns precisely
  with your Keytab Principal name.
