An **asymmetric JWT** refers to a JSON Web Token that is signed and verified using an asymmetric cryptographic algorithm (a
public/private key pair), such as **RS256** (RSA using SHA-256) or **ES256** (ECDSA using P-256 and SHA-256).

In contrast to symmetric signing (like HS256), where the exact same "secret phrase" is used to both create and verify the
token, asymmetric JWTs split these responsibilities between two different keys.

---

## The Core Concept: Public and Private Keys

Asymmetric JWT architecture relies on a mathematical relationship between two keys:

1. **The Private Key:** Kept strictly secret by the authentication server (Identity Provider). It is used exclusively to
   **sign** the JWT.
2. **The Public Key:** Shared openly with anyone who needs it (microservices, frontend apps, third-party APIs). It is used
   exclusively to **verify** that the token is authentic and untampered with.

---

## How It Works in Practice

1. **Token Generation (Signing):** A user logs into the Identity Provider (Auth Server). The server creates the JWT payload
   (e.g., user ID, roles) and uses its **Private Key** to generate a digital signature. The complete JWT is sent back to the
   client.
2. **Token Consumption (Verification):** The client sends this JWT to an API gateway or microservice to request data. The
   microservice fetches the Auth Server's **Public Key** and uses it to verify the signature.
3. **The Result:** If the verification succeeds, the microservice knows with 100% certainty that the token was created by the
   Auth Server and hasn't been altered, because only the Auth Server possesses the private key required to make that
   signature.

---

## Why Use Asymmetric JWTs? (The Advantages)

### 1. Superior Security in Microservices

In a symmetric setup (HS256), every single microservice needs to know the secret key to verify tokens. If a single
microservice is compromised, the attacker steals the secret and can now forge valid master tokens for the entire system. With
asymmetric setup, microservices only hold the **public key**. If a microservice is hacked, the attacker can verify tokens,
but they cannot _create_ or alter them.

### 2. Third-Party Integration

If you want external developers or partner companies to integrate with your API, you can share your public key freely. They
can securely verify your users' tokens without you ever risking your internal system's core security.

### 3. Centralized Control, Decentralized Verification

The central auth server retains total control over identity issuance. The rest of the network can dynamically scale and
verify identities independently without constantly pinging the auth server to ask, _"Is this token valid?"_

---

## The Trade-offs (The Disadvantages)

### 1. Performance Overhead

Asymmetric cryptography requires complex mathematical operations (like modular exponentiation or elliptic curve math).
Generating and verifying an asymmetric signature (like RS256) is noticeably slower and more CPU-intensive than symmetric
hashing (like HS256).

> _Note: ES256 (Elliptic Curve) is a popular modern choice because it offers the same security as RSA but with smaller key
> sizes and faster processing speeds._

### 2. Key Management Complexity

You have to securely manage, store, and rotate public/private key pairs. Typically, this is solved by using a **JWKS (JSON
Web Key Set)** endpoint—a public URL hosted by your auth server where microservices can automatically fetch the latest public
keys.

---

## Symmetric (HS256) vs. Asymmetric (RS256/ES256)

| Feature             | Symmetric (HS256)                                      | Asymmetric (RS256 / ES256)                   |
| ------------------- | ------------------------------------------------------ | -------------------------------------------- |
| **Keys Used**       | One single secret key                                  | A Private key and a Public key               |
| **Who can sign?**   | Anyone who can verify                                  | Only the holder of the Private Key           |
| **Who can verify?** | Anyone who can sign                                    | Anyone with access to the Public Key         |
| **Speed**           | Extremely fast                                         | Slower (CPU-intensive)                       |
| **Best Used For**   | Small, monolithic apps; internal, single-server setups | Large ecosystems, microservices, public APIs |
