**Kerberos** is a computer network authentication protocol designed to prove the identity of users or services across an
insecure network. It relies on a **trusted third party** to verify identities and allows mutual authentication—meaning the
user proves who they are to the server, and the server proves its identity back to the user.

Named after Cerberus, the three-headed dog from Greek mythology that guarded the gates of the underworld, Kerberos gets its
name because it relies on **three distinct heads** to function: the Client, the Server, and the Trusted Third Party.

---

## The Three Components: The Key Distribution Center (KDC)

In a Kerberos ecosystem, the trusted third party is called the **Key Distribution Center (KDC)**. It maintains a secure
database of all users, services, and their cryptographic keys. The KDC itself is split into two logical parts:

1. **Authentication Server (AS):** Verifies the initial login credentials of a user and issues a temporary credential called
   a **Ticket Granting Ticket (TGT)**.
2. **Ticket Granting Server (TGS):** Uses the TGT to issue specific service tickets allowing the user to access actual
   resources (like a file share or a database).

---

## How Kerberos Works: The Step-by-Step Flow

Instead of requiring you to type your password every time you want to access a different server on the network, Kerberos uses
a system of "tickets."

### Step 1: Initial Authentication (Getting the TGT)

- The client enters their username at their workstation.
- The workstation sends a request to the **Authentication Server (AS)** in the KDC.
- The AS checks its database. If the user exists, it sends back two things: a **Session Key** (encrypted with the user's
  password hash) and a **Ticket Granting Ticket (TGT)** (encrypted with a key only the KDC knows).
- The client's machine prompts for the password, decrypts the session key, and stores the TGT safely in memory.

### Step 2: Requesting Access to a Service (Getting a Service Ticket)

- Now, the user wants to access a shared file server.
- The client sends a request to the **Ticket Granting Server (TGS)** containing the TGT and an "authenticator" (a timestamp
  encrypted with the session key).
- The TGS decrypts the TGT, verifies the timestamp (to ensure it isn't an old request being replayed by a hacker), and
  creates a **Service Ticket** specifically for that file server.

### Step 3: Accessing the Service

- The client sends the **Service Ticket** directly to the target file server.
- The file server decrypts the ticket using its own secret key shared with the KDC.
- If everything matches, the file server grants access to the user without ever seeing the user's password.

## Why Use Kerberos? (The Pros)

### 1. Strong Security & No Passwords in Transit

Passwords are never sent across the network in plain text, nor are they stored on the client machine after initial login.
Only heavily encrypted keys and tickets move across the wire.

### 2. Single Sign-On (SSO)

A user enters their password exactly once at the beginning of the day. For the rest of their session, the background
ticket-exchange process automatically authenticates them to every compatible printer, database, and file share on the
network.

### 3. Mutual Authentication

In modern networks, rogue servers can try to trick clients into sending data. Kerberos prevents this because the client
requires the server to decrypt a ticket and prove its own identity before any real data changes hands.

### 4. Resistance to Replay Attacks

Kerberos heavily relies on precise timestamps. If an attacker intercepts an authentication packet on the network and tries to
resend it later to gain unauthorized access, the server will reject it because the timestamp has expired (usually a 5-minute
window).

---

## The Catch (The Cons)

### 1. Single Point of Failure

If the central KDC goes down, the entire network grinds to a halt. No one can log in, and no services can verify identities.
In production environments, this requires building redundant, highly available KDC clusters.

### 2. Strict Time Synchronization

Because Kerberos tickets use timestamps to prevent replay attacks, all clocks on the network (clients, servers, and KDCs)
must be tightly synchronized. If a computer's clock drifts by more than 5 minutes out of sync, authentication will fail
completely.

### 3. Complex Setup and Maintenance

Configuring a Kerberos realm requires careful network planning, strict access controls, and administrative overhead compared
to simpler token-based systems.

---

## Where is Kerberos Used Today?

If you have ever worked in a corporate office, you have almost certainly used Kerberos. It is the underlying authentication
protocol used by **Microsoft Active Directory** to manage Windows domains. It is also natively supported by macOS and Linux
operating systems for enterprise enterprise network management.
