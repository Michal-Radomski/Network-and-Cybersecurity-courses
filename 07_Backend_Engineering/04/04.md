Originally defined by RFC 4122 (and heavily updated in 2024 under **RFC 9562**), UUIDs have evolved significantly. While the
original versions relied on MAC addresses and random numbers, the modern versions are designed specifically for distributed
databases and chronological sorting.

Every UUID is a 128-bit number, usually represented as a 36-character string split into five groups by hyphens (e.g.,
`xxxxxxxx-xxxx-Mxxx-Nxxx-xxxxxxxxxxxx`).

The `M` position always represents the **version** (1 through 8), and the `N` position represents the **variant** (usually
specifying the RFC layout).

Here is a breakdown of all UUID versions from v1 to v8.

---

## 🏛️ The Traditional Versions (RFC 4122)

### UUIDv1: Time & MAC Address (Chronological)

UUIDv1 is generated using the current timestamp (counted in 100-nanosecond intervals since October 15, 1582) and the
generating computer’s physical **MAC address**.

- **How it works:** It guarantees uniqueness because no two computers have the same MAC address, and time always moves
  forward.
- **The Flaw:** **Privacy and security.** Because the MAC address is baked into the ID, anyone who sees the UUID can identify
  the specific machine that created it and the exact time it was generated.

### UUIDv2: DCE Security (Domain/UID)

UUIDv2 is highly specialized and rarely used today. It is similar to v1 but replaces a portion of the timestamp with a local
user identifier (like a POSIX User ID or Group ID) and a security domain.

- **The Flaw:** It sacrifices timestamp resolution for local security data, leading to a much higher risk of ID collisions if
  generated quickly.

### UUIDv3: Name-based using MD5 (Deterministic)

UUIDv3 is generated by hashing a "namespace" (like a URL or Distinguished Name) and a specific string input using the **MD5**
algorithm.

- **How it works:** It is completely deterministic. If you input the namespace `URL` and the name `google.com` into a v3
  generator, it will output the exact same UUID every single time.
- **The Flaw:** MD5 is cryptographically broken and prone to collision vulnerabilities.

### UUIDv4: Purely Random (The Reigning King)

UUIDv4 is the most widely used version today. It abandons timestamps and names completely, replacing 122 of the 128 bits with
purely random (or pseudo-random) data.

- **How it works:** It relies on the sheer mathematical improbability of a collision. The odds of generating two identical v4
  UUIDs are so astronomical that it is practically impossible.
- **The Flaw:** As discussed previously, because it is entirely random, it causes severe database index fragmentation in
  relational databases.

### UUIDv5: Name-based using SHA-1 (Deterministic)

UUIDv5 is identical in concept to UUIDv3, but it replaces the insecure MD5 hashing algorithm with **SHA-1**.

- **Use Case:** It is still widely used today when you need to generate a unique, reproducible ID based on text input (e.g.,
  converting a user's email address into a consistent ID without exposing the email itself).

---

## 🚀 The Modern Versions (RFC 9562 / New in 2024)

To fix the database performance flaws of UUIDv4 and the privacy flaws of UUIDv1, the IETF officially standardized three new
versions.

### UUIDv6: Reordered Time (The v1 Fix)

UUIDv6 takes the exact same timestamp data from UUIDv1 but **reverses the byte order** so that the most significant time bits
come first.

- **Why it matters:** It provides chronological sortability for databases, making it index-friendly.
- **The Catch:** It still optionally uses a MAC address or random data for node identification, making it a "legacy bridge"
  version rather than the ideal modern choice.

### UUIDv7: Unix Epoch Time (The New Standard)

UUIDv7 is the modern replacement for UUIDv4 in application development. It splits the 128 bits into two main parts:

1. **48-bit Timestamp:** Uses standard Unix Epoch time in milliseconds (the same time format used by almost all programming
   languages).
2. **74 bits of randomness:** Provides robust collision protection.

- **Why it’s great:** It gives you the best of both worlds. It naturally sorts chronologically (like a ULID), making it
  incredibly fast for database indexing, but it remains fully compliant with standard UUID parsers and hides machine privacy.

### UUIDv8: Custom / Vendor-Specific

UUIDv8 is an open-ended format designed for teams who need a 128-bit identifier but want to implement their own custom
layout.

- **How it works:** Aside from the version and variant bits, the remaining bits can be configured however you want. You could
  use a custom timestamp, a node ID, and shard identifiers tailored to your specific cloud architecture.
