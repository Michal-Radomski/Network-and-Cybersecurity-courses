To understand how a request travels from a frontend client to a backend process, we have to look past the high-level code and
dive into the operating system kernel and network stack.

Here is the step-by-step lifecycle of a single HTTP/TCP request as it transitions from user space to kernel space, and back
again.

---

## Phase 1: The Frontend Departure (User Space)

The journey begins in the client's browser or mobile app.

```text
[ Browser / Frontend ] ──( HTTP Request )──> [ OS Network Stack ]

```

1. **Application Layer Construction:** The frontend app executes a network call (e.g., `fetch()` in Javascript). The browser
   constructs a raw HTTP request string (headers, method, body).
2. **DNS Resolution:** If the domain name isn't cached, the OS resolves the domain (e.g., `api.example.com`) to an IP address
   using DNS.
3. **The Socket Call:** The browser makes a system call (`sys_socket` and `sys_connect`) to the client OS kernel, requesting
   a TCP connection to the backend's IP address on a specific port (usually 80 or 443).

---

## Phase 2: The Network Highway

The request is broken down into small packets to cross the internet.

1. **Serialization:** The client OS kernel wraps the HTTP data in a TCP segment, wraps that in an IP packet, and translates
   it into raw electrical, optical, or radio signals via the Network Interface Card (NIC).
2. **Routing:** The packets hop across routers, switches, and fiber-optic cables across the globe until they arrive at the
   hosting provider's data center or cloud network.

---

## Phase 3: Arrival at the Server NIC & Kernel Space

This is where the magic happens. The packets hit the backend server hardware and enter the Linux/Unix kernel.

### 1. The Hardware Interrupt (NIC to Ring Buffer)

When packets arrive at the server’s physical Network Interface Card (NIC), the NIC copies the packet data directly into a
pre-allocated region of system memory (RAM) called the **Rx (Receive) Ring Buffer** using Direct Memory Access (DMA). Once
copied, the NIC fires a hardware interrupt (IRQ) to the CPU to say, _"Hey, I have new data."_

### 2. SoftIRQs and the `sk_buff`

To prevent the CPU from freezing up during high traffic, the kernel quickly handles the hardware interrupt and delegates the
actual processing to a asynchronous software interrupt (**SoftIRQ**), specifically via the `net_rx_action` function. The
kernel allocates a data structure called an `sk_buff` (socket buffer) to wrap the packet data as it moves up the network
stack.

### 3. Walking Up the Kernel Network Stack

The kernel inspects the packet layer by layer:

- **Layer 2 (Data Link):** Verifies the MAC address and strips the Ethernet header.
- **Layer 3 (Network):** Inspects the IP header to ensure the packet is destined for this server. If iptables/nftables
  (firewalls) allow it, the IP header is stripped.
- **Layer 4 (Transport):** Inspects the TCP header. It checks the sequence numbers, handles TCP checksums, and identifies the
  destination port (e.g., Port 443).

---

## Phase 4: The Socket Buffer and Context Switch

Once the kernel verifies the TCP data, it needs to hand it off to the application waiting for it (like NGINX, Node.js, or
Go).

```text
[ Kernel Space: TCP Buffer ]
       │
       ▼  (Context Switch via epoll / sys_read)
[ User Space: Backend Process ]

```

1. **Finding the Socket:** The kernel looks at its internal TCP socket table to find which process is listening on that
   specific destination port.
2. **The Wake-Up Call:** Most modern backend servers use non-blocking I/O driven by the `epoll` (Linux) or `kqueue`
   (BSD/macOS) system calls. The kernel adds the data to that specific socket’s receive buffer and notifies the sleeping
   backend process that data is ready to be read.
3. **The Context Switch:** The CPU halts what it's doing, switches from **Kernel Mode** to **User Mode**, and grants CPU time
   to the backend process.

---

## Phase 5: The Backend Process (User Space)

The request has finally arrived at its destination.

1. **Reading the Stream:** The backend process executes a `read()` or `recv()` system call to copy the data out of the
   kernel’s socket buffer and into its own user-space application memory.
2. **HTTP Parsing:** The backend runtime (e.g., Node.js V8 engine or a Go binary) parses the raw bytes back into an HTTP
   request object.
3. **Routing & Execution:** Your application code takes over. It matches the URL path (e.g., `/api/v1/users`) to a
   controller, interacts with a database, runs business logic, and prepares an HTTP response.

---

## The Journey Home

Once the backend process is done, the entire pipeline runs in exact reverse: The application writes the response to its
user-space buffer $\rightarrow$ executes a `write()` system call $\rightarrow$ the kernel wraps it in TCP/IP headers
$\rightarrow$ the NIC transmits the data back across the internet $\rightarrow$ the frontend browser receives it, decrypts
it, and renders the updated UI to the user.
