In traditional web applications, the client (like a browser) has to constantly ask the server, "Is there new data yet?" This
is called **polling**, and it can be incredibly inefficient.

The **Push Backend Design Pattern** flips this dynamic. Instead of the client begging for updates, the client establishes a
persistent connection with the server _once_. When the server detects a change or receives new data, it immediately "pushes"
that data down to the client in real-time.

---

## How It Works

1. **Handshake:** The client requests a long-lived connection with the server.
2. **Persistent Channel:** The server accepts and keeps this connection open (unlike standard HTTP requests that close
   immediately after a response).
3. **Server-Initiated Event:** When a state change occurs in the backend, the server broadcasts the payload directly through
   the open channel.

### Common Technologies Used

- **WebSockets:** A full-duplex, bidirectional communication protocol over a single TCP connection. Perfect for chat apps or
  multiplayer games.
- **Server-Sent Events (SSE):** A unidirectional protocol where only the server can push data to the client over standard
  HTTP. Great for live sports scores or stock tickers.
- **Long Polling:** An older fallback method where the server holds a request open until it has new data, closes it, and the
  client immediately opens a new one.

## Node.js Example (Using WebSockets)

Let's build a simple real-time push system using Node.js and the popular `ws` library. In this example, the server will
automatically push a live timestamp to all connected clients every 3 seconds.

### 1. Setup

First, initialize your project and install the WebSocket library:

```bash
npm init -y
npm install ws

```

### 2. The Server (`server.js`)

This backend creates a WebSocket server. It tracks connected clients and uses `setInterval` to actively push data to them.

```javascript
const { WebSocketServer } = require("ws");

// Create a WebSocket server listening on port 8080
const wss = new WebSocketServer({ port: 8080 });

console.log("Push server is running on ws://localhost:8080");

// Event: Triggered whenever a new client connects
wss.on("connection", (ws) => {
  console.log("A new client connected!");

  ws.send(JSON.stringify({ message: "Welcome to the real-time push server!" }));

  // Handle client disconnection
  ws.on("close", () => {
    console.log("Client disconnected.");
  });
});

// Simulate a backend data change every 3 seconds and PUSH it to all clients
setInterval(() => {
  const payload = {
    event: "SERVER_UPDATE",
    timestamp: new Date().toISOString(),
    systemLoad: Math.random().toFixed(2), // Simulating real-time metrics
  };

  console.log("Pushing updates to clients...");

  // Broadcast the payload to every connected client
  wss.clients.forEach((client) => {
    // Make sure the connection is still open before sending
    if (client.readyState === 1) {
      client.send(JSON.stringify(payload));
    }
  });
}, 3000);
```

### 3. The Client (`client.js`)

You can run this in a separate Node terminal to see the push mechanism in action.

```javascript
const WebSocket = require("ws");

// Connect to the push server
const ws = new WebSocket("ws://localhost:8080");

ws.on("open", () => {
  console.log("Connected to server. Waiting for pushes...");
});

// Event: Triggered automatically whenever the server pushes data
ws.on("message", (data) => {
  const parsedData = JSON.parse(data);
  console.log("Received Push Notification:", parsedData);
});
```

## Pros and Cons of the Push Pattern

| Pros                                                                                              | Cons                                                                                                                                           |
| ------------------------------------------------------------------------------------------------- | ---------------------------------------------------------------------------------------------------------------------------------------------- |
| **True Real-Time:** Minimal latency between data generation and client awareness.                 | **Resource Intensive:** Keeping thousands of concurrent TCP connections open drains server memory.                                             |
| **Lower Bandwidth:** Eliminates the overhead of thousands of empty HTTP request/response headers. | **Scaling Complexity:** Requires specialized infrastructure (like Redis Pub/Sub or load balancers with sticky sessions) to scale horizontally. |
