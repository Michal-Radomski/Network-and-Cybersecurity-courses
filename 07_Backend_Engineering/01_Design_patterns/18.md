To understand how these concepts tie together, it helps to look at them first as foundational networking principles, and then
see how they apply to modern architecture patterns like HTTP/2 proxying and Connection Pooling.

At their core, **Multiplexing** and **Demultiplexing** are about sharing a single resource to handle multiple pieces of data
efficiently.

---

### The Core Concepts

#### 1. Multiplexing (Many into One)

Multiplexing is the process of combining multiple separate data streams or requests into a **single signal or physical
connection** for transport.

- **The Analogy:** Think of a highway on-ramp. Cars coming from three different local roads (separate streams) all merge into
  a single highway lane (the shared resource) to travel across a bridge.
- **Why do it?** Creating and destroying connections is expensive. Multiplexing allows us to maximize the utility of a
  single, established connection.

#### 2. Demultiplexing (One into Many)

Demultiplexing is the exact reverse process. It takes that single, combined stream arriving at the destination, separates the
mixed data packets, and delivers them to their correct individual recipients.

- **The Analogy:** Once the cars cross the bridge, they hit an off-ramp split that directs them back to their specific
  destination neighborhoods.
- **Why do it?** Without demultiplexing, the receiver wouldn't know which packet of data belongs to which request or
  application.

---

### How They Apply to Software Architecture

When dealing with backends, proxies, and databases, you will see these concepts implemented in two distinct ways: **HTTP/2
Proxying** and **Connection Pooling**.

---

### 1. HTTP/2 Proxying (True Multiplexing)

In older HTTP/1.1 systems, a browser could only send one request per TCP connection at a time. If it needed 10 images, it had
to open multiple connections or queue them sequentially (causing "head-of-line blocking").

**HTTP/2 introduces true multiplexing over a single connection:**

- **The Flow:** A browser establishes **one single TCP connection** to a reverse proxy (like Nginx or Apache).
- **Multiplexing:** The browser breaks down dozens of separate requests (HTML, JS, CSS, images) into interleaved "frames" and
  sends them all simultaneously down that _single_ connection line.
- **Demultiplexing at the Proxy:** The reverse proxy receives this single tangled stream of frames, untangles them
  (demultiplexes), maps them back to their original distinct requests, and forwards them to your backend applications (often
  using standard HTTP/1.1 lines).

> **Key Takeaway:** HTTP/2 proxying allows **concurrent interleaving**—multiple active conversations happening at the exact
> same millisecond over one wire.

---

### 2. Connection Pooling (Reused Segregation)

Connection Pooling is a technique heavily used in database drivers (like PostgreSQL or MongoDB clients) to manage expensive
database connections. It solves a similar problem to multiplexing, but handles it differently.

Instead of opening and closing a new TCP connection every time an Express route needs to run a SQL query, a "pool" of
pre-opened, persistent connections is kept alive in memory.

- **The Flow:** Your backend application initializes a pool of, say, 10 open connections to the database.
- **The "Sharing" Mechanism:** When Request A arrives, it borrows Connection #1 from the pool, runs its query, and returns
  Connection #1 back to the pool. If Request B arrives at the exact same time, it cannot share Connection #1; it must grab
  Connection #2.
- **Is it Multiplexing?** Generally, **no**. Classic connection pooling is technically a form of _time-division reuse_, but
  not true multiplexing. On any single database connection wire, only _one_ query can be active at a single instant. The pool
  scales concurrency by adding more physical connections, not by weaving multiple queries into one wire.

---

### Summary of Differences

| Feature                 | HTTP/2 Proxying                                                                                              | Connection Pooling                                                                                          |
| ----------------------- | ------------------------------------------------------------------------------------------------------------ | ----------------------------------------------------------------------------------------------------------- |
| **Primary Goal**        | Send hundreds of frontend web assets instantly over the internet without network bottlenecks.                | Eliminate the CPU/Time overhead of constantly opening and closing database handshakes.                      |
| **Number of TCP Wires** | Strictly **one** persistent connection between client and proxy.                                             | A **set** of multiple persistent connections (e.g., 10–50 lines) kept alive in a pool.                      |
| **Concurrency Style**   | **Interleaved:** Frames of Request A and Request B are mixed together on the exact same wire simultaneously. | **Serialized per wire:** Connection #1 handles Request A. Request B must use Connection #2 or wait in line. |
| **Where it lives**      | On the transport layer between the Browser and Edge Server (Proxy).                                          | Inside the Application Backend running the database driver.                                                 |
