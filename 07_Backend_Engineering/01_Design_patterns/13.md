**Server-Sent Events (SSE)** is a HTTP-based web technology that allows a server to push real-time data updates to a client
over a single, long-lived connection.

Unlike traditional HTTP requests where the client must constantly ask for data (polling), SSE enables a one-way, continuous
stream of data from the server to the client. It is built natively into the web platform via the `EventSource` API.

---

### How It Works (The Lifecycle)

1. **The Handshake:** The client initiates a standard HTTP request to the server, asking for a stream (typically using the
   `EventSource` JavaScript API).
2. **The Open Connection:** The server responds with a special HTTP header: `Content-Type: text/event-stream`. It keeps this
   connection open indefinitely.
3. **The Continuous Stream:** Whenever the server has new data, it formats it as plain text matching a specific format
   (`data: your_message_here\n\n`) and pushes it down the open channel.
4. **Auto-Reconnect:** If the connection drops (due to network blips or server restarts), the client browser will
   **automatically attempt to reconnect** without you having to write custom retry logic.

---

### Key Characteristics

| Pros                                                                                                                                                                  | Cons                                                                                                                                                                                                            |
| --------------------------------------------------------------------------------------------------------------------------------------------------------------------- | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **Lightweight HTTP:** It operates entirely over standard HTTP/HTTPS, requiring no custom protocols, complex handshakes, or special firewall configurations.           | **One-Way Only (Unidirectional):** Data flows _only_ from server to client. If the client needs to send data back, it must issue a separate standard HTTP POST/PUT request.                                     |
| **Native Browser Support:** Most modern browsers have the `EventSource` API built-in, handling connection management, reconnection, and event parsing automatically.  | **Connection Limits (HTTP/1.1):** When using older HTTP/1.1 protocols, browsers limit the number of open SSE connections to just **6 per domain**. (This is bypassed using HTTP/2, where hundreds can coexist). |
| **Built-in Features:** Supports native event handling (sending different types of messages) and handles reconnection tracking via standard headers (`Last-Event-ID`). | **Text-Only:** SSE natively handles text data (usually JSON strings). It is not designed for streaming binary data like audio or raw video.                                                                     |

---

### Common Use Cases

SSE is perfect for applications that need real-time updates but don't require heavy bidirectional interaction from the user.

- **Live Feeds & Dashboards:** Real-time stock tickers, crypto prices, or social media activity timelines.
- **AI Completion Streaming:** Platforms like ChatGPT use SSE to stream responses character-by-character back to the user.
- **Notification Systems:** Pushing in-app alerts, friend requests, or system updates the second they happen.
- **IoT Monitoring:** Streaming live telemetry or sensor data from a server to a web interface.

---

### SSE vs. WebSockets vs. Long Polling

- **Long Polling** opens and closes a new HTTP connection for every single update, creating massive overhead.
- **WebSockets** provide a full-duplex, two-way connection, which is great for collaborative tools (like Google Docs) or
  multiplayer games, but comes with a heavier server architecture footprint.
- **SSE** is the elegant middle-ground: highly efficient, low-overhead, real-time delivery, tailored specifically for pushing
  data to the client.
