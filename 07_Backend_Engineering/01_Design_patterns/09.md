In backend development, **Short Polling** is a client-server communication pattern where the client repeatedly sends HTTP
requests to the server at fixed, frequent intervals to check for new data or status updates.

It is a "pull-based" approach, meaning the server never initiates communication; it only responds when asked.

---

### How It Works (The Lifecycle)

1. **The Request:** The client sends a standard HTTP request (e.g., `GET /api/status`) to the server.
2. **The Immediate Response:** The server checks its database or cache. Whether there is new data or not, it **immediately**
   responds to the client with the current state.
3. **The Wait:** The client receives the response and waits for a predefined interval (e.g., every 3 or 5 seconds).
4. **The Repeat:** Once the timer hits zero, the client sends another request, repeating the cycle.

---

### Key Characteristics

| Pros                                                                                                                           | Cons                                                                                                                                   |
| ------------------------------------------------------------------------------------------------------------------------------ | -------------------------------------------------------------------------------------------------------------------------------------- |
| **Simple to Implement:** Uses standard HTTP protocols and requires no specialized server-side setup (unlike WebSockets).       | **High Overhead:** Every single request involves TCP handshake overhead and HTTP headers, which wastes bandwidth.                      |
| **Stateless:** The server doesn't need to maintain an open connection with the client, making it easier to scale horizontally. | **Resource Intensive:** The server can get hammered with empty responses (`200 OK` with no new data), wasting CPU and database cycles. |
| **Good for Predictable Updates:** Works well if you know the data changes precisely on a schedule.                             | **Not True Real-Time:** Updates are delayed by up to the length of the polling interval.                                               |

---

### Common Use Cases

- **File Upload/Processing Progress:** Checking if a large video transcoding job or PDF generation is finished.
- **Order Tracking:** Checking the status of a food delivery order every 10 seconds.
- **Simple Dashboards:** Refreshing a view that doesn't require instantaneous, sub-second updates.

> **When to avoid:** If you need instant, bidirectional, high-frequency updates (like a chat app or a multiplayer game), look
> into **WebSockets**, **Server-Sent Events (SSE)**, or **Long Polling** instead.
