Node.js natively supports multiple versions of the HTTP protocol out of the box through its built-in modules.

Here is the breakdown of the supported versions:

### 1. HTTP/1.0 and HTTP/1.1 (Fully Supported)

- **Module:** `import http from 'node.js/http'` (and `https` for SSL/TLS)
- **Status:** This is the default, most widely used implementation in Node.js. It handles the classic request-response model,
  persistent connections (keep-alive), chunked transfer encoding, and pipeline requests.

### 2. HTTP/2 (Fully Supported)

- **Module:** `import http2 from 'node.js/http2'`
- **Status:** Fully stable and supported.
- **Details:** Node.js includes a native HTTP/2 engine (built on top of the `nghttp2` library). It natively supports advanced
  HTTP/2 features like **multiplexing** (sending multiple requests over a single TCP connection), header compression (HPACK),
  and server push. It also includes an HTTP/1.1 fallback mechanism (`allowHTTP1: true`) for clients that do not support
  HTTP/2.

### 3. HTTP/3 (Not Natively Supported Yet)

- **Status:** **In Progress / External Libraries Only**
- **Details:** While HTTP/3 (which runs over QUIC instead of TCP) is highly adopted by modern browsers, it is **not yet
  included** in the standard library of Node.js. An experimental QUIC/HTTP/3 implementation was worked on in the core
  codebase, but due to shifting dependencies, native support is still in development. If you need HTTP/3 in Node.js, you
  currently have to use third-party userland modules or place Node.js behind a reverse proxy like **Nginx** or **Cloudflare**
  that handles the HTTP/3 handshake for you.

Express.js works out of the box with **HTTP/1.0** and **HTTP/1.1**.

Because Express relies heavily on Node’s traditional internal `http` module, it does not support newer versions of HTTP
cleanly or natively.

---

### The Breakdown

#### ❌ HTTP/2 (Not Natively Supported)

Even with Express 5.x, Express **does not natively support HTTP/2**.

- **The Reason:** Node's native `http2` core module uses a completely separate architecture based on "Streams" rather than
  standard request/response socket emitters. Express's internal structures (`req` and `res` decorators) are built strictly
  around the HTTP/1 pipeline.
- **The Workaround:** If you want HTTP/2 directly inside your Node app, you either have to use a wrapper library like `spdy`
  (which is largely deprecated) or manually bridge Node's `http2` module to handle Express as a listener function.

#### ❌ HTTP/3 (No Support)

Since Node.js core doesn't natively bundle HTTP/3 yet, Express cannot support it either.

---

### How the Industry Handles This (The "Reverse Proxy" Pattern)

In modern production environments, developers rarely force Express to handle HTTP/2 or HTTP/3 directly. Instead, they use a
**Reverse Proxy** or a **Content Delivery Network (CDN)**.

```text
[ Browser ] --( HTTP/2 or HTTP/3 )--> [ Nginx / Cloudflare ] --( HTTP/1.1 )--> [ Express App ]

```

1. **The Edge Layer:** A reverse proxy like **Nginx**, **HAProxy**, or a managed service like **Cloudflare** sits in front of
   your application. It handles the cutting-edge, secure HTTP/2 or HTTP/3 handshakes with the client's browser.
2. **The Internal Layer:** The proxy strips away that layer and forwards the request to your Express backend locally using
   plain, ultra-fast **HTTP/1.1**.

This gives you the best of both worlds: your users get the lightning-fast multiplexing benefits of modern HTTP, while your
backend code remains simple, stable, and highly compatible with standard Express middleware.
