In backend architecture, the choice between a **Stateful** and a **Stateless** design determines how your servers store,
remember, and manage data about a user across multiple requests.

The fundamental difference lies in **where the memory lives**.

---

### 1. Stateless Backend (Memory lives with the Client)

In a stateless architecture, the server is completely forgetful. It does not store any information about past requests. Every
single HTTP request from a client is a "blank slate" and must contain all the information, credentials, and context necessary
for the server to understand and process it.

- **How it works:** Think of a fast-food drive-thru. If you want a burger, fries, and a drink, you have to order them. If you
  pull up to the window 5 minutes later and ask for "another one," the employee has no idea who you are or what you ordered
  previously. You must state your entire order again.
- **Implementation:** Usually driven by **JWTs (JSON Web Tokens)** or tokens stored in the browser. Every time the frontend
  hits an API, it attaches the JWT in the header. The server decodes it, verifies who you are, processes the request, and
  immediately forgets you again.

#### Pros & Cons of Stateless

- **🚀 Infinite Scalability:** Because no server holds unique user data, you can spin up 100 identical servers. A user's
  first request can go to Server A, and their second can go to Server B without any issues. Load balancing is incredibly
  easy.
- **🔧 Highly Fault-Tolerant:** If a server crashes mid-session, it doesn't matter. The client just retries the request
  against a healthy server, sending its token along.
- **⚠️ Payload Overhead:** Because the client must send authentication and context data with _every single request_, it
  increases network bandwidth overhead.

---

### 2. Stateful Backend (Memory lives with the Server)

In a stateful architecture, the server actively remembers the client. It creates a dedicated session for the user in its
local memory or disk upon the first interaction and keeps track of their state across subsequent requests.

- **How it works:** Think of sitting down at a luxury restaurant with a dedicated waiter. The waiter remembers your name,
  what wine you are drinking, and that you have a peanut allergy. You don’t have to remind them of your allergies every time
  they bring a new course.
- **Implementation:** Usually driven by **Server-side Sessions**. When you log in, the server creates a session object in its
  memory and sends back a `session_id` cookie to your browser. On subsequent requests, your browser sends just that tiny ID,
  and the server looks up your file in its internal memory.

#### Pros & Cons of Stateful

- **🔒 Enhanced Security & Control:** The server has absolute control over the session data. If a developer needs to
  instantly revoke a user's access, they can delete the session from the server's memory immediately.
- **📉 Harder to Scale (The "Sticky Session" Problem):** If User A's session data is stored in Server 1's RAM, User A _must_
  always be routed to Server 1. If a load balancer accidentally routes them to Server 2, Server 2 will say "I don't know who
  you are," forcing the user to log in again.
- **💥 Single Point of Failure:** If Server 1 crashes or goes down for maintenance, all user sessions stored in its RAM are
  completely wiped out, forcing everyone on that server to log out.

---

### Summary Comparison

| Feature                    | Stateless Backend                                 | Stateful Backend                                        |
| -------------------------- | ------------------------------------------------- | ------------------------------------------------------- |
| **Where is state stored?** | Client-side (JWT / Tokens)                        | Server-side (RAM / Disk / Session)                      |
| **Server Scaling**         | Extremely easy (Horizontal scaling)               | Difficult (Requires sticky sessions or external caches) |
| **Request Overhead**       | Higher (Must send full token/data every time)     | Lower (Only sends a tiny Session ID)                    |
| **Session Revocation**     | Difficult (Tokens are valid until they expire)    | Instant (Server can delete the session instantly)       |
| **Common Use Cases**       | REST APIs, Microservices, Serverless (AWS Lambda) | Traditional Monoliths, Real-time MMO Games, Live Chat   |

### The Modern Compromise: Hybrid Stateless

To get the best of both worlds, modern architectures often deploy **Stateless Application Servers** backed by a centralized,
blazing-fast in-memory database like **Redis**.

The app servers themselves remain completely stateless (they don't store anything in their own RAM), but they use the
incoming Session ID to instantly grab the user's state from the shared Redis layer. This keeps the application servers
perfectly scalable while retaining the benefits of a stateful user experience.
