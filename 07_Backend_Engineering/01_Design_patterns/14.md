Here is a clean example of the **Server-Sent Events (SSE)** design pattern using TypeScript and Express.

In this setup, the client opens a single connection to the server. The server keeps this connection open and pushes data
(like live stock prices) to the client every few seconds without the client ever needing to ask again.

### 1. The Server (`server.ts`)

To implement SSE in Express, we must manually set specific HTTP headers to tell the browser to expect a continuous stream,
and we must format our text payload exactly as `data: <message>\n\n`.

```typescript
import express, { Request, Response } from "express";

const app = express();
const PORT = 3000;

// Endpoint that serves the real-time event stream
app.get("/api/live-updates", (req: Request, res: Response) => {
  // 1. Set SSE-specific HTTP headers
  res.writeHead(200, {
    "Content-Type": "text/event-stream",
    "Cache-Control": "no-cache",
    Connection: "keep-alive",
  });

  console.log("Client connected to SSE stream.");

  // 2. Start pushing data to the client at an interval
  const intervalId = setInterval(() => {
    const stockUpdate = {
      ticker: "AAPL",
      price: (150 + Math.random() * 10).toFixed(2),
      timestamp: new Date().toISOString(),
    };

    // CRITICAL: SSE format requires "data: <string>\n\n" to trigger the client event
    res.write(`data: ${JSON.stringify(stockUpdate)}\n\n`);
  }, 3000); // Send updates every 3 seconds

  // 3. Clean up if the client closes the browser tab or disconnects
  req.on("close", () => {
    console.log("Client disconnected from SSE stream.");
    clearInterval(intervalId);
    res.end();
  });
});

app.listen(PORT, () => {
  console.log(`SSE Server running on http://localhost:${PORT}`);
});
```

### 2. The Frontend / Client (`index.html`)

On the client-side, modern web browsers have native support for SSE via the `EventSource` API. You don't need external
libraries like Axios for this.

```html
<!DOCTYPE html>
<html lang="en">
  <head>
    <meta charset="UTF-8" />
    <title>SSE Live Dashboard</title>
  </head>
  <body>
    <h1>Live Stock Ticker</h1>
    <div id="ticker-box">Waiting for updates...</div>

    <script>
      // 1. Establish connection to the SSE endpoint
      const eventSource = new EventSource("http://localhost:3000/api/live-updates");

      // 2. Listen for incoming generic messages from the server
      eventSource.onmessage = (event) => {
        const rawData = JSON.parse(event.data);

        const targetDiv = document.getElementById("ticker-box");
        targetDiv.innerHTML = `
                <p><strong>Ticker:</strong> ${rawData.ticker}</p>
                <p><strong>Price:</strong> $${rawData.price}</p>
                <p><strong>Last Updated:</strong> ${rawData.timestamp}</p>
                <hr/>
            `;
        console.log("Received update:", rawData);
      };

      // 3. Handle errors or connection drops (EventSource auto-reconnects natively!)
      eventSource.onerror = (error) => {
        console.error("SSE Connection lost. Browser is attempting auto-reconnect...", error);
      };
    </script>
  </body>
</html>
```

---

### Key Takeaways of This Code

- **No Infinite Loops:** Unlike long polling, the client (`EventSource`) calls the server **once**. The connection stays
  open.
- **`res.write` instead of `res.json`:** Standard Express apps use `res.json()` or `res.send()`, which terminates the
  request. SSE uses `res.write()` to continuously pour data down the open pipe.
- **The Magic Double Newline (`\n\n`):** The browser uses `\n\n` as the delimiter to know that a specific packet of data is
  complete and ready to be parsed. Missing one `\n` will cause the client to sit waiting indefinitely.
