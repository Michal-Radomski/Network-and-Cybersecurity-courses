For a live public bus tracking map that updates every 10 seconds, **Short Polling** is actually the most practical, reliable,
and cost-effective design pattern you can use.

While real-time technologies like WebSockets or Server-Sent Events (SSE) sound flashy, they are usually overkill for this
specific use case. Here is a breakdown of why short polling wins here, along with how to structure it.

---

### Why Short Polling is the Best Choice

1. **Perfect Interval Alignment:** Your data source (the transit authority's API) likely updates its bus GPS coordinates on a
   fixed schedule (usually every 10 to 30 seconds). If the underlying data only changes every 10 seconds, keeping a constant
   WebSocket connection open is a waste of server resources.
2. **Stateless Scalability:** If your app becomes popular and thousands of users open the map, a WebSocket architecture
   forces your servers to hold thousands of open TCP connections in memory. With Short Polling, requests hit the server, get
   the coordinates, and instantly close. This makes it incredibly easy to scale your backend horizontally or cache the
   responses using a CDN (like Cloudflare) or Redis.
3. **Simplicity:** It requires no complex connection-drop lifecycle management on the client or server. It's just a standard
   `GET` request repeating on a timer.

---

### How to Structure the Backend Architecture

To ensure your backend doesn't melt when thousands of users poll every 10 seconds, you should implement a **Cached Short
Polling** pattern.

Do **not** let individual user requests trigger a direct fetch to the third-party transit API or your main database. That
will cause your app to get rate-limited or crash. Instead, decouple the data fetching from the user polling:

```text
[Transit API] --(Every 10s)--> [Background Worker] --> [Redis Cache] <-- (Every 10s Poll) -- [Multiple Users]

```

1. **The Background Worker:** Run a single, isolated cron job or background worker process that fetches the fresh bus
   coordinates from the Transit Authority API exactly once every 10 seconds.
2. **The Cache:** The worker processes the coordinates and overwrites a single key in a fast, in-memory database like
   **Redis** (or a local in-memory variable if your backend is a single monolith).
3. **The Express API Endpoint:** When your users' apps poll your backend every 10 seconds, your Express route simply reads
   the data straight out of Redis and hands it to them.

This ensures that whether you have 1 user or 10,000 users looking at the map, your backend only ever makes _one_ request to
the underlying data source every 10 seconds.

---

### A Quick Client-Side Implementation Note

When implementing the 10-second timer on your map frontend, avoid using a naive `setInterval`. If a user has a laggy cellular
connection, `setInterval` will keep firing new requests even if the previous ones are still loading, stacking up network
traffic.

Instead, use **Recursive `setTimeout**`:

```javascript
async function pollBusLocations() {
  try {
    const response = await fetch("/api/buses");
    const busData = await response.json();

    // Update your map markers here
    updateMapMarkers(busData);
  } catch (error) {
    console.error("Failed to fetch bus data", error);
  } finally {
    // Wait 10 seconds AFTER the request completes before forcing the next one
    setTimeout(pollBusLocations, 10000);
  }
}

// Start the loop when the map component mounts
pollBusLocations();
```
