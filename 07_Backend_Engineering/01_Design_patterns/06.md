If you are building high-performance network servers or storage applications in Linux, you eventually run into a massive
bottleneck: **I/O waiting times**.

Both `epoll` and `io_uring` are Linux kernel subsystems designed to handle I/O efficiently, but they represent two different
generations of architecture. `epoll` is the reliable, battle-tested veteran, while `io_uring` is the modern, ultra-fast
newcomer.

Here is a breakdown of how they work, how they differ, and why `io_uring` is changing the game.

---

## 1. What is epoll? (The Event Notifier)

Introduced in Linux 2.6, `epoll` (Event Poll) is an **asynchronous event notification** mechanism. It is primarily used for
network sockets.

Instead of having your program constantly check 10,000 network connections to see if any data has arrived (which destroys CPU
performance), you hand those connections over to `epoll`.

### How epoll Works:

1. You register a list of file descriptors (like network sockets) with the kernel.
2. Your application calls `epoll_wait()`, which puts your program to sleep.
3. When data arrives on a socket, the kernel wakes up your application and says, _"Hey, socket #42 is ready to be read!"_
4. Your application then issues a standard `read()` or `recv()` system call to actually get the data.

> **The Catch:** `epoll` doesn't actually do the I/O. It just tells you _when_ you can do the I/O without blocking.
> Furthermore, it does not work well with regular disk files (disk I/O always blocks in `epoll`).

---

## 2. What is io_uring? (The True Asynchronous Engine)

Introduced by Jens Axboe in Linux 5.1 (2019), `io_uring` is a **true asynchronous I/O interface**. It was designed to solve
the performance bottlenecks of `epoll` and older async systems (like `AIO`).

Instead of telling you when a socket is _ready_, `io_uring` allows you to say: _"Go read from this socket, put the data into
this buffer, and let me know when you are completely finished."_

### How io_uring Works:

It uses two lockless **ring buffers** (circular queues) shared directly between the Linux kernel and user space:

1. **Submission Queue (SQ):** Your application writes I/O requests (read, write, accept, open, etc.) into this queue.
2. **Completion Queue (CQ):** The kernel processes the requests in the background and pushes the results (success, bytes
   read, errors) into this queue for your application to consume.

Because these queues are in shared memory, you can submit and reap I/O requests **without making expensive system calls**.

---

## Key Differences: epoll vs. io_uring

| Feature                  | `epoll`                                                                                | `io_uring`                                                                      |
| ------------------------ | -------------------------------------------------------------------------------------- | ------------------------------------------------------------------------------- |
| **Primary Model**        | **Readiness notification** (Tells you when you _can_ read/write).                      | **Asynchronous execution** (Does the read/write for you).                       |
| **System Call Overhead** | High. Requires system calls to wait (`epoll_wait`) and to do the I/O (`read`/`write`). | Extremely low. Can even run in "kernel polling mode" for **zero** system calls. |
| **Supported I/O Types**  | Sockets, pipes, ttys. (Terrible for disk files).                                       | Everything. Network sockets, disk files, timers, and even `epoll` itself.       |
| **Data Copying**         | Requires copying data across the user/kernel boundary during the read/write phase.     | Can utilize registered buffers to minimize or eliminate data copying.           |
| **Complexity**           | Relatively simple, industry-standard (used by Nginx, Node.js, Redis).                  | Higher complexity; requires careful memory management of the queues.            |

---

## Why io_uring is a Massive Upgrade

To understand why `io_uring` is faster, look at the **System Call tax**.

Every time your program makes a traditional system call (like `read()`), the CPU has to switch contexts from "User Mode" to
"Kernel Mode" and back. Since security mitigations for vulnerabilities like Spectre and Meltdown were introduced, these
context switches became significantly more expensive.

- With `epoll`, handling 1,000 packets might require hundreds of context switches.
- With `io_uring`, you can batch 1,000 requests into the Submission Queue, submit them all at once (or let the kernel poll
  them automatically), and handle them with **zero context switches**.

### Summary: Which should you use?

- **Use `epoll**` if you are maintaining legacy software, need maximum portability across older Linux kernels, or are writing
  basic network code where "good enough" performance suffices.
- **Use `io_uring**` if you are building cutting-edge, high-throughput applications (like modern databases, web servers, or
  storage engines) where you need to squeeze every ounce of IOPS out of fast NVMe drives and high-speed network cards.
