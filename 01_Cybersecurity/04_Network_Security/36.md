Think of a **hypervisor** as the "traffic cop" or the "OS for operating systems."

In a traditional setup, you have one physical computer running one Operating System (OS), like Windows. A hypervisor is a
thin layer of software that sits between the physical hardware and the operating systems, allowing you to run multiple,
independent OSs (called **Virtual Machines** or VMs) on that single piece of hardware.

The hypervisor’s job is to carve up the physical resources—like CPU, RAM, and storage—and hand them out to each VM so they
don't crash into each other.

---

### The Two Types of Hypervisors

Hypervisors are generally categorized into two types based on where they "sit" in the system.

#### Type 1: Bare Metal (Native)

A Type 1 hypervisor is installed **directly** on the physical hardware. There is no Windows or macOS underneath it. It is
extremely fast and secure because it has a direct line to the hardware.

- **Best for:** Data centers, enterprise servers, and cloud computing.
- **Examples:** VMware ESXi, Microsoft Hyper-V, and Xen.

#### Type 2: Hosted

A Type 2 hypervisor runs as an **application** on top of a normal operating system (like Windows or Linux). It’s easy to set
up, but it's slightly slower because it has to go through the "Host" OS to reach the hardware.

- **Best for:** Software developers, security researchers, and students.
- **Examples:** Oracle VirtualBox, VMware Workstation, and Parallels Desktop.

---

### Why do we need them?

1.  **Efficiency:** Instead of having 10 physical servers running at 10% capacity, you can have one powerful server running
    10 virtual ones at 90% capacity. This saves massive amounts of electricity and space.
2.  **Isolation:** If one VM catches a virus or crashes, the others (and the physical hardware) remain completely unaffected.
3.  **Portability:** Since a VM is just a set of files managed by the hypervisor, you can move an entire server from a laptop
    in New York to a data center in London in minutes.
4.  **Snapshots:** You can save the "state" of a machine. If you're about to test a risky setting, you take a snapshot; if
    the system breaks, you click a button to travel back in time to when it worked.

### Key Terminology

- **Host:** The physical machine providing the resources.
- **Guest:** The Virtual Machine (VM) running on the hypervisor.
- **VMM:** Another name for a hypervisor (Virtual Machine Monitor).
