**TLS (Transport Layer Security)** is the cryptographic protocol that keeps the internet secure. It’s the "S" in **HTTPS**
and is responsible for encrypting data sent between a web browser and a server, ensuring that hackers can't eavesdrop on or
tamper with your information.

Think of it as a secure, private tunnel built over an otherwise public highway.

---

## The Two Core Layers of TLS

The TLS protocol operates in two main phases:

1. **The TLS Handshake Protocol:** This happens first. The client (your browser) and the server introduce themselves, verify
   each other's identities, agree on the encryption algorithms (cipher suites), and generate secret keys.
2. **The TLS Record Protocol:** Once the handshake is done, this layer takes over. It uses the secret keys generated during
   the handshake to encrypt and compress the actual data being sent back and forth.

---

## The TLS 1.3 Handshake Process (Step-by-Step)

TLS 1.3 is the modern standard (released in 2018). It is much faster and more secure than older versions (like TLS 1.2),
reducing the handshake from two "round trips" to just **one round trip (1-RTT)**.

Here is exactly how a modern TLS 1.3 handshake works:

### 1. Client Hello

The client starts the conversation by sending a **Client Hello** message to the server. This message includes:

- The TLS version it supports (e.g., TLS 1.3).
- A list of supported **cipher suites** (encryption algorithms).
- A random string of bytes (Client Random).
- **Key Share:** The client guesses which encryption algorithm the server will prefer and preemptively sends its
  cryptographic key share.

### 2. Server Hello & Cryptographic Choices

The server receives the message and replies with the **Server Hello**. In this step, the server:

- Chooses the highest mutually supported TLS version and cipher suite.
- Sends its own random string of bytes (Server Random).
- Sends its own **Key Share**. By combining the client's key share and the server's key share, _both sides can now calculate
  the session key_ without ever sending the actual key over the network (using Diffie-Hellman key exchange).

### 3. Server Authentication & Finished

Immediately after the Server Hello, the server sends a few more bundled messages (all encrypted using the new key they just
calculated):

- **Certificate:** The server sends its SSL/TLS certificate (proving it actually owns the website domain).
- **Certificate Verify:** A digital signature proving the server owns the private key associated with that certificate.
- **Finished:** A message stating it has completed its part of the handshake.

### 4. Client Verification & Finished

The client does some quick homework:

- It checks the server’s certificate against trusted Certificate Authorities (CAs) built into the browser to make sure the
  site isn't a fake.
- If the certificate is valid, the client sends its own **Finished** message, encrypted with the session key.

### 5. Secure Session Established

The handshake is complete! Both parties now use the symmetric **session keys** to encrypt and decrypt all data (HTML, images,
API requests) for the rest of the browsing session.

---

## Why is TLS so important?

- **Encryption:** It hides the data being transferred from eavesdroppers (like someone watching your traffic on public
  Wi-Fi).
- **Authentication:** It ensures that you are actually connecting to the real website (e.g., `google.com`) and not a
  scammer's copy.
- **Integrity:** It uses a Message Authentication Code (MAC) to ensure that data isn't altered or corrupted while traveling
  across the web.
