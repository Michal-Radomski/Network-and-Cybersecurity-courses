There is actually a slight misunderstanding about how `bcrypt` works under the hood!

Despite using the same line of code for every user (`const salt = await bcrypt.genSalt(10);`), **the salt generated will be
completely different every single time it is run.** Here is exactly what that code is doing:

---

## 1. The Number `10` is the "Work Factor" (Not a fixed ID)

The number you pass into `genSalt(10)` is the **Work Factor** (or rounds). It tells the CPU how many times to loop the
hashing math.

- It grows exponentially: a work factor of `10` means the algorithm runs $2^{10}$ (1,024) rounds to calculate the salt and
  hash.
- This is designed to intentionally slow down computers just enough (a fraction of a second) so that hackers cannot guess
  millions of passwords per second if they steal your database.

---

## 2. A New Random Salt is Generated Every Time

Every time a new user registers and your code calls `bcrypt.genSalt(10)`, the library generates a brand-new,
cryptographically secure **random 16-byte string** behind the scenes.

If you were to `console.log(salt)` for three different users, you would see three completely unique strings, like this:

```text
User 1 Salt: $2b$10$Eixomg2M7...
User 2 Salt: $2b$10$9pRhK1WvX...
User 3 Salt: $2b$10$Z3yUaBcDe...

```

_(Notice the `$10$` in the strings above? That's bcrypt automatically saving your "Work Factor" directly into the salt string
so it knows how many rounds to use later)._

---

## You Don't Even Need `genSalt` Separately

Because `bcrypt` is designed to be as foolproof as possible, you don't actually need to generate the salt in a separate line
of code. You can pass the work factor directly into the `hash` function, and it will generate the random salt and hash the
password all in one go:

```javascript
// This does both steps automatically!
const hashedPassword = await bcrypt.hash(password, 10);
```

When you look at the final `hashedPassword` saved in your database, bcrypt automatically glues the unique salt and the final
hash together into one string. When a user logs in, `bcrypt.compare()` reads that string, extracts the salt, hashes the
incoming password with it, and checks for a match.

---

---

When you look at a password hashed by `bcrypt` in your database, it looks like a single, chaotic string of text. However, it
is actually a highly structured string divided by dollar signs (`$`).

`bcrypt` knows exactly where the salt ends and the hash begins because the string is split into strict, fixed-length
sections.

---

## Anatomy of a Bcrypt Hash

Here is what a typical bcrypt string looks like:

```text
$2b$10$Eixomg2M7174kbGAf6b57u7T3/49U4M77mS

```

If we break this down into its four distinct components, you can see exactly how `bcrypt` reads it:

| Section                       | Content      | Meaning to Bcrypt                                                     |
| ----------------------------- | ------------ | --------------------------------------------------------------------- |
| **`$2b`**                     | Identifier   | Tells the library that the `bcrypt` algorithm version "2b" was used.  |
| **`$10`**                     | Cost Factor  | Tells the library it needs to run $2^{10}$ (1,024) rounds of hashing. |
| **`$Eixomg2M7174kbGAf6b57u`** | **The Salt** | The next **exactly 22 characters** are the unique random salt.        |
| **`7T3/49U4M77mS...`**        | **The Hash** | Everything after the 22-character salt is the actual hashed password. |

---

## How `bcrypt.compare()` Works Under the Hood

When a user tries to log in, you pass two things to `bcrypt.compare(plainPassword, hashedPasswordFromDB)`:

1. The plaintext password they just typed into the login form.
2. The full structured hash string from your database.

Because the structure is always identical, `bcrypt` performs these steps automatically:

1. **Parses the string:** It looks at the database string, skips the version and cost factor, and grabs the next **exactly 22
   characters** to extract the original salt.
2. **Re-hashes:** It takes the plaintext password the user just typed, combines it with that extracted salt, and runs it
   through the exact same number of rounds (e.g., 10).
3. **Matches:** It checks if the resulting hash matches the remainder of the string from the database. If they match
   perfectly, the login is successful!
