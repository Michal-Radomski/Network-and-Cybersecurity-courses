In the context of digital certificates, **V3 extensions** refer to the additional data fields introduced in **Version 3 of
the X.509 certificate standard**.

When the X.509 standard was first created (Versions 1 and 2), certificates only contained basic information: who issued it,
who owns it, the public key, and the expiration date. As the internet grew, those basic fields weren't enough. Version 3
introduced **Extensions** to allow organizations to add custom data, specify strict security rules, and expand what a
certificate can do.

Without V3 extensions, the modern secure internet could not function. Here are the most common and important ones used today:

---

## 1. Subject Alternative Name (SAN)

This is by far the most widely used V3 extension on the web.

- **What it does:** It allows a single digital certificate to protect multiple different domain names, subdomains, or IP
  addresses.
- **Why it matters:** In the old days, a certificate could only protect the exact name in the "Common Name" field (e.g.,
  `example.com`). If you wanted to secure `www.example.com`, `api.example.com`, and `example.net`, you had to buy three
  separate certificates. The SAN extension lets you bundle them all into one.

---

## 2. Basic Constraints

This extension is a vital security gatekeeper for the Chain of Trust.

- **What it does:** It specifies whether the certificate belongs to a **Certificate Authority (CA)** (which is allowed to
  sign and issue other certificates) or an **End-Entity** (a final website or user certificate, which cannot issue other
  certificates).
- **Why it matters:** It prevents a massive security loophole. If you buy a valid certificate for your personal blog
  (`myblog.com`), the `Basic Constraints` extension will say `IsCA: FALSE`. This tells web browsers that if your blog's
  certificate tries to sign a fake certificate for `google.com`, the browser should instantly reject it.

---

## 3. Key Usage & Extended Key Usage (EKU)

These extensions define the exact cryptographic purposes for which the certificate's public key can legally be used.

- **Key Usage:** Defines core cryptographic actions. For example, it might state the key can only be used for
  `Digital Signature` (proving identity) or `Key Encipherment` (encrypting data).
- **Extended Key Usage (EKU):** Defines specific application scenarios. Common settings include:
- `Server Authentication` (the certificate is for a web server/website).
- `Client Authentication` (the certificate is for a user logging into a network).
- `Code Signing` (the certificate is used by a software developer to sign an `.exe` or app installer so operating systems
  know it doesn't contain malware).

---

## 4. Authority Information Access (AIA)

This extension helps applications verify the certificate's validity in real-time.

- **What it does:** It provides URLs pointing to where the issuing CA's actual certificate can be downloaded, and more
  importantly, it provides the URL for the CA's **OCSP (Online Certificate Status Protocol) responder**.
- **Why it matters:** When your browser connects to a site, it reads the AIA extension to find out exactly where to send a
  quick ping to ask, _"Is this specific certificate still valid, or has it been stolen and revoked?"_

---

## Critical vs. Non-Critical Extensions

Every V3 extension is marked with a flag indicating whether it is **Critical** or **Non-Critical**.

- **Critical:** If an extension is marked as critical (like _Basic Constraints_), an application reading the certificate
  **must** understand and enforce that extension. If the application doesn't know what that extension means, it is forced to
  reject the entire certificate out of safety.
- **Non-Critical:** If an application doesn't recognize a non-critical extension, it can safely ignore it and continue
  processing the rest of the certificate.
