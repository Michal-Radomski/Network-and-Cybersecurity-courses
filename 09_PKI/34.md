**TLS 1.3 (Transport Layer Security 1.3)** is the modern standard protocol used to secure communications over the internet.
Compared to its predecessor (TLS 1.2), TLS 1.3 drastically simplified its protocol architecture to achieve two main goals:
**faster connection speeds** and **bulletproof security**.

The architecture of TLS 1.3 is layered directly on top of the reliable Transport Layer (TCP) and is fundamentally split into
two main operational layers: the **Handshake Protocols** (the management layer) and the **Record Protocol** (the data carrier
layer).

---

## 1. The High-Level Architecture

The TLS 1.3 architecture is divided cleanly into control-plane protocols and a data-plane protocol.

```text

|                           Handshake Protocols                         |
|  +--------------------+  +--------------------+  +-----------------+  |
|  | Handshake Protocol |  | Alert Protocol     |  | Heartbeat /     |  |
```

## 2. The Core Components

### A. The TLS Record Protocol (The Base Layer)

The Record Protocol is the workhorse of TLS. It handles the formatting, packaging, and cryptographic protection of all data
passing through the tunnel.

It takes higher-level messages (whether they are handshake negotiations or actual website data like HTTP traffic), processes
them, and hands them down to TCP. Its responsibilities include:

- **Fragmentation:** Slicing large chunks of data into manageable records (maximum $2^{14}$ bytes or 16 KB).
- **AEAD Only Encryption:** Unlike TLS 1.2, which allowed older, vulnerable encryption modes, TLS 1.3 _only_ allows
  **Authenticated Encryption with Associated Data (AEAD)** ciphers (such as AES-GCM or ChaCha20-Poly1305). This guarantees
  both data privacy and data integrity at the same time.
- **Type Hiding:** In TLS 1.3, the true payload type (whether a packet is a handshake, alert, or application data) is
  encrypted _inside_ the record payload. To network eavesdroppers, every record looks identically encrypted, preventing
  traffic analysis attacks.

### B. The Handshake Protocol (The Management Layer)

Before any secure data can flow, the client and server must agree on cryptographic keys. The TLS 1.3 handshake architecture
was completely redesigned to execute in just **one round-trip (1-RTT)**, down from the two round-trips (2-RTT) required in
TLS 1.2.

The architecture achieves this speed via an optimistic **"Guess-and-Share"** design:

1. **Client Hello + Key Share:** The client assumes the server supports modern elliptic curves. It immediately sends its
   identity parameters along with its cryptographic **Key Share** (e.g., its public key calculated using a curve like P-256
   or X25519) right in the very first packet.
2. **Server Hello + Key Share:** The server receives the client's guess. If it agrees, it replies with _its_ Key Share, its
   digital certificate to prove its identity, and a finished confirmation signature.
3. **Instant Encryption:** Because both sides now have each other's key shares, they can mathematically calculate the shared
   secret. From this point onward, all traffic—including the trailing parts of the handshake itself—is encrypted.

#### Zero Round-Trip Time (0-RTT) Mode

The TLS 1.3 architecture also introduces a specialized sub-protocol for resuming prior connections. If a client has connected
to a server recently, it can save a **Pre-Shared Key (PSK)** token. When reconnecting, the client can bundle encrypted
application data directly inside the _Client Hello_ packet, resulting in a **0-RTT** connection delay.

### C. The Alert Protocol

This architectural component handles exceptional events. If something goes wrong—such as a corrupted packet, an expired
certificate, or a sudden connection termination—the Alert Protocol transmits a message indicating the severity (Warning or
Fatal) and a specific error code, instantly shutting down the session cleanly if a Fatal alert is triggered.

## 3. Major Architectural Differences: TLS 1.2 vs. TLS 1.3

To fully understand the TLS 1.3 architecture, it helps to see what was intentionally removed to fix past structural flaws:

| Architectural Element       | TLS 1.2                                                                               | TLS 1.3                                                                               |
| --------------------------- | ------------------------------------------------------------------------------------- | ------------------------------------------------------------------------------------- |
| **Handshake Latency**       | 2 Round-Trips (2-RTT)                                                                 | 1 Round-Trip (1-RTT)                                                                  |
| **Cipher Suite Structure**  | Bulky. Combines key exchange, signature, and encryption types.                        | Minimalist. Contains **only** the AEAD encryption and hashing algorithm.              |
| **Static RSA Key Exchange** | Allowed (Vulnerable to retroactively breaking data if a private key is stolen later). | **Removed.** Perfect Forward Secrecy (PFS) via Ephemeral Diffie-Hellman is mandatory. |
| **Handshake Encryption**    | Handshake data is sent in clear text.                                                 | Handshake data is encrypted as early as possible.                                     |
| **Insecure Cryptography**   | Supported MD5, SHA-1, RC4, 3DES, and CBC ciphers.                                     | Completely pruned. All weak cryptographic primitives are banned.                      |
