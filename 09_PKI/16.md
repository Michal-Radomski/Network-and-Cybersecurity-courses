The **TLS 1.3 Handshake** is the modern protocol used to establish a secure, encrypted connection between a client (like your
web browser) and a server.

Compared to older versions like TLS 1.2, TLS 1.3 was redesigned from scratch to be **faster** and **more secure**. The
biggest change is speed: while TLS 1.2 required **two round trips** (2-RTT) of communication to set up encryption, TLS 1.3
cuts this down to just **a single round trip** (1-RTT).

---

## Step-by-Step: The 1-RTT Handshake

Because TLS 1.3 dropped support for legacy, insecure cryptographic algorithms, the browser can confidently guess what
parameters the server will want to use. This allows the handshake to happen in just one exchange:

### 1. Client Hello (The Guess)

The browser starts the connection by sending a message containing:

- The TLS version (1.3).
- A list of supported modern **Cipher Suites**.
- **The "Key Share":** The client doesn't just say what algorithms it supports; it goes ahead and generates its half of the
  cryptographic key exchange (using Elliptic Curve Diffie-Hellman) right away, assuming the server will agree.

### 2. Server Hello & Keys (The Agreement)

The server receives the request, selects a matching cipher suite from the list, and responds with:

- Its chosen cipher suite.
- **The Server's "Key Share":** Its own half of the cryptographic key exchange.
- **The Encrypted Extensions:** Extra parameters about the session.

> ⚡ **The Secret Is Born:** At this exact microsecond, both the client and server combine their respective key shares. Using
> the Diffie-Hellman math, they calculate a shared secret "Session Key." **Everything sent after this point in the handshake
> is fully encrypted.**

### 3. Authentication & Finished

Still within that same response packet, the server sends:

- **The Website Certificate:** To prove its identity.
- **Certificate Verify:** A digital signature proving the server owns the private key associated with that certificate.
- **Finished:** A message confirming its part of the handshake is done.

### 4. Client Finished

The browser verifies the server's certificate and signature. If everything looks good, it sends its own **Finished** message.

The secure channel is now wide open, and regular application data (like the website's HTML) can begin flowing immediately.

---

## The Ultimate Speed Feature: 0-RTT (Resumption)

If you have visited a website recently, TLS 1.3 allows an even faster reconnection method called **0-RTT (Zero Round Trip
Time)**.

- **How it works:** During your first visit, the server gives your browser a special "ticket."
- **The next visit:** When you reconnect, your browser sends that ticket along with your encrypted web request (like
  `GET /index.html`) in the **very first packet**.
- **The benefit:** The server decrypts the ticket, verifies it, and sends the website data back immediately. There is zero
  delay waiting for a handshake to finish.

> ⚠️ **Note:** While 0-RTT is incredibly fast, it is vulnerable to **Replay Attacks** (where a hacker intercepting your
> network copies that first packet and sends it again). Because of this, developers usually only use 0-RTT for safe requests
> that don't change data, like viewing a page, rather than submitting a payment form.

---

## Summary of Improvements

| Feature                | TLS 1.2                       | TLS 1.3                        |
| ---------------------- | ----------------------------- | ------------------------------ |
| **Handshake Speed**    | 2 Round Trips (2-RTT)         | 1 Round Trip (1-RTT)           |
| **Reconnection Speed** | 1-RTT                         | 0-RTT                          |
| **Cipher Suite Size**  | Long, complex names           | Short, simplified names        |
| **Privacy**            | Certificate sent in plaintext | Certificate is fully encrypted |
