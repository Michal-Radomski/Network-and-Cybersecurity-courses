In cryptography, **hashing**, **salting**, and **peppering** are techniques used to protect sensitive data—most commonly,
user passwords stored in a database.

Here is how each one works and why they are used together.

---

## 1. Hashing: The One-Way Blender

**Hashing** takes an input (like a password) and runs it through a mathematical algorithm to produce a fixed-length string of
characters, called a **hash**.

```text
"Password123"  --->  [ SHA-256 Algorithm ]  --->  ef92b778bafe421e...

```

### Key Characteristics:

- **One-Way:** Hashing is a one-way street. You can easily turn a password into a hash, but it is mathematically impossible
  to reverse the hash back into the original password.
- **Deterministic:** The same password will _always_ produce the exact same hash.
- **Deterministic Problem:** Because it is deterministic, if two users have the exact same password (e.g., `Password123`),
  they will have the exact same hash in the database. If a hacker steals the database, they can use pre-computed lists of
  common password hashes (called **Rainbow Tables**) to instantly crack them.

---

## 2. Salt: The Unique Seasoning

To fix the deterministic problem of hashing, we use a **salt**. A salt is a long, completely random string of characters
generated for _every single user_.

Before hashing the password, the system glues the salt to it.

```text
Password:  "Password123"
Salt:      "qX9!zL2p"

Combined:  "Password123qX9!zL2p"  --->  [ Hash Function ]  --->  Unique Hash

```

### How it helps:

- **Unique Hashes:** If two users have the password `Password123`, they get different random salts, resulting in completely
  different hashes in the database.
- **Database Storage:** The salt is stored **directly in the database** right next to the user's hash. When the user logs in
  later, the system grabs their salt, glues it to the typed password, hashes it, and checks if it matches.
- **Defense:** This completely destroys the effectiveness of pre-computed Rainbow Tables. A hacker has to crack each password
  one by one, which takes an immense amount of time.

---

## 3. Pepper: The Secret Ingredient

A **pepper** is very similar to a salt, but with one massive difference in how it is handled and stored.

While a salt is unique per user and stored openly in the database, a pepper is **secret, global, and stored outside the
database** (usually in the application's source code, an environment variable, or a secure hardware security module).

```text
Password:  "Password123"
Salt:      "qX9!zL2p"       (Stored in DB)
Pepper:    "SecretPepper"   (Stored in Code/Environment)

Combined:  "Password123qX9!zL2pSecretPepper"  --->  [ Hash Function ]  --->  Final Hash

```

### How it helps:

- **The "Database Leak" Defense:** Imagine a hacker manages to exploit a SQL injection vulnerability and downloads your
  entire database. They now have all the user hashes and all the salts.
- Because they _don't_ have the server's source code or environment variables, they don't have the **pepper**. Without it,
  they still cannot attempt to crack the hashes offline, because they are missing a crucial piece of the puzzle.

---

## Summary: The Final Recipe

| Concept    | What it is                               | Where it is stored                 | Main Purpose                                                          |
| ---------- | ---------------------------------------- | ---------------------------------- | --------------------------------------------------------------------- |
| **Hash**   | A one-way cryptographic fingerprint.     | Database                           | To avoid storing plaintext passwords.                                 |
| **Salt**   | A random string unique to each user.     | Database (next to hash)            | To make identical passwords look different and defeat Rainbow Tables. |
| **Pepper** | A secret string shared by all passwords. | Application Configuration (Secure) | To protect hashes even if the database is entirely stolen.            |
