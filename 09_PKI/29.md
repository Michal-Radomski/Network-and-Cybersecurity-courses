If you prefer to handle the process manually without letting Certbot automatically modify your Apache configuration files,
you can use Certbot's `certonly` mode. This fetches the certificates from Let's Encrypt and saves them to your server,
leaving you in complete control of your Apache virtual host files.

Here is the step-by-step manual process.

---

## Step 1: Obtain the Certificates (Without Installing)

Run Certbot with the `certonly` flag and use the `--webroot` plugin. This method proves you own the domain by placing a
temporary file in your website's public folder, which Let's Encrypt validates over the internet.

Run the following command (replace `/var/www/html` with your actual web root directory and `example.com` with your domain):

```bash
sudo certbot certonly --webroot -w /var/www/html -d example.com -d www.example.com

```

### What happens next:

- Certbot will ask for your email address and agreement to the terms of service (if you haven't done this before).
- It will communicate with Let's Encrypt, verify the files, and download the certificates.
- Upon success, your certificates will be stored in: `/etc/letsencrypt/live/example.com/`

Inside that directory, you will find four files. The two you need for Apache are:

1. `fullchain.pem` (Your public certificate combined with the Let's Encrypt intermediate certificate).
2. `privkey.pem` (Your secret private key).

---

## Step 2: Manually Configure Apache for HTTPS

Now that you have the files, you need to point Apache to them.

### 1. Enable the Apache SSL Module

Before Apache can use SSL certificates, you must enable its SSL module and the rewrite module (for forwarding traffic):

```bash
sudo a2enmod ssl
sudo a2enmod rewrite

```

### 2. Edit Your Virtual Host File

Open your Apache configuration file for your domain (usually located in `/etc/apache2/sites-available/`). For example:

```bash
sudo nano /etc/apache2/sites-available/example.com.conf

```

### 3. Add the Port 443 (HTTPS) Block

Inside this file, you need to create or update a `<VirtualHost *:443>` block to handle secure traffic. It should look like
this:

```apache
<VirtualHost *:443>
    ServerName example.com
    ServerAlias www.example.com
    DocumentRoot /var/www/html

    SSLEngine on
    SSLCertificateFile /etc/letsencrypt/live/example.com/fullchain.pem
    SSLCertificateKey /etc/letsencrypt/live/example.com/privkey.pem

    <Directory /var/www/html>
        Options Indexes FollowSymLinks
        AllowOverride All
        Require all granted
    </Directory>

    ErrorLog ${APACHE_LOG_DIR}/error.log
    CustomLog ${APACHE_LOG_DIR}/access.log combined
</VirtualHost>

```

### 4. (Optional) Add HTTP to HTTPS Redirect

To make sure users typing `http://example.com` are pushed to the secure version, add or update the port 80 block in the same
file:

```apache
<VirtualHost *:80>
    ServerName example.com
    ServerAlias www.example.com
    DocumentRoot /var/www/html

    RewriteEngine on
    RewriteCond %{SERVER_NAME} =www.example.com [OR]
    RewriteCond %{SERVER_NAME} =example.com
    RewriteRule ^ https://%{SERVER_NAME}%{REQUEST_URI} [END,NE,R=permanent]
</VirtualHost>

```

Save and close the file (`Ctrl+O`, `Enter`, then `Ctrl+X`).

---

## Step 3: Enable the Site and Restart Apache

If you created a new configuration file, ensure it is enabled in Apache:

```bash
sudo a2ensite example.com.conf

```

Test your Apache configuration for any syntax errors or typos:

```bash
sudo apache2ctl configtest

```

If it says `Syntax OK`, restart Apache to apply the new configuration and activate SSL:

```bash
sudo systemctl restart apache2

```

---

## Step 4: Setting up Automated Renewal

Even when you obtain certificates manually, the Certbot background service will still try to renew them automatically before
the 90 days are up. Because you used the `--webroot` plugin, Certbot remembers how it fetched the certificate and can renew
it seamlessly without human intervention.

Test the renewal process manually to ensure it won't fail down the road:

```bash
sudo certbot renew --dry-run

```

If the dry run succeeds, your manual configuration is complete, secure, and ready to run indefinitely.
