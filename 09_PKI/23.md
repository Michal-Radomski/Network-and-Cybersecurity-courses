When a Certificate Authority (CA) "signs" a public key, it doesn't actually scramble or encrypt the public key itself.
Instead, it takes the public key and identity data, hashes it, and then uses its private key to generate a unique digital
seal (the signature) that gets attached to the file.

Here is exactly how the cryptographic process works behind the scenes, broken down step-by-step.

---

## The Signing Process (Step-by-Step)

Imagine the CA is creating a digital certificate for a website. The website has sent over its information along with its own
public key.

### Step 1: Bundling the Data

The CA organizes all the plain-text information that needs to be signed into a standard format (the X.509 structure). This
bundle includes:

- The user/website's **Public Key**
- The user's identity (e.g., `www.example.com`)
- The expiration date
- The CA’s own identity

### Step 2: Hashing the Bundle

The CA runs this entire bundle of plain-text data through a cryptographic hash function (usually **SHA-256**).

- This mathematical function turns the entire dataset into a unique, fixed-length string of characters called a **Data Hash**
  (or thumbprint).
- _Why?_ It is much faster and more efficient to sign a short hash than a massive file.

### Step 3: Encrypting the Hash (The Actual Signing)

The CA takes that **Data Hash** and encrypts it using the **CA’s own Private Key**.

- The mathematical output of this encryption is the **Digital Signature**.
- Only the CA’s private key could have created this specific signature for this specific data hash.

### Step 4: Appending the Signature

The CA attaches this new **Digital Signature** to the bottom of the original plain-text bundle. The complete package—the
original data plus the CA's signature—is what we call a **Digital Certificate**.

```text
 [ Plain-Text Data + Public Key ]
               |
               v (Run through SHA-256)
          [ Data Hash ]
               |
               v (Encrypted with CA's Private Key)
      [ Digital Signature ]
               |
               +---> Appended to original data ---> [ Final Digital Certificate ]

```

---

## How it Looks Cryptographically (The Math)

Using standard asymmetric cryptography, the math looks like this:

$$Signature = \text{Encrypt}_{CA\_Private\_Key}(\text{Hash}(Data))$$

When a browser wants to verify this signature later, it does the inverse operation using the CA's widely available **Public
Key**:

$$Decrypted\_Hash = \text{Decrypt}_{CA\_Public\_Key}(Signature)$$

The browser then recalculates the hash of the plain-text data itself:

$$New\_Hash = \text{Hash}(Data)$$

If the $Decrypted\_Hash$ matches the $New\_Hash$, the browser knows two things with absolute certainty:

1. The data has not been modified by even a single letter (integrity).
2. It was definitely signed by that specific CA (authenticity).
