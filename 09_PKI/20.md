A digital certificate isn't just a random string of text; it is a highly structured file. Think of it as a digital ID card or
a high-tech passport.

Most digital certificates follow a strict international standard called **X.509**.

---

## 1. What a Certificate Looks Like (The Structure)

To a human, a raw certificate file looks like a scrambled block of text wrapped in specific headers. Depending on how it is
encoded, it generally looks like one of two things:

### PEM Format (Human-Readable Text)

This is the most common format you will see. It uses **Base64 ASCII encoding**, which means the binary data is translated
into text so it can be easily copied and pasted.

```text
-----BEGIN CERTIFICATE-----
MIIETjCCAzagAwIBAgIRAIvw9da8A3B8bV7vN3K8dBkwDQYJKoZIhvcNAQELBQAw
gY8xCzAJBgNVBAYTAlVTMRMwEQYDVQQIEwpDYWxpZm9ybmlhMRYwFAYDVQQHEw1T
YW4gRnJhbmNpc2NvMRUwEwYDVQQKEwxFeGFtcGxlIExMQzETMBEGA1UEAxMKRXhh
bXBsZSBDQTAeFw0yNjA2MTkwMDAwMDBaFw0yNzA2MTkwMDAwMDBaMGExCzAJBgNV
...[Hundreds of lines of random-looking alphanumeric characters]...
K3n8v9xP9zQ2wE4rR/5tXm9vB8s=
-----END CERTIFICATE-----

```

### DER Format (Binary)

If you were to open a DER file in a standard text editor, it would look like a corrupted mess of unreadable symbols and
strange characters (like `4^▒`). This is because it is stored in pure **binary format** (0s and 1s), which computers read
directly.

---

## 2. What is Included Inside a Certificate?

When a browser or operating system decodes that block of text, it uncovers a neat, organized set of fields defined by the
X.509 standard.

Here is exactly what is contained inside a standard digital certificate:

### Data Fields

- **Version:** Identifies which version of the X.509 standard the certificate uses (usually Version 3, which allows for
  custom extensions).
- **Serial Number:** A unique number assigned by the Certificate Authority (CA) to distinguish this certificate from all
  others it has issued.
- **Algorithm ID (Signature Algorithm):** The cryptographic algorithm the CA used to sign the certificate (e.g.,
  `SHA-256 with RSA`).
- **Issuer:** The identity of the CA that created and signed the certificate (e.g., _DigiCert Global Root CA_).
- **Validity Period:** The exact "Not Before" and "Not After" dates and times, defining when the certificate becomes active
  and when it expires.
- **Subject:** The identity of the owner of the certificate. For a website, this includes the **Common Name (CN)**, which is
  the actual domain name (e.g., `www.example.com`). It can also include the organization's name and location.
- **Subject's Public Key Info:** The actual public key being certified, along with the algorithm used to create it (e.g., RSA
  or ECC). **This is the core payload of the certificate.**

### X.509 Extensions (Version 3 additions)

- **Subject Alternative Name (SAN):** Allows the certificate to protect multiple domains (e.g., `example.com`,
  `m.example.com`, and `example.net` all on one certificate).
- **Key Usage:** Defines what the certificate can legally be used for (e.g., "Digital Signature," "Key Encipherment").

### The Digital Seal

- **Certificate Signature:** This is the cryptographic signature generated by the CA. The CA takes all the data listed above,
  hashes it, and encrypts it using the CA's own **private key**.

When your browser connects to a secure site, it reads the public key info, checks the expiration dates, and uses the CA's
well-known public key to verify that the **Certificate Signature** is valid. If it matches, the browser knows the data hasn't
been tampered with.
