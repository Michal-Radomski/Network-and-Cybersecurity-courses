## What is X.509?

**X.509** is a widely accepted international standard for defining the format of **public key certificates**. These
certificates are the digital equivalent of a driver's license or passport, used to securely prove ownership of a public
cryptographic key.

Managed by the International Telecommunication Union (ITU), X.509 is the foundational cornerstone of the **Public Key
Infrastructure (PKI)** used to secure internet traffic. When you see the padlock icon in your browser bar, X.509 certificates
are working behind the scenes.

---

## How It Works

An X.509 certificate binds a identity (like a website domain name, organization, or person) to a public key. A trusted third
party called a **Certificate Authority (CA)** digitally signs the certificate to verify its authenticity.

Here is the general lifecycle:

1. **Generation:** A user or server generates a key pair (a public key and a private key).
2. **Request:** The user sends a Certificate Signing Request (CSR) along with their public key to a CA.
3. **Issuance:** The CA verifies the user's identity and signs the certificate using the CA's own private key.
4. **Verification:** When you visit a secure site, your browser checks the CA's signature on the certificate. If the CA is
   trusted by your browser, the connection is deemed secure.

---

## Key Components of an X.509 Certificate

An X.509 certificate contains specific data fields to ensure its validity and security. The most common version in use today
is **Version 3**, which allows for custom extensions.

| Field                       | Description                                                                                               |
| --------------------------- | --------------------------------------------------------------------------------------------------------- |
| **Version**                 | Identifies which X.509 version applies (usually v3).                                                      |
| **Serial Number**           | A unique identifier assigned by the CA to distinguish it from other certificates.                         |
| **Algorithm ID**            | The cryptographic algorithm used by the CA to sign the certificate (e.g., SHA-256 with RSA).              |
| **Issuer**                  | The identity of the Certificate Authority that issued and signed the certificate.                         |
| **Validity Period**         | The "Not Before" and "Not After" dates establishing when the certificate is active and when it expires.   |
| **Subject**                 | The entity (website, company, individual) that owns the public key.                                       |
| **Subject Public Key Info** | The actual public key being certified, along with the algorithm it uses.                                  |
| **Extensions (v3)**         | Additional attributes, such as alternative domain names (SANs) or constraints on how the key can be used. |
| **Certificate Signature**   | The actual digital signature created by the CA's private key to prevent tampering.                        |

## Common Use Cases

X.509 certificates are everywhere in modern digital security:

- **HTTPS/TLS:** Securing web browsing by encrypting data sent between your browser and websites.
- **Email Encryption (S/MIME):** Signing and encrypting emails to ensure privacy and verify the sender.
- **Code Signing:** Allowing software developers to digitally sign applications so users know the software hasn't been
  modified by a third party.
- **SSH and VPNs:** Authenticating devices and users connecting to secure corporate networks.

> **Note:** While X.509 defines the _format_ of the certificate, the actual file extensions you see in the wild can vary
> depending on how they are encoded (e.g., `.pem`, `.crt`, `.der`, `.pfx`).
