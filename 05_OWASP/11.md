**Insufficient Logging and Monitoring** (now expanded into **Security Logging and Monitoring Failures** in the 2021 OWASP
Top 10) is often called the "silent" vulnerability. Unlike a SQL injection that leaves a clear hole, this is a failure of
visibility.

It represents the gap between a breach occurring and the organization actually noticing it. According to many industry
reports, the average time to detect a breach (dwell time) is often over 200 days—largely due to poor logging and monitoring.

---

### What Defines the Failure?

This vulnerability occurs when critical security events are not recorded, or when logs are generated but never analyzed or
acted upon. Common "missing" pieces include:

- **Invisible Actions:** Failing to log unsuccessful login attempts, password changes, or high-value transactions.
- **Local-Only Logs:** Storing logs only on the local server. If an attacker gains access, they can simply delete the logs to
  hide their tracks.
- **Vague Alerts:** Generating logs that lack context (e.g., "Error occurred") without specifying _who_ did it or _where_ it
  happened.
- **Threshold Blindness:** Not having alerts for suspicious patterns, such as 500 failed logins in one minute from a single
  IP.

---

### The Impact: "Dwell Time"

The danger isn't the lack of logs itself, but what that lack allows an attacker to do. Without proper monitoring, an attacker
can:

1. **Persist:** Stay in the system for months or years.
2. **Pivot:** Move laterally to more sensitive systems without being detected.
3. **Exfiltrate:** Slowly leak data in small batches that don't trigger basic bandwidth alarms.
4. **Tamper:** Modify or destroy data, and the company won't know which records are still trustworthy.

---

### What Should Be Logged?

A robust security posture requires logging at multiple levels of the application stack.

| Event Category       | Examples to Capture                                                                  |
| -------------------- | ------------------------------------------------------------------------------------ |
| **Authentication**   | Success, failure, and multi-factor authentication (MFA) bypass attempts.             |
| **Access Control**   | Unauthorized attempts to access restricted URLs or administrative functions.         |
| **Input Validation** | Hits on WAF rules or massive amounts of unexpected characters (potential injection). |
| **System Events**    | Critical service restarts, configuration changes, and API key usage.                 |

---

### Best Practices for Mitigation

To move from "blind" to "aware," organizations should implement the following:

- **Centralization:** Stream all logs to a central, secure location (like a SIEM—Security Information and Event Management
  system) that developers and attackers cannot easily modify.
- **Standardization:** Use a common log format (like JSON or Syslog) so that automated tools can easily parse the data.
- **Active Monitoring:** Use "Heartbeat" monitoring to ensure that your logging systems are actually running and haven't been
  disabled by an intruder.
- **Incident Response:** Logs are useless if no one is watching. Establish an Incident Response Plan (IRP) so that when a
  "Severity 1" alert triggers at 3:00 AM, there is a clear protocol for who to call.

---

### A Common Pitfall: Log Injection

Be careful not to trust the data you are logging! If you log a user's input directly, an attacker might provide a malicious
string (like `\nAdmin logged in`) to forge log entries or execute a **Log4Shell** style attack. Always neutralize or sanitize
data before it hits the log file.
