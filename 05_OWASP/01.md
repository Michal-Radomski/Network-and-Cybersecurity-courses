SQL Injection (SQLi) is a type of cyberattack where an attacker inserts malicious SQL code into an input field, which is then
executed by the backend database. This happens when an application fails to properly sanitize user input, allowing the
attacker to manipulate the original query's logic.

---

## How the Attack Works

In a vulnerable application, user input is often concatenated directly into a string. Imagine a login query like this:

`SELECT * FROM users WHERE username = '` + **userInput** + `' AND password = '` + **password** + `'`

If an attacker enters `' OR '1'='1` as the username, the resulting query becomes:

```sql
SELECT * FROM users WHERE username = '' OR '1'='1' AND password = '...';

```

Because `'1'='1'` is always true, the database may return the first user in the table (usually the admin), allowing the
attacker to bypass authentication without a password.

### Common SQLi Goals

- **Data Exfiltration:** Stealing sensitive information (emails, passwords, credit card numbers).
- **Data Modification:** Deleting or updating records.
- **Authentication Bypass:** Logging in as another user.
- **Remote Code Execution:** In some cases, taking over the entire database server.

---

## Prevention: The TypeScript Way

The most effective way to prevent SQLi is to **never concatenate user input into queries**. Instead, use prepared statements
or an Object-Relational Mapper (ORM).

### 1. Using Prepared Statements (Parameterized Queries)

With libraries like `pg` (PostgreSQL) or `mysql2`, you use placeholders (`?` or `$1`). The database driver treats the input
as **data**, not as executable code.

```typescript
import { Client } from "pg";

async function getUser(userId: string) {
  const client = new Client();
  await client.connect();

  // UNSAFE: NEVER DO THIS
  // const query = `SELECT * FROM users WHERE id = ${userId}`;

  // SAFE: Using parameterized queries
  const query = "SELECT * FROM users WHERE id = $1";
  const values = [userId];

  try {
    const res = await client.query(query, values);
    return res.rows[0];
  } catch (err) {
    console.error(err.stack);
  } finally {
    await client.end();
  }
}
```

### 2. Using an ORM (Prisma/TypeORM)

ORMs automatically handle parameterization for you, making it much harder to accidentally write a vulnerable query.

```typescript
import { PrismaClient } from "@prisma/client";

const prisma = new PrismaClient();

async function findUserByEmail(email: string) {
  // Prisma handles escaping and parameterization automatically
  return await prisma.user.findUnique({
    where: {
      email: email,
    },
  });
}
```

### 3. Input Validation

Always validate and sanitize input on the server side using libraries like **Zod** or **Joi**. If you expect a number, ensure
the input is a number before it ever touches your database logic.

```typescript
import { z } from "zod";

const UserInputSchema = z.object({
  userId: z.string().uuid(), // Ensures the input is a valid UUID
});

function processRequest(input: any) {
  const result = UserInputSchema.safeParse(input);
  if (!result.success) {
    throw new Error("Invalid Input");
  }
  // Proceed to database call
}
```

## Summary Table

| Method                  | Safety Level   | Why?                                                       |
| ----------------------- | -------------- | ---------------------------------------------------------- |
| **Concatenation**       | ❌ Dangerous   | Input is interpreted as SQL commands.                      |
| **Escaping Strings**    | ⚠️ Risky       | Easy to miss an edge case or use the wrong character set.  |
| **Prepared Statements** | ✅ Recommended | Separates code from data at the driver level.              |
| **ORMs**                | ✅ Recommended | Abstracts the SQL entirely and uses parameters by default. |
