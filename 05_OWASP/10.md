**Using Components with Known Vulnerabilities** (now part of the **Vulnerable and Outdated Components** category in the 2021
OWASP Top 10) is a risk that stems from the modern "building block" approach to software development.

Today, most applications are not written from scratch; they are assembled using open-source libraries, frameworks, and APIs.
If any of those underlying pieces have a security flaw, your entire application inherits that risk.

---

### The Nature of the Risk

This vulnerability is unique because it isn't necessarily a mistake in _your_ code, but rather a failure in your **Supply
Chain Management**.

- **Prevalence:** It is extremely common. A typical modern application can have hundreds of dependencies.
- **Transitive Dependencies:** You might use "Library A," but Library A relies on "Library B," which relies on "Library C." A
  vulnerability in Library C still puts you at risk.
- **Discovery:** Hackers don't have to find a new bug in your app; they just look at your "Ingredients List" (like a
  `package.json` or `pom.xml` file) and check if any of those versions have publicly disclosed vulnerabilities (CVEs).

---

### Common Attack Scenarios

| Scenario                     | How it happens                                                                                        |
| ---------------------------- | ----------------------------------------------------------------------------------------------------- |
| **Outdated Frameworks**      | Using an old version of Spring, Struts, or Express that has a known Remote Code Execution (RCE) flaw. |
| **Unpatched CMS**            | Running a WordPress or Drupal site without applying the latest security updates for plugins.          |
| **Insecure Transitive Deps** | Your app is secure, but a deep-level logging library you use is vulnerable to data injection.         |
| **Abandoned Projects**       | Using a library that is no longer maintained, meaning new bugs will never be fixed.                   |

---

### The "Equifax" Lesson

One of the most famous examples of this risk was the 2017 Equifax breach. The attackers didn't find a zero-day exploit; they
used a known vulnerability in the **Apache Struts** framework. A patch had been available for months, but Equifax had not
updated their systems, leading to the theft of data for over 140 million people.

---

### How to Defend Your Application

Prevention in this category is all about **Software Composition Analysis (SCA)** and constant vigilance.

1. **Inventory Management:** Maintain a "Software Bill of Materials" (SBOM). You can't protect what you don't know you have.
2. **Automated Scanning:** Use tools like `npm audit`, `Snyk`, or `GitHub Dependabot` to automatically scan your dependencies
   for known CVEs during development.
3. **Remove the Unused:** If a library isn't being used, remove it. It’s an unnecessary attack surface.
4. **Virtual Patching:** If a library is vulnerable but you can't update it immediately, use a Web Application Firewall (WAF)
   to block the specific exploit patterns targeted at that library.
5. **Policy-Based Selection:** Only use components from trusted, active sources that have a clear process for reporting and
   fixing security bugs.

---

### Pro-Tip: The "Latest" Trap

While staying updated is good, blindly updating to the "latest" version the second it drops can occasionally expose you to
**Supply Chain Attacks** (where a malicious actor hijacks a popular library and pushes a "poisoned" update). It’s often best
to wait a few days or use a "stable" version unless the update is a critical security patch.
