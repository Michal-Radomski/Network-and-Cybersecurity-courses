Insecure Deserialization is a critical security vulnerability that occurs when untrusted or manipulated data is used by an
application to inflict a DOS attack, execute code, or bypass authentication. While it was a standalone category in the 2017
OWASP Top 10, it is now part of the **Software and Data Integrity Failures** category in the most recent 2021 update.

To understand this vulnerability, we first need to define the two core processes involved:

1. **Serialization:** Converting an object in memory (like a user profile or a shopping cart) into a format that can be
   stored or transmitted (like JSON, XML, or binary data).
2. **Deserialization:** Taking that stored format and rebuilding it back into a live object in the application’s memory.

---

### How the Attack Works

The vulnerability arises when an application deserializes data from an untrusted source without sufficient validation. An
attacker can "poison" the serialized data, modifying object properties or even injecting malicious "gadgets" (segments of
existing code) that execute during the reconstruction process.

#### The "Magic" of Gadget Chains

In many programming languages (like Java, PHP, or Python), objects have "magic methods" that trigger automatically during
deserialization (e.g., `readObject()` in Java or `__wakeup()` in PHP). Attackers chain these methods together—known as a
**Gadget Chain**—to reach a dangerous sink, such as a command that executes a shell script.

---

### Common Impact

| Impact Type                     | Description                                                                            |
| ------------------------------- | -------------------------------------------------------------------------------------- |
| **Remote Code Execution (RCE)** | The most severe outcome. The attacker runs arbitrary commands on the server.           |
| **Privilege Escalation**        | An attacker modifies a serialized cookie to change their role from `user` to `admin`.  |
| **Denial of Service (DoS)**     | Sending a "billion laughs" style payload or a massive object that exhausts CPU/Memory. |
| **Data Access**                 | Bypassing access controls to read sensitive database records or files.                 |

---

### Real-World Example: The "Admin" Cookie

Imagine an application stores a user's session in a serialized PHP object within a cookie:

- **Legitimate Cookie:** `O:4:"User":2:{s:8:"username";s:5:"alice";s:7:"isAdmin";b:0;}`
- **Malicious Cookie:** `O:4:"User":2:{s:8:"username";s:5:"alice";s:7:"isAdmin";b:1;}`

By simply changing the boolean value of `isAdmin` from `0` to `1` and re-sending the cookie, the attacker tricks the
application into granting them administrative rights upon deserialization.

---

### Prevention and Mitigation

The most effective way to prevent this is to **never accept serialized objects from untrusted sources.** If you must
transport data, use a pure data format like JSON or XML that does not include object logic.

- **Integrity Checks:** Use digital signatures (like HMAC) to ensure the serialized data hasn't been tampered with.
- **Strict Type Constraints:** During deserialization, enforce a strict "allow-list" of expected classes.
- **Isolation:** Run the deserialization process in a low-privilege environment or container.
- **Monitoring:** Log deserialization exceptions and failures, which are often early indicators of an attack.
