## Sensitive Data Exposure in Web Applications

**Sensitive Data Exposure** occurs when an application fails to adequately protect critical information, allowing
unauthorized parties to access it. This isn't usually the result of a complex "hack" into a database; rather, it’s often a
failure of encryption—either in transit (moving over the internet) or at rest (stored on a server).

In the modern cybersecurity landscape, this is frequently categorized under **Broken Access Control** or **Cryptographic
Failures** in the OWASP Top 10.

---

### 1. What Qualifies as "Sensitive Data"?

Data is considered sensitive if its disclosure could result in identity theft, financial loss, or a breach of privacy. Common
examples include:

- **PII (Personally Identifiable Information):** Names, home addresses, and government IDs.
- **Financial Data:** Credit card numbers, CVVs, and banking details.
- **Health Information:** Medical records and insurance details.
- **Authentication Credentials:** Plaintext passwords, session tokens, and API keys.

---

### 2. Common Vulnerability Scenarios

#### A. Data in Transit (The "Man-in-the-Middle")

If a web application uses **HTTP** instead of **HTTPS**, data travels across the network in plaintext. Anyone sitting on the
same Wi-Fi network or controlling a router in the middle can "sniff" the traffic and read everything from login credentials
to credit card numbers.

#### B. Data at Rest (The Unencrypted Database)

If a database is breached and the information inside is stored in plaintext, the damage is absolute. A common mistake is
storing passwords without a strong cryptographic hash, or storing credit card numbers without encryption.

#### C. Insecure Backups and Logs

Sometimes the main application is secure, but the **backups** or **server logs** are stored in a publicly accessible
directory. Developers might accidentally log sensitive POST data (like passwords) to a text file for debugging and forget to
disable it in production.

---

### 3. How Exposure Happens: Real-World Examples

| Scenario                | The Mistake                                        | The Consequence                                                   |
| ----------------------- | -------------------------------------------------- | ----------------------------------------------------------------- |
| **Cleartext Protocols** | Using FTP or Telnet instead of SFTP or SSH.        | Credentials intercepted via packet sniffing.                      |
| **Weak Hashing**        | Using MD5 or SHA-1 for passwords.                  | Hackers can "crack" the hashes in seconds using rainbow tables.   |
| **Missing Headers**     | Failing to use `Strict-Transport-Security` (HSTS). | Users can be downgraded to an insecure HTTP connection.           |
| **Browser Caching**     | Not setting `Cache-Control: no-store`.             | Sensitive data is saved on a shared public computer's hard drive. |

---

### 4. Prevention Strategies

To protect sensitive data, developers should adopt a "Defense in Depth" approach:

1. **Classify Your Data:** Identify which data is sensitive according to privacy laws (GDPR, CCPA) and treat it with higher
   security tiers.
2. **Don't Store What You Don't Need:** The most secure data is the data you never collected. Discard PII as soon as it's no
   longer necessary.
3. **Encrypt Everything in Transit:** Use **TLS 1.2 or 1.3** for all connections and enforce it with HSTS headers.
4. **Encrypt Everything at Rest:**

- **Passwords:** Use "salted" slow-hashing algorithms like **Argon2** or **bcrypt**.
- **General Data:** Use strong encryption standards like **AES-256**.

5. **Disable Autocomplete and Caching:** Ensure that sensitive forms (like credit card inputs) tell the browser not to store
   the data.

---

> **Note:** Sensitive Data Exposure is often the "payload" of other attacks. For instance, a **SQL Injection** might be the
> method used to enter a system, but the **Sensitive Data Exposure** (lack of encryption) is what makes the stolen data
> useful to the attacker.
