**Broken Access Control** has claimed the #1 spot on the **OWASP Top 10** for a reason: it is the most common and often most
damaging security vulnerability in modern web applications.

At its core, it occurs when an application fails to properly enforce restrictions on what authenticated users are allowed to
do. While **authentication** confirms _who_ you are, **access control (authorization)** determines _what_ you can do. When
this is "broken," users can act outside of their intended permissions.

---

## Common Types of Access Control Failures

Broken access control isn't just one bug; it’s a category of several different logic failures.

### 1. Vertical Privilege Escalation

This happens when a lower-privileged user (like a "Customer") gains access to functions reserved for higher-privileged users
(like an "Admin").

- **Example:** A regular user discovers they can access the admin panel simply by typing `example.com/admin` into their
  browser because the page doesn't check for an admin session.

### 2. Horizontal Privilege Escalation (IDOR)

Insecure Direct Object Reference (IDOR) occurs when a user accesses the resources of another user with the same level of
privilege.

- **Example:** A user logged in as `User A` changes the URL from `example.com/account/105` to `example.com/account/106` and
  is suddenly able to view `User B`’s private data.

### 3. Metadata Manipulation

Attackers may tamper with pieces of data—like JSON Web Tokens (JWTs), cookies, or hidden form fields—that the application
uses to make authorization decisions.

- **Example:** A user modifies a cookie that says `isAdmin=false` to `isAdmin=true` to bypass security checks.

### 4. CORS Misconfiguration

Cross-Origin Resource Sharing (CORS) allows servers to specify who can access their resources. If misconfigured (e.g., using
a wildcard `*`), it can allow unauthorized domains to perform actions on behalf of a logged-in user.

---

## Why It Happens

Most access control issues stem from a **"Permit by Default"** mindset rather than a **"Deny by Default"** one. Other causes
include:

- **Relying on "Security through Obscurity":** Assuming a user won't find a hidden URL.
- **Client-side enforcement:** Checking permissions in JavaScript (which can be bypassed) instead of on the server.
- **Lack of Centralization:** Hardcoding access checks into every individual page rather than using a central security
  filter.

---

## How to Prevent It

Securing an application requires a disciplined approach to authorization logic:

| Strategy                       | Action                                                                                                             |
| ------------------------------ | ------------------------------------------------------------------------------------------------------------------ |
| **Deny by Default**            | All resources should be inaccessible unless explicitly allowed.                                                    |
| **Centralize Logic**           | Use a single, well-tested library or middleware to handle all authorization checks.                                |
| **Server-Side Validation**     | Never trust the client. Re-verify the user's identity and permissions on every single request.                     |
| **Disable Directory Browsing** | Ensure web server metadata (like `.git` folders) and directory listings aren't public.                             |
| **Log and Alert**              | Monitor for repeated 403 (Forbidden) errors, which could indicate an attacker is probing for IDOR vulnerabilities. |

---

> **The Golden Rule:** Access control is only effective if it is enforced on the **server-side** for every single request
> that touches sensitive data or functions.
