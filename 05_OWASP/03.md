Broken authentication is a critical security vulnerability that occurs when an application's functions related to **session
management** or **user identity** are implemented incorrectly. This allows attackers to compromise passwords, keys, or
session tokens, or to exploit other implementation flaws to assume other users' identities.

In the OWASP Top 10, this is often grouped under **Identification and Authentication Failures**.

---

## How It Works

At its core, authentication is how a web app verifies you are who you say you are. If the "gatekeeper" is lazy or
predictable, an attacker can bypass the lock entirely.

### Common Vulnerabilities

- **Credential Stuffing:** Using lists of leaked usernames and passwords from other breaches to gain access, assuming users
  reuse passwords.
- **Brute Force Attacks:** Testing thousands of password combinations because the app lacks rate-limiting or "account
  lockout" features.
- **Permissive Password Requirements:** Allowing weak passwords like `password123` or `admin`.
- **Session Fixation:** An attacker provides a victim with a specific Session ID, and once the victim logs in, the attacker
  uses that same ID to hijack the account.
- **Exposed Session IDs:** Putting session tokens in the URL (e.g., `example.com/dashboard?session_id=12345`), which can be
  leaked via browser history or "Referer" headers.

---

## Real-World Scenarios

### 1. The "Remember Me" Flaw

An application might store a user's ID in a persistent cookie without any encryption or signature. If the cookie is
`user_id=101`, an attacker can simply change it to `user_id=102` to log in as someone else.

### 2. Poor Session Timeout

If a user logs out of a public computer but the application doesn't actually invalidate the session on the server, the next
person to use that computer can hit the "Back" button and stay logged in.

### 3. Weak Password Reset

An app that asks for a "Favorite Color" as the only way to reset a password is fundamentally broken because that information
is easily found or guessed.

## How to Prevent It

To build a secure authentication system, developers generally follow these best practices:

| Feature                | Best Practice                                                          |
| ---------------------- | ---------------------------------------------------------------------- |
| **Multi-Factor (MFA)** | Implement MFA to ensure a stolen password isn't enough to gain access. |
| **Password Hashing**   | Use strong, salted algorithms like **Argon2** or **bcrypt**.           |
| **Rate Limiting**      | Block or slow down requests after multiple failed login attempts.      |
| **Session Security**   | Use built-in session managers that generate long, random IDs.          |
| **Cookie Flags**       | Set `HttpOnly` (prevents JS access) and `Secure` (requires HTTPS).     |

---

### Important Distinction

Broken Authentication is different from **Broken Access Control**.

- **Authentication:** "Are you really User A?"
- **Access Control:** "User A is logged in, but are they allowed to delete User B's files?"

Failure in the first usually leads to a total failure in the second.
