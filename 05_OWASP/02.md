## Understanding Command Injection

**Command Injection** is a critical security vulnerability that occurs when an application passes unsafe user-supplied data
(forms, cookies, HTTP headers, etc.) to a system shell. An attacker can use this to execute arbitrary operating system (OS)
commands on the server, often with the privileges of the application.

This typically happens when a developer intends to use a system utility (like `ping` or `mkdir`) and concatenates user input
directly into the command string.

### How the Attack Works

Attackers use **shell metacharacters** to chain commands together. Common separators include:

- **Linux/Unix:** `;`, `&&`, `||`, `|`, ```, `$( )`
- **Windows:** `&`, `&&`, `||`, `|`

**Example Scenario:** An app takes a filename to delete it: `rm ` + `user_input`. If a user enters `file.txt; rm -rf /`, the
shell executes:

1. `rm file.txt`
2. `rm -rf /` (The disaster)

---

## Prevention Strategies

The golden rule of web security is: **Never trust user input.**

1. **Avoid Shell Execution:** Use built-in language APIs (like `fs.unlink` in Node.js) instead of spawning a shell command.
2. **Input Validation:** Use strict allow-lists (e.g., only alphanumeric characters).
3. **Use `execFile` or `spawn`:** These functions treat arguments as data, not executable code, preventing shell
   interpretation.
4. **Escaping/Sanitization:** If you _must_ use a shell, use libraries specifically designed to escape shell characters.

## TypeScript Implementation Examples

Below are examples of how to safely handle system-level tasks in a Node.js/TypeScript environment.

### 1. The Safe Way (Using `execFile`)

`execFile` is generally safer than `exec` because it does not spawn a shell by default. The arguments are passed as an array,
so the OS treats them as literal strings rather than part of a command.

```typescript
import { execFile } from "child_process";

/**
 * Safely runs a 'ping' command.
 * @param target - User provided IP or hostname
 */
function safePing(target: string): void {
  // 1. Strict Validation (Allow-list)
  const dnsRegex = /^[a-zA-Z0-9.-]+$/;
  if (!dnsRegex.test(target)) {
    throw new Error("Invalid target format");
  }

  // 2. Use execFile: Arguments in the array are NOT parsed by a shell
  // Windows: 'ping.exe' | Linux: 'ping'
  const command = process.platform === "win32" ? "ping.exe" : "ping";
  const args = process.platform === "win32" ? ["-n", "1", target] : ["-c", "1", target];

  execFile(command, args, (error, stdout, stderr) => {
    if (error) {
      console.error(`Error: ${error.message}`);
      return;
    }
    console.log(`Output: ${stdout}`);
  });
}
```

### 2. Handling Windows vs. Linux Paths

If you are generating directories or files based on user input, avoid `mkdir` commands. Use the `fs` module instead.

```typescript
import * as fs from "fs";
import * as path from "path";

/**
 * Safely creates a user directory.
 * Prevents "Directory Traversal" and "Command Injection".
 */
function createUserDirectory(userInput: string): void {
  // Sanitize: Remove everything except alphanumeric and underscores
  const safeName = userInput.replace(/[^a-z0-9_]/gi, "");

  // Define a base path to prevent escaping the intended folder
  const baseDir = path.join(__dirname, "uploads");
  const finalPath = path.join(baseDir, safeName);

  if (!fs.existsSync(finalPath)) {
    // No shell involved, no injection possible
    fs.mkdirSync(finalPath, { recursive: true });
    console.log(`Directory created at: ${finalPath}`);
  }
}
```

### 3. Comparison Table: `exec` vs. `execFile`

| Feature      | `child_process.exec`                          | `child_process.execFile`            |
| ------------ | --------------------------------------------- | ----------------------------------- |
| **Shell**    | Spawns a shell (e.g., `/bin/sh` or `cmd.exe`) | Does **not** spawn a shell          |
| **Risk**     | High (Shell interprets `;`, `&`, etc.)        | Low (Arguments are passed directly) |
| **Use Case** | Complex shell piping/redirection              | Standard executable tasks           |
| **Security** | Requires manual, complex escaping             | Secure by design                    |

> **Security Note:** Even with `execFile`, you should still implement **Input Validation**. While it prevents command
> injection, it doesn't prevent "Argument Injection" (where an attacker passes a valid flag to the program that changes its
> behavior in an unintended way).
