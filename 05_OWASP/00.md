The **OWASP Top 10** is the gold-standard awareness document for developers and web application security. It represents a
broad consensus on the most critical security risks to web applications.

As of 2026, the community has transitioned to the **2025/2026 Updated List**, which reflects the modern shift toward
cloud-native environments and the software supply chain.

---

### 1. Broken Access Control

This remains the #1 risk. It occurs when users can act outside of their intended permissions.

- **The Risk:** An attacker accessing another user's account, viewing sensitive files, or gaining admin privileges.
- **Example:** Changing a URL from `example.com/app/user/123` to `.../user/124` to see someone else’s data.
- **Prevention:** Deny by default. Enforce record-level ownership and use unique, unguessable identifiers (UUIDs).

### 2. Security Misconfiguration

This jumped significantly in recent rankings due to the complexity of cloud and containerized environments.

- **The Risk:** Using default passwords, leaving unnecessary features enabled, or misconfigured cloud storage (e.g., open S3
  buckets).
- **Example:** A web server showing verbose error messages that reveal the database version and internal file paths.
- **Prevention:** Automate the "hardening" process and remove any unused features or samples from the software.

### 3. Cryptographic Failures

Focuses on the protection of data in transit and at rest.

- **The Risk:** Using weak hashing (like MD5), transmitting data over unencrypted HTTP, or failing to rotate keys.
- **Example:** Storing user passwords in plain text or using a predictable encryption key.
- **Prevention:** Use modern, strong algorithms (AES-256, Argon2) and ensure TLS is enforced everywhere.

### 4. Injection

While legendary (like SQL Injection), it has dropped slightly because modern frameworks prevent it by default.

- **The Risk:** Untrusted data is sent to an interpreter as part of a command or query.
- **Example:** A login form where entering `' OR 1=1 --` bypasses the password check.
- **Prevention:** Use parameterized queries (Prepared Statements) and strictly validate all user input.

### 5. Insecure Design

This is a "Shift Left" category focusing on flaws in the **architecture** rather than just the code.

- **The Risk:** The design itself is flawed, even if the code is written perfectly.
- **Example:** A banking app that doesn't require MFA for high-value transfers, or a "forgot password" flow that leaks if an
  email exists.
- **Prevention:** Use threat modeling during the design phase and follow secure design patterns.

### 6. Software Supply Chain Failures

An evolution of "Vulnerable and Outdated Components," this now covers the entire CI/CD pipeline.

- **The Risk:** Using a third-party library that contains a backdoor or a build server that has been compromised.
- **Example:** The SolarWinds or XZ Utils incidents, where malicious code was injected into legitimate software updates.
- **Prevention:** Use a Software Bill of Materials (SBOM), verify digital signatures, and scan dependencies automatically.

### 7. Identification and Authentication Failures

Deals with how the application identifies the user.

- **The Risk:** Credential stuffing (using leaked passwords from other sites) or weak session management.
- **Example:** Allowing "123456" as a password or not expiring a session token after a user logs out.
- **Prevention:** Implement Multi-Factor Authentication (MFA) and enforce strong password complexity.

### 8. Software and Data Integrity Failures

This focuses on code and data that is used without verifying its integrity.

- **The Risk:** Insecure deserialization or malicious updates.
- **Example:** An application that accepts serialized objects from a user, allowing them to execute remote code on the
  server.
- **Prevention:** Ensure digital signatures are used to verify that data or code hasn't been tampered with.

### 9. Security Logging and Monitoring Failures

You can't stop what you can't see.

- **The Risk:** Not logging failed login attempts or critical errors, leading to "silent" breaches that last for months.
- **Example:** An attacker tries 10,000 passwords, but the system never alerts the admin.
- **Prevention:** Log all authentication, authorization, and server-side validation failures. Set up real-time alerts.

### 10. Mishandling of Exceptional Conditions

(A recent addition/refinement) focuses on how the system reacts when things go wrong.

- **The Risk:** A system "failing open" (granting access when an error occurs) or leaking secrets in a crash report.
- **Example:** A database connection error causes the application to bypass a permission check entirely.
- **Prevention:** Ensure "fail-safe" defaults and sanitize all error messages before they reach the user.

---

> **Note:** The OWASP Top 10 is a starting point, not a complete security program. It is designed to help you prioritize the
> most likely and impactful threats to your application.
