Both **GPG keys** and **SSH keys** are tools used in cryptography to secure data and verify identity, but they are built for
entirely different jobs.

Here is a breakdown of what a GPG key is and how it differs from an SSH key.

---

## What is a GPG Key?

**GPG (GNU Privacy Guard)** is a free implementation of the OpenPGP standard. It is primarily used for **encrypting data**
(like emails or files) and **signing digital content** to prove it hasn't been tampered with.

A GPG key always comes as a pair:

- **Public Key:** You share this with the world. Others use it to encrypt messages to you or to verify your digital
  signature.
- **Private Key:** You keep this strictly secret. You use it to decrypt messages sent to you or to digitally sign your own
  work.

### Common Uses for GPG:

- **Signing Git Commits:** Verifying on GitHub/GitLab that code actually came from you and not an impersonator.
- **Email Encryption:** Sending secure, unreadable emails that only the intended recipient can decrypt.
- **Signing Software Packages:** Linux distributions use GPG to verify that software updates are authentic and safe.

---

## What is the Difference Between GPG and SSH Keys?

While both use asymmetric cryptography (public/private key pairs), their core purposes and how they manage trust are
fundamentally different.

| Feature                  | GPG Key                                                                                 | SSH Key                                                                             |
| ------------------------ | --------------------------------------------------------------------------------------- | ----------------------------------------------------------------------------------- |
| **Primary Purpose**      | **Data Integrity & Confidentiality** (Encrypting files/emails and signing data/code).   | **Authentication** (Logging into remote servers securely without a password).       |
| **Main Use Case**        | Signing Git commits, encrypting emails, verifying software packages.                    | Connecting to a remote server via terminal, pushing code to GitHub.                 |
| **How Trust is Handled** | **Web of Trust (WoT):** Users sign each other's keys to verify identities peer-to-peer. | **Identity-based:** A specific server explicitly trusts your specific public key.   |
| **Scope**                | Tied to an **identity** (usually your name and email address).                          | Tied to a **machine/session** (your specific computer accessing a specific server). |
| **Lifetime**             | Often includes an expiration date and can contain multiple subkeys.                     | Generally does not expire automatically (unless configured by an admin).            |

### The Analogy: Passport vs. House Key

- **GPG is like a Passport:** It is tied to your identity (Name, Email). It has an expiration date, and authorities (or
  friends) stamp it to prove you are who you say you are. You use it to sign contracts (Git commits) and seal envelopes
  (encrypt files).
- **SSH is like a House Key:** It doesn't care who you are; it just unlocks a specific door. You place your public key on a
  server's door, and when your computer shows up with the matching private key, the door opens.

---

### Can they be used interchangeably?

Technically, they use similar underlying math (like RSA or Ed25519), and you _can_ actually configure GPG to act as an SSH
authentication agent. However, out of the box, you cannot use an SSH key to sign a GPG message or vice versa, because their
file formats and metadata are entirely different.
