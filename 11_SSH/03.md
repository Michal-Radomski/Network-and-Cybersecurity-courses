Yes, you can absolutely have **multiple pairs** of SSH keys in your `~/.ssh` directory. In fact, it is a very common best
practice to use different keys for different services (like one for GitHub, one for a work server, and another for a personal
VPS).

Here is how Ubuntu (and the SSH client in general) figures out which key to use.

---

### 1. The Default Approach: "Try Everything"

By default, if you don't tell Ubuntu which key to use, it will look in your `~/.ssh` directory and automatically try standard
default filenames one by one, such as:

- `id_rsa`
- `id_ecdsa`
- `id_ed25519`

If you have custom-named keys (e.g., `github_key` and `work_server_key`), SSH won't know they exist automatically unless you
tell it, or unless it tries every key currently loaded into your **SSH Agent** (a background program that holds your
decrypted keys).

However, relying on SSH to try every key can cause issues. If you have too many keys, the remote server might reject your
connection for making too many failed authentication attempts.

---

### 2. The Smart Approach: The `config` File

The absolute best way to manage multiple keys and tell Ubuntu exactly which key goes where is by creating an **SSH config
file**.

You can create a file named `config` (no extension) inside your `~/.ssh/` directory:

```bash
nano ~/.ssh/config

```

Inside this file, you define specific rules for different services. For example:

```text
# Rules for GitHub
Host github.com
    HostName github.com
    User git
    IdentityFile ~/.ssh/github_key

# Rules for a specific work server
Host work-server
    HostName 192.168.1.50
    User ubuntu
    IdentityFile ~/.ssh/work_key
    IdentitiesOnly yes

```

### Why this works perfectly:

- **`Host`**: This is a nickname you use in your terminal (e.g., `ssh work-server`).
- **`IdentityFile`**: This tells Ubuntu exactly which private key file to use for this specific connection.
- **`IdentitiesOnly yes`**: This is a crucial setting. It forces Ubuntu to _only_ use the key specified in `IdentityFile`,
  preventing it from guessing or trying other keys.

Now, when you run `ssh work-server`, Ubuntu reads the config file, sees that it needs to use `~/.ssh/work_key`, and connects
seamlessly without any guesswork.
