The **Google Hacking Database (GHDB)** is a categorized, publicly accessible collection of Google search queries (commonly
known as "Google Dorks") designed to uncover exposed, sensitive data or misconfigured assets indexed on the public web.

Originally compiled in 2004 by cybersecurity researcher Johnny Long, the database is currently maintained by **Offensive
Security (OffSec)** as part of the Exploit Database (Exploit-DB).

---

## How the Database Is Categorized

The GHDB organizes queries into distinct categories based on what type of exposure the search term targets:

- **Files Containing Passwords:** Queries that search for inadvertently exposed plain-text credentials, configuration files,
  or backup archives.
- **Sensitive Directories:** Searches that reveal unlinked server directories or open directory listings (`Index of /`).
- **Web Server Detection:** Queries aimed at identifying specific web servers, operating systems, or service versions.
- **Vulnerable Files / Applications:** Queries that find known vulnerable web applications, outdated CMS portals, or
  unpatched software interfaces.
- **Pages Containing Login Portals:** Searches that map out administrative login interfaces across subdomains.
- **Network or Hardware Data:** Queries targeting connected hardware devices, such as network cameras, printer management
  pages, or router interfaces.

---

## Dual-Use Purpose

Like many cybersecurity tools, the database serves two main functions depending on who is using it:

- **Defensive Auditing (Blue Teams & System Administrators):** Security teams use the GHDB to check if their organization's
  domains, credentials, or internal documents have been accidentally indexed by Google, allowing them to fix configuration
  errors.
- **Passive Reconnaissance (Red Teams & Attackers):** Security testers and threat actors use the cataloged queries during
  initial reconnaissance to gather intelligence without directly interacting with or scanning the target server's
  infrastructure.

---

## Remediation Best Practices

To prevent sensitive assets from being cataloged in databases like the GHDB, organizations typically implement the following
controls:

1. **Strict Access Controls:** Place administrative panels, database files, and internal staging environments behind proper
   authentication rather than relying on obscurity.
2. **Proper Header Tags:** Use HTTP headers (such as `X-Robots-Tag: noindex`) or `<meta name="robots" content="noindex">`
   tags on pages that should not appear in search engine indexes.
3. **Google Search Console Removal Tools:** Request the immediate removal of accidentally indexed sensitive pages or cached
   URL copies via Google's administrative console.
