**ARP** stands for **Address Resolution Protocol**.

If the Default Gateway is the "exit door" of your network, ARP is the tool that helps devices find each other _inside_ the
room before they even reach the door.

In short: **IP addresses** are used to route data across different networks (the internet), but **MAC addresses** (the
physical hardware address burned into a network card) are used to deliver data to the exact machine on a local network. ARP
is the bridge between them. It translates an IP address into a MAC address.

---

## The "Who Has This?" Analogy

Imagine you are in a crowded classroom. You have a physical letter to give to "John Smith," but you don't know what John
Smith looks like.

1. You stand up and shout to the whole room: **"Who is John Smith? Please tell me your seat number!"** (This is an ARP
   Request).
2. Everyone hears you, but only the real John Smith stands up and replies: **"I am John Smith, and I am in Seat 4B."** (This
   is an ARP Reply).
3. You write down "John Smith = Seat 4B" in your notebook so you don't have to shout next time. (This is the ARP Cache).

---

## How the Protocol Works

When a computer wants to talk to an IP address on its local network, it goes through three steps:

### 1. Checking the ARP Cache

Before sending anything, the computer checks its own temporary internal list called the **ARP Cache** (or ARP Table). If it
already knows the MAC address for that IP, it sends the data immediately.

### 2. The ARP Request (The Broadcast)

If the IP isn't in the cache, the computer sends out an **ARP Request**.

- Because it doesn't know the target's MAC address, it uses a destination MAC address of `FF:FF:FF:FF:FF:FF` (the universal
  broadcast address).
- Every single device on that local subnet receives and reads this packet. The packet says: _"Who has IP 192.168.1.50? Tell
  192.168.1.10."_

### 3. The ARP Reply (The Unicast)

Every device inspects the packet. If the IP isn't theirs, they ignore it. However, the device that actually owns
`192.168.1.50` responds with an **ARP Reply**. This reply is a _unicast_ (sent directly back to the requester, not broadcast
to everyone). It says: _"Hey, I have that IP! My MAC address is 00:AA:BB:CC:DD:EE."_

## Anatomy of an ARP Packet

An ARP packet is incredibly lightweight because it doesn't carry internet data (like web pages or files); it only carries
addressing information.

Inside an ARP packet, you will find these key fields:

- **Hardware Type:** The type of network protocol being used (almost always **Ethernet**).
- **Protocol Type:** The network layer protocol being mapped (almost always **IPv4**).
- **Hardware & Protocol Size:** The length (in bytes) of the MAC and IP addresses.
- **Opcode (Operation Code):** A number that tells the receiving device what kind of ARP packet this is.
- `1` = ARP Request
- `2` = ARP Reply

- **Sender Hardware Address:** The MAC address of the device sending the packet.
- **Sender Protocol Address:** The IP address of the device sending the packet.
- **Target Hardware Address:** The MAC address of the destination. _(In an ARP Request, this field is left blank or filled
  with zeros, because the sender doesn't know it yet)._
- **Target Protocol Address:** The IP address of the destination.

Once the ARP packet completes its job, the original data packet (the web request, the email, etc.) is finally wrapped in an
Ethernet frame with the correct destination MAC address and fired across the cable.
