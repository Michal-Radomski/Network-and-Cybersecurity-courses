When data travels across multiple routers—such as across a large corporate network or the global Internet—the process is
called **multi-hop routing**.

Instead of a single router making a direct delivery, multiple routers form a chain. Each individual router is only
responsible for handing the data off to the next router in line (the **"Next Hop"**) until the packet finally reaches a
router that is directly connected to the destination network.

Here is how multi-hop routing looks in practice, tracing a packet's journey across a chain of routers.

---

## The Network Topology

Imagine a user on **Network A** trying to access a server on **Network Z**. They are separated by three different routers:

```text
[Host A] ---> (Router 1) ---> (Router 2) ---> (Router 3) ---> [Server Z]

```

- **Host A IP:** `192.168.1.5`
- **Server Z IP:** `10.0.0.99`

---

## The Packet's Journey (Step-by-Step)

### Hop 1: Host A to Router 1

Host A realizes the Server’s IP (`10.0.0.99`) is on a remote network. It packages the data and sends it to its local gateway,
Router 1.

- **Layer 3 (IP):** Source = `192.168.1.5` | Destination = `10.0.0.99`
- **Layer 2 (MAC):** Source = Host A | Destination = Router 1

### Hop 2: Router 1 to Router 2

Router 1 receives the data and strips away the Layer 2 MAC headers to look at the IP destination (`10.0.0.99`). It checks its
routing table. The table says: _"To reach 10.0.0.0/24, send the packet to Router 2."_

- **Layer 3 (IP):** Source = `192.168.1.5` | Destination = `10.0.0.99` _(Unchanged)_
- **Layer 2 (MAC):** Source = Router 1 | Destination = **Router 2** _(Rewritten)_

### Hop 3: Router 2 to Router 3

Router 2 receives the packet, strips the MAC layer, and checks its routing table. Its table says: _"To reach 10.0.0.0/24,
forward to Router 3."_

- **Layer 3 (IP):** Source = `192.168.1.5` | Destination = `10.0.0.99` _(Unchanged)_
- **Layer 2 (MAC):** Source = Router 2 | Destination = **Router 3** _(Rewritten)_

### Hop 4: Router 3 to Server Z

Router 3 strips the MAC layer and looks at the IP destination. It sees that `10.0.0.0/24` is **directly connected** to its
own local interface. It uses ARP to find Server Z's MAC address and delivers the packet directly.

- **Layer 3 (IP):** Source = `192.168.1.5` | Destination = `10.0.0.99` _(Unchanged)_
- **Layer 2 (MAC):** Source = Router 3 | Destination = **Server Z** _(Rewritten)_

---

## What a Multi-Router Routing Table Looks Like

For this to work, the intermediate routers (like Router 2) don't need to know the exact physical location of Server Z. They
only need to know which direction to point the packet.

An entry in **Router 2's** routing table would look like this:

| Destination Network | Subnet Mask     | Next Hop (Gateway)      | Output Interface |
| ------------------- | --------------- | ----------------------- | ---------------- |
| `10.0.0.0`          | `255.255.255.0` | `172.16.2.3` (Router 3) | `WAN_port2`      |

---

## Two Golden Rules of Multi-Hop Routing

1. **The IP Envelope Never Changes:** The source and destination IP addresses remain exactly the same throughout the entire
   journey. They ensure the packet knows its ultimate origin and final destination.
2. **The MAC Layer changes at _every single hop_:** Layer 2 hardware addresses (MAC addresses) are strictly local. They are
   stripped off and rewritten by every single router along the path to get the packet across the immediate physical link to
   the next router.

### The Time-to-Live (TTL) Safety Net

To prevent a packet from looping forever if there is a mistake in the routing tables, the IP header contains a **TTL
(Time-to-Live)** value (e.g., 64). Every single router the packet passes through subtracts `1` from this value. If the TTL
hits `0` before reaching the destination, the current router drops the packet and sends an error back to the host. This
prevents broken networks from being choked by endless, looping data.
